# functional-parallelism-api

**Parent:** [[content/L1/functional-algebra-and-concurrency-patterns|functional-algebra-and-concurrency-patterns]] — The functional API structure is an algebra governed by laws, such as `map(unit(1))(_ + 1) == unit(2)` and `map(y)(id) == y`. Concurrency relies on a non-blocking `Par` representation, solving a deadlock issue in `fork` when using fixed-size thread pools (e.g., `Executors.newFixedThreadPool(1)`) by requiring a fully non-blocking design, often implemented using Actors for non-blocking `map2` and ensuring that combinations like `parMap(List.range(1, 100000))(math.sqrt(_))` do not require a massive number of threads.

To construct a parallel computation, a simple initial implementation is the `unit` combinator: `def unit[A](a: A): Par[A] = es => new Future[A] { def apply(cb: A => Unit): Unit = cb(a) }`. This implementation avoids needing an explicit `ExecutorService` because it simply passes the available value `a` to the continuation `cb`. This immediately releases any latch and makes the result available if the continuation is part of the `run` implementation.

For true parallelism, the `fork` combinator is needed. The definition is: `def fork[A](a: => Par[A]): Par[A] = es => new Future[A] { def apply(cb: A => Unit): Unit = eval(es)(a(es)(cb)) }`. Here, the `eval` helper function, defined as `def eval(es: ExecutorService)(r: => Unit): Unit = es.submit(new Callable[Unit] { def call = r })`, handles the actual execution. When the `Future` returned by `fork` receives its continuation `cb`, the `eval` helper submits a task to the `ExecutorService` to asynchronously evaluate the by-name argument `a`. Once the argument `a` is evaluated and produces a `Future[A]`, the system registers `cb` to be invoked when that `Future` has its resulting value `A`.

Implementing a non-blocking `map2` requires a more advanced approach than using low-level `java.util.concurrent` primitives, as one must ensure that `map2` runs both `Par[A]` and `Par[B]` arguments in parallel and invokes the combining function `f` only after both results arrive, preventing race conditions. The recommended solution uses a non-blocking concurrency primitive called an **Actor**. An Actor acts as a concurrent process that processes messages sequentially, only using a thread when a message arrives. This mechanism is useful for writing concurrent code susceptible to race conditions or deadlocks.

One concrete example of an Actor is an `echoer` that processes messages received via an `ExecutorService` (e.g., `val S = Executors.newFixedThreadPool(4)`). The usage demonstrates that the `echoer` does not occupy a thread when it has no messages to process.

To implement `map2` without blocking, the following structure uses an Actor to manage the combination: 
`def map2[A,B,C](p: Par[A], p2: Par[B])(f: (A,B) => C): Par[C] = es => new Future[C] { def apply(cb: C => Unit): Unit = { ... } }`

The core logic relies on passing the Actor as a continuation to both `p` and `p2`. On the `A` side, the result is wrapped in `Left`, and on the `B` side, it is wrapped in `Right` (using the `Either` data type to track the source). The Actor logic manages state using two mutable variables, `ar: Option[A]` and `br: Option[B]`. The Actor processes the first message received (either `Left(a)` or `Right(b)`), stores it, and waits for the second result. When the second result arrives, the Actor calls the combining function `f` with both stored results and passes the final `C` result to the original continuation `cb` using `eval(es)`.

This non-blocking `map2` implementation successfully allows running `parMap` (e.g., `parMap(List.range(1, 100000))(math.sqrt(_))`) for large inputs, such as 100,000 iterations, and collecting the results using `run(Executors.newFixedThreadPool(2))(p)` without requiring 100,000 JVM threads, demonstrating that the law of forking now holds even for fixed-size thread pools.
