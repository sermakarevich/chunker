# parallel-combinator-deadlock-analysis

**Parent:** [[content/L1/functional-programming-advanced-concepts|functional-programming-advanced-concepts]] — Functional programming principles govern complex behavior through algebraic laws, utilizing specialized ADTs like `Option` and `Either` for explicit error handling. Efficiency is maximized via *non-strictness* and `Stream[+A]`, which facilitates automatic loop fusion by deferring computation, enabling safe handling of infinite streams and the corecursive generation of sequences using `unfold`. Concurrency is managed using the `Par[A]` type, which requires adherence to formal laws; specific implementations of `fork` must avoid blocking methods like `Future.get` when operating on fixed-size thread pools to prevent deadlocks.

The functional API must be structured as an algebra governed by specific laws or axioms. These laws define the expected behavior of operations. For instance, two key laws are that mapping a unit value of 1 using the `_ + 1` function must equal `unit(2)` (i.e., `map(unit(1))(_ + 1) == unit(2)`), and that mapping a value using the identity function must equal the original value (i.e., `map(y)(id) == y`). Defining these formal laws is critical because hidden assumptions or side effects can compromise compositionality, making it difficult to treat API components as black boxes. For example, if the law stating that `fork(x)` is equivalent to `x` for all expressions `x` and all choices of `ExecutorService` does not hold, general-purpose combinators like `parMap` would lose their soundness, which could potentially cause deadlocks. A specific subtle bug occurs in most implementations of `fork` when using an `ExecutorService` backed by a fixed-size thread pool, such as those created by `Executors.newFixedThreadPool`. If the maximum number of threads in the pool is limited to one, the following code sequence will likely deadlock: first defining `val a = lazyUnit(42 + 1)`, followed by `val S = Executors.newFixedThreadPool(1)`, and finally calling `println(Par.equal(S)(a, fork(a)))`. The deadlock occurs because the `fork` implementation uses `def fork[A](a: => Par[A]): Par[A] = es => es.submit(new Callable[A] { def call = a(es).get })`. This structure causes the outer `Callable` to be submitted and handled by the single available thread. Within that same thread, the code then submits and blocks, waiting for the result of a second `Callable` via `a(es).get`. Because the thread pool is restricted to only one thread, no available threads exist to execute the second `Callable`, causing the deadlock. Developers should re-examine the law `fork(x) == x` to test the implementation's robustness, determining if the law holds for the specific implementation or if the law needs refinement, perhaps by stipulating a requirement for unbounded thread pools. Alternative implementations for `fork` include two options: 1) `def fork[A](a: => Par[A]): Par[A] = es => fa(es)`: This avoids deadlock but fails to genuinely fork a separate logical thread for evaluating `fa`, meaning that calling `fork(hugeComputation)(es)` would execute `hugeComputation` in the main thread. This variant is, however, useful because it allows the computation delay until it is actually needed and is named `delay`. 2) To genuinely run arbitrary computations over fixed-size thread pools, a fully non-blocking representation of `Par` is required. The core problem with the current `Par` representation is that a value cannot be retrieved from a `Future` without the current thread blocking on its `get` method. Therefore, a `Par` representation that avoids resource leaks and blocking must ensure that the implementations of `fork` and `map2` never call any method that blocks the current thread, such as `Future.get`.
