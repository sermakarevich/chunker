# general-purpose-io-type

**Parent:** [[content/L1/advanced-fp-algebraic-design|advanced-fp-algebraic-design]] — Advanced functional programming mandates using algebraic structures (ADTs, Functor, Applicative, Monad) to treat computation as a first-class value, ensuring type safety and referential transparency by encapsulating side effects. Effect management is achieved using the `Free[F, A]` monad, where $F$ is the protocol, enabling interpreters (e.g., `runConsolePar`) to translate abstract I/O (like `ConsoleIO[A]`) into concrete structures (`Par[A]`) suitable for both trampolined and non-blocking asynchronous execution.

To establish a general methodology for writing programs that perform I/O, developers can formulate an algebraic data type. For any set of I/O operations the developer wishes to support, an algebraic data type can be constructed, with case classes representing individual operations. For instance, one could create a `Files` data type for file I/O, a `DB` data type for database access, and use an existing type like `Console` for standard input and output interactions. For any such data type $F$, the developer can generate a free monad `Free[F, A]` in which to write the program logic. These programs can be tested individually and later compiled down to a lower-level I/O type, referred to as `Async`: `type IO[A] = Free[Par, A]`. This `IO` type supports both trampolined sequential execution (due to the `Free` structure) and asynchronous execution (due to the `Par` monad). By defining the main program using this single, most general type, all individual effect types are brought together, requiring only a translation mechanism from any given $F$ to `Par`. 

When the Java Virtual Machine (JVM) executes the main program, it expects a `main` method with a specific signature, and the return type must be `Unit`, implying that side effects are expected. However, the program can delegate to a `pureMain` program that is entirely pure. In this scenario, the main method's sole function is to interpret the `pureMain` program, thereby executing the defined effects. 

The `unsafePerformIO` method, which is defined as `def unsafePerformIO[A](a: IO[A])(pool: ExecutorService): A = Par.run(pool)(run(a)(parMonad))`, handles the execution of the `IO` action, performing the effect by converting the `IO[A]` to a `Par[A]` and then extracting the value `A`. 

Developers must distinguish between effects and side effects. The `pureMain` program itself is designed to contain no side effects; it should be a referentially transparent expression of type `IO[Unit]`. By placing the performance of effects entirely within the `main` method, which exists outside the scope of the `pureMain` program, the developer ensures that the program exhibits effects but not side effects. 

However, despite the flexibility of the `IO` monad and the advantage of having I/O actions as first-class values, the `IO` type fundamentally only offers the same level of abstraction as ordinary imperative programming. Consequently, writing efficient, streaming I/O will generally require monolithic loops.
