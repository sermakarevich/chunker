# parser-algebra-completeness

**Parent:** [[content/L1/functional-algebra-synthesis|functional-algebra-synthesis]] — The synthesis details how functional programming APIs are designed using algebras, detailing specific combinators and types like Option, Either, Stream, Par, and flatMap. It covers advanced topics such as managing non-strictness for optimal data transformation, resolving concurrency deadlocks using the Actor model, enabling context-sensitive parsing via `flatMap`, and structuring comprehensive error reporting mechanisms.


The design of a functional API requires structuring it as an algebra, which is a collection of data types, functions over those types, and formal laws or axioms defining expected behavior. While primitives like `unit`, `map`, `map2`, `listOfN`, `many`, and `char` are useful, these combinators are not sufficient for parsing all context-free grammars, including JSON. The limitation arises because the choice of one parser might depend on the result of a preceding parser—a characteristic known as context sensitivity. For example, to parse a single digit (like '4') followed by a number of 'a' characters (e.g., valid inputs include "0", "1a", "2aa", or "4aaaa"), the second parser must depend on the result of the first parser (the second parser depends on its context). This dependency cannot be captured using the product combinator. To solve this expressiveness limitation, the API introduces a new primitive: `flatMap`. The signature for `flatMap` is defined as `def flatMap[A,B](p: Parser[A])(f: A => Parser[B]): Parser[B]`. This signature implies an ability to sequence parsers where each parser in the chain depends on the output of the previous one. Subsequent exercises involve using `flatMap` and other combinators to write the context-sensitive parser, implementing `product` and `map2` in terms of `flatMap`, and finally expressing the `map` combinator in terms of `flatMap` and/or other combinators.
