# combinator-generalization-choice-and-flatmap

**Parent:** [[content/L1/functional-programming-algebra-design|functional-programming-algebra-design]] — The synthesis details advanced functional programming concepts, including using specialized ADTs like `Option` and `Either` for type-safe error handling; implementing non-blocking concurrency via Actors for primitives like `map2`; achieving memory efficiency through non-strictness and `Stream` types; and generalizing combinators (like `choice` to `flatMap`) to form a coherent mathematical algebra. Furthermore, it introduces property-based testing (PBT) using generators (`Gen`) to define system behavior abstractly.

To illustrate this concept, consider the function `choice[A]`, which selects between two forking computations based on the result of an initial parallel computation: `def choice[A](cond: Par[Boolean])(t: Par[A], f: Par[A]): Par[A]`. This function constructs a computation that proceeds with the result of `t` if `cond` evaluates to `true`, or with the result of `f` if `cond` evaluates to `false`. While a simple, blocking implementation involves waiting for the result of `cond` (using `run(es)(cond).get`) to determine whether to run `t` or `f` (`es => if (run(es)(cond).get) t(es) else f(es)`), this blocking approach is suboptimal. 

To achieve greater generality, developers should generalize the choice mechanism beyond simply selecting between two computations (`t` and `f`). The concept can be extended to selecting among $N$ parallel computations, using `choiceN[A](n: Par[Int])(choices: List[Par[A]]): Par[A]`. This function takes a parallel computation `n` and uses its result to select one parallel computation from the list of provided `choices`, representing a step toward generalizing the original `choice` combinator.

Next, the choice structure can be generalized further by replacing the specific `List` container with a generic container. For example, instead of a list of computations, one could use a `Map[K, Par[V]]` where the keys `K` determine the selection. This led to the definition of `choiceMap[K,V](key: Par[K])(choices: Map[K,Par[V]]): Par[V]`. By observing the core requirement, it becomes clear that both the pair of arguments (Boolean selection in `choice`) and the list container (Int selection in `choiceN`) are merely serving as mechanisms to provide a function of type `Int => Par[A]` or `Boolean => Par[A]`. 

By creating a more general signature, the `chooser` combinator is defined: `def chooser[A,B](pa: Par[A])(choices: A => Par[B]): Par[B]`. The implementation of this new primitive, `chooser`, allows for the implementation of the previous specialized combinators: `choice` and `choiceN` can both be implemented using `chooser`. 

To advance the generalization, the authors observe that this operation is not merely a `chooser`. This pattern—an initial parallel computation `pa` that determines which subsequent parallel computation runs—is recognized as a functional pattern typically implemented using `flatMap` or `bind`. The signature is generalized to `def flatMap[A,B](a: Par[A])(f: A => Par[B]): Par[B]`. Further generalization suggests adding an even simpler combinator, `join`, which converts a nested parallel computation `Par[Par[X]]` to `Par[X]`: `def join[A](a: Par[Par[A]]): Par[A]`. This definition exists purely based on the required signature. The next step involves implementing `join` and then demonstrating how to implement both `flatMap` using `join` and `join` using `flatMap`.
