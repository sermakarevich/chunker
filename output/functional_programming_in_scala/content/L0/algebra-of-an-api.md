# algebra-of-an-api

**Parent:** [[content/L1/functional-api-algebra-and-parallel-combinator-laws|functional-api-algebra-and-parallel-combinator-laws]] — The design of a functional API must be treated as an algebra governed by laws, such as the mandatory requirement that `map(unit(1))(_ + 1) == unit(2)` and that `map(y)(id) == y`. The parallel combinator `fork` must satisfy the law `fork(x) == x`, but its current implementation risks deadlocking when using a fixed-size thread pool (like `Executors.newFixedThreadPool(1)`) because the outer `Callable` blocks on the inner computation's `get` call within the same single thread.

The process of defining a functional API involves treating the API as an algebra—an abstract set of operations combined with specific axioms or properties that are assumed to be true. This approach allows developers to focus on lining up types and reasoning formally, much like simplifying an algebraic equation. Identifying and formalizing these expected properties, or 'laws,' is crucial because any choice of law places constraints on what the operations can mean and determines possible implementation choices. For instance, considering the law `map(unit(1))(_ + 1) == unit(2)` illustrates how a law can be used as a test case for a library. This law suggests that mapping a unit value of 1 using the `_ + 1` function should be equivalent to receiving the result of `unit(2)`. Two `Par` objects are considered equivalent if, for any valid `ExecutorService` argument, calling the `get` method on their respective `Future` results yields the same value. This law can be generalized to `map(unit(x))(f) == unit(f(x))` for any choice of input value `x` and function `f`. This general law dictates that the implementation of `unit` must only pass along its received value and that the `ExecutorService` must not make assumptions or change behavior based on the submitted `Callable` objects. More concretely, this law prohibits typecasing (like `downcasting` or `isInstanceOf` checks) within the implementations of `map` and `unit`. These concepts can be further simplified by substituting the identity function (`id[A]`, defined as `def id[A](a: A): A = a`) for `f`, leading to the law `map(y)(id) == y`, where `y` represents `unit(x)`. This simplified law demonstrates that the mention of `unit` was extraneous to the primary function of `map`. Because of the parametricity of `map`, the authors can perform an inverse substitution, concluding that the original law `map(unit(x))(f) == unit(f(x))` must hold, a derivation termed a 'free theorem.' Another property to consider is `fork(x) == x`, which states that wrapping a computation `x` using `fork` should not change the result of the parallel computation; it should only execute `x` asynchronously in a logical thread separate from the main thread. Writing down such laws can place strong constraints on the implementations of `map`, `unit`, or `ExecutorService.submit`, requiring developers to thoroughly analyze potential corner cases and prove the law holds. The process of deducing these laws through algebraic manipulation and formal testing guides the design of the functional API.
