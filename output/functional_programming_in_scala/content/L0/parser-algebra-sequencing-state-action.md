# parser-algebra-sequencing-state-action

**Parent:** [[content/L1/parser-algebra-enhancements-2|parser-algebra-enhancements-2]] — The parser algebra was significantly enhanced by introducing a stack-based error structure (`ParseError(stack: List[(Location,String)])`), which allows developers to track error context and is managed by a `push` helper. State tracking requires the `Result` type, which now explicitly carries `charsConsumed: Int`. The new `flatMap` combinator enables context-sensitive parsing by advancing the input location and calculating cumulative consumption, while the `attempt` combinator, coupled with the `isCommitted` flag in `Failure`, controls commitment during ambiguous parsing and the `or` combinator's fallthrough logic.

To effectively implement the parser algebra, which treats parsers as state actions, a new mechanism is required to track the consumed input characters. Initially, the design used a `Location` object. To enhance this, a helper function, `toError`, was added to `Location` to facilitate error reporting: `def toError(msg: String): ParseError = ParseError(List((this, msg)))`. To support parser sequencing, which involves running multiple parsers sequentially (e.g., parsing "abra" followed by "cadabra"), a parser must indicate exactly how many characters it consumed upon success. Therefore, the definition of the parser type was updated from its original form to a new result type, `Result[+A]`. The updated signature for a parser is now `type Parser[+A] = Location => Result[A]`. The `Result` type is defined as a sealed trait with two primary constructors: `Success[+A](get: A, charsConsumed: Int)` and `Failure(get: ParseError)`. In the event of successful parsing, the `Success` case not only returns the parsed value of type `A` but also includes `charsConsumed: Int`, which represents the number of characters of input the parser consumed. This consumed character count is critical because the calling combinators use this information to accurately update the `Location` state for subsequent parsing steps. This understanding—that a parser is essentially a state action that can either fail or succeed while providing both a resulting value and the necessary state updates—is crucial for designing a representation that supports all stipulated combinators and formal laws.
