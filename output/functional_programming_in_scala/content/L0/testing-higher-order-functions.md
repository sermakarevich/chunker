# testing-higher-order-functions

**Parent:** [[content/L1/functional-algebraic-design-and-concurrency-patterns|functional-algebraic-design-and-concurrency-patterns]] — The API is designed algebraically, requiring explicit use of ADTs like `Option` and `Either` for error handling, and utilizing lazy `Stream`s for memory-efficient data transformation, complemented by `Par` for non-blocking parallel computation (modeled using the Actor pattern). Testing extends to generating complex functions using specialized generators, and parser creation is streamlined by treating parsers as first-class values in a combinator library, avoiding monolithic code generators like Yacc or ANTLR.

Our library seems quite expressive, but developers currently lack a robust mechanism for testing higher-order functions. While the library provides various methods for generating data using generators, generating functions for testing remains challenging. For example, the `takeWhile` function, which returns the longest prefix of an input whose elements all satisfy a predicate, can be tested using the property that for any list `s: List[A]` and any predicate `f: A => Boolean`, the expression `s.takeWhile(f).forall(f)` evaluates to true. In the Scala standard library, the method `forall` on `List` and `Stream` accepts a function `f: A => Boolean` and returns a Boolean. To test higher-order functions, developers must consider ways to generate functions. For instance, supposing a developer has a `Gen[Int]` and needs to produce a `Gen[String => Int]`, one approach is to generate constant functions that ignore the input string and delegate to the underlying `Gen[Int]`, defined as `def genStringIntFn(g: Gen[Int]): Gen[String => Int] = g map (i => (s => i))`. However, this method only generates constant functions that ignore their input, which is insufficient for testing complex behaviors like those required by `takeWhile`, as such testing requires generating functions that return a Boolean, not just constant values. To address this, a challenging design exercise involves generating functions that utilize their input argument to select which integer to return, leading to potential library extensions. The field of functional design offers many opportunities for generating test functions. Developers are encouraged to: 1) Write properties specifying the behavior of functions like `take`, `drop`, `filter`, and `unfold` for both `List` and `Stream`. 2) Write a sized generator for producing the Tree data type (defined in Chapter 3), and then use this to specify the behavior of the fold function for Tree, while considering API improvements to simplify this process. 3) Write properties to specify the behavior of the `sequence` function defined for `Option` and `Either`. The functions within the library exhibit deep structural connections. For instance, the `map` function defined for `Par`—`def map[A,B](a: Par[A])(f: A => B): Par[B]`—shares a signature with the function defined for `Gen` (`def map[B](f: A => B): Gen[B]`) and other types like `Option`, `List`, `Stream`, and `State`. This parallelism suggests that these functions not only share similar signatures but also satisfy analogous laws in their respective domains, indicating fundamental patterns across the functional solutions for different problems.
