# handling-errors-with-option-type

**Parent:** [[content/L1/scala-functional-data-structures-and-higher-order-functions|scala-functional-data-structures-and-higher-order-functions]] — The comprehensive synthesis details advanced Scala concepts, including persistent functional data structures, the non-tail-recursive nature of `foldRight` (which causes stack overflows on large inputs), and the necessity of implementing tail-recursive `foldLeft` for large lists. It also covers advanced function generalization like `flatMap` and `zipWith`, and modern type-safe error handling using the `Option[+A]` ADT instead of exceptions or sentinel values.

To improve error handling, the technique discussed replaces the practice of throwing exceptions with returning a specialized value that indicates whether an exceptional condition occurred. This approach leverages generic types and higher-order functions to encapsulate common error handling and error propagation patterns, ensuring the solution is completely type-safe and requires minimal syntactic overhead, unlike the boilerplates associated with checked exceptions or C-style error codes. 

Before this technique, an example function, `mean(xs: Seq[Double])`, computed the mean of a sequence of doubles. This function is considered a partial function because it is undefined when the input sequence `xs` is empty. If the program relies on exceptions, the initial definition of `mean` throws an `ArithmeticException` when `xs.isEmpty` is true. In contrast, returning a sentinel value (like `Double.NaN` or `null`) or forcing the caller to supply a default value (like in `mean_1(xs: IndexedSeq[Double], onEmpty: Double): Double`) presents several issues: 1) Error propagation is silent, allowing the caller to forget necessary checks, which might cause subsequent code errors only detected much later; 2) This approach introduces significant boilerplate code at call sites, requiring explicit checks whether the caller received a "real" result; 3) It fails for polymorphic code. For example, a function like `max[A](xs: Seq[A])(greater: (A,A) => Boolean): A`, which finds the maximum value in a sequence using a custom comparison function, cannot invent a sentinel value of type `A` if the input is empty, especially if `A` is a primitive type like `Double` or `Int` (because `null` is only valid for non-primitive types). 4) It forces special policies or calling conventions on the callers, making it difficult to pass the function to general higher-order functions that must treat all arguments uniformly.

An alternative to exceptions is to introduce a new generic type, `Option[+A]`, to represent explicitly that a function may not always have an answer. This type is defined using the following structure: `sealed trait Option[+A]`, which contains two cases: `case class Some[+A](get: A)` for defined values, and `case object None extends Option[Nothing]` for undefined values. By implementing the mean function using `Option[Double]`, the function definition becomes: `def mean(xs: Seq[Double]): Option[Double] = if (xs.isEmpty) None else Some(xs.sum / xs.length)`. The resulting type signature, `Option[Double]`, now accurately reflects the possibility that the result may not always be defined, while ensuring that `mean` is a total function which always returns a value of the declared type. This mechanism forces the calling code to explicitly handle the possibility of failure, which is a distinct advantage over relying on runtime exceptions or sentinel values. The `Option` data type is frequently used in the Scala standard library, such as when looking up a key in a map (which returns `Option`) or defining methods like `headOption` and `lastOption` for non-empty sequences, which return an `Option` containing the first or last elements.
