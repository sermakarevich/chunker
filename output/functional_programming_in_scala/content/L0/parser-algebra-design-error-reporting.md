# parser-algebra-design-error-reporting

**Parent:** [[content/L1/parser-algebra-enhancements-2|parser-algebra-enhancements-2]] — The parser algebra was significantly enhanced by introducing a stack-based error structure (`ParseError(stack: List[(Location,String)])`), which allows developers to track error context and is managed by a `push` helper. State tracking requires the `Result` type, which now explicitly carries `charsConsumed: Int`. The new `flatMap` combinator enables context-sensitive parsing by advancing the input location and calculating cumulative consumption, while the `attempt` combinator, coupled with the `isCommitted` flag in `Failure`, controls commitment during ambiguous parsing and the `or` combinator's fallthrough logic.

The design of the parser algebra must support advanced error reporting. Initially, reporting a ParseError only included a single `Location` and a `String` message. To improve granularity, the error mechanism should store a stack of errors: `case class ParseError(stack: List[(Location,String)])`. This change allows the structure to record what the parser was doing when it failed.

Developers propose introducing a function `scope` to manage error context. The `scope` combinator, defined as `def scope[A](msg: String)(p: Parser[A]): Parser[A]`, does not replace labels but simply adds additional information to the error stack if the parser `p` fails. Specifically, if `run(p)(s)` yields `Left(e1)`, then `run(scope(msg)(p))(s)` yields `Left(e2)`, where the head of `e2.stack` will be the message `msg`, and the rest of the stack (`e2.stack.tail`) will be the original error `e1`.

To further manage error reporting, a primitive combinator is needed for an 'or' operation that respects error state. When the parser `p` is defined using the `or(p1, p2)` combinator, and both `p1` and `p2` fail on the same input, the desired behavior is not merely a global convention but one that can be controlled by the programmer. For example, if `p` is constructed from `p1` and `p2`, and both fail on the input "abra cAdabra" (where 'A' is capitalized in cAdabra), the result is a question of which error to report. If the desired outcome is to report the error from `p1` (the "magic spell" branch) after successfully parsing the "abra" word, the implementation must commit to that branch, ignoring errors from `p2` (the "gibberish" branch) even if they occur. Conversely, in other scenarios, the programmer might wish to allow the parser to consider the next branch if the first one fails. 

To address this, a new combinator, `attempt`, is introduced: `def attempt[A](p: Parser[A]): Parser[A]`. This combinator delays the commitment to the parse until the parser successfully runs. It must satisfy the property that running `attempt(p)` and then running `p2` when `p` fails midway through examining the input must revert the commit, allowing `p2` to be run. The `attempt` combinator should be used when the grammar is ambiguous and multiple tokens might need to be examined before the grammar can resolve the ambiguity and commit to a single parsing branch.

Finally, the implementation of the parser algebra, `Parsers`, can be finalized by removing the abstract type parameter and setting the signature to `trait Parsers[Parser[+_]]`.

The initial attempt at implementing the `string` combinator suggests a concrete representation for `Parser[A]`: `type Parser[+A] = String => Either[ParseError,A]`. Using this type, the `string` combinator can be implemented as: `def string(s: String): Parser[A] = (input: String) => if (input.startsWith(s)) Right(s) else Left(Location(input).toError("Expected: " + s))`. This demonstrates how the parser combinator library can be structured around a function that receives the input string and returns either a `ParseError` or the successfully parsed value.
