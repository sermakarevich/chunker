# parser-combinator-algebra-design

**Parent:** [[content/L1/functional-algebra-synthesis|functional-algebra-synthesis]] — The synthesis details how functional programming APIs are designed using algebras, detailing specific combinators and types like Option, Either, Stream, Par, and flatMap. It covers advanced topics such as managing non-strictness for optimal data transformation, resolving concurrency deadlocks using the Actor model, enabling context-sensitive parsing via `flatMap`, and structuring comprehensive error reporting mechanisms.


The parser combinator library, which defines parsers as first-class values, provides several advanced parsing tasks for consideration. These tasks include defining parsers that count specific characters: a parser recognizing zero or more 'a' characters, which returns the count of seen 'a's; a parser recognizing one or more 'a' characters, which also returns the count of seen 'a's but must fail (and potentially report a customized error message like "Expected one or more 'a'") if the input does not start with 'a'; and a parser recognizing zero or more 'a' followed by one or more 'b', which returns a pair representing the counts of each character (e.g., 4 for 'a's and 1 for 'b'). Furthermore, developers should consider optimizing the parsing process: if the goal is only to count the characters in a sequence of zero or more 'a's, building a `List[Char]` and then extracting its length is inefficient; a more direct approach is needed.

Regarding the core algebra, the following theoretical questions guide development: Are various forms of repetition primitives in the algebra, or can they be defined using simpler combinators? Because the library introduces the `ParseError` type, the current algebra lacks any mechanisms for allowing programmers to control or specify detailed error messages. Developers should explore how to enhance the `ParseError` API to support meaningful error reporting. The associativity of the OR operator (`|`) must also be considered: Do the expressions `a | b` and `b | a` mean the same thing? If they are commutative, this must be documented. Likewise, is the expression `a | (b | c)` equivalent to `(a | b) | c`? If this associativity holds, determining if it is a fundamental primitive law or derivable from simpler principles is crucial. Finally, developers are encouraged to propose a set of formal laws to specify the entire algebra, noting that the laws do not need to be exhaustive, but should capture expected behaviors for any parsers implementation.
