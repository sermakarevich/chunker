# purely-functional-error-handling-principles

**Parent:** [[content/L1/functional-fp-laziness-and-error-handling|functional-fp-laziness-and-error-handling]] — Functional programming improves efficiency and robustness by using non-strictness (laziness) to fuse multiple data transformations into a single pass, and by employing algebraic data types like `Option` and `Either` to represent potential failures as values rather than relying on runtime exceptions.

In this chapter, the text noted problems associated with using exceptions and introduced fundamental principles for purely functional error handling. While the chapter focused on the algebraic data types `Option` and `Either`, the core concept is that program designers can represent exceptions as ordinary values and use higher-order functions to encapsulate common patterns of handling and propagating errors. This general idea—of representing effects as values—is a concept that will be revisited frequently throughout the book in various forms. Although the reader is not expected to be fluent with all the higher-order functions detailed in this chapter, the reader should now have sufficient familiarity to begin writing independent functional code that includes proper error handling. Given these new tools, exceptions should be reserved only for genuinely unrecoverable conditions. Finally, the chapter briefly introduced the notion of a non-strict function, referencing the `orElse`, `getOrElse`, and `Try` functions. The next chapter will examine non-strictness in more detail, explaining why non-strictness is important and how it provides greater modularity and efficiency in functional programs.
