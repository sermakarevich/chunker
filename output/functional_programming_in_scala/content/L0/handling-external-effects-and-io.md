# handling-external-effects-and-io

**Parent:** [[content/L1/monadic-structure-for-external-effects|monadic-structure-for-external-effects]] — To manage side effects and achieve true functional purity, computational structures are refactored using the `IO` monad, which models effects as pure, composable values. This concept generalizes through `TailRec[A]` to solve stack overflow issues via trampolining, eventually culminating in the highly abstract `Free[F[_],A]` structure, where the type parameter $F$ acts as an executable protocol (e.g., `Par` for concurrency or a custom `Console[A]` ADT) that governs the description of all interactions.

To understand the concept of handling external effects, the process of analyzing an impure function, $f$, of type $	ext{A} 	o 	ext{B}$ is useful. Developers can decompose $f$ into two parts: 1) a pure function of type $	ext{A} 	o 	ext{D}$, where $	ext{D}$ is a description of the result of $f$, and 2) an impure function of type $	ext{D} 	o 	ext{B}$. This second function acts as an interpreter for the descriptions generated by the first. By repeatedly applying this strategy, the process involves making more functions pure and pushing the side effects outward. Eventually, this decomposition reaches built-in procedures, like `println` (which has type $	ext{String} 	o 	ext{Unit}$), which necessitate side effects. 

To handle this, developers can introduce a new data type called $	ext{IO}$: `trait IO { def run: Unit }`. For example, the procedure `PrintLine` is defined as `def PrintLine(msg: String): IO = new IO { def run = println(msg) }`. Using this, a function like `contest(p1: Player, p2: Player)` can be refactored to return an `IO` value: `def contest(p1: Player, p2: Player): IO = PrintLine(winnerMsg(winner(p1, p2)))`. In this refactored state, the `contest` function itself is pure; it only produces an `IO` value, which is a pure description of an action that needs to take place. The only part that actually executes the side effect is the `run` method of the `IO` interpreter. The `contest` function's responsibilities are limited to composing the program parts: calculating the winner, determining the result message using `winnerMsg`, and instructing `PrintLine` to display the message on the console. The responsibility of interpreting the effect and manipulating the console belongs solely to the `run` method on the $	ext{IO}$ type.

Initially, it might seem that the $	ext{IO}$ type adds little value beyond merely satisfying the requirements of referential transparency. However, the true value of the $	ext{IO}$ type lies in the algebra it provides—the set of useful operations and programs that can be defined, leading to manageable laws for reasoning about larger programs. We can define the following operations for $	ext{IO}$: `def ++(io: IO): IO = new IO { def run = { self.run; io.run } }`, where `self` refers to the outer $	ext{IO}$ instance. The only property that is immediately apparent is that $	ext{IO}$ forms a Monoid, because `empty` acts as the identity and `++` is the associative operation. This allows a list of `IO` computations to be reduced to a single `IO`, regardless of whether the reduction is done via left or right folding.
