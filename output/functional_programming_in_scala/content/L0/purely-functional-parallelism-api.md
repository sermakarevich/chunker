# purely-functional-parallelism-api

**Parent:** [[content/L1/functional-par-design-patterns|functional-par-design-patterns]] — Functional programming manages non-fatal control flow using types like `Option[+A]` and `Either[+A, +B]`, while efficiency is achieved through non-strictness and the specialized lazy structure `Stream[+A]`. Parallel computations are modeled abstractly by `Par[A]`, requiring combinators like `map2` and `fork` to ensure concurrent execution and maintain API purity, allowing complex operations like `sum` to be recursively defined and generalized through the derivation of `map` from `map2`.

The proposed functional design for parallel computations defines `Par[A]` as a conceptual type representing a parallel computation. This approach models the computation abstractly, allowing the structure to be independent of the actual mechanism for implementing parallelism. The execution of the computation is deferred until the function `run[A](a: Par[A]): A` is called, which is responsible for executing the parallel task and extracting the resulting value. By making `Par` a pure data structure, `run` must incorporate the underlying mechanisms for parallelism, such as spawning new threads or delegating tasks to a thread pool, or utilizing an external resource like an `ExecutorService` (specifically, `def run[A](s: ExecutorService)(a: Par[A]): A`).

To maintain purity in the API while allowing effectful execution at the end, the designers defined a set of fundamental combinators:

1.  **`unit[A](a: A): Par[A]`**: Creates a parallel computation that immediately yields the value `a`. This function does not require the `ExecutorService`.
2.  **`map2[A,B,C](a: Par[A], b: Par[B])(f: (A,B) => C): Par[C]`**: Combines the results of two parallel computations, `a` and `b`, using a binary function `f`. This implementation requires passing the `ExecutorService` to both `a` and `b` to obtain the intermediate `Future` results (`af` and `bf`), then applying `f` to the results of `af.get` and `bf.get`, and finally wrapping the result in a `UnitFuture`. (Note: The original design acknowledged that this specific implementation does not respect timeouts.)
3.  **`fork[A](a: => Par[A]): Par[A]`**: Marks a computation for concurrent evaluation. This function utilizes the `ExecutorService` to submit a `Callable[A]` task, ensuring that the evaluation is executed in a separate logical thread, thereby preserving potential parallelism. The definition `def fork[A](a: => Par[A]): Par[A] = es => es.submit(new Callable[A] { def call = a(es).get })` is the simplest, most natural implementation. 

These methods define the functional API, which remains pure even though its implementation ultimately relies on effects when the user calls `run` and provides the `ExecutorService` machinery. The core principle is that the user programs against a pure interface, while the implementation handles the necessary side effects (e.g., submitting tasks to the `ExecutorService`) only when the system reaches the `run` function.
