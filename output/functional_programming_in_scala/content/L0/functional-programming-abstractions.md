# functional-programming-abstractions

**Parent:** [[content/L1/functional-fp-architecture-context|functional-fp-architecture-context]] — The design of functional programming necessitates managing computations as first-class values, utilizing special types ('effects') like Option, Either, Par, and Stream to handle potential failures and side effects without violating referential transparency. Applicative funtors combine independent results using fixed structure (e.g., `map2`), while Monads are required when the result of one calculation influences subsequent steps (e.g., `flatMap`). State management across all traversable functors (like List, Tree) is generalized using the `mapAccum` function, which allows for writing reusable core combinators such as `toList` and `zipWithIndex` for any structure.

In this chapter, the discussion moves to identifying broader abstract structures within the written libraries, building upon the concept of the monoid. The initial goal was to factor out code duplication across various implemented combinator libraries, which led to the discovery of two new abstract interfaces: Functor and Monad. Applicative functors represent a generalization of monads that are less expressive but generally more compositional. The functions `unit` and `map` enable lifting values and functions, while `map2` and `apply` provide the power to lift functions of higher arities. Traversable functors generalize the repeated usage of `sequence` and `traverse` functions. Collectively, the `Applicative` and `Traversable` abstractions allow developers to construct complex nested and parallel traversals from simple elements, requiring only that these simple elements be written once. The authors advise that as readers continue writing functional code, they should learn to spot instances of these abstractions and how to make better use of them. While this chapter marks the end of part 3, the material presented is just an introduction to the vast world of functional abstractions; techniques developed here can be applied to discover new structures like arrows, categories, and comonads. For further study, the material linked in the chapter notes is recommended. 

Part 4 will complete the functional programming story, focusing on how Functional Programming (FP) can express all imaginable programs, including those that mutate data in place and interact with the external world (like writing to files or reading from databases). The authors plan to begin by examining the most straightforward method for handling external effects, using an I/O monad. This I/O monad is a simplistic embedding of an imperative programming language into a functional language. Similarly, the general approach can handle local effects and mutation, which will be introduced in chapter 14. Both chapters will motivate the development of more composable strategies for managing effects. In chapter 15, the final chapter, the discussion will introduce a library for streaming I/O, detailing how to write compositional and modular programs that incrementally process I/O streams. The overall goal of Part 4 is not to cover every relevant technique for handling I/O and mutation but to introduce essential ideas and equip the reader with a conceptual framework for future learning. Although readers will encounter problems that do not look exactly like the examples discussed, completing Part 4, alongside parts 1–3, will put the reader in a good position to apply FP to any programming task.

**External Effects and I/O**

In this chapter, the discussion will extend what was learned about monads and algebraic data types to handle external effects, such as reading from databases and writing to files. The goal is to develop a monad for I/O, named `IO`, to allow developers to handle external effects in a purely functional way. The chapter will make a critical distinction between effects and side effects. The `IO` monad provides a straightforward way to embed imperative programming with I/O effects within a pure program, thus preserving referential transparency. It achieves this by clearly separating effectful code—code that necessitates an effect on the outside world—from the rest of the program. Furthermore, this approach will illustrate a key technique for managing external effects: using pure functions to compute a description of an effectful computation, which is then executed by a separate interpreter that actually performs the effects. This process is essentially crafting an embedded domain-specific language (EDSL) for imperative programming, a powerful technique utilized throughout the remainder of Part 4. The primary objective is to equip developers with the skills necessary to craft their own EDSLs for describing effectful programs.
