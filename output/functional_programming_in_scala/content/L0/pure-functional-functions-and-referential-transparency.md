# pure-functional-functions-and-referential-transparency

**Parent:** [[content/L1/functional-programming-purity-side-effects|functional-programming-purity-side-effects]] — The initial impure `buyCoffee` method charges the credit card directly, an observable side effect, which is refactored into the pure `buyCoffees` function that returns a `Charge` data type, enabling composable and testable logic.

When examining what constitutes a pure function, it is necessary to address the fact that programs must interact with the external world, requiring the submission of a `Charge` for processing by some external system. Furthermore, other useful programs may necessitate side effects or mutation. The text proposes that as the reader works through the book, the reader will discover that many programs that seem to necessitate side effects actually have a functional analogue. In other cases, the text suggests methods to structure code so that effects occur but are not observable. For instance, a program can mutate data that is declared locally within the body of a function if the program ensures that the data cannot be referenced outside that function, or the program can write to a file as long as no enclosing function can observe this occurrence.

To formalize the notion of side effects and purity, the book discusses the concept of a pure function. The initial understanding stated that Functional Programming (FP) means programming with pure functions, and a pure function is defined as one that lacks side effects. A function `f` with an input type `A` and an output type `B` (written in Scala as `A => B`, pronounced “A to B” or “A arrow B”) is a computation that relates every value `a` of type `A` to exactly one value `b` of type `B`. This means that `b` is determined solely by the value of `a`. For example, a function `intToString` having the type `Int => String` will take every integer to a corresponding string. Crucially, any changing state of an internal or external process is irrelevant to computing the result `f(a)`.
