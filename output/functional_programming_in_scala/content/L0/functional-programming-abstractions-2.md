# functional-programming-abstractions-2

**Parent:** [[content/L1/functional-fp-abstraction-architecture|functional-fp-abstraction-architecture]] — Functional programming achieves architectural rigor by establishing a hierarchy of abstractions (Functor, Applicative, Monad), which treat computation as first-class values using primitives like `unit` and `flatMap`. Complex state and effects are managed via specialized types—`Stream` (for lazy evaluation and memoization), `Par` (for non-blocking concurrency via the Actor model), and `Free[F[_], A]` (for separating the description of side effects from their execution via trampolining).

The core abstractions Functor, Applicative, and Monad manage computation flow by treating computation as first-class values. The Monoid abstraction, which is foundational, requires a type $A$, an associative binary operation $	ext{op}$ (satisfying $	ext{op}(	ext{op}(x,y), z) = 	ext{op}(x, 	ext{op}(y,z))$ for all $x, y, z: A$), and an identity element $	ext{zero}$ (satisfying $	ext{op}(x, 	ext{zero}) = x$ and $	ext{op}(	ext{zero}, x) = x$). Monoids are useful because they allow complex calculations to be assembled from simpler parts and facilitate parallel computation (e.g., the product of two monoids, $A$ and $B$, is also a monoid, and a monoid can exist for merging key-value Maps, provided the value type is also a monoid).

Functional programming introduces three major abstractions building upon each other: Functor, Applicative, and Monad.

The **Functor** abstraction generalizes the `map` function, allowing `map` to be defined over any type constructor $F[_]$. Implementing the `Functor` trait means a type constructor (like `List`, `Option`, or `F`) is a functor. A core law for a functor is the identity law: mapping over a structure $x$ with the identity function must equal $x$. Using `Functor`, complex structures such as the product type $F[(A, B)]$ can derive functions like `distribute`, which performs an `unzip` operation.

The **Applicative** abstraction builds upon `Functor` and is generally more compositional than `Monad`. The `Applicative` trait defines two primitive combinators: `unit` and `map2` (which combines results from two independent computations via $	ext{map2}[A,B,C](fa: F[A], fb: F[B])(f: (A, B) => C): F[C]$). An `Applicative` instance must implement `map` using `map2` and `unit`, and it also defines `traverse` (a function that performs a computation for every element in a list) and `sequence`. The `Applicative` pattern is adequate when combining results from independent computations, meaning the result of one computation does not influence subsequent computations.

The **Monad** abstraction is the most general interface, extending both `Applicative` and `Functor`. It defines a minimal set of primitives: `unit` and `flatMap`. The `Monad` instance requires implementing `flatMap` (also known as `join` or `compose`), which allows the result of one computation to influence subsequent computations. This dependency on results makes the `Monad` crucial when the computation structure is context-sensitive. The `Monad` trait ensures that all implementations of `Monad[F[_]]` automatically gain `map` and `map2` through the `flatMap` method, specifically: $	ext{def map2}[A,B,C](ma: F[A], mb: F[B])(f: (A, B) => C): F[C] = 	ext{flatMap}(ma)(a => 	ext{map}(mb)(b => f(a, b)))$.

**Key Relationships and Advanced Topics:**

*   **Hierarchy:** All Monads are Functors, but not all Functors are Monads. The `Applicative` trait mandates that all `Applicative` instances are also `Functor` instances. Furthermore, providing default implementations for the `Applicative` methods within the `Monad` trait confirms that all monads are also applicative functors.
*   **Computation Control:** When dealing with external effects, the `IO` monad (or its generalized forms like `Free[F[_], A]`) can be viewed as an embedding of an imperative language into a pure one. The `IO` monad's fundamental value lies in its ability to transform computations into ordinary values, allowing these values to be passed, stored, and reused. To prevent `StackOverflowError`, the `IO` monad redesigns control flow using data constructors (`Return`, `Suspend`, `FlatMap`), implementing a mechanism called trampolining, which processes the sequence of effects in a single loop. The `State` monad (which explicitly tracks and propagates an internal state $S$) and the `Reader` monad (which models a computation dependent on an external environment $R$) show how monads provide structure for sequencing operations, differentiating them from the `Id` monad, which merely wraps a value without altering it.
