# functional-programming-side-effects-buycoffee

**Parent:** [[content/L1/functional-programming-side-effects|functional-programming-side-effects]] — Functional Programming improves modularity and testability by refactoring functions from impure methods with side effects (like directly charging a credit card) into pure functions that return values (like a `Charge` data type), enabling complex business logic such as batching charges using functions like `coalesce` and `buyCoffees`.

To understand the benefits of functional programming (FP), the following example compares implementing a purchase function, `buyCoffee`, both with and without side effects. Initially, the task is to write a Scala program for handling purchases at a coffee shop. The original, impure approach involves a side effect: the `buyCoffee` method (within the `Cafe` class) directly charges the credit card (`cc.charge(cup.price)`). This external interaction, which requires contacting a credit card company via a web service, authorizing the transaction, and persisting a record, causes the code to be difficult to test because testing cannot involve actual charges. Furthermore, this initial implementation is difficult to reuse; for example, if Alice orders 12 cups of coffee, calling `buyCoffee` 12 times charges the payment system 12 separate times, incurring extra processing fees. 

**Functional Improvement (Listing 1.2):**
The revised, functional approach, demonstrates the separation of concerns. The `buyCoffee` method now returns a `(Coffee, Charge)` pair, where `Charge` is a newly introduced data type containing a `CreditCard` and an amount. Instead of executing the charge through a side effect, the function constructs a `Charge` object: `(cup, Charge(cc, cup.price))`. This means the `buyCoffee` function merely creates the charge value, and whatever system processes the payment is not involved. 

This design significantly improves testability and reusability. Since `Charge` is a data type, it can be passed around and combined using the specialized `combine` function, which handles merging charges for the same `CreditCard` (if `cc == other.cc`) or throwing an exception if the cards differ. To purchase $n$ cups of coffee, the new function `buyCoffees(cc: CreditCard, n: Int): (List[Coffee], Charge)` uses higher-order functions like `List.fill(n)(buyCoffee(cc))`, `unzip`, and `charges.reduce((c1,c2) => c1.combine(c2))` to efficiently calculate the total charge by combining all individual charges. The function body is: `def buyCoffees(cc: CreditCard, n: Int): (List[Coffee], Charge) = { val purchases: List[(Coffee, Charge)] = List.fill(n)(buyCoffee(cc)) val (coffees, charges) = purchases.unzip (coffees, charges.reduce((c1,c2) => c1.combine(c2))) }`. This solution is trivial to test because the `Cafe` class is completely ignorant of how the resulting `Charge` values will be processed. Additionally, making `Charge` a first-class value allows for the assembly of complex business logic, such as writing the `coalesce` function to group a list of individual charges by the credit card and combine them into a single total charge: `def coalesce(charges: List[Charge]): List[Charge] = charges.groupBy(_.cc).values.map(_.reduce(_ combine _)).toList`. This function, which uses anonymous functions like `_.cc` and `_ combine _`, highlights the composability of FP. The overall benefit is that the function is reusable and testable without needing mock objects or interfaces. Generally, this separation shows how a function with side effects can be refactored to a pure core, which then returns a value (like `Charge`) that is handled by an outer layer of the program to process the actual effects, thereby preserving the core logic while separating the concern of creation from the interpretation or processing of the value.
