# stream-transducers-and-incremental-io

**Parent:** [[content/L1/advanced-functional-algebra-stream-transducers|advanced-functional-algebra-stream-transducers]] — Advanced functional programming relies on treating computation as a first-class value, using algebraic structures like `Option` and `Either` for explicit error handling and `Stream` for non-strict, memory-efficient data processing. When transitioning to file I/O, the inherent resource safety flaws of lazy I/O are resolved by introducing the `Process[F[_],O]` stream transducer model, which guarantees resource cleanup via explicit finalization blocks and allows high-level combinators like `zipWithIndex` and `exists` to drive stateful, controlled transformations over file resources.

To manage side effects and achieve truly functional purity, developers often use the `IO` monad, which models external effects as pure, composable values. Although the `IO` monad simplifies structured side effect handling, constructing programs using the `IO` monad often leads to monolithic loops. These monolithic loops are difficult to compose, making the resulting code hard to extend later. For instance, if a program needs to count elements and return a value as soon as the count reaches 40,000, this action might occur between several I/O actions. Intertwining the core algorithm with I/O concerns poses a significant barrier to composition. This difficulty is evident when considering variations of the original scenario: checking if the number of non-empty lines in a file exceeds 40,000, or finding a line index before 40,000 where the first letters of consecutive lines spell out "abracadabra". While passing a function of type `String => Boolean` could address the first case, the second case requires modifying the loop to maintain additional state, resulting in code that is not only uglier but also likely tricky to implement correctly.

In general, efficient code written in the `IO` monad typically involves monolithic loops, which lack composability. A better approach is to use a `Stream[String]` to analyze the lines, which allows assembling the program from existing combinators like `zipWithIndex` and `exists`. For example, to check if the index plus one (`_._2 + 1`) is greater than or equal to 40,000, the expression `lines.zipWithIndex.exists(_._2 + 1 >= 40000)` is much cleaner. If the goal is to consider only non-empty lines, the expression becomes `lines.filter(!_.trim.isEmpty).zipWithIndex.exists(_._2 + 1 >= 40000)`. Furthermore, to find the substring "abracadabra" among the first letters of lines, the approach involves chaining three methods: filtering non-empty lines, taking only the first 40,000 lines (`take(40000)`), and then mapping each line to its first character (`map(_.head)`), finally using `indexOfSlice("abracadabra".toList)`.

However, the challenge arises when transitioning from analyzing an in-memory `Stream[String]` to reading from an actual file. The initial attempt to resolve this was writing a function, `def lines(filename: String): IO[Stream[String]] = IO { val src = io.Source.fromFile(filename); src.getLines.toStream append { src.close; Stream.empty } }`, which is called lazy I/O. The problem with lazy I/O is that the resulting `Stream[String]` inside the `IO` monad is not truly a pure value because the stream's elements are read from the file when forced. Specifically, if the argument to `indexOfSlice` does not exist as a subsequence of the input, the function returns -1. While lazy I/O appears appealing because it partially restores compositional style, it is problematic for several reasons: 

1. **Resource Safety**: The file resource will only be released if the entire stream is traversed. If the traversal is terminated early (for example, if `exists` finds a match), the resource remains open, leading to a leak. 
2. **Double Traversal**: There is no mechanism to prevent traversing the same `Stream` after the file has been closed. If the stream is memoized, excessive memory usage will result. If the stream is not memoized, attempting to traverse the stream again will cause an attempt to read from a closed file handle. 
3. **Concurrency**: Since forcing elements of the stream involves I/O side effects, two concurrent threads traversing the same `Stream` may exhibit unpredictable behavior. 
4. **Compositional Knowledge**: In real-world scenarios, developers cannot assume full knowledge of how the `Stream[String]` is used, as it might be passed to uncontrolled functions that store it for a long time before examination. Therefore, the developer must know something about the value's origin, which violates the principle of composition where developers should only need to know the value's type.
