# functional-concurrency-modeling

**Parent:** [[content/L1/functional-abstraction-patterns|functional-abstraction-patterns]] â€” Functional programming mandates explicit state management via generalizations like `State[S,A]` (e.g., `Rand[A] = State[RNG, A]`) and utilizes combinators such as `map`, `map2`, and `sequence` to model state transitions abstractly. Furthermore, functional APIs for parallel computations (`Par[A]`) and lazy streams (`Stream[A]`) solve issues of side effects, excessive intermediate data structure allocation, and forced sequential execution, respectively, by ensuring non-strict evaluation and controlled side-effect timing.

When considering Java's `java.lang.Thread` and `Runnable` classes, a partial API excerpt transcribed into Scala reveals the following structure: `trait Runnable` with a method `def run: Unit`, and `class Thread(r: Runnable)` with methods `def start: Unit` and `def join: Unit`. The `start` method begins running the `Runnable` object in a separate thread, and the `join` method blocks the calling thread until the `Runnable` object finishes execution. A problem arises with both types because neither method returns a meaningful value; therefore, if a developer wants to obtain information from a `Runnable` object, it must incorporate a side effect, such as mutating state for inspection. This reliance on side effects compromises compositionality because developers cannot manipulate `Runnable` objects generally without knowing details about their internal behavior. Furthermore, the `Thread` class has the disadvantage of mapping directly onto operating system threads, which are a scarce resource. Instead, it is preferable to create as many â€ślogical threadsâ€ť as are natural for the problem and later manage the mapping onto actual OS threads. This concept can be addressed by using utilities like `java.util.concurrent.Future`, `ExecutorService`, and friends. For instance, the `ExecutorService` class provides a method `def submit[A](a: Callable[A]): Future[A]`, and the `Future[A]` trait provides a method `def get: A`. Although these primitives help abstract over physical threads, they remain at a lower level of abstraction than the library the text intends to create. Specifically, the `Future.get` method blocks the calling thread until the `ExecutorService` has completed the execution, and the API offers no mechanism for composing futures. Though the library implementation can build on top of these tools, they lack a modular and compositional API suitable for direct use in functional programs. Developers must consider the definition of `unit` and `get`: `unit` could begin evaluating its argument immediately in a separate (logical) thread, or it could simply hold onto its argument until `get` is called and begin evaluation then. Achieving parallelism requires that `unit` begins evaluating its argument concurrently and returns immediately. If `unit` begins evaluating its argument concurrently, then calling `get` arguably breaks referential transparency. For example, replacing the variables `sumL` and `sumR` with their full definitions illustrates this: `Par.get(Par.unit(sum(l))) + Par.get(Par.unit(sum(r)))`. If `unit` starts evaluating its argument immediately, the subsequent call to `get` waits for that evaluation to complete. Consequently, the two sides of the addition sign (`+`) will not run in parallel if the variables `sumL` and `sumR` are simply inlined. This demonstrates that `unit` has a definite side effect, but only with regard to `get`. In this scenario, `unit` simply returns a `Par[Int]`, which represents an asynchronous computation. However, the moment the developer passes that `Par[Int]` object to `get`, the system explicitly waits for the computation to finish, thereby exposing the side effect. Therefore, the goal is to avoid calling `get`, or at least delay calling it until the final step. The objective is to enable combining asynchronous computations without waiting for the individual computations to finish. Because Scala evaluates function arguments strictly from left to right, if `unit` delays execution until `get` is called, the system will first spawn the parallel computation and then wait for it to finish before spawning the second parallel computation. This ensures that the entire computation remains effectively sequential.` ,
