# algebraic-laws-for-parallel-combinators

**Parent:** [[content/L1/functional-programming-advanced-concepts|functional-programming-advanced-concepts]] — Functional programming principles govern complex behavior through algebraic laws, utilizing specialized ADTs like `Option` and `Either` for explicit error handling. Efficiency is maximized via *non-strictness* and `Stream[+A]`, which facilitates automatic loop fusion by deferring computation, enabling safe handling of infinite streams and the corecursive generation of sequences using `unfold`. Concurrency is managed using the `Par[A]` type, which requires adherence to formal laws; specific implementations of `fork` must avoid blocking methods like `Future.get` when operating on fixed-size thread pools to prevent deadlocks.

Functional programming requires treating the API as an algebra governed by specific laws or axioms. These laws define the expected behavior of operations, such as the mandatory law that `map(unit(1))(_ + 1)` must equal `unit(2)`, and the law that `map(y)(id)` must equal `y`. Defining these formal laws is critical because hidden assumptions or side effects can compromise compositionality, making it difficult to treat API components as black boxes. For instance, if the law requiring `fork(x)` to be equivalent to `x` does not hold for all expressions `x` and all choices of `ExecutorService`, general-purpose combinators like `parMap` would lose their soundness, potentially causing deadlocks. A specific subtle bug occurs in most implementations of `fork` when using an `ExecutorService` backed by a fixed-size thread pool (such as those created by `Executors.newFixedThreadPool`). If the maximum number of threads in the pool is limited to one, the following code example will likely deadlock: `val a = lazyUnit(42 + 1)` followed by `val S = Executors.newFixedThreadPool(1)` and `println(Par.equal(S)(a, fork(a)))`. The deadlock arises because the `fork` implementation uses `def fork[A](a: => Par[A]): Par[A] = es => es.submit(new Callable[A] { def call = a(es).get })`. This structure causes the outer `Callable` to be submitted and handled by the single available thread. However, within that same thread, the code then submits and blocks, waiting for the result of a second `Callable` via `a(es).get`. Because the thread pool is limited to only one thread, no available threads exist to execute the second `Callable`, causing the deadlock. To test the robustness of the `fork` implementation, developers should re-examine the law `fork(x) == x` to determine if it holds for the specific implementation or if the law needs refinement, for example, by stipulating a requirement for unbounded thread pools. Alternative implementations for `fork` include: 1) `def fork[A](a: => Par[A]): Par[A] = es => fa(es)`: This avoids deadlock but fails to genuinely fork a separate logical thread for evaluating `fa`, meaning that calling `fork(hugeComputation)(es)` would execute `hugeComputation` in the main thread. This variant, however, is useful because it allows the computation delay until it is actually needed, and is named `delay`. 2) To genuinely run arbitrary computations over fixed-size thread pools, a fully non-blocking representation of `Par` is required. The core problem with the current `Par` representation is that a value cannot be retrieved from a `Future` without the current thread blocking on its `get` method. A representation of `Par` that avoids resource leaks and blocking requires that the implementations of `fork` and `map2` never call any method that blocks the current thread, such as `Future.get`.
