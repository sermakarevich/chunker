Within Scala, a data type's instance can be considered a subtype of another data type's instance if the contained types satisfy a subtype relationship; for example, if `Dog` is a subtype of `Animal`, then `List[Dog]` is a subtype of `List[Animal]`. This principle generalizes: for any types X and Y, if X is a subtype of Y, then `List[X]` is a subtype of `List[Y]`. The original type parameter definition, `sealed trait List[+A]`, uses `+A` to indicate that A is a covariant parameter. Using this annotation is beneficial because `Nil` extends `List[Nothing]`, and since `Nothing` is a subtype of all types, `Nil` can therefore be treated as a `List[Int]`, a `List[Double]`, and so on, fulfilling the desired behavior.

Developers can use pattern matching on functional data structures. For example, the `sum` function, which accepts `List[Int]` arguments, is defined as: `def sum(ints: List[Int]): Int = ints match { case Nil => 0; case Cons(x,xs) => x + sum(xs) }`. This implementation states that the sum of an empty list (`Nil`) is 0, and the sum of a non-empty list is the first element (`x`) plus the sum of the remaining elements (`xs`). Similarly, the `product` function, which accepts `List[Double]` arguments, is defined as: `def product(ds: List[Double]): Double = ds match { case Nil => 1.0; case Cons(0.0, _) => 0.0; case Cons(x,xs) => x * product(xs) }`. These definitions are recursive, a common pattern when writing functions that operate over recursive data types like `List`, which refers to itself recursively in its `Cons` data constructor. When matching a list like `Cons(1, Cons(2, Cons(3, Nil)))`, the structure is used in code such as `List(1, 2, 3) match { case Cons(h, _) => h }`, which results in 1.

Functional data structures are operated on using pure functions, meaning they must not change data in place or perform any side effects; thus, these structures are, by definition, immutable. For instance, concatenating two lists using the syntax `a ++ b` yields a new list without modifying the original lists, $a$ and $b$. The focus is on the singly linked list, which is the `List` data type in Scala's standard library. Defining the type requires a trait: `sealed trait List[+A]`, and `sealed` ensures that all implementations of the trait must be declared in the file. Two data constructors define the two possible list forms: the empty list, denoted by the case object `Nil` (which extends `List[Nothing]`), or a non-empty list, denoted by the case class `Cons[+A](head: A, tail: List[A])` (which extends `List[A]`). The type parameter `+A` indicates that A is a covariant parameter.

Data sharing in functional data structures arises because, when data is immutable, functions that modify a data structure, such as adding or removing elements from a list, do not require physical copying of the data. Adding an element 1 to the front of an existing list, say `xs`, simply creates a new list, `Cons(1, xs)`. Since lists are immutable, the operation reuses the original list `xs`, which is called data sharing. Similarly, removing the element from the front of a list defined as `mylist = Cons(x, xs)` merely involves returning its tail, `xs`; the original list `mylist` remains available and unharmed. Functional data structures are therefore **persistent**, meaning that existing references are never changed by operations on the data structure.

**Advanced List Recursion and Generalization:**
When generalizing operations like `sum` and `product` over lists, developers can use the `foldRight` function, which serves as a powerful higher-order function (HOF). The signature for `foldRight` is `def foldRight[A, B](as: List[A], z: B)(f: (A, B) => B): B`. This function processes the list by pattern matching: it uses `z` as the initial value and `f` (a function) to combine the list element (`A`) with the accumulated result (`B`). For example, implementing `sum2` is done by calling `foldRight(ns, 0)((x, y) => x + y)`, where 0 is the initial value, and the lambda function `(x, y) => x + y` dictates the addition operation.

**Handling Errors: Option and Either**

To improve error handling, a technique discussed replaces throwing exceptions with returning a specialized value indicating if an exceptional condition occurred. This approach leverages generic types and higher-order functions to encapsulate common error handling and error propagation patterns, ensuring the solution is completely type-safe and requires minimal syntactic overhead. 

Before this technique, an example function, `mean(xs: Seq[Double])`, computed the mean of a sequence of doubles. This function is a partial function because it is undefined when the input sequence `xs` is empty. If the program relies on exceptions, the initial definition of `mean` throws an `ArithmeticException` when `xs.isEmpty` is true. Returning a sentinel value (like `Double.NaN` or `null`) or forcing the caller to supply a default value presents several issues: 1) Error propagation is silent, allowing the caller to forget necessary checks; 2) This approach introduces significant boilerplate code at call sites; 3) It fails for polymorphic code; 4) It forces special policies or calling conventions on the callers.

An alternative to exceptions is to introduce a new generic type, `Option[+A]`, to represent explicitly that a function may not always have an answer. This type is defined using the structure: `sealed trait Option[+A]`, which contains two cases: `case class Some[+A](get: A)` for defined values, and `case object None extends Option[Nothing]` for undefined values. By implementing `mean` using `Option[Double]`, the function definition becomes: `def mean(xs: Seq[Double]): Option[Double] = if (xs.isEmpty) None else Some(xs.sum / xs.length)`. This resulting type signature, `Option[Double]`, accurately reflects the possibility that the result may not always be defined, ensuring that `mean` is a total function which always returns a value of the declared type. This mechanism forces the calling code to explicitly handle the possibility of failure, which is a distinct advantage over relying on runtime exceptions or sentinel values. The `Option` data type is frequently used in the Scala standard library, such as when looking up a key in a map or defining methods like `headOption` and `lastOption` for non-empty sequences.

The `Option` data type can be conceptually understood as a specialized list capable of holding at most one element. Consequently, many functions previously discussed for `List` also have analogous implementations for `Option`. The authors place these functions within the body of the `Option` trait for stylistic consistency, allowing invocation using the syntax `obj.fn(arg1)` or `obj fn arg1`, rather than the traditional standalone function syntax `fn(obj, arg1)`.
