# non-blocking-par-future

Efficiency is maximized via *non-strictness* and `Stream[+A]`, which facilitates automatic loop fusion by deferring computation, enabling safe handling of infinite streams and the corecursive generation of sequences using `unfold`. Concurrency is managed using the `Par[A]` type, which requires adherence to formal laws; specific implementations of `fork` must avoid blocking methods like `Future.get` when operating on fixed-size thread pools to prevent deadlocks.

To create a non-blocking representation of the `Par` type, the design introduces a custom `Future` structure, shifting away from using `java.util.concurrent.Future`, which requires blocking to obtain a value. The `Future[A]` trait is declared with a private method `apply(k: A => Unit)`, restricted to the `fpinscala.parallelism` package. The `Par[A]` type is then defined as `ExecutorService => Future[A]`. While the signature of `Par` looks identical to previous versions, its implementation relies on the new non-blocking `Future`. Instead of calling a `get` method to obtain the result, the user provides the new `Future`'s `apply` method a function, `k`, which accepts the result of type `A` and uses it to perform an effect. This function, `k`, acts as a continuation or callback.

By marking `apply` as `private[parallelism]`, the developers ensure that the API remains pure and that all established laws hold, as outside code cannot access this method. Although the internal mechanism of the `Future` is somewhat imperative (using `A => Unit`), the side effects employed are confined and are not observable to code that uses `Par`.

For users, the implementation of the `run` function provides a way to extract the result of a `Par[A]`. The `run` function takes an `ExecutorService` and a `Par[A]` computation, and since it must return a value of type `A`, it necessarily blocks the calling thread until the value is available. The process involves creating a mutable, thread-safe `AtomicReference[A]` to store the result and using a `CountDownLatch(1)`. The `run` implementation calls the `Par` computation, passing a continuation that sets the result in the `AtomicReference` and calls `countDown` on the `CountDownLatch` when the result is received. Finally, the `run` function calls `latch.await` and returns `ref.get`. This design confirms that the calling thread will block until the result of type `A` becomes available and the latch is released. Because of this blocking nature, the authors advise users to only call `run` when they definitely need to wait for a result, or alternatively, modify the API to expose the `apply` method on `Par` directly so that users can register asynchronous callbacks.
