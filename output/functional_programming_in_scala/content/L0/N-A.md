# N-A

**Parent:** [[content/L1/functional-fp-architecture-context|functional-fp-architecture-context]] — The design of functional programming necessitates managing computations as first-class values, utilizing special types ('effects') like Option, Either, Par, and Stream to handle potential failures and side effects without violating referential transparency. Applicative funtors combine independent results using fixed structure (e.g., `map2`), while Monads are required when the result of one calculation influences subsequent steps (e.g., `flatMap`). State management across all traversable functors (like List, Tree) is generalized using the `mapAccum` function, which allows for writing reusable core combinators such as `toList` and `zipWithIndex` for any structure.

The Applicative functor algebra is governed by a final law called naturality. To illustrate this, we can look at an example using the `Option` type.

Retrieving employee names and annual pay:

Given `case class Employee(name: String, id: Int)` and `case class Pay(rate: Double, hoursPerYear: Double)`, the function `format` defined as `def format(e: Option[Employee], pay: Option[Pay]): Option[String]` can combine results using `Applicative[Option]`'s `map2` function:
```scala
val F: Applicative[Option] = ...
def format(e: Option[Employee], pay: Option[Pay]): Option[String] = 
F.map2(e, pay) { (e, pay) =>
  s"{e.name} makes ${pay.rate * pay.hoursPerYear}"
}
val e: Option[Employee] = ...
val pay: Option[Pay] = ...
format(e, pay)
```
Here, the code applies a transformation to the result of `map2`—first, extracting the name from `Employee`, and second, extracting the yearly wage from `Pay`. Alternatively, the developer could apply these transformations separately before calling `format`, providing `format` with an `Option[String]` and `Option[Double]` instead of `Option[Employee]` and `Option[Pay]`.

Refactoring `format`:
```scala
val F: Applicative[Option] = ...
def format(name: Option[String], pay: Option[Double]): Option[String] = 
F.map2(e, pay) { (e, pay) => s"$e makes $pay" } 
val e: Option[Employee] = ...
val pay: Option[Pay] = ...
format(
  F.map(e)(_.name),
  F.map(pay)(pay => pay.rate * pay.hoursPerYear)
)
```
In the refactored version, `format` now accepts the employee name as an `Option[String]` (rather than extracting the name from an `Option[Employee]`), and similarly for the pay amount. This demonstrates that the `Applicative` behavior allows applying transformations either before or after combining values with `map2`. The naturality law states that these two approaches yield the same result.

Formally, the naturality law mandates the equivalence:
`map2(a,b)(productF(f,g)) == product(map(a)(f), map(b)(g))`
Where `productF` is a function combining two functions, defined as: `def productF[I,O,I2,O2](f: I => O, g: I2 => O2): (I,I2) => (O,O2) = (i,i2) => (f(i), g(i2))`

The applicative laws are generally straightforward sanity checks, similar to the monad laws, ensuring that `unit`, `map`, and `map2` function consistently and predictably.

**CHAPTER 12: Applicative and Traversable Functors**

**12.6 Traversable Functors**

Developers discovered Applicative functors by observing that `traverse` and `sequence` did not depend directly on `flatMap`. To generalize these operations, the goal is to find other types beyond `List` that are **Traversable**. The core problem is that `Applicative` abstract interfaces, such as `Applicative`, appear in concrete interfaces, such as `List`, `Map`, and `Tree`. The `Traverse` trait was introduced to abstract over this generalization:

```scala
trait Traverse[F[_]] {
  def traverse[G[_]:Applicative,A,B](fa: F[A])(f: A => G[B]): G[F[B]] =
    sequence(map(fa)(f))
  def sequence[G[_]:Applicative,A](fga: F[G[A]]): G[F[A]] = 
    traverse(fga)(ga => ga)
}
```

The `Traverse` trait generalizes the behavior seen in `traverse` and `sequence` when they operate on `List`. Concrete implementations of `Traverse` are required for `List`, `Option`, and `Tree` (where `Tree[+A]` is defined as `case class Tree[+A](head: A, tail: List[Tree[A]])`).

When analyzing the general signatures of `sequence` and `traverse`, specific instances can be deduced:
*   `List[Option[A]] => Option[List[A]]`: A call to `Traverse[List].sequence` with `Option` as the `Applicative` returns `None` if any element in the input list is `None`; otherwise, it returns the original list wrapped in `Some`.
*   `Tree[Option[A]] => Option[Tree[A]]`: A call to `Traverse[Tree].sequence` with `Option` as the `Applicative` returns `None` if any `Tree` in the input collection is `None`; otherwise, it returns the original `Tree` wrapped in `Some`.
