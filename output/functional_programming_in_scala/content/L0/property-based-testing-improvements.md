# property-based-testing-improvements

**Parent:** [[content/L1/pbt-dependency-refinement-and-parallelism-testing|pbt-dependency-refinement-and-parallelism-testing]] — The Prop structure was refined to accept `MaxSize` in addition to `TestCases` and `RNG`, changing the signature from `(TestCases, RNG) => Result` to `(MaxSize, TestCases, RNG) => Result`. This adaptation was necessary to correctly structure property generation for `forAll` and ensure proper handling of constraints like non-empty lists and parallel computations.

In chapter 8, the discussion details enhancing the application of property-based testing to verify the behavior of parallel computations. For instance, to verify the behavior of `List.sorted` (API documentation link: http://mng.bz/Pz86), one can write a property like `List(2,1,3).sorted` equals `List(1,2,3)`. To express formal laws for parallel computations, such as the law `map(unit(1))(_ + 1) == unit(2)`, developers first noted that using `Prop.forAll(Gen.unit(Par.unit(1)))(i => Par.map(i)(_ + 1)(ES).get == Par.unit(2)(ES).get)` expresses the test, but the detailed implementation is verbose, cluttered, and obscures the test's true intention. 

To simplify this, developers introduced a combinator `def check(p: => Boolean): Prop` on the `Prop` companion object. One initial implementation of `check` uses `forAll(unit(()))(_ => p)` to drive the evaluation of the Boolean argument `p`. However, the developers realized that even with memoization, the test runner would still generate and test multiple test cases, which is undesirable. A more efficient and semantically correct approach for `check` is to construct a `Prop` that ignores the number of test cases: `def check(p: => Boolean): Prop = Prop { (_, _, _) => if (p) Passed else Falsified("()", 0) }`. Although this is simpler, running `run(check(true))` still prints that the property passed `testCases` times, even if the property is only tested once. Therefore, a new result type, `Proved`, was proposed: `case object Proved extends Result`. This required modifying the test runner's `run` function to handle `Proved` by printing `+ OK, proved property.` instead of the standard passed message. 

For demonstrating this enhanced usability, the core test of equality between two parallel computations, such as `Par.map(Par.unit(1))(_ + 1)` and `Par.unit(2)`, was refined. Initially, this involved running both sides separately using `p(ES).get` and `p2(ES).get`. A first improvement was defining an `equal` combinator: `def equal[A](p: Par[A], p2: Par[A]): Par[Boolean] = Par.map2(p,p2)(_ == _)`. This allowed running a single parallel computation to compare both values. Further refinement suggested moving the running of the `Par` values into a specialized function, `forAllPar`. This function signature is `def forAllPar[A](g: Gen[A])(f: A => Par[Boolean]): Prop`. 

To improve the readability of the `forAllPar` implementation, the developers introduced two syntactic improvements: first, the combinator `def **[B](g: Gen[B]): Gen[(A,B)] = (this map2 g)((_,_))` was defined to cleanly combine two generators, $S$ and $g$. Secondly, they introduced a custom extractor object `object **` with an `unapply` function to enable pattern matching syntax for combining generators, allowing `forAllPar` to be written as `forAll(S ** g) { case s ** a => f(a)(s).get }`, where $S$ is a `Gen[ExecutorService]` that varies between fixed-size thread pools (1–4 threads) and an unbounded thread pool, and $g$ is the generator for the input value $A$. This clean syntax allows the property test to be written as: `val p2 = checkPar { equal (Par.map(Par.unit(1))(_ + 1), Par.unit(2)) }`, which greatly improves the code's readability and intent clarity. The authors noted that a similar `forAllPar` version for sized generators should also be considered.
