# property-based-testing-and-sized-generators

**Parent:** [[content/L1/functional-programming-patterns-and-pbt|functional-programming-patterns-and-pbt]] — Functional programming mandates total type safety via ADTs like `Option` and `Either`, and utilizes non-strictness (laziness) via `Stream` to optimize data transformations and handle infinite sequences. Concurrency is managed by `Par[A]`, which relies on the non-blocking Actor model and specific laws (like `fork(x) == x`) to manage parallelism without resource exhaustion. Property-Based Testing (PBT) uses the `Gen[A]` and `Prop` types, where `Prop` is structured as `case class Prop(run: (TestCases, RNG) => Result)` to track testing state accurately.

To understand the requirements for the `Prop` data type, which represents a property, developers must first review the `Gen` representation. The `Gen` representation already provides information regarding the requirements for `Prop`. Initially, `Prop` was defined using the structure: `trait Prop { def check: Either[(FailedCase, SuccessCount), SuccessCount] }`. However, this initial definition lacked critical information regarding the number of test cases the property must examine before it can be considered passed. To abstract over this dependency, developers introduced two types: `type TestCases = Int` and `type Result = Either[(FailedCase, SuccessCount), SuccessCount]`, leading to an initial structure of `case class Prop(run: TestCases => Result)`. Subsequently, it was noted that when a property passes, the number of successful test cases (`SuccessCount`) is implied to be equal to the argument passed to `run`. Since the caller of `run` learns no new information from the success count in the right case of the `Either`, the `Result` type was simplified. The new `Result` type became `type Result = Option[(FailedCase, SuccessCount)]`, and the corresponding `Prop` case class was updated to `case class Prop(run: TestCases => Result)`. Although using `None` to indicate failure was previously established, using `None` to represent the absence of a failure is counter-intuitive. Therefore, a dedicated data type was created to clarify intent, named `Result`. This `Result` sealed trait defines: - `case object Passed extends Result` which indicates that all tests passed and has `def isFalsified = false`. - `case class Falsified(failure: FailedCase, successes: SuccessCount) extends Result` which indicates that the property failed and has `def isFalsified = true`. This new `Result` type leads to the question of whether `Prop` can now be sufficiently represented. The design then re-examines the implementation of `forAll`: `def forAll[A](a: Gen[A])(f: A => Boolean): Prop`. It is realized that `forAll` requires more information than the current `Prop` definition provides. Specifically, if `Prop.run` needs to generate random test cases using the current `Gen` representation, it requires access to a Random Number Generator (RNG). Therefore, the dependency on randomness must be propagated to the `Prop` definition, updating the signature to: `case class Prop(run: (TestCases, RNG) => Result)`. By including `(TestCases, RNG)` as parameters to `Prop.run`, the `Prop` type gains sufficient information to implement `forAll`. The concrete implementation of `forAll` is provided as: `def forAll[A](as: Gen[A])(f: A => Boolean): Prop = Prop { (n, rng) => randomStream(as)(rng).zip(Stream.from(0)).take(n).map { case (a, i) => try { if (f(a)) Passed else Falsified(a.toString, i) } catch { case e: Exception => Falsified(buildMsg(a, e), i) } }.find(_.isFalsified).getOrElse(Passed)`.
