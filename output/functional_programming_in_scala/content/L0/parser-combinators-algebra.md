# parser-combinators-algebra

**Parent:** [[content/L1/functional-algebraic-design-and-concurrency-patterns|functional-algebraic-design-and-concurrency-patterns]] — The API is designed algebraically, requiring explicit use of ADTs like `Option` and `Either` for error handling, and utilizing lazy `Stream`s for memory-efficient data transformation, complemented by `Par` for non-blocking parallel computation (modeled using the Actor pattern). Testing extends to generating complex functions using specialized generators, and parser creation is streamlined by treating parsers as first-class values in a combinator library, avoiding monolithic code generators like Yacc or ANTLR.

The design of a functional API requires modeling the API as an algebra, which consists of a collection of data types, functions over these types, and formal laws or axioms defining expected behavior. The goal is to define an interface, or `Parsers[ParseError, Parser[+_]]` trait, that facilitates building parsers, while remaining agnostic about the specific implementations of `ParseError` and `Parser`. The trait defines the core functionality: a `run` method, `def run[A](p: Parser[A])(input: String): Either[ParseError, A]`, which executes a parser `p` against a given `input` string and returns either a `ParseError` or the successful result `A`. It also provides basic combinators, starting with `char: def char(c: Char): Parser[Char]`, which creates a parser that succeeds only if the input is exactly the character `c` and returns `c`. Furthermore, the trait includes `def string(s: String): Parser[String]`, creating a parser for an entire string. To handle logical alternatives, the `Parsers` trait defines a polymorphic combinator `def or[A](s1: Parser[A], s2: Parser[A]): Parser[A]`, which combines two parsers such that the resulting parser succeeds if either input parser succeeds. Developers can enhance the syntax by adding infix operators, allowing two parsers to be combined using the `|` operator (e.g., `s1 | s2` is equivalent to `or(s1, s2)`).
