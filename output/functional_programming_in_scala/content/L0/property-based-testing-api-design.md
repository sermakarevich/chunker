# property-based-testing-api-design

**Parent:** [[content/L1/functional-programming-algebra-design|functional-programming-algebra-design]] — The synthesis details advanced functional programming concepts, including using specialized ADTs like `Option` and `Either` for type-safe error handling; implementing non-blocking concurrency via Actors for primitives like `map2`; achieving memory efficiency through non-strictness and `Stream` types; and generalizing combinators (like `choice` to `flatMap`) to form a coherent mathematical algebra. Furthermore, it introduces property-based testing (PBT) using generators (`Gen`) to define system behavior abstractly.

In chapter 7, the discussion covered the design of a functional library for expressing parallel computations, introducing the concept that a good API should form an algebra—a collection of data types, functions over these types, and crucially, formal laws or properties that define relationships between these functions. The goal of this chapter is to introduce a simple but powerful library for property-based testing. The general goal of a property-based testing library is to decouple the specification of program behavior from the creation of test cases. The programmer specifies the behavior and provides high-level constraints on the test cases; the framework then automatically generates test cases that satisfy these constraints and runs tests to ensure that the program behaves as specified.

In ScalaCheck, a property-based testing library, a property definition looks like this: `val intList = Gen.listOf(Gen.choose(0,100))` defines a generator of lists of integers between 0 and 100. A property specifies behavior using the `forAll` function, such as `forAll(intList)(ns => ns.reverse.reverse == ns)`, which asserts that reversing a list twice yields the original list. Another example of a property is `forAll(intList)(ns => ns.headOption == ns.reverse.lastOption)`, which asserts that the first element equals the last element after reversal. These properties are checked by invoking `prop.check` (e.g., `scala> prop.check` outputs `+ OK, passed 100 tests.`), or by testing failing properties (e.g., `scala> failingProp.check` outputs `! Falsified after 6 passed tests.`).

The `intList` variable holds a generator of type `Gen[List[Int]]`, which knows how to generate test data of type `List[Int]`. Developers can combine and compose generators—for example, using the `&&` operator to combine two properties—and the `forAll` function creates a property by combining a generator `Gen[A]` with a predicate `A => Boolean`, asserting that all values generated by the generator satisfy the predicate. This set of properties forms a partial specification of the correct behavior of the function being tested.

The goal of property-based testing is not necessarily to fully specify program behavior, but to increase confidence in the code. Developers should perform a cost-benefit analysis to determine if additional work to complete properties is worthwhile. The library has features such as test case minimization, which, if a property fails, attempts to find the smallest failing test case, and exhaustive test case generation, which allows checking all values within a small domain, providing an actual proof rather than just the absence of evidence.

In designing a custom property-based testing library, developers must first determine the required data types, primitives, and laws. For inspiration, the authors analyze the ScalaCheck examples and observe that the `Gen` object (short for generator) must be polymorphic in some type. Thus, `Gen.choose(0,100)` returns a `Gen[Int]`, and `Gen.listOf` is likely a function with the signature `Gen[A] => Gen[List[A]]`. Since `Gen.listOf` should not care about the type of the generator it receives, the authors define it as `def listOf[A](a: Gen[A]): Gen[List[A]]`. Upon examining this signature, the authors recognize that the generator needs explicit information regarding the list size to be implemented. This leads to the proposal of a combinator with the signature `def listOfN[A](n: Int, a: Gen[A]): Gen[List[A]]`. The authors advise keeping in mind that not having to explicitly specify sizes gives the test runner the flexibility to choose test case sizes, which is beneficial for features like test case minimization.
