# non-strictness-and-lazy-evaluation

**Parent:** [[content/L1/functional-fp-laziness-and-error-handling|functional-fp-laziness-and-error-handling]] — Functional programming improves efficiency and robustness by using non-strictness (laziness) to fuse multiple data transformations into a single pass, and by employing algebraic data types like `Option` and `Either` to represent potential failures as values rather than relying on runtime exceptions.

The previous discussion detailed standard functional list operations like `map`, `filter`, and `foldLeft`/`foldRight`, noting that each operation independently traverses the entire input list and constructs a new output list. For instance, executing a sequence like `List(1, 2, 3, 4).map(_ + 10).filter(_ % 2 == 0).map(_ * 3)` involves intermediate list creation and multiple passes. The evaluation of this chain reveals that `map(_ + 10)` creates `List(11, 12, 13, 14)`, which is then passed to `filter(_ % 2 == 0)` to create `List(12, 14)`, and finally this result is passed to `map(_ * 3)` to produce the final list `List(36, 42)`. This sequential execution demonstrates that every transformation generates a temporary list used only as input for the subsequent transformation before being discarded. The core problem is efficiency: multiple passes and the allocation of temporary data structures (intermediate lists) are undesirable. Ideally, a sequence of transformations, such as removing odd-numbered cards followed by flipping over queens in a deck of cards, should occur in a single pass. To achieve this, the programming technique of automatic loop fusion—which involves fusing sequences of transformations—is necessary. This fusion can be accomplished through the use of non-strictness, or laziness. Non-strictness is a property of a function, meaning that the function may choose not to evaluate one or more of its arguments. Conversely, a strict function always evaluates all its arguments. By default, any function definition in Scala, including all previously defined functions, is strict. For example, invoking `square(41.0 + 1.0)` will result in `square` receiving the fully evaluated value of `42.0`. If one attempts to invoke `square(sys.error("failure"))`, the system error is triggered before the `square` function executes its body. Functional concepts like the short-circuiting Boolean operators `&&` and `||` are examples of non-strict operations; for instance, `false && { println("!!"); true }` evaluates to `false` without printing anything, and `true || { println("!!"); false }` evaluates to `true` without printing anything. Furthermore, the `if` control construct in Scala can be viewed as a non-strict function, as it does not necessarily evaluate both the true and false branches. Scala provides explicit syntax for non-strict functions using the `def if2[A](cond: Boolean, onTrue: () => A, onFalse: () => A): A = if (cond) onTrue() else onFalse()`. Here, arguments intended to be passed unevaluated must use the empty function literal syntax `() => A`, representing a function that accepts zero arguments and returns type `A`. A value of type `() => A` is conceptually equivalent to `Function0[A]`. These function literals, when passed as arguments, must be explicitly evaluated using empty parentheses (e.g., `onTrue()` or `onFalse()`) to retrieve a result. However, Scala provides a more convenient, syntactic alias for non-strict arguments using the arrow `=>` before the type, demonstrated by the definition `def if2[A](cond: Boolean, onTrue: => A, onFalse: => A): A = if (cond) onTrue else onFalse`. When using the `=>` syntax, Scala automatically wraps the expression in a thunk (an unevaluated form of an expression).
