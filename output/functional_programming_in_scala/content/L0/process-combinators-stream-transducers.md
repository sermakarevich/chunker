# process-combinators-stream-transducers

**Parent:** [[content/L1/advanced-functional-algebra-stream-transducers|advanced-functional-algebra-stream-transducers]] — Advanced functional programming relies on treating computation as a first-class value, using algebraic structures like `Option` and `Either` for explicit error handling and `Stream` for non-strict, memory-efficient data processing. When transitioning to file I/O, the inherent resource safety flaws of lazy I/O are resolved by introducing the `Process[F[_],O]` stream transducer model, which guarantees resource cleanup via explicit finalization blocks and allows high-level combinators like `zipWithIndex` and `exists` to drive stateful, controlled transformations over file resources.

To manage side effects and achieve true functional purity, developers frequently use the `IO` monad, which models external effects as pure, composable values. This approach, detailed in the context of type-system theory, generalises to abstract structures like `Free[F[_],A]`, where the type parameter $F$ serves as an executable protocol governing all interactions. While `IO` is effective, the core challenge involves handling stream processing and stateful computations, such as counting lines in a file or finding a specific line index. Standard stream transducers often lead to monolithic loops. A better approach is to utilize combinators like `zipWithIndex` and `exists` on a `Stream[String]`, such as using the expression `lines.zipWithIndex.exists(_._2 + 1 >= 40000)` to check if the index plus one (`_._2 + 1`) is greater than or equal to 40,000. If only non-empty lines must be considered, the expression becomes `lines.filter(!_.trim.isEmpty).zipWithIndex.exists(_._2 + 1 >= 40000)`. To locate the substring "abracadabra" among the first letters of lines, one must chain three methods: filtering non-empty lines, taking only the first 40,000 lines (`take(40000)`), and then mapping each line to its first character (`map(_.head)`), followed by using `indexOfSlice("abracadabra".toList)`. 

However, problems arise when transitioning from analyzing an in-memory `Stream[String]` to reading from an actual file. The initial attempt to resolve this was writing a function, `def lines(filename: String): IO[Stream[String]] = IO { val src = io.Source.fromFile(filename); src.getLines.toStream append { src.close; Stream.empty } }`, which uses lazy I/O. This lazy I/O approach causes issues because the resulting `Stream[String]` inside the `IO` monad is not truly a pure value, as the stream's elements are read from the file only when forced. Specific issues with this method include:

1. **Resource Safety**: The file resource is only released if the entire stream is traversed. If the traversal is terminated early (for example, when `exists` finds a match), the resource remains open, causing a leak.
2. **Double Traversal**: There is no mechanism to prevent traversing the same `Stream` after the file has been closed. If the stream is memoized, excessive memory usage occurs. If the stream is not memoized, attempting to traverse the stream again will cause an attempt to read from a closed file handle.
3. **Concurrency**: Since forcing elements of the stream involves I/O side effects, two concurrent threads traversing the same `Stream` may exhibit unpredictable behavior.
4. **Compositional Knowledge**: Developers cannot assume full knowledge of how the `Stream[String]` is used, as it might be passed to uncontrolled functions that store it for a long time before examination. Therefore, the developer must know something about the value's origin, which violates the principle of composition where developers should only need to know the value's type.

To solve these issues and restore a high-level style similar to `Stream` and `List` when performing I/O, the concept of stream transducers, or stream processors, is introduced. A stream transducer specifies a transformation from one sequence to another. To maintain functional purity and handle complex state, dedicated combinators are developed. The `Process[I,O]` type models a stream transducer, allowing construction of complex stream transformations by composing values. For instance, `def liftOne[I,O](f: I => O): Process[I,O] = Await { case Some(i) => Emit(f(i)); case None => Halt() }` converts a single function into a `Process[I,O]`. To transform an entire stream, the combinator `repeat` is implemented: `def repeat: Process[I,O] = go(this)`, where `go(p)` recursively calls `Emit(h, go(t))` or `Await(recv)` to continue the process. 

Specific stream transformations include:

*   **Mapping**: The function `lift[I,O](f: I => O): Process[I,O] = liftOne(f).repeat` allows mapping any function `f` over the entire input stream.
*   **Filtering**: The `filter[I](p: I => Boolean): Process[I,I]` function awaits input and emits the input only if it satisfies the predicate `p` before recursively calling `repeat`, ensuring continuation until the input stream is exhausted.
*   **Stateful Accumulation**: The `sum: Process[Double,Double]` process demonstrates tracking an internal state (the running total). The process uses an inner function `go(acc: Double): Process[Double,Double]` that awaits the next double `d` and emits a running total (`d+acc`), then recursively calls `go(d+acc)` with the new accumulated total.

These common patterns are generalized into the `loop` combinator: `def loop[S,I,O](z: S)(f: (I,S) => (O,S)): Process[I,O]` accepts an initial state `z` and a function `f` that takes the input `I` and the current state `S`, producing a new output `O` and a new state `S`. Finally, more advanced transformations like `map`, `++` (concatenation), and `flatMap` are defined, demonstrating that `Process[I,_]` forms a Monad, which allows composing transformations like `filter(_ % 2 == 0) |> lift(_ + 1)` (a pipeline) to execute complex stream operations, such as expressing `count` and `mean` in terms of the foundational `loop` combinator.
