# functional-api-algebra-laws

**Parent:** [[content/L1/functional-algebra-and-concurrency-patterns|functional-algebra-and-concurrency-patterns]] — The functional API structure is an algebra governed by laws, such as `map(unit(1))(_ + 1) == unit(2)` and `map(y)(id) == y`. Concurrency relies on a non-blocking `Par` representation, solving a deadlock issue in `fork` when using fixed-size thread pools (e.g., `Executors.newFixedThreadPool(1)`) by requiring a fully non-blocking design, often implemented using Actors for non-blocking `map2` and ensuring that combinations like `parMap(List.range(1, 100000))(math.sqrt(_))` do not require a massive number of threads.

The functional API design requires treating the API as an algebra, governed by formal laws or axioms. These laws dictate the expected behavior of operations. Two crucial examples of these laws are: 1) Mapping a unit value of 1 using the `_ + 1` function must equal `unit(2)` (i.e., `map(unit(1))(_ + 1) == unit(2)`). 2) Mapping a value using the identity function must equal the original value (i.e., `map(y)(id) == y`). Defining these formal laws is crucial because hidden assumptions or side effects can compromise compositionality, making it difficult to treat API components as black boxes. For instance, if the law stating that `fork(x)` must be equivalent to `x` for all expressions `x` and all choices of `ExecutorService` does not hold, general-purpose combinators like `parMap` would lose their soundness, potentially causing deadlocks. Specifically, a subtle bug occurs in most implementations of `fork` when the program uses an `ExecutorService` backed by a fixed-size thread pool, such as one created by `Executors.newFixedThreadPool`. If the maximum number of threads in the pool is restricted to one, running the following sequence will likely cause a deadlock: first defining `val a = lazyUnit(42 + 1)`, followed by `val S = Executors.newFixedThreadPool(1)`, and finally calling `println(Par.equal(S)(a, fork(a)))`. The deadlock happens because the `fork` implementation utilizes the definition `def fork[A](a: => Par[A]): Par[A] = es => es.submit(new Callable[A] { def call = a(es).get })`. This structure causes the outer `Callable` to be submitted and handled by the single available thread; within that same thread, the code then submits and blocks while waiting for the result of a second `Callable` via `a(es).get`. Since the thread pool is restricted to only one thread, no available threads exist to execute the second `Callable`, causing the deadlock. Developers must re-examine the law `fork(x) == x` to test the implementation's robustness, determining if the law holds for the specific implementation or if the law needs refinement, perhaps by stipulating a requirement for unbounded thread pools. Developers also have two alternative options for implementing `fork`: 1) Implementing `def fork[A](a: => Par[A]): Par[A] = es => fa(es)` avoids the deadlock but does not genuinely fork a separate logical thread for evaluating `fa`, meaning that calling `fork(hugeComputation)(es)` would execute `hugeComputation` in the main thread. This variant, however, remains useful because it allows the computation delay until it is actually needed and is named `delay`. 2) For genuinely running arbitrary computations over fixed-size thread pools, a fully non-blocking representation of `Par` is required. The fundamental issue with the current `Par` representation is that a value cannot be retrieved from a `Future` without the current thread blocking on its `get` method. Therefore, a `Par` representation that avoids resource leaks and blocking must ensure that the implementations of `fork` and `map2` never call any method that blocks the current thread, such as `Future.get`.
