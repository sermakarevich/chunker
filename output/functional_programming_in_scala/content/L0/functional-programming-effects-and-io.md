# functional-programming-effects-and-io

**Parent:** [[content/L1/functional-algebraic-type-theory|functional-algebraic-type-theory]] — Advanced functional programming uses algebraic structures (Functors, Applicatives, Monads, Free[F,A]) to model computation as first-class, manipulable values, ensuring type safety and referential transparency by explicitly tracking all potential effects, including those related to I/O, concurrency (using Par and the Actor model), and local state mutation (using the ST monad).

While the `IO` monad serves as an effective way to handle external effects and simplify impure code, it is not the only approach. There are other potential effects that may be non-observable, depending on the specific context and observer. For instance, consider the standard Scala side effect visible even in seemingly pure functional programs: object creation. Evaluating `val b = Foo("hello") == Foo("hello")` yields `true` using the `==` function, which is expected. However, evaluating `val c = Foo("hello") eq Foo("hello")` yields `false`, because the two calls to `Foo("hello")` produce distinct object references in memory. If a developer stores the result of `Foo("hello")` in a value `x`, evaluating `x eq x` yields `true`. This demonstrates that the apparent side effect of object creation is only observable through the reference equality method (`eq`). This suggests that, for most programs, object creation is not an actual side effect. This leads to a more general definition of referential transparency: an expression $e$ is referentially transparent with regard to a program $p$ if every occurrence of $e$ in $p$ can be replaced by the result of evaluating $e$ without altering the meaning of $p$. Furthermore, the definition must account for the fact that not all programs observe the same effects. Consequently, an effect of $e$ is non-observable by $p$ if that effect does not affect the referential transparency of $e$ with regard to $p$. In Scala, evaluating an expression $e$ to its normal form—which can be achieved by assigning it to a `val v`—is taken as the standard for evaluation. Thus, $e$ is referentially transparent with regard to $p$ if rewriting $p$ by replacing every appearance of $e$ with $v$ does not change the meaning of $p$.

The core challenge remains defining what constitutes a side effect and, subsequently, what defines the meaning of a program. When discussing referential transparency, the context is paramount, as the context determines what aspects of a program's meaning are considered relevant. For example, the `timesTwo` method, defined as `def timesTwo(x: Int) = { if (x < 0) println("Got a negative number") x * 2 }`, has a definite side effect on standard output. If a program that calls `timesTwo(1)` is replaced by simply using `2`, the computed result is the same, but the meaning of the program has changed because the `println` side effect was lost. This highlights that the program's meaning depends on whether the change in standard output is considered an observable behavior within the current context. The fundamental point is that programmers must decide which effects are important enough to track; this choice is a value judgment. Although specialized mechanisms like the `IO` monad can track specific effects (e.g., `println` calls), this can be overkill, particularly for temporary debugging logs. Instead, programmers should adopt the policy of tracking only those effects that program correctness genuinely depends on. If a program involves file I/O, file I/O effects must be tracked in the type system to the extent feasible. Static type information can inform developers about the types of effects involved, thereby guiding decisions on their importance within a given context.

In addition to these theoretical concerns, implementing programs that perform I/O using the `IO` monad (as shown in Listing 15.1 for counting lines in a file) presents practical issues. While using low-level primitives like `while` loops and mutable `Iterator`s allows the program to be incremental and terminate early, the code is not inherently resource-safe. Programmers must manually ensure the closing of file handles (e.g., using `finally` blocks). Failure to do so, or the closing of files outside of a `finally` block when an exception occurs, results in a resource leak. Since a file handle is a scarce resource, an entire program relying on resource-safe execution must guarantee that resources like file handles, network sockets, and database connections are closed upon both normal termination and exceptional exit. The `IO` monad's reliance on the developer to manage resource safety is a problem, suggesting a need for a library that ensures resource safety by construction.
