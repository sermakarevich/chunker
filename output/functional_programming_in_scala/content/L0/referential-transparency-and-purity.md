Referential transparency (RT) and purity are crucial concepts for functional programming. An expression $e$ is referentially transparent if, for all programs $p$, all occurrences of $e$ within $p$ can be replaced by the result of evaluating $e$ without affecting the meaning of $p$. Similarly, a function $f$ is pure if the expression $f(x)$ is referentially transparent for all referentially transparent values $x$. 

### Referential Transparency, Purity, and the Substitution Model

Consider the initial, impure `buyCoffee` example: `def buyCoffee(cc: CreditCard): Coffee = { val cup = new Coffee(); cc.charge(cup.price); cup }`. Although the return type of the call `cc.charge(cup.price)` (which might be Scala's `Unit`, equivalent to `void` in other languages) is discarded by `buyCoffee`, the result of evaluating `buyCoffee(aliceCreditCard)` will merely be the `cup` object, which is equivalent to `new Coffee()`. For `buyCoffee` to be pure, by the definition of RT, the program $p$ operating on `buyCoffee(aliceCreditCard)` must behave identically to the program $p$ operating on `new Coffee()`, for any program $p$. This condition does not hold because the program `new Coffee()` performs no actions, whereas `buyCoffee(aliceCreditCard)` will contact the credit card company and authorize a charge. This difference represents an observable failure of purity.

Referential transparency mandates the invariant that everything a function does must be represented by the value that the function returns, according to the function's result type. This constraint enables the substitution model, which is a simple and natural mode of reasoning about program evaluation. When expressions are referentially transparent, computation proceeds much like solving an algebraic equation: the evaluation process fully expands every part of an expression, replacing all variables with their referents, and then reduces the expression to its simplest form. At every step, computation proceeds by substituting equals for equals. In essence, RT enables equational reasoning about programs.

We can illustrate the substitution model with examples. In the Scala interpreter (REPL), standard string operations show referential transparency. For instance, when defining `val x = "Hello, World"`, and later calculating `val r1 = x.reverse` and `val r2 = x.reverse`, the results $r1$ and $r2$ are the same. If we substitute the definition of $x$ into the expressions, the results remain unchanged, confirming that $x$ is referentially transparent. Furthermore, $r1$ and $r2$ are also referentially transparent.

Contrast this with operations that violate RT, such as using the `append` function on a `java.lang.StringBuilder` class. If one first initializes `val x = new StringBuilder("Hello")` and then calls `val y = x.append(", World")`, calculating `val r1 = y.toString` and `val r2 = y.toString` yields the same result. However, if one attempts to substitute the call to `append` into the expression, evaluating `val r1 = x.append(", World").toString` and `val r2 = x.append(", World").toString`, the outcomes differ: $r1$ yields "Hello, World", while $r2$ yields "Hello, World, World". This difference proves that `StringBuilder.append` is not a pure function because its side effect (mutating the `StringBuilder` referenced by $x$) alters the program's state during the second evaluation.

Understanding RT allows the substitution model because it simplifies reasoning about program behavior: the effects of evaluation are purely local (affecting only the expression currently being evaluated), meaning programmers do not need to mentally simulate sequences of state updates. Conversely, the substitution model is simple to reason about because functional programs are modular and composable.
