# lazy-streams-in-scala

**Parent:** [[content/L1/functional-fp-laziness-and-error-handling|functional-fp-laziness-and-error-handling]] — Functional programming improves efficiency and robustness by using non-strictness (laziness) to fuse multiple data transformations into a single pass, and by employing algebraic data types like `Option` and `Either` to represent potential failures as values rather than relying on runtime exceptions.

To improve the efficiency and modularity of functional programs, the following section explores using laziness, using lazy lists (or streams) as an example. The goal is to see how chains of transformations on streams can be fused into a single pass using laziness. A stream is defined using the `Stream[+A]` trait, which has two data constructors: `Empty extends Stream[Nothing]` for the empty stream, and `Cons[+A](h: () => A, t: () => Stream[A])` for a non-empty stream. Since the head (`h`) and tail (`t`) of a non-empty stream are non-strict, they must be explicitly forced as thunks rather than being defined by-name parameters.

In Scala, the `Stream` object provides a smart constructor, `cons[A](hd: => A, tl: => Stream[A]): Stream[A]`, designed to create non-empty streams. This constructor caches the head and tail as lazy values, ensuring that the underlying thunk for the head and tail is only executed once when first forced. The `Stream` object also provides `def empty[A]: Stream[A] = Empty` for creating empty streams and `def apply[A](as: A*): Stream[A]` for constructing a stream from multiple elements. When using `apply`, if the sequence of elements `as` is empty, the method returns `Empty`; otherwise, it recursively uses `cons` to construct the stream from the head element and the rest of the list.

When traversing or examining a `Stream`, one must explicitly force the underlying thunks. For example, the function `headOption: Option[A]` extracts the head: it returns `None` if the stream is `Empty`, or `Some(h())` if the stream is `Cons(h, t)`, explicitly forcing the head thunk using `h()`. This capability of `Stream` to evaluate only the portion that is actually demanded—and specifically, not evaluating the tail of the `Cons` node—is considered useful.

Furthermore, a dedicated smart constructor ensures that cached values for the head and tail of the `Cons` node are memoized, preventing repeated computation. While the empty smart constructor merely returns `Empty` (annotated as `Stream[A]`), the non-empty smart constructor `def cons[A](hd: => A, tl: => Stream[A]): Stream[A]` (defined in the `Stream` object) takes care of memoizing the by-name arguments for both the head and tail of the `Cons` node. This pattern guarantees that the underlying thunk will only perform its calculation once upon the first force, and subsequent forces return the cached lazy value. This same mechanism is utilized in the `Stream.apply` function: when constructing a stream from elements `as`, Scala automatically wraps the `as.head` and `apply(as.tail: _*)` expressions in thunks, ensuring they are not evaluated until the `Stream` is explicitly forced.
