**Parent:** [[content/L1/functional-programming-principles|functional-programming-principles]] — Functional programming ensures type safety and non-mutation using patterns like Option[+A] and Either[+A, +B] for explicit error handling, and employs techniques like non-strictness (laziness) to achieve automatic loop fusion and optimize data transformations by avoiding intermediate lists. To maintain referential transparency, mutable state, such as random number generation, must be managed by defining an interface (e.g., `RNG`) that forces methods to return both the resulting value and the new state, rather than using side effects.

Summary: In the preceding chapter, non-strictness was introduced as a fundamental technique enabling efficient and modular functional programs. Non-strictness separates the description of an expression from the time and manner of its evaluation, allowing a description to be reused in multiple contexts while evaluating different portions of the expression to obtain different results. 

Purely Functional State: In this section, the text demonstrates how to write purely functional programs that manipulate state, using the generation of random numbers as an example. The standard library class `scala.util.Random` provides an imperative API that relies on side effects, such as calling `rng.nextDouble` or `rng.nextInt(10)` on a newly created `scala.util.Random` object named `rng` to generate random numbers. Because the object `rng` has an internal state that updates after each method invocation (e.g., `rng.nextDouble` or `rng.nextInt`), these methods are not referentially transparent. This means that calling these methods as side effects makes the code difficult to test, as the sequence of generated values is not reliably reproducible. 

To address this, one suggested solution involves passing the random number generator object, `rng: scala.util.Random`, as an explicit argument to a method, such as `def rollDie(rng: scala.util.Random): Int = rng.nextInt(6)`.
