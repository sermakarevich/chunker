# purely-functional-parallelism-combinators

**Parent:** [[content/L1/functional-par-design-patterns|functional-par-design-patterns]] — Functional programming manages non-fatal control flow using types like `Option[+A]` and `Either[+A, +B]`, while efficiency is achieved through non-strictness and the specialized lazy structure `Stream[+A]`. Parallel computations are modeled abstractly by `Par[A]`, requiring combinators like `map2` and `fork` to ensure concurrent execution and maintain API purity, allowing complex operations like `sum` to be recursively defined and generalized through the derivation of `map` from `map2`.

To combine parallel computations, the design process must consider methods for combining the results of two asynchronous computations. By defining a new higher-order function, `Par.map2`, it can combine the results of two parallel computations, $sum(l)$ and $sum(r)$, into a single overall computation: `Par.map2(sum(l), sum(r))(_ + _)`. When implementing the generalized `sum` function, the function must return `Par[Int]` instead of `Int` to prevent the loss of information. The recursive definition of `sum` must be: if the input sequence `ints` has a size less than or equal to 1, the function returns `Par.unit(ints.headOption.getOrElse(0))`; otherwise, the function splits `ints` into two parts, `l` and `r`, and computes `Par.map2(sum(l), sum(r))(_ + _)`. The `Par.map2` function is a new higher-order function for combining two parallel computations. The most general signature for `Par.map2` is required. When considering if the `unit` function should accept its argument lazily, it is observed that accepting the argument lazily does not appear to provide a benefit in the context of the recursive sum calculation. The choice regarding whether `Par.map2` should take its arguments lazily is critical: `map2` must run both sides of the computation in parallel to ensure that the two computations are treated as independent. To achieve this meaning, the design must ensure that the two computations are inherently independent and can run concurrently. An analysis of what happens if `map2` is strict in both arguments, while evaluating a sequence such as `sum(IndexedSeq(1, 2, 3, 4))`, reveals that because Scala evaluates arguments from left to right, the process would strictly construct the entire left half of the computation tree before starting the right half. This sequential construction contradicts the desired parallel behavior. The goal is to make `map2` lazy such that it initiates immediate parallel execution for both arguments, thereby eliminating any priority given to either side. A more advanced technique, `Par.fork`, can explicitly control when a parallel computation is started. This function allows the programmer to wrap an argument, instructing that the given `Par` value should be executed in a separate logical thread. By using `Par.fork(sum(l))` and `Par.fork(sum(r))` within the `map2` call, the `sum` function can maintain a strict `map2` while explicitly guaranteeing that the two parallel tasks are executed asynchronously. This approach separates the concerns of combining results from the concern of asynchronous execution, which avoids attaching an arbitrary global policy for parallelism to the `map2` combinator. Furthermore, incorporating `Par.fork` allows the `unit` function to become strict without sacrificing expressiveness. The function `lazyUnit[A](a: => A): Par[A]` serves as a derived combinator, using `unit` and `fork` to implement the behavior of `unit` asynchronously. The choice of whether `Par.fork` or `get` should be responsible for initiating thread creation is an architectural decision. It is more appropriate to give `get` the responsibility of creating threads and submitting execution tasks because this design choice gives the calling code fine-grained control over the parallelism strategy used in different parts of the program, instead of relying on a single, globally accessible, and potentially inadequate resource. Reasoning about the required information to spawn a parallel task, rather than knowing the exact implementation of `Par`, led to this conclusion.
