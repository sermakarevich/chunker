# parser-algebra-state-tracking

**Parent:** [[content/L1/parser-algebra-enhancements-2|parser-algebra-enhancements-2]] — The parser algebra was significantly enhanced by introducing a stack-based error structure (`ParseError(stack: List[(Location,String)])`), which allows developers to track error context and is managed by a `push` helper. State tracking requires the `Result` type, which now explicitly carries `charsConsumed: Int`. The new `flatMap` combinator enables context-sensitive parsing by advancing the input location and calculating cumulative consumption, while the `attempt` combinator, coupled with the `isCommitted` flag in `Failure`, controls commitment during ambiguous parsing and the `or` combinator's fallthrough logic.

To effectively implement the parser algebra, which models parsers as state actions, developers must introduce a mechanism to track the characters consumed during parsing. Initially, the design used a `Location` object for this purpose. To improve error reporting granularity, a helper function, `toError`, must be added to `Location`, allowing the creation of a `ParseError` by calling `def toError(msg: String): ParseError = ParseError(List((this, msg)))`. To support parser sequencing—the sequential execution of multiple parsers (for example, parsing "abra" followed by "cadabra")—a parser must explicitly indicate the number of characters it consumed upon successful execution. Therefore, the parser type definition was updated from its original form to a new result type, `Result[+A]`. The updated signature for a parser is now `type Parser[+A] = Location => Result[A]`. The `Result` type is defined as a sealed trait with two primary constructors: `Success[+A](get: A, charsConsumed: Int)` and `Failure(get: ParseError)`. If parsing succeeds, the `Success` case provides both the parsed value of type `A` and `charsConsumed: Int`, which tracks the count of input characters consumed. This consumed character count is critical because the calling combinators use this information to accurately update the `Location` state for subsequent parsing steps. Understanding this principle—that a parser is essentially a state action that can either fail or succeed while providing both a resulting value and the necessary state updates—is crucial for designing a representation that supports all required combinators and formal laws.
