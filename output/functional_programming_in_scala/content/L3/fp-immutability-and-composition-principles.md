# fp-immutability-and-composition-principles

**Parent:** [[index]]

Functional Programming (FP) is a paradigm centered on computing via functions, emphasizing purity and avoiding external state changes. At its theoretical root, a pure function $f$ mapping $A$ to $B$ (`A => B`) must depend exclusively on its input $a$, ensuring that for every value $a$, it maps to exactly one value $b$. This deterministic nature is underpinned by the concept of Referential Transparency (RT), which allows an expression $e$ to be reliably replaced everywhere by its evaluated result without changing the program's meaning. RT enables a powerful substitution model, simplifying modularity and making program evaluation comparable to solving an algebraic equation.

While Scala is sometimes classified as an impure functional programming language because it permits mixed function types, the core philosophy, advocated by Paul Chiusano and Rúnar Bjarnason in *Functional Programming in Scala* (Manning Shelter Island, copyright 2015; ISBN 9781617290657), mandates systematically treating the code as if it were purely functional. They argue that side effects—such as classical I/O operations, exceptions, or mutation—are primarily useful for convenience or interoperability, but overusing them signifies poor practice. The structural goal is to transform systems, like the simulated coffee shop purchase system, from one reliant on side effects into pure, composable cores.

### The Journey from Impurity to Pure Composition

The initial, impure design of the `Cafe` class's `buyCoffee` method included a direct side effect: `cc.charge(cup.price)`. This external interaction, simulating communication with a credit card company, made the function difficult to test because actual charges were triggered. A subsequent attempt at modular improvement involved passing a `Payments` object (`p: Payments`) to `buyCoffee`, which improved testability. However, the authors noted that this design required forcing a concrete class into an interface, and mock implementations of `Payments` might still harbor mutable internal state, necessitating complex checks.

Furthermore, the initial structure suffered from significant reusability hurdles. For example, if Alice ordered 12 cups, calling the impure `buyCoffee` 12 times would execute 12 separate, costly transactions, rather than batching them. This limitation necessitated specialized, complex logic like `BatchingPayments` or methods such as `closeBatch` to manage transactions in groups.

The genuinely functional solution involved radically restructuring `buyCoffee` to return a tuple of type `(Coffee, Charge)`. Here, `Charge` is defined as a new, first-class data type containing both the `CreditCard` and the total amount. Instead of executing a side effect, the function merely constructs the charge value: `(cup, Charge(cc, cup.price))`. This separation of the payment system from the value construction enhances both testability and reusability. Because `Charge` is a first-class value, complex business logic can be composed using a specialized `combine` function, which handles merging multiple charges for the same `CreditCard` (if `cc == other.cc`) or throwing an exception if the cards mismatch.

This leads to the higher-order function `buyCoffees(cc: CreditCard, n: Int): (List[Coffee], Charge)`. It efficiently calculates the total charge for $n$ cups using standard Scala functional capabilities: `List.fill(n)(buyCoffee(cc))`, followed by `unzip` and the `charges.reduce((c1,c2) => c1.combine(c2))` method. The enhanced composability is further shown by the `coalesce` function, which groups arbitrary charges by credit card (`charges.groupBy(_.cc).values.map(_.reduce(_ combine _)).toList`) and reduces them to a single total charge. The use of anonymous functions, such as `_.cc` and `_ combine _`, highlights FP's core ability to compose complex operations.
