# functional-algebra-design-patterns-2

**Parent:** [[content/L3/functional-algebra-design-principles|functional-algebra-design-principles]] — The design utilizes algebraic structures like Functor, Applicative, and Monad (requiring `unit` and `flatMap`) to enforce functional composition across diverse domains, including error handling (`Option`, `Either`), memory-efficient stream processing (`Stream`), and controlled state management in concurrency (`Par`). Advanced systems model external effects via generalized types like `Free[F[_],A]` (replacing `IO`), which separates program description from execution using trampolining, and manage parsing state using structured combinators that track consumption and error stacks.

The development of advanced functional API design—spanning stream processing, error handling, concurrency, and parsing—is founded on the principle of treating computation itself as a first-class value. This necessitates modeling the entire API as a rigorous functional algebra, comprising data types, functions, and strictly defined mathematical laws (axioms). To ensure absolute type safety and prevent runtime failures, best practices mandate the explicit encapsulation of all potential side effects, including error states, utilizing specialized Algebraic Data Types (ADTs) and Higher-Order Functions (HOFs). 

### Algebraic Structures for Control and Error Management

For managing non-fatal control flow, two critical ADTs are utilized: `Option[+A]` and `Either[+A, +B]`. The `Option` type models potential presence or absence of a value, defined by `case class Some[+A](get: A)` for a defined state, and `case object None extends Option[Nothing]` for an undefined state. This structure is essential for defining *total functions*; for example, calculating the mean of an empty sequence must return `None` when using `Option[Double]`, elevating potential failure to a static, compile-time contract. The `Either` type provides a structured, two-sided error mechanism: successful results are wrapped in `Right[+A]`, while failure or error is explicitly carried by `Left[+B]`. These HOFs abstract complex error propagation patterns, ensuring reliable, compile-time checked error handling.

In the specific domain of parsers, the initial `ParseError` structure, which only held a single `Location` and a `String` message, was insufficient. The system was critically upgraded to support a **stack of errors**: `case class ParseError(stack: List[(Location,String)])`. This allows the parser to record the complete context and history of failures. To manage this programmatically, a `push` helper function creates a new `ParseError` by prepending the new error pair to the existing stack: `def push(loc: Location, msg: String): ParseError = copy(stack = (loc,msg) :: stack)`. Furthermore, the parser result must track input consumption. The `Result` sealed trait formalizes this, having two constructors: `Success[+A](get: A, charsConsumed: Int)` and `Failure(get: ParseError)`. The inclusion of `charsConsumed: Int` is paramount, as combinators must use this count to correctly advance the `Location` for subsequent parsing steps. The updated parser signature is: `type Parser[+A] = Location => Result[A]`. This design also incorporates commitment status, extended to `case class Failure(get: ParseError, isCommitted: Boolean) extends Result[Nothing]`. The `attempt` combinator, defined as `def attempt[A](p: Parser[A]): Parser[A] = s => p(s).uncommit`, leverages a `uncommit` function to ensure that a failure does not irreversibly commit the parser state.

### Advanced Combinators and State Management

Key combinators like `flatMap` are crucial for context-sensitive parsing, enabling the selection of a second parser whose definition depends on the successful result of the first. Implementing `flatMap` requires precise state transition management: if the first parser succeeds with value `a` and consumes `n` characters, the input location must be advanced using `advanceBy(n: Int)` (incrementing the offset) before calling the second parser `g(a)`. Success is managed via `advanceSuccess(n: Int)`: `def advanceSuccess(n: Int): Result[A] = this match { case Success(a,m) => Success(a,n+m); case _ => this }`. The structure of `flatMap` is: `def flatMap[A,B](f: Parser[A])(g: A => Parser[B]): Parser[B] = s => f(s) match { case Success(a,n) => g(a)(s.advanceBy(n)).addCommit(n != 0); case e@Failure(_,_) => e }`. Complementing this, the `or` combinator logic respects commitment status: if the first parser `x` fails uncommitted, the second parser `y` is attempted, following the core implementation: `def or[A](x: Parser[A], y: => Parser[A]): Parser[A] = s => x(s) match { case Failure(e,false) => y(s); case r => r }`. Additionally, the `scope` combinator, `def scope[A](msg: String)(p: Parser[A]): Parser[A]`, allows contextual error management by setting the head of the error stack to `msg` if `p` fails, while retaining the previous context.

### Data Flow Abstractions: Functor, Monad, and Streams

Across various domains, algebraic coherence is achieved by defining complex operations from minimal primitives. The **Functor** trait defines `def map[A,B](fa: F[A])(f: A => B): F[B]`, which allows useful functions like `distribute` (for product types) and `codistribute` (for sum/coproduct types) to operate generically. A Functor must satisfy the identity law: `map(x)(a => a) == x`.

Efficiency in data transformation relies on **non-strictness** (laziness). Standard `List` processing, like transforming `List(1, 2, 3, 4)` via `map(_ + 10).filter(_ % 2 == 0).map(_ * 3)`, generates three intermediate lists: `List(11, 12, 13, 14)`, `List(12, 14)`, and `List(36, 42)`, causing high memory overhead. This is solved using `Stream[+A]`, a specialized lazy list trait. Structurally, `Stream` is defined by `Empty` and `Cons[+A](h: () => A, t: () => Stream[A])`. The use of empty function literals ensures both the head (`h`) and tail (`t`) are non-strict (thunks), guaranteeing memoization—evaluation only occurs once upon first demand. The `Stream` object uses the smart constructor `cons[A](hd: => A, tl: => Stream[A]): Stream[A]` to manage this. Core combinators like `map`, `filter`, and `flatMap` chain transformations without intermediate results. The most powerful stream builder is the corecursive function `unfold[A, S](z: S)(f: S => Option[(A, S)]): Stream[A]`, which accepts an initial state $z$ and function $f$, terminating when $f$ returns `None`. `unfold` supports defining infinite streams using `constant[A](a: A)` or `from(n: Int)`, and implementing combinators like `zipWith[B](s2: Stream[B]): Stream[(A, B)]`. Structural utilities include `def startsWith[A](s: Stream[A]): Boolean` and `def tails: Stream[Stream[A]]` (which yields a stream of all suffixes, e.g., for `Stream(1,2,3)`, it is `Stream(Stream(1,2,3), Stream(2,3), Stream(3), Stream()))`).

Following the Functor, the **Monad** aims to unify disparate computational contexts (e.g., `Parser`, `Gen`, `Par`, `Option`) by providing the `flatMap` mechanism. The final `Monad` trait requires `unit` and `flatMap`, generalizing functions like `map2` via: 
$$ 	ext{map2}[A,B,C](	ext{fa}: F[A], 	ext{fb}: F[B])(f: (A,B) 	o C): F[C] = 	ext{flatMap}(	ext{fa})(a 	o 	ext{map}(	ext{fb})(b 	o f(a, b))) $$ 
Implementing `Monad` for `Gen` allows developers to gain `map` and `map2` automatically. This monadic pattern enables advanced operations like `sequence` (List[F[A]] $	o$ F[List[A]]), `traverse` (List[A] $	o$ F[List[B]]), and `replicateM` (Int $	o$ F[List[A]]). 

### Concurrency and Parallelism (Par) and Monoids

Concurrency and parallelism are managed using `Par[A]`, representing a non-blocking computation. This moves away from standard Java blocking mechanisms like `java.util.concurrent.Future`, which mandate a blocking `.get()` call. The core mechanism utilizes a continuation function `k: A => Unit`, ensuring that side effects are confined. The formal laws are governed by the **Actor** model. To prevent deadlocks, such as those observed when executing `println(Par.equal(S)(a, fork(a)))` on a fixed-size thread pool of size one, the Actor processes incoming messages (e.g., `Left(a)` or `Right(b)`), storing state (e.g., `ar: Option[A]` and `br: Option[B]` for `map2`). It waits for the second message before executing the combining function $f$. This non-blocking design allows combinators like `parMap(List.range(1, 100000))(math.sqrt(_))` to utilize a limited thread pool (e.g., two threads) efficiently, thereby validating the forking law even under resource constraints. 

Algebraically, the **Monoid** is a structure defined by a type $A$, an associative binary operation `op`, and an identity element `zero`. Monoids facilitate sophisticated composition. For instance, the **Map Merge Monoid** allows defining a monoid for `Map[K, V]` where $V$ is itself a monoid. This is implemented by `mapMergeMonoid[K,V]`, which defines the merge operation using `foldLeft` over the union of key sets, combining values using the nested monoid operation `V.op`.

In summary, the entire design relies on defining complex primitives (like `map2`, `flatMap`, `product`) algebraically in terms of a minimal set of foundational building blocks (`unit` and `flatMap`), guaranteeing robust, compile-time checked functional composition across diverse contexts.
