# functional-laziness-and-error-handling

**Parent:** [[content/L3/fp-immutability-and-composition-principles|fp-immutability-and-composition-principles]] — FP utilizes Referential Transparency and first-class functions to manage state, transforming functions into compositions of pure values, such as the `Charge` tuple, which avoids side effects like the initial `cc.charge(cup.price)` during coffee purchases.

Functional programming deeply embeds the concept of computation itself being treatable as a value. Instead of designing systems where potential exceptions are handled as mandatory control flow paths, best practices involve encapsulating these effects—including error states—using specialized algebraic data types (ADTs) and higher-order functions (HOFs). This approach is fundamental for ensuring complete type safety, forcing the developer to explicitly handle all possible outcomes and thus preventing the silent failures common with traditional sentinel values or simple runtime checks. The general principle adopted throughout functional programming is the representation of exceptions as ordinary, first-class values.

To manage non-fatal control flow, two specialized types are paramount: `Option[+A]` and `Either[+A, +B]`. The `Option` type is designed to model the presence or absence of a value, having two distinct cases: `case class Some[+A](get: A)` for a defined value, and `case object None extends Option[Nothing]` for an undefined state. This design is crucial because it allows functions that might not always yield a result, such as calculating the mean of a sequence, to be designated *total functions*. For instance, if a mean calculation function uses `Option[Double]`, it will explicitly return `None` when the input sequence is empty. This makes the potential failure state an inherent, static part of the function's contract, representing a massive improvement over the unreliable reliance on runtime exceptions and boilerplate `try-catch` blocks. Complementing this, the `Either` type offers a structured mechanism to carry one of two specific outcomes: success, typically represented by `Right[+A]`, or a failure/error value, often represented by `Left[+B]`. By leveraging these HOFs, complex error propagation patterns can be abstracted away, significantly minimizing boilerplate code while guaranteeing robust, compile-time checked error handling. These concepts of handling non-fatal control flow—whether failure or success—are universal and generalize far beyond simple options and left/right structures.

Efficiency concerns in functional programming often revolve around chaining multiple data transformations (e.g., applying sequential `map`, `filter`, and subsequent `map` calls) on data structures like standard Scala `List`. Executing this sequence traditionally requires multiple passes over the initial data and, critically, generates numerous temporary, intermediate data structures (intermediate lists). For example, processing `List(1, 2, 3, 4)` through the sequence `map(_ + 10).filter(_ % 2 == 0).map(_ * 3)` would first yield `List(11, 12, 13, 14)`, then create a second list, `List(12, 14)`, and finally a third, `List(36, 42)`. This overhead of excessive memory allocation and computational burden due to multiple passes is highly undesirable. The solution is the implementation of automatic loop fusion, which is achieved through the employment of *non-strictness*, or laziness.

Non-strictness is a core functional property meaning a function is allowed to not evaluate all its arguments. Conversely, a strict function always evaluates every argument provided. While Scala treats function definitions as strict by default, the contrast is best seen in short-circuiting Boolean operators like `&&` and `||`, which only evaluate arguments as required. The standard `if` control construct also exhibits non-strict behavior; it does not necessarily evaluate both the true and false branches. To enforce controlled non-strictness, Scala introduced `if2[A]`: `def if2[A](cond: Boolean, onTrue: () => A, onFalse: () => A): A = if (cond) onTrue() else onFalse()`. Here, the empty function literal `() => A` (conceptually `Function0[A]`) ensures the provided block (e.g., `onTrue` or `onFalse`) only runs if the conditional check mandates it. A more syntactically convenient alias uses the arrow `=>` (e.g., `def if2[A](cond: Boolean, onTrue: => A, onFalse: => A): A = if (cond) onTrue else onFalse`), automatically wrapping the expression in a 'thunk' (an unevaluated form of an expression) that must be explicitly forced for evaluation.

This concept finds its most robust and explicit implementation in `Stream[+A]`, a specialized trait designed to model lazy lists. A stream has two constructors: `Empty extends Stream[Nothing]` for the empty stream, and `Cons[+A](h: () => A, t: () => Stream[A])` for a non-empty one. The definition using `() => A` and `() => Stream[A]` is structurally critical, forcing both the head (`h`) and tail (`t`) components to be non-strict (thunks). This guarantees that the value of the head and the computation of the tail are evaluated only once upon the first demand, a mechanism known as memoization. The `Stream` object provides the smart constructor `cons[A](hd: => A, tl: => Stream[A]): Stream[A]`, which manages this memoization. The `Stream.apply[A](as: A*)` constructor builds streams from multiple elements `as`; if `as` is empty, it returns `Empty`; otherwise, it recursively uses `cons`, automatically wrapping both the initial head and subsequent tail components in thunks, thereby preserving laziness. 

Accessing stream components requires explicit handling of thunks. For instance, `headOption: Option[A]` returns `None` if the stream is `Empty`, or `Some(h())` if it is `Cons(h, t)`, explicitly calling `h()` to force the head thunk while crucially *not* forcing the tail. To facilitate inspection, utilities like `take(n)` (returns the first $n$ elements), `drop(n)` (skips the first $n$ elements), and `takeWhile(p: A => Boolean)` (returns starting elements satisfying predicate $p$) are available. 

Advanced stream operations benefit immensely from laziness. Core combinators such as `map`, `filter`, `append`, and `flatMap` chain transformations without instantiating intermediate results. For example, `Stream(1,2,3,4).map(_ + 10).filter(_ % 2 == 0).toList` demonstrates that the computation generates elements one by one, saving memory by allowing the garbage collector to reclaim space for values (like 11 and 13) as soon as they are no longer needed. This efficiency is paramount when dealing with infinite streams. An infinite stream of ones, `val ones: Stream[Int] = Stream.cons(1, ones)`, can be safely processed: `ones.take(5).toList` yields `List(1, 1, 1, 1, 1)`, and `ones.exists(_ % 2 != 0)` returns `true` immediately, as it terminates upon finding the first satisfying element. However, care must be taken; operations like `ones.forAll(_ == 1)` will cause an infinite loop (stack overflow) because the predicate never finds a non-matching element that allows termination. 

Developers can generate specialized streams using advanced methods. `def constant[A](a: A): Stream[A]` generates an infinite stream of $a$. `def from(n: Int): Stream[Int]` generates an infinite sequence of integers starting from $n$. `def fibs` generates the infinite Fibonacci sequence (0, 1, 1, 2, 3, 5, 8, etc.). A general, corecursive function is `def unfold[A, S](z: S)(f: S => Option[(A, S)]): Stream[A]`. `unfold` takes an initial state $z$ and a function $f$ that determines both the next state and the next value, terminating when $f$ returns `None`. The power of `unfold` is its ability to redefine core builders: `def fibs` (EXERCISE 5.12), `def from` (EXERCISE 5.12), `def constant` (EXERCISE 5.12), and `def ones` (EXERCISE 5.12). Using `unfold` to define `constant` and `ones` maintains constant memory usage, improving upon simple recursive definitions. Furthermore, `unfold` can implement other combinators, including `map` (EXERCISE 5.13), `take` (EXERCISE 5.13), `takeWhile` (EXERCISE 5.13), and `zipWith[B](s2: Stream[B]): Stream[(A, B)]` (EXERCISE 5.13), which follows the Chapter 3 `zipWith` pattern. A paired function, `zipAll[B](s2: Stream[B]): Stream[(Option[A], Option[B])]` (EXERCISE 5.13), continues traversing as long as either input stream has elements, utilizing `Option` to track stream exhaustion. 

Advanced combinators add structural depth. `def startsWith[A](s: Stream[A]): Boolean` (EXERCISE 5.14) checks if stream $s$ is a prefix of another (e.g., `Stream(1,2,3)` starts with `Stream(1,2)` returns `true`). From this, `def hasSubsequence[A](s: Stream[A]): Boolean` (EXERCISE 5.14) can be implemented by checking if any tail of the stream starts with $s$: `tails exists (_ startsWith s)`. An even more powerful structural method is `def tails: Stream[Stream[A]]` (EXERCISE 5.15`), which returns a stream of all suffixes of the input sequence. For instance, for `Stream(1,2,3)`, `tails` generates `Stream(Stream(1,2,3), Stream(2,3), Stream(3), Stream())`.
