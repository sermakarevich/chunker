# functional-programming-design-patterns

**Parent:** [[content/L3/functional-fp-design-principles|functional-fp-design-principles]] — Functional programming mandates type safety by modeling computation as first-class values, using specialized ADTs like `Option[+A]` (for optional values) and `Either[+A, +B]` (for explicit error handling, Right/Left). Efficiency is maintained via non-strictness, which is explicitly implemented using `Stream[+A]` for lazy lists, enabling core combinators like `map` and `filter` to avoid intermediate data structure creation. Stateful and concurrent logic is managed using combinators such as `map2` (combining two actions) and `fork` (asynchronous execution), utilizing principles derived from the Actor model to prevent deadlocks and maintain type-safe composition in parallel contexts.

The principles of robust, modern software design, particularly within the functional programming paradigm, are founded on core concepts that ensure type safety, manage side effects, and optimize computational efficiency through sophisticated abstractions. At its heart, functional programming treats computation itself as a value, mandating the explicit handling of potential side effects and error states to maintain complete type safety. This approach replaces reliance on mandatory control flow paths for exceptions or simple sentinel values with specialized algebraic data types (ADTs) and higher-order functions (HOFs). 

### Handling Control Flow and State

To model computations that may not always yield a result, the `Option[+A]` type is essential. It explicitly defines two distinct states: `case class Some[+A](get: A)` for a present value, and `case object None extends Option[Nothing]` for an undefined state. This design allows functions, such as calculating the mean of an empty sequence, to be correctly designated as *total functions*. For instance, a mean function returning `Option[Double]` guarantees that an empty input results in a statically defined `None`. Complementing this, the `Either[+A, +B]` type provides a structured way to carry one of two specific outcomes: success, typically encapsulated by `Right[+A]`, or failure/error, represented by `Left[+B]`. Leveraging these HOFs minimizes boilerplate while guaranteeing robust, compile-time checked error handling. 

State management follows a similar pattern. Since functional structures prohibit mutation, state transitions must be handled explicitly to preserve **referential transparency**. In domains like Random Number Generation (RNG), generating distinct random integers requires passing the updated state. The process is formalized by defining a type alias, `type Rand[+A] = RNG => (A, RNG)`. This 'state action' represents a computation that accepts an initial `RNG` state, yields a value of type `A`, and concurrently returns the updated `RNG` state. For example, the function `randomPair` demonstrates this necessity: it receives `rng: RNG`, uses it for the first call (`val (i1, rng2) = rng.nextInt`), and then critically uses the returned state (`rng2`) as the input for the second call (`val (i2, rng3) = rng2.nextInt`), yielding the final state (`rng3`) along with the results `((i1, i2), rng3)`. 

To simplify this manual state passing, specialized combinators are developed. The **unit action** `def unit[A](a: A): Rand[A] = rng => (a, rng)` accepts the initial `RNG` state and passes it through unchanged, returning constant value `a`. The **`map` combinator** allows transforming the output of a state action without altering the state. Since `Rand[A]` is functionally equivalent to `RNG => (A, RNG)`, `map` is implemented via function composition: `def map[A,B](s: Rand[A])(f: A => B): Rand[B] = rng => { val (a, rng2) = s(rng); (f(a), rng2) }`. For instance, transforming `nonNegativeInt` into `nonNegativeEven` uses `map(nonNegativeInt)(i => i - i % 2)`.

When combining two independent state actions, **`map2`** is necessary. This takes two actions, `ra` and `rb`, and a combining function `f: (A, B) => C`, resulting in `def map2[A,B,C](ra: Rand[A], rb: Rand[B])(f: (A, B) => C): Rand[C]`. To generate pairs, `both[A,B]` is a specialization: `def both[A,B](ra: Rand[A], rb: Rand[B]): Rand[(A,B)] = map2(ra, rb)((_, _))`. Generating an (Int, Double) pair uses `val randIntDouble: Rand[(Int, Double)] = both(int, double)`.

Advanced combination structures include **`sequence`**, which collects a list of state actions `fs: List[Rand[A]]` into a single coherent state transition: `def sequence[A](fs: List[Rand[A]]): Rand[List[A]]`. This pattern is utilized to generate a list of random integers.

For conditional and sequential control flow, the powerful **`flatMap`** combinator is crucial. Its signature is `def flatMap[A,B](f: Rand[A])(g: A => Rand[B]): Rand[B]`. `flatMap` allows an initial random value of type `A` to determine which subsequent state action, `g: A => Rand[B]`, is executed. This capability is key for complex logic, such as implementing `nonNegativeLessThan(6)`. The `rollDie` function demonstrates the need for transformation: the initial definition `def rollDie: Rand[Int] = nonNegativeLessThan(6)` requires correction using `map` to achieve the correct result (between 1 and 6): `def rollDie: Rand[Int] = map(nonNegativeLessThan(6))(_ + 1)`.

These combinators—`unit`, `map`, `map2`, `sequence`, and `flatMap`—provide a unified, functional way to manage stateful computations, guaranteeing determinism and explicit control over state flow, regardless of the specific state type (be it `RNG`, or a generalized `State[S, A]`).

### Computational Efficiency and Laziness (Streams)

When addressing performance bottlenecks in data transformations, traditional sequence processing (like chaining `map`, `filter`, and `map` on a standard Scala `List`) is highly inefficient because it requires multiple passes over the initial data and generates numerous temporary, intermediate structures. The solution lies in leveraging **non-strictness**, or laziness. Non-strictness is a property allowing a function to not evaluate all its arguments, contrasting with strict functions. This concept is evident in short-circuiting operators like `&&` and `||`, or the specialized `if2[A]`: `def if2[A](cond: Boolean, onTrue: () => A, onFalse: () => A): A = if (cond) onTrue() else onFalse()`. Using the empty function literal `() => A` guarantees execution only when the condition dictates it.

The concept is best implemented using `Stream[+A]`, a specialized trait modeling lazy lists. A stream is structurally defined by two components: `Empty extends Stream[Nothing]` and `Cons[+A](h: () => A, t: () => Stream[A])`. The use of `() => A` and `() => Stream[A]` forces both the head (`h`) and the tail (`t`) to be non-strict (thunks). This mechanism ensures that the head value and the computation of the tail are evaluated only once upon first demand—a process known as memoization. The smart constructor `cons[A](hd: => A, tl: => Stream[A]): Stream[A]` manages this memoization. Furthermore, `Stream.apply[A](as: A*)` automatically wraps components in thunks, preserving laziness. Accessing components is explicit: `headOption: Option[A]` calls `h()` but crucially *does not* force the tail.

Advanced stream operations benefit profoundly from laziness. Combinators like `map`, `filter`, `append`, and `flatMap` chain transformations without creating intermediate results. Processing `Stream(1,2,3,4).map(_ + 10).filter(_ % 2 == 0).toList` generates elements one by one, allowing the garbage collector to reclaim memory for discarded intermediate values (like 11 and 13).

This efficiency is vital for infinite streams. An infinite stream of ones, `val ones: Stream[Int] = Stream.cons(1, ones)`, can be safely handled. Operations like `ones.take(5).toList` yield `List(1, 1, 1, 1, 1)`, and `ones.exists(_ % 2 != 0)` terminates immediately upon finding `true`. However, `ones.forAll(_ == 1)` results in an infinite loop because the predicate never finds a non-matching element to permit termination. Developers can generate streams using `constant[A](a: A)`, `from(n: Int)`, or `fibs`. The most generalized method is `unfold[A, S](z: S)(f: S => Option[(A, S)]): Stream[A]`. `unfold` accepts an initial state $z$ and function $f$, terminating when `None` is returned. `unfold` redefines core builders (e.g., `fibs`, `from`, `constant`, `ones`) while maintaining constant memory usage. 

Structurally, `Stream` supports advanced combinators. `def startsWith[A](s: Stream[A]): Boolean` checks if $s$ is a prefix of another stream. This leads to `hasSubsequence[A](s: Stream[A])`, implemented by checking `tails exists (_ startsWith s)`. The utility method **`tails: Stream[Stream[A]]`** generates a stream of all suffixes of the input sequence (e.g., for `Stream(1,2,3)`, it yields `Stream(Stream(1,2,3), Stream(2,3), Stream(3), Stream())`).

### Immutability and Parallelism

Beyond functional abstraction, core data management principles persist. Data structures in functional programming are fundamentally immutable; operations, such as concatenating two lists using `a ++ b`, always produce a new structure without modifying the originals. The standard list type, `List[+A]`, is defined by `Nil` and `Cons[+A](head: A, tail: List[A])`. While the principles of data sharing make structures **persistent** (elements are reused), efficiency considerations exist: calculating `init` cannot be implemented in constant time like `tail` because the tail replacement requires copying all preceding `Cons` objects. Developers can generalize list operations using `foldRight`: `def foldRight[A, B](as: List[A], z: B)(f: (A, B) => B): B`, which processes the list by pattern matching, combining the element with the accumulated result.

In the context of parallel computing, functional approaches advocate designing highly modular and composable APIs, moving away from direct reliance on concurrency primitives like `java.lang.Thread`. Since primitives like `Runnable` and `Thread` use side effects (e.g., state mutation) and lack return values, they compromise compositionality. A goal is to design 'logical threads' that abstract beyond scarce physical OS resources. To maintain true non-blocking parallelism, the abstraction must solve the side-effect problem inherent in blocking calls like `Future.get`. The mechanism must delay the side-effect until the final step, ensuring that parallel components are defined without sequentially calling `get`.

In summary, functional programming establishes comprehensive mechanisms—from `Option`/`Either` for control flow to `Stream` for memory-efficient iteration and state monads (`Rand[A]`) for reproducible state transitions—all united by the principle of immutability and explicit state handling, making complex systems composable, predictable, and scalable, whether sequential or parallel.
