# advanced-functional-algebra-design

**Parent:** [[content/L3/functional-algebra-design-principles|functional-algebra-design-principles]] — The design utilizes algebraic structures like Functor, Applicative, and Monad (requiring `unit` and `flatMap`) to enforce functional composition across diverse domains, including error handling (`Option`, `Either`), memory-efficient stream processing (`Stream`), and controlled state management in concurrency (`Par`). Advanced systems model external effects via generalized types like `Free[F[_],A]` (replacing `IO`), which separates program description from execution using trampolining, and manage parsing state using structured combinators that track consumption and error stacks.

The overarching design principle for modern functional programming APIs is to model the entire system—not just the type signatures—as a functional algebra. This approach mandates that the API must comprise defined data types, associated functions over those types, and, critically, a set of rigorously formal laws or axioms that define the expected behavior. The primary objective is to achieve total type safety and prevent silent or hidden failures by mandating the explicit encapsulation of all potential effects, including error states. This is achieved using specialized Algebraic Data Types (ADTs) and Higher-Order Functions (HOFs).

For managing non-fatal control flow and error states, the core mechanisms are `Option[+A]` and `Either[+A, +B]`. The `Option` type models the potential presence or absence of a value, having two explicit cases: `case class Some[+A](get: A)` for a defined value, and `case object None extends Option[Nothing]` for an undefined state. This structure is indispensable for defining *total functions*, which guarantees, for instance, that calculating the mean of an empty sequence must deterministically return `None` when using `Option[Double]`. The `Either` type provides a structured, two-sided mechanism for handling errors: `Right[+A]` carries the successful result, while `Left[+B]` represents the failure state. These HOFs abstract complex error propagation patterns, elevating potential failure states to a mandatory, compile-time contract, thereby vastly improving upon reliance on manual `try-catch` blocks and sentinel values.

Efficiency in data processing is fundamentally tied to the concept of *non-strictness*, or laziness. Traditional operations on standard collections like `List` are highly inefficient when chaining multiple transformations (e.g., `map(_ + 10).filter(_ % 2 == 0).map(_ * 3)`) applied to `List(1, 2, 3, 4)`. Such execution requires multiple distinct passes and generates numerous temporary intermediate data structures: `List(11, 12, 13, 14)`, then `List(12, 14)`, and finally `List(36, 42)`, resulting in intensive memory allocation overhead. This is mitigated by employing laziness. While concepts like short-circuiting Boolean operators (`&&`, `||`) or the `if` construct demonstrate this principle (e.g., `def if2[A](cond: Boolean, onTrue: () => A, onFalse: () => A): A = if (cond) onTrue() else onFalse()`), the mechanism is most robustly and explicitly realized in `Stream[+A]`, a specialized trait for lazy lists.

A `Stream` is defined by two constructors: `Empty extends Stream[Nothing]` and `Cons[+A](h: () => A, t: () => Stream[A])`. The use of empty function literals (`() => A` and `() => Stream[A]`) is structurally critical, forcing both the head (`h`) and tail (`t`) components to be non-strict (thunks). This guarantees that both the head value and the tail computation are evaluated only once upon first demand, a process known as memoization. The `Stream` object utilizes the smart constructor `cons[A](hd: => A, tl: => Stream[A]): Stream[A]`, which manages this memoization. Accessing stream components necessitates explicit thunk handling; for example, `headOption: Option[A]` returns `None` if the stream is `Empty`, or `Some(h())` if it is `Cons(h, t)`, ensuring `h()` is called without forcing the tail computation.

Stream utility methods include `take(n)` (retrieving the first $n$ elements), `drop(n)` (skipping the first $n$ elements), and `takeWhile(p: A => Boolean)` (returning elements that satisfy a predicate $p$). Advanced core combinators—`map`, `filter`, `append`, and `flatMap`—chain transformations seamlessly without generating intermediate results, a capability crucial for handling infinite streams. For instance, processing an infinite sequence defined by `val ones: Stream[Int] = Stream.cons(1, ones)` and then calling `ones.take(5).toList` generates the results sequentially, allowing the garbage collector to reclaim memory space efficiently. The most general and powerful stream builder is the corecursive function `unfold[A, S](z: S)(f: S => Option[(A, S)]): Stream[A]`, which accepts an initial state $z$ and a function $f$ that determines the next state and value, terminating when $f$ returns `None`. This function supports the definition of `constant[A](a: A)` (an infinite stream of $a$) and `from(n: Int)` (an infinite sequence starting from $n$). Furthermore, `Stream` supports structural methods such as `startsWith[A](s: Stream[A]): Boolean` (checking prefix relationship) and `tails` (returning a stream of all suffixes; e.g., for `Stream(1,2,3)`, this yields `Stream(Stream(1,2,3), Stream(2,3), Stream(3), Stream())`). A paired function, `zipAll[B](s2: Stream[B]): Stream[(Option[A], Option[B])]`, traverses as long as either input stream contains elements, using `Option` to track exhaustion.

### Modeling Concurrency and Parallelism (Par)

Concurrency and parallelism are managed using the `Par[A]` data type, specifically designed to represent a non-blocking computation that moves away from standard Java blocking mechanisms like `java.util.concurrent.Future`, which require calling `.get()`. Instead, `Par[A]` utilizes a private method `apply(k: A => Unit)` that accepts a continuation function $k$, which handles the result $A$ and performs subsequent effects. The core mathematical laws of parallelism are vital for compositionality, including `map(unit(1))(_ + 1) == unit(2)` and `map(y)(id) == y`.

Formalizing the `fork` combinator requires ensuring the law `fork(x) == x` holds, meaning wrapping computation $x$ does not change its result; it merely executes asynchronously. A failure scenario illustrates this: using an `ExecutorService` with a fixed pool size of one (e.g., `Executors.newFixedThreadPool(1)`), running `println(Par.equal(S)(a, fork(a)))` for `val a = lazyUnit(42 + 1)` will cause a deadlock. This is because the internal `fork` implementation attempts to submit and handle the outer `Callable` on the single available thread, and within that same thread, it blocks waiting for the result of a second `Callable` via `a(es).get`. Since the pool is limited to one, no resources remain for the second computation.

To maintain true non-blocking execution, `fork` and `map2` must never call blocking methods like `Future.get()`. The ideal solution for genuine parallel execution on fixed-size thread pools is to model the API using the **Actor** model. An Actor acts as a concurrent process managing state (e.g., tracking two mutable variables: `ar: Option[A]` and `br: Option[B]` for `map2`). It processes incoming messages sequentially: upon receiving the first message (e.g., `Left(a)` or `Right(b)`), it stores it and waits for the second result. Upon receiving the second result, the Actor executes the combining function $f$ and propagates the final $C$ result. This non-blocking Actor design enables operations like `parMap(List.range(1, 100000))(math.sqrt(_))` to run efficiently using only a fixed-size thread pool (e.g., two threads) without allocating 100,000 JVM threads, thereby successfully validating the law of forking even under severe resource constraints. Complex combinations are managed by `map2[A,B,C](p: Par[A], p2: Par[B])(f: (A,B) => C): Par[C]`, relying on the foundational relationship that `flatMap` and `join` relate to `map` and `map2`.

### Property-Based Testing Utilities (Gen, Prop, and PBT) 

To facilitate Property-Based Testing (PBT), two core concepts are introduced: `Gen[A]` (Generator) and `Prop` (Property). The `forAll` function initiates the property check: `def forAll[A](g: Gen[A])(f: A => Boolean): Prop`. Structurally, the definition of `Prop` has undergone significant refinement due to the need for explicit reporting of test status. Initially, attempts were made with complex `Either` types, but this led to simplification. The final, robust structure uses a dedicated sealed trait `Result` to manage reporting: - `case object Passed` indicates success and has `def isFalsified = false`. - `case class Falsified(failure: FailedCase, successes: SuccessCount)` indicates failure and has `def isFalsified = true`. Since `forAll` must generate random test cases, the `Prop` definition must propagate the dependency on randomness. Thus, the signature was updated to: `case class Prop(run: (TestCases, RNG) => Result)`. The function signature for `forAll` reflects this, and the internal implementation uses `randomStream[A](g: Gen[A])(rng: RNG)` to generate an infinite stream of $A$ values by repeatedly sampling $g$. 

Regarding `Gen`, it is defined as: `case class Gen[A](sample: State[RNG, A])`. This allows basic functions like `choose` (generating integers in the range `start` to `stopExclusive`), `unit` (always generating value $a$), and `listOfN` (generating lists of length $n$). For generating dependent values, the crucial function is `flatMap`, which allows one generator to use its output value to determine the subsequent generator. This ability enables generalizing `listOfN` to support variable lengths by implementing `listOfN(size: Gen[Int]): Gen[List[A]]` using `flatMap`.

For implementation detail, `forAll` is defined as: `def forAll[A](as: Gen[A])(f: A => Boolean): Prop = Prop { (n, rng) => randomStream(as)(rng).zip(Stream.from(0)).take(n).map { case (a, i) => try { if (f(a)) Passed else Falsified(a.toString, i) } catch { case e: Exception => Falsified(buildMsg(a, e), i) } }.find(_.isFalsified).getOrElse(Passed)`. The helper function `buildMsg[A](s: A, e: Exception)` constructs a detailed failure message string combining details from the failed value and the exception.

The PBT framework also supports enhancing usability via the `Prop` companion object's helper method `run`, which simplifies the testing workflow by accepting default arguments for all three dependencies: `maxSize` (defaulting to 100), `testCases` (defaulting to 100), and `rng` (defaulting to `RNG.Simple(System.currentTimeMillis)`). 

Beyond value properties, PBT is extended to verify concurrency and parallel computations, necessitating specialized combinators. A significant readability improvement was achieved by introducing the specialized combinator `def check(p: => Boolean): Prop` on the `Prop` companion object, which allows expressing abstract laws, such as `map(unit(1))(_ + 1) == unit(2)`, in a highly readable manner.

These functional design paradigms, whether applied to data structures (`Stream`), parallelism (`Par`), or testing methods (`Prop`), underscore the necessity of defining the API structure using an algebraic approach, ensuring that core combinators like `map` share analogous mathematical laws across their diverse domains (e.g., `Par` map shares signature with `Gen` map, `Stream` map, and `Option` map).

### Advanced Topics: Parsers and Generics

When extending PBT, developers are encouraged to consider defining sized generators for specialized data structures, such as the Tree type, and using this to specify the behavior of the fold function for that structure, leveraging the API's structural properties.

In the domain of parsing, the library adopts the parser combinator approach, treating parsers as first-class algebraic values. It defines an interface, or `Parsers[ParseError, Parser[+_]]` trait, which facilitates building parsers while remaining agnostic about `ParseError` and `Parser` implementations. Basic combinators include `char: def char(c: Char): Parser[Char]` and `def string(s: String): Parser[String]`. The polymorphic combinator `def or[A](s1: Parser[A], s2: Parser[A]): Parser[A]` handles logical alternatives. Furthermore, `def listOfN[A](n: Int, p: Parser[A]): Parser[List[A]]` recognizes exactly $n$ repetitions of $p$. Such a structured, algebraic definition allows complex APIs, even those dealing with context-sensitive grammars where parser choice depends on preceding results, to be built robustly by defining the algebra first, and then implementing the concrete data representation.
