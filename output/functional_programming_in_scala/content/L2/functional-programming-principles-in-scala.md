# functional-programming-principles-in-scala

**Parent:** [[content/L3/fp-immutability-and-composition-principles|fp-immutability-and-composition-principles]] — FP utilizes Referential Transparency and first-class functions to manage state, transforming functions into compositions of pure values, such as the `Charge` tuple, which avoids side effects like the initial `cc.charge(cup.price)` during coffee purchases.

Functional Programming (FP) represents a rigorous programming style that emphasizes computation via functions, often referred to as "programming with functions." At its core, pure functional programming dictates that a function's output must be determined exclusively by its inputs, making it fundamentally ignorant of any external or internal state changes. Formally, a pure function `f` mapping type $A$ to $B$ (`A => B`) must map every value $a$ of type $A$ to exactly one value $b$ of type $B$, meaning the computation of $b$ depends only on $a$. This strict adherence to determinism is formally enabled by the concept of Referential Transparency (RT), where an expression $e$ can be reliably replaced everywhere by its evaluated result without altering the program's meaning. RT enables a powerful substitution model, allowing program evaluation to proceed like solving an algebraic equation, which inherently simplifies modularity and composability.

While the theoretical ideal of pure FP restricts functions to behave as binary relations, mapping strictly to results, Scala is classified as an impure functional programming language because it permits both pure and impure functions without syntactic or type distinctions. However, the authors of *Functional Programming in Scala*, Paul Chiusano and Rúnar Bjarnason, strongly advocate for the systematic use of pure functions. They argue that side effects—such as exceptions, classical I/O operations, or mutation—are useful for reasons of interoperability, efficiency, or convenience, but that overusing them constitutes poor programming style. Structurally, they treat Scala as if it were a pure functional language, systematically eliminating or mitigating signs of impurity, such as mutable variables, exceptions, and classical input/output, proving that all concepts can be expressed purely.

The strength of FP lies in its ability to increase modularity through pure functions. This inherent modularity makes pure functions significantly easier to test, reuse, parallelize, generalize, and reason about, while also making them less prone to bugs. The key conceptual advancement is the separation of the core business logic (value creation) from the external effects (value interpretation or processing). This refactoring strategy transforms a function that initially required side effects into a pure core function that merely returns a specialized value, such as a `Charge` object.

### The Pedagogical and Practical Demonstration: From Impure to Pure

The book’s material uses a simulated coffee shop purchase system to demonstrate this shift. The initial approach, encapsulated in a `Cafe` class (Listing 1.1), was impure: the `buyCoffee` method used a direct side effect, exemplified by the line `cc.charge(cup.price)`. Charging the credit card requires external interactions (e.g., communicating with a credit card company, authorizing a transaction, and persisting a detailed record), thus constituting a non-observable side effect. Due to this side effect, the initial code was difficult to test because test procedures could not involve making actual charges.

The attempt at modular improvement (Listing 1.2) involved passing a `Payments` object (`p: Payments`) into `buyCoffee`: `def buyCoffee(cc: CreditCard, p: Payments): Coffee = { val cup = new Coffee(); p.charge(cc, cup.price); cup }`. While this regained some testability because `Payments` could be modeled as an interface, the authors criticized this approach. Passing `Payments` required forcing a concrete class into an interface, and any mock implementation of `Payments` might contain internal mutable state that necessitated explicit, complex checks to confirm the state was correctly modified after the call to `charge`.

Furthermore, the initial structure faced critical reusability issues. If a customer, Alice, ordered 12 cups, calling the impure `buyCoffee` 12 times would trigger 12 separate, expensive transactions, instead of consolidating them. Solving this required specialized, complex logic, such as writing a `BatchingPayments` implementation or having the `buyCoffee` function explicitly signal the completion of a batch via a method like `closeBatch`.

### The Pure Functional Solution

The genuinely functional improvement achieved maximum separation of concerns. The `buyCoffee` method was redesigned to return a tuple of type `(Coffee, Charge)`, where `Charge` is a new, first-class data type that holds both the `CreditCard` and the total amount. Instead of executing a side effect, the function merely constructs the charge value: `(cup, Charge(cc, cup.price))`. The payment system remains conceptually separate from this construction, allowing the pure function to generate the charge value without invoking any external commands. This fundamental change greatly enhances both testability and reusability.

Since `Charge` is a first-class data value, complex business logic can be composed using a specialized `combine` function. This function handles merging multiple charges for the same `CreditCard` (if `cc == other.cc`) or throws an exception if the cards differ. To efficiently calculate the total charge for $n$ cups of coffee, the new higher-order function, `buyCoffees(cc: CreditCard, n: Int): (List[Coffee], Charge)`, utilizes the Scala standard library's functional capabilities: `List.fill(n)(buyCoffee(cc))`, `unzip`, and the `charges.reduce((c1,c2) => c1.combine(c2))` method. Its full body is defined as: `def buyCoffees(cc: CreditCard, n: Int): (List[Coffee], Charge) = { val purchases: List[(Coffee, Charge)] = List.fill(n)(buyCoffee(cc)) val (coffees, charges) = purchases.unzip (coffees, charges.reduce((c1,c2) => c1.combine(c2))) }`.

This pure approach is trivially testable because the `Cafe` class is isolated from how the resulting `Charge` values are processed. The enhanced composability is further demonstrated by the `coalesce` function, which groups an arbitrary list of individual charges by the credit card (`charges.groupBy(_.cc).values.map(_.reduce(_ combine _)).toList`) and reduces them into a single total charge. The use of anonymous functions, such as `_.cc` and `_ combine _`, highlights the profound composability inherent to FP.

### Implementation Details and Structural Guidance

The book, *Functional Programming in Scala*, authored by Paul Chiusano and Rúnar Bjarnason and published by Manning Shelter Island (copyright 2015, ISBN 9781617290657), serves as a comprehensive deep dive into FP principles. It is structured into four parts: Part 1 (Introduction to FP, starting p. 1), Part 2 (Functional Design and Combinator Libraries, starting p. 93), Part 3 (Common Structures in Functional Design, starting p. 173), and Part 4 (Effects and I/O, starting p. 227). The content is designed to be read sequentially, although the sections allow for comfortable breaks.

The overall structure is designed to teach the principles of FP, with Scala serving merely as the vehicle for instruction. Part 4, which covers real-world applications, I/O (including working with databases, files, or video displays), and mutable state, is intended to be understood only after the reader has built a strong foundational understanding of the functional style established in Parts 1, 2, and 3.

From a pedagogical standpoint, the authors emphasize that mastering FP demands active participation through coding, not just theoretical understanding. Each chapter generally introduces a new idea or technique, explains it with an example, and reinforces it with a sequence of exercises. Readers are strongly advised to complete the dedicated exercise source code, and support materials (hints and answers) are available at https://github.com/fpinscala/fpinscla. For further discussion, support channels include the scala-functional Google group (https://groups.google.com/forum/#!topic/scala-functional/) and the #fp-in-scala IRC channel on irc.freenode.net.

Syntactically, the book provides detailed guidance: a method is initiated using `def`, parameters are listed as `(param: Type)`, and the return type is specified with `: Type`. The single equals sign `=` separates the signature from the definition, and the result of the right-hand side is implicitly returned. Moreover, the authors teach advanced functional features, such as the treatment of functions as first-class values, which allows them to be passed as arguments to other functions (Higher-Order Functions or HOFs), a concept vital for highly abstract and purely functional code. The core concepts covered include defining purely functional data structures, organizing small functional programs, handling errors, and managing state, before escalating to generalized libraries and techniques.

The authors initially conceived the audience as intermediate-level Java or C programmers, but assert that the material is suitable for programmers from any language or professional background, emphasizing that motivation and curiosity are paramount. While the principles taught are language-agnostic, readers needing deeper Scala syntax knowledge should consult official resources like http://scala-lang.org/documentation/books.html.
