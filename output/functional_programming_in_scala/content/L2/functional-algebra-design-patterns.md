# functional-algebra-design-patterns

**Parent:** [[content/L3/functional-fp-design-principles|functional-fp-design-principles]] — Functional programming mandates type safety by modeling computation as first-class values, using specialized ADTs like `Option[+A]` (for optional values) and `Either[+A, +B]` (for explicit error handling, Right/Left). Efficiency is maintained via non-strictness, which is explicitly implemented using `Stream[+A]` for lazy lists, enabling core combinators like `map` and `filter` to avoid intermediate data structure creation. Stateful and concurrent logic is managed using combinators such as `map2` (combining two actions) and `fork` (asynchronous execution), utilizing principles derived from the Actor model to prevent deadlocks and maintain type-safe composition in parallel contexts.

The design of modern functional programming languages mandates treating computation itself as a first-class value. This foundational principle fundamentally alters how control flow and side effects are managed, requiring the API to be modeled as a functional algebra—a set of data types, functions, and rigorously defined laws. To ensure complete type safety and prevent silent failures, best practices eschew traditional mechanisms like runtime exceptions or sentinel values, instead mandating the explicit encapsulation of all potential effects, including error states, using specialized Algebraic Data Types (ADTs) and Higher-Order Functions (HOFs). 

### Managing Control Flow and Errors (ADTs)

For managing non-fatal control flow, two specialized types are paramount: `Option[+A]` and `Either[+A, +B]`. The `Option` type models the potential presence or absence of a value, defined by two cases: `case class Some[+A](get: A)` for a defined value, and `case object None extends Option[Nothing]` for an undefined state. This design is vital for defining *total functions*. For instance, calculating the mean of a sequence must return `None` when the input sequence is empty. By using `Option[Double]`, the potential failure is elevated to a static, compile-time contract, vastly improving upon boilerplate `try-catch` blocks. Complementing this, the `Either` type provides a structured, two-sided mechanism: success is carried by `Right[+A]`, and failure/error is represented by `Left[+B]`. These HOFs abstract complex error propagation patterns, ensuring reliable, compile-time checked error handling.

### Optimizing Data Transformation with Laziness (Streams)

Efficiency concerns frequently arise when chaining multiple data transformations—such as sequential `map`, `filter`, and subsequent `map` calls—on standard collections like `List`. Traditional execution requires multiple passes over the initial data and generates numerous temporary, intermediate data structures. For example, processing `List(1, 2, 3, 4)` through the sequence `map(_ + 10).filter(_ % 2 == 0).map(_ * 3)` would necessitate creating three separate intermediate lists: `List(11, 12, 13, 14)`, followed by `List(12, 14)`, and finally `List(36, 42)`. This intensive memory allocation and computational burden is mitigated by employing *non-strictness*, or laziness.

Non-strictness dictates that a function is permitted to not evaluate all its arguments, contrasting with strict functions. While short-circuiting Boolean operators (`&&`, `||`) and the `if` construct demonstrate this concept (e.g., `def if2[A](cond: Boolean, onTrue: () => A, onFalse: () => A): A = if (cond) onTrue() else onFalse()`), the concept is most robustly and explicitly implemented in `Stream[+A]`, a specialized trait for lazy lists. A `Stream` is defined by two constructors: `Empty extends Stream[Nothing]` for an empty stream, and `Cons[+A](h: () => A, t: () => Stream[A])` for a non-empty one. Structurally, the use of empty function literals (`() => A` and `() => Stream[A]`) is structurally critical, forcing both the head (`h`) and tail (`t`) components to be non-strict (thunks). This guarantees that both the head value and the tail computation are evaluated only once upon the first demand, a process known as memoization. The `Stream` object provides the smart constructor `cons[A](hd: => A, tl: => Stream[A]): Stream[A]`, which manages this memoization. The `Stream.apply[A](as: A*)` constructor builds streams recursively using `cons`, automatically wrapping both initial head and subsequent tail components in thunks.

Accessing stream components requires explicit thunk handling; for example, `headOption: Option[A]` returns `None` if the stream is `Empty`, or `Some(h())` if it is `Cons(h, t)`, explicitly calling `h()` while crucially *not* forcing the tail. Utility methods include `take(n)` (retrieving the first $n$ elements), `drop(n)` (skipping the first $n$ elements), and `takeWhile(p: A => Boolean)` (returning elements satisfying predicate $p$).

Advanced stream operations benefit immensely from laziness. Core combinators like `map`, `filter`, `append`, and `flatMap` chain transformations without creating intermediate results. For example, `Stream(1,2,3,4).map(_ + 10).filter(_ % 2 == 0).toList` demonstrates computation generating elements sequentially, saving memory by allowing the garbage collector to reclaim space. This efficiency is critical when dealing with infinite streams. One can safely process `val ones: Stream[Int] = Stream.cons(1, ones)` using `ones.take(5).toList` (yielding `List(1, 1, 1, 1, 1)`) or `ones.exists(_ % 2 != 0)` (returning `true` immediately). However, developers must avoid infinite loops, such as with `ones.forAll(_ == 1)`, which causes a stack overflow because the predicate never finds a non-matching element allowing termination. Developers can generate specialized streams using `constant[A](a: A)` (infinite stream of $a$) or `from(n: Int)` (infinite sequence starting from $n$). The most generalized and powerful builder is the corecursive function `unfold[A, S](z: S)(f: S => Option[(A, S)]): Stream[A]`, which takes an initial state $z$ and a function $f$ determining the next state and value, terminating when $f$ returns `None`. `unfold` can define builders like `constant`, `from`, and `ones`, and implement combinators like `map`, `take`, `takeWhile`, and `zipWith[B](s2: Stream[B]): Stream[(A, B)]`. A paired function, `zipAll[B](s2: Stream[B]): Stream[(Option[A], Option[B])]`, traverses as long as either input stream has elements, using `Option` to track exhaustion.

Structural methods include `def startsWith[A](s: Stream[A]): Boolean` (checking prefix relationship) and the critical `def tails: Stream[Stream[A]]` (returning a stream of all suffixes; e.g., for `Stream(1,2,3)`, this yields `Stream(Stream(1,2,3), Stream(2,3), Stream(3), Stream())`).

### Concurrency and Parallelism (Par)

Concurrency and parallelism are managed using the `Par[A]` data type, which is fundamentally designed to represent a non-blocking computation. To achieve this, the design must move away from standard Java blocking mechanisms like `java.util.concurrent.Future`, which require calling `.get()` to obtain a value. Instead, `Par[A]` utilizes a private method `apply(k: A => Unit)` that accepts a continuation function `k`, which handles the result `A` and performs subsequent effects. While the internal mechanism involves side effects (`A => Unit`), these are confined and not observable to the user code.

The challenge of formalizing `fork` is highlighted by the requirement that the law `fork(x) == x` must hold, asserting that wrapping a computation `x` using `fork` does not alter its result; it merely executes `x` asynchronously. If this law fails, general-purpose combinators like `parMap` lose soundness, potentially causing deadlocks. A specific, concrete failure case occurs when utilizing an `ExecutorService` backed by a fixed-size thread pool, such as `Executors.newFixedThreadPool(1)`. If the pool size is one, the sequence: `val a = lazyUnit(42 + 1)` followed by `val S = Executors.newFixedThreadPool(1)` and running `println(Par.equal(S)(a, fork(a)))` will deadlock. This happens because the internal `fork` implementation, `es => es.submit(new Callable[A] { def call = a(es).get })`, submits and handles the outer `Callable` on the sole available thread, and within that same thread, it blocks waiting for the result of a second `Callable` via `a(es).get`. Since the thread pool is limited to one, no threads are available for the second computation, causing the deadlock.

To circumvent this, alternative implementations were proposed: 1) `def fork[A](a: => Par[A]): Par[A] = es => fa(es)` (a form of `delay`), which avoids the deadlock but fails to genuinely initiate a separate logical thread for evaluating `fa`. Instead, `fork(hugeComputation)(es)` executes `hugeComputation` on the main thread, although it effectively delays the computation until needed. 2) The ideal solution for genuinely parallel execution on fixed-size thread pools requires a fully non-blocking `Par` representation. This means that `fork` and `map2` must never call methods that block the current thread, such as `Future.get`.

The core mathematical laws governing parallelism are crucial for maintaining compositionality. Examples include: `map(unit(1))(_ + 1) == unit(2)` (mapping a unit value of 1 using `_ + 1` equals `unit(2)`) and `map(y)(id) == y` (mapping a value `y` using identity returns `y`).

Sophisticated combinators like `map2[A,B,C](p: Par[A], p2: Par[B])(f: (A,B) => C): Par[C]` necessitate advanced concurrency management. The recommended solution uses an **Actor** model. An Actor functions as a concurrent process that manages state (e.g., maintaining two mutable variables, `ar: Option[A]` and `br: Option[B]` for `map2`) and processes messages sequentially. The Actor processes the first message (e.g., `Left(a)` or `Right(b)`), stores it, and waits for the second result. Upon receiving the second result, the Actor executes the combining function `f` and propagates the final `C` result. This non-blocking design ensures that operations like `parMap(List.range(1, 100000))(math.sqrt(_))` can run efficiently using a fixed-size thread pool (e.g., two threads) without requiring 100,000 JVM threads, successfully validating the law of forking even under resource constraints. This rigorous framework requires understanding that the `flatMap` and `join` primitives, for instance, relate fundamentally to the `map` and `map2` combinators, allowing complex API structures to be built from a minimal, algebraically coherent set of primitives. This entire structure culminates in the necessity of using Property-Based Testing (PBT), where properties are defined abstractly (e.g., `ns.reverse.reverse == ns`) and checked using a generator (`Gen`), providing increased developer confidence and acting as a powerful form of verification.
