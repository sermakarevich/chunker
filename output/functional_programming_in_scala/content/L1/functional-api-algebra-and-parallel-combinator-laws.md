# functional-api-algebra-and-parallel-combinator-laws

**Parent:** [[content/L2/functional-paradigm-architecture|functional-paradigm-architecture]] — Functional programming abstracts computations into first-class values, using ADTs like Option and Either to handle non-fatal errors. Efficiency is achieved via non-strictness (laziness), implemented using `Stream[+A]` to fuse operations and handle infinite sequences, while parallelism is managed by the `Par[A]` type, which defers execution until a terminal `run` call and uses combinators like `map2` and `fork` to maintain purity and manage resource-intensive effects.

Functional API design requires adopting an algebraic approach, treating the API as a set of operations coupled with specific axioms, or 'laws.' This formal method of defining behavior is critical because side effects and hidden assumptions can severely compromise compositionality, making it difficult to treat components as true black boxes. By establishing these laws, developers can focus on formal reasoning and ensuring the overall structural integrity of the system. For instance, if the fundamental law positing that `fork(x)` is equivalent to `x` for all expressions `x` and all choices of `ExecutorService` does not hold, general-purpose combinators like `parMap` would lose their soundness, potentially leading to highly dangerous issues, such as system-level deadlocks. 

Concrete examples of these laws guide implementation choices. One property is the generalization of how `map` interacts with unit values: `map(unit(x))(f) == unit(f(x))`. The law `map(unit(1))(_ + 1) == unit(2)` demonstrates this. The core concept is that the implementation of `unit` must strictly pass along its received value, and the underlying `ExecutorService` must not make behavioral changes based on submitted `Callable` objects. These laws place strong constraints on the implementations of combinators like `map` and `unit`, prohibiting techniques such as typecasing (e.g., `downcasting` or `isInstanceOf` checks) within their implementations. Furthermore, this leads to a simplified law: `map(y)(id) == y`, where `y` represents `unit(x)`, which proves that the mention of `unit` was logically extraneous to the primary function of `map` due to the parametricity of `map`.

Another foundational law is `map(y)(id) == y` and the law `map(unit(x))(f) == unit(f(x))`. A direct, concrete law to test is `map(unit(1))(_ + 1) == unit(2)`. This establishes that mapping a unit value of 1 using the function `_ + 1` should be mathematically equivalent to simply receiving the result of `unit(2)`.

The combinator `fork` itself must adhere to the law `fork(x) == x`, which asserts that wrapping a computation `x` using `fork` should not alter the result of the parallel computation; it must only execute `x` asynchronously in a separate logical thread from the main thread. Writing down such laws necessitates developers to thoroughly analyze potential corner cases and prove the law's validity. This process guides the design of the functional API.

However, the practical implementation of `fork` poses significant challenges, particularly when utilizing an `ExecutorService` backed by a fixed-size thread pool, such as one created by `Executors.newFixedThreadPool`. If the maximum number of threads is limited to one, a specific code structure designed to test the law `fork(x) == x` will likely deadlock. Consider the sequence: `val a = lazyUnit(42 + 1)` and `val S = Executors.newFixedThreadPool(1)`. Running `println(Par.equal(S)(a, fork(a)))` demonstrates this critical failure. The deadlock occurs because the `fork` implementation, `def fork[A](a: => Par[A]): Par[A] = es => es.submit(new Callable[A] { def call = a(es).get })`, results in the outer `Callable` being submitted and handled by the sole available thread. Within that same thread, the executed code then submits and blocks, waiting for the result of a second `Callable` via `a(es).get`. Since the thread pool is restricted to one thread, there are no available threads to execute this second internal computation, resulting in a deadlock.

To avoid deadlocks while still maintaining the structure of parallel computation, alternative implementations for `fork` were considered. One such variant, `def fork[A](a: => Par[A]): Par[A] = es => fa(es)`, successfully avoids the deadlock but fails to genuinely initiate a separate logical thread for evaluating `fa`. This specific variant only allows the computation delay until it is actually required, meaning that `fork(hugeComputation)(es)` would execute `hugeComputation` on the main thread, rather than truly paralleling it. The inherent difficulty lies in the fact that, for a value to be retrieved from a `Future`, the current thread must block on its `get` method, and the representation of `Par` must be capable of avoiding resource leaks and blocking at all times. Achieving a non-blocking representation for `Par` that truly supports `fork` and `map2` requires a fundamental overhaul of how parallel results are represented.

In analogy with other functional programming constructs, the `map` combinator is required to be structure-preserving. It must only alter the value contained within the parallel computation (`Par[A]`) without changing the overall structure of the computation itself.
