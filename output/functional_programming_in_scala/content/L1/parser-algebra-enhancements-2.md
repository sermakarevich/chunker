# parser-algebra-enhancements-2

**Parent:** [[content/L2/functional-algebra-design-patterns-2|functional-algebra-design-patterns-2]] — The advanced functional API is built on a functional algebra, requiring the explicit use of ADTs like `Option` and `Either` for controlled error/control flow, and relies on `Stream[+A]` for lazy, memory-efficient data transformation by avoiding intermediate lists. Concurrency uses `Par[A]` governed by the non-blocking Actor model, while complex state and structure composition is unified by the Monad type class (requiring `unit` and `flatMap`) and specialized algebraic structures like the Monoid (e.g., `mapMergeMonoid`).

The development of advanced parser combinator libraries requires a sophisticated foundational design that models computation, state changes, and failure modes using rigorous functional concepts. A core principle is treating computation as a first-class value, necessitating the modeling of the entire API as a functional algebra. To ensure type safety and avoid runtime exceptions, the design mandates the explicit use of Algebraic Data Types (ADTs) and Higher-Order Functions (HOFs) for managing side effects and control flow.

**I. Modeling State and Errors using ADTs**

For managing non-fatal control flow, two specialized types are essential: `Option[+A]` (modeling potential presence or absence of a value with `Some[+A]` and `None`) and `Either[+A, +B]` (providing structured success (`Right[+A]`) and failure (`Left[+B]`) containers). These types enforce compile-time contracts, dramatically improving upon manual error handling with `try-catch` blocks.

In the context of parsing, the state of the parser must account for consumed input characters and sophisticated error context. The original rudimentary `ParseError` structure, which held only a single `Location` and a `String` message, was insufficient. The system was upgraded to support a **stack of errors**: `case class ParseError(stack: List[(Location,String)])`. This critical enhancement allows the parser to record the context and history of failures, showing what the parser was attempting when it broke.

To manage this error context programmatically, a helper function `push` was introduced. Given a `Location` and a message, `push` creates a new `ParseError` instance, prepending the new error pair to the existing stack: `def push(loc: Location, msg: String): ParseError = copy(stack = (loc,msg) :: stack)`. The existing `Location` object retains the full input string and its current offset.

Furthermore, to support sequential parsing and maintain accurate state tracking, the parser type definition was fundamentally revised. Originally, a parser might only return a value `A`. Now, the result must track input consumption. The result is encapsulated in the `Result` sealed trait, which has two primary constructors: `Success[+A](get: A, charsConsumed: Int)` and `Failure(get: ParseError)`. The inclusion of `charsConsumed: Int` is paramount, as calling combinators must use this count to correctly advance the `Location` for subsequent parsing steps. The updated signature for a parser is thus: `type Parser[+A] = Location => Result[A]`. The `Location` object already provides the mechanisms for tracking the full input string and the current offset.

**II. Core Combinators and Ambiguity Management**

The library expands its set of primitives to include `flatMap` as a new combinator. `flatMap` is essential for context-sensitive parsing, as it enables the selection of a second parser whose definition depends on the successful result of the first parser. Implementing `flatMap` requires multiple state transitions: if the first parser succeeds with value `a` and consumes `n` characters, the input location must first be advanced using a helper function `advanceBy(n: Int)` (which increments the offset) before calling the second parser `g(a)`. The overall success result must then be calculated using `advanceSuccess(n: Int)`: `def advanceSuccess(n: Int): Result[A] = this match { case Success(a,m) => Success(a,n+m); case _ => this }`. If the first parser fails, `flatMap` propagates the failure immediately. The implementation is defined as: `def flatMap[A,B](f: Parser[A])(g: A => Parser[B]): Parser[B] = s => f(s) match { case Success(a,n) => g(a)(s.advanceBy(n)).addCommit(n != 0); case e@Failure(_,_) => e }`.

This structure also necessitates specialized handling for committed versus uncommitted failure states. The `Result` type is extended to accommodate this by including a `Boolean` flag in the `Failure` constructor: `case class Failure(get: ParseError, isCommitted: Boolean) extends Result[Nothing]`. This commitment status is managed by a helper function `uncommit`, which cancels commitment upon failure: `def uncommit: Result[A] = this match { case Failure(e,true) => Failure(e,false); case _ => this }`. The `attempt` combinator leverages this mechanism, defined as `def attempt[A](p: Parser[A]): Parser[A] = s => p(s).uncommit`, ensuring that failing to parse does not irreversibly commit the parser state.

Error handling in the context of alternatives (`or`) is similarly refined. If a parser `p` is constructed using `or(p1, p2)`, and both `p1` and `p2` fail on the same input, the programmer might need control over which error is reported. If the goal is to force commitment to `p1`'s error after partial success, the `isCommitted` flag is crucial. The general `or` combinator logic respects this: if `x` succeeds, the expression succeeds; if `x` fails in a committed state, the process fails immediately; otherwise, if `x` fails uncommitted, the second parser `y` is attempted. The core implementation for `or` is: `def or[A](x: Parser[A], y: => Parser[A]): Parser[A] = s => x(s) match { case Failure(e,false) => y(s); case r => r }`.

Additionally, a new developer-proposed combinator, `scope`, is used to manage error context without replacing labels. Defined as `def scope[A](msg: String)(p: Parser[A]): Parser[A]`, it ensures that if the parser `p` fails when running through `scope(msg)`, the resulting error stack (`e2`) has its head set to the message `msg`, while the tail (`e2.stack.tail`) retains the original error context (`e1`).

**III. Principles of Concurrency, Laziness, and Parsing (Advanced Topics)**

The foundational design principles applicable across parsing and other functional domains, such as stream processing and concurrency, are critical. In the realm of programming language design, functional concepts like laziness and non-strictness are crucial for memory efficiency. Processing collections using `Stream[+A]`—a specialized lazy list—avoids the costly allocation of numerous temporary, intermediate data structures, which happens with standard collections (e.g., processing `List(1, 2, 3, 4)` through `map(_ + 10).filter(_ % 2 == 0).map(_ * 3)`). Structurally, `Stream` is defined by `Empty` and `Cons[+A](h: () => A, t: () => Stream[A])`. The use of empty function literals (`() => A`) forces both the head and tail components to be non-strict (thunks), guaranteeing evaluation occurs only once upon first demand (memoization). Advanced stream operations like `map`, `filter`, and `flatMap` chain transformations efficiently. The most generalized builder is `unfold[A, S](z: S)(f: S => Option[(A, S)]): Stream[A]`, capable of generating infinite sequences (e.g., `constant[A](a: A)` or `from(n: Int)`).

Concurrency is managed via `Par[A]`, designed to represent non-blocking computations. This system moves beyond standard blocking mechanisms like `java.util.concurrent.Future`, which require a blocking `.get()` call. For instance, attempting `println(Par.equal(S)(a, fork(a)))` with a fixed-size thread pool of size one results in a deadlock because the internal `fork` implementation blocks waiting for a result, preventing the second computation from starting. The ideal solution mandates a fully non-blocking `Par` representation where `fork` and `map2` never call blocking methods. This is achieved using the Actor model, where the Actor processes incoming messages (e.g., `Left(a)` or `Right(b)`), stores state (using mutable variables like `ar: Option[A]` and `br: Option[B]` for `map2`), and executes the combining function `f` only when the second message arrives, ensuring sequential, non-blocking state management.

In both parsing and concurrency, the concept of algebraic coherence is maintained by defining complex operations from minimal, fundamental primitives. For parsing, `flatMap` and `join` relate fundamentally to `map` and `map2` combinators. In the broader design context, the entire framework relies on Property-Based Testing (PBT), where abstract properties (e.g., `ns.reverse.reverse == ns`) are verified using a generator (`Gen`), greatly increasing developer confidence.
