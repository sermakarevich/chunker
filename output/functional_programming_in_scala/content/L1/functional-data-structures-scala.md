# functional-data-structures-scala

**Parent:** [[content/L2/scala-functional-advanced-concepts|scala-functional-advanced-concepts]] — Functional programming in Scala relies on immutability and data sharing, utilizing the `sealed` trait `List[+A]` with constructors `Nil` and `Cons` and supporting covariant parameters (`+A`). Key advanced techniques include pattern matching, using HOFs like `foldRight` (for summation/product) and `foldLeft` (for stack safety), and composing `map`, `filter`, and `flatMap` for efficient data manipulation.

The principles of Functional Programming in Scala emphasize immutability and the use of pure functions, leading to the implementation of highly efficient and predictable data structures. Central to this approach is the concept of functional data structures, which are inherently immutable because they operate solely using pure functions, meaning they cannot modify data in place or cause side effects. This immutability is exemplified by basic types like integers (e.g., 3 or 4) and collections, such as the list. When concatenating two lists using `a ++ b`, the resulting list is newly created without altering the original inputs, $a$ and $b$.

### The Scala `List` Data Structure and Variance

The standard library’s `List` type provides a robust foundation for these structures. Defining `List` requires advanced Scala concepts, beginning with a `sealed trait List[+A]`. The use of `sealed` is crucial because it dictates that all implementations of the trait must reside within the same file. The structure uses two data constructors: the empty list, represented by the case object `Nil` (which extends `List[Nothing]`), and a non-empty list, represented by the case class `Cons[+A](head: A, tail: List[A])` (which extends `List[A]`). The `+A` annotation signifies that `A` is a covariant parameter, a feature that is essential for maintaining type safety and flexibility. This covariance allows the system to treat `Nil`—since `Nothing` is a subtype of all types—as instances of various types, such as `List[Int]` or `List[Double]`, thereby supporting the desired behavior for diverse list types.

Furthermore, the principle of variance dictates that for any types X and Y, if X is a subtype of Y, then `List[X]` is automatically considered a subtype of `List[Y]`. This mechanism is foundational to how Scala handles type relationships within collections.

### Utility and Pattern Matching

Developers utilize companion objects, which are conventionally placed alongside the data type they support, to house utility functions. The `List` companion object, for instance, provides functions like `def fill[A](n: Int, a: A): List[A]`, which constructs a list containing `n` copies of element `a`. Operating on lists requires pattern matching, a powerful mechanism for deconstructing data structures. The functions `sum` and `product`, housed in the `List` companion object, demonstrate this. The `sum` function, specifically accepting `List[Int]` arguments, is defined recursively: `def sum(ints: List[Int]): Int = ints match { case Nil => 0; case Cons(x,xs) => x + sum(xs) }`. This implementation states that the sum of an empty list (`Nil`) is 0, and the sum of a non-empty list is its head element (`x`) plus the sum of the rest of the list (`xs`). Similarly, the `product` function, accepting `List[Double]` arguments, is defined as: `def product(ds: List[Double]): Double = ds match { case Nil => 1.0; case Cons(0.0, _) => 0.0; case Cons(x,xs) => x * product(xs) }`. These definitions are intrinsically recursive, which is the standard pattern for writing functions that operate over recursive data types like `List`, given that `Cons` refers to `List[A]` in its definition of the `tail`.

Pattern matching functions evaluate by descending into the structure of the provided expression (the scrutinee). The process involves matching against patterns, which can include literals (e.g., `3`) or binding a subexpression. For instance, `List(1,2,3) match { case Cons(h,_) => h }` utilizes a data constructor pattern to bind the head element to `h`, resulting in 1. The underscore variable pattern (`_`) is useful for matching any expression or for ignoring multiple parts of the target structure. When matching a full list structure like `Cons(1, Cons(2, Cons(3, Nil)))`, the structure must be accurately represented in the match statement.
