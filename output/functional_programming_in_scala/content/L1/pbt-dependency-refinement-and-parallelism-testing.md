# pbt-dependency-refinement-and-parallelism-testing

**Parent:** [[content/L2/advanced-functional-algebra-design|advanced-functional-algebra-design]] — Functional design requires treating computation and API logic as a functional algebra, employing specialized ADTs like `Option` and `Either` for compile-time error handling, and using `Stream` for memory-efficient lazy data transformations. Concurrency is managed via `Par` using the non-blocking Actor model to avoid deadlocks, and testing is achieved through `Prop` and `Gen` in Property-Based Testing, which uses a structured `Result` trait for detailed pass/fail reporting.

The design and evolution of advanced property-based testing (PBT) libraries necessitate meticulous handling of computational dependencies, state, and parallelism, leading to continuous refinements in the core data types and combinators. Central to this system is the `Prop` data structure, which encapsulates a property check, fundamentally defined by a `run` function. Initially, `Prop` lacked sufficient information regarding the constraints of its execution. To ensure the system could reliably test properties across various scenarios, including those requiring multiple test cases and handling differing input sizes, the structure was refined. Specifically, the `Prop` case class was updated to `case class Prop(run: (MaxSize, TestCases, RNG) => Result)`. Here, `MaxSize` is defined as `type MaxSize = Int`, ensuring the property function `run` explicitly accepts the maximum size allowed, alongside the number of test cases (`TestCases`) and the random number generator (`RNG`). Consequently, the key method `forAll` was updated to match this signature, accepting a generator that inherently provides size information: `def forAll[A](g: Int => Gen[A])(f: A => Boolean): Prop = Prop { (max, n, rng) => ... }`. This revised implementation structures the generation process by iterating over sizes, ranging from 0 up to $\min(n, max) + 1$. For each size $i$, it generates a set of random cases and combines them into a single property (`prop`) using the logical $\&\&$ operator. The final execution of the property is triggered by calling `prop.run(max, n, rng)`.

To enhance developer usability, a helper method `run` was introduced on the `Prop` companion object. This function simplifies the testing workflow by accepting a `Prop` value and providing default arguments for all three dependencies: `maxSize` (defaulting to 100), `testCases` (defaulting to 100), and `rng` (defaulting to `RNG.Simple(System.currentTimeMillis)`). This `run` function internally calls `p.run(maxSize, testCases, rng)` and employs pattern matching on the resulting `Result` to provide clear, readable console output. It handles either the failure details (printing the `Falsified(msg, n)` structure) or an OK message for successful execution (`Passed`).

The library's ability to model abstract laws is crucial when dealing with complex operations like finding the maximum value of a list (`List.max`). When defining a property `maxProp` for the `max` function—which takes a list of randomly generated small integers using `val smallInt = Gen.choose(-10, 10)`—the initial attempt revealed a failure because the standard library's `max` function crashes when provided an empty list. This critical dependency forced a structural change: to guarantee a non-empty list for testing, the developer had to define a dedicated generator function, `listOf1`, and update the property definition for `max` to utilize this new generator, thereby ensuring the derived property generator correctly handles the non-empty constraint.

Beyond standard value properties, PBT is extended to verify concurrency and parallel computations, necessitating dedicated combinators.
