# functional-programming-advanced-combinators

**Parent:** [[content/L2/functional-paradigm-architecture|functional-paradigm-architecture]] — Functional programming abstracts computations into first-class values, using ADTs like Option and Either to handle non-fatal errors. Efficiency is achieved via non-strictness (laziness), implemented using `Stream[+A]` to fuse operations and handle infinite sequences, while parallelism is managed by the `Par[A]` type, which defers execution until a terminal `run` call and uses combinators like `map2` and `fork` to maintain purity and manage resource-intensive effects.

The design of functional programming languages heavily emphasizes treating computations as values, which fundamentally shifts how system design handles potential failures and parallel execution. A core principle is encapsulating effects, including error states, using specialized algebraic data types (ADTs) and higher-order functions (HOFs). This approach guarantees complete type safety and forces developers to explicitly handle all possible outcomes, thereby preventing the common pitfalls of silent failures associated with traditional sentinel values or simple runtime checks. The general best practice is representing exceptions as ordinary, first-class values.

To manage non-fatal control flow, two primary specialized types are paramount: `Option[+A]` and `Either[+A, +B]`. The `Option` type models the mere presence or absence of a value through two distinct cases: `case class Some[+A](get: A)` for a defined value, and `case object None extends Option[Nothing]` for an undefined state. This design is crucial because it allows functions that might not always yield a result—such as calculating the mean of a sequence—to be designated *total functions*. For instance, if a mean calculation function uses `Option[Double]`, it will explicitly return `None` when the input sequence is empty, making the potential failure state an inherent, static part of the function's contract. Complementing this, the `Either` type provides a structured mechanism to carry one of two specific outcomes: success, conventionally represented by `Right[+A]`, or a failure/error value, often represented by `Left[+B]`. By leveraging these HOFs, complex error propagation patterns can be abstracted away, leading to significant minimization of boilerplate code while guaranteeing robust, compile-time checked error handling. These concepts of handling non-fatal control flow—whether failure or success—are universal and generalize beyond simple option and left/right structures.

Beyond error handling, functional programming addresses efficiency concerns related to data transformations, particularly when chaining multiple operations like `map`, `filter`, and subsequent `map` calls on standard data structures such as `List`. Executing such a sequence traditionally forces multiple passes over the initial data and, critically, generates numerous temporary, intermediate data structures. For example, processing the sequence `List(1, 2, 3, 4)` through `map(_ + 10).filter(_ % 2 == 0).map(_ * 3)` first yields `List(11, 12, 13, 14)`, then creates a second list, `List(12, 14)`, and finally a third, `List(36, 42)`. This massive overhead of memory allocation and computational burden due to multiple passes is highly undesirable. The architectural solution is the implementation of automatic loop fusion, achieved through the employment of *non-strictness*, or laziness.

Non-strictness is a core functional property, meaning a function is permissible to not evaluate all its arguments. Conversely, a strict function must always evaluate every argument provided. While Scala treats function definitions as strict by default, the contrast is observable in short-circuiting Boolean operators like `&&` and `||`, which only evaluate arguments as required, and within the standard `if` control construct, which does not necessarily evaluate both the true and false branches. To enforce controlled non-strictness, Scala introduced `if2[A]`: `def if2[A](cond: Boolean, onTrue: () => A, onFalse: () => A): A = if (cond) onTrue() else onFalse()`. Using the empty function literal `() => A` (or its alias `=> A`) ensures the provided block only runs if the conditional check mandates it, as it wraps the expression in a 'thunk' (an unevaluated form of an expression). This concept finds its most robust and explicit implementation in `Stream[+A]`, a specialized trait designed to model lazy lists. A stream is constructed using two constructors: `Empty extends Stream[Nothing]` for the empty stream, and `Cons[+A](h: () => A, t: () => Stream[A])` for a non-empty one. The definition utilizing `() => A` and `() => Stream[A]` is structurally critical because it mandates that both the head (`h`) and tail (`t`) components must be non-strict (thunks). This guarantees that the value of the head and the computation of the tail are evaluated only once upon the first demand, a process known as memoization. The `Stream` object provides the smart constructor `cons[A](hd: => A, tl: => Stream[A]): Stream[A]`, which manages this memoization.

Accessing stream components requires explicit handling of thunks. For instance, `headOption: Option[A]` returns `None` if the stream is `Empty`, or `Some(h())` if it is `Cons(h, t)`, requiring explicit calling of `h()` to force the head thunk while crucially *not* forcing the tail. Utilities such as `take(n)` (returns the first $n$ elements), `drop(n)` (skips the first $n$ elements), and `takeWhile(p: A => Boolean)` (returns starting elements satisfying predicate $p$) are readily available. Advanced stream operations benefit immensely from laziness. Core combinators—including `map`, `filter`, `append`, and `flatMap`—chain transformations without instantiating intermediate results. For example, `Stream(1,2,3,4).map(_ + 10).filter(_ % 2 == 0).toList` demonstrates that the computation generates elements sequentially, conserving memory by allowing the garbage collector to reclaim space for values (such as 11 and 13) as soon as they are no longer needed. This efficiency is vital when dealing with infinite streams. An infinite stream of ones, defined as `val ones: Stream[Int] = Stream.cons(1, ones)`, can be safely processed: `ones.take(5).toList` yields `List(1, 1, 1, 1, 1)`, and `ones.exists(_ % 2 != 0)` returns `true` immediately upon finding the first matching element, as it terminates early. However, care must be taken; operations like `ones.forAll(_ == 1)` will lead to an infinite loop (stack overflow) because the predicate never finds a non-matching element necessary for termination.

Developers can construct specialized streams using advanced methods. A `constant[A](a: A)` function generates an infinite stream of $a$. A `from(n: Int)` function generates an infinite sequence of integers starting from $n$. Furthermore, `def fibs` generates the infinite Fibonacci sequence (0, 1, 1, 2, 3, 5, 8, etc.). A powerful, general, corecursive function is `def unfold[A, S](z: S)(f: S => Option[(A, S)]): Stream[A]`. `unfold` takes an initial state $z$ and a function $f$ that determines both the next state and the next value, terminating when $f$ returns `None`. The utility of `unfold` is immense, allowing for the definition of core builders like `def fibs`, `def from`, and `def constant` while maintaining constant memory usage, which surpasses simple recursive definitions. Moreover, `unfold` can implement other complex combinators, such as `map`, `take`, `takeWhile`, and `zipWith[B](s2: Stream[B]): Stream[(A, B)]` (following the Chapter 3 `zipWith` pattern). A related function, `zipAll[B](s2: Stream[B]): Stream[(Option[A], Option[B])]`, continues traversing as long as either input stream has elements, using `Option` to track stream exhaustion.

Advanced stream structures also include `def startsWith[A](s: Stream[A]): Boolean` (checking if stream $s$ is a prefix of another), which can, in turn, enable the definition of `def hasSubsequence[A](s: Stream[A]): Boolean` by checking if any tail of the stream starts with $s$: `tails exists (_ startsWith s)`. An even more potent structural method is `def tails: Stream[Stream[A]]`, which returns a stream of all suffixes of the input sequence. For instance, given `Stream(1,2,3)`, `tails` generates the sequence `Stream(Stream(1,2,3), Stream(2,3), Stream(3), Stream())`.

Focusing on parallel computation, the discussion addresses which combinators should be considered truly primitive abstractions. The complexity of defining `parMap` as a new primitive combinator is highlighted, particularly concerning adherence to timeout requirements, making the reuse of established primitives advantageous. The function signature for `parMap` is `def parMap[A,B](ps: List[A])(f: A => B): Par[List[B]]`. Implementing this requires a two-step process. First, developers utilize `asyncF(f)` to map the input list `ps` to a list of parallel computations (`List[Par[B]]`). The `asyncF` combinator is crucial because it converts a standard function `A => B` into a function with the signature `A => Par[B]`, enabling the easy generation of $N$ parallel computations. Second, the resultant `List[Par[B]]` must be converted into the `Par[List[B]]` type required by the return signature. This necessity mandates the definition of a function called `sequence`, which has the signature `def sequence[A](ps: List[Par[A]]): Par[List[A]]`. Using these established combinators, the concrete implementation for `parMap` is: `def parMap[A,B](ps: List[A])(f: A => B): Par[List[B]] = fork { val fbs: List[Par[B]] = ps.map(asyncF(f)); sequence(fbs) }`. It is important to note that when this implementation executes, `parMap` returns immediately, even if the input list `ps` is very large. However, the actual asynchronous computation only runs when the user calls `run`, at which point the system spawns $N$ parallel computations and waits for all of them to complete, collecting the final results into a list.

Following `parMap`, the next task is the implementation of `parFilter`, which filters elements of a list in parallel, having the signature `def parFilter[A](as: List[A])(f: A => Boolean): Par[List[A]]`. Reusing existing combinators for this function is advantageous as it prevents the duplication of complex logic, specifically regarding timeout handling. Finally, the text encourages experimentation with writing other useful parallel computations, suggesting three specific advanced tasks: 1) developing a more general version of the parallel summation function (previously discussed) and applying it to efficiently find the maximum value of an `IndexedSeq` in parallel; 2) creating a function that accepts a list of paragraphs (`List[String]`) and calculates the total number of words across all paragraphs in parallel, while generalizing this function as much as possible; and 3) implementing `map3`, `map4`, and `map5` using the existing `map2` combinator.
