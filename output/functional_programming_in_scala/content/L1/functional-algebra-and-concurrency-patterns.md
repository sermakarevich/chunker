# functional-algebra-and-concurrency-patterns

**Parent:** [[content/L2/functional-algebra-design-patterns|functional-algebra-design-patterns]] — Functional programming mandates treating computation as a first-class value and utilizing ADTs like `Option` and `Either` to handle non-fatal control flow and errors explicitly. For performance, specialized `Stream` types enforce non-strictness (laziness), avoiding the creation of intermediate data structures when chaining transformations like `map` and `filter` on collections such as `List(1, 2, 3, 4)`. Concurrency is managed via `Par[A]`, which requires sophisticated, non-blocking primitives—such as using the Actor model for `map2`—to prevent deadlocks, ensuring that foundational laws like `fork(x) == x` hold even when executing 100,000 iterations with a fixed-size thread pool.

Functional programming paradigms emphasize the treatment of computation itself as a first-class value, fundamentally reshaping how systems handle effects and potential failures. Instead of relying on traditional mechanisms like mandatory control flow paths for exception handling or unreliable runtime checks, best practices mandate encapsulating all potential effects—including error states—using specialized Algebraic Data Types (ADTs) and Higher-Order Functions (HOFs). This approach guarantees complete type safety, forcing developers to explicitly address all possible outcomes, thereby eliminating the silent failures often associated with sentinel values. The core principle involves representing exceptions as ordinary, first-class values.

To manage non-fatal control flow robustly, two types are paramount: `Option[+A]` and `Either[+A, +B]`. The `Option` type models the presence or absence of a value through two states: `case class Some[+A](get: A)` for a defined value, and `case object None extends Option[Nothing]` for an undefined state. This design is essential for defining *total functions*; for instance, a function calculating the mean of a sequence must return `None` when the input is empty. This elevates the potential failure state to a static, compile-time contract, representing a substantial improvement over the use of boilerplate `try-catch` blocks. Complementing this, the `Either` type provides a structured mechanism to carry one of two specific outcomes: success, conventionally represented by `Right[+A]`, or a failure/error value, typically represented by `Left[+B]`. By utilizing these HOFs, complex error propagation patterns are abstracted, guaranteeing reliable, compile-time checked error handling.

Efficiency concerns in functional programming often arise when chaining multiple data transformations (e.g., applying sequential `map`, `filter`, and subsequent `map` calls) on structures like standard Scala `List`. Traditional execution requires multiple passes over the initial data, generating numerous temporary, intermediate data structures. For example, processing `List(1, 2, 3, 4)` through the sequence `map(_ + 10).filter(_ % 2 == 0).map(_ * 3)` involves creating three intermediate lists: `List(11, 12, 13, 14)`, then `List(12, 14)`, and finally `List(36, 42)`. This excessive memory allocation and computational burden is highly undesirable.

The solution is the implementation of automatic loop fusion, achieved through non-strictness, or laziness. Non-strictness dictates that a function is allowed to not evaluate all its arguments, contrasting with strict functions. While Scala generally treats function definitions as strict, the concept is best demonstrated by short-circuiting Boolean operators like `&&` and `||`, and the `if` control construct, which does not necessarily evaluate both true and false branches. To enforce controlled non-strictness, Scala introduced `if2[A]`: `def if2[A](cond: Boolean, onTrue: () => A, onFalse: () => A): A = if (cond) onTrue() else onFalse()`. The use of the empty function literal `() => A` ensures that the provided block only runs if the condition mandates it. A more convenient alias uses the arrow `=>` (e.g., `def if2[A](cond: Boolean, onTrue: => A, onFalse: => A): A = if (cond) onTrue else onFalse`), which automatically wraps the expression in a 'thunk' (unevaluated form) that requires explicit forcing.

This concept is most robustly implemented in `Stream[+A]`, a specialized trait designed to model lazy lists. A stream has two constructors: `Empty extends Stream[Nothing]` for the empty stream, and `Cons[+A](h: () => A, t: () => Stream[A])` for a non-empty one. Structurally, the use of `() => A` and `() => Stream[A]` is critical, forcing both the head (`h`) and tail (`t`) components to be non-strict (thunks). This guarantees that both the head value and the tail computation are evaluated only once upon the first demand, a mechanism known as memoization. The `Stream` object provides the smart constructor `cons[A](hd: => A, tl: => Stream[A]): Stream[A]`. The `Stream.apply[A](as: A*)` constructor builds streams from multiple elements, recursively using `cons` and automatically wrapping both the initial head and subsequent tail components in thunks. 

Accessing stream components requires explicit thunk handling; `headOption: Option[A]` returns `None` if the stream is `Empty`, or `Some(h())` if it is `Cons(h, t)`, explicitly calling `h()` while crucially *not* forcing the tail. Utility methods include `take(n)` (first $n$ elements), `drop(n)` (skips first $n$ elements), and `takeWhile(p: A => Boolean)` (returns starting elements satisfying predicate $p$).

Advanced stream operations benefit immensely from laziness. Core combinators like `map`, `filter`, `append`, and `flatMap` chain transformations without creating intermediate results. `Stream(1,2,3,4).map(_ + 10).filter(_ % 2 == 0).toList` demonstrates that the computation generates elements sequentially, saving memory by allowing the garbage collector to reclaim space for values (like 11 and 13) as soon as they are no longer needed. This efficiency is vital for infinite streams; `val ones: Stream[Int] = Stream.cons(1, ones)` can be safely processed, with `ones.take(5).toList` yielding `List(1, 1, 1, 1, 1)` and `ones.exists(_ % 2 != 0)` returning `true` immediately, as it terminates upon finding the first satisfying element. Caution must be exercised, however, as `ones.forAll(_ == 1)` will cause an infinite loop (stack overflow) because the predicate lacks a non-matching element for termination.

Developers can generate specialized streams using advanced methods. `def constant[A](a: A): Stream[A]` generates an infinite stream of $a$, and `def from(n: Int): Stream[Int]` generates an infinite sequence starting from $n$. The corecursive function `def unfold[A, S](z: S)(f: S => Option[(A, S)]): Stream[A]` is powerful, taking an initial state $z$ and a function $f$ determining the next state and value, terminating when $f$ returns `None`. `unfold` can define `constant`, `from`, and `ones` while maintaining constant memory usage. Furthermore, `unfold` can implement `map`, `take`, `takeWhile`, and `zipWith[B](s2: Stream[B]): Stream[(A, B)]`. A paired function, `zipAll[B](s2: Stream[B]): Stream[(Option[A], Option[B])]`, traverses as long as either stream has elements, using `Option` to track exhaustion. Structural depth is provided by `def startsWith[A](s: Stream[A]): Boolean` (checking if $s$ is a prefix of $s'$), and the highly useful `def tails: Stream[Stream[A]]` (returning a stream of all suffixes; e.g., for `Stream(1,2,3)`, this generates `Stream(Stream(1,2,3), Stream(2,3), Stream(3), Stream())`).

Regarding concurrent API design, the API must be viewed as an algebra governed by specific formal laws or axioms. Two key examples are: 1) `map(unit(1))(_ + 1) == unit(2)`, showing mapping a unit value of 1 by `_ + 1` results in `unit(2)`. 2) `map(y)(id) == y`, confirming that mapping a value `y` using the identity function retains the original value. Defining these laws is vital because hidden side effects compromise compositionality, making API components untreatable as black boxes. For example, if the law stating that `fork(x)` is equivalent to `x` for all expressions `x` and all `ExecutorService` choices fails, combinators like `parMap` lose soundness, potentially leading to deadlocks.

A subtle but critical bug arises in most `fork` implementations when using an `ExecutorService` backed by a fixed-size thread pool, specifically those from `Executors.newFixedThreadPool`. If the pool size is limited to one, the sequence of actions—defining `val a = lazyUnit(42 + 1)`, initializing `val S = Executors.newFixedThreadPool(1)`, and calling `println(Par.equal(S)(a, fork(a)))`—will likely deadlock. The deadlock mechanism is derived from the implementation: `def fork[A](a: => Par[A]): Par[A] = es => es.submit(new Callable[A] { def call = a(es).get })`. This structure causes the outer `Callable` to occupy the single available thread. While running in that thread, the code then submits and blocks waiting for the result of the second `Callable` via `a(es).get`. Since the thread pool is exhausted, no threads are available for the second task, causing the deadlock. Developers must re-examine the law `fork(x) == x` to assess the implementation's robustness, perhaps stipulating the need for unbounded thread pools.

Alternative strategies for `fork` include: 1) Using `def fork[A](a: => Par[A]): Par[A] = es => fa(es)`. This variant avoids deadlock but fails to genuinely fork a separate logical thread for evaluating `fa`, meaning calling `fork(hugeComputation)(es)` executes `hugeComputation` in the main thread. It is useful, however, because it delays computation until needed and is named `delay`. 2) To genuinely execute arbitrary computations over fixed-size thread pools, a fully non-blocking representation of `Par` is mandatory. The core issue is that a value cannot be retrieved from a `Future` without the current thread blocking on `Future.get`. Therefore, a `Par` representation that avoids resource leaks and blocking must ensure that `fork` and `map2` never invoke methods that block the current thread, such as `Future.get`.

Complementary to this concurrency model, the functional API also introduces concepts like the `unit` combinator (`def unit[A](a: A): Par[A] = es => new Future[A] { def apply(cb: A => Unit): Unit = cb(a) }`), which is simple and avoids explicit `ExecutorService` usage. For true parallelism, the `fork` combinator utilizes `def eval(es: ExecutorService)(r: => Unit): Unit = es.submit(new Callable[Unit] { def call = r })` to submit a task to the `ExecutorService` and asynchronously evaluate the by-name argument `a` when the returned `Future` is ready to execute the continuation `cb`. 

Implementing a non-blocking `map2` requires sophisticated concurrency management, as it must ensure both `Par[A]` and `Par[B]` run in parallel and invoke the combining function `f` only upon the arrival of both results, preventing race conditions. The recommended solution utilizes a non-blocking concurrency primitive: the **Actor**. An Actor operates as a concurrent process that manages messages sequentially and only consumes a thread when a message is received. An example is an `echoer` process. The specific implementation of `map2` uses the Actor to manage state, maintaining two mutable variables, `ar: Option[A]` and `br: Option[B]`. The Actor processes the first message (wrapped as `Left(a)` or `Right(b)`), stores it, and waits for the second result. Upon receiving the second result, the Actor executes the combining function `f` with both stored results and propagates the final `C` result to the original continuation `cb` using `eval(es)`. This non-blocking design successfully allows `parMap` (e.g., running 100,000 iterations of `math.sqrt(_)` using `List.range(1, 100000)`) to operate efficiently, collecting results via `run(Executors.newFixedThreadPool(2))(p)` without requiring 100,000 JVM threads, thus demonstrating that the law of forking holds even for fixed-size thread pools.
