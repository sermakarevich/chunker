# functional-programming-principles

**Parent:** [[content/L2/functional-programming-design-patterns|functional-programming-design-patterns]] — Functional programming achieves robust design by managing side effects explicitly through structures like `Option[A]` and `Either[A, B]` for type-safe control flow, and by using `Stream[A]` and the `unfold` function for memory-efficient lazy computation. State changes are controlled via the `Rand[A]` type alias and combinators like `map` and `flatMap`, which guarantee referential transparency by explicitly passing the updated state through all computations.

The foundation of robust, modern software design, particularly within the functional programming paradigm, rests on several key concepts: managing computation effects, ensuring data immutability, and optimizing algorithmic performance through laziness. Fundamentally, functional programming treats computation itself as a value, advocating for the encapsulation of potential side effects and error states. Instead of relying on mandatory control flow paths for exceptions, best practices utilize specialized algebraic data types (ADTs) and higher-order functions (HOFs) to maintain complete type safety. This approach forces the developer to explicitly account for all possible outcomes, thereby eliminating the risk of silent failures associated with traditional sentinel values or simple runtime checks. 

To manage non-fatal control flow, two specialized types are essential: `Option[+A]` and `Either[+A, +B]`. The `Option` type models the mere presence or absence of a value, possessing two distinct states: `case class Some[+A](get: A)` for a defined value, and `case object None extends Option[Nothing]` for an undefined state. This design allows functions that might not always yield a result, such as calculating the mean of an empty sequence, to be correctly designated as *total functions*. For example, if a mean function uses `Option[Double]`, it statically returns `None` for an empty input, making the potential failure an inherent, compile-time contract. Complementing this, the `Either` type provides a structured mechanism to carry one of two specific outcomes: success, typically via `Right[+A]`, or a failure/error value, often represented by `Left[+B]`. Leveraging these HOFs permits abstracting complex error propagation patterns, minimizing boilerplate code while guaranteeing robust, compile-time checked error handling. 

Beyond handling control flow, efficiency concerns in functional programming often necessitate addressing performance bottlenecks related to data transformations. A common issue arises when chaining multiple data transformations—like sequential `map`, `filter`, and subsequent `map` calls—on structures such as standard Scala `List`. Such a traditional sequence requires multiple passes over the initial data and, critically, generates numerous temporary, intermediate data structures. For instance, processing `List(1, 2, 3, 4)` through `map(_ + 10).filter(_ % 2 == 0).map(_ * 3)` first creates `List(11, 12, 13, 14)`, then a second list, `List(12, 14)`, and finally a third, `List(36, 42)`. This excessive overhead of memory allocation and computation due to multiple passes is highly undesirable. The solution is the implementation of automatic loop fusion, achieved through the employment of *non-strictness*, or laziness. 

Non-strictness is a core functional property indicating that a function is permitted to not evaluate all its arguments. While a strict function evaluates every argument provided (e.g., `square(41.0 + 1.0)` yields `square` receiving the fully evaluated value of `42.0`), the contrast is clearest in short-circuiting Boolean operators like `&&` and `||`, or the standard `if` construct. To enforce controlled non-strictness, Scala introduced `if2[A]`: `def if2[A](cond: Boolean, onTrue: () => A, onFalse: () => A): A = if (cond) onTrue() else onFalse()`. The use of the empty function literal `() => A` ensures the provided block only executes if the conditional check dictates it. A more convenient alias uses the arrow `=>` (e.g., `def if2[A](cond: Boolean, onTrue: => A, onFalse: => A): A = if (cond) onTrue else onFalse`), automatically wrapping the expression in a 'thunk' (an unevaluated form of an expression) that must be explicitly forced for evaluation. 

This concept finds its most explicit implementation in `Stream[+A]`, a specialized trait modeling lazy lists. A stream has two constructors: `Empty extends Stream[Nothing]` for the empty stream, and `Cons[+A](h: () => A, t: () => Stream[A])` for a non-empty one. The structural definition using `() => A` and `() => Stream[A]` is critically important, as it forces both the head (`h`) and tail (`t`) components to be non-strict (thunks). This guarantees that the head value and the computation of the tail are evaluated only once upon the first demand, a process called memoization. The `Stream` object provides the smart constructor `cons[A](hd: => A, tl: => Stream[A]): Stream[A]`, managing this memoization. The `Stream.apply[A](as: A*)` constructor builds streams from elements `as`; if `as` is empty, it returns `Empty`; otherwise, it recursively uses `cons`, automatically wrapping both the initial head and subsequent tail components in thunks, thus preserving laziness. 

Accessing stream components requires explicit thunk handling. For example, `headOption: Option[A]` returns `None` if the stream is `Empty`, or `Some(h())` if it is `Cons(h, t)`, explicitly forcing the head thunk (`h()`) but crucially *not* forcing the tail. Utilities available include `take(n)` (returns the first $n$ elements), `drop(n)` (skips the first $n$ elements), and `takeWhile(p: A => Boolean)` (returns starting elements satisfying predicate $p$). 

Advanced stream operations benefit immensely from laziness. Core combinators like `map`, `filter`, `append`, and `flatMap` chain transformations without creating intermediate results. For instance, `Stream(1,2,3,4).map(_ + 10).filter(_ % 2 == 0).toList` demonstrates that computation generates elements one by one, saving memory by allowing the garbage collector to reclaim space for intermediate values (like 11 and 13) as soon as they are no longer needed. This efficiency is vital when dealing with infinite streams. An infinite stream of ones, `val ones: Stream[Int] = Stream.cons(1, ones)`, can be safely processed: `ones.take(5).toList` yields `List(1, 1, 1, 1, 1)`, and `ones.exists(_ % 2 != 0)` returns `true` immediately upon finding the first satisfying element. Conversely, operations like `ones.forAll(_ == 1)` will cause an infinite loop (stack overflow) because the predicate never finds a non-matching element to allow termination. 

Developers can generate specialized streams using methods like `constant[A](a: A)` (an infinite stream of $a$), `from(n: Int)` (an infinite sequence of integers starting from $n$), and `fibs` (the infinite Fibonacci sequence: 0, 1, 1, 2, 3, 5, 8, etc.). A general, corecursive function is `unfold[A, S](z: S)(f: S => Option[(A, S)]): Stream[A]`. `unfold` accepts an initial state $z$ and a function $f$ determining both the next state and the next value, terminating when $f$ returns `None`. The utility of `unfold` is its capacity to redefine core builders such as `fibs`, `from`, `constant`, and `ones`, while maintaining constant memory usage. Furthermore, `unfold` can implement other combinators, including `map` (EXERCISE 5.13), `take` (EXERCISE 5.13), `takeWhile` (EXERCISE 5.13), and `zipWith[B](s2: Stream[B]): Stream[(A, B)]` (EXERCISE 5.13), which follows the Chapter 3 `zipWith` pattern. A paired function, `zipAll[B](s2: Stream[B]): Stream[(Option[A], Option[B])]` (EXERCISE 5.13), continues traversal as long as either input stream possesses elements, utilizing `Option` to track stream exhaustion. 

Advanced combinators add structural depth. `def startsWith[A](s: Stream[A]): Boolean` (EXERCISE 5.14) checks if stream $s$ is a prefix of another (e.g., `Stream(1,2,3)` starts with `Stream(1,2)` returns `true`). This leads to `def hasSubsequence[A](s: Stream[A]): Boolean` (EXERCISE 5.14), implemented by checking if any tail of the stream starts with $s$: `tails exists (_ startsWith s)`. An even more powerful method is `def tails: Stream[Stream[A]]` (EXERCISE 5.15`), which returns a stream of all suffixes of the input sequence. For `Stream(1,2,3)`, `tails` generates `Stream(Stream(1,2,3), Stream(2,3), Stream(3), Stream())`. 

Simultaneously, the principles of managing state and immutability must be upheld even when dealing with concepts like randomness. Since data structures in functional programming are fundamentally immutable and operations must use pure functions, data cannot be changed in place, and side effects are forbidden. For example, concatenating two lists using `a ++ b` produces a new list without altering the original structures $a$ and $b$. The core data type is the singly linked list, `List`, defined by a sealed trait `List[+A]` and its constructors: `Nil` (the empty list, extending `List[Nothing]`) and `Cons[+A](head: A, tail: List[A])` (a non-empty list). While the principles of data sharing make structures **persistent** (adding or removing elements reuses the original list structure), efficiency limitations exist: the `init` function, which returns all but the last element, cannot be implemented in constant time like `tail` because replacing the tail requires copying all preceding `Cons` objects. Developers can generalize list operations using `foldRight`: `def foldRight[A, B](as: List[A], z: B)(f: (A, B) => B): B`. This processes a list by pattern matching, using `z` as the initial value and `f` to combine the element (`A`) with the accumulated result (`B`), as shown when calculating `sum2` via `foldRight(ns, 0)((x, y) => x + y)`. 

When state changes are necessary, they must be explicitly managed to restore **referential transparency**. In standard libraries, classes like `scala.util.Random` use an imperative API and rely on side effects (e.g., `rng.nextDouble`) on an internal state (`rng`), making the sequence of generated values non-reproducible and difficult to test. The simple solution of passing the `rng: scala.util.Random` object as an explicit argument still contains a flaw, as the side effects remain hidden. The functional remedy involves modifying the API to return both the newly generated value *and* the new state. For instance, a robust interface should be defined as `trait RNG { def nextInt: (Int, RNG) }`. This method must generate an `Int` and return both the random number and a new `RNG` object. This pattern ensures that the old state remains unmodified, cleanly separating the computation of the next state from its communication. The provided `SimpleRNG` class implements this using a linear congruential generator algorithm: `SimpleRNG(seed: Long)` extends `RNG` and defines `nextInt: (Int, RNG) = { val newSeed = (seed * 0x5DEECE66DL + 0xBL) & 0xFFFFFFFFFFFFL; val nextRNG = SimpleRNG(newSeed); val n = (newSeed >>> 16).toInt }`. When called, `SimpleRNG(42).nextInt` yields the pair `(n1: Int = 16159453, rng2: RNG = SimpleRNG(1059025964525))`. A subsequent call using `rng2` yields `(n2: Int = -1281479697, rng3: RNG = SimpleRNG(197491923327988))`. Crucially, since the result depends only on the input state and not external factors, this API is pure. While this pure state computation incurs efficiency loss compared to mutation, that loss can be mitigated by using efficient purely functional data structures, and sometimes mutation can be safely performed without compromising referential transparency.
