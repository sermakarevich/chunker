# functional-programming-purity-side-effects

**Parent:** [[content/L2/functional-programming-principles-in-scala|functional-programming-principles-in-scala]] — The book *Functional Programming in Scala* systematically teaches FP principles using Scala as a vehicle, demonstrating how to refactor impure code (like a coffee shop charge system) into pure functional forms that achieve superior testability and reusability by separating core business logic from external effects.

The conceptual foundation of functional programming (FP) rests upon the principle of purity, which dictates that a function's output is determined solely by its inputs, regardless of any external or internal state changes. A pure function, in the context of a language like Scala, is one that has no observable side effects. Formally, a function `f` with input type `A` and output type `B` (`A => B`) must relate every value `a` of type `A` to exactly one value `b` of type `B`. This means that computing `b` depends only on `a`. Crucially, any internal or external state change is irrelevant to computing the result `f(a)`; the function's only observable effect is the computation of its result.

This strict definition of purity is formalized and enabled by the concept of Referential Transparency (RT). An expression $e$ is referentially transparent if, for all programs $p$, every occurrence of $e$ within $p$ can be replaced by the result of evaluating $e$ without altering the meaning of $p$. Similarly, a function $f$ is pure if the expression $f(x)$ is referentially transparent for all referentially transparent values $x$. RT enables the powerful substitution model, which treats program evaluation like solving an algebraic equation: computation proceeds by fully expanding every part of an expression, replacing all variables with their referents, and then reducing the expression to its simplest form. Because expressions are referentially transparent, computation maintains an equational reasoning about programs, simplifying modularity and composability.

However, real-world programs must interact with the external world, necessitating side effects (like communicating a `Charge` to an external system), and other useful programs may involve state mutation. The text proposes that many programs appearing to necessitate side effects actually possess a functional analogue, or that effects can be structured so they are not observable. For instance, data can be locally mutated if the program ensures that the data cannot be referenced outside the function's scope, or file writing can occur provided no enclosing function observes this occurrence.

### The Cost of Impurity: A Purchase Example

To illustrate the conflict between necessity and purity, consider the initial, impure implementation of a `buyCoffee` method within a `Cafe` class. This method, typed as `def buyCoffee(cc: CreditCard): Coffee = { val cup = new Coffee(); cc.charge(cup.price); cup }`, directly charges the credit card (`cc.charge(cup.price)`). This interaction—which requires contacting a credit card company, authorizing the transaction, and persisting a record—constitutes an observable side effect. Because of this, testing the function is problematic, as the test must not involve actual charges. Furthermore, the design limits reusability: if Alice orders 12 cups of coffee, calling the impure `buyCoffee` 12 times charges the payment system 12 separate times, generating extra processing fees.

### The Pure Refactoring Strategy

Functional improvements involve separating the core logic from the external effects. The revised, functional approach demonstrates this separation by making `buyCoffee` return a `(Coffee, Charge)` pair, where `Charge` is a dedicated data type containing both the `CreditCard` and the total amount. Instead of performing the side effect, the function merely constructs the charge value: `(cup, Charge(cc, cup.price))`. The payment processing system is kept ignorant of this construction, allowing the pure function to generate the charge value without running any external commands. This fundamental shift significantly enhances both testability and reusability.

The `Charge` data type allows for the assembly of complex business logic. For example, a specialized `combine` function handles merging multiple charges for the same credit card (if `cc == other.cc`) or throwing an exception if the cards differ. To calculate the total charge for $n$ cups of coffee, the new function `buyCoffees(cc: CreditCard, n: Int): (List[Coffee], Charge)` utilizes higher-order functions like `List.fill(n)(buyCoffee(cc))`, `unzip`, and `charges.reduce((c1,c2) => c1.combine(c2))`. The resulting function body is: `def buyCoffees(cc: CreditCard, n: Int): (List[Coffee], Charge) = { val purchases: List[(Coffee, Charge)] = List.fill(n)(buyCoffee(cc)) val (coffees, charges) = purchases.unzip (coffees, charges.reduce((c1,c2) => c1.combine(c2))) }`. This entire solution is pure and trivial to test because the `Cafe` class remains completely isolated from how the resulting `Charge` values will eventually be processed. 

This purity allows for the creation of the `coalesce` function, which groups a list of individual charges by the credit card (`charges.groupBy(_.cc).values.map(_.reduce(_ combine _)).toList`) and combines them into a single total charge. This highlights the profound composability of FP, enabling the function to be reusable without needing mock objects or interfaces.
