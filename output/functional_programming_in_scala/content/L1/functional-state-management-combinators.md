# functional-state-management-combinators

**Parent:** [[content/L2/functional-programming-design-patterns|functional-programming-design-patterns]] — Functional programming achieves robust design by managing side effects explicitly through structures like `Option[A]` and `Either[A, B]` for type-safe control flow, and by using `Stream[A]` and the `unfold` function for memory-efficient lazy computation. State changes are controlled via the `Rand[A]` type alias and combinators like `map` and `flatMap`, which guarantee referential transparency by explicitly passing the updated state through all computations.

The philosophy of purely functional programming dictates that managing state transitions must be done explicitly and transparently. Rather than relying on mechanisms that allow functions to silently mutate internal state (e.g., using private fields within a class like `Foo` containing `FooState s`), functional best practices require a design where every method explicitly accepts the current state and returns both the resulting value and the computed next state. For example, a trait defining such an action would require methods like `def bar: (Bar, Foo)` and `def baz: (Int, Foo)`. This disciplined approach forces the calling program to take full responsibility for passing the newly computed next state (`Foo`) through the remainder of the execution flow.

This pattern is most visibly demonstrated in random number generation (RNG). When using a standard RNG, the result of a computation depends entirely on the initial seed or state. To ensure functional purity and reproducibility, any function that generates multiple random values, such as generating two distinct random integers, must chain the state transitions. The function `randomPair` illustrates this necessity: `def randomPair(rng: RNG): ((Int, Int), RNG) = { val (i1, rng2) = rng.nextInt; val (i2, rng3) = rng2.nextInt; ((i1, i2), rng3) }`. Critically, the state returned by the first call (`rng2`) must be used as the input state for the second call, and the final state (`rng3`) must be returned to the caller. This rigorous state passing mechanism ensures that the entire operation is deterministic, yielding the final state along with the results.

To simplify this repetitive, manual state passing, functional programmers introduce specialized abstractions. The central tool is the type alias `type Rand[+A] = RNG => (A, RNG)`, which represents a state action. A value of type `Rand[A]` is conceptually a program that accepts an initial `RNG` state, uses that state to generate a value of type `A`, and concurrently yields the updated `RNG` state. Given an existing method, such as `RNG`'s `nextInt`, it can be wrapped into this type, creating `val int: Rand[Int] = _.nextInt`. 

Combinators are developed to handle the composition of these state actions, abstracting away the underlying state mechanism. The fundamental action is the **unit action**, defined as `def unit[A](a: A): Rand[A] = rng => (a, rng)`. This accepts an initial state `rng` and passes it through without modification, returning the constant value `a`. 

The **`map` combinator** allows for pure transformations of the result without altering the state. Since `Rand[A]` is functionally equivalent to `RNG => (A, RNG)`, `map` is implemented as function composition: `def map[A,B](s: Rand[A])(f: A => B): Rand[B] = rng => { val (a, rng2) = s(rng); (f(a), rng2) }`. For instance, the calculation `def nonNegativeEven: Rand[Int] = map(nonNegativeInt)(i => i - i % 2)` demonstrates how `map` transforms the output of `nonNegativeInt` while correctly propagating the state (`rng2`).

To combine results from two independent state actions, the **`map2` combinator** is necessary. This function takes two state actions, `ra: Rand[A]` and `rb: Rand[B]`, and a combining function `f: (A, B) => C`, returning a new action. It is defined as `def map2[A,B,C](ra: Rand[A], rb: Rand[B])(f: (A, B) => C): Rand[C]`. This pattern leads to utility functions like `both[A,B](ra: Rand[A], rb: Rand[B]): Rand[(A,B)] = map2(ra, rb)((_, _))`. For example, generating an (Int, Double) pair is achieved by `val randIntDouble: Rand[(Int, Double)] = both(int, double)`. More generalized combinations, such as `both(int, double)` and `both(double, int)`, can be formed. For sequence generation, the **`sequence` combinator** combines a list of state actions into a single coherent state transition, represented by `def sequence[A](fs: List[Rand[A]]): Rand[List[A]]`. This enables the entire `ints` function to be generated using `sequence` on a list of state actions.

These combinators (`unit`, `map`, `map2`, `sequence`) are inherently generalized; they do not depend on the state being specifically an `RNG`. This generality necessitates a more powerful abstraction: **`flatMap`**. The signature is `def flatMap[A,B](f: Rand[A])(g: A => Rand[B]): Rand[B]`. `flatMap` allows for a powerful control flow structure: it generates an initial random value of type `A` using the `Rand[A]` action, and then uses the resulting value `A` as input to determine which subsequent state action, `g: A => Rand[B]`, should be executed. This conditional execution capability is crucial for complex logic like implementing `nonNegativeLessThan`. 

Demonstrating the superiority of `flatMap` over `map` and `map2`, the functional model improves utility functions like `rollDie`. The initial definition `def rollDie: Rand[Int] = nonNegativeLessThan(6)` can fail (e.g., returning 0 when tested with `SimpleRNG(5)`). The correct, robust definition must use `map` to transform the result: `def rollDie: Rand[Int] = map(nonNegativeLessThan(6))(_ + 1)`.

In summary, the entire state management API—including `unit`, `map`, `map2`, `sequence`, and `flatMap`—provides a unified, functional way to handle stateful computations, guaranteeing referential transparency and explicit control over state flow, regardless of the specific state type (RNG, FooState, etc.).
