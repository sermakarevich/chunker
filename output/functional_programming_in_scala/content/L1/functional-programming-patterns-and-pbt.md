# functional-programming-patterns-and-pbt

**Parent:** [[content/L2/advanced-functional-algebra-design|advanced-functional-algebra-design]] — Functional design requires treating computation and API logic as a functional algebra, employing specialized ADTs like `Option` and `Either` for compile-time error handling, and using `Stream` for memory-efficient lazy data transformations. Concurrency is managed via `Par` using the non-blocking Actor model to avoid deadlocks, and testing is achieved through `Prop` and `Gen` in Property-Based Testing, which uses a structured `Result` trait for detailed pass/fail reporting.

The principles of functional programming mandate treating computation itself as a first-class value, fundamentally shaping how control flow and side effects are managed. To ensure total type safety and prevent issues like silent failures, APIs must model themselves as functional algebras, involving defined data types, functions, and rigorously established laws. Traditional mechanisms such as runtime exceptions or sentinel values are actively eschewed, replaced by mandatory explicit encapsulation of all potential effects, including error states. This is achieved through specialized Algebraic Data Types (ADTs) and Higher-Order Functions (HOFs).

For managing non-fatal control flow, two specialized ADTs are foundational: `Option[+A]` and `Either[+A, +B]`. The `Option` type models the potential presence or absence of a value using two cases: `case class Some[+A](get: A)` for a defined value, and `case object None extends Option[Nothing]` for an undefined state. This structure is vital for defining *total functions*; for example, calculating the mean of an empty sequence must return `None` when using `Option[Double]`. This elevates the potential failure state to a static, compile-time contract, significantly improving upon using boilerplate `try-catch` blocks. Complementing this, the `Either` type offers a structured, two-sided mechanism: success is carried by `Right[+A]`, and failure or error is represented by `Left[+B]`. These HOFs abstract complex error propagation patterns, thus ensuring reliable, compile-time checked error handling.

Efficiency demands in data processing are managed by employing *non-strictness*, or laziness, especially when chaining multiple data transformations. When standard collections like `List` are processed—for instance, using the sequence `map(_ + 10).filter(_ % 2 == 0).map(_ * 3)` on `List(1, 2, 3, 4)`—traditional execution requires multiple passes and generates numerous temporary intermediate data structures: `List(11, 12, 13, 14)`, then `List(12, 14)`, and finally `List(36, 42)`. This intense memory allocation burden is mitigated by embracing non-strictness. While conceptual examples include short-circuiting Boolean operators (`&&`, `||`) and the `if` construct, the concept is most robustly and explicitly realized in `Stream[+A]`, a specialized trait for lazy lists.

A `Stream` is defined by two constructors: `Empty extends Stream[Nothing]` and `Cons[+A](h: () => A, t: () => Stream[A])`. Structurally, the use of empty function literals (`() => A` and `() => Stream[A]`) is critical, forcing both the head (`h`) and tail (`t`) components to be non-strict (thunks). This guarantees that both the head value and the tail computation are evaluated only once upon first demand, a mechanism known as memoization. The smart constructor `cons[A](hd: => A, tl: => Stream[A]): Stream[A]` manages this memoization. Accessing stream components requires explicit thunk handling; for instance, `headOption: Option[A]` returns `None` if the stream is `Empty`, or `Some(h())` if it is `Cons(h, t)`, explicitly calling `h()` without forcing the tail. Utility methods include `take(n)` (to retrieve the first $n$ elements), `drop(n)` (to skip the first $n$ elements), and `takeWhile(p: A => Boolean)` (returning elements that satisfy predicate $p$).

Advanced stream operations leverage laziness immensely. Core combinators like `map`, `filter`, `append`, and `flatMap` chain transformations without generating intermediate results. Processing `Stream(1,2,3,4).map(_ + 10).filter(_ % 2 == 0).toList` demonstrates computation generating elements sequentially, thereby saving memory by allowing the garbage collector to reclaim space. This efficiency is crucial when handling infinite streams. One can safely process `val ones: Stream[Int] = Stream.cons(1, ones)` using `ones.take(5).toList` (yielding `List(1, 1, 1, 1, 1)`) or `ones.exists(_ % 2 != 0)` (returning `true` immediately). Developers must, however, avoid infinite loops, such as with `ones.forAll(_ == 1)`, which results in a stack overflow because the predicate never finds a non-matching element allowing termination. Specialized streams can be generated using `constant[A](a: A)` (for an infinite stream of $a$) or `from(n: Int)` (for an infinite sequence starting from $n$). The most general and powerful builder is the corecursive function `unfold[A, S](z: S)(f: S => Option[(A, S)]): Stream[A]`, taking an initial state $z$ and a function $f$ to determine the next state and value, terminating when $f$ returns `None`. The `unfold` function supports building `constant`, `from`, and `ones`, and implementing combinators like `map`, `take`, `takeWhile`, and `zipWith[B](s2: Stream[B]): Stream[(A, B)]`. A paired function, `zipAll[B](s2: Stream[B]): Stream[(Option[A], Option[B])]`, traverses as long as either input stream contains elements, using `Option` to track exhaustion.

Structurally, the `Stream` includes `def startsWith[A](s: Stream[A]): Boolean` (for checking prefix relationship) and the critical `def tails: Stream[Stream[A]]` (which returns a stream of all suffixes; e.g., for `Stream(1,2,3)`, this sequence is `Stream(Stream(1,2,3), Stream(2,3), Stream(3), Stream())`).

### Concurrency and Parallelism (Par)

Concurrency and parallelism are managed using the `Par[A]` data type, designed specifically to represent a non-blocking computation. To achieve this, the design must depart from standard Java blocking mechanisms like `java.util.concurrent.Future`, which necessitate calling `.get()` to obtain a value. Instead, `Par[A]` employs a private method `apply(k: A => Unit)` that accepts a continuation function `k`, which handles the result $A$ and performs subsequent effects. While the internal mechanism may involve side effects ($A 	o Unit$), these effects are confined and undetectable to the user code.

The formalization of `fork` introduces key laws, notably that `fork(x) == x` must hold, asserting that wrapping a computation $x$ using `fork` does not alter its result; it merely executes $x$ asynchronously. If this law is violated, general combinators like `parMap` lose soundness, potentially leading to deadlocks. A specific failure case illustrates this: utilizing an `ExecutorService` with a fixed pool size of one (e.g., `Executors.newFixedThreadPool(1)`), running the sequence `val a = lazyUnit(42 + 1)` followed by `val S = Executors.newFixedThreadPool(1)` and then `println(Par.equal(S)(a, fork(a)))` will cause a deadlock. This occurs because the internal `fork` implementation, `es => es.submit(new Callable[A] { def call = a(es).get })`, submits and handles the outer `Callable` on the sole available thread, and within that same thread, it blocks waiting for the result of a second `Callable` via `a(es).get`. Since the thread pool is limited to one, no resources remain for the second computation, resulting in the deadlock.

To circumvent this, two alternative solutions were analyzed: 1) `def fork[A](a: => Par[A]): Par[A] = es => fa(es)` (a form of `delay`), which avoids the deadlock but fails to genuinely initiate a separate logical thread for evaluating $fa$. Instead, `fork(hugeComputation)(es)` executes `hugeComputation` on the main thread, delaying the computation until it is needed. 2) The ideal solution for genuine parallel execution on fixed-size thread pools mandates a fully non-blocking `Par` representation. This requires that `fork` and `map2` never call methods that block the current thread, such as `Future.get`.

The core mathematical laws governing parallelism, such as `map(unit(1))(_ + 1) == unit(2)` and `map(y)(id) == y`, are vital for maintaining compositionality. Sophisticated combinators, like `map2[A,B,C](p: Par[A], p2: Par[B])(f: (A,B) => C): Par[C]`, necessitate advanced concurrency management. The recommended model solution is the **Actor** model. An Actor acts as a concurrent process managing state (e.g., two mutable variables: `ar: Option[A]` and `br: Option[B]` for `map2`) and processes incoming messages sequentially. The Actor processes the first message (e.g., `Left(a)` or `Right(b)`), stores it, and waits for the second result. Upon receiving the second result, the Actor executes the combining function $f$ and propagates the final $C$ result. This non-blocking design enables operations like `parMap(List.range(1, 100000))(math.sqrt(_))` to run efficiently using a fixed-size thread pool (e.g., two threads) without consuming 100,000 JVM threads, thereby successfully validating the law of forking even under severe resource constraints. This comprehensive framework relies on understanding that `flatMap` and `join` relate fundamentally to `map` and `map2`, allowing complex API structures to be constructed from a minimal, algebraically coherent set of primitives. Ultimately, this structure necessitates the use of Property-Based Testing (PBT), where properties are defined abstractly (e.g., `ns.reverse.reverse == ns`) and checked using a generator (`Gen`), serving as a powerful form of developer verification.

### Property-Based Testing Utilities (Gen and Prop)

To facilitate Property-Based Testing (PBT), two core concepts are introduced: `Gen[A]` (Generator) and `Prop` (Property). The `forAll` function initiates the property check: `def forAll[A](a: Gen[A])(f: A => Boolean): Prop`. The resulting property is stored in the `Prop` type. Initially, `Prop` was theorized as a trait with `def check: Unit` and `def &&(p: Prop): Prop = ???`. However, requiring two separate `check` method calls for composition via `&&` would produce two distinct reports, which was deemed incorrect. To allow composition, `check` had to return a meaningful value, not just perform a side effect. If `Prop` were represented as a non-strict Boolean, any standard Boolean function (AND, OR, NOT, XOR) would work, but this was considered insufficient because failure details might require tracking both the count of successful tests and the specific failing arguments. Conversely, success reporting often requires knowing the total number of tests run.

Initial attempts led to `trait Prop { def check: Either[(FailedCase, SuccessCount), SuccessCount] }`. This definition could be refined further using `type TestCases = Int` and `type Result = Either[(FailedCase, SuccessCount), SuccessCount]]`, giving an initial structure of `case class Prop(run: TestCases => Result)`. However, when a property passes, the `SuccessCount` is implied to equal the argument passed to `run`, and since the caller learns nothing new from the success count in the `Right` case of the `Either`, the `Result` type was simplified to `type Result = Option[(FailedCase, SuccessCount)]`, and the `Prop` case class updated to `case class Prop(run: TestCases => Result)`.

Crucially, using `None` to indicate failure was deemed counter-intuitive for representing the absence of a failure. Therefore, a dedicated sealed trait, `Result`, was created to clarify intent: - `case object Passed extends Result` indicates all tests passed and has `def isFalsified = false`. - `case class Falsified(failure: FailedCase, successes: SuccessCount) extends Result` indicates failure and has `def isFalsified = true`. This improved clarity led to re-examining `forAll`'s requirements. Since `forAll` must generate random test cases, the `Prop` definition must propagate the dependency on randomness. Thus, the signature was updated to: `case class Prop(run: (TestCases, RNG) => Result)`. By including `(TestCases, RNG)` as parameters to `Prop.run`, the `Prop` type gains the necessary information to implement `forAll` correctly.

Regarding the `Gen` representation, it is defined as: `case class Gen[A](sample: State[RNG, A])`. This allows for the implementation of functions like `choose` (generating integers in the range `start` to `stopExclusive`), `unit` (always generating value $a$), and `listOfN` (generating lists of length $n$). For implementing generators that depend on another value (e.g., generating pairs `(String, String)` where the second string's characters are limited by the first), the `flatMap` function is required. This allows one generator to use its generated value to determine the subsequent generator. This capability can generalize `listOfN` to support variable lengths by implementing `listOfN(size: Gen[Int]): Gen[List[A]]` using `flatMap`.

To provide a concrete implementation, the `forAll` function was defined as: `def forAll[A](as: Gen[A])(f: A => Boolean): Prop = Prop { (n, rng) => randomStream(as)(rng).zip(Stream.from(0)).take(n).map { case (a, i) => try { if (f(a)) Passed else Falsified(a.toString, i) } catch { case e: Exception => Falsified(buildMsg(a, e), i) } }.find(_.isFalsified).getOrElse(Passed)`. This complex implementation relies on `randomStream[A](g: Gen[A])(rng: RNG)`, which generates an infinite stream of $A$ values by repeatedly sampling $g$ using the random number generator $rng$, defined as `Stream.unfold(rng)(rng => Some(g.sample.run(rng)))`. Furthermore, the utility function `buildMsg[A](s: A, e: Exception)` constructs a detailed failure message string, combining details from the failed value and the exception.
