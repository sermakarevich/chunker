# functional-fp-laziness-and-error-handling

**Parent:** [[content/L2/functional-laziness-and-error-handling|functional-laziness-and-error-handling]] — Functional programming handles potential failures by using specialized ADTs like `Option[+A]` (for presence/absence) and `Either[+A, +B]` (for success/error), enforcing compile-time safety. For efficiency, it utilizes *non-strictness* (laziness), implemented via `Stream[+A]` (which uses memoization and thunks for its `Cons` constructor), enabling core combinators like `map` and `filter` to process data in a single pass without intermediate memory overhead.

Functional programming principles emphasize representing computation, including potential exceptions, not as mandatory execution paths but as values. The core concept revolves around designing systems where effects, such as error states, are encapsulated using algebraic data types and higher-order functions (HOFs). This approach ensures complete type safety and forces the calling code to explicitly handle all possible outcomes, preventing silent failures that often occur with sentinel values or simple checks. 

To manage potential failure states, specialized types like `Option[+A]` and `Either[+A, +B]` are utilized. The `Option` type represents the possibility of a value existing, containing two defined cases: `case class Some[+A](get: A)` for defined values, and `case object None extends Option[Nothing]` for undefined values. This design makes functions that might not always yield a result, such as computing the mean of a sequence, *total functions* (meaning they always return a declared value type). For instance, a function calculating the mean using `Option[Double]` will return `None` if the input sequence is empty, thereby making the possibility of failure part of the function's static contract. This mechanism is a significant improvement over the prior reliance on runtime exceptions, which can lead to unrecoverable states or require cumbersome `try-catch` blocks.

Beyond `Option`, the `Either` type provides a structured way to return one of two possible values, allowing developers to carry both a success value (often represented by `Right[+A]`) and an error value (often represented by `Left[+B]`). By leveraging these HOFs, common error propagation patterns can be abstracted away, greatly minimizing boilerplate code while maintaining robust, compile-time guaranteed error handling.

These concepts of handling non-fatal control flow—such as failure or success—are generalizable. The general principle is that program designers can represent exceptions as ordinary values, a concept that will be revisited frequently throughout functional programming.

One of the primary efficiency concerns in functional programming relates to chaining multiple transformations on data structures, such as applying a sequence of `map`, `filter`, and subsequent `map` calls. If executed traditionally, this sequence requires multiple passes over the input data and generates numerous temporary, intermediate data structures (intermediate lists). For example, the sequence `List(1, 2, 3, 4).map(_ + 10).filter(_ % 2 == 0).map(_ * 3)` first creates `List(11, 12, 13, 14)` in the first step, then creates `List(12, 14)` in the second step, and finally produces `List(36, 42)` in the third. This excessive memory allocation and computational overhead due to multiple passes are undesirable. To solve this, the concept of automatic loop fusion is introduced, achieved by employing *non-strictness*, or laziness. 

Non-strictness is a fundamental property of a function, meaning it has the option to not evaluate all its arguments. Conversely, a strict function always evaluates all arguments passed to it. By default, Scala treats all function definitions as strict. The contrast between strict and non-strict evaluation is exemplified by the short-circuiting Boolean operators `&&` and `||`, which only evaluate arguments as needed. The standard `if` control construct in Scala also exhibits non-strict behavior; it does not necessarily evaluate both the true and false branches. To achieve controlled non-strictness, Scala provides the `if2[A]` function, which accepts arguments using the empty function literal syntax `() => A`. This syntax, conceptually equivalent to `Function0[A]`, guarantees that the provided block of code (e.g., `onTrue` or `onFalse`) will only run if the conditional check demands it. The `if2` signature is: `def if2[A](cond: Boolean, onTrue: () => A, onFalse: () => A): A = if (cond) onTrue() else onFalse()`. The use of empty parentheses (`onTrue()` or `onFalse()`) is required to explicitly force the evaluation and retrieve the result from these thunks. 

Scala offers a more syntactically convenient alias for non-strict arguments using the arrow `=>` (e.g., `def if2[A](cond: Boolean, onTrue: => A, onFalse: => A): A = if (cond) onTrue else onFalse`). When the `=>` syntax is used, Scala automatically wraps the expression in a 'thunk' (an unevaluated form of an expression), which must subsequently be evaluated. 

This concept of laziness is exemplified by `Stream[+A]`, a specialized type designed to model lazy lists or streams. A stream is defined by the `Stream[+A]` trait, which has two constructors: `Empty extends Stream[Nothing]` for the empty stream, and `Cons[+A](h: () => A, t: () => Stream[A])` for a non-empty stream. Crucially, the head (`h`) and tail (`t`) constructors are written as *non-strict* (i.e., they take thunks), meaning they are only evaluated when explicitly requested. The `Stream` object provides smart constructors, notably `cons[A](hd: => A, tl: => Stream[A]): Stream[A]`, which automatically handles memoization. This memoization pattern ensures that the underlying thunk for either the head or the tail will execute only once upon the first attempt to force the value, and all subsequent accesses return the cached, computed result. 

When examining a `Stream`, evaluation involves explicitly forcing the underlying thunks. For example, the `headOption: Option[A]` method demonstrates this by returning `None` if the stream is `Empty`, or `Some(h())` if it is `Cons(h, t)`, explicitly calling `h()` to force the head thunk. The `Stream` object also includes a `def apply[A](as: A*): Stream[A]` smart constructor for creating streams from multiple elements. This constructor handles the recursive construction and ensures that both the initial head and subsequent tail components are wrapped in thunks, thus maintaining the lazy evaluation property. 

These streaming mechanisms are essential for achieving functional efficiency, as they allow chains of transformations to operate on data in a single pass without the overhead of intermediate memory allocations, thus greatly improving modularity and efficiency in functional programs.
