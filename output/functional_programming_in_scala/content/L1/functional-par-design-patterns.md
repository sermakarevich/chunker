# functional-par-design-patterns

**Parent:** [[content/L2/functional-paradigm-architecture|functional-paradigm-architecture]] — Functional programming abstracts computations into first-class values, using ADTs like Option and Either to handle non-fatal errors. Efficiency is achieved via non-strictness (laziness), implemented using `Stream[+A]` to fuse operations and handle infinite sequences, while parallelism is managed by the `Par[A]` type, which defers execution until a terminal `run` call and uses combinators like `map2` and `fork` to maintain purity and manage resource-intensive effects.

Functional programming relies on core principles that abstract computation, treating it as a first-class value, thereby ensuring type safety and managing complex control flow using specialized structures and techniques. The general approach dictates that potential exceptions should not be handled as mandatory control flow paths, but rather that these effects, including error states, must be encapsulated using Algebraic Data Types (ADTs) and Higher-Order Functions (HOFs). This system forces developers to explicitly account for all possible outcomes, thus mitigating the risk of silent failures often associated with traditional sentinel values or simple runtime checks. The fundamental pattern is representing exceptions as ordinary, first-class values.

To manage non-fatal control flow, two primary specialized types are essential: `Option[+A]` and `Either[+A, +B]`. The `Option` type models simple presence or absence, featuring two defined cases: `case class Some[+A](get: A)` for an existing value, and `case object None extends Option[Nothing]` for an undefined state. This design allows functions that may not always yield a result, such as calculating the mean of a sequence, to be classified as *total functions*. For example, if a mean calculation uses `Option[Double]`, it statically returns `None` when the input sequence is empty, establishing the potential failure state as an inherent, static part of the function's contract. Complementing this, the `Either` type provides a structured mechanism to carry one of two distinct outcomes: a success outcome, typically wrapped in `Right[+A]`, or a failure/error value, often contained within `Left[+B]`. Leveraging these HOFs allows complex error propagation patterns to be abstracted away, significantly reducing boilerplate code while guaranteeing compile-time checked error handling.

Regarding performance, especially when chaining multiple data transformations (e.g., `map`, `filter`, and subsequent `map`) on structures like standard Scala `List`, execution traditionally mandates multiple passes over the initial data and, critically, generates numerous temporary, intermediate data structures (intermediate lists). For instance, processing `List(1, 2, 3, 4)` through the sequence `map(_ + 10).filter(_ % 2 == 0).map(_ * 3)` requires generating three sequential lists: `List(11, 12, 13, 14)`, then `List(12, 14)`, and finally `List(36, 42)`. This overhead of excessive memory allocation and computational burden due to multiple passes is highly undesirable. The functional solution is the implementation of automatic loop fusion, which is achieved through the employment of *non-strictness*, or laziness.

Non-strictness is a functional property allowing a function to not evaluate all its arguments; conversely, a strict function always evaluates every argument. While Scala treats definitions as strict by default, the concept is illustrated by short-circuiting Boolean operators like `&&` and `||`, and the standard `if` control construct, which does not necessarily evaluate both branches. To enforce controlled non-strictness, `if2[A]` is introduced: `def if2[A](cond: Boolean, onTrue: () => A, onFalse: () => A): A = if (cond) onTrue() else onFalse()`. The use of the empty function literal `() => A` ensures the provided block only executes if the condition mandates it. A more syntactically convenient alternative uses the arrow `=>` (e.g., `def if2[A](cond: Boolean, onTrue: => A, onFalse: => A): A = if (cond) onTrue else onFalse`), automatically wrapping the expression in a 'thunk'.

This concept is robustly implemented in `Stream[+A]`, a specialized trait designed for modeling lazy lists. A stream is constructed using two forms: `Empty extends Stream[Nothing]` for the empty stream, and `Cons[+A](h: () => A, t: () => Stream[A])` for a non-empty one. The structural necessity of using `() => A` and `() => Stream[A]` for both the head (`h`) and tail (`t`) components forces them to be non-strict (thunks). This guarantees that both the head's value and the tail's computation are evaluated only once upon the first demand, a process known as memoization. The smart constructor `cons[A](hd: => A, tl: => Stream[A]): Stream[A]` manages this memoization. The `Stream.apply[A](as: A*)` constructor builds streams from multiple elements; if `as` is empty, it returns `Empty`; otherwise, it recursively uses `cons`, automatically wrapping both the initial head and subsequent tail components in thunks, thereby preserving laziness.

Accessing stream components requires explicit handling of thunks. For instance, `headOption: Option[A]` returns `None` if the stream is `Empty`, or `Some(h())` if it is `Cons(h, t)`, forcing the head thunk (`h()`) while crucially *not* forcing the tail. Utilities like `take(n)` (returns the first $n$ elements), `drop(n)` (skips the first $n$ elements), and `takeWhile(p: A => Boolean)` (returns starting elements satisfying predicate $p$) are available.

Advanced stream operations benefit greatly from laziness. Core combinators such as `map`, `filter`, `append`, and `flatMap` chain transformations without instantiating intermediate results. For example, `Stream(1,2,3,4).map(_ + 10).filter(_ % 2 == 0).toList` demonstrates that the computation generates elements sequentially, saving memory by allowing the garbage collector to reclaim space for values (like 11 and 13) as soon as they are no longer needed. This efficiency is paramount for handling infinite streams. An infinite stream of ones, `val ones: Stream[Int] = Stream.cons(1, ones)`, can be safely processed: `ones.take(5).toList` yields `List(1, 1, 1, 1, 1)`, and `ones.exists(_ % 2 != 0)` immediately returns `true` upon finding the first satisfying element. However, caution is required: operations like `ones.forAll(_ == 1)` will cause an infinite loop (stack overflow) because the predicate never finds a non-matching element to allow termination.

Developers can generate specialized streams using advanced methods. `def constant[A](a: A): Stream[A]` generates an infinite stream of $a$. `def from(n: Int): Stream[Int]` generates an infinite sequence of integers starting from $n$. `def fibs` generates the infinite Fibonacci sequence (0, 1, 1, 2, 3, 5, 8, etc.). A general, corecursive function is `def unfold[A, S](z: S)(f: S => Option[(A, S)]): Stream[A]`. `unfold` takes an initial state $z$ and a function $f$ that determines both the next state and the next value, terminating when $f$ returns `None`. The power of `unfold` is that it can redefine core builders: `def fibs` (EXERCISE 5.12), `def from` (EXERCISE 5.12), `def constant` (EXERCISE 5.12), and `def ones` (EXERCISE 5.12). Using `unfold` to define `constant` and `ones` preserves constant memory usage, improving upon simple recursive definitions. Furthermore, `unfold` can implement other combinators, including `map` (EXERCISE 5.13), `take` (EXERCISE 5.13), `takeWhile` (EXERCISE 5.13), and `zipWith[B](s2: Stream[B]): Stream[(A, B)]` (EXERCISE 5.13), which follows the Chapter 3 `zipWith` pattern. A paired function, `zipAll[B](s2: Stream[B]): Stream[(Option[A], Option[B])]` (EXERCISE 5.13), continues traversing as long as either input stream has elements, using `Option` to track stream exhaustion. 

Structurally, `def startsWith[A](s: Stream[A]): Boolean` (EXERCISE 5.14) checks if stream $s$ is a prefix of another (e.g., `Stream(1,2,3)` starts with `Stream(1,2)` returns `true`). From this, `def hasSubsequence[A](s: Stream[A]): Boolean` (EXERCISE 5.14) is implemented by checking if any tail of the stream starts with $s$: `tails exists (_ startsWith s)`. A more powerful structural method is `def tails: Stream[Stream[A]]` (EXERCISE 5.15), which returns a stream of all suffixes of the input sequence. For `Stream(1,2,3)`, `tails` generates `Stream(Stream(1,2,3), Stream(2,3), Stream(3), Stream())`.

Functional design for parallel computation introduces `Par[A]`, a conceptual type representing an abstract parallel computation. Execution is deferred until the function `run[A](a: Par[A]): A` is called, which manages the underlying mechanisms, such as using an `ExecutorService` (specifically, `def run[A](s: ExecutorService)(a: Par[A]): A`). By keeping `Par` pure, the user programs against a pure interface, while the `run` function handles the necessary side effects (e.g., task submission). 

Three fundamental combinators define this functional API:

1.  **`unit[A](a: A): Par[A]`**: Creates a parallel computation that immediately yields the value `a`. This function does not require an `ExecutorService`.
2.  **`map2[A,B,C](a: Par[A], b: Par[B])(f: (A,B) => C): Par[C]`**: Combines results from two parallel computations, `a` and `b`, using a binary function `f`. Its implementation requires passing the `ExecutorService` to both `a` and `b` to obtain intermediate `Future` results (`af` and `bf`), which are then used to apply `f` to `af.get` and `bf.get`, finally wrapping the result in a `UnitFuture`. (It is noted that this specific implementation does not respect timeouts.)
3.  **`fork[A](a: => Par[A]): Par[A]`**: Marks a computation for concurrent evaluation. This function utilizes the `ExecutorService` to submit a `Callable[A]` task, ensuring the computation runs in a separate logical thread, thereby preserving potential parallelism. The canonical implementation is `def fork[A](a: => Par[A]): Par[A] = es => es.submit(new Callable[A] { def call = a(es).get })`.

These methods define the pure functional API, relegating effectful execution to the terminal `run` call.

To combine parallel results, a higher-order function `Par.map2` can be defined, allowing the combination of two parallel computations, $sum(l)$ and $sum(r)$, into a single overall computation: `Par.map2(sum(l), sum(r))(_ + _)`. To prevent loss of information, the generalized `sum` function must return `Par[Int]` rather than `Int`. The recursive definition of `sum` is structured as follows: if the input sequence `ints` has a size of 1 or less, the function returns `Par.unit(ints.headOption.getOrElse(0))`; otherwise, it splits `ints` into two parts, `l` and `r`, and computes `Par.map2(sum(l), sum(r))(_ + _)`. 

Crucially, the design choice for `Par.map2`'s arity regarding laziness is pivotal. To ensure both computations are treated as independent and can run concurrently, `map2` must be lazy. An analysis shows that if `map2` were strict in both arguments, when evaluating `sum(IndexedSeq(1, 2, 3, 4))`, the left-to-right evaluation order would strictly construct the entire left half of the computation tree before starting the right half, contradicting true parallel behavior. The goal is to make `map2` lazy so it initiates immediate parallel execution for both arguments, eliminating prioritization. A more advanced mechanism, `Par.fork`, can explicitly manage parallel task initiation. By using `Par.fork(sum(l))` and `Par.fork(sum(r))` within the `map2` call, the `sum` function can maintain a strict `map2` while guaranteeing the two parallel tasks execute asynchronously. This separates the concerns of combining results from asynchronous execution, avoiding the attachment of an arbitrary global policy for parallelism to the `map2` combinator. Furthermore, incorporating `Par.fork` allows the `unit` function to become strict without sacrificing expressiveness. A derived combinator, `lazyUnit[A](a: => A): Par[A]`, serves this purpose, utilizing `unit` and `fork` to implement asynchronous behavior.

In terms of foundational combinators, `map` can be derived from `map2`. Specifically, `def map[A,B](pa: Par[A])(f: A => B): Par[B] = map2(pa, unit(()))((a,_) => f(a))`. This derivation highlights that `map2` is strictly more powerful than `map`, illustrating how a seemingly primitive function can be expressed using a more foundational primitive.

Beyond `map2`, a function `parMap` can be designed to combine $N$ parallel computations. Unlike `map2`, which only handles two, `parMap` must process a list of parallel computations (`List[A]`) using a function `f: A => B` to return `Par[List[B]]`: `def parMap[A,B](ps: List[A])(f: A => B): Par[List[B]]`. Although defining `parMap` as a new primitive is possible, the structure of the problem suggests that existing combinators, combined with assumptions about underlying data representation, might allow for more efficient implementations. The general pattern shown by `map` and `sortPar` demonstrates the powerful utility of `map2` as the central mechanism for value transformation within the parallel context. The overall architecture emphasizes maintaining purity in the API while strategically isolating effectful execution to the `run` function through sophisticated type-level abstractions and combinators.
