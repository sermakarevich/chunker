# functional-programming-advanced-concepts

**Parent:** [[content/L2/functional-algebra-design-patterns|functional-algebra-design-patterns]] — Functional programming mandates treating computation as a first-class value and utilizing ADTs like `Option` and `Either` to handle non-fatal control flow and errors explicitly. For performance, specialized `Stream` types enforce non-strictness (laziness), avoiding the creation of intermediate data structures when chaining transformations like `map` and `filter` on collections such as `List(1, 2, 3, 4)`. Concurrency is managed via `Par[A]`, which requires sophisticated, non-blocking primitives—such as using the Actor model for `map2`—to prevent deadlocks, ensuring that foundational laws like `fork(x) == x` hold even when executing 100,000 iterations with a fixed-size thread pool.

The design of robust functional programming languages hinges on treating computation itself as a first-class value, which fundamentally alters how control flow and side effects are managed. Instead of relying on traditional sentinel values or runtime exceptions, best practices mandate encapsulating potential effects—including error states—using specialized algebraic data types (ADTs) and higher-order functions (HOFs). This guarantees complete type safety by forcing developers to explicitly handle all possible outcomes, thereby preventing silent failures. 

To manage non-fatal control flow, two crucial specialized types are utilized: `Option[+A]` and `Either[+A, +B]`. The `Option` type explicitly models the presence or absence of a value, defined by two distinct cases: `case class Some[+A](get: A)` for a defined value, and `case object None extends Option[Nothing]` for an undefined state. This design is vital because it allows functions that may not always yield a result, such as calculating the mean of a sequence, to be designated *total functions*. For instance, if a mean calculation function uses `Option[Double]`, it statically returns `None` when the input sequence is empty, making the potential failure an inherent part of the function’s contract, which is vastly superior to relying on boilerplate `try-catch` blocks or runtime exceptions. Complementing this, the `Either` type provides a structured mechanism to carry one of two specific outcomes: success, typically represented by `Right[+A]`, or a failure/error value, often represented by `Left[+B]`. By leveraging these HOFs, complex error propagation patterns can be abstracted, significantly minimizing boilerplate code while guaranteeing compile-time checked error handling. These concepts generalize far beyond simple options and left/right structures, concerning the handling of non-fatal control flow, whether success or failure.

Furthermore, efficiency concerns in functional programming often relate to chaining multiple data transformations (e.g., sequential `map`, `filter`, and subsequent `map` calls) on standard Scala `List`. Executing such a sequence traditionally requires multiple passes over the initial data and generates numerous temporary, intermediate data structures. For example, processing `List(1, 2, 3, 4)` through `map(_ + 10).filter(_ % 2 == 0).map(_ * 3)` first produces `List(11, 12, 13, 14)`, then creates a second list, `List(12, 14)`, and finally a third, `List(36, 42)`. This massive overhead of excessive memory allocation and computational burden due to multiple passes is highly undesirable. The robust solution is the implementation of automatic loop fusion, which is achieved through the employment of *non-strictness*, or laziness. 

Non-strictness is a core functional property meaning a function is permitted to not evaluate all its arguments. In contrast, a strict function always evaluates every argument. While Scala treats function definitions as strict by default, the concept is clearest in short-circuiting Boolean operators like `&&` and `||`, and in the standard `if` construct, which does not necessarily evaluate both the true and false branches. To enforce controlled non-strictness, Scala introduced `if2[A]`: `def if2[A](cond: Boolean, onTrue: () => A, onFalse: () => A): A = if (cond) onTrue() else onFalse()`. Here, the empty function literal `() => A` ensures the provided block only runs if the conditional check mandates it. A more convenient alias uses the arrow `=>` (e.g., `def if2[A](cond: Boolean, onTrue: => A, onFalse: => A): A = if (cond) onTrue else onFalse`), automatically wrapping the expression in a 'thunk' (an unevaluated form). This concept achieves its most robust and explicit implementation in `Stream[+A]`, a specialized trait designed to model lazy lists. 

A `Stream` has two constructors: `Empty extends Stream[Nothing]` for the empty stream, and `Cons[+A](h: () => A, t: () => Stream[A])` for a non-empty one. The use of `() => A` and `() => Stream[A]` is structurally critical because it forces both the head (`h`) and tail (`t`) components to be non-strict (thunks). This guarantees that the value of the head and the computation of the tail are evaluated only once upon the first demand, a mechanism known as memoization. The `Stream` object provides the smart constructor `cons[A](hd: => A, tl: => Stream[A]): Stream[A]`, which manages this memoization. `Stream.apply[A](as: A*)` builds streams from multiple elements `as`; if `as` is empty, it returns `Empty`; otherwise, it recursively uses `cons`, automatically wrapping both the initial head and subsequent tail components in thunks, thereby preserving laziness. Accessing stream components requires explicit handling of thunks; for example, `headOption: Option[A]` returns `None` if the stream is `Empty`, or `Some(h())` if it is `Cons(h, t)`, forcing the head thunk while crucially *not* forcing the tail. Utilities include `take(n)` (returns the first $n$ elements), `drop(n)` (skips the first $n$ elements), and `takeWhile(p: A => Boolean)` (returns starting elements satisfying predicate $p$).

Advanced stream operations benefit immensely from laziness. Combinators such as `map`, `filter`, `append`, and `flatMap` chain transformations without instantiating intermediate results. For example, `Stream(1,2,3,4).map(_ + 10).filter(_ % 2 == 0).toList` demonstrates computation generating elements one by one, saving memory by allowing the garbage collector to reclaim space for values (like 11 and 13) as soon as they are no longer needed. This efficiency is paramount when dealing with infinite streams. An infinite stream of ones, `val ones: Stream[Int] = Stream.cons(1, ones)`, can be safely processed using methods like `ones.take(5).toList` (yielding `List(1, 1, 1, 1, 1)`) or `ones.exists(_ % 2 != 0)` (returning `true` immediately upon finding the first satisfying element). However, one must be careful; operations like `ones.forAll(_ == 1)` will cause an infinite loop (stack overflow) because the predicate never finds a non-matching element that allows termination. Developers can generate specialized streams using methods like `constant[A](a: A)` (generates an infinite stream of $a$) or `from(n: Int)` (generates an infinite sequence of integers starting from $n$). A general, corecursive function is `def unfold[A, S](z: S)(f: S => Option[(A, S)]): Stream[A]`. This function takes an initial state $z$ and a function $f$ determining both the next state and the next value, terminating when $f$ returns `None`. The power of `unfold` is that it can define core builders like `fibs`, `from`, `constant`, and `ones`, while maintaining constant memory usage. Moreover, `unfold` can implement other combinators, including `map`, `take`, `takeWhile`, and `zipWith[B](s2: Stream[B])` (following the Chapter 3 `zipWith` pattern). A paired function, `zipAll[B](s2: Stream[B]): Stream[(Option[A], Option[B])]`, continues traversing as long as either input stream has elements, utilizing `Option` to track stream exhaustion.

Structurally advanced methods include `def startsWith[A](s: Stream[A]): Boolean` (checking if stream $s$ is a prefix of another, e.g., `Stream(1,2,3)` starts with `Stream(1,2)` returns `true`). From this, `def hasSubsequence[A](s: Stream[A]): Boolean` can be implemented by checking if any tail of the stream starts with $s$: `tails exists (_ startsWith s)`. The most powerful structural method is `def tails: Stream[Stream[A]]` (which returns a stream of all suffixes of the input sequence). 

Beyond lazy lists, advanced functional design tackles concurrent computation using types like `Par[A]`, which is implemented as `ExecutorService => Future[A]` to ensure a non-blocking representation. To achieve this, the design introduced a custom `Future` structure, moving away from `java.util.concurrent.Future` because the latter requires blocking to obtain a value. The `Par[A]` type uses a private method `apply(k: A => Unit)` (restricted to the `fpinscala.parallelism` package) where the user provides a continuation function, `k`. This function accepts the result `A` and uses it to perform an effect. Although the internal mechanism uses `A => Unit` (an imperative side effect), these side effects are confined and not observable to the user code. 

While the implementation of the `run` function provides a way to extract the result of a `Par[A]`, it necessarily blocks the calling thread until the value is available. This process involves creating a mutable, thread-safe `AtomicReference[A]` and using a `CountDownLatch(1)`. The `run` implementation executes the `Par` computation, passing a continuation that sets the result in the `AtomicReference` and calls `countDown` on the `CountDownLatch` upon reception. Finally, the `run` function calls `latch.await` and returns `ref.get`. Due to this inherent blocking nature, developers are advised to only call `run` when synchronous waiting is required, or alternatively, modify the API to expose the `apply` method directly to allow for asynchronous callbacks. 

Addressing concurrency further, the functional API must be governed by formal laws. Key laws include that `map(unit(1))(_ + 1)` must equal `unit(2)` and that `map(y)(id)` must equal `y`. These laws are critical for maintaining compositionality and treating API components as black boxes. For example, if the law stating that `fork(x)` is equivalent to `x` for all expressions `x` and all choices of `ExecutorService` does not hold, general-purpose combinators like `parMap` would lose soundness, potentially leading to deadlocks. A critical deadlock bug occurs in most implementations of `fork` when using an `ExecutorService` backed by a fixed-size thread pool, such as those from `Executors.newFixedThreadPool`. If the pool size is limited to one, the sequence `val a = lazyUnit(42 + 1)` followed by `val S = Executors.newFixedThreadPool(1)` and `println(Par.equal(S)(a, fork(a)))` will deadlock. This is because the `fork` implementation (`es => es.submit(new Callable[A] { def call = a(es).get })`) submits and handles the outer `Callable` on the single available thread, but then blocks within that same thread waiting for the result of a second `Callable` via `a(es).get`. Since the pool is limited to one thread, no threads exist to execute the second `Callable`, causing the deadlock. Developers must re-examine the law `fork(x) == x` to ensure robustness, perhaps by stipulating a requirement for unbounded thread pools. Alternative implementations for `fork` include: 1) `def fork[A](a: => Par[A]): Par[A] = es => fa(es)` (named `delay`), which avoids the deadlock but fails to genuinely fork a separate logical thread, meaning `hugeComputation` runs in the main thread. This variant is useful because it allows the computation delay until it is needed. 2) To genuinely run arbitrary computations over fixed-size thread pools, a fully non-blocking representation of `Par` is mandatory. Any `Par` representation that avoids resource leaks and blocking must ensure that `fork` and `map2` never call any method that blocks the current thread, such as `Future.get`.
