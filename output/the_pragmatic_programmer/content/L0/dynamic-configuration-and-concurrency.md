# dynamic-configuration-and-concurrency

**Parent:** [[content/L1/software-design-patterns-and-development-workflows|software-design-patterns-and-development-workflows]] — Software design alternatives to inheritance include Interfaces/Protocols (Traits) and Delegation, which break coupling and enforce necessary behaviors without requiring shared code. Professional workflows mandate structured methods like PERT (optimistic, most likely, pessimistic estimates) and Iterative Estimation, requiring detailed logging and root cause analysis for any estimate error exceeding 50%. Tooling proficiency requires mastering the command shell (WYSIAYG), achieving 'editor fluency' via keyboard mastery and extensions, and utilizing Version Control System (VCS) branching for isolated, audited development.

A critical design consideration for achieving highly available applications is ensuring that configuration remains dynamic. The application should not require stopping and restarting to change a single parameter. Instead, developers should use a configuration service: components of the application can register for notifications of parameter updates, and the service can send messages containing new values when changes occur. Regardless of the specific format, configuration data dictates the runtime behavior of an application, and changing configuration values should not necessitate rebuilding the code.

Lacking external configuration makes the code less adaptable and flexible. The failure to adapt can have severe consequences, comparable to the extinction of the dodo on Mauritius due to human interaction.

Developers must also exercise caution when applying configuration principles. They should avoid **overdoing** the configuration; for instance, configuring every single field in an application, as one early client did, can create a coding nightmare requiring the implementation of both the field and all the administrative code needed to save and edit it. Furthermore, developers should not push complex design decisions to configuration merely out of laziness. If a team has a genuine debate about a feature's implementation or whether the feature should be the user's choice, the developer should prototype the solution in one way and obtain feedback on whether the decision is sound.

Regarding system concurrency, developers must understand the difference between concurrency and parallelism. Concurrency occurs when the execution of two or more pieces of code appear to run simultaneously. Parallelism, conversely, means that the code physically runs at the same time, which requires hardware capability such as multiple cores, multiple CPUs, or connected computers. To achieve concurrency, the operating environment must be able to switch execution between different parts of the code while it is running, often implemented using techniques like fibers, threads, and processes.

Every decently sized system contains concurrent aspects, whether explicit or buried within a library. Concurrency is essential for any application designed to interact with the real world because such systems are inherently asynchronous—whether users are interacting, data is being fetched, or external services are being called. If a developer forces the process to be strictly serial, running one thing only after the previous one completes, the system will feel sluggish and will not utilize the full power of the underlying hardware.
