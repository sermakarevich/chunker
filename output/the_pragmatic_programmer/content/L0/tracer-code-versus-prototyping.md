# tracer-code-versus-prototyping

**Parent:** [[content/L1/software-architecture-risk-management|software-architecture-risk-management]] — The core architectural principles stress achieving flexibility by adhering to concepts like decoupling and abstracting third-party APIs to maintain reversibility. Development risk is managed using two distinct methods: Prototyping, which generates disposable code for low-cost exploration (e.g., whiteboarding a UI), and Tracer Code, which creates a lean, structurally sound framework that guides the overall system integration from the start.

The concept of prototyping is for exploring specific aspects of the final system, allowing developers to analyze and expose risk and find chances for correction at a greatly reduced cost. For instance, a developer producing an application to help shippers determine how to pack odd-sized boxes into containers might prototype the user interface by coding only enough to make the interface responsive to user actions using a UI tool. Developers might also prototype algorithms, such as those for optimal packing, by coding functional tests in a high-level, forgiving language like Python, and low-level performance tests in a language closer to the machine. Once the user agrees on the layout, the developer can throw away the prototype and recode the component with the business logic using the target language. In any case, once a decision is made, the developer must restart and code the algorithms in their final environment, interfacing to the real world. Prototyping generates disposable code.

However, the tracer code approach addresses a fundamentally different problem: establishing how the application, as a whole, hangs together. When developers need to show users how the system interactions will work in practice, or when developers need an architectural skeleton upon which to hang code, they should use tracer code. For example, a developer might construct a tracer consisting of a trivial implementation of the container packing algorithm (such as first-come, first-served) and a simple but working user interface. Once all components are integrated into the application, the resulting framework can be used to demonstrate the system to both users and developers. Over time, the developer adds new functionality and completes stubbed routines, but the underlying framework remains intact, ensuring the system continues to behave as it did when the initial tracer code was completed.

This distinction is critical: prototyping generates disposable code, whereas tracer code is lean, complete, and forms a structural part of the final system. One should think of prototyping as the reconnaissance and intelligence gathering that precedes the firing of a single tracer bullet. 

**Guidance on Prototyping:**

Prototypes can be built in various ways, and not only with code. For example, a user interface can be prototyped as a drawing on a whiteboard, a nonfunctional mock-up using a paint program, or with an interface builder, and Post-it notes are effective for prototyping dynamic items like workflow and application logic. Prototypes are designed to answer only a few questions, making them cheaper and faster to develop than production applications. When prototyping a UI, developers can afford incorrect results or data, and when investigating computational or performance aspects, developers can accept a poor or even nonexistent UI. However, if the development environment demands that the developer cannot ignore details, the developer should consider if a tracer bullet style of development is more appropriate.

**What to Prototype:**

A developer can prototype anything that carries risk: any unproven, experimental, or doubtful element, or anything that is absolutely critical to the final system. Potential prototype targets include architecture, new functionality in an existing system, the structure or contents of external data, third-party tools or components, performance issues, and user interface design.

**Guidance on Implementation:**

When building a prototype, developers should remember that the goal is learning, not producing production-ready code. The developer can ignore details regarding correctness (using dummy data where appropriate), completeness (functioning only in a very limited sense, perhaps with one preselected input and one menu item), robustness (incomplete or missing error checking, leading to potential crashes), and style (the code should not have excessive comments or documentation). Furthermore, developers can use high-level scripting languages, such as Python or Ruby, for prototypes, as these languages are less likely to hinder the development process.
