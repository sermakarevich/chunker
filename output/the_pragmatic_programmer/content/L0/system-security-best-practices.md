# system-security-best-practices

**Parent:** [[content/L1/software-engineering-mastery-and-security-principles|software-engineering-mastery-and-security-principles]] — Professional proficiency requires mastery of foundational tools, adopting structured estimation techniques like PERT and iterative cycles, and achieving fluency in command shells and advanced editors. System design demands a defensive, transformational approach utilizing concepts such as Principle of Least Privilege, minimizing attack surface area, and implementing advanced architectural patterns like PubSub and Actor Model, alongside specific details like encrypting all sensitive data and sanitizing external input.

When developing software, programmers must adopt a defensive approach because perfect software does not exist. A professional programmer must proactively adopt defensive coding practices, recognizing that every system failure is a risk. These practices include validating all provided information, using assertions to detect bad data, distrusting data from potential attackers or trolls, checking for consistency, and applying constraints on database columns. Furthermore, developers must account for both internal errors and deliberate malicious external attacks. An essential aspect of system design is minimizing the attack surface area, which is the total sum of all access points where an attacker can enter data, extract data, or invoke service execution. Since smaller, simpler code is better because complexity increases the attack surface, developers must never trust external data; developers must always sanitize external data before passing it to a database, view rendering, or other processing. Similarly, developers must be cautious with unauthenticated services, as any user in the world can call these services, which creates an opportunity for a denial-of-service attack. It is also vital to keep the number of authorized users at an absolute minimum, and if deployment credentials are compromised, the entire product is compromised. When developers report data, developers must ensure that the reported data is appropriate for the user's authorization level, truncating or obfuscating potentially risky information, such as Social Security or government ID numbers, to prevent revealing private information. Overall, system security requires adherence to basic principles: 1) Minimize Attack Surface Area, 2) Principle of Least Privilege, 3) Secure Defaults, 4) Encrypt Sensitive Data, and 5) Maintain Security Updates. Implementing the Principle of Least Privilege means using the least amount of privilege for the shortest time necessary. Instead of automatically granting the highest permission level, such as root or Administrator, developers should take only the necessary level, perform the minimum work, and quickly relinquish the permission. For example, on Unix-derived systems, the login program starts with root privileges but drops the high level of privilege to that of the user as soon as authentication is complete. Following the Secure Defaults principle dictates that the default settings on an application or for a user should be the most secure values, even if they are not the most user-friendly. Finally, developers must encrypt sensitive data, such as personally identifiable information, financial data, passwords, or other credentials, ensuring that the data is never left in plain text, whether stored in a database or an external file.
