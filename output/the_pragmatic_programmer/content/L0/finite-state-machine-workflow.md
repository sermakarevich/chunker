# finite-state-machine-workflow

**Parent:** [[content/L1/professional-mastery-and-system-design|professional-mastery-and-system-design]] — Professional competency requires mastery of multifaceted tools, from physical implements and sophisticated editor proficiencies to advanced architectural patterns like PubSub and Event Streams; techniques range from PERT and Iterative Estimation to using 'chop' for debugging and enforcing API wrappers for resources to prevent global state coupling.

A Finite State Machine (FSM) dictates that every state transition results in both a new state and the name of an action. The action name is subsequently used to select the code to execute before looping back. While the provided code implementation is very basic, it successfully achieves the required functionality, and many other variants are possible. For instance, the transition table could utilize anonymous functions or function pointers for the actions, or the developers could wrap the code that implements the state machine in a separate class, allowing that class to manage its own internal state. Furthermore, the developers should not feel obligated to process all state transitions simultaneously. For example, when guiding a user through an application signup process, there are likely numerous sequential transitions: the user enters details, the system validates the user's email, the user agrees to the 107 different legislated warnings that online applications must provide, and so on. Using external storage to maintain the state and relying on that state to drive the state machine is an effective method for handling these kinds of complex workflow requirements.
