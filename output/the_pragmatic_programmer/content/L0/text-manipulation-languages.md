# text-manipulation-languages

**Parent:** [[content/L1/professional-software-engineering-workflows|professional-software-engineering-workflows]] — Professional mastery requires mastering specialized tools and methodologies, including the systematic use of PERT (optimistic, likely, pessimistic estimates) and iterative development to manage project uncertainty; developers must achieve 'editor fluency' and leverage specialized text manipulation languages like Python and Ruby to build complex workflows, while defensive programming principles, notably Design by Contract (DBC) defined by Bertrand Meyer, mandate explicitly guaranteeing module boundaries and expected states.

The concept of text manipulation is essential for programmers, who manipulate text in the same way woodworkers shape wood. While basic tools—such as shells, editors, and debuggers—are specialized for one or two specific jobs, general-purpose text manipulation tools are needed to perform transformations not readily handled by the basic toolset. These text manipulation languages, such as Python or Ruby, are powerful and versatile, functioning akin to routers in woodworking; they are described as noisy, messy, and somewhat brute force, and mistakes with them can ruin entire pieces. However, when used properly, these tools exhibit surprising finesse and subtlety. Mastering these languages takes time, but they are invaluable enabling technologies. Utilizing these languages allows programmers to quickly hack up utilities and prototype ideas that would otherwise take five to ten times longer using conventional programming languages. For instance, Kernighan and Pike built the same program in five different languages in *The Practice of Programming* (KP99), with the Perl version being the shortest at 17 lines, compared with 150 lines in C. With a text manipulation language like Perl, a programmer can manipulate text, interact with programs, communicate over networks, drive web pages, perform arbitrary precision arithmetic, and write programs with unconventional syntax. 

To illustrate the wide-ranging applicability of these tools, the authors used Ruby and Python for tasks related to creating the book itself. Specifically, the build system for the Pragmatic Bookshelf is written in Ruby, where authors, editors, layout people, and support staff use Rake tasks to coordinate the building of PDFs and ebooks. The text manipulation process was also used for code inclusion and highlighting: to avoid duplicating code (and thus guaranteeing forgetting to update an example when the corresponding program was changed), a simple Ruby script is invoked when formatting the book. This script extracts a named segment of a source file, performs syntax highlighting, and converts the result into the typesetting language. Furthermore, a website update script extracts the table of contents and uploads it to the book’s webpage, and another script extracts and uploads book sections as samples. To include equations, a Python script converts LaTeX math markup into formatted text. For index generation, since indexes are typically separate documents, a Ruby script was used to collate and format the entries directly within the text itself. In reality, the Pragmatic Bookshelf is built around text manipulation, and following the advice to keep everything in plain text maximizes the benefits of using these languages to manipulate the text.
