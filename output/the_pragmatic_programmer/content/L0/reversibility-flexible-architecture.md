# reversibility-flexible-architecture

**Parent:** [[content/L1/system-design-principles-orthogonality-reversibility|system-design-principles-orthogonality-reversibility]] — The synthesis emphasizes that designing systems using Orthogonality (decoupling components so changes in one do not affect others), Reversibility (avoiding irreversible decisions by abstracting services), and DRY principles requires adopting specific coding practices. Techniques include writing 'shy code,' passing context explicitly rather than using global variables, employing the Strategy pattern for similar functions, and using formal specifications like OpenAPI for external APIs, while proactively preventing knowledge duplication across the development team through structured communication and peer review.

This section discusses Reversibility, a concept that advises against making critical, irreversible decisions when designing software. The text highlights that while engineers and management tend to favor singular, easy answers, the real world is constantly changing, meaning that any fact or design choice can change over time. When project teams become hampered by the myopic notion that only one approach exists, they risk encountering unpleasant surprises, such as being forced to recode an entire project due to a company standardizing on a different database (e.g., forcing the use of PDQ instead of a planned vendor's database).

Critical decisions are often not easily reversible. Committing to a specific vendor's database, architectural pattern, or deployment model can result in a course of action that cannot be undone without significant expense. The core advice is that developers should not let critical decisions solidify, as time passes and the project progresses, committing the team to a narrower version of reality that limits options. The problem is that critical decisions lack reversibility.

To achieve flexibility, developers should adhere to principles like DRY, decoupling, and using external configuration. For example, instead of assuming a relational database from Vendor A, developers should abstract the concept of a database so that it simply provides persistence as a service. This abstraction allows the flexibility to switch technologies midstream. Similarly, a project starting as a browser-based application should plan for a possible later shift to a mobile application by stripping out HTML rendering and replacing it with an API.

The key principle is that the difficulty in making a decision should not lead to the mistake of assuming that any decision is cast in stone. Instead, developers should view decisions as if they are written in the sand at the beach, knowing that a big wave can wipe them out at any time. This emphasis on reversibility leads to the advice of creating a Flexible Architecture by abstracting third-party APIs behind internal abstraction layers and breaking code into components, even if the application eventually deploys on a single, massive server. 

The section concludes with a quantum mechanics analogy involving Schrödinger's cat: if a cat is placed in a closed box with a radioactive particle (with a 50% chance of fissioning), the correct answer, according to Schrödinger, is that the cat is both dead and alive until the box is opened. This analogy suggests that every decision in code evolution results in a different version of the future, prompting developers to consider how many possible futures their code must support and the difficulty of supporting them when the time arrives.
