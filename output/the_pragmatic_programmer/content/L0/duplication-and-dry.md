# duplication-and-dry

**Parent:** [[content/L1/developer-pragmatic-principles-communication-design|developer-pragmatic-principles-communication-design]] — Developers must employ second-order thinking, treating natural language as a programming language and applying principles like DRY and ETC, to ensure that all knowledge, including business rules and APIs, has a single, authoritative source representation across code, documentation, and specifications, while tailoring communication pitches to the specific needs of the target audience.

Analyzing code for duplication reveals that merely having identical function bodies does not constitute a DRY (Don't Repeat Yourself) violation if the functions validate separate concepts. For example, in a wine ordering application, if a programmer writes `def validate_age(value): validate_type(value, :integer); validate_min_integer(value, 0)` and `def validate_quantity(value): validate_type(value, :integer); validate_min_integer(value, 0)`, a code review critique stating that the functions are a DRY violation is incorrect. The functions validate two separate things (age and quantity) that happen to share the same validation rules; thus, this is a coincidence, not knowledge duplication. 

In contrast, duplication of knowledge, such as business rules, can cause significant maintenance overhead. For instance, if a function calculates fees, and the fees are determined by factors such as a $20 charge for each returned check, a $10 charge for each overdraft day exceeding three days, and a 50% reduction if the average account balance is over $2,000, the intended fee structure must be captured both in the documentation comment and the code. If a customer changes a fee, the developer must update both locations, leading to an eventual discrepancy and violating DRY principles. The recommended refactoring simplifies this by extracting the calculation into a function named `calculate_account_fees(account)` where the name describes the function's purpose, and the details are retained within the source code: `fees = 20 * account.returned_check_count; fees += 10 * account.overdraft_days if account.overdraft_days > 3; fees /= 2 if account.average_balance > 2_000;`. This ensures the name reflects the action and the details are in the source code, adhering to DRY. 

Data structures themselves can fall subject to the DRY principle. If a class, such as `Line`, defines a `start` point, an `end` point, and a `length`, the length calculation is inherently dependent on the start and end points. To remedy this duplication, the length attribute should be treated as a calculated field rather than a stored one: `class Line { Point start; Point end; double length() { return start.distanceTo(end); } };`. While developers might later violate DRY for performance, such as caching data to avoid repeated expensive operations, they must localize the impact of this violation, ensuring that only the methods within the class are responsible for maintaining internal consistency, preventing the violation from being exposed outside the module. Furthermore, when any module exposes a data structure, the developer must use accessor functions to read and write the object's attributes (e.g., `void setStart(Point p) { this.start = p; calculateLength(); }` and `double getLength() { return length; }`). This practice, which aligns with Meyer’s Uniform Access principle from *Object-Oriented Software Construction* [Mey97], requires that all services offered by a module are available through a uniform notation, regardless of whether the services are implemented via storage or computation. 

When the code interacts with the outside world, such as other libraries via APIs, remote service calls, or external data sources, the developer frequently encounters DRY violations because the code must replicate knowledge present in the external entity (e.g., needing to know the external API, the schema, or the meaning of error codes). This duplication—where both the developer's code and the external entity must know the representation of their interface—is inevitable but manageable through several strategies. For internal APIs, developers should use tools that specify the API in a neutral format, allowing the generation of documentation, mock APIs, functional tests, and API clients, ideally storing all APIs in a central, shared repository. For external APIs, the use of formal specifications like OpenAPI is beneficial, allowing import into local API tools for reliable integration. If no such specification exists, the developer should create and publish one, which can help others maintain it. When dealing with data sources, many data sources allow developers to introspect the data schema, enabling the generation of container code directly from the schema, minimizing manual code duplication. An alternative, often preferred method is to avoid representing external data in a fixed structure (like a struct or class) and instead store it in a key/value data structure (such as a map, hash, dictionary, or object). This approach requires a second layer of validation: a simple table-driven validation suite that verifies the key/value map contains the minimum required data in the necessary format. 

Perhaps the most difficult type of duplication to detect and manage occurs between different developers on a project. Whole sets of functionality might be unintentionally duplicated across the team, and this duplication can remain undetected for years, leading to severe maintenance issues. Historically, one U.S. state's audit for Y2K compliance revealed over 10,000 programs containing different versions of Social Security Number validation code. To mitigate this, developers must build a strong, collaborative team with effective communication. At the module level, common functionality or data that does not belong to an obvious area of responsibility can be implemented multiple times over. To address this, developers should encourage active and frequent communication, such as running daily scrum standup meetings or setting up forums (like Slack channels) to discuss common problems. Establishing a central location in the source tree for utility routines and scripts and appointing a team member as the project librarian to facilitate knowledge exchange are also valuable strategies. Finally, developers must commit to reviewing other people's source code and documentation, whether informally or during code reviews, recognizing that the process is reciprocal and mutually beneficial.
