# concurrency-and-shared-resources

**Parent:** [[content/L1/professional-mastery-methodologies|professional-mastery-methodologies]] — Professional mastery requires adopting advanced structured methodologies, including PERT (Scenario-Based Estimation) and Iterative Estimation, and mastering tools like the command shell and VCS. Crucially, developers must handle architectural complexities like temporal coupling, understanding that concurrency (software mechanism) differs from parallelism (hardware concern), while strictly managing shared state using mechanisms like mutexes or leveraging structured methods like activity diagrams to ensure robust, concurrent systems.

The problem of managing concurrent access to shared resources often requires mechanisms for exclusive access, which may include library implementations such as mutexes (for mutual exclusion), monitors, or semaphores. While some languages implement concurrency support directly into the language itself, such as Rust, which enforces data ownership—ensuring that only one variable or parameter holds a reference to any particular piece of mutable data at a time—functional languages also aim to simplify concurrency by encouraging the use of immutable data. However, even functional languages face challenges because they must eventually interact with the real, mutable world. Because concurrency in a shared resource environment is inherently difficult, managing it manually presents significant challenges, leading to the recommendation that developers should seek alternative methods to gain the benefits of concurrency without encountering these difficulties. Relevant related topics include Orthogonality, Decoupling, and Programming by Coincidence.
