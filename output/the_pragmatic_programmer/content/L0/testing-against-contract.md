# testing-against-contract

**Parent:** [[content/L1/professional-system-mastery-and-design-patterns|professional-system-mastery-and-design-patterns]] — Professional mastery requires adopting structured methodologies like PERT (Scenario-Based Estimation) for project planning, achieving editor fluency through muscle memory, and utilizing advanced architectural patterns (e.g., PubSub, Actor Model) while implementing rigorous software testing via Design by Contract and property-based testing; all systems must also adhere to defensive security practices such as minimizing attack surface area and applying the Principle of Least Privilege.

Component-based development aims to establish a system where generic software components can be available and combined easily, similar to how common integrated circuits (ICs) are combined. However, this approach relies on using components that are known to be reliable and having standardized interconnect standards, common voltages, and timing. Just as hardware chips are designed to be tested—not only at the factory or upon installation but also in the field—software must integrate testability from its inception. Software developers must thoroughly test every piece before connecting the components. On the hardware side, a full Built-In Self Test (BIST) feature or a Test Access Mechanism (TAM) provides a test harness for external environments to provide stimuli and collect responses from the chip. Software replicates this by using unit testing: exercising a module by establishing an artificial environment, invoking the module's routines, and checking the results against known values or comparing them against results from previous test runs (known as regression testing). Once individual 'software ICs' are assembled into a complete system, unit test facilities can be used for large-scale checking, which is termed Ruthless and Continuous Testing. To properly conduct unit testing, developers must first decide what aspects of the module require testing. Historical practices of merely running random data through the code and checking print statements are inadequate. Instead, testing should focus on checking the module against its documented contract, a concept introduced in Design by Contract. Testing against contract ensures that a given unit honors its promise, revealing both whether the code meets the stated contract and whether the contract itself accurately reflects the desired functionality, especially across a wide range of test cases and boundary conditions. For a numerical example, such as a square root routine, the documented contract defines pre-conditions (argument must be $\ge 0$) and post-conditions (the absolute difference between the result squared and the original argument must be less than $\epsilon \times \text{argument}$). This contract dictates the necessary test cases: passing a negative argument to ensure rejection; passing an argument of zero to confirm acceptance (a boundary value); and passing values between zero and the maximum expressible argument to verify the residual calculation. Assuming the routine performs its own pre- and postcondition checking, basic test scripts can exercise the square root function, including calling `assertWithinEpsilon(my_sqrt(0), 0)`, `assertWithinEpsilon(my_sqrt(2.0), 1.4142135624)`, `assertWithinEpsilon(my_sqrt(64.0), 8.0)`, and `assertWithinEpsilon(my_sqrt(1.0e7), 3162.2776602)`, while also asserting that `my_sqrt(-4.0)` raises an exception. When a module is dependent on several others, such as a module A utilizing a DataFeed and a LinearRegression component, the testing process must proceed by testing subcomponents first: 1) test DataFeed's contract fully; 2) test LinearRegression's contract fully; and 3) test A's contract, which relies on the other two contracts but does not directly expose them. This sequence allows developers to quickly isolate a problem—if A fails, the developer can concentrate the debugging effort on module A or its interaction with the subcomponents, saving time re-examining the subcomponents. Ultimately, thorough testing against contract is necessary to avoid creating a “time bomb”—a failure that remains unnoticed until an inconvenient time later in the project. Regarding testing execution, ad-hoc testing, which involves manually running code, must be formalized; if the code breaks once, the developer must add the test to the existing unit test suite. Furthermore, even the most comprehensive test suite may miss bugs; therefore, developers should test the software after deployment using real-world data. While software lacks physical test pins, developers can gain insight into the internal state using log files containing consistent, regular trace messages, which can be parsed to deduce processing time or logic paths.
