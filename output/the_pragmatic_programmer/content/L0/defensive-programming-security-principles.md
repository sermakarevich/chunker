# defensive-programming-security-principles

**Parent:** [[content/L1/professional-system-mastery-and-design-patterns|professional-system-mastery-and-design-patterns]] — Professional mastery requires adopting structured methodologies like PERT (Scenario-Based Estimation) for project planning, achieving editor fluency through muscle memory, and utilizing advanced architectural patterns (e.g., PubSub, Actor Model) while implementing rigorous software testing via Design by Contract and property-based testing; all systems must also adhere to defensive security practices such as minimizing attack surface area and applying the Principle of Least Privilege.

While developers should treat software development as an organic activity requiring continuous adaptation and monitoring, a core principle is adopting a defensive approach because perfect software does not exist. A professional programmer must proactively adopt defensive coding practices, recognizing that every system failure is a risk. These practices include validating all provided information, using assertions to detect bad data, distrusting data from potential attackers or trolls, checking for consistency, and applying constraints on database columns. 

Furthermore, developers must account for both internal errors and deliberate malicious external attacks. An essential aspect of system design is minimizing the attack surface area, which is the total sum of all access points where an attacker can enter data, extract data, or invoke service execution. Smaller, simpler code is better because complexity increases the attack surface, making the system more porous. Developers must never trust external data; they must always sanitize it before passing it to a database, view rendering, or other processing. Similarly, developers should be cautious with unauthenticated services, as any user in the world can call these services, which creates an opportunity for a denial-of-service attack. It is also vital to keep the number of authorized users at an absolute minimum, and if deployment credentials are compromised, the entire product is compromised. When reporting data, developers must ensure that the data reported is appropriate for the user's authorization level, truncating or obfuscating potentially risky information, such as Social Security or government ID numbers, to prevent revealing private information.

Security requires adherence to basic principles: 1) Minimize Attack Surface Area, 2) Principle of Least Privilege, 3) Secure Defaults, 4) Encrypt Sensitive Data, and 5) Maintain Security Updates.

Implementing the Principle of Least Privilege means using the least amount of privilege for the shortest time necessary. Instead of automatically granting the highest permission level, such as root or Administrator, developers should take only the necessary level, perform the minimum work, and quickly relinquish the permission. For example, on Unix-derived systems, the login program starts with root privileges but drops the high level of privilege to that of the user as soon as authentication is complete. This principle should guide design by partitioning sensitive resources into different categories so that individual users only have permissions for certain categories. 

Following the Secure Defaults principle dictates that the default settings on an application or for a user should be the most secure values, even if they are not the most user-friendly. Finally, developers must encrypt sensitive data, such as personally identifiable information, financial data, passwords, or other credentials, ensuring they are never left in plain text, whether stored in a database or an external file.
