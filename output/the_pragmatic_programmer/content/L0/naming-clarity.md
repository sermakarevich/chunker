When developing software, developers must remember that naming is crucial because names reveal a lot about intent and belief. The proper naming of applications, subsystems, modules, functions, and variables is highly important because names convey the role an element plays within the code. When creating code, developers should pause and consider the motivation for creating a specific element; this practice shifts attention from immediate problem-solving to the bigger picture. Considering the role of a variable or function prompts developers to reflect on what is special about the element and what it interacts with. Often, this deeper reflection reveals that the intended action lacks logical consistency if the name is inappropriate.

The human brain processes written words very quickly, demonstrating that words have a high cognitive priority. This phenomenon, illustrated by the Stroop effect, shows that reading written words is easier than recognizing colors. Developers must ensure that the names used in the code live up to this cognitive priority. For instance, instead of naming a variable `user` (which is functionally meaningless), using names like `customer` or `buyer` provides constant reminders of the person's role, improving clarity. Similarly, when defining an instance method that calculates a discount, the method name and parameter name should clearly convey the intent. For example, instead of `public void deductPercent(double amount)`, a better pattern is `public void applyDiscount(Percentage discount)`. Changing the parameter type from a generic `double` to a defined `Percentage` type explicitly documents the expected range (e.g., between 0 and 100, or 0.0 and 1.0).

Clarity is also achieved by giving functions context-specific names. If a module calculates a Fibonacci number, calling the function `Fib.fib(n)` might be confusing, as it relies on the context of the module. Using a name like `Fib.of(0)` or `Fib.nth(20)` clarifies the function's intent regardless of the calling context.

While writing clear code, developers must distinguish between code clarity and project branding. Projects and teams often use obscure, 'clever' names (e.g., Pokémon or Marvel superheroes). Regarding local programming conventions, some languages require specific names: for instance, in the C language, `i`, `j`, and `k` are traditionally used for loop increment variables, and `s` for a character string. Developers must honor the local culture of the programming language or environment, as applying a convention from one environment (like using `i` for a string variable in Clojure) is incorrect. Furthermore, while language features may accept a subset of Unicode characters, developers should only use such characters if the community expects them.
