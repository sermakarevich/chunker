# property-based-testing-overview

**Parent:** [[content/L1/professional-system-mastery-and-design-patterns|professional-system-mastery-and-design-patterns]] — Professional mastery requires adopting structured methodologies like PERT (Scenario-Based Estimation) for project planning, achieving editor fluency through muscle memory, and utilizing advanced architectural patterns (e.g., PubSub, Actor Model) while implementing rigorous software testing via Design by Contract and property-based testing; all systems must also adhere to defensive security practices such as minimizing attack surface area and applying the Principle of Least Privilege.

Testing is not merely a separate department or staff function; testing is an integral part of programming. When developing software, testing, design, and coding all contribute to the overall programming process. To validate a function, developers should write unit tests by considering typical potential problems based on their knowledge of the function being tested. However, a potential problem arises if a developer writes both the original code and the accompanying tests, as this setup risks expressing an incorrect assumption in both places. If the code passes the tests, it only demonstrates that the code operates according to the developer's understanding, which may be flawed.

While one workaround is for different people to write the tests and the code under test, this is undesirable because, as discussed in Topic 41, the primary benefit of considering tests is how that process informs the code development. Instead, developers should favor an alternative: having the computer, which lacks human preconceptions, perform testing.

This leads to the use of Contracts, Invariants, and Properties. Design by Contract (Topic 23) establishes that code should operate under explicit contracts: the caller meets specific conditions when providing input, and the module guarantees certain outputs. Additionally, code invariants describe properties that must remain true for a piece of state after passing through a function. For example, if a list is sorted, its resulting length must equal the original length.

Developers can use these contracts and invariants—grouping them together and calling them properties—to automate testing through property-based testing. This technique allows developers to validate assumptions automatically. For instance, for a sorted list, one established property is that the result must maintain the same size as the original. Another property is that no element in the result can be greater than the element that follows it.

When using property-based testing, such as with the Hypothesis tool and pytest in Python, the principles are universal. The testing process runs the defined tests multiple times (e.g., 100 times), passing a different, randomly generated list into each test run. This process simulates running numerous individual tests with varying lengths and contents.

Property-based testing requires defining data generation rules using a mini-language based on calls to functions in a strategies module. For example, generating all even numbers between 10 and 20 requires mapping integers between 10 and 20 using a multiplier of 2. Developers can also compose types, such as generating lists of natural numbers that have a maximum length of 100 elements.
