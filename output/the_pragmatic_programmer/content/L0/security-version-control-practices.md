# security-version-control-practices

**Parent:** [[content/L1/developer-mastery-principles|developer-mastery-principles]] — Professional proficiency requires systematic mastery ranging from physical tools (like a woodworker's basic set) to architectural patterns (like PubSub and Actor Model). Developers must enforce strict security hygiene by never checking in credentials (API keys, SSH keys) to VCS, using third-party providers for authentication, and adhering to NIST guidelines (e.g., minimum 64 character password length) while promoting long entropy and timely security updates. Code clarity demands domain-specific naming (e.g., `applyDiscount(Percentage discount)`), respecting local language conventions, and utilizing advanced technical skills like the 'chop' technique for debugging and 'zip' operations for event streams.

When developing software, developers must treat Version Control Systems (VCS) as a project-wide time machine, capable of restoring code to previous states, including documentation, phone number lists, makefiles, build and release procedures, and small shell scripts. Although the VCS is critical for collaboration, developers must remember that the process of handling project source files over a network or using cloud storage is unreliable for collaboration, as this practice frequently leads to teams corrupting each other’s work, losing changes, breaking builds, and causing interpersonal conflicts. Additionally, keeping the main repository on a network or cloud drive is problematic because the VCS uses interacting files and directories, and if two instances simultaneously make changes, the overall state can become corrupted, potentially causing irreparable damage.

Regarding sensitive data, developers must never check in secrets, API keys, SSH keys, encryption passwords, or other credentials alongside the source code in version control. Instead, these keys and secrets must be managed separately, generally using configuration files or environment variables as part of the build and deployment processes.

Concerning application security, good security practices can often run counter to common sense or common practice. For instance, strict password policies actually lower security. Recommended guidelines from the NIST dictate the following regarding passwords:

*   **Length:** Do not restrict password length to less than 64 characters; NIST recommends 256 as a good maximum length.
*   **Truncation:** Do not truncate the user’s chosen password.
*   **Characters:** Do not restrict special characters such as `[]();&%$#` or `/`. The NIST recommends accepting all printable ASCII characters, space, and Unicode.
*   **Hints:** Do not provide password hints to unauthenticated users, or prompt for specific types of information (for example, asking for "what was the name of your first pet?").
*   **Functionality:** Developers should not disable the paste function in the browser, as crippling browser and password manager functionality encourages users to create simpler, shorter passwords that are much easier to compromise. Both the U.S. NIST and the U.K. National Cyber Security Centre specifically require verifiers to allow paste functionality for this reason.
*   **Composition Rules:** Do not impose other composition rules, such as mandating a particular mix of upper and lower case, numerics, or special characters, or prohibiting repeating characters.
*   **Password Changes:** Developers should not arbitrarily require users to change their passwords after a certain length of time; this should only happen for a valid reason, such as following a confirmed breach.

To encourage strong security, developers must promote long, random passwords with a high degree of entropy, as artificial constraints limit entropy and expose user accounts to takeover. Furthermore, developers must prioritize maintaining security updates, because deferring an update until a later time leaves the system vulnerable to a known exploit. This principle applies to every network-connected device, including phones, cars, appliances, personal laptops, developer machines, build machines, production servers, and cloud images. The largest data breaches in history have been caused by systems that were behind on their updates. 

When managing cryptography, developers should follow the rule: never do it themselves. Even simple common practices are wrongheaded. Developers must rely only on reliable resources: well-vetted, thoroughly examined, well-maintained, frequently updated, and preferably open-source libraries and frameworks. For authentication, instead of attempting to build a custom login system using passwords or biometric authentication, developers should follow the pragmatic approach of using a third-party authentication provider, which can be an off-the-shelf service run in-house or a third party in the cloud. Authentication services are often available from email, phone, or social media providers, and relying on these providers is superior because they dedicate all their time to keeping their systems secure.
