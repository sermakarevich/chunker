Professional systems design recommends using three methods—Interfaces and Protocols, Delegation, and Mixins/Traits—as alternatives to traditional class inheritance. Interfaces and protocols, which are also known as traits, allow developers to specify sets of required behaviors for a class, regardless of the language. For instance, a `Car` class might implement `Drivable` and `Locatable` behaviors, where `Drivable` specifies methods like `double getSpeed()` and `void stop()`, and `Locatable` specifies methods like `Coordinate getLocation()` and `boolean locationIsValid()`. These declarations do not generate actual code; they merely mandate that any class implementing the interface must contain all specified methods. This power is amplified because developers can use the interfaces as types; consequently, any class implementing a specific interface is compatible with that type. For example, if both `Car` and `Phone` implement `Locatable`, they can all be stored in a list of `Locatable` items, allowing developers to process the list while being certain that every item has the methods `getLocation()` and `locationIsValid()`. 

Interfaces and protocols provide polymorphism without requiring inheritance. Delegation addresses a common issue with inheritance: if a parent class has many methods (e.g., 20 methods), but the subclass only needs two, the subclass object still carries the remaining 18 methods, causing the class to lose control of its own interface. Instead of using inheritance, developers can implement delegation. For example, an `Account` class can delegate its persistence responsibilities to a `Persister` object by storing a reference to the persister. This approach effectively breaks the coupling to the framework's API, allowing the developers to control the necessary API and creating a clean separation from the framework's constraints.
