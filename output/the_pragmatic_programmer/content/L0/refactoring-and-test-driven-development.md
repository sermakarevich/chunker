# refactoring-and-test-driven-development

**Parent:** [[content/L1/software-architecture-and-refactoring-lifecycle|software-architecture-and-refactoring-lifecycle]] — Software development is an iterative, continuous process (like gardening) that prioritizes refactoring—the disciplined restructuring of code without altering external behavior—triggered by issues like duplication or drifting requirements. Advanced system design requires adopting patterns like Publish/Subscribe and Reactive Programming for decoupling, utilizing the Actor Model or Blackboard system for concurrency, and always preserving accurate records when performing multiple estimations or monitoring program state.

Refactoring, at its core, is the process of redesigning code. Programmers can redesign anything that the programmer or the development team designed using new facts, deeper understandings, or changing requirements. However, simply removing large quantities of code without a plan can create problems worse than the initial state.

Therefore, refactoring must be an activity that development teams undertake slowly, deliberately, and carefully. Martin Fowler offers several simple tips for refactoring safely:
1. Do not attempt to refactor and add new functionality simultaneously.
2. Developers must ensure that good tests exist before starting refactoring; the developer should run these tests as often as possible to quickly identify any broken components.
3. The developer should take short, deliberate steps, such as moving a field from one class to another, splitting a method, or renaming a variable. Refactoring often involves numerous localized changes that combine to result in a larger-scale change, and testing after each step prevents prolonged debugging.

Automatic Refactoring: This capability, once confined to the Smalltalk world, is now available in many IDEs and mainstream languages. These IDEs can assist with renaming variables and methods, splitting extensive routines into smaller ones, automatically propagating required changes, and supporting code movement via drag and drop. However, Martin Fowler's emphasis on maintaining good regression tests remains the key to safe refactoring.

If the intended change goes beyond simple refactoring and requires changing external behavior or interfaces, the developer should deliberately break the build so that old clients fail to compile. This failure signals what needs updating. When the developer encounters a piece of code that is not correct, the developer should fix it. To manage difficult changes: if the change causes pain now but will cause even greater pain later, the developer should proceed. This echoes the lesson from Software Entropy: the development team must not tolerate broken windows.

Testing and Designing for Testability: The major benefits of testing arise when the developer thinks about and writes the tests, rather than merely running them. When a developer is writing new code, thinking about how a test would be written helps guide the process. For example, if a developer needs to write a function querying a database to return a list of people who watch more than 10 videos a week, the process of considering tests guides the developer to crucial architectural improvements: the developer should pass the database instance into the function rather than relying on a global one, and the developer should pass the name of the field being tested as a parameter. Thinking through the test cases allows the developer to view the system from the client's perspective, rather than merely the author's.

Advanced testing techniques include Test-Driven Development (TDD), which follows a short cycle: (1) deciding on a small piece of functionality to add, (2) writing a test that will pass once that functionality is implemented, (3) running all tests and verifying that the only failure is the newly written test, (4) writing the smallest amount of code necessary to make the test pass, and (5) refactoring the code while verifying that the tests still pass. The goal is to execute this cycle within minutes. While TDD is valuable for beginners, developers must avoid becoming slaves to the process, such as by spending undue time ensuring 100% test coverage or writing redundant tests (e.g., writing a failing test merely to reference a class name).

In design, the core principle is that a test is the first user of the code. Testing helps identify functional patterns and forces the developer to understand the code fully, which simplifies the function. By thinking about potential boundary conditions and required error handling before writing the code, the developer can structure the function to simplify the logic.

Bottom-Up vs. Top-Down Design: When designing systems, two schools of design exist: top-down and bottom-up. The top-down approach recommends starting with the overall problem and systematically breaking it into smaller, manageable pieces, which are then expressed in code.
