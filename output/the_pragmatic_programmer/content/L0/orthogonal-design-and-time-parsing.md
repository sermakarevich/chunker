Architecturally, developers should adopt a transformational perspective, favoring explicit APIs, using FSMs or PubSub for decoupling, and utilizing the Actor Model or Blackboard systems for concurrent state management. For example, transferring information between machines requires considering time versus distance, with a 1 TGB tape transfer over a 1 Gbps line taking roughly 9,000 seconds (2.5 hours) and necessitating a distance of almost 9 miles to outperform a 3.5 mph courier.

The concept of orthogonality in class design suggests that a class should focus on a single, specific task, like splitting lines, and should ignore irrelevant details, such as the original source of the lines. This design not only simplifies development but also increases flexibility, allowing a component like Split2 to handle lines read from a file, generated by another routine, or passed in via the environment. When designing in Object-Oriented (OO) languages, developers must be aware that features like multiple inheritance, exceptions, operator overloading, and parent-method-overriding (via subclassing) offer significant opportunities to increase coupling in nonobvious ways. While a class coupling code to data (cohesion) is often beneficial, poorly focused classes can still result in ugly interfaces. In contrast, functional languages encourage writing numerous small, decoupled functions that can be combined in various ways to solve problems. Although this approach works well in theory and often in practice, a specific type of coupling can occur: when functions transform data, the output of one function may become the required input for the next. If developers are not careful, changing the data format of a function can cause failures anywhere in the subsequent transformational stream. To enhance system design, developers should also consider non-technical methods. For instance, simulating a process by drawing cartoons of a car, a phone, and a house on a whiteboard, and placing Post-it notes describing the content on the clickable areas, allows for refinement of the drawings and placement of Post-it notes as the meeting progresses. Furthermore, to ensure a language is extendable, developers can make a parser table driven, where each entry contains a command letter, a flag indicating if an argument is required, and the name of the routine to call to handle that command. For example, a system could define commands like 'P' (with argument required, calling `doSelectPen`), 'U' (no argument, calling `doPenUp`), 'D' (no argument, calling `doPenDown`), 'N' (argument required, calling `doPenDir`), 'E' (argument required, calling `doPenDir`), 'S' (argument required, calling `doPenDir`), and 'W' (argument required, calling `doPenDir`). The main program logic involves reading a line, finding the command, retrieving the argument if needed, and calling the handler function. The function that looks up a command performs a linear search of the table, returning either the matching entry or `NULL` if the command is unknown. Finally, reading the numeric argument can be accomplished using `sscanf`. Regarding language specification, a time specification can be defined using Backus-Naur Form (BNF) rules: `time ::= hour ampm | hour : minute ampm | hour : minute`.
