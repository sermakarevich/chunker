# breaking-temporal-coupling

**Parent:** [[content/L1/professional-mastery-methodologies|professional-mastery-methodologies]] — Professional mastery requires adopting advanced structured methodologies, including PERT (Scenario-Based Estimation) and Iterative Estimation, and mastering tools like the command shell and VCS. Crucially, developers must handle architectural complexities like temporal coupling, understanding that concurrency (software mechanism) differs from parallelism (hardware concern), while strictly managing shared state using mechanisms like mutexes or leveraging structured methods like activity diagrams to ensure robust, concurrent systems.

The concept of temporal coupling refers to a design flaw in software architecture where the code imposes a mandatory sequence on events that is not fundamentally required to solve the problem. Developers often approach programming with a linear mindset, assuming that one must execute a method before another, or that actions must occur in a strict order (e.g., 'Tick must happen before Tock'). This linear approach, while natural, leads to temporal coupling, which severely limits flexibility and realism. To build resilient systems, developers must design architectures that allow for concurrency and that decouple time and order dependencies. Achieving this reduces time-based dependencies in various areas, including workflow analysis, architecture, design, and deployment, resulting in systems that are easier to reason about and potentially respond faster and more reliably.

To analyze potential concurrency in a system's workflow, an activity diagram, which consists of rounded boxes representing actions, is an effective tool. Arrows leaving an action indicate that a subsequent action can start once the first action completes, while a thick line called a synchronization bar indicates that all actions leading into the bar must complete before proceeding along any outgoing arrows. Any action with no incoming arrows can be started at any time. Activity diagrams can be used to identify activities that could be performed in parallel but are not currently designed that way.

For example, while writing a robotic piña colada maker's steps sequentially (1. Open blender; 2. Open piña colada mix; 3. Put mix in blender; 4. Measure 1/2 cup white rum; 5. Get pink umbrellas; 6. Add 2 cups of ice; 7. Serve), a bartender would lose their job following those steps strictly in order. However, the actions can be performed in parallel. Analyzing the workflow reveals that top-level tasks (1, 2, 4, 10, and 11) can all happen concurrently up front, and tasks 3, 5, and 6 can happen in parallel later. Identifying these dependencies is crucial for optimization.

In the context of technical writing and content generation, a book is built by feeding plain text through a pipeline of processors. Some processors look for specific constructs (like bibliography citations, index entries, or special markup for tips), while others operate on the document as a whole. Since many processors must access external information (such as reading or writing files, or piping through external programs), the slow processing time allows for exploiting concurrency: in fact, each step in the pipeline executes concurrently, reading from the previous step and writing to the next. Similarly, for processor-intensive tasks like converting mathematical formulae, which can take up to 500ms per equation due to historical reasons, the process can utilize parallelism. Since each formula conversion is independent of the others, developers convert each formula in its own parallel process and collect the results back into the book as they become available. This approach significantly speeds up book construction on multicore machines.

Activity diagrams only suggest potential areas of concurrency; they do not determine if exploiting these areas is worthwhile. For example, the piña colada example shows that running all potential initial tasks concurrently would require a bartender with five hands. The design process must guide the identification of activities that take time, but not time within the code itself. Opportunities for concurrent work include querying a database, accessing an external service, or waiting for user input, as these activities normally would stall a program. Utilizing these waiting periods for other productive tasks represents a major opportunity for concurrency.

It is vital to distinguish between concurrency and parallelism: concurrency is a software mechanism, whereas parallelism is a hardware concern. If multiple processors are available, either locally or remotely, developers can reduce the overall time by splitting work into relatively independent chunks and processing each chunk in parallel, followed by combining the results. A practical example is the Elixir compiler, which starts by splitting the project into modules and compiling each in parallel. If one module depends on another, its compilation pauses until the dependent module's build results are available. Upon completion of the top-level module, all dependencies have been compiled, resulting in a speedy compilation that utilizes all available cores.

When applying these concepts to developing applications, developers must first identify places where concurrency and parallelism can benefit the system, and then tackle the challenging process of implementing them safely.
