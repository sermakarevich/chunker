# resource-management-finish-what-you-start

**Parent:** [[content/L1/professional-software-engineering-practices|professional-software-engineering-practices]] — The synthesis details professional software practices, covering Design by Contract (DBC) for guaranteeing module correctness through defined preconditions, postconditions, and invariants; mandatory resource safety using the 'Finish What You Start' principle (e.g., using scoped blocks for file handling); advanced debugging techniques like the 'chop' binary search for isolating errors; and operational proficiency in the command shell and editors, which requires achieving 'editor fluency' to minimize the distance between thought and execution.

To manage resources, developers frequently handle items with limited availability, including memory, transactions, threads, network connections, and files. Most typically, resource usage follows a predictable pattern: the developer allocates the resource, utilizes it, and subsequently deallocates it. However, many developers lack a consistent plan for resource allocation and deallocation. Therefore, the following tip, 'Finish What You Start,' suggests that the function or object responsible for allocating a resource must also be responsible for deallocating it. 

Consider a flawed example written in Ruby: the `read_customer` function opens a file and reads customer information, while `write_customer` rewinds and writes the result back to the file. The `update_customer` function orchestrates this process. Although `update_customer` appears clean—it reads a record, updates the balance, and writes the record back—the underlying problem is that `read_customer` and `write_customer` are tightly coupled because they share the instance variable `@customer_file`. The `read_customer` function stores the file reference in `@customer_file`, and `write_customer` uses that stored reference to close the file. Crucially, this shared variable does not appear in the `update_customer` routine itself.

This sharing mechanism creates maintenance difficulties. If a programmer receives a specification change—for example, requiring that the balance is updated only if the new value is not negative—the programmer modifies `update_customer` using the shared variable. During testing, the code seems fine, but in production, the code eventually collapses after several hours due to too many open files. This occurs because `write_customer` is not reliably called under all circumstances, preventing the file from being closed.

Attempting to fix this by adding a special case to `update_customer` (e.g., `else: @customer_file.close`) forces the shared variable `@customer_file` to be managed manually. This approach solves the immediate problem but results in three routines (`read_customer`, `write_customer`, and `update_customer`) being coupled through `@customer_file`. Tracking the file's open/close status becomes extremely difficult, leading to a state of imbalance.

Applying the 'Finish What You Start' principle involves refactoring the code to pass the file object as a parameter, eliminating the reliance on the shared instance variable. The refactored structure first opens the file in `update_customer`, passing the open file reference to both `read_customer` and `write_customer`. Finally, `update_customer` closes the file after completing its tasks. This centralizes the resource management, ensuring the open and close operations occur in the same location. Furthermore, modern languages, such as Ruby, allow scoping the resource's lifetime to an enclosed block (using `File.open(...) do |file| ... end`). When using this block construct, the file variable automatically goes out of scope and is guaranteed to be closed, thereby eliminating the need for explicit manual closing statements and ensuring resource safety.
