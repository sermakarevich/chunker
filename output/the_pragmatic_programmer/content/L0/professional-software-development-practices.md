# professional-software-development-practices

**Parent:** [[index]]

When developing software, programmers must view the process not as a static construction effort, but as an organic activity requiring continuous careful thought, judgment, and monitoring. Programmers should critically assess all code, including their own, because constant room for improvement exists in both the code and the design. Testing is valuable not only for finding bugs; testing also gathers feedback on the code, including aspects of the design, the API, and coupling. The primary benefits of testing arise when a developer considers and writes the tests, rather than simply running them. Furthermore, programmers must be aware of potential semantic drift while coding, acknowledging that the names chosen can define the reality of the system. The core principle of writing professional software is maintaining situational awareness, similar to a safe driver who constantly reviews the situation and prepares for the unexpected. When a developer encounters design challenges or writer's block, the developer should attempt prototyping. Prototyping involves looking for aspects of the project to explore—for example, investigating a new framework's data binding or a novel algorithm's edge cases—and the developer should follow this process by writing 'I’m prototyping' on a sticky note and dedicating time to the exploration. 

Programmers must remember that the understanding of a system is paramount. When working with existing code, especially code written by other people, a developer should not mechanically read the code and make notes on seemingly important details. Instead, the developer should conduct an experiment. When the developer spots something that seems strange, the developer should jot it down and continue this process to look for underlying patterns. By analyzing these patterns, the developer may find that the job of understanding the code becomes significantly easier, allowing the developer to consciously apply the patterns that the original author applied tacitly. Furthermore, this process might allow the developer to learn something entirely new. 

Beyond just the code, developers should also listen to the developer's intuition (gut) when coding, recognizing that this skill applies to the overall system design as well. If a design feels wrong, or if a requirement creates unease, the developer should stop and analyze those feelings. If the developer is in a supportive environment, the developer should express these feelings out loud and explore them, as the chances are that a critical issue is lurking nearby. The developer must listen to intuition to avoid problems before they manifest. 

Regarding the development process, developers must also acknowledge the challenge of working in 'minefields'—environments filled with potential traps. Remembering the story of a weary soldier probing a seemingly safe field for mines demonstrates the danger of drawing false conclusions based on lucky or circumstantial evidence. Developers must avoid 'programming by coincidence,' which relies on luck and accidental successes, and instead adopt a deliberate approach. 

Accidents of implementation occur when developers rely on undocumented error or boundary conditions resulting from how the code is written. For example, if a developer calls a routine with bad data, the routine responds in a particular way, and the developer codes based on that response, even if the original author did not intend for the routine to function that way. If the routine is later 'fixed,' the developer's code may break. Calling components in the wrong sequence or in the wrong context is also a related problem. 

When writing code that other people will call, developers must rely only on documented behavior and must use the basic principles of good modularization and hiding the implementation details behind small, well-documented interfaces. If documented behavior is unavailable, the developer must explicitly document the assumption being made. Furthermore, the developer must recognize that developers are prone to seeing patterns and causes even when these are merely coincidences, such as in finance or gaming. An intermittent error visible every 1,000 requests in a log file might be a race condition or a simple bug. Any perceived pattern must be proven through testing, not assumed. If a project involves temporal inconsistencies, such as logging data from hardware collection units across various time zones, developers must use a proper model of time handling instead of compensating for flaws with repeated additions or subtractions (for example, using '+1' or '-1').
