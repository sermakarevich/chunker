# advanced-event-handling-patterns

**Parent:** [[content/L1/professional-mastery-and-system-design|professional-mastery-and-system-design]] — Professional competency requires mastery of multifaceted tools, from physical implements and sophisticated editor proficiencies to advanced architectural patterns like PubSub and Event Streams; techniques range from PERT and Iterative Estimation to using 'chop' for debugging and enforcing API wrappers for resources to prevent global state coupling.

Although State Machines are a topic that developers sometimes underutilize, the concepts discussed here address problems that go beyond state handling. The Observer Pattern describes a mechanism involving a source of events, called the observable, and a list of interested clients, or observers. An observer registers its interest with the observable, typically by passing a reference to a function that the observable will call later. When an event occurs, the observable iterates through its list of registered observers and calls the specific function for each one, passing the event as a parameter. For example, a `Terminator` module in Ruby can register multiple callbacks to be executed before terminating the application; each registered callback can use the notification to tidy up temporary resources, commit data, or perform other actions. The process of creating an observable is simple: the developer pushes a function reference onto a list and subsequently calls those functions when an event occurs. The Observer/Observable pattern has been widely used, especially in user interface systems, where callbacks inform the application of user interactions. However, the Observer Pattern has two problems: because each observer must register with the observable, the process introduces coupling. Furthermore, because typical implementations handle callbacks synchronously and inline within the observable, this design can introduce performance bottlenecks. Publish/Subscribe (PubSub) generalizes the Observer Pattern while solving the coupling and performance issues. In the PubSub model, publishers and subscribers communicate via channels, which are implemented by a separate body of code—this detail is hidden from the application code. Every channel has a name. Subscribers register interest in one or more named channels, and publishers write events to these channels. Unlike the Observer Pattern, the communication between the publisher and subscriber is handled outside the application code, making it potentially asynchronous. Because cloud service providers often offer PubSub offerings, and every popular language typically has at least one PubSub library, developers should use this technology. PubSub is effective for decoupling the handling of asynchronous events, allowing code to be added and replaced, potentially while the application is running, without altering existing code. The drawback of PubSub is that it can be difficult to visualize the system's flow, as a developer cannot look at a publisher and immediately determine which subscribers are involved with a specific message. Compared to the Observer Pattern, PubSub provides a significant reduction in coupling by abstracting through a shared interface (the channel); however, it is fundamentally just a message passing system. To handle complex systems that respond to combinations of events, developers need mechanisms that add a time dimension to event processing. Reactive Programming, which is familiar to those who have used a spreadsheet, involves values reacting when the cells they reference change. Frameworks like React and Vue.js facilitate this data-level reactivity in web browsers. Streams provide a way to treat events as a collection of data. This allows developers to manipulate, combine, filter, and perform all standard collection operations on the event data. Event streams can also be asynchronous, meaning the code receives the opportunity to respond to events as they arrive. The current standard for reactive event handling is defined on the site http://reactivex.io, which documents a language-agnostic set of principles. An example demonstrates how to zip two streams together: one stream containing a list of five animal names and a second stream acting as an interval timer generating an event every 500ms. The resulting combined stream only emits a value when data is available on both inputs, thus emitting a value every half second. A second example illustrates that event streams can fetch data asynchronously from a remote site, such as `https://reqres.in`, by performing a GET request to `users/«id»`. By using `mergeMap` in the RxJS library, the system processes three separate requests for user IDs 3, 2, and 1 in parallel. This processing is highly effective because the first request, for id 2, took 82ms, followed by requests for id 3 and id 1, which returned 50ms and 51ms later, respectively. Event streams are valuable because they unify synchronous and asynchronous processing through a common API. Event streams are pervasive, governing events like button clicks, timer expirations, or file lines matching a pattern. Any code crafted around events tends to be more responsive and better decoupled than linear counterparts. This concept is related to Topic 28, Decoupling, and Topic 36, Blackboards.
