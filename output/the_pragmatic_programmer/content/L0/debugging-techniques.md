# debugging-techniques

**Parent:** [[content/L1/developer-productivity-and-workflow|developer-productivity-and-workflow]] — Software development requires mastery of editor fluency for text manipulation, utilizing VCS for comprehensive project management, and employing systematic debugging techniques like 'chop' and tracing statements to identify bugs efficiently.

When encountering a bug, developers can utilize the 'chop' technique, which is conceptually similar to performing a binary search on a sorted array. The core idea is to efficiently locate the source of an error within a large system. First, if a programmer is dealing with a massive stack trace and needs to pinpoint the function that mangled a value, the developer should select a stack frame approximately in the middle of the trace. Determining if the error is present in that middle frame allows the developer to narrow the focus to the frames preceding it or the frames following it. Repeating this 'chop' process—selectively analyzing the middle of the remaining stack trace—can pinpoint the origin of the bug in a maximum of six attempts, even with 64 frames.

Developers can apply a similar systematic process when investigating bugs that only appear on specific datasets. A developer should split the entire dataset into two parts and check if the problem recurs when feeding one half or the other through the application. This process of repeatedly dividing the data set must continue until a minimal set of values that consistently exhibit the problem is identified.

Furthermore, if a team introduced a bug across a series of releases, a developer can use this methodical approach by creating a test that causes the current release to fail. The developer should then choose a release version exactly halfway between the current failure point and the last known working version, running the test to determine which half contains the problematic change. Successfully implementing this technique is one of the key advantages of utilizing robust version control systems; some advanced version control systems can automate this search by selecting potential releases based on the test results.

Beyond the 'chop' method, specialized techniques enhance debugging ability. While debuggers typically focus on a program's current state, sometimes a developer needs to observe the state of a program or a data structure over time. Tracing statements, which are simple diagnostic messages printed to the screen or a file (e.g., "got here" or "value of x = 2"), represent a primitive technique compared to modern IDE-style debuggers but remain highly effective for diagnosing certain classes of errors that standard debuggers cannot detect. Tracing is especially invaluable in time-sensitive systems, such as concurrent processes, real-time systems, and event-based applications. Developers can use tracing statements to descend into the code by adding them sequentially down the call tree; for example, tracking a resource leak, such as unbalanced file opens and closes, involves tracing each open and each close event into a log file. The developer can then process this log file using text processing tools or shell commands to easily locate the initial offending open.

For simple, yet highly effective problem diagnosis, a developer should employ the technique of "Rubber Ducking." This involves explaining the problem step-by-step to another person, who should merely listen or follow along. The simple act of verbalizing assumptions and explaining the code's intended function often causes the root cause of the problem to become visible. If a person is unavailable, using a rubber duck, teddy bear, or potted plant can serve the same purpose.

Debugging code often requires a process of elimination, as the code base may mix application code (written by the developer and others), third-party components (such as databases, web frameworks, or specialized algorithms), and the underlying platform environment (operating system, libraries, and compilers). When investigating a bug, the developer should generally assume that the bug resides within the application code under development, rather than assuming the problem is with a third-party library, even if the issue may ultimately be external. If a problem does exist with a third party, the developer must still first eliminate their own application code before filing the bug report.

When a system fails after a minor change, the developer must assume the newly changed element is responsible, directly or indirectly, regardless of how implausible the cause seems. If an overall system failure is detected, the developer must systematically check all boundary conditions, especially those related to the assumed-safe components. When encountering a 'surprising' bug, the developer must accept that one or more of the assumptions guiding the initial investigation is incorrect. Instead of simply fixing the issue, the developer must determine why the failure was not caught earlier by amending the unit or other tests to prevent recurrence. The developer must also check for other potential locations in the code that might be susceptible to the same bug and fix them proactively, and finally, if the bug resulted from one person's incorrect assumption, the developer should discuss the problem with the entire team to prevent others from sharing the same misunderstanding.
