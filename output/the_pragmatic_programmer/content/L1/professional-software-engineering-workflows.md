# professional-software-engineering-workflows

**Parent:** [[content/L2/professional-mastery-workflows|professional-mastery-workflows]] — Professional mastery demands systematic skill building across physical, operational, and digital domains, ranging from foundational tools (like a woodworker's rules, gauges, saws, planes, and chisels) to advanced software practices such as utilizing Program Evaluation Review Technique (PERT) for estimating project scope, and achieving 'editor fluency' in text manipulation. Key digital competencies include mastering the command shell (WYSIAYG vs. WYSIWYG), deploying Version Control Systems (VCS) with branching and CI/CD pipelines, applying defensive coding principles like Design by Contract (DBC), and architecting systems using decoupled patterns like Publish/Subscribe (PubSub) and Reactive Programming.

The professional mastery of technical disciplines, whether in manual trades, advanced software development, or technical writing, necessitates a systematic, multi-layered approach to developing specialized expertise. This journey begins with the physical toolkit, requiring foundational items—such as a woodworker’s initial selection of rules, gauges, saws, planes, and chisels. These tools must be chosen for their longevity, distinct function with minimal overlap, and ergonomic suitability. Critical to this phase is developing meticulous care for the tools, recognizing that each possesses unique quirks, requiring specialized handling like specific sharpening or specialized holding angles. Over time, the utility of the original basic toolset remains high, suggesting that improving both tool quality and the depth of knowledge regarding optimal use directly enhances professional productivity. The relationship eventually transcends mere equipment, becoming a profound extension of the user's mind that translates abstract thoughts into tangible physical results. While advanced specialized tools, such as laser-guided miter saws or dovetail jigs, may eventually be acquired, the fundamental mastery of the basics is paramount.

Building upon this resource management principle, professionals must systematically expand their 'toolkit' to encompass operational workflows, particularly in project scheduling and programming. Due to inherent uncertainty, setting project duration requires structured methodologies. Two core techniques are employed: Scenario-Based Estimation and Iterative Estimation. Scenario-Based Estimation, formalized as Program Evaluation Review Technique (PERT) and drawing inspiration from the U.S. Navy's planning for the Polaris submarine project, dictates that professionals must rarely provide a single definitive number. Instead, a spectrum of possibilities must be offered, covering diverse operational scenarios. For any given task, three specific estimates are required: an optimistic figure, a most likely figure, and a pessimistic figure. By arranging these tasks into a dependency network and applying fundamental statistics, one can calculate the statistically likely best and worst possible project timelines. This use of ranges significantly mitigates estimation errors that arise from simply padding the timeline out of general uncertainty, thereby substantially improving the overall project estimate. 

In contrast, Iterative Estimation, or 'Eating the Elephant,' recognizes that the most reliable timeline is derived from acquiring direct, practical experience *on* the project itself. This process is achieved through incremental development, which mandates the repeated completion of a detailed cycle using very small, isolated slices of functionality. Each cycle must systematically include checking requirements, performing risk analysis (with mandatory priority given to the riskiest items early), designing, implementing, integrating, and validating the results with the end-users. Initially, while a general idea of the total number of iterations or the approximate duration of each might exist, basing the initial guess on this information is unreliable for all projects except the most trivial. Unless the project is an exact mimicry of a previous endeavor using the identical team and technology stack, guesswork is inherently flawed. Therefore, the professional must first complete the coding and testing for the initial functionality, formally concluding the first iteration. The accrued practical experience then enables the informed refinement of the initial guess regarding both the total number of required iterations and the specific scope of work for every subsequent one. This refinement improves with every cycle, and the professional's confidence in the final schedule grows directly in correlation with the refinement. Management must understand that the team's ultimate productivity, environment, and effort will determine the final schedule. Consequently, the professional must formally refine the schedule as an integral part of every single iteration, and the most accurate advice for communicating the timeline to stakeholders is to state, “I’ll get back to you.” Furthermore, for all estimation tasks, adequate time must be allocated to slow down and carefully work through the descriptive steps; estimates delivered under duress (such as at a coffee machine) are frequently inaccurate. A detailed logging system is mandatory, requiring the rigorous tracking of the accuracy of every prediction. If the observed error exceeds 50%, a thorough investigation into the specific root causes of the inaccuracy is required. These root causes might stem from selecting parameters that do not accurately reflect the problem's reality or relying on an incorrect underlying model. Uncovering the precise root cause is crucial, as it leads directly to significantly better subsequent estimations. When an overall estimate involves multiple subestimates, a separate, meticulous record of these subestimates must be maintained to observe the degree of accuracy in the predictions.

These professional methodologies extend deeply into digital programming workflows, where the command shell serves as the primary workbench for text-based professionals. While Graphical User Interface (GUI) or Integrated Development Environments (IDEs) accelerate routine tasks like file movement or project deployment, relying solely on them drastically limits capability, particularly restricting the automation of common tasks or the combination of disparate tools into customized macros. A critical operational distinction exists: GUI environments typically adhere to WYSIWYG (what you see is what you get), whereas the command shell provides WYSIAYG (what you see is all you get). Furthermore, GUI environments are generally confined to functional boundaries set by their designers; extending the model often encounters significant limitations unless the designer has provided explicit integration hooks.

To maximize efficiency, developers must achieve deep familiarity with the command shell. Mastery involves running utilities and combining them using pipes, which allows for the construction of complex macro commands. Essential shell configurations include setting color themes and configuring a personalized prompt (which can display real-time data like version control status, time, or a shortened directory name). Furthermore, the utility of aliases (e.g., `alias apt-up='sudo apt-get update && sudo apt-get upgrade'`) and functions is paramount. Automation can be extended through advanced completion mechanisms, allowing the shell to offer context-specific suggestions based on the current working directory. Developing shell proficiency requires treating the shell as a personalized, optimized workspace. 

Core programming proficiency centers on 'editor fluency' in text manipulation, representing the capacity to operate multiple powerful editors with maximum effortlessness, minimizing the mental distance between an idea and its physical manifestation. Fluency involves more than just knowing commands; it includes the ability to move and select content precisely by character, word, line, and paragraph. It also demands navigating by various syntactic units (e.g., matching delimiters, functions, or modules) and performing code refactoring tasks like reindenting or commenting/uncommenting entire blocks using a single command. Complex features to master include managing undo/redo history, splitting the editor into multiple panels, and executing advanced searches using regular expressions. Advanced capabilities involve temporarily creating multiple cursors based on a selection or pattern match, enabling parallel text edits, and integrating project error display and test execution within the editor. 

Achieving fluency is a continuous, targeted improvement process. Instead of learning every available command, the professional must identify repetitive actions and critically ask, 'there must be a better way.' This builds muscle memory through sustained, ideally multiple daily, repetition. Many powerful editors utilize a basic core expanded by extensions. When facing a limitation, the professional's first action must be searching for an existing extension; if none exists, the professional must learn the editor's extension language to automate the task, potentially writing only one or two lines of code, which can scale to publishing a full-blown extension for the wider community. Targeted self-improvement challenges include: 1) Disabling autorepeat to build key sequences for selection and movement by characters, words, lines, and blocks. 2) Restricting all editing for one full week exclusively to keyboard input, eliminating reliance on a mouse. 3) Creating a compilation list of desired features to guide the search for plugins. 4) If no plugin or extension exists, building the functionality from scratch.

Integral to development is the use of Version Control Systems (VCS), which are indispensable for maintaining the integrity of source code and all digital artifacts, including documentation, memos, shell scripts, and build procedures. VCS functions as a project-wide time machine, greatly surpassing the utility of a simple 'undo.' Its use is paramount for collaboration, as simple file sharing or cloud storage risks team members overwriting work, leading to data loss and conflicting builds. Furthermore, placing a main repository on a network or cloud drive is dangerous because multiple simultaneous changes risk corrupting the entire state. VCS mitigates this by tracking every change, offering not only basic mistake reversal but also valuable data for auditing, bug tracking, and performance analysis. Users can precisely track metrics such as who altered a specific line, the difference between current and previous weeks' versions, the total quantity of lines changed during a release, and which specific files are modified most frequently. 

One of VCS's most powerful features is **branching**. Developers can create a separate branch at any historical point, ensuring that all work executed there remains fully isolated from all other development lines. This is vital for team structures, allowing one developer to work on Feature A on a dedicated branch while a teammate simultaneously works on Feature B in another, eliminating any chance of interference. While many VCS are decentralized, establishing a central repository is recommended for maximizing integration benefits. This central hosting environment should ideally provide robust features: excellent security and access control, an intuitive UI, CLI capability for comprehensive automation, automated builds and tests, robust branch merging support (e.g., pull requests), integrated issue management for metrics tracking, and good reporting tools like Kanban boards. Many advanced teams configure their VCS such that a simple push to a designated branch automatically triggers the system build, runs associated tests, and, if all checks pass, deploys the new code to production, thereby ensuring reliability. Such advanced systems necessitate proactive learning of recovery skills, including documenting all necessary system configurations and installed applications in text files, which must be hosted in a VCS separate from the primary laptop. Developers should implement advanced workflows such as feature branches, pull/merge requests, continuous integration (CI), continuous build pipelines, and continuous deployment (CD).

Beyond text editing, specialized language tools, such as Python or Ruby, are critical for general-purpose text manipulation, akin to routers in woodworking. These languages are powerful and versatile, though described as noisy, messy, and somewhat brute force; mistakes with them can ruin entire pieces. However, when correctly applied, they exhibit surprising finesse and subtlety. Mastering these languages, though requiring time, yields invaluable enabling technologies, allowing programmers to quickly hack up utilities and prototype ideas that would otherwise take five to ten times longer using conventional programming languages. For example, Kernighan and Pike demonstrated this in *The Practice of Programming* (KP99), where they built the same program in five different languages, with the Perl version being the shortest at 17 lines compared to 150 lines in C. A text manipulation language like Perl allows a programmer to manipulate text, interact with programs, communicate over networks, drive web pages, perform arbitrary precision arithmetic, and write programs using unconventional syntax. The authors used Ruby and Python for tasks related to creating the Pragmatic Bookshelf itself. The build system for this bookshelf is written in Ruby, utilizing Rake tasks for coordinating the building of PDFs and ebooks. The text manipulation process was used for code inclusion and highlighting: a simple Ruby script is invoked when formatting the book to extract a named segment of a source file, perform syntax highlighting, and convert the result into the typesetting language. Furthermore, a website update script extracts and uploads the table of contents, and a separate script extracts and uploads book sections as samples. For equation inclusion, a Python script converts LaTeX math markup into formatted text. Index generation was handled by a Ruby script to collate and format entries directly within the text itself, illustrating that the entire bookshelf structure relies on text manipulation, and keeping everything in plain text maximizes these benefits. 

Complementary to development workflows is the maintenance of the 'Engineering Daybook,' a physical journal concept utilized by early electronics and mechanical engineers. This daybook records activities, learned concepts, ideas sketched in the margins, readings from meters, and any work-related observation. Developers can adapt this practice to document meeting notes, current project status, recorded variable values during debugging, reminders, wild ideas, and general observations. The physical nature of writing is posited to offer three primary benefits: 1) It is more reliable than human memory, allowing recall of specific details (e.g., the name and phone number of a company regarding a power supply problem). 2) It provides a place to store ideas not immediately relevant, allowing the developer to concentrate on the main task while preserving great ideas. 3) It acts as a kind of 'rubber duck,' assisting the brain by providing a necessary pause for reflection, which can sometimes lead to the realization that the topic being noted is incorrect. The text advocates maintaining this daybook using paper, rather than a digital file or wiki, due to the fundamental difference in the act of writing versus typing. Beyond record-keeping, the philosophy of pragmatic programming emphasizes writing defensively, as perfect software does not exist. Accepting this reality is crucial for preventing wasted energy pursuing an impossible dream. Defensive coding practices mandate validating all provided information, using assertions to detect bad data, distrusting data from potential attackers or trolls, checking for consistency, and applying constraints on database columns. Specific defensive measures include using Design by Contract (DBC), applying Dead Programs Tell No Lies, using Assertive Programming to actively verify assumptions while the code runs, employing resources balancing techniques when dealing with system resources like memory or files, and always proceeding in small steps. 

Design by Contract (DBC) is a powerful technique developed by Bertrand Meyer in *Object-Oriented Software Construction* for the Eiffel language. Its focus is explicitly documenting and agreeing upon the rights and responsibilities of software modules to guarantee program correctness. A correct program is defined as one that performs no more and no less than it claims to do. DBC requires every function and method to define its expected state before execution and its guaranteed state upon conclusion, formalized into three categories: 1. **Preconditions:** These define what must be true before a routine can be called; the calling party is responsible for meeting these explicit requirements. 2. **Postconditions:** These specify what the routine guarantees to achieve—the state of the world upon its completion, mandating that the routine must conclude (disallowing infinite loops). 3. **Class invariants:** These ensure that a class maintains a condition that is always true from any caller's perspective, which must hold true upon the routine's exit and return to the caller. If either party fails to uphold the terms, a predefined remedy is invoked (e.g., raising an exception). DBC requires developers to be strict about what they accept and to promise as little as possible in return. This concept applies across functional, object-oriented, or procedural programming languages. While it is distinct from Test-Driven Development (TDD) or property-based testing, DBC provides distinct advantages because it does not require any setup or mocking, and it explicitly defines parameters for success or failure for all possible cases, unlike testing which can only target one specific instance. For instance, when implementing a deposit function in a bank account, two preconditions (deposit amount > 0.00, account must be open and valid) and a postcondition (new transaction can be found among account's transactions) must be defined; failing to meet these triggers a runtime exception like a `java.lang.AssertionError`.
