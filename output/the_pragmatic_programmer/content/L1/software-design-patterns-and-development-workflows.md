# software-design-patterns-and-development-workflows

**Parent:** [[content/L2/systematic-professional-mastery-workflow|systematic-professional-mastery-workflow]] — Professional proficiency demands mastery across physical tool use (gauges, saws, chisels), advanced workflows (PERT, Iterative Estimation, requiring logging and systematic refinement), and complex digital architecture. Key digital skills include mastering the command shell (WYSIAYG vs. WYSIWYG), achieving editor fluency through sustained repetition, and leveraging advanced systems like VCS (branching, CI/CD) and reactive programming (PubSub, Event Streams, Actor Model) to manage coupling and asynchronous complexity. Debugging requires systematic techniques such as the 'chop' binary search and using diagnostic 'tracing statements'.

The evolution of software design principles emphasizes moving beyond reliance on traditional class inheritance, which introduces structural vulnerabilities and coupling issues. While foundational concepts, such as those observed in Simula 67 (1969) and later used in Smalltalk, initially provided frameworks for incorporating common functionality—allowing developers to save keystrokes or express a 'kind-of' relationship between classes—these methods present significant problems. Inheritance introduces coupling not only between the child and its immediate parent, but also across the entire ancestry chain. This coupling means that a top-level function operating on a child class (e.g., calling `my_ride.move_at(30)`) can fail if an ancestor class (like `Vehicle`) changes its API—for example, renaming `move_at` to `set_velocity` or changing an instance variable from `@speed` to `@velocity`—even if the calling code only interacts with the `Car` class. Furthermore, defining types solely through deep class hierarchies is problematic, as these structures can become complex and fragile. The difficulty of multiple inheritance, which was a major hurdle for C++ in the 1990s, led many modern object-oriented languages to omit this feature entirely.

To mitigate these architectural risks, professional system design strongly recommends three alternative techniques: Interfaces and Protocols, Delegation, and Mixins/Traits. Interfaces and protocols, which are also known as traits, define mandatory sets of behaviors for a class without generating actual code. For instance, a `Car` class might implement both `Drivable` and `Locatable` interfaces. The `Drivable` interface mandates methods like `double getSpeed()` and `void stop()`, while `Locatable` specifies `Coordinate getLocation()` and `boolean locationIsValid()`. Crucially, these declarations allow developers to use interfaces as types; if both `Car` and `Phone` implement `Locatable`, they can be managed in a single list of `Locatable` items, guaranteeing that all elements possess the required methods. These mechanisms provide polymorphism without the inherent risks of physical inheritance.

Complementing interfaces is the technique of Delegation. Delegation addresses a specific issue with inheritance: if a parent class defines many methods (e.g., 20), but the subclass only needs two, the subclass object still retains the remaining 18 methods, causing the class to lose control of its required interface. Instead of relying on inheritance, developers can delegate responsibilities—such as persistence—by having the subclass object store a reference to a separate service object (e.g., an `Account` class delegating persistence to a `Persister` object). This approach effectively breaks the coupling to the framework's API, allowing for a clean separation and control over the necessary application interface, regardless of underlying framework constraints.

These specialized techniques are critical for maintaining architectural flexibility, especially when designing highly available applications where configuration must remain dynamic. Developers must rely on configuration services that allow components to register for parameter updates and receive messages when changes occur, preventing the need to stop and restart the application merely to change a single parameter. The failure to build in external configuration drastically limits adaptability, with consequences comparable to the extinction of the dodo.

System development also involves critical concepts related to concurrency and resource management. Developers must first distinguish between concurrency—the appearance of multiple code pieces running simultaneously through execution switching (using fibers, threads, or processes)—and parallelism, which requires actual physical simultaneous execution capability (multiple cores or CPUs). Since virtually all modern systems are inherently asynchronous (when interacting with users, fetching data, or calling external services), forcing a strictly serial process will result in a sluggish system that fails to utilize the underlying hardware's full power. 

A major architectural challenge in parallel programming is managing shared state. Shared state is defined broadly as any situation where two or more code segments hold references to the same piece of mutable data, and managing it is highly error-prone. Furthermore, developers must be cautious of temporal coupling: this occurs when code imposes an unnatural sequence on events, such as requiring multiple back-end service accesses one after the other, thereby compromising flexibility.

To overcome these difficulties, the use of the actor model is recommended. In this model, independent processes communicate over defined channels using simple semantics while strictly sharing no mutable data. Additionally, blackboards are useful systems that function as a combination of an object store and a smart publish/subscribe broker, offering significant decoupling when correctly implemented. 

Beyond code structure, professional proficiency requires advanced skills in workflow management, estimation, and tooling. When estimating complex tasks, relying on a single number is insufficient. Instead, techniques like Program Evaluation Review Technique (PERT) require providing a spectrum of possibilities, demanding three specific estimates for every task: optimistic, most likely, and pessimistic. The professional then arranges these tasks into a dependency network to calculate the statistically likely best and worst-case project timelines, which significantly improves the overall estimate by mitigating errors caused by arbitrary padding. 

For greater accuracy, Iterative Estimation, or 'Eating the Elephant,' is the preferred method. This approach involves incremental development, demanding the repeated completion of a detailed cycle using small, isolated slices of functionality. Each full cycle must systematically include: checking requirements, performing risk analysis (mandating early priority on the riskiest items), designing, implementing, integrating, and validating results with end-users. Initially, guesswork regarding total duration is unreliable, unless the project is an exact mimicry using the identical team and technology stack. The process begins by completing the coding and testing for the first iteration, and the accrued practical experience subsequently allows for the informed refinement of both the total number of required iterations and the specific scope of subsequent work. Critically, the professional must formally refine the schedule as an integral part of every single iteration, and for communicating timelines to stakeholders, the advice is to state, “I’ll get back to you.”

Regardless of the estimation method, the professional must allocate adequate time to slow down, as estimates delivered under duress are inaccurate. A detailed logging system is mandatory, tracking the accuracy of every prediction. If the observed error exceeds 50%, a thorough investigation into the root causes is required. If the overall estimate involves multiple subestimates, a meticulous record of these subestimates must be maintained to observe the degree of accuracy in predictions.

In the digital workflow, mastering the command shell is paramount. While Graphical User Interfaces (GUIs) and Integrated Development Environments (IDEs) speed up routine tasks (like file movement), they limit capability—particularly preventing the automation of common tasks or combining disparate tools. A key distinction is that GUIs follow WYSIWYG (what you see is what you get), whereas the command shell provides WYSIAYG (what you see is all you get). To maximize efficiency, mastery involves combining utilities using pipes to construct complex macro commands. Key shell configurations include setting color themes, configuring a prompt (which can display information like version control status, time, or a shortened directory name), and utilizing aliases (e.g., `alias apt-up='sudo apt-get update && sudo apt-get upgrade'`) and functions. The professional must also utilize advanced completion mechanisms for context-specific suggestions.

Core to programming is 'editor fluency,' which is the capacity for effortless operation across powerful editors, minimizing the mental distance between an idea and its physical manifestation. Proficiencies include precise content selection by character, word, line, and paragraph; navigating by syntactic units (e.g., matching delimiters, functions, or modules); performing code refactoring (like reindenting or commenting/uncommenting blocks) with a single command; and managing advanced features like multi-panel splitting, undo/redo history, and regex searches. Advanced skills include temporarily creating multiple cursors for parallel text edits and integrating project error display within the editor.

Achieving fluency requires targeted, sustained self-improvement, which involves identifying repetitive actions and implementing better solutions. Rather than learning every command, the focus should be on building muscle memory through repetition. When facing a limitation, the professional must first search for an existing extension; only if none exists should they learn the editor's extension language to automate the task. Targeted challenges include: 1) Disabling autorepeat to force key sequence learning for selection/movement; 2) Restricting all editing for one full week exclusively to keyboard input; 3) Creating a compilation list of desired features to guide plugin searching; 4) If no plugin or extension exists, building the functionality from scratch.

Version Control Systems (VCS) are indispensable for maintaining the integrity of all digital artifacts, including source code, documentation, and shell scripts. VCS acts as a project-wide time machine, surpassing a simple 'undo.' It is crucial for collaboration because simple file sharing risks data loss and conflicting builds. VCS mitigates this risk by tracking every change, providing auditing, bug tracking, and performance analysis capabilities. Users can precisely track metrics such as who altered a specific line, the difference between current and previous weeks' versions, the total quantity of lines changed during a release, and the files modified most frequently. A powerful capability is **branching**, allowing a developer to isolate all work on Feature A on one dedicated branch while a teammate works on Feature B simultaneously. Although many VCS are decentralized, establishing a central repository is recommended for maximizing integration benefits. This central host should ideally provide robust features: excellent security/access control, intuitive UI, CLI for comprehensive automation, automated builds and tests, robust branch merging support (like pull requests), integrated issue management, and good reporting tools (e.g., Kanban boards). Advanced teams often configure VCS so that a simple push to a designated branch automatically triggers the system build, runs associated tests, and deploys the new code to production.

Finally, systematic debugging requires the 'chop' technique, which is conceptually paralleling a binary search. When confronting a massive stack trace, the developer selects a stack frame near the middle; determining if the error exists there narrows the focus down to the preceding or subsequent frames. Repeating this methodical process can pinpoint the bug's origin in a maximum of six attempts, even with 64 frames. In addition to showing instantaneous state, developers often use **tracing statements**—simple diagnostic messages—to observe data or program state over extended periods of time.
