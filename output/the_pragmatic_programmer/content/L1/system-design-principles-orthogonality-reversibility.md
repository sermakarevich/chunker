# system-design-principles-orthogonality-reversibility

**Parent:** [[content/L2/pragmatic-programming-design-principles|pragmatic-programming-design-principles]] — Sustainable software development requires systemic control against 'software rot' by prioritizing the fixing or containment of minor flaws, while individual developers must manage a 'knowledge portfolio' through continuous investment in skills, diverse learning (including nontechnical subjects and new programming languages annually), and adopting advanced communication techniques that tailor a message to the audience. Core architectural principles—Orthogonality, Reversibility, and the DRY principle—mandate that code must be built to be Easy to Change (ETC), ensuring that knowledge, whether in APIs or business rules (like calculating complex account fees), is defined in a single, authoritative location to prevent systemic failure and maintain system flexibility.

System design emphasizes several critical, overlapping principles—Orthogonality, Reversibility, and DRY (Don't Repeat Yourself)—to ensure the resulting software is maintainable, flexible, and resilient to change. These principles guide developers toward minimizing interdependencies and localizing impact, thereby reducing development risk and increasing long-term productivity.

### The Principle of Orthogonality

Orthogonality, initially a term borrowed from geometry, describes two lines meeting at right angles, signifying independence in vector terms. In computing, it signifies a type of decoupling where changes in one component do not affect any of the others. For instance, in a well-designed system, the database code should be orthogonal to the user interface; this means the interface can change without affecting the database, and the database can be swapped without changing the interface. A hallmark of orthogonal design is that components are self-contained, having single, well-defined purposes. When systems are highly interdependent, as demonstrated by the complexity of a helicopter's controls (where lowering the collective pitch lever requires compensating adjustments to the right stick and right pedal, which in turn affects all other controls), any local fix is nearly impossible. The goal is to ensure that components are isolated such that a developer can change one component without worrying about the rest, provided the component’s external interfaces are preserved.

Practically, this means structuring systems into cooperating, orthogonal modules, which can be conceptualized using layered approaches. In such a model, each layer only utilizes abstractions provided by the layers beneath it. A practical test of orthogonality is mapping out the components and determining if changing the requirements for a single function affects only one module; if so, the system adheres to this principle. When considering complex systems, like controlling a heating plant, the ability to switch from a Graphical User Interface (GUI) to a mobile interface without altering the underlying control logic demonstrates orthogonality. Furthermore, developers must ensure decoupling not only from other modules but also from external, uncontrollable factors. It is crucial to avoid using identifiers such as telephone numbers, postal codes, Social Security Numbers, government IDs, email addresses, or domains, as these are external identifiers prone to change (e.g., area code reassignment). Developers must also choose third-party toolkits carefully, ensuring they do not impose unnecessary changes. An object persistence scheme is considered orthogonal if it is transparent and does not require developers to access or create objects in a special way. The Enterprise Java Beans (EJB) system exemplifies this, expressing transaction start and end declaratively via annotations outside the work methods, allowing the same application code to run in different transaction environments. This approach mirrors the Decorator Pattern, which adds functionality without altering core components.

### Coding Practices and Design Patterns

Orthogonality requires continuous vigilance against unintentional functional duplication and tight coupling. Techniques to maintain this include: 

1. **Decoupling:** Developers should write 'shy code'—modules that do not reveal unnecessary details or rely on the implementations of other modules. Following the Law of Demeter suggests that if a developer needs to change an object’s state, they must prompt the object to perform the action itself, keeping code isolated. 
2. **Avoiding Global Data:** Global variables, even read-only ones, tie the code to all other components sharing that data. Code is generally more manageable if required context is explicitly passed into modules, often as parameters to constructors or contained within structures passed by reference. Developers must exercise caution with the Singleton pattern, as it can mimic global variables and introduce unnecessary linkage.
3. **Addressing Repetitive Functions:** When encountering similar functions, particularly those sharing common code at the start and end but differing in central algorithms, this repetition signals a structural problem. The Strategy pattern is recommended as a superior implementation approach. 
4. **Refactoring:** Developers must adopt a critical habit of refactoring—looking for opportunities to reorganize code structure to improve orthogonality. When conducting bug fixes, developers must assess the fix's localization; if changes are scattered throughout the system, the design is likely not orthogonal and requires refactoring. Unit tests serve as a test of orthogonality: if a unit test requires importing a large percentage of the system's code, the module is not sufficiently decoupled.

### Reversibility and Flexibility in Decision Making

Complementing orthogonality is the principle of Reversibility, which counsels against making critical, irreversible design decisions. The reality of software development is that the external environment is constantly changing, meaning that no design choice is permanent. Committing to a specific vendor's database, architectural pattern, or deployment model can create a course of action that is prohibitively expensive or impossible to reverse.

To maintain flexibility, developers must abstract concepts. Instead of assuming a relational database from Vendor A, they should abstract the concept of persistence, treating it merely as a service. This abstraction allows the technology to be switched midstream. Similarly, a browser-based application should plan for a future mobile shift by stripping out HTML rendering and replacing it with a dedicated API. This abstract approach is crucial because the difficulty of making a decision should not lead to the mistake of assuming it is 'cast in stone.' Developers should treat all decisions as if they are written in the sand, subject to the next big wave.

### Handling External Systems and Knowledge Duplication

Orthogonality and DRY principles are particularly challenged when code interacts with external services, APIs, or data sources. When interacting with external APIs, the developer often encounters DRY violations because the code must replicate external knowledge (like schemas or error codes). For internal APIs, using tools to specify the API in a neutral format is recommended, allowing the generation of documentation, mock APIs, functional tests, and clients, ideally housed in a central, shared repository. For external APIs, formal specifications like OpenAPI are beneficial, but if none exist, the developer must create and publish one. 

When dealing with data sources, developers should aim to minimize manual code duplication. While some sources allow introspection to generate container code directly from the schema, an alternative method is often preferred: storing external data in a key/value structure (like a map, hash, or object). This mandates a second layer of validation—a simple table-driven suite—to ensure the key/value map contains the minimum required data in the necessary format. 

The most difficult area for maintaining orthogonality and DRY is often within a development team itself. Historically, a U.S. state's Y2K compliance audit revealed over 10,000 programs with differing versions of Social Security Number validation code, demonstrating how whole sets of functionality can be unintentionally duplicated across a team. To combat this, teams must foster strong collaboration through communication (e.g., daily scrum standup meetings, dedicated Slack channels) and establishing a central location in the source tree for utility routines. Appointing a project librarian to facilitate knowledge exchange, coupled with mandatory peer code reviews (both informal and formal), is critical for preventing systemic knowledge duplication.
