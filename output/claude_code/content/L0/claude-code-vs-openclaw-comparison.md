# claude-code-vs-openclaw-comparison

**Parent:** [[content/L1/agent-architecture-comparison|agent-architecture-comparison]] — Claude Code delegates to isolated subagents via AgentTool with six built-in types (Explore, Plan, etc.), while OpenClaw supports advanced multi-agent routing and customizable sub-agent delegation with up to 5 nesting depths, managing security through differing trust models (deny-first vs. single trusted operator).

The design of Claude Code utilizes a mostly append-only state, meaning context compaction never modifies or deletes previously written transcript lines. Claude Code's "checkpoints" are specifically file-history checkpoints for the `--rewind-files` flag, stored in the directory `~/.claude/filehistory/<sessionId>/.` These checkpoints provide file-level snapshots for reverting filesystem changes, and they do not constitute a generic checkpoint store. The following section provides a comparative analysis by contrasting Claude Code's design choices with those of OpenClaw, an architecturally independent, open-source AI agent system that answers many of the same recurring design questions regarding loop architecture, safety, extensibility, context management, delegation, and persistence. OpenClaw operates as a persistent WebSocket gateway daemon that connects approximately two dozen messaging surfaces, including WhatsApp, Telegram, Slack, Discord, and Signal, to an embedded agent runtime, and it includes companion applications on macOS, iOS, and Android.

**Table 3: Architectural comparison: Claude Code vs. OpenClaw across six design dimensions.**

| Feature | Claude Code | OpenClaw |
| :--- | :--- | :--- |
| **System scope** | CLI/IDE coding harness, ephemeral persistence process | Persistent WS gateway daemon, multichannel control plane |
| **Trust model** | Deny-first per-action rule evaluation with hooks and optional ML classifier; utilizes 7 permission modes; supports a graduated trust spectrum | Uses a single trusted operator per gateway; implements DM pairing and allows inbound channels; provides opt-in sandboxing with configurable scope (per-agent, per-session, or shared) and multiple backends |
| **Agent runtime** | Iterative async generator (`queryLoop()`) as system center | Pi-agent runner embedded inside gateway RPC dispatch; utilizes per-session queue serialization (with an optional global lane) |
| **Extension architecture** | 4 mechanisms with graduated context costs: MCP, plugins, skills, hooks | Manifest-first plugin system with 12 capability types and central registry; separate skills layer; built-in MCP via openclaw-mcp (server and outbound client registry) |
| **Memory and context** | CLAUDE.md 4-level hierarchy; 5-layer compaction pipeline; LLM-based memory scan | workspace bootstrap files (AGENTS.md, SOUL.md, TOOLS.md, IDENTITY.md, USER.md, plus conditionally BOOTSTRAP.md, HEARTBEAT.md, and MEMORY.md); separate memory system (MEMORY.md, daily notes, optional DREAMS.md); auto-compaction with pluggable providers; optional hybrid search (vector + keyword, conditional on embedding provider); experimental dreaming for long-term promotion |
| **Task-delegating subagents** | (e.g., Explore, Plan, general-purpose); utilizes worktree isolation; final response text returned to parent | Two separate concerns: (a) multi-agent routing with isolated agents, distinct workspaces, and binding-based channel dispatch; (b) sub-agent delegation with configurable nesting depth (max 5, default 1, recommended 2) and thread-bound sessions |
