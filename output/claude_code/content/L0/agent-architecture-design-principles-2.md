# agent-architecture-design-principles-2

**Parent:** [[index]]

The discussion of agent architectures and design principles is underpinned by several core design questions. Regarding reasoning, the core principle adopted by Claude Code is that the Large Language Model (LLM) performs the required reasoning about actions, while the dedicated operational harness executes those actions. The model generates structured tool-use blocks, and the surrounding harness is responsible for parsing these blocks, checking permissions, and dispatching them to the specific tools. This physical separation ensures that the model cannot bypass the system's safety mechanisms—including sandboxing, permission checks, or deny-first rules—because the model's only permitted interface to the outside world is the structured `tool_use` protocol, which the harness must validate before execution. This model is evidenced by source code analysis estimating that approximately 98.4% of Claude Code's codebase constitutes the operational infrastructure, leaving only about 1.6% for actual decision logic. Regarding the number of execution engines, Claude Code utilizes a single `queryLoop()` function for all interactions, regardless of whether the user is operating through an interactive terminal, a headless CLI invocation, or the Agent SDK. This design choice prioritizes uniformity and simplicity. For the default safety posture, the system implements deny-first evaluation coupled with human escalation. This rule set mandates that deny rules always override ask rules, which in turn override allow rules, and any unrecognized actions must be escalated to the user. To achieve a layered defense-in-depth, multiple independent safety mechanisms operate in parallel: the permission rules, PreToolUse hooks, the auto-mode ML classifier (when enabled), and optional shell sandboxing. Alternative systems, such as SWE-Agent and OpenHands, rely on container-based isolation to contain execution, while Aider primarily uses Git-based rollback for safety.
