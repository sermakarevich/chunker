# subagent-orchestration-details

**Parent:** [[content/L1/agent-architecture-comparison|agent-architecture-comparison]] — Claude Code delegates to isolated subagents via AgentTool with six built-in types (Explore, Plan, etc.), while OpenClaw supports advanced multi-agent routing and customizable sub-agent delegation with up to 5 nesting depths, managing security through differing trust models (deny-first vs. single trusted operator).

Multi-agent orchestration is a key design dimension for coding agents, with potential choices spanning parent-child hierarchies, peer-based conversation frameworks (Wu et al., 2024), and graph-structured workflow engines (LangChain, Inc., 2024). Claude Code’s delegation architecture implements the isolated subagent boundaries principle, alongside aspects of deny-first with human escalation (permission override) and reversibility-weighted risk assessment (subagent tool restrictions). When Claude determines that an authentication test fix requires exploring the authentication module’s structure, it delegates this exploration to a subagent. The delegation mechanism is the `AgentTool` (found at `AgentTool.tsx`), which retains `Task` as a legacy alias. The model invokes `Agent` using a structured input that must include the delegation prompt, an optional subagent type, and configuration for isolation mode, permission overrides, and the working directory.

**The Agent Tool and Delegation Criteria**

The `AgentTool` input schema uses feature-gated fields, thus omitting optional parameters when their backing features are disabled. The isolation field offers `['worktree', 'remote']` for internal users and only `['worktree']` for external users, a setting that is determined at build time. The `cwd` field is gated by a feature flag, and the `run_in_background` field is omitted when background tasks are disabled or when `fork-subagent` mode is enabled. Claude Code supports up to six built-in subagent types, depending on active feature flags and the entrypoint: 1) **Explore:** primarily a read/search-oriented investigation, containing a deny-list for write and edit tools. 2) **Plan:** creates structured plans, and execution proceeds through the standard permission model. 3) **General-purpose:** is broadly capable and used when explicitly requested (note: omitting the type may route to the `fork-subagent` path instead). 4) **Claude Code Guide:** assists with onboarding and documentation, and possesses its own `permissionMode` override. 5) **Verification:** executes validation checks (such as test suites or linting). 6) **Statusline-setup:** is specialized for terminal status line configuration.

Beyond these built-in types, users define custom subagents using `.claude/agents/*.md` files, and plugins contribute agent definitions via `loadPluginAgents.ts`. The markdown body of each custom agent file serves as the agent’s system prompt, and the YAML frontmatter specifies configuration fields, including description, tools (allowlist), disallowedTools, model, effort, permissionMode, mcpServers, hooks, maxTurns, skills, memory scope, background flag, and isolation mode. Additionally, JSON-formatted agent definitions support the same fields plus a prompt as an explicit field (via `loadAgentsDir.ts`). This setup means a custom agent can function as a fully configured, isolated sub-system with its own tools, model, permissions, hooks, memory scope, and isolation mode. While `AgentTool` and `SkillTool` both sit alongside each other in the base tool pool as meta-tools, they differ fundamentally: `SkillTool` injects instructions into the current context window, whereas `AgentTool` spawns a new, isolated one. Consequently, most subagent invocations require a self-contained prompt because the default path does not inherit the parent’s conversation history (the `fork-subagent` path is an exception). Conversation-based frameworks that share full transcript histories avoid the need for a self-contained prompt but risk context explosion as the number of agents increases.

**Isolation Architecture**

Subagent isolation supports multiple modes (via `AgentTool.tsx`): 1) **Worktree:** creates a temporary git worktree, providing the subagent with its own copy of the repository to modify without affecting the parent’s working tree. 2) **Remote** (internal-only): launches in a remote Claude Code Remote environment, which always runs in the background. 3) **In-process** (default): shares the filesystem with the parent but operates within an isolated conversation context.

The permission override logic for subagents (managed in `runAgent.ts`) involves several specific rules. When a subagent defines a `permissionMode`, the override is applied unless the parent agent is already operating in `bypassPermissions`, `acceptEdits`, or `auto` mode, since these parent modes always take precedence because they represent explicit user decisions about the safety/autonomy trade-off. For asynchronous agents, the system determines whether to avoid prompts through a cascade: first checking `explicit canShowPermissionPrompts`, then checking `bubble` mode (which always shows prompts because it escalates to the parent terminal), and finally checking the `default` mode (where synchronous agents show prompts, but asynchronous agents do not). Background agents that can show prompts must set `awaitAutomatedChecksBeforeDialog: true`, ensuring that the classifier and hooks resolve before interrupting the user.

These isolation modes occupy different points in a design space. Container-based isolation (utilized by SWE-Agent and OpenHands (Yang et al., 2024; Wang et al., 2024b)) offers stronger resource boundaries but requires container infrastructure. In contrast, context-only isolation (used by conversation-based frameworks like AutoGen (Wu et al., 2024)) shares the filesystem but separates conversation histories. Claude Code’s worktree-based isolation provides filesystem-level separation with zero external dependencies, utilizing Git’s built-in mechanism rather than introducing container orchestration.
