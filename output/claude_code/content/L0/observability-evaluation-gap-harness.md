# observability-evaluation-gap-harness

**Parent:** [[content/L1/claude-code-operational-architecture-2|claude-code-operational-architecture-2]] — Claude Code utilizes a highly complex, multi-layered operational harness (98.4% of code) structured around three commitments: graduated layering, append-only auditability, and model judgment within a deterministic framework; it manages safety via a seven-stage pipeline across seven permission modes (e.g., `acceptEdits`, `auto`) and context memory via a five-stage compaction pipeline.

The observability–evaluation adoption gap—the discrepancy between the ability to observe agent actions and the use of formal evaluation methods—remains unresolved by existing sources, making its resolution an architectural challenge for the operational harness rather than a capability limitation of the model. Empirical research has characterized this gap at several resolutions: Cemri et al. (2025) catalogued fourteen failure modes spanning system-design issues, inter-agent misalignment, and task verification. Furthermore, Pathak et al. (2025) developed a benchmark of agent trajectories specifically for anomaly detection in traces, Yao et al. (2024) exposed consistency gaps using the passk metric (which calculates the probability that all $k$ independent trials succeed), and Kapoor et al. (2024) argued that current agent benchmarks lack holdouts and cost controls, thereby limiting what observability can actually diagnose. Regarding the permission pipeline and tool-orchestration layers analyzed in Sections 4 and 5, two architectural questions persist. First, the sources have not settled whether the scaffolding suggested by Rajasekaran (2026)—such as generator–evaluator separation, sprint contracts, or post-hoc checks (building upon Madaan et al.'s (2023) self-refine pattern)—should be integrated inside the operational harness (for example, as additional hook events alongside the 27 documented hooks in Section 6) or should reside outside the harness as a separate evaluation layer.
