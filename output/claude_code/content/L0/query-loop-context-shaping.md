# query-loop-context-shaping

**Parent:** [[content/L1/agent-architecture-and-context-management|agent-architecture-and-context-management]] — Claude Code implements a highly structured, layered architecture comprising five subsystems (surface, core, safety/action, state, backend). Its context management uses five mandatory shapers—Budget reduction, Snip, Microcompact, Context collapse, and Auto-compact—executed sequentially before every model call. Safety relies on a deny-first rule evaluation passing through seven independent gates, including sandboxing and hook-based interception, thereby ensuring resilience regardless of user vigilance.

The `queryLoop()` function is implemented as an AsyncGenerator, which yields `StreamEvent`, `RequestStartEvent`, `Message`, `TombstoneMessage`, and `ToolUseSummaryMessage` events as the loop progresses. This generator-based design facilitates streaming output to the User Interface (UI) layer while simultaneously maintaining a single synchronous control flow within the loop.

Claude Code's reactive loop follows the ReAct pattern: the model generates reasoning and tool invocations, the system harness executes the actions, and the resulting data feeds the next iteration. While alternative orchestration patterns include explicit graph-based routing (LangChain, Inc., 2024), which defines control flow as a state machine with typed edges, and tree-search methods (Zhou et al., 2023) that explore multiple action trajectories before committing, Claude Code primarily uses the orchestrator-workers pattern for subagent delegation (Section 8) while maintaining a core reactive loop, which is consistent with Anthropic’s documentation (Schluntz and Zhang, 2024) on five composable workflow patterns (prompt chaining, routing, parallelization, orchestrator-workers, and evaluator-optimizer). This reactive design prioritizes simplicity and low latency by committing to one action sequence per turn, thereby trading away search completeness.

### Tool Dispatch and Streaming Execution

When the model response includes `tool_use` blocks, the system selects between two execution paths. The primary path utilizes `StreamingToolExecutor`, which starts executing tools immediately as they stream in from the model response, thereby reducing latency for responses containing multiple tools. The fallback path uses the `runTools()` function found in `toolOrchestration.ts`, which iterates over partitions generated by `partitionToolCalls()`. Both execution paths classify tools as either concurrent-safe or exclusive. The system serializes state-modifying operations, such as shell commands, while read-only operations can execute concurrently.

The `StreamingToolExecutor` manages concurrent execution through two mechanisms: 1) A **Sibling abort controller**, which immediately terminates all other in-flight subprocesses if any single Bash tool encounters an error, preventing them from completing naturally. 2) A **Progress-available signal**, which wakes up the `getRemainingResults()` consumer when new output becomes available. Results are buffered and emitted in the sequence in which the tools were received; consequently, the output order remains consistent even when the tools run in parallel. This concurrent-read, serial-write execution model provides a middle ground between fully serial dispatch and more aggressive speculative approaches, such as PASTE (Sui et al., 2026), which predicts and pre-executes future tool calls while the model is generating, thus masking tool latency through speculation.

The tool result collection phase processes updates from either the streaming executor or the synchronous `runTools()` generator. Each update can contain a tool result, an attachment, or a progress event. A special check is implemented to detect `hook_stopped_continuation` attachments: if a PostToolUse hook signals that the turn should not continue, the system sets a `shouldPreventContinuation` flag. Subsequently, results are normalized for the Anthropic API using `normalizeMessagesForAPI()`, which filters the results to retain only user-type messages.

### Pre-Model Context Shapers

Five distinct context shapers execute sequentially in `query.ts` before every model call, with each shaper operating on the `messagesForQuery` array. The shapers are designed so that earlier steps apply lighter reductions, preceding later steps that apply broader compaction.

1. **Budget reduction (`applyToolResultBudget()`):** Enforces specific per-message size limits on tool results. The shaper replaces any oversized outputs with content references. Tools that are exempted (i.e., those where `maxResultSizeChars` is not finite) retain their complete output. The content replacements generated by this process are persisted for both agent and session query sources, enabling reconstruction if the session resumes. Budget reduction must run before microcompact because microcompact operates exclusively using `tool_use_id` and never inspects the content; this dependency ensures the two shapers compose cleanly.
2. **Snip (`snipCompactIfNeeded()`, gated by `HISTORY_SNIP`):** This is a lightweight trim that removes older history segments, returning an object containing the remaining messages, the count of freed tokens, and the boundary message. The value `snipTokensFreed` is explicitly passed to auto-compact because the main token counter calculates context size using the usage field on the most recent assistant message, and that message survives snip with its pre-snip `input_tokens` still attached; thus, snip’s savings are invisible to the counter unless passed through explicitly.
3. **Microcompact:** This shaper provides fine-grained compression. It always runs a time-based path and optionally runs a cache-aware path (gated by `CACHED_MICROCOMPACT`). When the cached path is enabled, boundary messages are deferred until after the API response so that the system can utilize the actual `cache_deleted_input_tokens` rather than estimations. Microcompact returns `{messages, compactionInfo}`, where `compactionInfo` may include pending `CacheEdits`.
4. **Context collapse (Gated by `CONTEXT_COLLAPSE`):** This mechanism performs a read-time projection over the conversation history. The collapsed view is not stored in the REPL array; instead, summary messages reside in a dedicated collapse store. By using `applyCollapsesIfNeeded()`, the model sees the collapsed version via the `messagesForQuery` array, while the full history remains available for reconstruction. This architectural detail ensures that context collapse does not mutate the REPL’s stored history.
5. **Auto-compact:** This final shaper, triggered by `compactConversation()` in `compact.ts`, performs a full model-generated summary. This function first executes PreCompact hooks, then creates a summary request using `getCompactPrompt()`, and finally calls the model to produce a compressed summary. The resultant summary feeds into `buildPostCompactMessages()` (in `compact.ts`). Auto-compact only executes if the context still exceeds the predefined pressure threshold after all four preceding shapers have successfully run.

### Recovery Mechanisms and Stop Conditions

The query loop incorporates several recovery and termination mechanisms for various edge cases:

*   **Max output tokens escalation:** If the API response hits the output token cap, the system may retry by using an escalated limit, provided that a `GrowthBook` flag is set and no existing override or environment-variable cap is present. The loop allows up to three recovery attempts per turn (`MAX_OUTPUT_TOKENS_RECOVERY_LIMIT = 3`).
*   **Reactive compaction (Gated by `REACTIVE_COMPACT`):** If the context approaches capacity, reactive compact executes, summarizing only enough information to free up necessary space. The `hasAttemptedReactiveCompact` flag ensures this function fires at most once per turn.
*   **Prompt-too-long handling:** If the API returns a `prompt_too_long` error, the loop first attempts context collapse overflow recovery and then reactive compaction. Only if both of these recovery steps fail does the system terminate with the specific reason: `'prompt_too_long'`.
*   **Streaming fallback:** The `onStreamingFallback` callback handles potential streaming API issues, enabling the loop to retry the process using an alternative strategy.
*   **Fallback model:** The `fallbackModel` parameter allows the system to switch to an alternative model if the primary model encounters a failure.

Multiple conditions can terminate the loop: 1) **No tool use:** This is the primary stop condition, which occurs when the model produces only plain text content. 2) **Max turns:** The configurable `maxTurns` limit is reached. 3) **Context overflow:** The API returns a `prompt_too_long` error. 4) **Hook intervention:** A PostToolUse hook sets the `hook_stopped_continuation` flag. 5) **Explicit abort:** The `abortController` signal fires.
