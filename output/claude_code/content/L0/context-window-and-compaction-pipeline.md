# context-window-and-compaction-pipeline

**Parent:** [[content/L1/agent-architecture-mechanisms|agent-architecture-mechanisms]] — The agent's highly layered architecture employs a 'deny-first' authorization pipeline, managed through three mechanisms (declarative rules, global trust modes, and hooks), and five permission modes (e.g., `plan`, `acceptEdits`, `auto`) to enforce safety; context assembly is non-static, starting with a four-level CLAUDE.md memory hierarchy and evolving during the turn through late injections; and extensibility is achieved via four core mechanisms (MCP servers, Plugins, Skills, Hooks) plus agent spawning, which collectively manage context state and tool reach through sophisticated meta-tools and a five-layer compaction process.

The context window is built from several sources, some of which are included during initial assembly and others that are injected later during a conversation turn. 

Initially, the context window includes:

1. **System Prompt:** This incorporates any output style modifications and content from the `--append-system-prompt` flag. 
2. **Environment Info:** The function `getSystemContext()` (in `context.ts`) retrieves the git status (which is skipped if the execution is in remote mode or if git instructions are disabled) and an optional cache-breaking injection for internal builds (which is gated by the `BREAK_CACHE_COMMAND` flag). The system memoizes this information once per session.
3. **CLAUDE.md Hierarchy:** The function `getUserContext()` (in `context.ts`) loads the four-level instruction file hierarchy. The system memoizes this as well. Furthermore, the CLAUDE.md files are plain-text Markdown, which makes the content inspectable and editable by the user, prioritizing auditability over expressiveness. Instead of using embeddings or a vector similarity index for memory retrieval, Claude Code uses an LLM-based scan of memory-file headers to select up to five relevant files on demand, surfacing the files at a file granularity rather than an entry granularity. The CLAUDE.md files follow a multi-level loading hierarchy, defining four memory types: Managed memory (e.g., `/etc/claude-code/CLAUDE.md` on Linux) for OS-level policy for all users; User memory (`~/.claude/CLAUDE.md`) for private global instructions; Project memory (`CLAUDE.md`, `.claude/CLAUDE.md`, and `.claude/rules/*.md` in project roots) for instructions checked into the codebase; and Local memory (`CLAUDE.local.md` in project roots) for gitignored, private project-specific instructions. The system’s file discovery traverses from the current directory up to the root, checking for all project and local memory files in each directory, with files closer to the current directory having higher priority (and thus, being loaded later). The system loads files in a “reverse order of priority,” meaning that later-loaded files receive more model attention. For root-to-CWD directories, unconditional rules from `.claude/rules/*.md` load eagerly at startup. For nested directories below the Current Working Directory (CWD), even unconditional rules are loaded lazily when the agent reads files in matching directories, meaning the model’s instruction set can evolve during a conversation as new parts of the codebase are explored. The CLAUDE.md content is delivered as user context (a user message), not as system prompt content, which means model compliance with these instructions is probabilistic rather than guaranteed. This separation creates a deliberate distinction between guidance (CLAUDE.md, probabilistic) and enforcement (permission rules, deterministic).
4. **Path-scoped rules:** These are conditional and directory-matched rules that load lazily when the agent reads files in matching directories.
5. **Auto memory:** The system asynchronously prefetches contextually relevant memory entries.
6. **Tool metadata:** This includes skill descriptions, MCP tool names, and deferred tool definitions (which are available via `ToolSearch` on demand).
7. **Conversation history:** This carries forward existing history, which is subject to compaction.
8. **Tool results:** These include outputs from file reads, command outputs, and subagent summaries.
9. **Compact summaries:** These replace older history segments.

When the system generates a turn, the context window composition process is managed by the following steps:

1. **System Prompt Assembly:** The process at `query.ts` combines the system context with the base prompt using `asSystemPrompt(appendSystemContext(systemPrompt, systemContext))()`. The system also prepends the user context (CLAUDE.md and date) to the message array via `prependUserContext()`. 
2. **Late Injections:** Several context sources are injected late, after the main window is constructed. These include relevant-memory prefetch (managed by `query.ts`), MCP instructions deltas (only new or changed server instructions), agent listing deltas, and background agent task notifications. Consequently, the context window is not static at the time of assembly but can grow during the turn.

Furthermore, the system utilizes a five-layer compaction pipeline, which manages context pressure through graduated compression in `query.ts`. Instead of using a single strategy, Claude Code applies five layers sequentially, with each layer having increasing aggressiveness. Three of these layers are gated by feature flags, and the budget reduction layer is always active, while auto-compact is user-configurable. This layered design reflects a lazy-degradation principle: the system applies the least disruptive compression first, escalating only when cheaper strategies are insufficient. The five layers are:

1. **Budget reduction:** This layer enforces per-tool-result size limits. (This layer is always active).
2. **Snip:** This mechanism (gated by `HISTORY_SNIP`) is a lightweight trim that removes older history segments.
3. **Microcompact:** This mechanism (gated by `CACHED_MICROCOMPACT`) provides fine-grained, cache-aware compression.
4. **Context collapse:** This mechanism (gated by `CONTEXT_COLLAPSE`) performs a read-time virtual projection over the conversation history. 
5. **Auto-compact:** This mechanism (enabled by default but can be disabled) performs a full, model-generated summary. 

When the `buildPostCompactMessages()` function (in `compact.ts`) returns the compacted output structure, it includes the following elements: `[boundaryMarker, ...summaryMessages, ...messagesToKeep, ...attachments, ...hookResults]`. The `boundaryMarker` is annotated with preserved-segment metadata, recording `headUuid`, `anchorUuid`, and `tailUuid` to enable read-time chain patching. Because compaction never modifies or deletes previously written transcript lines, this process is mostly append-only, only appending new boundary and summary events to the transcript.
