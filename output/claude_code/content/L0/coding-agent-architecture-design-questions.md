# coding-agent-architecture-design-questions

**Parent:** [[content/L1/claude-code-agent-architecture|claude-code-agent-architecture]] — Claude Code, an agentic coding tool, is designed around five core values (Authority, Safety, Reliability, Capability Amplification, Contextual Adaptability), which guide the implementation of architectural principles like deny-first evaluation and five-layer context compaction. It operates a core while-loop loop and relies on complex surrounding subsystems including a seven-mode permission system, four extensibility mechanisms, and an append-oriented session store.

This section details four recurring design questions that underpin production coding agents, with each question corresponding to one or more design principles listed in Table 1. For each question, the original text provides Claude Code's implemented answer, discusses plausible alternatives, and demonstrates the progressive implementation through Sections 4 to 9.

**1. Where does reasoning live?**
In Claude Code, the model performs reasoning about the required actions, while the dedicated harness is responsible for executing those actions. The model emits structured `tool_use` blocks as part of its response; the harness then parses these blocks, checks permissions, dispatches them to the specific tool implementations, and collects the results (as shown in `query.ts`). This architecture ensures that the model never directly accesses the filesystem, runs shell commands, or makes network requests. This physical separation introduces a critical security consequence: because reasoning and enforcement utilize separate code paths, a compromised or adversarially manipulated model cannot bypass the sandboxing, permission checks, or deny-first rules implemented within the harness. The model's only permitted interface to the outside world is the structured `tool_use` protocol, which the harness must validate before execution.

Community analysis of the extracted source code estimates that approximately 1.6% of Claude Code's codebase constitutes AI decision logic, leaving the remaining 98.4% dedicated to operational infrastructure. This ratio effectively demonstrates how minimal the core agent reasoning layer is. Alternative designs, such as Devin and LangGraph (LangChain, Inc., 2024), invest significantly more heavily in scaffolding-side reasoning, for instance, by maintaining explicit planning and task-tracking structures or by routing control flow through developer-defined state graphs.

**2. How many execution engines?**
Claude Code utilizes a single `queryLoop()` function, which executes irrespective of whether the user is interacting through an interactive terminal, a headless CLI invocation, the Agent SDK, or an IDE integration (all implemented in `query.ts`). In this model, only the rendering and user-interaction layer is allowed to vary. In contrast, other systems adopt mode-specific engines; for example, an IDE integration might follow a distinct code path compared to a CLI tool, thereby sacrificing uniformity for optimized surface-specific functionality.

**3. What is the default safety posture?**
Claude Code's default safety posture is established using deny-first with human escalation. This rule set mandates that deny rules override ask rules, which in turn override allow rules, and any unrecognized actions are escalated to the user instead of being allowed silently (as implemented in `permissions.ts`). Furthermore, multiple independent safety layers operate in parallel: the permission rules, PreToolUse hooks, the auto-mode classifier (when enabled), and optional shell sandboxing. Since any single layer can block an action, this approach successfully combines the principles of deny-first with human escalation and defense in depth with layered mechanisms, as defined in Table 1. Alternative approaches, such as SWE-Agent and OpenHands (Yang et al., 2024; Wang et al., 2024b), instead rely on container-based isolation to contain arbitrary execution, while Aider (Gauthier, 2024) primarily uses git-based rollback as its key safety mechanism.

**4. What is the binding resource constraint?**
For Claude Code, the primary binding resource constraint is the context window, which is 200K for older models and 1M for the Claude 4.6 series. To manage this, five distinct context-reduction strategies execute before every model call (`query.ts`), and several other subsystem decisions are employed to limit context consumption, including lazy loading of instructions, deferred tool schemas, and summary-only subagent returns (as detailed in Section 7). The five-layer pipeline is necessary because no single compaction strategy can address all types of context pressure. These five strategies are: Budget reduction (targeting individual tool outputs that exceed size limits); Snip (handling temporal depth); Microcompact (reacting to cache overhead); Context collapse (managing very long histories); and Auto-compact (performing semantic compression as a last resort). Each layer operates using a different cost-benefit trade-off, ensuring that earlier, cheaper layers execute before the more costly ones. Alternative architectures may identify other resources, such as compute budget (limiting the number of model calls or tool invocations) or working memory (maintaining an explicit scratchpad rather than depending on the conversation history), as their primary bottleneck.
