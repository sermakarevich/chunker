# agent-system-values-and-principles

**Parent:** [[content/L1/claude-code-agent-architecture|claude-code-agent-architecture]] — Claude Code, an agentic coding tool, is designed around five core values (Authority, Safety, Reliability, Capability Amplification, Contextual Adaptability), which guide the implementation of architectural principles like deny-first evaluation and five-layer context compaction. It operates a core while-loop loop and relies on complex surrounding subsystems including a seven-mode permission system, four extensibility mechanisms, and an append-oriented session store.

The core design of the agent system is driven by five primary values: Human Decision Authority, Safety/Security/Privacy, Reliable Execution, Capability Amplification, and Contextual Adaptability. These values inform the system's architecture and design principles.

**Safety, Security, and Privacy:** The system is designed to protect users' code, data, and infrastructure from harm, even if the user is inattentive or makes mistakes. This value differs from Human Decision Authority, which concerns the user’s power to choose, by focusing instead on the system’s obligation to protect the user even when that power lapses. Anthropic’s safe-agents framework separately highlights securing agent interactions and protecting privacy across extended interactions as core commitments. Specifically, the auto-mode threat model (Hughes, 2026) targets four risk categories: overeager behavior, honest mistakes, prompt injection, and model misalignment.

**Reliable Execution:** Reliable Execution dictates that the agent must accomplish what the user truly intended, maintain coherence over time, and enable verification of the agent's work before declaring success. This value encompasses both single-turn correctness (determining if the agent interpreted the request faithfully) and long-horizon dependability (maintaining coherence across context window boundaries, session resumption, and multi-agent delegation). Anthropic’s product documentation (Anthropic, 2026d) describes a three-phase loop that the agent repeats until the task is complete: gather context, take action, and verify results. Furthermore, the agent design guidance (Schluntz and Zhang, 2024) stresses that assessing progress at each step must be grounded in "truth from the environment." Similarly, the harness-design guidance (Rajasekaran, 2026) notes that agents tend to confidently praise the work, even when the quality is mediocre, suggesting the need for separating generation from evaluation.

**Capability Amplification:** This value ensures the system materially increases the amount of work a human can accomplish per unit of effort and cost. Anthropic’s internal survey (Huang et al., 2025), discussed in Section 1, indicates that the architecture enables qualitatively new workflows, meaning approximately 27% of tasks represented work that would not otherwise have been attempted. The system's creators describe the tool as “a Unix utility rather than a traditional product,” built from the smallest building blocks that are “useful, understandable, and extensible” (Cherny and Wu, 2025). The architecture prioritizes investing in deterministic infrastructure—such as context management, tool routing, and recovery—over decision scaffolding (like explicit planners or state graphs). The premise guiding this choice is that increasingly capable models benefit more from a rich operational environment than from frameworks that restrict the model's choices.

**Contextual Adaptability:** This value requires the system to fit the user’s specific context—including the user's project, tools, conventions, and skill level—and demands that the relationship between the user and the agent improves over time. The extension architecture (CLAUDE.md, skills, MCP, hooks, plugins) provides configurable options at multiple levels of context cost (Sections 6 and 7). Longitudinal data (McCain et al., 2026) demonstrates that the human-agent relationship evolves: the auto-approve rate increases from approximately 20% when there are fewer than 50 sessions to over 40% by 750 sessions. This pattern describes an autonomy that is “co-constructed by the model, the user, and the product,” meaning the system is designed for trust trajectories rather than fixed trust states. The transfer of the Mechanical Composition Protocol (MCP) to the Linux Foundation’s Agentic AI Foundation (The Linux Foundation, 2025) further reflects the ecosystem dimension of this value.

***

**Mapping of Values to Principles and Mechanisms:**
The five values map to specific architectural decisions: 
* **Human Decision Authority** motivates the adoption of deny-first evaluation, the graduated trust spectrum, append-only state (for auditable history), externalized programmable policy, and values-over-rules (Sections 5 to 7 and 9). 
* **Safety, Security, and Privacy** motivates defense in depth, deny-first defaults, reversibility-weighted assessment, externalized policy, and isolated subagent boundaries (Sections 5 and 8). 
* **Reliable Execution** motivates context-as-scarce-resource management, append-only durable state, graceful recovery, isolated subagent boundaries, and defense in depth (Sections 4 and 7 to 9). 
* **Capability Amplification** motivates minimal scaffolding, composable extensibility, reversibility-weighted risk, context management, and graceful recovery (Sections 4 to 6). 
* **Contextual Adaptability** motivates transparent file-based memory, composable extensibility, the graduated trust spectrum, and externalized programmable policy (Sections 5 to 7).

These mappings also reveal what the architecture deliberately avoids: imposing explicit planning graphs on the model’s reasoning, providing a single unified extension mechanism, or restoring all session-scoped trust-related state across session resume. 

**Table 1: Design Principles, Values, and Design Questions:** The principles serve the following values and answer specific design questions:

* **Deny-first with human escalation:** Serves Authority, Safety. Question: Should unrecognized actions be allowed, blocked, or escalated to the human? (Relevant Sections: 5, 8, 9).
* **Graduated trust spectrum:** Serves Authority, Adaptability. Question: Should the permission level be fixed, or should users traverse a spectrum over time? (Relevant Section: 5).
* **Defense in depth with layered mechanisms:** Serves Safety, Authority, Reliability. Question: Should the system use a single safety boundary or multiple overlapping ones using different techniques? (Relevant Sections: 3, 5).
* **Externalized programmable policy:** Serves Safety, Authority, Adaptability. Question: Should the policy be hardcoded, or should the system use externalized configurations with lifecycle hooks? (Relevant Sections: 5, 6).
* **Context as scarce resource with progressive management:** Serves Reliability, Capability. Question: What is the binding resource constraint, and how should the system manage it (single-pass truncation or graduated pipeline)? (Relevant Sections: [Missing] -> Assuming 4/5).
* **Append-only durable state:** Serves Reliability, Authority. Question: Should the state be mutable, use checkpoint snapshots, or utilize append-only logs? (Relevant Sections: 4, 9).
* **Minimal scaffolding, maximal operational harness:** Serves Capability, Reliability. Question: Should the system invest in scaffolding-side reasoning or in operational infrastructure that allows the model to reason freely? (Relevant Sections: 3, 4).
* **Values over rules:** Serves Capability, Authority. Question: Should the system rely on rigid decision procedures or on contextual judgment backed by deterministic guardrails? (Relevant Sections: 3, 5, 7).
* **Composable multi-mechanism extensibility:** Serves Capability, Adaptability. Question: Should the system use one unified extension API, or layered mechanisms at different context costs? (Relevant Section: 6).
* **Reversibility-weighted risk assessment:** Serves Capability, Safety. Question: Should the oversight be the same for all actions, or should it be lighter for reversible and read-only actions? (Relevant Sections: [Missing] -> Assuming 4/5). 
* **Transparent file-based configuration and memory:** Serves Adaptability, Authority. Question: Should the system use an opaque database, embedding-based retrieval, or user-visible, version-controllable files? (Relevant Section: 7).
* **Isolated subagent boundaries:** Serves Reliability, Safety, Capability. Question: Should subagents share the parent’s context and permissions, or should they operate in isolation? (Relevant Section: 8).
* **Graceful recovery and resilience:** Serves Reliable, Capability. Question: Should the system fail hard on errors, or should it silently recover and reserve human attention for unrecoverable situations? (Relevant Sections: 4, 5, 8).
