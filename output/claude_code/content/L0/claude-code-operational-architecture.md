# claude-code-operational-architecture

**Parent:** [[index]]

Claude Code, an agentic coding tool developed by Anthropic, was designed with an architecture that prioritizes investment in a rich operational harness over limiting the Large Language Model's inherent decision-making processes. Source code analysis estimates that approximately 98.4% of Claude Code's codebase constitutes the operational harness, leaving only about 1.6% for actual decision logic. This structural separation adheres to the 'Values over Rules' principle, mandating that the operational harness must create conditions enabling the model to make informed decisions rather than dictating the model's choices. To ensure system reliability and control, the system employs a layered safety architecture that forces every tool invocation through a multi-stage authorization pipeline, which operates independently of human vigilance. This pipeline involves three primary components: declarative rules, global trust modes, and programmable hooks. The decision process first begins with a Policy Core assessment, determining the outcome as Deny, Allow, or Ask. If the model requests a tool, the process executes tool pre-filtering, which strips all blanket-denied tools from the model's view before building the tool pool. Next, the PreToolUse hook fires, allowing a hook to return a `permissionDecision` (deny or ask), a `permissionDecisionReason`, or an `updatedInput` to modify tool parameters. Following this, the deny-first rule engine evaluates the request, ensuring that any deny rule always takes precedence over an allow rule. Finally, the Permission handler checks the runtime context, which can be **Coordinator** (for multi-agent coordination), **Swarm worker** (for worker agents), **Speculative classifier** (when the `BASH_CLASSIFIER` feature flag is active and the tool is a `BashTool`), or **Interactive** (the standard user path). Furthermore, Claude Code supports an extensible architecture including Mechanical Composition Protocol (MCP) servers, plugins, skills, and hooks. The system organizes these extensions by increasing context cost: hooks consume zero context, skills consume low context, and MCP servers consume high context. Context management utilizes a five-layer compaction pipeline, which is necessary because the context window acts as the primary binding resource constraint. The five sequential shapers are: 1. **Budget reduction:** This layer enforces per-tool-result size limits and is always active. 2. **Snip:** A lightweight trim, gated by `HISTORY_SNIP`, that removes older history segments. 3. **Microcompact:** A fine-grained, cache-aware compression, gated by `CACHED_MICROCOMPACT`. 4. **Context collapse:** Performs a read-time projection over the conversation history, gated by `CONTEXT_COLLAPSE`. 5. **Auto-compact:** Performs a full, model-generated summary, enabled by default. For complex workflows, Claude Code manages subagent delegation and orchestration. Subagents, which include built-in types (such as Explore, Plan, General-purpose, Claude Code Guide, Verification, and Statusline-setup) and user-defined types (defined using `.claude/agents/*.md` files), operate within isolated contexts. The delegation mechanism uses the `AgentTool` meta-tool and requires structured input defining the delegation prompt, an optional subagent type, and specific configurations for isolation mode, permission overrides, and the working directory. The system supports three isolation modes: **Worktree** (creates a temporary git worktree for filesystem-level separation), **Remote** (internal-only, launching in a remote Claude Code Remote environment), and **In-process** (shares the filesystem but operates within an isolated conversation context). The subagents return only summary-only results to the parent agent, and the parent agent manages the state by appending new boundary and summary events, keeping the transcript mostly append-only.
