# agent-architecture-design-analysis

**Parent:** [[index]]

Coding agents are systems designed to perform actions like running shell commands, editing files, and calling external services on behalf of the user. These systems are generally motivated by five human values: Human Decision Authority, Safety/Security/Privacy, Reliable Execution, Capability Amplification, and Contextual Adaptability. The architecture of Claude Code, an agentic coding tool developed by Anthropic, is analyzed against these five values and thirteen design principles, which guide the system's design choices. 

Structurally, the core of Claude Code is a simple `while-loop` that calls the model, runs tools, and repeats the cycle. However, the majority of the codebase resides in surrounding subsystems: a permission system with seven modes and an ML-based classifier; a five-layer compaction pipeline for context management; four extensibility mechanisms (Mechanical Composition Protocol/MCP, plugins, skills, and hooks); a subagent delegation and orchestration mechanism; and append-oriented session storage.

A comparison of Claude Code with OpenClaw, an independent open-source AI agent system, demonstrates how the same recurring design questions yield different architectural answers based on the deployment context. Specifically, the shift observed is from per-action safety evaluation (Claude Code) to perimeter-level access control (OpenClaw), from a single Command Line Interface (CLI) loop to an embedded runtime within a multi-channel gateway (OpenClaw), and from context-window extensions to gateway-wide capability registration.

Through source-level code analysis and architectural comparison, the analysis identifies specific design choices and trade-offs. The design philosophy mandates that the model should perform reasoning about required actions, while a dedicated harness executes those actions. This separation is a critical security measure because the model's only permitted interface to the outside world is the structured tool-use protocol, which the harness must validate before execution, ensuring the model cannot directly access the filesystem, run shell commands, or make network requests.

**Design Principles and Recurring Challenges:**

1. **Safety and Permissions:** The system's default safety posture is based on deny-first rule evaluation with human escalation. This rule set mandates that deny rules always override ask rules, which in turn override allow rules, and any unrecognized actions are escalated to the user. Multiple independent safety layers operate in parallel: the permission rules, PreToolUse hooks, the auto-mode ML classifier (when active), and optional shell sandboxing. This layering is necessary because any single layer can block an action, combining deny-first with human escalation and defense in depth. The system supports seven permission modes: `plan` (requiring explicit user approval), `default` (standard interactive use mode), `acceptEdits` (automatically approving certain filesystem shell commands like `mkdir`, `rmdir`, `touch`, `rm`, `mv`, `cp`, and `sed`, but requiring explicit approval for others), `auto` (an ML classifier evaluation), `dontAsk` (allowing no prompting while enforcing deny rules), and `bypassPermissions` (bypassing most prompts while maintaining safety-critical checks). The `auto` mode is conditionally included when the `TRANSCRIPT_CLASSIFIER` feature flag is active, while the `bubble` mode is used internally for subagent escalation.

2. **Context Management:** The primary binding resource constraint for Claude Code is the context window (200K for older models, 1M for Claude 4.6 series). To manage this, the system utilizes a five-layer compaction pipeline, which applies multiple sequential strategies before escalating. The five layers are: 
*   **Budget reduction:** Enforces per-tool-result size limits (always active). 
*   **Snip:** A lightweight trim that removes older history segments (gated by `HISTORY_SNIP`). 
*   **Microcompact:** Provides fine-grained, cache-aware compression (gated by `CACHED_MICROCOMPACT`). 
*   **Context collapse:** Performs a read-time virtual projection over the conversation history (gated by `CONTEXT_COLLAPSE`). 
*   **Auto-compact:** Performs a full, model-generated summary (enabled by default). 
This graduated compaction ensures that the system applies the least disruptive strategy first.

3. **Extensibility and State:** The architecture employs four distinct extension mechanisms: Mechanical Composition Protocol (MCP) servers, plugins, skills, and hooks. The system organizes these extensions by increasing context cost: hooks consume zero context, skills consume low context, and MCP servers consume high context. Additionally, the system uses an append-oriented state model, storing session transcripts as mostly append-only JSONL files, ensuring that context compaction never modifies or deletes previously written transcript lines. Subagent delegation reinforces this isolation, utilizing `AgentTool` to spawn isolated subagents (e.g., Explore, Plan, General-purpose, Claude Code Guide, Verification, Statusline-setup) that operate within isolated contexts and return only summary-only results to the parent agent.

**Future Directions:** The analysis identifies six open design directions for future agent systems: the observability–evaluation gap, cross-session persistence of state and relationship, harness boundary evolution, horizon scaling, external governance constraints, and reframing the evaluative lens as a design question. Furthermore, the analysis notes a major open question for agent builders: how to preserve human capability over time, treating long-term sustainability as a first-class design problem rather than a downstream evaluation metric.
