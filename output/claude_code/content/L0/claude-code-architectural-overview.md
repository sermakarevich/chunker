# claude-code-architectural-overview

**Parent:** [[content/L1/claude-code-operational-harness-design|claude-code-operational-harness-design]] — Claude Code structures its architecture around an operational harness (98.4% of code) that manages state and enforces permissions, allowing the LLM to remain a stateless completion endpoint. Safety is ensured via a multi-stage pipeline checked by the Policy Core, which handles contexts including Coordinator, Swarm worker, Speculative classifier, and Interactive. Context management uses a five-layer compaction pipeline (Budget reduction, Snip, Microcompact, Context collapse, Auto-compact), and subagents utilize the `AgentTool` meta-tool with three isolation modes: Worktree, Remote, and In-process.

Claude Code, an agentic coding tool developed by Anthropic, has an architecture designed to prioritize investment in a rich operational harness over limiting the model's inherent decision-making processes. The system’s core agent loop acts as the operational harness, while the Large Language Model (LLM) is utilized solely as a stateless completion endpoint. Analyzing the source code reveals that approximately 98.4% of Claude Code's codebase constitutes the operational harness—which includes permission gates, tool routing, and context management systems—leaving only about 1.6% for actual decision logic. This structural separation means that the architectural complexity is intended not to constrain the model's decisions, but rather to enable them. This design is guided by the 'Values over Rules' principle, which stipulates that the operational harness must create conditions that allow the model to make informed decisions rather than dictating those choices. 

To ensure reliability and control, the system employs a layered safety architecture, which passes every tool invocation through a multi-stage authorization pipeline. This pipeline is designed to operate independently of human vigilance, as Anthropic’s auto-mode analysis found that users approve roughly 93% of permission prompts, making interactive confirmation unreliable as a sole safety mechanism. The pipeline involves three primary components: declarative rules, global trust modes, and programmable hooks. The decision process proceeds through a Policy Core assessment which determines the outcome as Deny, Allow, or Ask. If the model requests a tool, the process first executes tool pre-filtering, which strips blanket-denied tools from the model's view before tool pool assembly. Next, the PreToolUse hook fires; this hook can return a `permissionDecision` (deny or ask), a `permissionDecisionReason`, or an `updatedInput` to modify tool parameters. Following this, the deny-first rule engine evaluates the request, ensuring that any deny rule always takes precedence over an allow rule. Finally, the Permission handler checks the runtime context, which can be **Coordinator** (for multi-agent coordination), **Swarm worker** (for worker agents), **Speculative classifier** (when `BASH_CLASSIFIER` is enabled and the tool is a `BashTool`), or **Interactive** (the standard user path).

Beyond safety, the system features highly detailed mechanisms for extensibility and context management. The core agent loop uses four distinct extension mechanisms: the Mechanical Composition Protocol (MCP) servers, plugins, skills, and hooks. The system organizes these extensions by increasing context cost: hooks consume zero context, skills consume low context, and MCP servers consume high context. Furthermore, the system implements a five-layer compaction pipeline to manage context pressure, as the context window is a primary binding resource constraint. These five sequential shapers are: 1) **Budget reduction**, which enforces per-tool-result size limits (always active); 2) **Snip**, a lightweight trim that removes older history segments (gated by `HISTORY_SNIP`); 3) **Microcompact**, providing fine-grained, cache-aware compression (gated by `CACHED_MICROCOMPACT`); 4) **Context collapse**, which performs a read-time projection over the conversation history; and 5) **Auto-compact**, which performs a full, model-generated summary (and is enabled by default).

To handle complex workflows, Claude Code supports subagent delegation and orchestration. Subagents, which include built-in types (e.g., Explore, Plan, General-purpose, Claude Code Guide, Verification, Statusline-setup) and custom user-defined types (using `.claude/agents/*.md` files), operate within isolated contexts. The delegation mechanism utilizes the `AgentTool` meta-tool and requires a structured input that defines the delegation prompt, an optional subagent type, and specific configurations for isolation mode, permission overrides, and the working directory. The system supports three isolation modes: **Worktree** (creating a temporary git worktree, providing filesystem-level separation), **Remote** (internal-only, launching in a remote Claude Code Remote environment), and **In-process** (sharing the filesystem but operating within an isolated conversation context). The subagents return only summary-only results to the parent agent, and the state is managed by appending new boundary and summary events, ensuring the transcript remains mostly append-only.
