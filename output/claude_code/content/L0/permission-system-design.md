# permission-system-design

**Parent:** [[content/L1/agent-architecture-and-context-management|agent-architecture-and-context-management]] — Claude Code implements a highly structured, layered architecture comprising five subsystems (surface, core, safety/action, state, backend). Its context management uses five mandatory shapers—Budget reduction, Snip, Microcompact, Context collapse, and Auto-compact—executed sequentially before every model call. Safety relies on a deny-first rule evaluation passing through seven independent gates, including sandboxing and hook-based interception, thereby ensuring resilience regardless of user vigilance.

When Claude Code executes a tool, such as running `npm test` via `BashTool` to reproduce an authentication test failure, the request passes through the permission pipeline. Every tool invocation must pass through the system's permission system, which defaults to either denying or asking the user for permission rather than silently allowing the action. This default behavior is motivated by Anthropic’s auto-mode analysis (Hughes, 2026), which found that users approve approximately 93% of permission prompts, indicating that approval fatigue renders interactive confirmation an unreliable sole safety mechanism. Therefore, the system maintains safety independently of user vigilance through architectural commitments, including deny-first evaluation, blanket-deny pre-filtering, and sandboxing, all of which operate regardless of user attentiveness. 

According to the Figure 4 overview, the permission gate design incorporates three primary mechanisms that shape policy: declarative rules, global trust modes, and programmable hooks, with each mechanism being independently configurable. The decision process flows through distinct components: the Agent receives a request that undergoes a Policy Core assessment, which determines the outcome as Deny, Allow, or Ask. The User/Auto Classifier is used to determine the execution path, while the Rules system and Tool Use mechanism are checked by the Permissions Decision layer. 

In terms of specific mechanisms, seven permission modes exist within the type definitions: 1. **plan:** Requires the model to create a detailed plan, and execution proceeds only after explicit user approval. 2. **default:** Represents the standard, interactive use mode, requiring user approval for most operations. 3. **acceptEdits:** Automatically approves edits within the working directory and certain filesystem shell commands (specifically `mkdir`, `rmdir`, `touch`, `rm`, `mv`, `cp`, `sed`), but other shell commands still require explicit approval. 4. **auto:** An ML-based classifier evaluates requests that do not pass fast-path checks, and this mode is gated by the `TRANSCRIPT_CLASSIFIER` feature flag. 5. **dontAsk:** This mode allows no prompting, but the system still enforces deny rules. 6. **bypassPermissions:** This mode bypasses most permission prompts, yet safety-critical checks and rules that are bypass-immune still apply. 7. **bubble:** This is an internal-only mode designed for subagent permission escalation to the parent terminal.

Of these seven modes, the five externally visible modes—`acceptEdits`, `bypassPermissions`, `default`, `dontAsk`, and `plan`—are defined in the `EXTERNAL_PERMISSION_MODES` array. The `auto` mode is conditionally included only when the `TRANSCRIPT_CLASSIFIER` feature flag is active, while the `bubble` mode exists only in the type union and is used internally for subagent permission escalation (as discussed in Section 8). 

Permission rules are always evaluated in deny-first order, managed by the `permissions.ts` file. The `toolMatchesRule()` function enforces that a deny rule always takes precedence over an allow rule, even if the allow rule is more specific (for example, a broad deny rule like 'deny all shell commands' cannot be overridden by a narrow allow rule like 'allow npm test'). The system's rule system supports both tool-level matching (by tool name) and content-level matching (by matching specific tool input patterns, such as `Bash(prefix:npm)`).

The seven modes collectively span a graduated autonomy spectrum, progressing from `plan` (where the user must approve all plans before execution) through `default` and `acceptEdits`, up to `bypassPermissions` (which offers minimal prompting). This gradient reflects the core design tension that as the system's autonomy increases, the system must shift its security mechanism from interactive user approval to automated safety checks.
