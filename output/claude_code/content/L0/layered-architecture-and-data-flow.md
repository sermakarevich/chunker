# layered-architecture-and-data-flow

**Parent:** [[content/L1/agent-architecture-and-context-management|agent-architecture-and-context-management]] — Claude Code implements a highly structured, layered architecture comprising five subsystems (surface, core, safety/action, state, backend). Its context management uses five mandatory shapers—Budget reduction, Snip, Microcompact, Context collapse, and Auto-compact—executed sequentially before every model call. Safety relies on a deny-first rule evaluation passing through seven independent gates, including sandboxing and hook-based interception, thereby ensuring resilience regardless of user vigilance.

The overall data flow operates along a left-to-right spine. First, a user submits a request through an interface, which enters the agent loop. The agent loop then proposes actions to the permission system; if the actions are approved, they reach the tools, which interact with the execution environment and return `tool_result` messages back to the loop. Separately, the state and persistence systems operate alongside the agent loop, recording transcripts and loading data from prior sessions.

Initialization of the application entry point, `main()` in `main.tsx`, initializes security settings (specifically including `NoDefaultCurrentDirectoryInExePath` to prevent Windows PATH hijacking), registers signal handlers for graceful shutdown, and subsequently dispatches the program to the appropriate execution mode.

The system architecture is highly layered and comprises five major subsystems: the surface layer, the core layer, the safety/action layer, the state layer, and the backend layer. 

**Surface Layer:** This layer comprises entry points and rendering components. The `src/entrypoints/` directory contains startup paths, including the Agent SDK entry with `coreTypes.ts`, `controlSchemas.ts`, and `coreSchemas.ts`. The `src/screens/` directory composes full-screen layouts, and `src/components/` provides terminal UI building blocks using the `ink` framework. The interactive CLI launches a terminal UI that features real-time streaming, permission dialogs, and progress indicators. For single-shot processing, the headless CLI (invoked via `claude -p`) creates a `QueryEngine` instance. Furthermore, the Agent SDK emits typed events using async generators.

**Core Layer:** This layer manages the agent loop and the compaction pipeline. The `queryLoop()` async generator in `query.ts` implements the iterative agent loop. This function consumes assembled context from the state layer and dispatches tool requests to the safety/action layer. Before every model call, a compaction pipeline—composed of five sequential shapers (found in `query.ts:365–453`)—manages context pressure. These five shapers are: budget reduction, snip, microcompact, context collapse, and auto-compact, reflecting details found in Sections 4.3 and 7.3.

**Safety/Action Layer:** This layer handles permission management, hooks, extensibility, tools, sandboxing, and subagents. The permission system (`permissions.ts`) executes deny-first rule evaluation, supporting up to seven permission modes (if counting internal-only bubble and feature-gated auto, found in `types/permissions.ts`). This system also incorporates an integrated auto-mode ML classifier (`yoloClassifier.ts`), which performs a two-stage fast-filter and chain-of-thought evaluation of tool safety (Section 5). A hook pipeline, defined by 27 event types (detailed in `coreTypes.ts` and `types/hooks.ts`), can block, rewrite, or annotate tool requests; specifically, 5 of these 27 hooks are safety-related, while the remaining 22 serve lifecycle and orchestration purposes (Section 6). An extensibility subsystem allows plugins and skills to register tools and hooks into the runtime. The `assembleToolPool()` function in `tools.ts` merges both built-in and Mechanical Composition Protocol (MCP)-provided tools. All approved shell commands must pass through a shell sandbox (`shouldUseSandbox.ts`) that restricts filesystem and network access independently of the permission system. Subagent spawning, managed by `AgentTool` (`AgentTool.tsx`, `runAgent.ts`), is dispatched through the same `buildTool()` factory as all other tools. Subagents re-enter the `queryLoop()` with an isolated context window and return only a summary to the parent agent (Section 8).

**State Layer:** This layer manages context assembly, runtime state, persistence, memory, and sidechains. Context assembly is explicitly defined as a memoized state loader, not a routing hub. The function `getSystemContext()` (`context.ts`) computes the session-level system context, including the git status, and `getUserContext()` (`context.ts`) loads the CLAUDE.md hierarchy and the current date. Both contexts are cached for reuse: the system context is appended to the system prompt, while the user context is added as a user-context message. The `src/state/` directory manages the runtime application state. Session transcripts are stored as mostly append-only JSONL files at project-specific paths using `sessionStorage.ts`. The CLAUDE.md + memory subsystem provides a four-level instruction hierarchy (defined in `claudemd.ts`), ranging from managed settings to directory-specific files, plus auto-memory entries that Claude writes during conversations (Section 7.2). Sidechain transcripts (in `sessionStorage.ts:247`) store each subagent’s conversation in a separate file, thereby preventing the subagent's content from inflating the parent context (Section 8.3). The global prompt history is maintained in `history.jsonl` (`history.ts`). Operations to resume or fork reconstruct the session state from transcripts using `conversationRecovery.ts`.

**Backend Layer:** This layer handles execution backends and external resources. It supports shell command execution with optional sandboxing (using `BashTool.tsx` and `PowerShellTool.tsx`), remote execution support (in `src/remote/`), and MCP server connections across multiple transport variants. These variants include stdio, SSE, HTTP, WebSocket, SDK, and IDE-specific adapters (managed in `services/mcp/client.ts`). Finally, 42 tool subdirectories located in `src/tools/` implement concrete tool logic.

**QueryEngine and Query Flow Clarification:** The documentation for `QueryEngine.ts` states that the `QueryEngine` owns the query lifecycle and session state for a conversation. It extracts the core logic from the `ask()` method into a standalone class usable by both the headless/SDK path and, potentially, the REPL. The class serves as a conversation wrapper for non-interactive surfaces, not the execution engine itself. A `QueryEngineConfig` is required for its constructor, which accepts initial messages, an abort controller, and a file-state cache, along with other per-conversation state. The `submitMessage()` method is an async generator that orchestrates a single turn. Crucially, the shared query path lives in `query()` (`query.ts`), which wraps an internal `queryLoop()`. The `QueryEngine` delegates its work to this `query()` function. This distinction is architecturally significant because the interactive CLI also calls `query()`, entirely bypassing the `QueryEngine` class.

**Safety and Permission Layers Detail:** The implementation enforces a safety-by-default principle through seven independent layers, requiring a request to pass through all applicable layers, as any single layer can block the request. These layers are: 1) **Tool pre-filtering** (`tools.ts`): Removes blanket-denied tools from the model's view before invocation, preventing the model from attempting to call them. 2) **Deny-first rule evaluation** (`permissions.ts`): Mandates that deny rules always supersede allow rules, even if the allow rule is more specific. 3) **Permission mode constraints** (`types/permissions.ts`): The active mode determines the baseline handling for requests that do not match any explicit rule. 4) **Auto-mode classifier**: An ML-based classifier evaluates tool safety, potentially denying requests that the rule system would otherwise permit. 5) **Shell sandboxing** (`shouldUseSandbox.ts`): Allows approved shell commands to execute within a sandbox that restricts filesystem and network access. 6) **Not restoring permissions on resume** (`conversationRecovery.ts`): Ensures that session-scoped permissions are not restored when a session resumes or forks. 7) **Hook-based interception** (`types/hooks.ts`): PreToolUse hooks can modify permission decisions, while PermissionRequest hooks can resolve decisions asynchronously alongside the user dialog (or before it, in coordinator mode).
