# claude-code-vs-openclaw-architecture

**Parent:** [[content/L1/agent-architecture-comparison|agent-architecture-comparison]] — Claude Code delegates to isolated subagents via AgentTool with six built-in types (Explore, Plan, etc.), while OpenClaw supports advanced multi-agent routing and customizable sub-agent delegation with up to 5 nesting depths, managing security through differing trust models (deny-first vs. single trusted operator).

The comparison contrasts Claude Code, a CLI coding harness bound to a single repository session, with OpenClaw, a persistent control plane designed for multi-channel personal assistance. These two systems occupy different regions of the agent design space, demonstrating how the same recurring design questions yield different architectural answers when the deployment context changes.

**1. System Scope and Deployment Model:** Claude Code operates as an ephemeral Command Line Interface (CLI) process constrained to a single repository. Every Claude Code session starts and ends within the terminal. In contrast, OpenClaw functions as a persistent daemon (default port 18789, running on loopback-only) that manages all messaging surface connections and coordinates clients, tools, and device nodes through a typed WebSocket protocol. This divergence in system scope is the most fundamental architectural difference, influencing all subsequent design considerations. Compositionally, OpenClaw can host Claude Code, OpenAI Codex, and Gemini CLI as external coding harnesses via its Agent Client Protocol (ACP) integration, making the two systems stackable.

**2. Trust Model and Security Architecture:** The systems address distinct threat models. Claude Code assumes an untrusted model operating within a trusted developer’s machine. Consequently, the system utilizes a deny-first permission system (detailed in Section 5), employs an ML classifier for automated safety assessment, and establishes a graduated autonomy spectrum through seven permission modes. OpenClaw, conversely, assumes a single trusted operator per gateway instance. Its security architecture focuses on identity and access control (including DM pairing codes, sender allowlists, and gateway authentication) rather than on evaluating every single action. For tool policy, OpenClaw uses configurable allow/deny lists per agent, departing from a centralized classifier. OpenClaw offers sandboxing as an opt-in feature with multiple backends (Docker, SSH, or OpenShell) and a configurable scope (per-agent, per-session, or shared); although a non-main mode can sandbox all non-main sessions when enabled, sandboxing is not active by default. The OpenClaw security documentation explicitly advises that hostile multi-tenant isolation on a shared gateway is not a supported security boundary. This highlights a core design choice: Claude Code establishes its trust boundary between the model and the execution environment, while OpenClaw places the boundary at the gateway perimeter.

**3. Agent Runtime and Tool Orchestration:** Both systems implement agentic loops, but they occupy different structural positions. In Claude Code, the `queryLoop()` asynchronous generator (Section 4) constitutes the system’s center; all interfaces feed into this loop, which directly manages context assembly, model calls, tool dispatch, and recovery. For OpenClaw, the agent runtime—an embedded Pi-agent core—resides within a larger gateway dispatch layer. The gateway’s agent Remote Procedure Call (RPC) first validates parameters and resolves sessions, returning immediately; the embedded runner then executes the agentic loop while emitting lifecycle and stream events back through the gateway protocol. To prevent tool and session races across the multi-channel surface, OpenClaw serializes all runs through per-session queues and an optional global lane. Although both systems adhere to the ReAct pattern (Yao et al., 2022), Claude Code’s loop is the control plane itself, whereas OpenClaw’s loop is a component within a larger control plane.

**4. Extension Architecture:** Claude Code utilizes four extension mechanisms: the Mechanical Composition Protocol (MCP), plugins, skills, and hooks. These mechanisms are organized by increasing context cost: hooks consume zero context, skills consume low context, and MCP servers consume high context. All four extend a single agent’s context window and tool surface. OpenClaw implements a manifest-first plugin system encompassing four architectural layers (discovery, enablement, runtime loading, and surface consumption) and twelve capability types, which include text inference, speech, media understanding, image/music/video generation, web search, and messaging channels. Plugins register capabilities into a central registry, and the gateway reads this registry to expose tools, channels, provider setups, hooks, HTTP routes, CLI commands, and services. OpenClaw also includes a separate skills layer with multiple sources (workspace, project-level, personal, managed, and bundled directories, with workspace skills having the highest precedence) and provides a public registry (ClawHub). Additionally, OpenClaw supports MCP through built-in `openclaw-mcp` commands (which manage both server and outbound client registries). The key architectural difference is that Claude Code’s extensions modify one single agent’s action surface, while OpenClaw’s plugins extend the gateway’s overall capability surface across all agents.

**5. Memory, Context, and Knowledge Management:** Both systems prioritize using transparent file-based memory rather than opaque databases. Claude Code loads a four-level CLAUDE.md hierarchy and manages context pressure using a five-layer compaction pipeline (Section 7). For memory retrieval, Claude Code employs an LLM-based scan of file headers. OpenClaw, conversely, injects workspace bootstrap files into the system prompt at the start of a session: five core files (AGENTS.md, SOUL.md, TOOLS.md, IDENTITY.md, USER.md), plus conditionally BOOTSTRAP.md, HEARTBEAT.md, and MEMORY.md, with large files being truncated. The separate memory system manages three file types: MEMORY.md for long-term durable facts, date-stamped daily notes (formatted as `memory/YYYY-MMDD.md`), and an optional DREAMS.md for summarizing background consolidation sweeps. When a user configures an embedding provider, memory search utilizes hybrid retrieval, combining vector similarity with keyword matching. An experimental dreaming system performs background consolidation, scoring candidates and promoting only qualified items from short-term recall into long-term memory. Furthermore, before compaction, OpenClaw automatically reminds the agent to save important notes to memory files, thus preventing context loss. Both systems share the commitment of keeping memory user-visible and editable, but OpenClaw invests more heavily in structured long-term memory promotion (dreaming, daily notes, and memory search), while Claude Code focuses more on graduated context compression via its five-layer system. OpenClaw also supports pluggable compaction providers and session pruning, although its compaction pipeline is less graduated than Claude Code’s five-layer system.

**6. Multi-agent Architecture and Routing:** This dimension reveals the starkest divergence. Claude Code implements multi-agent functionality through task delegation: the parent agent spawns subagents (such as Explore, Plan, general-purpose, and custom types) that operate within isolated context windows and are restricted to specific tool sets, returning only summary-only results. Worktree isolation provides filesystem-level separation for these subagents. OpenClaw separates two distinct concerns. First, it handles multi-agent routing, allowing a single gateway to host multiple fully isolated agents—each with its own workspace, authentication profile, session store, and model configuration—and routes them to specific channels or senders via deterministic binding rules. Second, it manages sub-agent delegation, which allows background runs within a single agent to be spawned with a configurable nesting depth (maximum 5, default 1, recommended 2), thread-bound sessions on supported channels, and configurable tool policy by depth. OpenClaw’s project vision explicitly rejects using agent-hierarchy frameworks as a default architecture. This distinction is critical because Claude Code’s subagents are subordinate workers within one user’s coding session, whereas OpenClaw’s multi-agent routing creates genuinely independent agent instances serving different users or purposes through different channels.
