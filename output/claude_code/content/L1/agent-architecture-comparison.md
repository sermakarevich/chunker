# agent-architecture-comparison

**Parent:** [[index]]

The fields of advanced AI agent systems—including Claude Code and OpenClaw—are defined by complex trade-offs in architecture, trust modeling, context management, and orchestration. These systems, while addressing common goals like task automation and long-term memory, utilize fundamentally divergent approaches rooted in their deployment contexts: Claude Code operates as an ephemeral, single-repository CLI/IDE coding harness, whereas OpenClaw functions as a persistent, multi-channel WebSocket gateway daemon designed for personal assistance across diverse messaging services.

**Agent Orchestration and Delegation:**

Claude Code’s multi-agent workflow is based on task delegation, employing a robust `AgentTool` mechanism (which formerly used `Task` as an alias) to launch specialized, isolated subagents. The model initiates this process by invoking `Agent` using a structured input that must specify the delegation prompt, an optional subagent type, and configurations for isolation mode, permission overrides, and the target working directory. The core principle here is the isolated subagent boundaries principle. Built-in subagent types include six dedicated roles: **Explore** (a read/search-oriented investigation with a deny-list for write/edit tools), **Plan** (for creating structured plans following standard permission models), **General-purpose** (a broadly capable type used when explicitly requested, noting that omitting the type might default to the `fork-subagent` path), **Claude Code Guide** (for onboarding and documentation, possessing its own `permissionMode` override), **Verification** (for executing checks like linting or test suites), and **Statusline-setup** (specialized for terminal status configuration). Beyond these, users can define custom subagents using `.claude/agents/*.md` files, or contribute them via `loadPluginAgents.ts`. Custom agent definitions are highly configurable: the markdown body serves as the system prompt, while the YAML frontmatter allows specification of description, `tools` (allowlist), `disallowedTools`, `model`, `effort`, `permissionMode`, `mcpServers`, `hooks`, `maxTurns`, `skills`, `memory` scope, `background` flag, and `isolation` mode. Furthermore, JSON-formatted agent definitions can support the same fields plus an explicit prompt via `loadAgentsDir.ts`. Both `AgentTool` and `SkillTool` exist as meta-tools, but they are distinct: `SkillTool` injects instructions into the current context window, whereas `AgentTool` spawns a new, isolated process. Consequently, most subagent calls demand a self-contained prompt because the default path does not inherit the parent’s full conversation history, though the `fork-subagent` path is an exception. For multi-agent routing, OpenClaw’s architecture is vastly different. It separates two distinct concerns: (a) **multi-agent routing**, which enables a single gateway to manage multiple, fully isolated agents, each possessing its own workspace, authentication profile, session store, and model configuration, routing them via deterministic binding rules; and (b) **sub-agent delegation**, which allows background runs within a single agent to be spawned. OpenClaw offers configurable nesting depth (defaulting to 1, maximum 5, recommended 2) and thread-bound sessions on supported channels, explicitly rejecting agent-hierarchy frameworks as its default model. This means that while Claude Code's subagents are subordinate workers within one coding session, OpenClaw's multi-agent framework supports genuinely independent agent instances for different users or purposes through different communication channels.

**Isolation and Permissions:**

Claude Code implements subagent isolation through multiple modes defined in `AgentTool.tsx`: 1) **Worktree** (creates a temporary Git worktree, giving the subagent its own copy of the repository to prevent parent worktree contamination); 2) **Remote** (internal-only, launching in a remote Claude Code Remote environment, which always runs asynchronously in the background); 3) **In-process** (the default mode, sharing the filesystem but maintaining conversational isolation). Permission control is highly regulated (managed in `runAgent.ts`). A subagent's defined `permissionMode` override is respected unless the parent agent is already operating in `bypassPermissions`, `acceptEdits`, or `auto` mode, as these parent modes signify explicit user decisions about autonomy. The system uses a two-tier permission scoping model when `allowedTools` is set via `runAgent()`, preserving SDK-level permissions derived from the `--allowedTools` flag. For asynchronous operations, a cascade exists to determine if prompts are avoided: checking `explicit canShowPermissionPrompts`, then `bubble` mode (which always prompts due to parent terminal escalation), and finally the `default` mode (synchronous agents prompt, but asynchronous ones do not). OpenClaw adopts a trust model that assumes a single trusted operator per gateway instance, focusing on identity verification (like DM pairing codes and sender allowlists) rather than continuous, granular action evaluation. OpenClaw supports opt-in sandboxing with multiple backends (Docker, SSH, OpenShell) and configurable scope (per-agent, per-session, or shared), although hostiles multi-tenant isolation on a shared gateway is not a supported security boundary. Claude Code, conversely, establishes its trust boundary specifically between the LLM and the execution environment, utilizing a deny-first system bolstered by an ML classifier and seven defined permission modes, providing a graduated trust spectrum.
