# agent-architecture-and-context-management

**Parent:** [[index]]

The architecture of Claude Code is meticulously layered, comprising five major subsystems—the surface layer, the core layer, the safety/action layer, the state layer, and the backend layer—to ensure robust functionality and comprehensive safety. This design ensures that the overall data flow progresses along a strict left-to-right spine: a user submission enters the agent loop, which proposes actions, which must pass through the permission system, and if approved, reach the tools for execution in the environment, returning `tool_result` messages back to the loop. The entire system initiation is handled in `main()` within `main.tsx`, which initializes security settings, specifically including `NoDefaultCurrentDirectoryInExePath` to prevent Windows PATH hijacking, registers signal handlers for graceful shutdown, and then dispatches the program to the relevant execution mode.

**Surface and Core Functionality:**
The system's user-facing components reside in the `src/entrypoints/` directory, containing startup paths such as the Agent SDK entry (which utilizes `coreTypes.ts`, `controlSchemas.ts`, and `coreSchemas.ts`). The `src/screens/` directory composes full-screen layouts, while `src/components/` provides terminal UI building blocks using the `ink` framework. Interaction can occur through a specialized interactive CLI, which features real-time streaming, permission dialogs, and progress indicators, or via a headless CLI (invoked as `claude -p`) which instantiates a `QueryEngine` instance. For programmatic use, the Agent SDK emits typed events using async generators. The core agent logic is managed by the `queryLoop()` async generator in `query.ts`. This function implements the iterative agent loop, consuming assembled context from the state layer and dispatching tool requests to the safety/action layer. This generator-based design facilitates streaming output to the User Interface (UI) layer while maintaining a single, synchronous control flow within the loop.

**Safety, Actions, and Execution:**
The system employs a highly redundant, safety-by-default principle, requiring an action to pass through multiple independent layers. The `permissions.ts` file enforces a deny-first rule evaluation, where deny rules always supersede allow rules, even if the allow rule is more specific (e.g., 'deny all shell commands' overrides 'allow npm test'). This system supports up to seven permission modes: `plan` (requires explicit user approval for all plans), `default` (standard interactive mode requiring most user approvals), `acceptEdits` (automatically approves edits in the working directory and specific filesystem commands like `mkdir`, `rmdir`, `touch`, `rm`, `mv`, `cp`, `sed`, but requires approval for others), `auto` (an ML-based classifier evaluating requests that fail fast-path checks, gated by `TRANSCRIPT_CLASSIFIER`), `dontAsk` (allows no prompting but enforces deny rules), `bypassPermissions` (bypasses most prompts, but safety-critical checks still apply), and `bubble` (an internal mode for subagent escalation). Of these, the five externally visible modes are `acceptEdits`, `bypassPermissions`, `default`, `dontAsk`, and `plan`. The decision process involves a Policy Core assessment, which outputs Deny, Allow, or Ask. The flow also includes the `auto` mode and the `bubble` mode, controlled by the `TRANSCRIPT_CLASSIFIER` flag. Tool invocation must pass through the safety checks: 1) **Tool pre-filtering** (`tools.ts`) removes blanket-denied tools; 2) **Deny-first rule evaluation** (`permissions.ts`); 3) **Permission mode constraints** (`types/permissions.ts`); 4) **Auto-mode classifier** (ML-based evaluation); 5) **Shell sandboxing** (`shouldUseSandbox.ts`) restricts filesystem and network access for approved shell commands; 6) **Not restoring permissions on resume** (`conversationRecovery.ts`); and 7) **Hook-based interception** (`types/hooks.ts`) which allows PreToolUse hooks to modify decisions and PermissionRequest hooks to resolve them asynchronously. Because any single layer can block an action, the system adheres to defense in depth.

**Tool Execution and Subagents:**
When the model response contains `tool_use` blocks, the system selects an execution path. The primary path uses `StreamingToolExecutor`, which executes tools as they stream in, minimizing latency for multiple tools. The fallback path uses `runTools()` from `toolOrchestration.ts`. The `StreamingToolExecutor` manages concurrency using two mechanisms: a **Sibling abort controller**, which terminates all other in-flight subprocesses if one Bash tool encounters an error; and a **Progress-available signal**, which wakes the `getRemainingResults()` consumer when output is available. Tools are classified as either concurrent-safe or exclusive, and state-modifying operations (like shell commands) are serialized, while read-only operations can run concurrently. Subagent delegation is managed by `AgentTool` (`AgentTool.tsx`, `runAgent.ts`) and dispatched through the main `buildTool()` factory, which is the same factory used for all other tools. Subagents re-enter the `queryLoop()` with an isolated context window and only return a summary to the parent agent, maintaining separation of concerns.

**Context Management and Compaction Pipeline:**
Context window management is a multi-stage process ensuring the system remains functional even with large histories. Before every model call, five distinct context-reduction shapers execute sequentially within `query.ts` on the `messagesForQuery` array. These shapers are: 1) **Budget reduction** (`applyToolResultBudget()`): Implements specific per-message size limits on tool results, replacing oversized outputs with content references. This must run before microcompact. 2) **Snip** (`snipCompactIfNeeded()`): A lightweight trim that removes older history segments, explicitly tracking the `snipTokensFreed` value. 3) **Microcompact:** Provides fine-grained compression, running a time-based path and optionally a cache-aware path (gated by `CACHED_MICROCOMPACT`). It returns `{messages, compactionInfo}`.

4. **Context collapse** (`applyCollapsesIfNeeded()`): Performs a read-time projection over the conversation history. The resulting summary view is visible via `messagesForQuery`, but the full history remains intact in a dedicated collapse store, ensuring the REPL array is not mutated. 5. **Auto-compact** (`compactConversation()`): The final, most comprehensive shaper. It executes only if the context still exceeds the predefined pressure threshold after the preceding four shapers. This function involves PreCompact hooks, a summary request via `getCompactPrompt()`, and a final model call to produce a compressed summary, which is then integrated via `buildPostCompactMessages()`.

**State Persistence and Backend:**
The system manages runtime state through the State Layer, which handles context assembly and persistence. Context assembly is a memoized state loader, not a routing hub. `getSystemContext()` computes session context, including git status, and `getUserContext()` loads the CLAUDE.md hierarchy and the current date. Both are cached: system context appends to the system prompt, and user context is added as a user-context message. Session transcripts are stored in mostly append-only JSONL files at project-specific paths using `sessionStorage.ts`. The CLAUDE.md + memory subsystem defines a four-level instruction hierarchy (from managed settings to directory-specific files) and includes auto-memory entries written by Claude. Sidechain transcripts, stored in `sessionStorage.ts:247`, are crucial for isolating subagent conversations from inflating the parent context. The global prompt history is maintained separately in `history.jsonl` (`history.ts`). Recovery, for resuming or forking a session, is managed by `conversationRecovery.ts`. The Backend Layer supports execution backends including shell command execution (via `BashTool.tsx` and `PowerShellTool.tsx`) with optional sandboxing, remote execution (`src/remote/`), and connections to MCP servers across multiple variants: stdio, SSE, HTTP, WebSocket, SDK, and IDE-specific adapters (managed in `services/mcp/client.ts`).

**Query Flow and Error Handling:**
While `QueryEngine.ts` owns the query lifecycle and session state, it is important to note that the interactive CLI calls `query()` (`query.ts`) directly, bypassing the `QueryEngine` class, which is reserved for non-interactive surfaces. The `submitMessage()` method of `QueryEngine` is an async generator orchestrating a single turn. The `queryLoop()` function is the central orchestrator. Termination conditions are comprehensive, including reaching `maxTurns`, receiving a `hook_stopped_continuation` flag, the `abortController` signal firing, or context overflow. If an API response fails due to `prompt_too_long`, the loop first attempts context collapse overflow recovery, then reactive compaction. Only if both fail does the system terminate with the specific error: `'prompt_too_long'`.

**Tool Dispatch Detail:**
The tool result collection process handles updates containing tool results, attachments, or progress events. A specific check for `hook_stopped_continuation` attachments sets a `shouldPreventContinuation` flag. Results are then normalized for the Anthropic API using `normalizeMessagesForAPI()`, which filters out all non-user-type messages. Regarding tool execution, if the model is using `tool_use` blocks, the system either streams results using `StreamingToolExecutor` or uses the synchronous `runTools()` generator. The architecture ensures that the output order remains consistent even when tools run in parallel, as the results are buffered and emitted in the sequence they were originally received.
