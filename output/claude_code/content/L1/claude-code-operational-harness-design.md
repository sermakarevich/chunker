# claude-code-operational-harness-design

**Parent:** [[index]]

The architectural philosophy of Claude Code establishes a foundation where a rich, deterministic operational harness is prioritized over limiting the Large Language Model's (LLM) inherent decision-making processes. This approach positions the core agent loop as the primary operational harness, which manages state, enforces permissions, and orchestrates workflows, while the LLM itself is utilized solely as a stateless completion endpoint. An analysis of the codebase reveals a structural imbalance: approximately 98.4% of Claude Code's codebase constitutes this operational infrastructure—encompassing permission gates, tool routing, and context management systems—leaving only about 1.6% dedicated to actual decision logic. This structural design choice signifies that the complexity is not meant to constrain the model's choices, but rather to enable and facilitate them, adhering to the design principle of 'Values over Rules,' which mandates that the operational environment must facilitate informed choices rather than dictating them.

To ensure reliability and maintain control within this highly autonomous model environment, the system implements a multi-layered safety architecture. This safety pipeline passes every tool invocation through a series of checks, designed to operate independently of human vigilance. This necessity arises because Anthropic's auto-mode analysis indicated that users approve roughly 93% of permission prompts, rendering interactive confirmation an unreliable sole safety measure. The core decision process traverses three primary components: declarative rules, global trust modes, and programmable hooks. When a tool is requested, the flow begins with a Policy Core assessment, which determines the outcome as Deny, Allow, or Ask. This is followed by tool pre-filtering, which actively strips any tools that have been globally denied from the model's available view, before the tool pool is assembled. Next, the PreToolUse hook fires, providing a potential decision—either a `permissionDecision` (deny or ask), a `permissionDecisionReason`, or an `updatedInput` to modify the tool's parameters. The subsequent step involves the deny-first rule engine, ensuring that any explicit deny rule takes precedence over any allow rule. The final check is performed by the Permission handler, which verifies the runtime context, which can be one of four specific modes: **Coordinator** (used for multi-agent coordination), **Swarm worker** (for worker agents), **Speculative classifier** (activated when `BASH_CLASSIFIER` is enabled and the tool invoked is a `BashTool`), or **Interactive** (representing the standard user path). 

Beyond safety, Claude Code features sophisticated mechanisms for extensibility and context management. The system incorporates four distinct extension mechanisms: Mechanical Composition Protocol (MCP) servers, plugins, skills, and hooks. These mechanisms are systematically organized by their respective context consumption cost, creating a clear architectural gradient: hooks consume zero context; skills consume low context; and MCP servers are the most resource-intensive, consuming high context. Furthermore, managing context pressure is critical due to the context window acting as a primary binding resource constraint. Claude Code employs a five-layer compaction pipeline, consisting of five sequential shapers: 1) **Budget reduction**, which is always active and enforces strict per-tool-result size limits; 2) **Snip**, a lightweight trimming process that removes older history segments (which is gated by a `HISTORY_SNIP` flag); 3) **Microcompact**, a fine-grained, cache-aware compression method (gated by `CACHED_MICROCOMPACT`); 4) **Context collapse**, which performs a specialized read-time projection over the conversation history; and 5) **Auto-compact**, which executes a full, model-generated summary and is enabled by default. 

To manage complex, multi-step workflows, Claude Code supports granular subagent delegation and orchestration. Subagents—which include several built-in types such as Explore, Plan, General-purpose, Claude Code Guide, Verification, and Statusline-setup, as well as custom user-defined types created via `.claude/agents/*.md` files—operate within isolated and dedicated contexts. The delegation process is formalized using the `AgentTool` meta-tool and demands a structured input. This input must define the delegation prompt, an optional subagent type, and specific configurations for isolation mode, permission overrides, and the working directory. The system provides three defined isolation modes to ensure separation of concerns: **Worktree** (this method creates a temporary git worktree, thereby ensuring filesystem-level separation); **Remote** (designed for internal use, launching within a remote Claude Code Remote environment); and **In-process** (which shares the filesystem while operating within an isolated conversation context). A key constraint managed by this delegation system is that subagents return only summary-only results to the parent agent, and the state is maintained by appending new boundary and summary events, thereby keeping the conversational transcript largely append-only. The entire design underscores an operating system-like abstraction, where the core agent loop functions as the kernel, and all other subsystems constitute the supporting operating system infrastructure, prioritizing the model's autonomy while providing robust, deterministic control layers for execution and safety.
