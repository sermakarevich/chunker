# agent-architecture-mechanisms

**Parent:** [[index]]

The architecture of the Claude Code agent is defined by a deeply layered, multi-stage safety, control, and extensibility framework, designed to balance high autonomy with granular control. This design is fundamentally separated into three domains: the Permission and Authorization Pipeline, the mechanism of Context Window Assembly, and the methods of Extension Surface management.

### Permission and Authorization Pipeline

The system adopts a ‘deny-first’ evaluation approach for all tool invocations, ensuring that a denial rule always takes precedence over an allow rule, even if the allow rule is more specific (e.g., a broad deny rule like 'deny all shell commands' cannot be overridden by a narrow allow rule like 'allow npm test'). This enforcement is managed primarily by the `permissions.ts` file. The overall policy gate is shaped by three independently configurable mechanisms: declarative rules, global trust modes, and programmable hooks.

The flow begins with **Pre-filtering**, where the `filterToolsByDenyRules()` function in `tools.ts` strips blanket-denied tools from the model’s view entirely during tool pool assembly, preventing the model from attempting forbidden calls. The core decision process then proceeds through several stages: 1) **PreToolUse Hooks**, which can execute to return a `permissionDecision` (deny or ask), a `permissionDecisionReason`, or provide an `updatedInput` to modify the parameters. These hooks run regardless of subsequent rule-based denials. 2) **Rule Evaluation**, where the deny-first rule engine assesses the request. 3) **Permission Handler**, which branches execution into four runtime paths: **Coordinator** (for multi-agent coordination, attempting automated resolution first), **Swarm Worker** (for participating agents), **Speculative Classifier** (when `BASH_CLASSIFIER` is enabled and the tool is `BashTool`, allowing instant approval if high confidence is returned), and **Interactive** (the standard fallback path requiring user approval). 

In the automated paths (Coordinator and background agents), the system prioritizes automated resolution over immediate user interaction. When a classifier or a deny rule blocks an action, the denial is treated as a routing signal, not a hard stop; the model receives the denial reason, revises its approach, and attempts a safer alternative in the next loop iteration. This is facilitated by the `PermissionDenied` hook event, enabling external code to observe and respond to denials programmatically, yielding a recovery-oriented design.

Five distinct permission modes govern the system's autonomy gradient: 1. **plan** (requiring explicit user approval for all plans); 2. **default** (the standard, interactive mode); 3. **acceptEdits** (automatically approving edits within the working directory and specific shell commands: `mkdir`, `rmdir`, `touch`, `rm`, `mv`, `cp`, `sed`, but requiring approval for others); 4. **auto** (an ML-based classifier evaluation, gated by the `TRANSCRIPT_CLASSIFIER` flag); 5. **dontAsk** (allowing no prompting while still enforcing deny rules). Additionally, `bypassPermissions` offers minimal prompting, while `bubble` is an internal-only mode for subagent escalation.

The Auto-Mode Classifier (`yoloClassifier.ts`), utilized when `TRANSCRIPT_CLASSIFIER` is active, evaluates the proposed tool invocation against the conversation transcript and the permission template, providing an allow, deny, or manual request. The architectural commitment to safety is layered: for instance, the system uses shell sandboxing—a protective layer for Bash and PowerShell commands (managed by `shouldUseSandbox.ts`)—which provides filesystem and network isolation independently of the application-level permission model. The two systems are independent; a command can be approved but still sandboxed, or denied and never reach the sandbox check. This layered structure is critical, though researchers at Adversa.ai noted a functional degradation when defense layers share failure modes: commands with over 50 subcommands force a single generic approval prompt instead of per-subcommand deny-rule checks, highlighting the structural tension between safety and performance.

### Context Window Assembly and Management

The context window is not static; its composition is managed by a multi-step process that can inject information both at startup and dynamically during the conversation turn. 

**Initial Assembly Sources:**
During initialization, the context receives input from: 1) **System Prompt** (incorporating optional output style modifications); 2) **Environment Info** (via `getSystemContext()`, which retrieves git status and an optional cache-breaking injection, memoized per session); 3) **CLAUDE.md Hierarchy** (via `getUserContext()`, loading a four-level instruction file hierarchy). The CLAUDE.md files are plaintext Markdown, prioritizing auditability. These files define four memory types: **Managed memory** (`/etc/claude-code/CLAUDE.md` for OS-level policy); **User memory** (`~/.claude/CLAUDE.md` for private global instructions); **Project memory** (found in project roots: `CLAUDE.md`, `.claude/CLAUDE.md`, and `.claude/rules/*.md`); and **Local memory** (`CLAUDE.local.md` for gitignored, private project instructions). The discovery process traverses from the CWD up to the root, with files closer to the CWD having higher priority, and the system loads them in a 'reverse order of priority,' meaning later-loaded files receive more model attention. For root-to-CWD directories, unconditional rules from `.claude/rules/*.md` load eagerly; for nested directories below CWD, even unconditional rules are loaded lazily, allowing the model’s instruction set to evolve as the codebase is explored. Crucially, the CLAUDE.md content is delivered as user context, making model compliance probabilistic rather than guaranteed, contrasting with the deterministic enforcement of permission rules. Additionally, the context incorporates **Path-scoped rules** (conditional, directory-matched, and loaded lazily), **Auto memory** (asynchronously prefetched relevant entries), **Tool metadata** (skill descriptions, MCP tool names, and deferred tool definitions), **Conversation history**, **Tool results**, and **Compact summaries**.

**Turn-Based Composition and Compaction:**
Context window composition proceeds in sequence: 1) **System Prompt Assembly** (`query.ts`) combines the system context with the base prompt, prepending the user context (CLAUDE.md and date) via `prependUserContext()`; 2) **Late Injections** inject relevant-memory prefetch, MCP instructions deltas (only new/changed server instructions), agent listing deltas, and background agent task notifications, meaning the context grows during the turn. The system manages context pressure using a five-layer compaction pipeline, designed as a lazy-degradation principle: 1) **Budget reduction** (always active, enforcing per-tool-result size limits); 2) **Snip** (gated by `HISTORY_SNIP`, a lightweight trim removing older history segments); 3) **Microcompact** (gated by `CACHED_MICROCOMPACT`, providing cache-aware compression); 4) **Context collapse** (gated by `CONTEXT_COLLAPSE`, performing a read-time virtual projection over history); and 5) **Auto-compact** (user-configurable, performing a full, model-generated summary). The process of compaction, handled by `buildPostCompactMessages()` in `compact.ts`, is append-only, appending new boundary and summary events while preserving the integrity of previous transcript lines by annotating preserved-segment metadata with `headUuid`, `anchorUuid`, and `tailUuid`.

### Extensibility Surface: Layers of Control

The agent supports highly composable, multi-mechanism extensibility, providing methods for the model to see what it can and how it can act. The extension surface is primarily built upon four mechanisms and separate agent definitions.

**1. MCP Servers (Model Context Protocol):** This is the main external tool integration path. Servers are configured across multiple scopes (project, user, local, enterprise, plus plugin and claude.ai servers). The Mechanical Composition Protocol client (`services/mcp/client.ts`) supports multiple transport types, including stdio, SSE, HTTP, WebSocket, SDK, and IDE-specific variants (`sse-ide`, `ws-ide`). MCP tools are merged into a single flat tool pool and are the first mechanism encountered during context assembly, alongside other context sources. 

**2. Plugins:** Functioning as both a packaging format and distribution mechanism, a plugin manifest (`utils/plugins/schemas.ts`) accepts ten component types: commands, agents, skills, hooks, MCP servers, LSP servers, output styles, channels, settings, and user configuration. The `pluginLoader` validates this manifest and routes each component to its dedicated registry: skills/commands use the `SkillTool` meta-tool; agents are consumed by `AgentTool`; hooks register in the hook registry; MCP/LSP servers are added to their standard configurations; and output styles modify response formatting. Plugins are the primary distribution vehicle, allowing a single package to extend the agent across multiple component types simultaneously.

**3. Skills:** Defined by a `SKILL.md` file with YAML frontmatter, skills can advertise numerous fields (display name, description, allowed tools, argument hints, model overrides, execution context like 'fork', associated agent definitions, effort levels, and shell configuration). When invoked, the `SkillTool` meta-tool injects the skill's instructions into the context. Skills can also define their own dynamic hooks, registering them upon invocation. 

**4. Hooks:** The source defines 27 distinct, well-documented hook events (`coreTypes.ts`). Five of these participate directly in the permission flow: `PreToolUse`, `PostToolUse`, `PostToolUseFailure`, `PermissionRequest`, and `PermissionDenied`. These hooks are robustly typed with specific Zod-validated output schemas (`types/hooks.ts`) supporting rich capabilities such as context injection, input modification, and result transformation (`updatedMCPToolOutput`). Other hooks cover session management (e.g., `SessionStart`, `SessionEnd`), user interaction (e.g., `UserPromptSubmit`), subagent lifecycle (e.g., `SubagentStart`, `TaskCreated`), and context management (e.g., `PreCompact`, `PostCompact`). Users configure persistent hook commands through settings and plugins using four types: shell commands (`type: command`), LLM prompt hooks (`type: prompt`), HTTP hooks (`type: http`), and agentic verifier hooks (`type: agent`).

**5. Tool Pool Assembly:** The `assembleToolPool()` function acts as the single source of truth. It involves five steps: 1) **Base tool enumeration** (`getAllBaseTools()`), which returns up to 54 tools (including 19 always-included tools like BashTool, FileReadTool, AgentTool, SkillTool, and 35 conditional tools); 2) **Mode filtering** (`getTools()`), which applies runtime checks (e.g., limiting to Bash, Read, Edit in `CLAUDE_CODE_SIMPLE` mode); 3) **Deny rule pre-filtering** (`filterToolsByDenyRules()`), stripping banned tools; 4) **MCP tool integration** (filtering and merging MCP tools); 5) **Deduplication** (built-in tools override MCP tools). This ensures consistency across all execution paths (REPL.tsx, AgentTool.tsx). Agent-based extensions, which create new, isolated context windows, are kept conceptually separate from these core four extension mechanisms.
