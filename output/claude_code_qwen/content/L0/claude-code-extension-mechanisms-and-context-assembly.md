# claude-code-extension-mechanisms-and-context-assembly

**Parent:** [[content/L1/claude-code-extension-context-subagent-architecture|claude-code-extension-context-subagent-architecture]] — Claude Code's architecture integrates four extension mechanisms (MCP, plugins, skills, hooks) with a 54-tool assembly pipeline, a 9-source context window, a 4-level CLAUDE.md memory hierarchy, and a 5-layer graduated compaction pipeline to manage bounded context. Subagent delegation uses worktree isolation and summary-only returns to prevent context explosion, while session persistence relies on mostly-append JSONL transcripts and explicit permission rebinding across resume/branch operations. Comparative analysis with OpenClaw reveals opposing architectural bets: Claude Code prioritizes ephemeral CLI execution, deny-first per-action safety, and single-agent context extension, whereas OpenClaw operates as a persistent gateway daemon emphasizing perimeter identity control, multi-agent routing, and shared plugin registries.

Claude Code integrates four distinct extension mechanisms that plug into the agent loop at specific injection points: context assembly controls what the model sees, model configuration controls what the model can reach, and execution controls whether and how an action runs.

MCP servers provide external service integration. Each connected server contributes tool definitions as MCPTool objects to an internal claudeai-proxy. Dedicated built-in tools ListMcpResourcesTool and ReadMcpResourceTool grant access to MCP resources.

Plugins function simultaneously as a packaging format and a distribution mechanism. The PluginManifestSchema (utils/plugins/schemas.ts) accepts exactly ten component types: commands, agents, skills, hooks, MCP servers, LSP servers, output styles, channels, settings, and user configuration. The plugin loader (utils/plugins/pluginLoader.ts) validates these manifests and routes each component to its respective registry: commands and skills surface through the SkillTool meta-tool, agents appear in definitions consumed by AgentTool, hooks merge into the hook registry, and MCP and LSP servers fold into their standard configurations. Output styles modify response formatting. A single plugin package can extend the system across multiple component types simultaneously, making plugins the primary distribution vehicle for third-party extensions.

Skills are defined by a SKILL.md file containing YAML frontmatter. The parseSkillFrontmatterFields() function (loadSkillsDir.ts) parses more than fifteen fields, including display name, description, allowed tools (which grant the skill access to additional tools), argument hints, model overrides, execution context ('fork' for isolated execution), associated agent definitions, effort levels, and shell configuration. Skills can define their own hooks, which register dynamically upon invocation. Bundled skills register in-memory at startup. When invoked, the SkillTool meta-tool injects the skill's instructions into the context.

The system defines twenty-seven hook events spanning tool authorization (PreToolUse, PostToolUse, PostToolUseFailure, PermissionRequest, PermissionDenied), session lifecycle (SessionStart, SessionEnd, Setup, Stop, StopFailure), user interaction (UserPromptSubmit, Elicitation, ElicitationResult), subagent coordination (SubagentStart, SubagentStop, TeammateIdle, TaskCreated, TaskCompleted), context management (PreCompact, PostCompact, InstructionsLoaded, ConfigChange), workspace events (CwdChanged, FileChanged, WorktreeCreate, WorktreeRemove), and notifications (coreTypes.ts, coreSchemas.ts). Fifteen of these events have event-specific output schemas with rich fields supporting permission decisions, context injection, input modification, MCP result transformation, and retry control (types/hooks.ts). Persisted hook commands configured via settings and plugins utilize four command types: shell commands (type: command), LLM prompt hooks (type: prompt), HTTP hooks (type: http), and agentic verifier hooks (type: agent) (schemas/hooks.ts). The runtime also supports non-persistable callback hooks (type: callback) used by the SDK and internal instrumentation (types/hooks.ts). Hook sources include settings.json, plugins, and managed policy at startup; skill hooks register dynamically upon invocation (utils/hooks.ts).

Tool Pool Assembly
The assembleToolPool() function located at tools.ts serves as the single source of truth for combining built-in tools with MCP tools. The assembly follows a five-step pipeline:
1. Base tool enumeration: getAllBaseTools() (tools.ts) returns an array of up to fifty-four tools. Nineteen tools are always included (such as BashTool, FileReadTool, AgentTool, SkillTool), and thirty-five more are conditionally included based on feature flags, environment variables, and user type. Anthropic-internal users receive additional internal tools. Worktree mode enables EnterWorktreeTool and ExitWorktreeTool. Agent swarms enable team tools. When embedded search tools are available in the Bun binary, dedicated GlobTool and GrepTool are omitted.
2. Mode filtering: getTools() (tools.ts) applies mode-specific filtering. In CLAUDE_CODE_SIMPLE mode, only Bash, Read, and Edit are available (or REPLTool in the REPL branch; plus coordinator tools if applicable). Each tool's isEnabled() method is called for runtime availability checks.
3. Deny rule pre-filtering: filterToolsByDenyRules() (tools.ts) strips blanket-denied tools from the model's view before any call.
4. MCP tool integration: MCP tools from appState.mcp.tools are filtered by deny rules and merged with built-in tools.
5. Deduplication: Tools are deduplicated by name, with built-in tools taking precedence over MCP tools.

Both REPL.tsx (via the useMergedTools hook) and AgentTool.tsx (when building the worker tool set) invoke this function, ensuring consistent assembly across all execution paths. At request time, deferred tools may be hidden from the model's context until explicitly queried via ToolSearch (tools.ts). Agent-based extension via custom agent definitions (.claude/agents/*.md) and plugin-contributed agents creates new, isolated context windows rather than extending the current one.

Rationale for Four Mechanisms
Because each additional extension mechanism increases the developer learning surface, the architecture intentionally uses four distinct mechanisms rather than consolidating them. The design recognizes that different kinds of extensibility impose different costs on the context window. A single mechanism cannot span the full range from zero-context lifecycle hooks to schema-heavy tool servers without forcing unnecessary trade-offs on extension authors. Each mechanism trades deployment complexity for a different kind of extensibility, with specific context costs and insertion points: MCP servers provide high-cost runtime tool integration at the model() tool pool insertion point; Plugins provide medium-cost multi-component packaging across all three insertion points; Skills provide low-cost domain-specific instructions and meta-tool invocation at the assemble() context injection point; Hooks provide zero-cost-by-default cross-cutting lifecycle control at the execute() pre/post tool insertion point. The graduated context-cost ordering means cheap extensions scale widely without exhausting the context window, while expensive ones reserve capacity for cases requiring new tool surfaces. This approach accommodates a broader range of extension patterns than single or two-tier frameworks but increases the learning curve for developers selecting a mechanism.

Context Construction and Memory
Claude Code manages its context window and persists user instructions by implementing two core principles: treating context as a scarce resource with progressive management, and using transparent file-based configuration and memory. 

Before the next model call, the agent loop assembles a context window from the tool pool, CLAUDE.md files, auto memory, and conversation history. The assembly draws from nine sources, some injected at initial assembly and others late during the turn:
1. System prompt, incorporating output style modifications and any --append-system-prompt flag content.
2. Environment info via getSystemContext() (context.ts): git status (skipped in remote mode or when git instructions are disabled) and an optional cache-breaking injection for internal builds (gated by BREAK_CACHE_COMMAND). This data memoizes once per session.
3. CLAUDE.md hierarchy via getUserContext() (context.ts): a four-level instruction file hierarchy. This also memoizes.
4. Path-scoped rules: conditional and directory-matched rules that load lazily when the agent reads files in matching directories.
5. Auto memory: contextually relevant memory entries prefetched asynchronously.
6. Tool metadata: skill descriptions, MCP tool names, and deferred tool definitions (via ToolSearch, on demand).
7. Conversation history: carried forward, subject to compaction.
8. Tool results: file reads, command outputs, subagent summaries.
9. Compact summaries: replacing older history segments.

The context window is not static at assembly time; it can grow during the turn via late injections such as relevant-memory prefetch (query.ts), MCP instructions deltas (only new or changed server instructions), agent listing deltas, and background agent task notifications.
The system prompt assembly at query.ts combines system context with the base prompt via asSystemPrompt(appendSystemContext(systemPrompt, systemContext))(). User context (CLAUDE.md and date) is prepended to the message array via prependUserContext(). This structural separation means CLAUDE.md content occupies a different position in the API request than the system prompt, which potentially affects model attention patterns.

CLAUDE.md Hierarchy and Auto Memory
The memory system operates on the principle that stored context should be inspectable and editable by the user. CLAUDE.md files use plain-text Markdown rather than structured configuration or opaque database entries. This transparency choice trades expressiveness for auditability: users can directly read, edit, version-control, and delete any instruction the agent sees. Alternative memory architectures illustrate this trade-off: retrieval-augmented approaches use embedding-based lookup to surface relevant prior context, gaining flexibility at the cost of inspectability; database-backed memory offers structured querying but requires additional infrastructure and remains opaque to version control. Claude Code's file-based approach makes every instruction directly readable, editable, and committable alongside the codebase. The system does not use embeddings or a vector similarity index for memory retrieval. Instead, it uses an LLM-based scan of memory-file headers to select up to five relevant files on demand, surfacing them at file granularity rather than entry granularity.

CLAUDE.md files follow a multi-level loading hierarchy defined in claudemd.ts, which specifies four memory types:
1. Managed memory (e.g. /etc/claude-code/CLAUDE.md on Linux): OS-level policy for all users.
2. User memory (~/.claude/CLAUDE.md): private global instructions.
3. Project memory (CLAUDE.md, .claude/CLAUDE.md, and .claude/rules/*.md in project roots): instructions checked into the codebase.
4. Local memory (CLAUDE.local.md in project roots): gitignored for private project-specific instructions.

File discovery traverses from the current directory up to root, checking for all project and local memory files in each directory. Files closer to the current directory have higher priority because they load later. Files load in "reverse order of priority," meaning later-loaded files receive more model attention. For root-to-CWD directories, unconditional rules from .claude/rules/*.md load eagerly at startup. For nested directories below the CWD, even unconditional rules load lazily when the agent reads files in matching directories. This design allows the model's instruction set to evolve during a conversation as new parts of the codebase are explored.
