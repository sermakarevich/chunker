# claude-md-delivery-compaction-subagent-persistence

**Parent:** [[content/L1/claude-code-extension-context-subagent-architecture|claude-code-extension-context-subagent-architecture]] — Claude Code's architecture integrates four extension mechanisms (MCP, plugins, skills, hooks) with a 54-tool assembly pipeline, a 9-source context window, a 4-level CLAUDE.md memory hierarchy, and a 5-layer graduated compaction pipeline to manage bounded context. Subagent delegation uses worktree isolation and summary-only returns to prevent context explosion, while session persistence relies on mostly-append JSONL transcripts and explicit permission rebinding across resume/branch operations. Comparative analysis with OpenClaw reveals opposing architectural bets: Claude Code prioritizes ephemeral CLI execution, deny-first per-action safety, and single-agent context extension, whereas OpenClaw operates as a persistent gateway daemon emphasizing perimeter identity control, multi-agent routing, and shared plugin registries.

Claude Code delivers CLAUDE.md instructions as user messages rather than system prompt content, as implemented in `context.ts`. This architectural choice means model compliance with CLAUDE.md guidance is probabilistic rather than guaranteed. Deterministic enforcement relies on permission rules evaluated in deny-first order, creating a deliberate separation between probabilistic guidance and deterministic enforcement. The `setCachedClaudeMdContent()` function caches loaded CLAUDE.md content for the auto-mode classifier to avoid an import cycle between the CLAUDE.md loader and the permission system.

Memory files support an `@include` directive processed by `processMemoryFile()` in `claudemd.ts` for modular instruction sets. Syntax variants include `@path`, `@./relative`, `@~/home`, and `@/absolute`. The directive functions only in leaf text nodes, not inside code blocks. The implementation pushes the including file first and appends included files after it, prevents circular references by tracking processed paths, and silently ignores non-existent files.

The five-layer compaction pipeline in `query.ts` implements graduated compression, reflecting a lazy-degradation principle that applies the least disruptive compression first and escalates only when cheaper strategies prove insufficient. This graduated approach contrasts with simpler frameworks that use single-pass truncation or a single summarization step. The trade-off is increased complexity: five interacting compression layers, several gated by feature flags, create behavior that is difficult for users to fully predict. While auto-compact produces a visible transcript summary and microcompact emits a boundary marker, context collapse operates without user-visible output. The five layers execute in this sequence: (1) Budget reduction (always active) enforces per-tool-result size limits; (2) Snip (gated by `HISTORY_SNIP`) performs lightweight older-history trimming; (3) Microcompact (gated by `CACHED_MICROCOMPACT`) applies fine-grained cache-aware compression; (4) Context collapse (gated by `CONTEXT_COLLAPSE`) creates a read-time virtual projection over history; (5) Auto-compact (enabled by default, user-configurable) generates a full model-generated summary.

The `buildPostCompactMessages()` function in `compact.ts` returns a compacted output structure containing a boundary marker, summary messages, messages to keep, attachments, and hook results. The boundary marker is annotated with preserved-segment metadata via `annotateBoundaryWithPreservedSegment()`, recording `headUuid`, `anchorUuid`, and `tailUuid` to enable read-time chain patching. This mostly-append design ensures compaction never modifies or deletes previously written transcript lines; it only appends new boundary and summary events.

The `compactConversation()` function in `compact.ts` fires Pre-compact hooks first to allow hook-injected custom instructions. A GrowthBook feature flag controls whether the compaction path reuses the main conversation’s prompt cache; a January 2026 code comment documents an experiment stating the false path is 98% a cache miss and costs approximately 0.76% of fleet `cache_creation`. After compaction, attachment builders re-announce runtime state such as plans, skills, and async agents from the live application state, because compaction discards prior attachment messages but not the underlying state.

Multi-agent orchestration in Claude Code implements the isolated subagent boundaries principle alongside deny-first with human escalation and reversibility-weighted risk assessment. Delegation occurs through the Agent tool (`AgentTool.tsx`), with Task retained as a legacy alias. The model invokes Agent with structured input containing the delegation prompt, an optional subagent type, and configuration for isolation mode, permission overrides, and working directory. The Agent tool input schema uses feature-gated fields. The isolation field offers `['worktree', 'remote']` for internal users and `['worktree']` for external users, determined at build time. The `cwd` field is feature-gated. The `run_in_background` field is omitted when background tasks are disabled or when fork-subagent mode is enabled.

Claude Code provides up to six built-in subagent types depending on feature flags and entrypoint: Explore (primarily read/search-oriented investigation with write and edit tools in its deny-list), Plan (creates structured plans with execution proceeding through the standard permission model), General-purpose (broadly capable, used when explicitly requested, though omitting the type may route to the fork-subagent path), Claude Code Guide (onboarding and documentation assistance with its own permissionMode override), Verification (runs validation checks such as test suites and linting), and Statusline-setup (specialized for terminal status line configuration). Beyond built-ins, users define custom subagents via `.claude/agents/*.md` files, and plugins contribute agent definitions via `loadPluginAgents.ts`. The markdown body of each custom agent file serves as the agent’s system prompt, while YAML frontmatter specifies configuration fields including description, tools (allowlist), disallowedTools, model, effort, permissionMode, mcpServers, hooks, maxTurns, skills, memory scope, background flag, and isolation mode. JSON-formatted agent definitions support the same fields plus an explicit prompt field (`loadAgentsDir.ts`). This configuration allows a custom agent to function as a fully configured, isolated sub-system with its own tools, model, permissions, hooks, memory scope, and isolation mode. The Agent tool sits alongside the Skill tool in the base tool pool as a meta-tool that dispatches to these definitions. Unlike the Skill tool, which injects instructions into the current context window, the Agent tool spawns a new, isolated context window. The trade-off is that most subagent invocations require a self-contained prompt because the default path does not inherit the parent’s conversation history, though the fork-subagent path is an exception.

Subagent isolation supports multiple modes defined in `AgentTool.tsx`. Worktree isolation creates a temporary git worktree, giving the subagent its own copy of the repository to modify without affecting the parent’s working tree. Remote isolation is internal-only and launches in a remote Claude Code Remote environment, always running in the background. In-process isolation is the default, sharing the filesystem with the parent but operating in an isolated conversation context.

The permission override logic in `runAgent.ts` applies a subagent’s defined permissionMode unless the parent is already in `bypassPermissions`, `acceptEdits`, or `auto` mode, because those parent modes always take precedence as explicit user decisions about the safety/autonomy trade-off. For async agents, the system determines prompt display through a cascade: explicit `canShowPermissionPrompts` first, then bubble mode (always show, as they escalate to the parent terminal), then the default (sync agents show prompts, async agents do not). Background agents that can show prompts set `awaitAutomatedChecksBeforeDialog: true`, ensuring the classifier and hooks resolve before interrupting the user.

Claude Code’s worktree-based isolation provides filesystem-level separation with zero external dependencies by leveraging Git’s built-in mechanism, contrasting with container-based isolation used by SWE-Agent and OpenHands, or context-only isolation used by conversation-based frameworks like AutoGen. When `allowedTools` is explicitly provided to `runAgent()`, a two-tier permission scoping model applies. SDK-level permissions from `--allowedTools` are preserved as explicit permissions from the SDK consumer that apply to all agents, while session-level rules are replaced with the subagent’s declared `allowedTools`. When `allowedTools` is not provided, the common AgentTool path inherits the parent’s session-level rules without replacement.

Each subagent writes its own transcript as a separate `.jsonl` file alongside a `.meta.json` metadata file (`sessionStorage.ts`, `runAgent.ts`). This sidechain design preserves subagent histories for debugging and auditing without inflating the parent’s session file. Only the subagent’s final response text and metadata return to the parent conversation context, respecting the context as bottleneck principle. The `runAgent()` function accepts 21 parameters covering agent definition, prompts, permissions, tools, model settings, isolation, and callbacks. The summary-only return model conserves context because conversation-based frameworks that share full transcript histories risk context explosion as agent count grows. Even isolated-context parallelism carries substantial cost; Claude Code’s agent teams consume approximately 7× the tokens of a standard session in plan mode, making summary-only return critical. For multi-instance coordination in agent teams, the harness uses file locking rather than a message broker, claiming tasks from a shared list via lock-file-based mutual exclusion with lock files stored at predictable filesystem paths. This approach trades throughput for zero-dependency deployment and full debuggability, allowing any agent’s state to be inspected by reading plain-text JSON files.

Session persistence implements the append-only durable state principle. Session-scoped permissions exist in memory only and are not serialized to the transcript, so resume operations rebuild the permission context from CLI arguments and disk settings; requests the rebuilt context does not recognize fall back to deny-first prompting. Claude Code’s persistence mechanisms write the conversation, including messages, tool results, and compact boundaries, to disk as events occur.

Session transcripts are stored as mostly append-only JSONL files at a project-specific path, with explicit cleanup rewrites as an exception. The `getTranscriptPath()` function in `sessionStorage.ts` computes the path as `join(projectDir, ${getSessionId()}.jsonl)`, where `projectDir` is determined by first checking `getSessionProjectDir()` (set by `switchSession()` during resume/branch) and falling back to `getProjectDir(getOriginalCwd())()`. Three persistence channels operate independently: (1) Session transcripts record conversation messages and compaction/metadata events, scoped to the project with one file per session; (2) Global prompt history stores user prompts only in `history.jsonl` at the Claude configuration home directory, with the `makeHistoryReader()` generator yielding entries in reverse order via `readLinesReverse()` to support Up-arrow and ctrl+r navigation; (3) Subagent sidechains store separate `.jsonl` and `.meta.json` files per subagent. Session transcripts store compaction markers, file-history snapshots, attribution snapshots, and content-replacement records alongside standard messages. This architecture ensures conversations outlive the context window because the disk transcript records everything while compaction recycles the live view, and it ensures conversations outgrow a single path because the append-only transcript allows users to rewind, resume, or fork into new branches without losing prior work. Compaction boundaries explicitly mark summary generation, enabling seamless rewinding, resuming, and branching operations.
