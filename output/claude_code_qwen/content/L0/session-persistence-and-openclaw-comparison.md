# session-persistence-and-openclaw-comparison

**Parent:** [[content/L1/claude-code-extension-context-subagent-architecture|claude-code-extension-context-subagent-architecture]] — Claude Code's architecture integrates four extension mechanisms (MCP, plugins, skills, hooks) with a 54-tool assembly pipeline, a 9-source context window, a 4-level CLAUDE.md memory hierarchy, and a 5-layer graduated compaction pipeline to manage bounded context. Subagent delegation uses worktree isolation and summary-only returns to prevent context explosion, while session persistence relies on mostly-append JSONL transcripts and explicit permission rebinding across resume/branch operations. Comparative analysis with OpenClaw reveals opposing architectural bets: Claude Code prioritizes ephemeral CLI execution, deny-first per-action safety, and single-agent context extension, whereas OpenClaw operates as a persistent gateway daemon emphasizing perimeter identity control, multi-agent routing, and shared plugin registries.

Session persistence and context compaction architecture separates live session state (the context window and compaction processes) from durable storage mechanisms (session transcripts, `history.jsonl`, subagent sidechains, and checkpoints). A deliberate design choice prioritizes auditability and simplicity over query power: every event is human-readable, version-controllable, and reconstructable without specialized tooling, whereas database-backed alternatives would introduce deployment dependencies and reduce transparency. The session identity system pairs a `sessionId` with a `sessionProjectDir`, setting both together during resume or branch operations. The transcript path must use the same active project directory that was used when messages were originally written to prevent hooks from querying the wrong directory.

The `--resume` flag rebuilds a conversation by replaying the transcript through `conversationRecovery.ts`, while forking creates a new session from an existing one via `commands/branch/branch.ts`. Neither resume nor fork restores session-scoped permissions; users must grant them again in the new session. This represents a deliberate safety-conservative design choice that treats sessions as isolated trust domains. Restoring previously granted permissions on resume would create a convenience benefit but risk carrying stale trust decisions into a changed context. The architecture opts for explicit re-granting over implicit persistence, accepting user friction as the necessary cost of maintaining the safety invariant that trust is always established within the current session.

The `compact_boundary` marker interacts directly with persistence. The `annotateBoundaryWithPreservedSegment()` function in `compact.ts` records `headUuid`, `anchorUuid`, and `tailUuid` within the boundary event. These UUIDs enable the session loader to patch the message chain at read time: preserved messages retain their original `parentUuids` on disk, and the loader uses the boundary metadata to link them correctly. This mostly-append architecture ensures compaction never modifies or deletes previously written transcript lines. The term "checkpoints" in this context refers exclusively to file-history checkpoints for the `--rewind-files` command, stored at `~/.claude/filehistory/<sessionId>/`, representing file-level snapshots for reverting filesystem changes rather than a generic checkpoint store.

A comparative analysis between Claude Code and OpenClaw evaluates their architectural divergence across six design dimensions. Table 3 summarizes the contrast across system scope, trust model, agent runtime, extension architecture, memory and context, and multi-agent routing. The comparison highlights how identical recurring design questions produce different architectural answers under different deployment contexts.

**System Scope and Deployment Model:** Claude Code operates as an ephemeral CLI process bound to a single repository, where each session starts and ends with the terminal. OpenClaw operates as a persistent daemon (default port 18789, loopback-only) that owns all messaging surface connections and coordinates clients, tools, and device nodes over a typed WebSocket protocol. This scope difference is the foundational architectural divergence, determining how every other design question is framed. OpenClaw can host Claude Code, OpenAI Codex, and Gemini CLI as external coding harnesses through its Agent Client Protocol (ACP), demonstrating a stackable compositional relationship.

**Trust Model and Security Architecture:** Claude Code addresses a threat model assuming an untrusted model operating within a trusted developer’s machine. Its security relies on deny-first per-action rule evaluation, hooks, an optional ML classifier, and seven permission modes that form a graduated autonomy spectrum. OpenClaw addresses a threat model assuming a single trusted operator per gateway instance. Its security begins with identity and access control (DM pairing codes, sender allowlists, gateway authentication) rather than per-action safety classification. Tool policy in OpenClaw uses configurable allow/deny lists per agent. Sandboxing is an opt-in feature with multiple backends (Docker, SSH, or OpenShell) and configurable scope (per-agent, per-session, or shared); a non-main mode can sandbox all non-main sessions when enabled. OpenClaw explicitly states that hostile multi-tenant isolation on a shared gateway is not a supported security boundary. Consequently, Claude Code places the trust boundary between the model and the execution environment, while OpenClaw places it at the gateway perimeter.

**Agent Runtime and Tool Orchestration:** Both systems implement agentic loops following the ReAct pattern, but occupy different architectural positions. In Claude Code, the `queryLoop()` async generator serves as the system’s center, managing context assembly, model calls, tool dispatch, and recovery for all interfaces. In OpenClaw, the agent runtime (an embedded Pi-agent core) sits inside a larger gateway dispatch layer. The gateway’s agent RPC validates parameters, resolves sessions, and returns immediately, while the embedded runner executes the agentic loop and emits lifecycle and stream events back through the gateway protocol. OpenClaw serializes runs through per-session queues and an optional global lane to prevent tool and session races across the multi-channel surface. In OpenClaw, the agent loop functions as a component within a control plane rather than the control plane itself.

**Extension Architecture:** Claude Code organizes its four extension mechanisms (MCP, plugins, skills, hooks) by graduated context cost: hooks consume zero context, skills consume low context, and MCP servers consume high context. All four mechanisms extend a single agent’s context window and tool surface. OpenClaw utilizes a manifest-first plugin system with four architectural layers (discovery, enablement, runtime loading, surface consumption) and twelve capability types including text inference, speech, media understanding, image/music/video generation, web search, and messaging channels. Plugins register capabilities into a central registry that the gateway reads to expose tools, channels, provider setup, hooks, HTTP routes, CLI commands, and services. OpenClaw maintains a separate skills layer with six precedence sources (workspace, project-level, personal, managed, bundled, and extra directories, with workspace skills highest), a public registry (ClawHub), and supports MCP through built-in `openclaw mcp` commands. The key distinction is that Claude Code extensions modify one agent’s action surface, whereas OpenClaw plugins extend a shared gateway surface across all agents.

**Memory, Context, and Knowledge Management:** Both systems prioritize transparent file-based memory. Claude Code loads a four-level CLAUDE.md hierarchy and manages context pressure through a five-layer compaction pipeline, using an LLM-based scan of file headers for memory retrieval. OpenClaw injects workspace bootstrap files into the system prompt at session start: five core files (AGENTS.md, SOUL.md, TOOLS.md, IDENTITY.md, USER.md) plus conditionally BOOTSTRAP.md, HEARTBEAT.md, and MEMORY.md, truncating large files. OpenClaw’s memory system manages three file types: MEMORY.md for long-term durable facts, date-stamped daily notes (memory/YYYY-MMDD.md), and an optional DREAMS.md for dreaming sweep summaries. When an embedding provider is configured, OpenClaw uses hybrid retrieval combining vector similarity with keyword matching. An experimental dreaming system performs background consolidation, scoring candidates and promoting qualified items from short-term recall into long-term memory. OpenClaw automatically reminds the agent to save important notes to memory files before compaction to prevent context loss. While both systems commit to user-visible, editable memory, OpenClaw invests in structured long-term memory promotion and pluggable compaction, whereas Claude Code invests in graduated context compression. OpenClaw’s compaction pipeline remains less graduated than Claude Code’s five-layer system.

**Multi-Agent Architecture and Routing:** This dimension reveals the starkest divergence. Claude Code’s multi-agent model relies on task delegation: the parent spawns subagents (Explore, Plan, general-purpose, and custom types) that operate in isolated context windows with restricted tool sets and return summary-only results, utilizing worktree isolation for filesystem-level separation. OpenClaw separates two distinct concerns. First, multi-agent routing allows a single gateway to host multiple fully isolated agents, each with its own workspace, authentication profiles, session store, and model configuration, routed to specific channels or senders via deterministic binding rules. Second, sub-agent delegation allows within a single agent, background runs to spawn with configurable nesting depth (maximum 5, default 1, recommended 2), thread-bound sessions, and configurable tool policy by depth. OpenClaw explicitly rejects agent-hierarchy frameworks as a default architecture. The distinction matters because Claude Code’s subagents function as subordinate workers within one user’s coding session, while OpenClaw’s multi-agent routing creates genuinely independent agent instances serving different users or purposes across different channels.

This comparison yields three core observations. First, the recurring design questions (where reasoning lives, what safety posture to adopt, how to manage context, how to structure extensibility) apply universally beyond coding agents; the answers vary strictly based on deployment context. Second, the systems make opposite architectural bets: Claude Code invests in graduated per-action safety evaluation while OpenClaw invests in perimeter-level identity and access control; Claude Code treats the agent loop as the architectural center while OpenClaw treats the gateway control plane as the center; Claude Code extensions modify a single context window while OpenClaw plugins extend a shared gateway surface. These inversions follow directly from differing trust models and deployment topologies. Third, the compositional relationship (OpenClaw hosting Claude Code via ACP) demonstrates that the AI agent design space is a layered architecture where gateway-level systems and task-level harnesses can compose, rather than a flat taxonomy of exclusive alternatives.
