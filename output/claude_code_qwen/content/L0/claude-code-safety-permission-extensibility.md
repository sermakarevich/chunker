# claude-code-safety-permission-extensibility

**Parent:** [[content/L1/claude-code-subsystem-decomposition-and-safety-enforcement|claude-code-subsystem-decomposition-and-safety-enforcement]] — Claude Code's architecture decomposes into strictly bounded subsystems where AgentTool spawns isolated subagents, the state layer memoizes context via context.ts and CLAUDE.md hierarchies, and queryLoop() in query.ts drives a reactive while-loop executing nine sequential steps with seven whole-object continue points. Context management relies on a five-shaper compaction pipeline and per-tool budgets, while a seven-layer permission pipeline (documented via a Hughes 2026 93% approval rate finding) enforces deny-first evaluation across seven modes using the auto-mode classifier in yoloClassifier.ts and 27 hook events. Extensibility layers across MCP servers, CLAUDE.md elements, model reach tools, and execution gate hooks operate on a strict context-cost ordering to preserve the bounded window constraint.

Claude Code's safety and extensibility architecture operates as follows:

Production coding agents adopt different safety architectures: layered policy enforcement, operating-system-level sandboxing, or version-control-based rollback. Claude Code combines layered policy enforcement with operating-system-level sandboxing. This design implements four specific architectural principles identified in the system's foundational design principles framework: deny-first evaluation with human escalation, a graduated trust spectrum, defense in depth using layered mechanisms, and reversibility-weighted risk assessment.

When the Claude model decides to execute a tool—for example, running the npm test command via BashTool to reproduce a failing authentication test—the request enters the permission pipeline. Every tool invocation passes through this permission system, and the default behavior is to deny or ask for confirmation rather than allow silently. This default stems from documented behavioral patterns: an auto-mode analysis conducted by Hughes in 2026 found that users approve approximately 93% of permission prompts. This high approval rate indicates that approval fatigue renders interactive confirmation behaviorally unreliable as a sole safety mechanism. Because users habitually approve requests without careful review, the system must maintain safety independently of human vigilance. This requirement motivates the architectural commitment to deny-first evaluation, blanket-deny pre-filtering, and sandboxing as independent layers that operate regardless of user attentiveness.

The permission system defines seven permission modes across its type definitions in the `types/permissions.ts` file: five external modes, one auto mode added conditionally, and one bubble mode included in the type union. The modes are:
1. plan: The Claude model must create a plan; execution proceeds only after user approval.
2. default: Standard interactive use. Most operations require user approval.
3. acceptEdits: Edits within the working directory and certain filesystem shell commands (mkdir, rmdir, touch, rm, mv, cp, sed) are auto-approved; other shell commands require approval.
4. auto: An ML-based classifier evaluates requests that do not pass fast-path checks. This mode is gated by the TRANSCRIPT_CLASSIFIER feature flag.
5. dontAsk: No prompting occurs, but deny rules remain enforced.
6. bypassPermissions: Skips most permission prompts, but safety-critical checks and bypass-immune rules still apply.
7. bubble: An internal-only mode used for subagent permission escalation to the parent terminal.

The five externally visible modes (acceptEdits, bypassPermissions, default, dontAsk, plan) are defined in the EXTERNAL_PERMISSION_MODES array. The auto mode is conditionally included only when the TRANSCRIPT_CLASSIFIER feature flag is active. The bubble mode exists in the type union but not in either mode array; it is used internally for subagent permission escalation. Permission rules are evaluated in deny-first order within `permissions.ts`. The `toolMatchesRule()` function checks deny rules first, meaning a deny rule always takes precedence over an allow rule, even when the allow rule is more specific. A broad deny rule cannot be overridden by a narrow allow rule. The rule system supports tool-level matching (by tool name) and content-level matching (matching specific tool input patterns, such as `Bash(prefix:npm)`).

The seven modes span a graduated autonomy spectrum, ranging from plan (user approves all plans before execution) through default and acceptEdits to bypassPermissions (minimal prompting). This gradient reflects a recurring design tension where increased autonomy shifts the system from interactive approval to automated safety checks. Alternative agent systems resolve this tension differently. SWE-Agent and OpenHands use Docker container isolation to sandbox the agent's entire execution environment rather than evaluating individual tool invocations. Aider relies on Git as a safety net, making all changes reversible through version control. Claude Code's approach layers multiple policy-enforcement mechanisms on top of optional container sandboxing, trading simplicity for fine-grained control over individual actions.

The full authorization pipeline proceeds through several sequential stages:
- Pre-filtering: Before any tool request reaches runtime evaluation, the `filterToolsByDenyRules()` function in `tools.ts` strips blanket-denied tools from the Claude model's view entirely at tool pool assembly time. The documentation specifies that this function uses the same matcher as the runtime permission check, so MCP server-prefix rules like `mcp__server` strip all tools from that server before the Claude model sees them. This prevents the Claude model from attempting to invoke forbidden tools.
- PreToolUse hook: Registered hooks fire as part of the permission pipeline. A PreToolUse hook can return a permissionDecision to deny or ask, or an updatedInput that modifies the tool's input parameters (defined in `types/hooks.ts`). A hook allow does not bypass subsequent rule-based denies or safety checks. In the interactive execution path, the user dialog is queued first and hooks run asynchronously. Coordinator and background-agent execution paths await automated checks before showing the dialog.
- Rule evaluation: The deny-first rule engine evaluates the request. MCP tools are matched by their fully qualified `mcp__server__tool` name, and server-level rules match all tools from that server.
- Permission handler: The handler in `useCanUseTool.tsx` branches into one of four execution paths based on runtime context:
  1. Coordinator: Used for multi-agent coordination mode. It attempts automated resolution (classifier, hooks, rules) before falling back to user interaction.
  2. Swarm worker: Handles worker agents in a multi-agent swarm with their own resolution logic.
  3. Speculative classifier: When the BASH_CLASSIFIER flag is enabled and the tool is BashTool, a speculative classifier races a pre-started classification result against a timeout. If the classifier returns with high confidence, the tool is approved instantly without user interaction.
  4. Interactive: The fallback path that presents the standard user approval dialog through the terminal UI.

In coordinator and some background execution paths, automated resolution is attempted before user interaction. In the standard interactive path, the dialog can appear first while hooks or classifier checks continue in parallel. When the classifier or a deny rule blocks an action, Claude Code treats the denial as a routing signal rather than a hard stop: the Claude model receives the denial reason, revises its approach, and attempts a safer alternative in the next loop iteration. The PermissionDenied hook event enables external code to observe and respond to these denials programmatically. This recovery-oriented design means that permission enforcement shapes the Claude model's behavior rather than simply halting it.

The auto-mode classifier resides in `yoloClassifier.ts` and participates in permission decisions when enabled. When TRANSCRIPT_CLASSIFIER is enabled, the classifier loads three prompt resources: a base system prompt, an external permissions template, and for Anthropic-internal users, a separate internal template. The `isUsingExternalPermissions()` function checks USER_TYPE and a forceExternalPermissions config flag to select the appropriate template.

Of the 27 hook events defined in the source code (`coreTypes.ts`), five participate directly in the permission flow, each with a specific Zod-validated output schema (`types/hooks.ts`):
- PreToolUse: Can return permissionDecision (deny or ask, though allow does not bypass subsequent checks), permissionDecisionReason, and updatedInput (modify parameters).
- PostToolUse: Can inject additionalContext and, for MCP tools, return updatedMCPToolOutput to modify results before they enter the context.
- PostToolUseFailure: Can inject additionalContext for error-specific guidance.
- PermissionDenied: Can provide retry guidance after auto-mode denials.
- PermissionRequest: Can return a decision of allow or deny. In coordinator and similar execution paths, this can resolve before the user dialog. In the standard interactive path, it can also run alongside the dialog.

For non-MCP tools, the tool_result is emitted before the PostToolUse hook fires. For MCP tools, the result is delayed until after post hooks have run, enabling updatedMCPToolOutput to take effect.

Shell sandboxing provides an additional layer of protection for Bash and PowerShell commands via the `shouldUseSandbox.ts` file. The `shouldUseSandbox()` function checks whether sandboxing is globally enabled, whether the invocation has opted out, and whether the command matches any exclusion patterns. When active, the sandbox provides filesystem and network isolation independent of the application-level permission model. A command can be permission-approved but still sandboxed, or permission-denied and never reach the sandbox check. The two systems operate on different axes: authorization versus isolation. The layered safety architecture rests on an independence assumption: if one layer fails, others catch the violation. However, several layers share common performance constraints. Security researchers at Adversa.ai in 2026 documented that commands with more than 50 subcommands fall back to a single generic approval prompt instead of running per-subcommand deny-rule checks, because per-subcommand parsing caused terminal UI freezes. This example demonstrates that defense-in-depth can degrade when its layers share failure modes, revealing a structural tension between safety and performance.

The permission pipeline governs whether a tool request executes. The extensibility architecture determines which tools exist in the first place by assembling the Claude model's action surface. A recurring design question for coding agents is how to structure the extension surface: as a single unified mechanism, a small number of specialized mechanisms, or a layered stack with different context costs. This architecture illustrates two design principles from the foundational design principles framework: composable multi-mechanism extensibility and externalized programmable policy. Returning to the running example task of repairing `auth.test.ts` after the `npm test` request has been mediated by the permission system, the next architectural question is which extension-enabled action surface is available for the repair. When a turn begins in Claude Code, the Claude model sees not just built-in tools like BashTool and FileReadTool, but also database query tools from an MCP server, a custom lint skill from `.claude/skills/`, and tools contributed by an installed plugin. These arrive through four mechanisms that extend the agent at different points of the loop: MCP servers provide external tool integration, plugins package and distribute bundles of components, skills inject domain-specific instructions, and hooks intercept the tool execution lifecycle. Anthropic's documentation presents a broader view that includes CLAUDE.md configuration and subagents alongside these four mechanisms. CLAUDE.md and subagents operate in different subsystems (context construction and delegation, respectively), but their context-cost ordering is architecturally significant because it reveals how each extension point trades off expressiveness against the bounded context window.

The four extension mechanisms are implemented in distinct source directories and serve different integration patterns:
- MCP servers: The Model Context Protocol is the primary external tool integration path. MCP servers are configured from multiple scopes: project, user, local, and enterprise, with additional plugin and claude.ai servers merged at runtime (`services/mcp/config.ts`). The MCP client (`services/mcp/client.ts`) supports multiple transport types: stdio, SSE, HTTP, WebSocket, SDK, plus IDE-specific variants (sse-ide, ws-ide).
- Context-cost ordering (CLAUDE.md elements): CLAUDE.md files load into context with files above the working directory loading at startup and subdirectory files loading on demand. Skill descriptions advertise skills so the Claude model calls SkillTool. MCP resources and prompts deliver non-tool content an MCP server pushes. Output style replaces the response-formatting system block. UserPromptSubmit hooks inject context or block on every user turn. SessionStart hooks provide one-shot context injection at session start.
- Model reach elements: Built-in tools provide Read/Edit/Bash capabilities shipped with the CLI. MCP tools provide tools from any MCP server in the same flat pool. SkillTool acts as a meta-tool that launches a skill by name. AgentTool acts as a meta-tool that spawns a sub-agent recursively.
- Execution gate elements: Permission rules provide declarative allow/deny/ask per call. PreToolUse hooks approve/block/rewrite a tool call. PostToolUse hooks mutate output or inject context after a call. Stop hooks force the loop to keep going at model stop. SubagentStop hooks perform the same function for sub-agents spawned via AgentTool. Notification hooks handle external side effects on user notifications.
