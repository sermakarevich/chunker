# claude-code-subsystem-decomposition-and-safety-enforcement

**Parent:** [[index]]

Claude Code’s architectural execution model decomposes into clearly bounded subsystems that separate reasoning from operational infrastructure. Subagent spawning is mediated by AgentTool (implemented in AgentTool.tsx and runAgent.ts), which routes through the standard buildTool() factory. When invoked, the subagent re-enters queryLoop() with a fully isolated context window and returns exclusively a summary to the parent process, establishing a strict boundary against context inflation. The state layer functions as a memoized state loader rather than a routing hub. Context assembly is handled by functions in context.ts: getSystemContext() computes session-level system context including git status, while getUserContext() loads the CLAUDE.md hierarchy and current date. Both are cached for reuse; system context appends to the system prompt, and user context adds a user-context message. Runtime state resides in src/state/, while session transcripts persist as mostly append-only JSONL files in sessionStorage.ts. The CLAUDE.md + memory subsystem (claudemd.ts) defines a four-level instruction hierarchy spanning managed settings to directory-specific files, supplemented by auto-memory entries written during conversations. Sidechain transcripts in sessionStorage.ts:247 isolate each subagent’s conversation in separate files. Global prompt history lives in history.ts as history.jsonl, with resume and fork operations reconstructing state via conversationRecovery.ts. The backend layer manages execution backends and external resources, comprising shell execution (BashTool.tsx, PowerShellTool.tsx), remote execution in src/remote/, MCP connections across stdio, SSE, HTTP, WebSocket, SDK, and IDE-specific transports via services/mcp/client.ts, and 42 tool subdirectories in src/tools/. QueryEngine documentation explicitly states it owns the query lifecycle and session state, extracting ask() logic for headless/SDK and future REPL use. Its constructor accepts a QueryEngineConfig (initial messages, abort controller, file-state cache, per-conversation state), and submitMessage() is an async generator orchestrating a single turn. Architecturally, the shared query path resides in query() (query.ts), which wraps an internal queryLoop(); the interactive CLI bypasses QueryEngine entirely, calling query() directly, confirming the loop function rather than the engine class as the core execution unit.

Turn execution operates as a reactive loop implementing a simple while-loop architecture that embodies minimal scaffolding with maximal operational harness, context as a scarce resource, and graceful recovery. Each turn follows a fixed nine-step sequence in query.ts: settings resolution destructures immutable parameters (system prompt, user context, permission callback, model configuration); mutable state initialization creates a single State object storing all mutable state across iterations, with the loop’s seven continue points overwriting this object in one whole-object assignment rather than mutating fields individually; context assembly calls getMessagesAfterCompactBoundary() to retrieve messages from the last compact boundary forward, ensuring compacted content maps to summaries; pre-model context shapers execute sequentially; a for await loop over deps.callModel() streams the response while passing assembled messages, the full system prompt, thinking configuration, the available tool set, an abort signal, the current model specification, and additional options including fast-mode settings, effort value, and fallback model; tool-use dispatch routes tool_use blocks to orchestration; the permission gate evaluates each request; tool results append as tool_result messages; and the loop halts when the response contains only text content. queryLoop() is defined as an AsyncGenerator yielding StreamEvent, RequestStartEvent, Message, TombstoneMessage, and ToolUseSummaryMessage events, enabling streaming UI output while preserving synchronous control flow. This reactive design follows the ReAct pattern, trading search completeness for simplicity and latency, primarily utilizing the orchestrator-workers workflow pattern for subagent delegation.

The context window acts as the binding constraint, enforced through four contextual design decisions beyond the compaction pipeline: CLAUDE.md lazy loading delays nested-directory instructions until those directories are read; deferred tool schemas include only names initially when ToolSearch is enabled; subagents return only summary text; and a per-tool-result budget caps individual outputs. The five-layer compaction pipeline executes sequentially before every model call. Budget reduction via applyToolResultBudget() enforces per-message size limits, replacing oversized outputs with content references for exempt tools where maxResultSizeChars is not finite, and persists replacements for resume reconstruction. snipCompactIfNeeded() (gated by HISTORY_SNIP) trims older history, returning {messages, tokensFreed, boundaryMessage}; tokensFreed explicitly plumbs savings to auto-compact because the main token counter derives context size from the usage field on the most recent assistant message, which survives snip with pre-snip input_tokens attached. Microcompact runs a time-based path and optionally a cache-aware path (gated by CACHED_MICROCOMPACT), deferring boundary messages until after the API response to use actual cache_deleted_input_tokens, returning {messages, compactionInfo} that may include pendingCacheEdits. Context collapse (gated by CONTEXT_COLLAPSE) operates as a read-time projection over history, replacing messagesForQuery via applyCollapsesIfNeeded() without mutating REPL stored history, persisting across turns via a collapse store. Auto-compact triggers only when pressure persists after previous shapers, executing PreCompact hooks in compact.ts, generating a summary request with getCompactPrompt(), calling compactConversation(), and feeding buildPostCompactMessages(). Recovery mechanisms handle edge cases: max output tokens escalation (subject to a GrowthBook flag and MAX_OUTPUT_TOKENS_RECOVERY_LIMIT = 3), reactive compaction (gated by REACTIVE_COMPACT, fires once per turn via hasAttemptedReactiveCompact), prompt_too_long handling (context-collapse overflow and reactive compact before terminating), onStreamingFallback for API issues, and fallbackModel switching. Loop termination occurs via no tool use, max turns, context overflow, hook_stopped_continuation intervention, or explicit abortController signaling.

Tool dispatch defaults to StreamingToolExecutor, executing tools as they stream in to reduce latency, with a fallback path using runTools() in toolOrchestration.ts iterating over partitions from partitionToolCalls(). Tools classify as concurrent-safe or exclusive; read-only operations run in parallel while state-modifying operations serialize. StreamingToolExecutor coordinates concurrency via a sibling abort controller that fires on any Bash tool error to terminate other subprocesses, and a progress-available signal waking getRemainingResults(). Results buffer and emit in request order to preserve Claude model expectations. Post-execution checks for hook_stopped_continuation attachments via PostToolUse hooks, normalizes results via normalizeMessagesForAPI(), and filters to user-type messages.

Safety operates through seven independent layers where any single layer blocks a request. A Hughes 2026 auto-mode analysis documented a 93% user approval rate, proving approval fatigue renders interactive confirmation behaviorally unreliable and motivating safety independent of human vigilance. The seven layers are: (1) Tool pre-filtering in tools.ts strips blanket-denied tools at assembly time; (2) Deny-first evaluation in permissions.ts ensures deny rules override allow rules regardless of specificity via toolMatchesRule(); (3) Permission mode constraints in types/permissions.ts set baseline handling; (4) Auto-mode classifier evaluates safety; (5) Shell sandboxing in shouldUseSandbox.ts restricts filesystem/network access independently of authorization; (6) conversationRecovery.ts prevents session-scoped permission restoration on resume/fork; (7) Hook-based interception in types/hooks.ts modifies decisions. Seven permission modes exist in types/permissions.ts: plan, default, acceptEdits, auto, dontAsk, bypassPermissions, and bubble. The five external modes reside in EXTERNAL_PERMISSION_MODES. Auto mode activates only when the TRANSCRIPT_CLASSIFIER feature flag is active. Bubble mode exists in the type union for internal subagent escalation but not in mode arrays. Rule evaluation matches tool-level names and content-level patterns (e.g., Bash(prefix:npm)), with MCP tools matching mcp__server__tool prefixes. The authorization pipeline branches in useCanUseTool.tsx into four paths: Coordinator (automated resolution before fallback), Swarm worker, Speculative classifier (BASH_CLASSIFIER flag races a pre-started classification against timeout for instant Bash approval), and Interactive (standard dialog). Denials function as routing signals prompting model revision, observable via the PermissionDenied hook. The auto-mode classifier in yoloClassifier.ts loads three prompt resources (base system prompt, external permissions template, internal template for Anthropic users via isUsingExternalPermissions() checking USER_TYPE and forceExternalPermissions) when TRANSCRIPT_CLASSIFIER enables.

Of the 27 hook events in coreTypes.ts, five participate in permission flow with Zod-validated schemas in types/hooks.ts: PreToolUse (returns permissionDecision, permissionDecisionReason, updatedInput), PostToolUse (injects additionalContext, updatedMCPToolOutput), PostToolUseFailure, PermissionDenied, and PermissionRequest. Non-MCP results emit before PostToolUse, while MCP results delay until after PostToolUse to enable updatedMCPToolOutput. Shell sandboxing in shouldUseSandbox.ts checks global opt, invocation opt, and exclusion patterns. Security researchers at Adversa.ai in 2026 documented that commands exceeding 50 subcommands fall back to a single generic approval prompt to avoid terminal UI freezes, demonstrating defense-in-depth degradation when layers share performance constraints.

Extensibility structures the action surface across four context-cost ordered mechanisms: MCP servers (project/user/local/enterprise/plugin/claude.ai scopes, stdio/SSE/HTTP/WebSocket/SDK/IDE transports via services/mcp/config.ts); CLAUDE.md elements (base files at startup, subdirectory files on demand, skill descriptions, MCP resources/prompts, output style, UserPromptSubmit hooks, SessionStart hooks); Model reach elements (built-in tools, flat MCP tool pool, SkillTool, AgentTool); and Execution gate elements (rules, PreToolUse/PostToolUse hooks, stop hooks). This layered architecture trades context-window pressure for fine-grained, composable control across a bounded execution loop.
