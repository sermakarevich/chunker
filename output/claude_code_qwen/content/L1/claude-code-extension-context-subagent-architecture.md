# claude-code-extension-context-subagent-architecture

**Parent:** [[index]]

The Claude Code extension and context architecture is structured around four distinct injection mechanisms, a graduated context-cost model, and a layered safety system that collectively define how the agent observes, reasons, and acts. The extension surface is deliberately partitioned into MCP servers, plugins, skills, and hooks to avoid consolidating fundamentally different integration patterns into a single mechanism, thereby reducing developer learning costs while accommodating varying context window penalties. MCP servers provide high-cost runtime tool integration by contributing MCPTool definitions to an internal claudeai-proxy, with dedicated ListMcpResourcesTool and ReadMcpResourceTool granting resource access. Plugins function as both a packaging format and distribution mechanism, validated against the PluginManifestSchema in utils/plugins/schemas.ts, which accepts exactly ten component types: commands, agents, skills, hooks, MCP servers, LSP servers, output styles, channels, settings, and user configuration. The pluginLoader.ts validator routes each component to its respective registry, with commands and skills surfacing through the SkillTool meta-tool, agents appearing in AgentTool definitions, hooks merging into the global registry, and output styles modifying response formatting. Skills are defined via SKILL.md YAML frontmatter, parsed by parseSkillFrontmatterFields() in loadSkillsDir.ts, which processes over fifteen fields including display name, description, allowed tools, argument hints, model overrides, execution context ('fork'), associated agent definitions, effort levels, and shell configuration. Skill hooks register dynamically upon invocation, and bundled skills register in-memory at startup. Hooks constitute the lowest-cost extension point, with twenty-seven events spanning tool authorization (PreToolUse, PostToolUse, PostToolUseFailure, PermissionRequest, PermissionDenied), session lifecycle, user interaction, subagent coordination, context management, workspace events, and notifications. Fifteen events carry event-specific output schemas, and runtime hooks draw from settings.json, plugins, managed policy, and dynamic skill registration, supporting four persisted command types (shell, prompt, http, agent) and non-persistable callback hooks for SDK instrumentation.

Tool pool assembly is centralized in assembleToolPool() within tools.ts, executing a five-step pipeline that serves as the single source of truth for the agent's action surface. Step one enumerates base tools via getAllBaseTools(), returning up to fifty-four tools: nineteen are always included (including BashTool, FileReadTool, AgentTool, SkillTool), while thirty-five load conditionally based on feature flags, environment variables, and user type, with Anthropic-internal users receiving additional internal tools. Worktree mode enables EnterWorktreeTool and ExitWorktreeTool, agent swarms enable team tools, and embedded search tools in the Bun binary cause GlobTool and GrepTool to be omitted. Step two applies mode filtering through getTools(), invoking each tool's isEnabled() method; in CLAUDE_CODE_SIMPLE mode, only Bash, Read, and Edit are available (or REPLTool in the REPL branch), plus coordinator tools. Step three strips blanket-denied tools via filterToolsByDenyRules(), removing them from the model's view before any call. Step four merges MCP tools filtered by deny rules, and step five deduplicates by name with built-ins taking precedence. Both REPL.tsx and AgentTool.tsx invoke this function, and at request time, deferred tools may be hidden until queried via ToolSearch.

Context construction precedes every model call by assembling nine sources: system prompt (with output style and --append-system-prompt), environment info via getSystemContext() (git status, cache-breaking injection gated by BREAK_CACHE_COMMAND), CLAUDE.md hierarchy via getUserContext(), path-scoped rules (lazy loading), auto memory (asynchronous prefetch via query.ts), tool metadata (skill descriptions, MCP names, deferred definitions), conversation history, tool results, and compact summaries. The window grows during a turn via late injections like relevant-memory prefetch, MCP instruction deltas, agent listing deltas, and background notifications. CLAUDE.md instructions are delivered as user messages rather than system prompt content via context.ts, making compliance probabilistic; deterministic safety relies on deny-first permission rules. Memory uses a four-level hierarchy defined in claudemd.ts: managed memory (/etc/claude-code/CLAUDE.md), user memory (~/.claude/CLAUDE.md), project memory (CLAUDE.md, .claude/CLAUDE.md, .claude/rules/*.md), and local memory (.claude/CLAUDE.md, gitignored). Traversal moves from CWD to root, with later-loaded files receiving higher model attention. Root-to-CWD .claude/rules/*.md load eagerly at startup, while nested rules load lazily on file reads. The @include directive in processMemoryFile() supports @path, @./relative, @~/home, and @/absolute syntax in leaf text nodes, pushing the including file first and appending included files while preventing circular references. Memory retrieval uses an LLM-based scan of file headers to select up to five relevant files, deliberately trading the flexibility of embedding-based lookup or database-backed memory for user-editable, version-controllable transparency. CLAUDE.md content is cached via setCachedClaudeMdContent() to avoid an import cycle with the permission system.

Context pressure is managed by a five-layer graduated compaction pipeline in query.ts that applies the least disruptive compression first. Layer one (always active) enforces per-tool-result size limits. Layer two (HISTORY_SNIP) trims older history. Layer three (CACHED_MICROCOMPACT) applies cache-aware compression. Layer four (CONTEXT_COLLAPSE) creates a read-time virtual projection. Layer five (auto-compact, default) generates a full summary. buildPostCompactMessages() in compact.ts returns a boundary marker, summary messages, kept messages, attachments, and hook results. annotateBoundaryWithPreservedSegment() records headUuid, anchorUuid, and tailUuid for read-time chain patching, ensuring a mostly-append architecture that never modifies or deletes old transcript lines. compactConversation() fires Pre-compact hooks first; a GrowthBook flag gates prompt cache reuse, with a January 2026 comment noting the false path yields a 98% cache miss and costs ~0.76% of fleet cache_creation. Post-compaction, attachment builders re-announce runtime state. Session persistence channels write to disk across three independent paths: session transcripts (join(projectDir, ${sessionId}.jsonl)), global history (history.jsonl with makeHistoryReader() and readLinesReverse() for reverse navigation), and subagent sidechains. Compaction boundaries mark summaries for rewinding and branching. The --rewind-files checkpoints are stored at ~/.claude/filehistory/<sessionId>/ for file-level snapshots. Resume replays via conversationRecovery.ts, and branching uses commands/branch/branch.ts. Neither restores session-scoped permissions, treating sessions as isolated trust domains and requiring explicit re-granting to prevent stale trust propagation.

Subagent orchestration delegates through AgentTool, supporting worktree, remote, and in-process isolation. Worktree isolation creates a temporary git worktree; remote isolation is internal-only and background-based. Subagent types include Explore (read/search with write/edit deny-listed), Plan (structured plans with standard permissions), General-purpose (broad, routes to fork when omitted), Claude Code Guide (onboarding with permissionMode override), Verification (tests/lint), and Statusline-setup. Custom agents are defined in .claude/agents/*.md with markdown system prompts and YAML frontmatter controlling description, tools, disallowedTools, model, effort, permissionMode, mcpServers, hooks, maxTurns, skills, memory scope, background, and isolation. Permission overrides apply the subagent's mode unless the parent uses bypassPermissions, acceptEdits, or auto. Async agents cascade prompt display via canShowPermissionPrompts, bubble mode, or default sync behavior, with awaitAutomatedChecksBeforeDialog ensuring classifier/hook resolution before interruption. Two-tier permission scoping applies when allowedTools is passed to runAgent(): SDK --allowedTools are preserved while session rules are replaced; otherwise, session rules inherit. Subagents write sidechain .jsonl and .meta.json files, returning only summary text to prevent context explosion, as agent teams consume ~7× tokens in plan mode. Multi-instance coordination uses filesystem-based lock files instead of message brokers for zero-dependency deployment and full debuggability.

Comparative analysis with OpenClaw highlights architectural inversions driven by scope and trust models. Claude Code operates as an ephemeral CLI process bound to a single repository, while OpenClaw is a persistent daemon on port 18789 (loopback-only) coordinating clients, tools, and devices via typed WebSocket. Claude Code hosts Claude Code, OpenAI Codex, and Gemini CLI via ACP. Trust diverges fundamentally: Claude Code assumes an untrusted model on a trusted machine, enforcing deny-first per-action evaluation, hooks, an ML classifier, and seven permission modes across a graduated autonomy spectrum. OpenClaw assumes a trusted operator per gateway, securing identity/access (DM pairing codes, sender allowlists, gateway auth), using configurable tool allow/deny lists, and offering opt-in sandboxing (Docker, SSH, OpenShell) scoped per-agent/session/shared, explicitly rejecting hostile multi-tenant isolation. Claude Code places trust between model and execution environment; OpenClaw places it at the gateway perimeter. Runtime architecture differs similarly: Claude Code's queryLoop() async generator is the system center, whereas OpenClaw's embedded Pi-agent core sits inside a gateway dispatch layer that validates parameters, resolves sessions, and serializes runs via per-session queues plus an optional global lane. Extension architectures oppose each other: Claude Code grades mechanisms by context cost (hooks=zero, skills=low, MCP=high) to modify one agent's surface, while OpenClaw uses a manifest-first plugin system with four layers and twelve capability types (text inference, speech, media, image/music/video generation, web search, messaging) that extend a shared gateway registry. Memory diverges in retrieval and compaction: Claude Code uses a four-level CLAUDE.md hierarchy, LLM header scanning (max five files), and five-layer graduated compaction; OpenClaw injects five core files plus conditionally BOOTSTRAP.md/HEARTBEAT.md/MEMORY.md, manages MEMORY.md, date-stamped daily notes, and DREAMS.md, employs hybrid vector/keyword retrieval, and implements an experimental dreaming system for background consolidation. Multi-agent design separates task delegation from routing: Claude Code delegates to isolated subagents with restricted tools and summary-only returns using worktree isolation, while OpenClaw separates multi-agent routing (hosting fully independent agents per workspace/auth/session/model/channel/sender) from sub-agent delegation (background runs, max nesting depth of five, thread-bound sessions, configurable tool policy). These inversions demonstrate that gateway-level systems and task-level harnesses compose architecturally rather than competing as flat alternatives, with Claude Code prioritizing graduated per-action safety and single-context extensibility, and OpenClaw prioritizing perimeter identity control and shared gateway surface extension.
