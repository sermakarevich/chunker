# claude-code-memory-persistence-subagents

**Parent:** [[index]]

Claude Code employs a sophisticated memory, context, and persistence layer designed to balance transparency, model compliance, and token efficiency. Central to this architecture is the `CLAUDE.md` hierarchy, a multi-level loading system defined in `claudemd.ts` that avoids vector databases in favor of plain-text Markdown for user-controllable memory. The hierarchy consists of four memory types: (1) Managed memory (OS-level policy, e.g., `/etc/claude-code/CLAUDE.md` on Linux); (2) User memory (private global instructions at `~/.claude/CLAUDE.md`); (3) Project memory (checked into the codebase at `CLAUDE.md`, `.claude/CLAUDE.md`, and `.claude/rules/*.md` in project roots); and (4) Local memory (gitignored private project-specific instructions in `CLAUDE.local.md`). 

File discovery traverses from the current directory upward to the root, loading files in "reverse order of priority," meaning files closer to the current directory are loaded later and thus receive more model attention. Unconditional rules from `.claude/rules/*.md` in root-to-CWD directories load eagerly at startup. However, for nested directories below the CWD, rules are loaded lazily when the agent reads files in matching directories, allowing the agent's instruction set to evolve during a session. To support modularity, `processMemoryFile()` in `claudemd.ts` supports an `@include` directive for leaf text nodes (excluding code blocks) with syntax variants `@path`, `@./relative`, `@~/home`, and `@/absolute`. Circular references are prevented by tracking processed paths, and non-existent files are silently ignored. 

All `CLAUDE.md` content is delivered as user context (a user message) rather than system prompt content, making model compliance probabilistic rather than guaranteed. This creates a deliberate separation between guidance (probabilistic `CLAUDE.md`) and enforcement (deterministic permission rules evaluated in deny-first order). To prevent import cycles between the loader and the permission system, the `setCachedClaudeMdContent()` function caches loaded content for the auto-mode classifier. 

To manage the context window bottleneck, Claude Code implements a five-layer compaction pipeline in `query.ts` using graduated compression. The layers are applied sequentially in order of increasing aggressiveness: (1) Budget reduction (always active, enforces per-tool-result size limits); (2) Snip (gated by `HISTORY_SNIP`, performs lightweight older-history trimming); (3) Microcompact (gated by `CACHED_MICROCOMPACT`, provides fine-grained cache-aware compression); (4) Context collapse (gated by `CONTEXT_COLLAPSE`, a read-time virtual projection over history); and (5) Auto-compact (enabled by default, produces a full model-generated summary). 

The `buildPostCompactMessages()` function in `compact.ts` manages the output structure: `[boundaryMarker, ...summaryMessages, ...messagesToKeep, ...attachments, ...hookResults]`. The boundary marker is annotated with preserved-segment metadata (headUuid, anchorUuid, and tailUuid) via `annotateBoundaryWithPreservedSegment()` to allow read-time chain patching. This ensures that compaction never modifies or deletes previously written transcript lines, only appending new events. The `compactConversation()` function triggers pre-compact hooks for custom instructions. A GrowthBook feature flag determines whether the compaction path reuses the main conversation's prompt cache; a January 2026 experiment showed that disabling this reuse (the "false" path) results in a 98% cache miss and costs approximately 0.76% of fleet `cache_creation`. After compaction, attachment builders re-announce runtime state (plans, skills, and async agents) because prior attachment messages are discarded, though the underlying state is preserved.

When delegating work, the `Agent` tool (`AgentTool.tsx`), also known by the legacy alias `Task`, serves as the primary mechanism. The model provides a structured input including a delegation prompt, optional subagent type, and configurations for isolation mode, permission overrides, and working directory. The `isolation` field offers `worktree` and `remote` for internal users, while external users are limited to `worktree`. The `cwd` field is gated by a feature flag, and the `run_in_background` field is omitted if background tasks are disabled or if fork-subagent mode is enabled. 

Claude Code provides six built-in subagent types: (1) Explore (read/search-oriented; write/edit tools are in its deny-list); (2) Plan (creates structured plans; execution follows the standard permission model); (3) General-purpose (broadly capable); (4) Claude Code Guide (onboarding/documentation assistance with a specific `permissionMode` override); (5) Verification (runs validation checks like linting and test suites); and (6) Statusline-setup (specialized for terminal status line configuration). Users can define custom subagents via `.claude/agents/*.md` files, and plugins contribute agents via `loadPluginAgents.ts`. For custom agents, the markdown body is the system prompt, and YAML frontmatter defines the description, tools (allowlist), disallowedTools, model, effort, permissionMode, mcpServers, hooks, maxTurns, skills, memory scope, background flag, and isolation mode. JSON-formatted agent definitions via `loadAgentsDir.ts` support these same fields plus a `prompt` field. 

Unlike the `SkillTool`, which injects instructions into the current window, the `AgentTool` spawns a new isolated conversation context. Most subagent invocations require a self-contained prompt because they do not inherit the parent's conversation history, except in the fork-subagent path. Subagent isolation supports three modes: (1) Worktree (creates a temporary git worktree for modifications without affecting the parent); (2) Remote (internal-only; launches in a remote Claude Code Remote environment in the background); and (3) In-process (default; shares the filesystem but uses an isolated conversation context). 

Permission override logic in `runAgent.ts` applies a subagent's `permissionMode` unless the parent is already in `bypassPermissions`, `acceptEdits`, or `auto` mode. For async agents, the system checks `canShowPermissionPrompts` and `bubble` mode (which always shows prompts as they escalate to the {parent terminal}). Sync agents show prompts, while async agents generally do not. Background agents capable of showing prompts set `awaitAutomatedChecksBeforeDialog: true` to ensure classifiers and hooks resolve before user interruption. When `allowedTools` is explicitly provided to `runAgent()`, the agent uses the subagent's declared `allowedTools` instead of session-level rules, though SDK-level permissions from `--allowedTools` are always preserved. 

Each subagent writes its own transcript as a separate `.jsonl` and `.meta.json` file via `sessionStorage.ts` and `runAgent.ts` to prevent subagent histories from inflating the parent's session file. Only the final response text and metadata return to the parent's context. Agent teams in plan mode consume approximately 7x more tokens than a standard session. Coordination between multi-instance agents is handled via lock-file-based mutual exclusion for claiming tasks from a shared list, avoiding message brokers or distributed coordination services to ensure zero-dependency deployment. 

Claude Code's persistence follows an append-only durable state principle. There are three independent persistence channels: (1) Session transcripts (project-scoped; stored as one file per session containing messages, compaction markers, file-history snapshots, attribution snapshots, and content-replacement records); (2) Global prompt history (stores only user prompts in `history.jsonl` at the configuration home directory; managed by `history.ts` and accessed via `makeHistoryReader()` and `readLinesReverse()` for Up-arrow/ctrl+r navigation); and (3) Subagent sidechains (separate `.jsonl` and `.meta.json` files per subagent). 

Session transcripts use an append-only JSONL format for auditability and human-readability. The `getTranscriptPath()` function in `sessionStorage.ts` computes the path by joining the `projectDir` (determined by `getSessionProjectDir()` or `getProjectDir(getOriginalCwd())()`) with the session ID (`${getSessionId()}.jsonl`). When using `--resume`, `conversationRecovery.ts` in `conversationRecovery.ts` rebuilds the conversation by replaying the transcript. Forking, via `commands/branch/branch.ts`, creates a new session from an existing one. Neither resume nor fork operations restore session-scoped permissions; this safety-conservative design treats sessions as isolated trust domains to prevent carrying stale trust decisions into changed contexts. 

To support the append-only nature of the compaction boundary, the `annotateBoundaryWithPreservedSegment()` function in `compact.ts` records `headUuid`, `anchorUuid`, and `tailUuid` to allow the agent to patch the message chain at read time. Additionally, the system implements "checkpoints" as file-history snapshots stored at `~/.claude/filehistory/<sessionId>/` for the `--rewind-files` command to revert filesystem changes. 

In comparison to OpenClaw, the two systems differ across six key dimensions: 
1. **System Scope**: Claude Code is an ephemeral CLI process bound to a repository session, whereas OpenClaw is a persistent WebSocket gateway daemon (port 18789, loopback-only) coordinating multi-channel messaging (WhatsApp, Telegram, Slack) via a typed WebSocket protocol. 
2. **Trust Model**: Claude Code uses a deny-first per-action rule evaluation with hooks, an ML classifier, and seven permission modes. OpenClaw assumes a single trusted operator and uses identity/access control (DM pairing codes, sender allowlists) and opt-in sandboxing (Docker, SSH, or OpenShell) with configurable scope. 
3. **Agent Runtime**: Claude Code's center is the `queryLoop()` async generator. Open uma OpenClaw's center is an embedded Pi-agent core inside a gateway dispatch layer, with runs serialized through per-session queues and an optional global lane. 
4. **Extension Architecture**: Claude Code's extensions are based on context cost (MCP, plugins, skills, hooks). OpenClaw uses a manifest-first plugin system (12 capability types) and a separate skills layer and MCP support via `openclaw mcp` commands. 
5. **Memory and Context**: Claude Code uses a four-level `CLAUDE.md` hierarchy and a five-layer compaction pipeline. OpenClaw injects workspace bootstrap files (`AGENTS.md`, `SOUL.md`, `TOOLS.md`, `IDENTITY.md`, `USER.md`) and the long-term memory is managed through `MEMORY.md`, daily notes, and `DREAMS.md` for dreaming sweep summaries, using hybrid vector/keyword search if an embedding provider is present. 
6. **Multi-agent and Routing**: Claude Code uses task-delegating subagents (Explore, Plan, General-purpose) with worktree isolation and return summaries only. OpenClaw separates multi-agent routing (independent agents with distinct workspaces and auth profiles) and sub-agent delegation (background runs with configurable nesting depth up to 5).
