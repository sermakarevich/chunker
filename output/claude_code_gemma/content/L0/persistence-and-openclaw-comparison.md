# persistence-and-openclaw-comparison

**Parent:** [[content/L1/claude-code-memory-persistence-subagents|claude-code-memory-persistence-subagents]] — Claude Code implements a four-level CLAUDE.md memory hierarchy, a five-layer graduated compaction pipeline for context management, and an append-only JSONL persistence system for sessions and subagent sidechains. It utilizes isolated subagents with three isolation modes (Worktree, Remote, In-process) and a sidechain transcript design to prevent parent session inflation.

Claude Code employs three independent persistence channels for durable storage: (1) Session transcripts, which are project-scoped and stored as one file per session, containing conversation records (user, assistant, attachment, and system messages) as well as compaction and other metadata events; (2) Global prompt history, which stores only user prompts in `history.jsonl` at the Claude configuration home directory, managed by `history.ts` and accessed via the `makeHistoryReader()` generator and `readLinesReverse()` to support Up-arrow and ctrl+r navigation; and (3) Subagent sidechains, consisting of separate `.jsonl` and `.meta.json` files for each subagent. 

Session transcripts utilize an append-only JSONL format to prioritize auditability and simplicity over query power, ensuring every event is human-readable and reconstructable without specialized tooling. Beyond simple messages, transcripts store compaction markers, file-history snapshots, attribution snapshots, and content-replacement records. The session identity system pairs a `sessionId` with a `sessionProjectDir`, both of which are set during resume or branch operations. To ensure hooks operate in the correct directory, the transcript path must use the same project directory that was active when the messages were originally written.

When using the `--resume` flag, `conversationRecovery.ts` rebuilds the conversation by replaying the transcript. Forking, handled by `commands/branch/branch.ts`, creates a new session from an existing one. Crucially, both resume and fork operations do not restore session-scoped permissions; users must re-grant permissions in the new session. This safety-conservative design treats sessions as isolated trust domains to prevent the risk of carrying stale trust decisions into a changed context.

To support persistence, the `compact_boundary` marker and the `annotateBoundaryWithPreservedSegment()` function in `compact.ts` record `headUuid`, `anchorUuid`, and `tailUuid` in the boundary event. These UUIDs allow the session loader to patch the message chain at read time, ensuring that preserved messages maintain their original `parentUuids` on disk while the loader links them correctly. This append-only design ensures that compaction never modifies or deletes previously written transcript lines. Additionally, Claude Code implements "checkpoints" as file-history snapshots for the `--rewind-files` command, stored at `~/.claude/filehistory/<sessionId>/` to revert filesystem changes.

In a comparative analysis with OpenClaw, an independent open-source AI agent system, the two differ across six dimensions:
1. **System Scope**: Claude Code is an ephemeral CLI process bound to a single repository session. OpenClaw is a persistent WebSocket gateway daemon (default port 18789, loopback-only) that coordinates multi-channel messaging (e.g., WhatsApp, Telegram, Slack) via a typed WebSocket protocol.
2. **Trust Model**: Claude Code uses a deny-first per-action rule evaluation with hooks, an ML classifier, and seven permission modes. OpenClaw assumes a single trusted operator per gateway and uses identity/access control (DM pairing codes, sender allowlists) and opt-in sandboxing (Docker, SSH, or OpenShell) with configurable scope (per-agent, per-session, or shared).
3. **Agent Runtime**: Claude Code's center is the `queryLoop()` async generator. OpenClaw uses an embedded Pi-agent core inside a gateway dispatch layer, where runs are serialized through per-session queues and an optional global lane.
4. **Extension Architecture**: Claude Code provides four mechanisms based on context cost (MCP, plugins, skills, hooks). OpenClaw uses a manifest-first plugin system with 12 capability types and a central registry, alongside a separate skills layer and MCP support via `openclaw mcp` commands.
5. **Memory and Context**: Claude Code utilizes a four-level `CLAUDE.md` hierarchy and a five-layer compaction pipeline. OpenClaw injects workspace bootstrap files (e.g., `AGENTS.md`, `SOUL.md`, `TOOLS.md`, `IDENTITY.md`, `USER.md`) and manages long-term memory through `MEMORY.md`, daily notes, and an optional `DREAMS.md` for dreaming sweep summaries, using hybrid vector/keyword search when an embedding provider is present.
6. **Multi-agent and Routing**: Claude Code uses task-delegating subagents (Explore, Plan, general-purpose) with worktree isolation and summary-only returns. OpenClaw separates multi-agent routing (independent agents with distinct workspaces and auth profiles) from sub-agent delegation (background runs with configurable nesting depth up to 5).
