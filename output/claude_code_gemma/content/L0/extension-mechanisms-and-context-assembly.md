# extension-mechanisms-and-context-assembly

**Parent:** [[content/L1/claude-code-architecture-deep-dive|claude-code-architecture-deep-dive]] — Claude Code utilizes a reactive ReAct loop (queryLoop) and a seven-layer safety system including deny-first rule evaluation and ML-based auto-mode classification. Context is managed via a five-layer compaction pipeline and a four-level CLAUDE.md hierarchy, while tools are assembled via a five-step pipeline and extended through MCP servers, plugins, and skills.

Claude Code's agent loop features three primary injection points: `assemble()` which controls what the model sees, `model()` which controls what the model can reach, and `execute()` which controls whether and how an action actually runs. The system employs four distinct extension mechanisms to manage the action surface and agent behavior:

1. **MCP Servers**: These provide external service integration using a multi-transport approach via an internal `claudeai-proxy`. Each connected server contributes tool definitions as `MCPTool` objects. The built-in tools `ListMcpResourcesTool` and `ReadMcpResourceTool` provide access to MCP resources. This mechanism has a high context cost due to the size of tool schemas.
2. **Plugins**: Plugins function as both a packaging format and a distribution mechanism. The `PluginManifestSchema` (in `utils/plugins/schemas.ts`) supports ten component types: commands, agents, skills, hooks, MCP servers, LSP servers, output styles, channels, settings, and user configuration. The `pluginLoader.ts` validates these manifests and routes components to registries: commands and skills use the `SkillTool` meta-tool, agents are used by `AgentTool`, hooks merge into the hook registry, MCP and LSP servers use standard configurations, and output styles modify response formatting. Plugins have a medium context cost.
3. **Skills**: Skills are defined by `SKILL.md` files with YAML frontmatter. The `parseSkillFrontmatterFields()` function in `loadSkillsDir.ts` parses over 15 fields, including display name, description, allowed tools, argument hints, model overrides, execution context ('fork' for isolated execution), associated agent definitions, effort levels, and shell configuration. Skills can define dynamic hooks that register upon invocation. When the `SkillTool` meta-tool invokes a skill, it injects the skill's instructions into the context. Skills have a low context cost because only frontmatter descriptions, not full content, remain in the prompt.
4. **Hooks**: The system defines 27 hook events across six categories: tool authorization (`PreToolUse`, `PostToolUse`, `PostToolUseFailure`, `PermissionRequest`, `PermissionDenied`), session lifecycle (`SessionStart`, `SessionEnd`, `Setup`, `Stop`, `StopFailure`), user interaction (`UserPromptSubmit`, `Elicitation`, `ElicitationResult`), subagent coordination (`SubagentStart`, `SubagentStop`, `TeammateIdle`, `TaskCreated`, `TaskCompleted`), context management (`PreCompact`, `PostCompact`, `InstructionsLoaded`, `ConfigChange`), and workspace events (`CwdChanged`, `FileChanged`, `WorktreeCreate`, `WorktreeRemove`) and notifications (defined in `coreTypes.ts` and `coreSchemas.ts`). Fifteen of these have event-specific output schemas in `types/hooks.ts` for permission decisions, context injection, and input modification. Persisted hook commands (configured via `settings.json` or plugins) use four types: shell commands, LLM prompt hooks, HTTP hooks, and agentic verifier hooks (defined in `schemas/hooks.ts`). Additionally, the system supports non-persistable callback hooks for the SDK and internal instrumentation. Hooks have a zero context cost by default.

**Tool Pool Assembly**
The `assembleToolPool()` function in `tools.ts` serves as the single source of truth for combining built-in and MCP tools via a five-step pipeline:
1. **Base tool enumeration**: `getAllBaseTools()` returns up to 54 tools. Nineteen are always included (e.g., `BashTool`, `FileReadTool`, `AgentTool`, `SkillTool`), and 35 are conditional based on feature flags, user type, or environment variables. Internal users get additional tools; Worktree mode enables `EnterWorktreeTool` and `ExitWorktreeTool`; Agent swarms enable team tools. `GlobTool` and `GrepTool` are omitted if embedded search tools are available in the Bun binary.
2. **Mode filtering**: `getTools()` applies mode-specific filtering. In `CLAUDE_CODE_SIMPLE` mode, only `Bash`, `Read`, and `Edit` are available (or `REPLTool` in the REPL branch, plus coordinator tools).
3. **Deny rule pre-filtering**: `filterToolsByDenyRules()` removes blanket-denied tools from the model's view.
4. **MCP tool integration**: MCP tools from `appState.mcp.tools` are filtered by deny rules and merged with built-in tools.
5. **Deduplication**: Tools are deduplicated by name, with built-in tools taking precedence over MCP tools.

`REPL.tsx` and `AgentTool.tsx` both use this assembly process. Deferred tools may be hidden from context until queried via `ToolSearch` in `tools.ts`. Custom agent definitions via `.claude/agents/*.md` and plugin agents create isolated context windows and are handled separately from these four mechanisms.

**Context Construction and Memory**
Claude Code assembles its context window from several sources: (1) the system prompt (including output style modifications and `--append-system-prompt` content), (2) environment info via `getSystemContext()` in `context.ts` (including git status and `BREAK_CACHE_COMMAND` injections), (3) the `CLAUDE.md` hierarchy via `getUserContext()` in `context.ts`, (4) path-scoped rules (loaded lazily when reading files in matching directories), (5) asynchronously prefetched auto memory, (6) tool metadata (skill descriptions, MCP tool names, and deferred tool definitions via `ToolSearch`), (7) conversation history (subject to compaction), (8) tool results (file reads, command outputs, subagent summaries), and (9) compact summaries. 

In `query.ts`, the system prompt is assembled using `asSystemPrompt(appendSystemContext(systemPrompt, systemContext)())`. User context from `CLAUDE.md` and the date is added via `prependUserContext()`. Late-injection sources include relevant-memory prefetch, MCP instruction deltas, agent listing deltas, and background agent task notifications.

**CLAUDE.md Hierarchy and Auto Memory**
To ensure transparency, `CLAUDE.md` files use plain-text Markdown rather than a database or embeddings. The system does not use a vector similarity index; it employs an LLM-based scan of memory-file headers to select up to five relevant files on demand. This allows users to directly read, edit, and version-control all instructions the agent sees.
