# claudemd-compaction-subagents-persistence

**Parent:** [[content/L1/claude-code-memory-persistence-subagents|claude-code-memory-persistence-subagents]] — Claude Code implements a four-level CLAUDE.md memory hierarchy, a five-layer graduated compaction pipeline for context management, and an append-only JSONL persistence system for sessions and subagent sidechains. It utilizes isolated subagents with three isolation modes (Worktree, Remote, In-process) and a sidechain transcript design to prevent parent session inflation.

Claude Code's CLAUDE.md files use a multi-level loading hierarchy defined in `claudemd.ts`. This hierarchy consists of four memory types: (1) Managed memory, which provides OS-level policy for all users (e.g., `/etc/claude-code/CLAUDE.md` on Linux); (2) User memory, which contains private global instructions located at `~/.claude/CLAUDE.md`; (3) Project memory, which includes instructions checked into the codebase at `CLAUDE.md`, `.claude/CLAUDE.md`, and `.claude/rules/*.md` in project roots; and (4) Local memory, which is gitignored and stores private project-specific instructions in `CLAUDE.local.md` in project roots. File discovery traverses from the current directory up to the root, checking for all project and local memory files in each directory; files closer to the current directory have higher priority and are loaded later. 

Files load in "reverse order of priority," meaning later-loaded files receive more model attention. For root-to-CWD directories, unconditional rules from `.claude/rules/*.md` load eagerly at startup. For nested directories below the current working directory (CWD), even unconditional rules are loaded lazily when the agent reads files in matching directories, allowing the model's instruction set to evolve during a conversation. CLAUDE.md content is delivered as user context (a user message) via `context.ts`, rather than as system prompt content. Because CLAUDE.md content is conversational context, model compliance is probabilistic rather than guaranteed. This creates a deliberate separation between guidance (probabilistic CLAUDE.md) and enforcement (deterministic permission rules evaluated in deny-first order). The `setCachedClaudeMdContent()` function caches loaded content for the auto-mode classifier to avoid an import cycle between the CLAUDE.md loader and the permission system. Memory files also support an `@include` directive for modular instruction sets via `processMemoryFile()` in `claudemd.ts`, supporting syntax variants `@path`, `@./relative`, `@~/home`, and `@/absolute` in leaf text nodes (not inside code blocks). The implementing file is pushed first, then included files are appended; circular references are prevented by tracking processed paths, and non-existent files are silently ignored.

To address the context window bottleneck, Claude Code implements a five-layer compaction pipeline in `query.ts` using graduated compression. This pipeline applies five layers in sequence with increasing aggressiveness, applying the least disruptive compression first. Three layers are gated by feature flags, budget reduction is always active, and auto-compact is user-configurable. The five layers are: (1) Budget reduction (always active), which enforces per-tool-result size limits; (2) Snip (gated by `HISTORY_SNIP`), which performs lightweight older-history trimming; (3) Microcompact (gated by `CACHED_MICROCOMPACT`), which provides fine-grained cache-aware compression; (4) Context collapse (gated by `CONTEXT_COLLAPSE`), which is a read-time virtual projection over history; and (5) Auto-compact (enabled by default, can be disabled), which produces a full model-generated summary.

The `buildPostCompactMessages()` function in `compact.ts` returns a compacted output structure: `[boundaryMarker, ...summaryMessages, ...messagesToKeep, ...attachments, ...hookResults]`. The boundary marker is annotated with preserved-segment metadata (headUuid, anchorUuid, and tailUuid) via `annotateBoundaryWithPreservedSegment()` to enable read-time chain patching. This design ensures compaction never modifies or deletes previously written transcript lines, only appending new boundary and summary events. The `compactConversation()` function in `compact.ts` triggers pre-compact hooks for custom instructions. A GrowthBook feature flag determines if the compaction path reuses the main conversation's prompt cache (a January 2026 experiment noted that the "false" path results in a 98% cache miss and costs ~0.76% of fleet `cache_creation`). After compaction, attachment builders re-announce runtime state (plans, skills, and async agents) from live app state, as compaction discards prior attachment messages but not the underlying state.

When Claude Code delegates work to subagents, it uses an isolated subagent boundaries principle. The `Agent` tool (`AgentTool.tsx`), with `Task` as a legacy alias, is the delegation mechanism. The model invokes `Agent` with a structured input including a delegation prompt, optional subagent type, and configurations for isolation mode, permission overrides, and working directory. The `Agent` tool input schema uses feature-gated fields; the `isolation` field offers `worktree` and `remote` for internal users and only `worktree` for external users. The `cwd` field is gated by a feature flag, and the `run_in_background` field is omitted when background tasks are disabled or when fork-subagent mode is enabled.

Claude Code provides up to six built-in subagent types: (1) Explore (read/search-oriented, with write/edit tools in its deny-list); (2) Plan (creates structured plans, with execution proceeding through the standard permission model); (3) General-purpose (broadly capable); (4) Claude Code Guide (onboarding and documentation assistance with a specific `permissionMode` override); (5) Verification (runs validation checks like test suites and linting); and (6) Statusline-setup (specialized for terminal status line configuration). Users can also define custom subagents via `.claude/agents/*.md` files, and plugins contribute agent definitions via `loadPluginAgents.ts`. For custom agents, the markdown body is the system prompt, and YAML frontmatter specifies the description, tools (allowlist), disallowedTools, model, effort, permissionMode, mcpServers, hooks, maxTurns, skills, memory scope, background flag, and isolation mode. JSON-formatted agent definitions support these same fields plus a `prompt` field via `loadAgentsDir.ts`. While `SkillTool` injects instructions into the current context window, `AgentTool` spawns a new, isolated one. Consequently, most subagent invocations require a self-contained prompt because the default path does not inherit the parent's conversation history (except for the fork-subagent path).

Subagent isolation supports three modes in `AgentTool.tsx`: (1) Worktree, which creates a temporary git worktree for repository modification without affecting the parent; (2) Remote (internal-only), which launches in a remote Claude Code Remote environment in the background; and (3) In-process (default), which shares the filesystem but operates in an isolated conversation context. Permission override logic in `runAgent.ts` applies a subagent's `permissionMode` unless the parent is already in `bypassPermissions`, `acceptEdits`, or `auto` mode, as these represent explicit user decisions. For async agents, the system checks `canShowPermissionPrompts`, then `bubble` mode (which always shows prompts as they escalate to the parent terminal), then the default (sync agents show prompts, async agents do not). Background agents that can show prompts set `awaitAutomatedChecksBeforeDialog: true` to ensure classifiers and hooks resolve before the user is interrupted. 

When `allowedTools` is explicitly provided to `runAgent()`, a two-tier permission scoping model applies: SDK-level permissions from `--allowedTools` are preserved for all agents, but session-level rules are replaced with the subagent's declared `allowedTools`. If `allowedTools` is not provided, the parent's session-level rules are inherited. 

Each subagent writes its own transcript as a separate `.jsonl` file with a `.meta.json` metadata file via `sessionStorage.ts` and `runAgent.ts`. This sidechain design prevents subagent histories from inflating the parent's session file. Only the subagent's final response text and metadata return to the parent's conversation context, respecting the "context as bottleneck" principle. Agent teams in plan mode consume approximately 7x the tokens of a standard session. For multi-instance coordination, the harness uses lock-file-based mutual exclusion for claiming tasks from a shared list, rather than a message broker or distributed coordination service, ensuring zero-dependency deployment and full debuggability through plain-text JSON files.

Regarding session persistence, Claude Code follows an append-only durable state principle. Session-scoped permissions live in memory only and are not serialized to the transcript, so resuming a session rebuilds the permission context from CLI args and disk settings. Session transcripts are stored as mostly append-only JSONL files at a project-specific path. The `getTranscriptPath()` function in `sessionStorage.ts` computes this path by joining the `projectDir` (determined by `getSessionProjectDir()` during resume/branch or falling back to `getProjectDir(getOriginalCwd())()`) with the session ID (`${getSessionId()}.jsonl`).
