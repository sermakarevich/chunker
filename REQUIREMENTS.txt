# Hierarchical Chunking System — Specification

## Goal

Implement a Python application that processes a document into a hierarchically connected
structure of logically self-sufficient chunks and multi-level summaries. The output is
designed for hierarchical agentic search, enabling traversal from broad concept summaries
down to fine-grained, context-enriched text chunks.

---

## Data Model

Define these entities before any implementation. All state is built around them.

```python
@dataclass
class Chunk:
    id: str                       # uuid
    source_span: tuple[int, int]  # (start_char, end_char) in original document
    original_text: str            # verbatim source text
    rewritten_text: str           # LLM-rewritten, self-sufficient version (used for embedding)
    summary: str                  # short summary of this chunk
    parent_block_id: str          # id of the L1 SummaryBlock this chunk belongs to
    forced_split: bool = False    # True if max size or attempts were exhausted
    level: int = 0                # always 0 for chunks

@dataclass
class SummaryBlock:
    id: str                          # uuid
    level: int                       # 1 = groups chunks, 2 = groups L1 blocks, etc.
    child_ids: list[str]             # ordered list of Chunk.id or SummaryBlock.id
    summary_text: str                # LLM-generated summary of this block
    parent_block_id: str | None      # None if this is the root block
```

The final output is a tree rooted at one or more top-level `SummaryBlock` nodes, with
`Chunk` nodes as leaves. All nodes carry `parent_block_id` / `child_ids` for bidirectional
traversal.

---

## Configuration

All thresholds are named parameters — no magic numbers in code.

```python
@dataclass
class ChunkerConfig:
    # Chunking
    split_strategy: str          # "words" | "paragraphs" | "sentences" | custom
    min_chunk_tokens: int        # minimum tokens before completeness is checked
    max_chunk_tokens: int        # hard ceiling; force-split if reached
    max_expansion_attempts: int  # max LLM completeness checks before forcing a split

    # Summarisation
    summary_aggregation_threshold: int  # cumulative token count of pending summaries
                                        # that triggers aggregation into a higher-level block
    min_group_size: int = 2             # minimum summaries per group (hard constraint)
    max_group_size: int = 8             # maximum summaries per group (soft hint to model)

    # Context injection
    context_budget_tokens: int   # max tokens of metadata injected per LLM call

    # LLM
    model: str
    temperature: float
```

## Processing Pipeline

### Phase 1 — Chunking (sequential, cursor-driven)

**Step 1.1 — Initial split**

Apply the configured split strategy to produce candidate chunk windows. This is a
read-only operation on the source text; no LLM calls yet.

Splitting is abstracted behind a `Splitter` interface:

```python
class Splitter(Protocol):
    def split(self, text: str, config: ChunkerConfig) -> list[str]: ...
```

Provided implementations: `WordSplitter`, `ParagraphSplitter`, `SentenceSplitter`.
Custom strategies can be injected.

**Step 1.2 — Completeness check loop**

For each candidate chunk window, call the LLM with the following inputs:

- The candidate chunk text
- Injected metadata context (see Context Injection section)
- Instruction: determine whether this chunk ends at a logically complete boundary,
  or whether the thought continues into the next chunk.

  Strong indicators that a chunk IS complete and a new one should begin:
  - A new paragraph that opens with a clearly different subject or concept
  - A transition phrase ("Another approach...", "In contrast...", "Moving on to...")
  - A section or subheading break

  A new paragraph alone is NOT sufficient — lists, examples, and continuations
  frequently start new paragraphs mid-thought. The topic shift must accompany it.

The LLM responds with:
```json
{
  "complete": true | false,
  "boundary_phrase": "exact verbatim phrase from the text where the new chunk begins"
}
```

**If `complete: false`:** expand the window by one split unit and repeat.
Stop expanding when either:
- The LLM returns `complete: true`
- `max_chunk_tokens` is reached → force a split at the last sentence boundary and
  emit a `WARN: forced_split` log entry. The chunk is marked `forced_split: true`.
- `max_expansion_attempts` is exhausted → same forced-split behaviour as above.

**If `complete: true`:** validate the `boundary_phrase`:
1. Search for the exact phrase as a substring in the remaining document text.
2. If not found, re-prompt the LLM once with an explicit instruction:
   "The phrase was not found verbatim in the source text. Return the boundary_phrase
   exactly as it appears in the document, character-for-character, including
   punctuation and capitalisation."
3. If the retry also fails to produce a matching substring, log `WARN: phrase_not_found`
   and split at the last sentence boundary instead. No fuzzy matching is performed —
   the retry addresses the actual root cause (model paraphrasing) more directly than
   approximate string matching would.

The cursor advances to the validated `boundary_phrase` position (or fallback position).

**Step 1.3 — Chunk rewrite**

Once a chunk boundary is confirmed, call the LLM to rewrite the chunk so it is
self-sufficient without requiring the surrounding document for comprehension.
This involves resolving pronoun references, clarifying implied subjects, and
prepending minimal framing where needed.

Both artifacts are stored:
- `original_text` — verbatim source, preserved for citation and auditing
- `rewritten_text` — used for embedding and retrieval

The LLM also produces a short `summary` (1–3 sentences) of the chunk at this step.

**Step 1.4 — Checkpoint**

After each confirmed chunk, write the `Chunk` object to the checkpoint store
(see Checkpointing section) before advancing the cursor. If the process is
interrupted, resume from the last checkpointed chunk.

---

### Phase 2 — Summary Aggregation (bottom-up, level by level)

Summary aggregation is triggered when the cumulative token count of pending summaries
at the current level exceeds `summary_aggregation_threshold`.

**Important:** aggregation groupings must be **contiguous runs**. The LLM is asked to
identify where one logical concept ends and the next begins — not to produce arbitrary
clusters. Non-contiguous groupings are rejected and the LLM is re-prompted with an
explicit contiguity constraint.

**Step 2.1 — Group summaries**

Present the ordered list of pending summaries to the LLM with injected metadata context.

Instruct the model that:
- Each group must contain at least `min_group_size` summaries (default: 2). A group
  of 1 is a degenerate case — it produces a higher-level block semantically identical
  to its single child, adding a level without adding information.
- As a soft guideline, prefer groups of no more than `max_group_size` summaries
  (default: 8) to avoid excessively coarse blocks. A single dense concept may
  legitimately exceed this — it is a hint, not a hard limit.
- All groups must be contiguous runs; reordering or skipping summaries is not permitted.

The LLM responds with:
```json
{
  "groups": [
    {
      "boundary_after_index": 3,
      "concept_label": "short label for this concept"
    }
  ]
}
```

`boundary_after_index` defines where one contiguous group ends. Groups are derived
from these boundaries — no explicit per-group membership list is needed.

**Post-response validation:**
- If any group has fewer than `min_group_size` children: reject the response and
  re-prompt with the constraint stated explicitly. This is a hard rejection.
- If any group exceeds `max_group_size` children: re-prompt asking the model to
  subdivide that group. This is a soft rejection — accept the original response
  if the model declines to subdivide.
- If hard validation fails twice consecutively: fall back to even-sized grouping
  (divide the summaries into `ceil(n / 4)` contiguous groups of roughly equal size)
  and log `WARN: grouping_fallback`.

**Step 2.2 — Generate block summary**

For each identified group, call the LLM to produce a `summary_text` for the
`SummaryBlock`. Inject the summaries of all child nodes as input.

**Step 2.3 — Recurse**

The new `SummaryBlock.summary_text` values at level N become the pending summaries
for level N+1. Repeat Steps 2.1–2.2 until the total pending summaries fall below
`summary_aggregation_threshold` or only one block remains (root).

**Short document handling:** if the entire document produces fewer pending summaries
than `summary_aggregation_threshold` after Phase 1, skip aggregation entirely. The
output is a flat list of chunks with no intermediate `SummaryBlock` layer.

## Context Injection

Every LLM call (completeness check, rewrite, grouping, block summary generation) receives
injected metadata context to ground the model in the document's broader narrative.

Context is assembled by the `ContextBuilder`, which respects `context_budget_tokens`
as a hard ceiling. Insertion follows this **priority order** (highest first):

1. **Same-level immediate predecessor** — the summary or `rewritten_text` of the
   chunk/block directly before the current one. This is the highest priority because
   it resolves local references, pronouns, and "as mentioned above" continuations.

2. **Latest summary from each higher level** — the most recent `summary_text` from
   L1, L2, … up to the root. These provide topic framing and document-level context.

3. **Earlier same-level context** — the predecessor before the immediate predecessor,
   then the one before that, walking backwards until the budget is exhausted.

If inserting the next item would exceed `context_budget_tokens`, that item is skipped
and the builder tries the next priority item. Partial insertion of a single item is
not permitted; items are inserted whole or not at all.

```python
class ContextBuilder:
    def build(
        self,
        current_id: str,
        level: int,
        state: ProcessingState,
        budget_tokens: int,
    ) -> str:
        """Returns the assembled context string for injection into the LLM prompt."""
        ...
```

**Note on hierarchy availability:** when processing early chunks (Phase 1), no
higher-level summaries exist yet. The builder silently omits unavailable levels.
Later chunks will receive richer context as the hierarchy fills in. This is
expected and acceptable — it is not a bug.

---

## Hierarchy Building Order

The system uses a **hybrid approach**:

- **Phase 1 (chunking)** is sequential and cursor-driven. Higher-level summaries are
  injected if they exist (built from previously completed blocks), but are not
  prerequisites. Early chunks are processed with whatever context is available.

- **Phase 2 (aggregation)** is a batch pass per level, executed after all chunks
  at the level below are complete. This ensures every grouping decision has access
  to the full peer set at that level.

This means the first time a full cross-document higher-level summary exists is after
the first aggregation pass. For very long documents this may happen mid-processing
and will be available for subsequent chunk rewrites.

---

## State Management

All mutable state lives in a `ProcessingState` object:

```python
@dataclass
class ProcessingState:
    document_id: str
    source_text: str
    cursor_position: int                          # current char offset in source_text
    chunks: list[Chunk]                           # completed chunks, ordered
    summary_blocks: dict[int, list[SummaryBlock]] # level → ordered blocks
    pending_summaries: list[str]                  # summaries awaiting next aggregation pass
    checkpoint_path: str
```

---

## Checkpointing and Resumability

After every completed `Chunk` or `SummaryBlock`, serialize the full `ProcessingState`
to `checkpoint_path` (JSON or msgpack). On startup, check for an existing checkpoint
matching `document_id`; if found, restore state and resume from `cursor_position`.

This ensures that a failure partway through a long document does not require reprocessing
from scratch.

---

## LLM Call Abstraction

All LLM interactions go through a single `LLMClient` interface to allow swapping
providers and simplifying testing:

```python
class LLMClient(Protocol):
    def complete(self, system: str, user: str, response_schema: dict) -> dict: ...
```

Structured JSON responses are enforced via the response schema at every call site.
All calls are wrapped in retry logic with exponential backoff (3 attempts by default).
Each call is logged with: call type, token counts, model response, and any warnings.

---

## Output Format

The final output is a JSON document suitable for ingestion by a search system:

```json
{
  "document_id": "...",
  "root_blocks": ["<SummaryBlock id>"],
  "blocks": {
    "<id>": {
      "type": "summary_block",
      "level": 2,
      "summary_text": "...",
      "child_ids": ["...", "..."],
      "parent_block_id": "..."
    }
  },
  "chunks": {
    "<id>": {
      "type": "chunk",
      "source_span": [0, 412],
      "original_text": "...",
      "rewritten_text": "...",
      "summary": "...",
      "parent_block_id": "...",
      "forced_split": false
    }
  }
}
```

The search consumer can:
- Embed `rewritten_text` of each `Chunk` for dense retrieval
- Embed `summary_text` of each `SummaryBlock` for coarse-level retrieval
- Traverse parent/child links for hierarchical re-ranking

---

## Search Interface Contract (consumer-facing)

The output hierarchy supports the following traversal patterns for agentic search:

- **Top-down**: start at root `SummaryBlock`, select the most relevant child block
  by summary similarity, recurse until reaching leaf chunks.
- **Flat chunk search**: embed and search only leaf `Chunk.rewritten_text` values,
  then expand upward via `parent_block_id` for context.
- **Level-targeted search**: search only at a specific level (e.g. L1 summaries) for
  coarse topic identification before drilling down.

---

## Logging and Observability

Every LLM call emits a structured log entry:

```json
{
  "event": "completeness_check | chunk_rewrite | summary_group | block_summary",
  "chunk_or_block_id": "...",
  "tokens_prompt": 412,
  "tokens_completion": 87,
  "context_items_injected": ["chunk:prev", "l1_summary:latest"],
  "decision": { ... },
  "warnings": ["forced_split", "phrase_not_found", "grouping_fallback"]
}
```

This enables offline quality auditing: you can replay every LLM decision and identify
where boundaries were forced, phrases failed to match, or grouping fell back to
even-sized splitting.

---

## Document Update Strategy

For the initial version, document updates trigger full reprocessing. The checkpoint
format is designed to make partial reprocessing feasible in a future version (matching
unchanged `source_span` ranges to previously computed chunks), but this is out of scope
for v1.
