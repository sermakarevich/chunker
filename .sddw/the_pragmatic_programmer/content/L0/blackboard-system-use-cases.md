# blackboard-system-use-cases

**Parent:** [[content/L1/professional-development-architecture-methods|professional-development-architecture-methods]] — Software development emphasizes mastery across tooling, process, and architecture. Key techniques include using PERT (requiring optimistic, most likely, and pessimistic estimates) and iterative development; leveraging command shells and advanced editors for maximum efficiency; and employing VCS features like branching and CI/CD for collaboration. Architecturally, developers should adopt a transformational perspective, favoring explicit APIs, using FSMs or PubSub for decoupling, and utilizing the Actor Model or Blackboard systems for concurrent state management. For example, transferring information between machines requires considering time versus distance, with a 1 TGB tape transfer over a 1 Gbps line taking roughly 9,000 seconds (2.5 hours) and necessitating a distance of almost 9 miles to outperform a 3.5 mph courier.

For image processing, if the work involves simple scheduling among parallel processes, a shared work queue may suffice. However, if feedback is involved—meaning that the results of one processed chunk affect other chunks, such as in machine vision applications or complex 3D image-warp transforms—developers should consider implementing a blackboard system. 

For group calendaring, a blackboard system may be suitable. Participants can post scheduled meetings and availability to the blackboard, which is appropriate because the entities function autonomously, feedback from decisions is important, and participants may come and go. The developers can partition this blackboard system based on the user performing the search: junior staff might only need to search the immediate office, human resources might need to search only English-speaking offices worldwide, and the CEO might need the entire scope. Regarding data formats, the developers have flexibility: they can ignore formats or languages they do not understand. Instead, the developers only need to understand different formats for offices that have meetings with each other, and they do not need to expose all participants to a full transitive closure of all possible formats. This practice reduces coupling only to where it is necessary, avoiding artificial constraints.

For a network monitoring tool, the structure is highly similar to the mortgage or loan application program. Here, users send trouble reports and automated systems report statistics, with both types of data posting to the blackboard. A human or software agent can analyze the blackboard to diagnose network failures: for instance, two errors on a line might be attributed to cosmic rays, but 20,000 errors suggest a hardware problem. Similar to how detectives solve a murder mystery, multiple entities can analyze the blackboard and contribute ideas to solve the network problems.
