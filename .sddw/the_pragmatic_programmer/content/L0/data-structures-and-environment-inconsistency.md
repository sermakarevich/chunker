# data-structures-and-environment-inconsistency

**Parent:** [[content/L1/software-engineering-mastery-and-security-principles|software-engineering-mastery-and-security-principles]] — Professional proficiency requires mastery of foundational tools, adopting structured estimation techniques like PERT and iterative cycles, and achieving fluency in command shells and advanced editors. System design demands a defensive, transformational approach utilizing concepts such as Principle of Least Privilege, minimizing attack surface area, and implementing advanced architectural patterns like PubSub and Actor Model, alongside specific details like encrypting all sensitive data and sanitizing external input.

The assumption that a list of key-value pairs requires a unique key is generally accurate, and hash libraries typically enforce this uniqueness either through the hash's internal behavior or by generating explicit error messages for duplicated keys. However, an array does not automatically enforce this constraint; the array can happily store duplicate keys unless the developer explicitly codes the system to prevent it. In such a scenario, the first key encountered that matches `DepositAccount` is accepted, and any subsequent matching entries are ignored. Because the order of entries in an array is not guaranteed, the retrieval results may be inconsistent, meaning that sometimes the operation works, and sometimes it does not.

Furthermore, the differences in machine environments between development and production should be treated as coincidence, not as defining characteristics of the system.
