**Contextual Information:**

*   **Naming and Clarity:** The naming of applications, subsystems, modules, functions, and variables is highly important because names convey the role an element plays within the code. Developers must pause while creating code and consider the motivation for creating a specific element; this practice shifts attention from immediate problem-solving to considering the bigger picture. Deeper reflection on the role of a variable or function prompts developers to consider what is special about the element and what it interacts with. Often, this deeper reflection reveals that the intended action lacks logical consistency if the name is inappropriate.
*   **Cognitive Load:** The human brain processes written words very quickly, demonstrating that words have a high cognitive priority. This phenomenon, illustrated by the Stroop effect, shows that reading written words is easier than recognizing colors. Therefore, developers must ensure that the names used in the code live up to this cognitive priority. For example, when defining an instance method that calculates a discount, the method name and parameter name should clearly convey the intent. Instead of using `public void deductPercent(double amount)`, a better pattern is to use `public void applyDiscount(Percentage discount)`, which explicitly documents the expected range (e.g., between 0 and 100, or 0.0 and 1.0) by changing the parameter type from a generic `double` to a defined `Percentage` type.
*   **Contextual Naming:** Clarity is also achieved by giving functions context-specific names. For example, if a module calculates a Fibonacci number, calling the function `Fib.fib(n)` might be confusing because it relies on the module's context. Using a name like `Fib.of(0)` or `Fib.nth(20)` clarifies the function's intent regardless of the calling context. The local programming conventions must also be honored: for instance, in the C language, developers traditionally use `i`, `j`, and `k` for loop increment variables, and `s` for a character string; applying a convention from one environment (like using `i` for a string variable in Clojure) is incorrect.
