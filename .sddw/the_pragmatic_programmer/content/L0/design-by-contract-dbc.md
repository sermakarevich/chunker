# design-by-contract-dbc

**Parent:** [[content/L1/professional-software-engineering-workflows|professional-software-engineering-workflows]] — Professional mastery requires mastering specialized tools and methodologies, including the systematic use of PERT (optimistic, likely, pessimistic estimates) and iterative development to manage project uncertainty; developers must achieve 'editor fluency' and leverage specialized text manipulation languages like Python and Ruby to build complex workflows, while defensive programming principles, notably Design by Contract (DBC) defined by Bertrand Meyer, mandate explicitly guaranteeing module boundaries and expected states.

Design by Contract (DBC) is a powerful, simple technique developed by Bertrand Meyer in *Object-Oriented Software Construction* for the Eiffel language. DBC focuses on explicitly documenting and agreeing upon the rights and responsibilities of software modules to guarantee program correctness. A correct program is defined as one that performs no more and no less than it claims to do; therefore, documenting and verifying this claim is the core objective of DBC. Every function and method within a software system must define its expected state before execution and its guaranteed state upon conclusion. Meyer formalized these expectations and claims into three categories:

1. **Preconditions:** These define what must be true before a routine can be called, serving as the routine’s explicit requirements. The calling party is responsible for ensuring that the preconditions are met and that good data is passed to the routine.
2. **Postconditions:** These specify what the routine guarantees to achieve—the state of the world when the routine finishes. If a routine has a postcondition, the routine must conclude, meaning infinite loops are disallowed.
3. **Class invariants:** These ensure that a class maintains a condition that is always true from the perspective of any caller. Although the invariant might not hold during the routine's internal processing, it must be true by the time the routine exits and control returns to the caller. (Note that a class cannot grant unrestricted write-access to any data member involved in the invariant.)

The contract established between a routine and any potential caller states that if the routine’s preconditions are met by the caller, the routine guarantees that all postconditions and invariants will be true upon completion. If either party fails to meet the terms of this contract, a predefined remedy is invoked (such as raising an exception or terminating the program); regardless of the outcome, failure to uphold the contract constitutes a bug, and therefore, preconditions should not be used for things like user-input validation.

Some languages, such as Clojure, support pre- and post-conditions, along with comprehensive instrumentation via `specs`. For instance, when implementing a function to accept a deposit on a bank account, one must define two preconditions: that the deposit amount must be greater than 0.00, and that the specified account must be open and valid. A postcondition is required to guarantee that the newly created transaction (the return value) can be found among the account's transactions. If the function is called with invalid input (e.g., a negative amount or an invalid account ID), the system will raise a runtime exception, such as a `java.lang.AssertionError`.

In other languages, features like Elixir's guard clauses offer a mechanism for dispatching function calls against multiple available bodies. This approach is preferable to simply checking inputs, as it prevents the function from being called entirely if the provided arguments fall outside the acceptable range.

Design by Contract (DBC) requires developers to be strict about what they accept before starting and to promise as little as possible in return. This concept applies regardless of whether the programming language is functional, object-oriented, or procedural. While Design by Contract and Test-Driven Development (TDD), property-based testing, and defensive programming are all approaches aimed at program correctness, DBC offers distinct advantages: it does not require any setup or mocking, and it explicitly defines the parameters for success or failure for all possible cases, unlike testing, which can only target one specific case at a time.
