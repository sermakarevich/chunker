# pragmatic-starter-kit-automation

**Parent:** [[content/L1/software-development-methodology-synthesis|software-development-methodology-synthesis]] — Effective software development mandates full automation of build, test, and deployment procedures (CI/CD), utilizing Version Control to trigger processes via commits and ensuring roles are not siloed through the 'tracer bullet' method. Teams must adopt systematic estimation techniques like PERT (optimistic/most likely/pessimistic figures) and Iterative Estimation, focusing on delivering working software on demand while resisting the 'cargo cult' trap.

To develop software effectively, teams should adopt an automated, consistent, and repeatable approach, ensuring that every aspect of the project, including the build, release procedure, and testing, is automatic. This commitment to automation is crucial because manual procedures introduce inconsistency and variability, making repeatability dependent on chance. Since software development should behave like modern car starting—where pressing a button automates a complex, formerly manual process—the build, test, and deployment procedures must be fully automated and repeatable on any capable machine. The concept of the Pragmatic Starter Kit identifies three interrelated, critical pillars that support every project: Version Control, Regression Testing, and Full Automation.

**1. Version Control:** The team must store all project assets, including the build and release configuration, under version control. Utilizing version control makes build machines ephemeral, allowing them to be created on demand as spot instances in the cloud. Crucially, version control must drive the build and release process. Therefore, build, test, and deployment must be triggered by commits or pushes to the version control system and built within a cloud container. The process of releasing to staging or production should be specified using a tag within the version control system, making the release process highly low-ceremony and enabling true continuous delivery that is not dependent on any single machine or developer.

**2. Regression Testing:** Pragmatic Programmers proactively seek out bugs using unit tests (fine, small nets) to catch minor defects, while integration tests (big, coarse nets) target major system failures. Testing must be implemented early and frequently. Because catching a bug is significantly easier when it is first introduced, developers must write a large number of unit tests, dedicating time to this effort, as the time spent creating test code is inexpensive in the long run and increases the chance of achieving near-zero defects. Passing tests provides a high degree of confidence that a piece of code is finished. The automatic build must run all available tests, and for accurate results, the test environment must closely mirror the production environment to prevent bugs from developing in gaps.

**3. Full Automation:** The overall process demands that build, test, and deployment are automated steps. Methodologies like Scrum (for project management) and eXtreme Programming (XP) (for technical practices) are starting points, but developers must investigate these approaches independently, being careful not to become over-invested in any single methodology, which can lead to calcification and restrict adaptability. Continuous automation is necessary for every recurring task, including build and release procedures, to ensure consistency and repeatability across the entire project.
