# orthogonality-system-design

**Parent:** [[content/L1/system-design-principles-orthogonality-reversibility|system-design-principles-orthogonality-reversibility]] — The synthesis emphasizes that designing systems using Orthogonality (decoupling components so changes in one do not affect others), Reversibility (avoiding irreversible decisions by abstracting services), and DRY principles requires adopting specific coding practices. Techniques include writing 'shy code,' passing context explicitly rather than using global variables, employing the Strategy pattern for similar functions, and using formal specifications like OpenAPI for external APIs, while proactively preventing knowledge duplication across the development team through structured communication and peer review.

The principle of orthogonality can be applied to system design through several practices, which aim to minimize interdependency among system components. Many developers are already familiar with the need to design orthogonal systems, sometimes using terms such as modular, component-based, or layered to describe this process. Systems should be constructed from a set of cooperating modules, ensuring that each module's functionality remains independent from the others. When organizing these components into layers, each layer provides a specific level of abstraction. This layered approach is beneficial for designing orthogonal systems because each layer utilizes only the abstractions provided by the layers beneath it, thereby increasing flexibility in changing underlying implementations without affecting the code. Furthermore, layering helps reduce the risk of runaway dependencies between modules. A simple test for orthogonal design involves mapping out the components and asking: if a developer dramatically changes the requirements for one particular function, how many modules are affected? In an orthogonal system, the answer should be 'one.' For example, changing a button on a Graphical User Interface (GUI) panel should not require changes to the database schema, and adding context-sensitive help should not change the billing subsystem. 

When considering a complex system, such as monitoring and controlling a heating plant, a requirement change from a graphical user interface to a mobile interface demonstrates orthogonality. In an orthogonally designed system, changes would only affect the modules related to the user interface; the underlying logic for controlling the plant would remain unchanged. If the system is structured carefully, it should be able to support both interfaces using the same underlying codebase. It is equally important to assess the system's decoupling from real-world changes. Developers must avoid using uncontrollable external identifiers, such as telephone numbers as customer identifiers. Because phone companies reassign area codes, or because postal codes, Social Security Numbers, government IDs, email addresses, and domains are all external identifiers that are subject to change at any time, developers should not rely on the properties of things the team cannot control. 

Developers must also preserve the orthogonality when introducing third-party toolkits and libraries. Care must be taken to choose technologies wisely. When incorporating a toolkit or library, developers should ask whether the toolkit imposes changes on the code that are unnecessary. An object persistence scheme is considered orthogonal if it is transparent; if it requires the developer to create or access objects in a special way, it is not. Isolating such implementation details from the application code helps simplify future vendor changes. The Enterprise Java Beans (EJB) system serves as an example of orthogonality, where application code in transaction-oriented systems typically must delineate the start and end of each transaction. EJB expresses this information declaratively as annotations outside the methods that perform the work, allowing the same application code to run in different EJB transaction environments with no changes. This approach is analogous to the Decorator Pattern, which involves adding functionality without changing the core components, a programming style that can be used in almost every programming language and requires discipline rather than a specific framework or library.

**Coding Practices for Orthogonality**

Writing code inherently risks reducing a program’s orthogonality. Developers must constantly monitor not only their immediate actions but also the overall context of the application, as unintentionally duplicating functionality in another module or expressing existing knowledge twice can occur. Several techniques maintain orthogonality:

1. **Keep the code decoupled:** Developers should write 'shy code'—modules that do not reveal unnecessary details to other modules and do not rely on other modules’ implementations. Developers can follow the Law of Demeter, discussed in Topic 28, Decoupling: if a developer needs to change an object’s state, the developer must prompt the object to perform the action. This practice keeps the code isolated from other code’s implementation, increasing the chances of remaining orthogonal.
2. **Avoid global data:** Every time code references global data, the code ties itself to all other components that share that data. Even global variables intended only for reading can cause problems (for instance, if the developer later needs to modify the code to support multithreading). Generally, code is easier to understand and maintain if the developer explicitly passes any required context into the modules. In object-oriented applications, context is often passed as parameters to objects' constructors; in other code, the developer can create structures containing the context and pass around references to those structures. Developers must be cautious with the Singleton pattern, which is a way of ensuring only one instance of an object of a particular class exists; while many people use singletons as a kind of global variable (especially in languages like Java that do not otherwise support globals), singletons can also lead to unnecessary linkage.
3. **Avoid similar functions:** When a developer encounters a set of functions that look similar (for example, sharing common code at the start and end but having different central algorithms), this repetition suggests structural problems. Developers should examine the Strategy pattern in Design Patterns for a better implementation approach.
4. **Refactoring:** Developers should develop the habit of being constantly critical of the code, looking for opportunities to reorganize the code to improve its structure and orthogonality. This process is called refactoring and is important enough that a dedicated section exists on the topic (see Topic 40, Refactoring).

**Testing and Documentation**

An orthogonally designed and implemented system is easier to test. Because the interactions between the system’s components are formalized and limited, a larger portion of system testing can be performed at the individual module level. This is beneficial because module-level (or unit) testing is considerably easier to specify and perform than integration testing, and the developer should aim to perform these tests automatically as part of the regular build process (see Topic 41, Test to Code). Writing unit tests serves as a test of orthogonality: developers should assess whether the unit test requires importing a large percentage of the system's code; if so, the module is not well decoupled from the rest of the system. Bug fixing provides another opportunity to assess orthogonality; when a developer encounters a problem, the developer must assess how localized the fix is. If changes are scattered throughout the entire system, the system is likely not orthogonally designed and coded, suggesting it is time to refactor. When making a change, the developer should assess if the fix addresses all issues or if other problems mysteriously arise, which is an opportunity to bring automation to bear. The system must use a version control system (and the developer will learn about this after reading Topic 19, Version Control) to tag bug fixes when checking the code back in after testing and should run monthly reports analyzing trends in the number of source files affected by each bug fix. 

Orthogonality also applies to documentation. The axes of documentation are content and presentation. With truly orthogonal documentation, the developer should be able to change the presentation dramatically without altering the content. Using a markup system such as Markdown, developers should focus only on the content while allowing any rendering tool to manage the presentation.
