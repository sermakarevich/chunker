# software-ethics-and-design

**Parent:** [[content/L1/software-professional-practices-and-ethics|software-professional-practices-and-ethics]] — Professional software development requires an ethical commitment to user protection and a focus on solving underlying business problems rather than just delivering features, necessitating full automation, state-coverage testing (like Property-Based Testing), and advanced architectural practices like using the Actor Model or PubSub to maintain decoupling.

When developing software, professionals must recognize that the software development process is not merely a mechanical process, but an organic activity requiring constant careful thought and judgment to ensure the resulting program can maintain a long, accurate, and productive life. Programmers should actively participate in the coding process, moving beyond mere implementation to critically assess all code, including their own, because constant room for improvement exists in both the code and the design. Testing is valuable not just for finding bugs; testing gathers feedback on the code, including aspects of the design, the API, and coupling. The primary benefits of testing arise when a developer considers and writes the tests, rather than simply running them. Furthermore, programmers must be aware of any potential semantic drift while coding, acknowledging that the names chosen can define the reality of the system. The core principle of writing professional software is maintaining situational awareness, akin to a safe driver constantly reviewing the situation and preparing for the unexpected. When encountering design challenges or writer's block, a developer should attempt prototyping, which involves looking for aspects of the project to explore—for example, a new framework's data binding or a novel algorithm's edge cases—and should follow this process by writing “I’m prototyping” on a sticky note and dedicating time to the exploration. 

Embedded devices, which control life-critical systems ranging from power plants to cars to medical equipment, often use an order of magnitude more computers than those used in laptops, desktops, and data centers. Even a simple central heating control system or home appliance can kill someone if it is poorly designed or implemented. Many nonembedded systems, such as social media or banking systems, can also cause great good or great harm: for instance, social media can promote peaceful revolution or foment ugly hate, and big data can destroy privacy. Since banking systems make loan decisions that change people’s lives, and any system can potentially snoop on users, developers must recognize that the difference between a utopian future and a nightmare dystopia can be subtle. Consequently, developers hold a staggering responsibility when building software, as the software weaves the very fabric of modern life, far exceeding the scope of early hobby programs on 8-bit CPUs or isolated batch business processes. 

The ethical compass dictates that developers must apply vigilance due to this unexpected power. Any piece of code delivered affects people, and developers have a duty to ask two questions about every piece of code delivered: 1) Have I protected the user? and 2) Would I use this myself? 

The first question, “Have I done my best to protect the users of this code from harm?” mandates checking for provisions to apply ongoing security patches (e.g., on a simple baby monitor), ensuring that even if an automatic central heating thermostat fails, the customer retains manual control, and ensuring that the developer stores only necessary data while encrypting anything personal. Developers bear responsibility if they cannot truthfully say that they tried to list all the consequences and protected the users from them.

The second question relates to the Golden Rule: whether the developer would be happy to be a user of the software and whether the developer is comfortable having details or movements shared. Because some inventive ideas may skirt ethical boundaries, developers involved in such a project bear the same responsibility as the sponsors. The fundamental ethical rule remains: Do not enable scumbags. When designing the future, developers must use their imagination and hopes to build a future that all people want to inhabit, recognizing when they are doing something against this ideal and having the courage to say “no!”
