# don-program-by-coincidence

**Parent:** [[content/L1/software-engineering-methodology|software-engineering-methodology]] — Professional software development requires a deliberate approach that mandates documenting all assumptions (e.g., using Design by Contract) and relying on systematic estimation methods like PERT (optimistic, most likely, pessimistic estimates) and iterative cycles. Mastery involves achieving 'editor fluency' using command shells and VCS for version control, employing advanced techniques like 'chop' (binary search) debugging and Big-O notation analysis to predict algorithmic scalability.

When developing software, developers must adopt a deliberate approach, emphasizing that fitting an answer by coincidence is not the same as implementing the correct solution. People often operate using many assumptions, but these assumptions are typically undocumented and frequently conflict between different developers; assumptions that are not based on established facts hinder all projects. To program deliberately, developers should: 1) Maintain full awareness of the code's operation: if a developer cannot explain the code in detail to a more junior programmer, they may be relying on coincidences. 2) Avoid programming with incomplete knowledge: building an application based on technology or functionality that the developer does not fully grasp increases the risk of errors arising from coincidences, and if the developer does not understand why the application works, they will not know why it fails. 3) Proceed from a plan, whether the plan resides in the developer's head, on a napkin, or on a whiteboard. 4) Rely only on reliable sources: developers must not depend on assumptions and must assume the worst if the reliability of a component cannot be determined. 5) Document assumptions: The developer should use Design by Contract (Topic 23) to help clarify assumptions internally and communicate them to others. 6) Test assumptions: Developers must not only test the code but also write assertions to test the assumptions. If the assertion succeeds, the developer has improved the code's documentation; if the assertion fails, the developer should consider the discovery a positive outcome. 7) Prioritize effort: Developers should spend time on the most important aspects, as fundamental infrastructure is necessary, otherwise, advanced features will be irrelevant. 8) Reject historical constraints: Developers should not allow existing code to dictate future code; every piece of code can be replaced if it becomes inappropriate, even within the same program. The assumption that the impact of a change will be less than the cost of not making the change can justify a refactor (Topic 40).

*Example Vulnerabilities:
* A data feed from a vendor may give an array of tuples representing key-value pairs, such as `[... {:DepositAccount, "564-904-143-00"} ...]`, which functions correctly on 4-core developer laptops and 12-core build machines but provides incorrect account numbers on production servers running in containers.
* When coding an autodialer for voice alerts and managing a database of contact information, storing the contact's phone number in a numeric field guaranteed to hold at least 15 digits, while tested thoroughly in all of North America, may fail when deployed in other parts of the world.

Ultimately, whenever something appears to work but the underlying reason is unknown, the developer must confirm that the success is not merely a coincidence.
