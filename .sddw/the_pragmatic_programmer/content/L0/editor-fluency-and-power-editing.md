# editor-fluency-and-power-editing

**Parent:** [[content/L1/professional-proficiency-workflow|professional-proficiency-workflow]] — Professional mastery requires systematic tool development, whether it's physical tools like gauges and chisels or digital tools like command shells and text editors. Methods for complex scheduling include PERT (using optimistic, most likely, and pessimistic estimates for tasks) and iterative development (repeatedly completing small functional slices). To maximize efficiency, programmers should master the command shell by customizing settings like prompts and creating aliases, and achieve editor fluency by learning core key sequences, extending the editor's capabilities via plugins, and building custom extensions.

To manipulate text as a primary workbench, professionals should use powerful editors, which function as extensions of the hand. Because text constitutes the basic raw material of programming, the ability to manipulate text must be achieved with maximum effortlessness.

While the first edition of the book recommended using a single editor for all purposes—including code, documentation, memos, and system administration—the current advice encourages professionals to use multiple editors, emphasizing the need to work toward fluency in each tool.

**Tip 27: Achieve Editor Fluency**

Developing editor fluency is important because the major benefit is that becoming fluent allows the professional to stop thinking about the mechanics of editing. This greatly reduces the distance between thinking an idea and having it appear in the editor's buffer, allowing thoughts to flow and benefiting programming. While the professional might gain an additional week of working time over the course of a year by making editing just 4% more efficient when working 20 hours a week, this is not the primary advantage.

**Defining Fluency:**
Fluency encompasses the ability to perform the following tasks while editing text and code:
*   **Text Manipulation:** Move and make selections by character, word, line, and paragraph.
*   **Code Editing:** Move by various syntactic units (e.g., matching delimiters, functions, modules).
*   **Code Refactoring:** Reindent code following changes, and comment/uncomment blocks of code using a single command.
*   **History:** Undo and redo changes.
*   **Workflow:** Split the editor window into multiple panels, and navigate between them.
*   **Navigation & Search:** Navigate to a particular line number, search for both strings and regular expressions, and repeat previous searches.
*   **Text Management:** Sort selected lines. 
*   **Advanced Features:** Temporarily create multiple cursors based on a selection or pattern match, and edit the text at each cursor in parallel.
*   **Integration:** Display compilation errors in the current project and run the current project's tests.

**Achieving Fluency (Moving Toward Fluency):**
The most pragmatic approach is not to learn every command of a powerful editor, but rather to learn the commands that improve the individual's productivity. The process for developing fluency follows these steps:
1.  **Identify Repetition:** When the professional finds themselves performing a repetitive action while editing, the professional must adopt the habit of thinking, “there must be a better way,” and then discover that efficient method.
2.  **Build Muscle Memory:** After discovering a useful feature, the professional must achieve muscle memory through repetition. Consciously seeking opportunities to use this new feature, ideally multiple times daily, solidifies the ability to use the feature without conscious thought.

**Expanding the Editor's Capabilities:**
Many powerful code editors use a basic core augmented by extensions. When the professional encounters an apparent limitation in the current editor, the professional should search for an extension that addresses the need. If the professional is unable to find an existing solution, the professional should dig into the editor's extension language to automate repetitive tasks, often requiring only one or two lines of code. Taking this process further, the professional can write a full-blown extension and, if the need is apparent, publish it so that other people can benefit. 

**Self-Improvement Challenges:**
To build fluency, professionals should undertake specific challenges: 1) Disable autorepeat and learn key sequences to move, select, and delete by characters, words, lines, and blocks. 2) For one week, edit using only the keyboard, eliminating reliance on a mouse or trackpad. 3) Look for integrations by compiling a list of desired features to bring into the editor, which may guide the search for existing plugins or extensions. 4) If no plugin or extension exists for a needed functionality, the professional must build it.
