# end-to-end-development-approach

**Parent:** [[content/L1/software-architecture-and-refactoring-lifecycle|software-architecture-and-refactoring-lifecycle]] — Software development is an iterative, continuous process (like gardening) that prioritizes refactoring—the disciplined restructuring of code without altering external behavior—triggered by issues like duplication or drifting requirements. Advanced system design requires adopting patterns like Publish/Subscribe and Reactive Programming for decoupling, utilizing the Actor Model or Blackboard system for concurrency, and always preserving accurate records when performing multiple estimations or monitoring program state.

In software design, two traditional approaches exist: top-down and bottom-up design. The bottom-up approach involves building code layer by layer, starting at the bottom to produce abstractions that are closer to the problem being solved. Developers then add successive layers, using higher-level abstractions, continuing until the final layer forms an abstraction that solves the entire problem. Conversely, the top-down approach assumes developers can express the entire requirement up front, which is often impossible. Similarly, the bottom-up approach assumes that developers can construct a list of abstractions that will eventually lead to a single top-level solution, but developers cannot decide on the functionality of these layers without knowing the ultimate direction of the solution.

Therefore, the recommended practice is to build software incrementally by constructing small pieces of end-to-end functionality, which requires learning about the problem as the development process proceeds. Developers must apply this accumulated learning as they continue to flesh out the code, involving the customer at each step and allowing the customer to guide the process. Although practicing Test-Driven Development (TDD) is valuable, developers must periodically stop to review the big picture. Becoming focused on the green "tests passed" message can be misleading, potentially leading developers to write extensive code that fails to advance the solution.

For instance, in 2006, Ron Jeffries, a prominent figure in the agility movement, documented his test-driven coding of a Sudoku solver. After five posts, Jeffries refined the underlying board representation and refactored the object model multiple times until satisfied, but ultimately abandoned the project. This example illustrates how a developer can become distracted by the minute details, reinforced by the positive feedback of passing tests. In contrast, Peter Norvig demonstrated an alternative approach, which was not driven by tests. Norvig began with a fundamental understanding of how such problems are traditionally solved using constraint propagation, focusing immediately on refining the algorithm. Norvig addressed the board representation using only a dozen lines of code that flowed directly from the discussion of notation. While tests can certainly assist in guiding development, without a clear destination or goal, the process risks leading the developer in circles.
