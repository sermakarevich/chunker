# orthogonality-principles

**Parent:** [[content/L1/system-design-principles-orthogonality-reversibility|system-design-principles-orthogonality-reversibility]] — The synthesis emphasizes that designing systems using Orthogonality (decoupling components so changes in one do not affect others), Reversibility (avoiding irreversible decisions by abstracting services), and DRY principles requires adopting specific coding practices. Techniques include writing 'shy code,' passing context explicitly rather than using global variables, employing the Strategy pattern for similar functions, and using formal specifications like OpenAPI for external APIs, while proactively preventing knowledge duplication across the development team through structured communication and peer review.

Orthogonality is a critical concept for creating systems that are easy to design, build, test, and extend. Although orthogonality is a critical concept, it is rarely taught directly; rather, people often encounter it as an implicit feature of various other methods and techniques. However, applying the principle of orthogonality directly will immediately improve the quality of systems produced.

**What is Orthogonality?**

Initially, 'orthogonality' is a term borrowed from geometry. Two lines are orthogonal if they meet at right angles, such as the axes on a graph, meaning the two lines are independent in vector terms. For example, if the number 1 on a diagram moves north, the movement does not change the amount of distance east or west, and if the number 2 moves east, it does not affect the north or south position.

In computing, the term signifies a type of independence or decoupling. Two or more things are orthogonal if changes in one component do not affect any of the others. In a well-designed system, the database code should be orthogonal to the user interface: this means one can change the interface without affecting the database, and one can swap databases without changing the interface.

**A Nonorthogonal System Example**

To illustrate, consider a helicopter tour of the Grand Canyon where the pilot faints. If the reader recalls knowing that helicopters have four basic controls—the cyclic (the stick held in the right hand), the collective pitch lever (pulled up to increase pitch on all blades, generating lift), the throttle (at the end of the pitch lever), and two foot pedals (which vary the amount of tail rotor thrust and help with turning)—the reader initially believes the operation is simple. However, the process is complex: lowering the collective pitch lever causes the helicopter's nose to drop, and the pilot starts to spiral down to the left. The reader then discovers that the system is nonorthogonal, as every control input has secondary effects. Lowering the left-hand lever requires adding compensating backward movement to the right-hand stick and pushing the right pedal. Critically, each of these adjustments affects all of the other controls again, resulting in an unbelievably complex system where every change impacts all other inputs.

**Benefits of Orthogonality**

As the helicopter example shows, nonorthogonal systems are inherently more complex to change and control because when components of any system are highly interdependent, no local fix exists.

1. **Eliminating Effects Between Unrelated Things:** Designers should aim to create components that are self-contained, meaning they should be independent and have a single, well-defined purpose (a concept Yourdon and Constantine call cohesion in *Structured Design: Fundamentals of a Discipline of Computer Program and Systems Design* [YC79]). When components are isolated from one another, a developer knows that one can change a component without worrying about the rest. As long as the developer does not change that component’s external interfaces, the developer can be confident that problems will not ripple through the entire system.

2. **Increased Productivity:** Developers gain productivity when writing orthogonal systems because changes are localized, which reduces development and testing time. It is easier for developers to write relatively small, self-contained components than to create a single large block of code. Simple components can be designed, coded, tested, and then forgotten, eliminating the need to keep changing existing code when new code is added. Furthermore, an orthogonal approach promotes reuse; if components have specific, well-defined responsibilities, they can be combined with new components in ways that were not envisioned by the original implementers. The more loosely coupled the systems, the easier the systems are to reconfigure and reengineer. Combining orthogonal components also increases productivity: if one component does one thing and another does a second thing, combining them yields a result that does both things; conversely, if the two components are not orthogonal, there will be overlap, and the resulting system will do less.

3. **Reduced Risk:** An orthogonal approach reduces inherent development risks by isolating diseased sections of code. If a module is malfunctioning (or 'sick'), the problem is less likely to spread symptoms throughout the rest of the system, and the module can be easily removed and replaced with something new and healthy. The resulting system is less fragile, meaning that making small changes and fixes to a particular area will restrict any generated problems to that area. Additionally, an orthogonal system is more likely to be tested, because designing and running tests on the components is easier. Finally, an orthogonal approach prevents developers from becoming too tightly tied to a particular vendor, product, or platform, because the interfaces to these third-party components will be isolated to smaller parts of the overall development.
