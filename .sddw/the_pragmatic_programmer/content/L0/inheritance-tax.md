# inheritance-tax

**Parent:** [[content/L1/software-engineering-workflows-and-architecture|software-engineering-workflows-and-architecture]] — Professional proficiency requires mastering tools and workflows, utilizing techniques like PERT (requiring optimistic, most likely, and pessimistic estimates) and iterative development (which demands formal schedule refinement and detailed logging of prediction errors over 50% inaccuracy). Furthermore, complex systems should be designed using a data-flow transformational model, not isolated components, and developers must mitigate the coupling risks associated with inheritance by favoring Interfaces, Delegation, and Mixins/traits.

When designing software, developers must consider the implications of using inheritance. The foundational concept of inheritance first appeared in Simula 67 in 1969. Initially, Simula used prefix classes, such as `link CLASS car` and `link CLASS bicycle`, where the `link` class served as a parent class to add functionality for linked lists. The mental model for Simula programmers was that the instance data and implementation of the `link` class were prepended to the implementations of `car` and `bicycle`, presenting a form of polymorphism where both classes contained the `link` code. After Simula, Smalltalk also utilized inheritance. Alan Kay, one of the creators of Smalltalk, noted that Smalltalk implemented inheritance using Lisp-like dynamics to experiment with 'differential programming.'

Developers today use inheritance for two primary reasons: 1) to save keystrokes by adding common functionality from a base class into child classes (e.g., `class User` and `class Product` both subclassing `ActiveRecord::Base`), or 2) to express the relationship between classes, viewing a class as a kind of another class (e.g., a `Car` is a kind of `Vehicle`).

Both methods of inheritance present problems. First, inheritance introduces coupling. The child class couples not only to the parent but also to the parent’s parent, and the code that uses the child is coupled to all ancestors. For example, if a top-level function calls `my_ride.move_at(30)` and the `move_at` method is defined in the parent `Vehicle` class, changing the `Vehicle` class API—such as renaming `move_at` to `set_velocity` or changing the instance variable `@speed` to `@velocity`—will break the parent class’s contract, even if the top-level code only interacts with the `Car` class.

Second, using inheritance to define types is problematic. While class hierarchies seem appealing, they can quickly grow into complex, fragile structures. This complexity is worsened by the multiple inheritance issue; modeling a `Car` as a `Vehicle` and also as an `Asset`, `InsuredItem`, and `LoanCollateral` would require multiple inheritance, a feature that C++ struggled with in the 1990s, leading many modern object-oriented (OO) languages to omit it.

Instead of relying on inheritance, developers should utilize three alternative techniques: Interfaces and protocols, Delegation, and Mixins and traits.
