# software-testing-and-automation

**Parent:** [[content/L1/software-professional-practices-and-ethics|software-professional-practices-and-ethics]] — Professional software development requires an ethical commitment to user protection and a focus on solving underlying business problems rather than just delivering features, necessitating full automation, state-coverage testing (like Property-Based Testing), and advanced architectural practices like using the Actor Model or PubSub to maintain decoupling.

The software development lifecycle includes several critical testing stages. **Unit testing** involves writing code that exercises individual modules. Since modules must function independently, all modules used in the project must pass their own unit tests before proceeding; otherwise, the interconnected parts are unlikely to work together effectively.

After all individual modules pass their unit tests, the next stage is **integration testing**. This process verifies that major system subsystems interact and function correctly together. When solid contracts are established and units are well tested, detecting integration issues becomes straightforward; otherwise, integration testing often becomes the single largest source of bugs. Integration testing is essentially an extension of unit testing, where the developer verifies that entire subsystems honor their contracts.

Beyond technical testing, validating the system against user needs is crucial. When an executable user interface or prototype becomes available, developers must determine if the product meets the functional requirements and addresses what the users actually need, rather than just what they stated they wanted. This validation process must account for end-user access patterns and how these patterns differ from developer test data.

Furthermore, **performance testing** (or stress testing) may be necessary to confirm that the software meets performance requirements under real-world conditions. Developers should ask whether the software is scalable for the expected number of users, connections, or transactions per second, and may need specialized testing hardware or software to simulate load realistically.

Finally, thorough testing requires testing the tests themselves. Because perfect software is impossible, perfect test software is also impossible. Developers should treat the set of test suites as an elaborate security system, which must be tested by actively trying to break it. After writing a test to detect a specific bug, the developer must deliberately cause the bug and verify that the test correctly complains, ensuring the test will reliably catch the bug in a real scenario.

To ensure the highest level of quality, the best practice is to **find bugs only once**. Once a human tester discovers a bug, automated tests must be modified immediately to check for that particular bug every single time, regardless of how trivial the bug seems. This prevents developers from spending time chasing bugs that automated tests could already find.

When aiming for comprehensive validation, developers should not rely solely on code coverage analysis tools, which track executed lines of code. While these tools provide a general sense of completeness, knowing that every line of code has been executed does not mean the program has been tested thoroughly. A function, for instance, that takes two integers (a and b) and calculates `a / (a + b)` has 1,000,000 logical states (for integers 0 to 999), 999,999 of which work correctly, but one state (where a + b equals zero) will cause failure. Developers must prioritize testing state coverage over code coverage. A valuable way to explore how code handles unexpected states is to use **Property-Based Testing**, which utilizes a computer to generate test data based on the defined contracts and invariants of the code under test.

Complementing all testing efforts is the necessity of **full automation**. Modern development mandates that all processes must be scripted and automatic, whether utilizing simple shell scripts with rsync and ssh or full-featured solutions like Ansible, Puppet, Chef, or Salt. Relying on any manual intervention breaks the process. For example, attempts to install add-on packages via manual instructions result in subtle, variable differences in machine configurations, causing bugs that appear only on specific developer machines. A shell script or program consistently executes the same instructions, in the same order, time after time, and all build and release procedures must be automated. If any manual steps are introduced, the entire system becomes brittle, thus requiring full automation in the build and deployment processes. Successfully combining version control, rigorous testing, and full automation provides the necessary stable foundation for development, allowing developers to concentrate on user delight.
