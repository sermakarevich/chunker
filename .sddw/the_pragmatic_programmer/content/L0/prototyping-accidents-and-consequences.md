# prototyping-accidents-and-consequences

**Parent:** [[content/L1/software-engineering-methodology|software-engineering-methodology]] — Professional software development requires a deliberate approach that mandates documenting all assumptions (e.g., using Design by Contract) and relying on systematic estimation methods like PERT (optimistic, most likely, pessimistic estimates) and iterative cycles. Mastery involves achieving 'editor fluency' using command shells and VCS for version control, employing advanced techniques like 'chop' (binary search) debugging and Big-O notation analysis to predict algorithmic scalability.

When designing software, developers must remember that prototypes are intended to fail, and that prototypes are ultimately discarded, regardless of whether they fail. Because of this reality, there is no downside to following the initial prototyping steps: first, creating a comment in the empty editor buffer that describes in one sentence what the developer wants to learn or do, and then starting to code. If the developer encounters doubts during the process, the developer should look at the sticky note for guidance. If a nagging doubt crystallizes into a solid concern in the middle of the coding process, the developer must address it immediately. If, upon reaching the end of the experiment, the developer still feels uneasy, the developer should restart the process with the 'walk and the talk and the time off' approach. However, the experience shows that at some point during the first prototype, the developer will feel a sense of urgency and enthusiasm for the task. At this stage, the developer knows the appropriate next steps: the developer must delete all the prototype code, throw away the sticky note, and fill the empty editor buffer with new, polished code.

When working with existing code, particularly code written by other people, developers must deal with varying instincts and decisions. Instead of mechanically reading the code and making notes on seemingly important details, the developer should conduct an experiment. When the developer spots something that seems strange, the developer should jot it down and continue this process to look for underlying patterns. By analyzing the patterns, the developer may find that the job of understanding the code becomes significantly easier, allowing the developer to consciously apply the patterns that the original author applied tacitly. Furthermore, this process might allow the developer to learn something entirely new.

Beyond just the code, listening to the developer's intuition (gut) when coding is an important skill to foster, but this skill applies to the overall system design as well. If a design feels wrong, or if a requirement creates unease, the developer should stop and analyze those feelings. If the developer is in a supportive environment, the developer should express these feelings out loud and explore them, as the chances are that a critical issue is lurking nearby. The developer must listen to intuition to avoid problems before they manifest.

Regarding the development process, developers must acknowledge the challenge of working in 'minefields'—environments filled with potential traps. Remembering the story of a weary soldier probing a seemingly safe field for mines demonstrates the danger of drawing false conclusions based on lucky or circumstantial evidence. Developers must avoid 'programming by coincidence,' which relies on luck and accidental successes, and instead adopt a deliberate approach.

Accidents of implementation occur when developers rely on undocumented error or boundary conditions resulting from how the code is written. For example, if a developer calls a routine with bad data, the routine responds in a particular way, and the developer codes based on that response, even if the original author did not intend for the routine to function that way. If the routine is later 'fixed,' the developer's code may break. Calling components in the wrong sequence or in the wrong context is also a related problem.

When writing code that other people will call, developers must rely on the basic principles of good modularization and hiding the implementation details behind small, well-documented interfaces. For routines that developers call, the developer must rely only on documented behavior; if documented behavior is unavailable, the developer must explicitly document the assumption being made. Furthermore, the developer must recognize that developers are prone to seeing patterns and causes even when they are merely coincidences, such as in finance or gaming. An intermittent error visible every 1,000 requests in a log file may be a race condition or a simple bug. Any perceived pattern must be proven through testing, not assumed.

If a project involves temporal inconsistencies, such as logging data from hardware collection units across various time zones, developers must use a proper model of time handling instead of compensating for flaws with repeated additions or subtractions (e.g., $+1$ or $-1$).

Ultimately, developers must always proactively validate all provided information and avoid building 'cargo cult' code, which merely imitates a form without addressing the underlying technical requirements.
