# concurrency-and-parallelism-architectures

**Parent:** [[content/L1/software-design-patterns-and-development-workflows|software-design-patterns-and-development-workflows]] — Software design alternatives to inheritance include Interfaces/Protocols (Traits) and Delegation, which break coupling and enforce necessary behaviors without requiring shared code. Professional workflows mandate structured methods like PERT (optimistic, most likely, pessimistic estimates) and Iterative Estimation, requiring detailed logging and root cause analysis for any estimate error exceeding 50%. Tooling proficiency requires mastering the command shell (WYSIAYG), achieving 'editor fluency' via keyboard mastery and extensions, and utilizing Version Control System (VCS) branching for isolated, audited development.

The discussion of concurrent and parallel programming introduces several critical architectural considerations. While developers frequently discuss general coupling (referring to dependencies that impede change), there is another type of coupling: temporal coupling. Temporal coupling occurs when code imposes a sequence on events that is not fundamentally required to solve the problem. For example, determining if code must access multiple back-end services sequentially, one after the other, leads to temporal coupling, which must be avoided to maintain flexibility. The difficulty in writing concurrent and parallel code stems partly from the fact that developers learned to program using sequential systems. Consequently, existing language features, which are safe when used sequentially, become liabilities when two or more things can happen simultaneously. A primary source of this issue is shared state. Shared state is not limited to global variables; rather, any situation where two or more chunks of code hold references to the same piece of mutable data constitutes shared state. While specialized topics exist to describe workarounds for shared state, these solutions are often error-prone. Better methods for building concurrent applications include using the actor model. In the actor model, independent processes communicate over channels using defined, simple semantics while sharing no data. This approach covers both the theory and practice of using actors and processes. Additionally, blackboards are systems that function as a combination of an object store and a smart publish/subscribe broker. Although blackboards initially struggled to gain traction, modern middleware layers increasingly utilize blackboard-like semantics, offering a significant degree of decoupling when correctly used. The use of concurrent and parallel code has transitioned from being exotic to being a requirement in modern software development.
