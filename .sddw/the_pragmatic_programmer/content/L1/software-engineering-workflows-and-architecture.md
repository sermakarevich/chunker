# software-engineering-workflows-and-architecture

**Parent:** [[content/L2/systematic-professional-mastery-workflow|systematic-professional-mastery-workflow]] — Professional proficiency demands mastery across physical tool use (gauges, saws, chisels), advanced workflows (PERT, Iterative Estimation, requiring logging and systematic refinement), and complex digital architecture. Key digital skills include mastering the command shell (WYSIAYG vs. WYSIWYG), achieving editor fluency through sustained repetition, and leveraging advanced systems like VCS (branching, CI/CD) and reactive programming (PubSub, Event Streams, Actor Model) to manage coupling and asynchronous complexity. Debugging requires systematic techniques such as the 'chop' binary search and using diagnostic 'tracing statements'.

The field of advanced software development demands a continuous process of systematic learning, where proficiency is built upon mastering specialized tools, workflows, and conceptual models. Professional capability requires not just possessing advanced tools—such as laser-guided miter saws or dovetail jigs—but developing meticulous care for foundational tools like rules, gauges, saws, planes, and chisels, recognizing that these basic items remain useful even when advanced ones are acquired. This relationship progresses until the tools become a profound extension of the mind, translating abstract thoughts into tangible results. Critically, this commitment extends beyond physical equipment to encompass operational workflows and programmatic thought processes. When approaching complex tasks, estimation requires structured methodologies, as inherent uncertainty makes single definitive number predictions unreliable. Two primary techniques exist: Scenario-Based Estimation and Iterative Estimation.

Scenario-Based Estimation, drawing inspiration from the U.S. Navy's planning for the Polaris submarine project and formalized as Program Evaluation Review Technique (PERT), dictates that professionals must provide a spectrum of possibilities rather than a single figure. Every task measured via PERT requires three specific estimates: an optimistic figure, a most likely figure, and a pessimistic figure. For example, estimating house painting might yield a low estimate of 10 hours (assuming ideal conditions and optimal paint coverage), a middle estimate of 18 hours, and a pessimistic estimate of 30 hours or more (necessitated by adverse weather). By arranging tasks into a dependency network and using fundamental statistics, professionals calculate the statistically likely best and worst possible project timelines. This systematic use of ranges significantly improves the overall project estimate by mitigating errors caused by generalized padding. When conducting estimation, developers must allocate adequate time to slow down, carefully working through descriptive steps, as estimates delivered under duress (such as at a coffee machine) are often inaccurate. Furthermore, a detailed logging system is mandatory, requiring the rigorous tracking of every prediction's accuracy; if the observed error exceeds 50%, a thorough investigation into the specific root causes of the inaccuracy—such as selecting parameters that do not accurately reflect reality or relying on an incorrect underlying model—is crucial for significantly improving subsequent estimations. When an overall estimate involves multiple subestimates, a separate and meticulous record must be maintained to track the degree of accuracy in those predictions.

Iterative Estimation, or 'Eating the Elephant,' recognizes that the most dependable method for setting a project timeline is gaining direct, practical experience *on* the project itself. This practice is implemented through incremental development, which demands the repeated completion of a detailed cycle using small, isolated slices of functionality. Each cycle must systematically include checking requirements, performing risk analysis (with mandatory priority given to the riskiest items early), designing, implementing, integrating, and validating the results with the end-users. While initial guesses for the total number of iterations or the duration of each are possible, they are unreliable for most projects. Consequently, the professional must first complete the coding and testing for the initial functionality, formally concluding the first iteration. The resulting practical experience then enables the informed refinement of the initial estimate regarding both the total number of required iterations and the specific scope of work for every subsequent cycle. This professional must formally refine the schedule as an integral part of every single iteration. For communicating the timeline to stakeholders, the most accurate advice is to state, “I’ll get back to you.”

Beyond workflow, modern software architecture requires developers to adopt a *transformational perspective*. Programs should be viewed as transforming data—converting an input into an output—rather than focusing only on isolated elements like classes, modules, data structures, algorithms, languages, or frameworks. Adopting this transformation model is vital for reducing coupling, as it allows data to become a peer to functionality, forming a pipeline structure: code $	o$ data $	o$ code $	o$ data. In this pipeline, the data itself is not tied to a specific class definition. When building such systems, data management should prioritize passing data as a continuous flow, minimizing storage in isolated pools. This allows for a significant reduction in coupling, making a function reusable anywhere its parameters match the output of another function, and results in cleaner, shorter, and flatter code.

This transformational model is further reinforced by advanced concepts in reactive programming and event handling. Reactive Programming treats values as reacting when the cells they reference change, utilized in web frameworks like React and Vue.js. Event streams provide a robust way to treat events as a collection of data, enabling developers to manipulate, combine, filter, and apply standard collection operations to event data. These streams can be asynchronous, allowing the code to respond to events as they arrive. The current standard for reactive event handling is defined on http://reactivex.io. For instance, combining two streams can be achieved with `zip`, which only emits a value when data is available on both inputs; or, using `mergeMap` in the RxJS library, the system can process multiple asynchronous requests—such as fetching user IDs 3, 2, and 1 from `https://reqres.in`—in parallel. This system processing is highly effective, exemplified by an 82ms request for id 2, followed by 50ms and 51ms requests for id 3 and id 1, respectively. Event streams are valuable because they unify synchronous and asynchronous processing through a common API and are pervasive, governing events from button clicks to file line matches, making code more responsive and better decoupled than linear alternatives. Furthermore, the principle of designing systems should mimic this flow by ensuring that when passing data between transformations, raw values are never used; instead, the values must be wrapped in a data structure that explicitly signals validity. In languages with type checking, this can be achieved using conventions like Haskell's `Maybe`, F#'s `Option`, or Elixir's tuple format, such as `{:ok, value}` or `{:error, reason}`. If an error occurs, the pattern dictates propagating that error down the pipeline, for instance, in an Elixir implementation where a function handles either the `{:ok, content}` tuple to proceed with finding matching lines or simply executes a second version that returns the error tuple.

Shifting focus to programmatic structure, developers must be aware of the pitfalls of using inheritance. While foundational concepts of inheritance appeared in Simula 67 (using prefix classes like `link CLASS car`), and later in Smalltalk (which used Lisp-like dynamics), developers today use it primarily to save keystrokes or express type relationships (e.g., a `Car` is a kind of `Vehicle`). However, inheritance introduces two significant problems. First, it introduces coupling; a child class couples not only to its immediate parent but also to the parent's ancestors, and the code utilizing the child is coupled to all ancestors. Changing a method's API (e.g., renaming `move_at` to `set_velocity`) in a high-level parent class (`Vehicle`) will break the contract for the child (`Car`), even if the calling code only interacts with `Car`. Second, defining types via class hierarchies is fragile; the complexity is worsened by the multiple inheritance problem, which C++ struggled with in the 1990s, leading modern OO languages to often omit the feature.

To mitigate these issues, developers should prioritize three alternative techniques: Interfaces and protocols, Delegation, and Mixins and traits. This systematic, forward-thinking approach is paralleled by advanced development practices regarding tooling and system integrity. The command shell must be mastered, moving beyond GUI/IDE reliance, because while GUIs adhere to WYSIWYG (what you see is what you get), the shell provides WYSIAYG (what you see is all you get). Shell mastery involves using utilities combined with pipes, setting color themes, configuring a prompt (which can display information like version control status or time), and utilizing aliases and functions. For text manipulation, 'editor fluency' is paramount, requiring the capacity to operate multiple powerful editors effortlessly, minimizing the mental distance between an idea and its physical manifestation. Fluency involves moving and selecting content precisely by character, word, line, or paragraph; navigating by syntactic units (e.g., matching delimiters); and managing complex features like advanced searches using regular expressions or temporarily creating multiple cursors for parallel edits. Achieving fluency is achieved by identifying repetitive actions and building muscle memory through sustained repetition, and when facing a limitation, searching for an extension or writing the extension's language code. Beyond the editor, Version Control Systems (VCS) are indispensable; they operate as a project-wide time machine, vastly improving upon simple 'undo.' VCS is crucial for collaboration because simple file sharing or cloud storage risks data loss and conflicting builds due to multiple simultaneous changes. VCS systems not only track every change for basic reversal but provide invaluable data for auditing, bug tracking, and performance analysis, allowing users to track metrics such as who altered a specific line or the total quantity of lines changed during a release. One of the most powerful features is **branching**, allowing work on Feature A to remain fully isolated from Feature B. For optimal integration, a central repository is recommended, ideally providing excellent security and access control, an intuitive UI, CLI capability for comprehensive automation, automated builds and tests, robust branch merging support (like pull requests), and integrated issue management tools. Advanced teams often configure their VCS such that a simple push to a designated branch automatically triggers the system build, runs associated tests, and deploys the new code, ensuring reliability. Furthermore, mastering debugging requires the 'chop' technique, which is a binary search method for isolating an error within a massive system. By selecting a stack frame near the middle of a massive stack trace, the developer determines if the error is present, narrowing the focus to the preceding or subsequent frames in a maximum of six attempts. To observe state over time, developers utilize **tracing statements**, simple diagnostic messages, supplementing the standard debugger's instantaneous state view.
