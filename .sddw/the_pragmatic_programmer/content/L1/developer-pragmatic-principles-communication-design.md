# developer-pragmatic-principles-communication-design

**Parent:** [[content/L2/pragmatic-programming-design-principles|pragmatic-programming-design-principles]] — Sustainable software development requires systemic control against 'software rot' by prioritizing the fixing or containment of minor flaws, while individual developers must manage a 'knowledge portfolio' through continuous investment in skills, diverse learning (including nontechnical subjects and new programming languages annually), and adopting advanced communication techniques that tailor a message to the audience. Core architectural principles—Orthogonality, Reversibility, and the DRY principle—mandate that code must be built to be Easy to Change (ETC), ensuring that knowledge, whether in APIs or business rules (like calculating complex account fees), is defined in a single, authoritative location to prevent systemic failure and maintain system flexibility.

Developing sophisticated software requires developers to move beyond merely predicting immediate outcomes (first-order thinking) and embrace second-order thinking, which involves considering the consequences of those consequences. A core skill in development is the ability to investigate existing problems by identifying underlying models and understanding how those models operate, as obtaining simple answers has become increasingly challenging. By merging an extensive personal knowledge portfolio with critical analysis applied to numerous technical articles, developers can successfully decipher highly complex technical answers.

Effective communication is foundational to development success. Possessing superior ideas, exquisite code, or pragmatic thinking remains functionally inert unless it can be successfully shared with others. A promising idea, for example, is described as an orphan without effective communication. This communication effort is multifaceted: developers spend considerable time in meetings, both listening and speaking; they work directly with end users to fully grasp their needs; they write code, which serves to transmit intentions to a machine and meticulously documents the developers' thought process for future colleagues; they compose proposals and memos used to request and justify resources, report current status, or suggest new technical directions; and finally, they engage in daily advocacy, modifying established practices and proposing innovations within their teams. Given that a significant portion of daily work involves communication, excelling at this task is mandatory. Developers must treat natural languages, including English, as if they were another programming language, adhering to principles such as honoring the DRY (Don't Repeat Yourself) principle, utilizing automation, and implementing similar thoughtful design principles.

Effective communication hinges on two crucial elements: first, **KNOWING YOUR AUDIENCE**, and second, **KNOWING WHAT TO SAY**. To achieve successful communication, one must deeply understand the recipient’s needs, interests, and existing capabilities. For instance, when presenting an update—such as changing a remote monitoring system to use a third-party message broker for status notifications—the communication 'pitch' must be tailored specifically to the audience. For end users, the highlight is the ability for their systems to now interoperate with other services. For marketing departments, this can be leveraged to boost sales figures. For development and operations managers, the benefit is that the care and maintenance of that system component will become the responsibility of another party. By crafting the appropriate pitch for each group, the presenter can manage to generate excitement across all stakeholder groups. Furthermore, the critical tactic in all forms of communication is actively gathering feedback; one must not passively wait for questions but must actively solicit them while also observing non-verbal cues like body language and facial expressions. The documentation process is a key area of communication, yet developers frequently treat it inadequately, often viewing it only as an unfortunate necessity or a low-priority task that management might neglect at the end of a project. Pragmatic Programmers, therefore, must integrate documentation as a core and integral component of the entire development lifecycle. When writing documentation, developers can minimize wasted time and duplication by keeping documentation proximate—directly within the source code itself. Consequently, all pragmatic principles used for writing code must also be applied to documenting it.

It is beneficial to embed documentation directly into the code rather than adding it afterward. While generating visually appealing documentation from source code comments is relatively simple, the text recommends adding comments to modules and exported functions specifically to aid other developers who use the code. However, excessive, mechanical commenting is detrimental, as developers should not assume that every function, data structure, or type declaration requires its own comment. Such over-commenting increases maintenance complexity because a single change then necessitates updating two separate locations. Therefore, developers must restrict non-API commentary to explaining *why* something was done, its underlying purpose, and its initial goal. Commenting the *how* aspect of the source code is redundant because the code itself already demonstrates this, thereby violating the DRY principle. Commenting source code instead provides an optimal opportunity to document elusive project elements that are otherwise difficult to capture, such as fundamental engineering trade-offs, the core reasoning behind design decisions, or alternative designs that were subsequently rejected.

Effective software design requires focusing on the **ETC principle: Easier to Change**. The authors caution against reliance on generalized principles or industry gurus, asserting that a well-designed system must ultimately adapt to its users. For computer code, this adaptability must be achieved by ensuring the code is easy to change. The ETC concept is powerful enough to encompass nearly every other design principle; for instance, decoupling is beneficial because isolating concerns makes each concern individually easier to modify, and the Single Responsibility Principle is useful because a requirement change only necessitates modifying a single, isolated module. Similarly, effective naming conventions make the code easier to read, which intrinsically makes the code easier to change. Importantly, ETC should be treated as a guiding *value*, not as an absolute *rule*. As a value, it serves to guide decision-making when selecting among various development pathways. Developers are encouraged to actively reinforce this value—whether writing code, conducting tests, or fixing bugs—by deliberately asking whether the action taken makes the overall system more or less changeable. Although common sense often points to the simplest path for future modifications, common sense can be insufficient. In these complex instances, two actions are recommended: First, developers must make the code *replaceable*, guaranteeing that the chunk of code will not become a future constraint; this means keeping the code both decoupled and cohesive. Second, developers should document this decision-making process—the situation, the available choices, and potential guesses about change—in an engineering daybook and place a tag within the source code. This record allows the developer to review their own reasoning later when the code inevitably needs modification.

Another critical principle guiding all design elements—including DRY, Orthogonality, Reversibility, and Decoupling—is ETC. Beyond structural design, the concept of duplication itself requires close attention. Although programmers use specifications, running code, and testing to gather, organize, maintain, and harness knowledge, this knowledge is inherently unstable and changes rapidly (due to events like client meetings, evolving government regulations, or failed test cases). This instability forces developers into a constant state of maintenance, requiring them to reorganize and re-express knowledge within the system. Many people mistakenly believe that maintenance only begins after an application deployment, defining it only as bug fixing and feature enhancement. In reality, developers are constantly in a state of maintenance because their own understanding evolves daily, and new or emerging requirements appear regardless of changes in the external environment. During maintenance, developers must modify the *representations* of things—the knowledge embedded in the application. The primary pitfall is the tendency to duplicate knowledge across the specifications, processes, and programs developed. This duplication creates a massive maintenance headache that begins even before the application launches. To reliably develop software and ensure ease of understanding and maintenance, developers must commit to the **DRY principle**: every piece of knowledge must possess a single, authoritative, unambiguous representation within the entire system.

The alternative to DRY is expressing the identical piece of knowledge in two or more distinct locations. If a developer modifies one instance, they must remember to modify all the others; otherwise, the program will inevitably fail due to contradiction. The danger here is not whether the developer will remember, but *when* they will forget. DRY is recognized as one of the most crucial tools in the pragmatic programmer’s kit. Importantly, DRY is not restricted solely to code; it pertains to the duplication of *knowledge* or *intent*. It means expressing the same idea in two different places, potentially using two entirely different methods.

To test if a system adheres to DRY, a developer should ask: if a single aspect of the code requires modification, are they forced to make that change across multiple locations and in multiple different formats? If the change must be applied to the code, the documentation, the database schema, or a data structure, the system violates the DRY principle. The pseudocode example demonstrates code duplication, such as the handling of negative numbers in printing a balance. This duplication can be solved by creating a reusable formatting function, `format_amount(value)`, which centralizes the logic. Similarly, merely having identical function bodies does not constitute a DRY violation if the functions validate separate concepts. For example, in a wine ordering application, `validate_age(value)` and `validate_quantity(value)` might both validate type and minimum integers, but since they validate two separate concepts, this shared logic is a coincidence, not knowledge duplication. However, if business rules, such as the calculation of fees (e.g., $20 per returned check, $10 per overdraft day over three, and a 50% reduction if the average balance exceeds $2,000), are captured both in the documentation comment and the code, changing a fee requires updating both locations, leading to potential contradiction. The recommended refactoring involves extracting this calculation into a function, `calculate_account_fees(account)`, keeping the details confined within the source code itself. This ensures the function name reflects the purpose and the specifics are maintained in one authoritative location, adhering to DRY.

Data structures also require DRY adherence. If a `Line` class stores a `start` point, an `end` point, and a `length`, the length calculation is derived from the start and end points. To avoid storing redundant information, the length attribute should be treated as a calculated field rather than a stored one: `class Line { Point start; Point end; double length() { return start.distanceTo(end); } };`. While performance might sometimes necessitate violating DRY (like caching), the impact of this violation must be localized, meaning only the methods within the class should manage internal consistency, preventing the inconsistency from escaping the module. When a module exposes a data structure, developers must implement accessor functions (e.g., `void setStart(Point p)` and `double getLength()`). This approach upholds Meyer’s Uniform Access principle, ensuring all services are accessed uniformly regardless of whether they are computed or stored.

When code interfaces with external systems (APIs, remote services, external data), DRY violations are common because the code must replicate external knowledge (like the API schema or error codes). This duplication is manageable: for internal APIs, developers should use neutral specification formats to generate documentation, mock APIs, tests, and client code, ideally keeping all APIs in a central repository. For external APIs, using formal specifications like OpenAPI is ideal, allowing integration into local tools. If no specification exists, the developer should create and publish one. For data sources, developers can use schema introspection to generate container code, or alternatively, they can avoid fixed structures (like a struct) and store data in a key/value map (like a hash or dictionary), requiring a supplemental table-driven validation suite to ensure minimum required data is present and correctly formatted. If the most challenging duplication occurs between different developers, whole functionalities might be duplicated across the team, remaining undetected for years. To mitigate this, a strong, collaborative team with excellent communication is necessary. Strategies include running daily scrum standups or utilizing forums like Slack channels to discuss common challenges, establishing a central source tree for utility routines, and designating a team member as the project librarian to aid knowledge exchange. Finally, committing to reviewing colleagues' code and documentation, whether informally or during formal reviews, is a reciprocal and mutually beneficial practice.

Online communication standards, especially for email, are crucial since email serves as a cornerstone for settling disputes, discussing contracts, and acting as evidence in court. Developers must maintain professional rigor: always proofread before hitting SEND, check meticulously for spelling errors and accidental auto-correct mishaps, and maintain a simple, clear format. Quotations should be kept to a minimum to avoid excessive length.

Finally, when considering future planning, developers must not limit thinking to first-order results; instead, they must use second-order thinking to assess subsequent consequences. This deep analysis capacity, combined with methodical communication and adherence to principles like DRY and ETC, forms the blueprint for resilient, maintainable, and highly influential software systems.
