# architectural-principles-and-risk-management

**Parent:** [[index]]

The synthesis of robust, scalable, and flexible software systems requires the continuous application of foundational architectural principles, including Orthogonality, Reversibility, and Dry (Don't Repeat Yourself). These principles guide developers toward minimizing interdependencies and localizing impact, thereby reducing development risk and increasing long-term productivity. System design must be approached not as a series of fixed decisions, but as a dynamic process of managing inherent technical risk in a fluid development environment.

### Structural Design Principles and Decoupling

Orthogonality, which signifies independence in component design (like two lines meeting at right angles in geometry), dictates that changes in one component must not affect others. For instance, the user interface must be orthogonal to the database; changing one should not impact the other. This structural goal requires structuring systems into cooperating, isolated modules, each with a single, well-defined purpose. When systems become highly interdependent, such as controlling a helicopter, attempting a local fix is nearly impossible because lowering the collective pitch lever requires compensating adjustments to the right stick and right pedal, which consequently affects all other controls. The core objective is to ensure developers can modify one component without worrying about the rest, provided the component’s external interfaces remain consistent. 

Practically, this involves developing layered approaches where each layer only uses abstractions provided by the layers beneath it. A rigorous test of orthogonality is to map components and confirm that changing a single function's requirements affects only one module. This principle must extend beyond internal modules, requiring developers to carefully manage external dependencies. It is crucial to avoid using external identifiers prone to change, such as telephone numbers, postal codes, Social Security Numbers, government IDs, email addresses, or domains, as these create unnecessary linkage. Similarly, developers must approach third-party toolkits cautiously, ensuring they do not impose unwanted structural constraints.

To maintain orthogonality, specific coding techniques are required. Developers must implement 'shy code'—modules that avoid revealing unnecessary details or relying on other modules' internal implementations. The Law of Demeter advises that if a developer needs to change an object’s state, they must prompt the object itself to perform the action, thereby keeping the code isolated. A significant anti-pattern is the use of global variables, including even read-only ones, as these tie the code to all sharing components. Code is generally more manageable when required context is explicitly passed in, usually as parameters to constructors or contained within structures passed by reference. The Singleton pattern must be used with caution, as it mimics the problematic nature of global variables. 

When structural duplication is detected, particularly in similar functions that share common start and end code but differ only in the central algorithm, the Strategy pattern is the superior solution. Furthermore, developers must maintain a critical habit of refactoring, treating every bug fix as an opportunity to assess localization; if changes are scattered, the design lacks orthogonality and needs refactoring. A unit test serves as a metric for orthogonality: if it requires importing a large percentage of the system's code, the module is insufficiently decoupled.

### Mitigating Irreversible Decisions and Abstraction

Complementing orthogonality is Reversibility, which mandates resisting critical, irreversible design decisions. Because the external environment is continuously changing, no design choice—be it a specific vendor's database, architectural pattern, or deployment model—can be assumed permanent. Committing early creates a costly or impossible path to reverse. To sustain flexibility, developers must abstract concepts. Instead of assuming a relational database from Vendor A, the focus must be on abstracting the *concept* of persistence as a service, allowing the underlying technology to be swapped midstream. For example, a browser-based application should preemptively plan for a future mobile shift by intentionally stripping out HTML rendering and replacing it entirely with a dedicated Application Programming Interface (API). The underlying guiding principle is that the difficulty of making a choice should not lead to the mistaken belief in its permanence, treating all decisions as if they are written in the sand.

This necessity for abstraction is particularly challenging when integrating external services. When interacting with APIs, developers often face DRY violations because they must replicate external knowledge, such as schemas or error codes. While formal specifications like OpenAPI are useful for external APIs, if none exist, the developer must create and publish one. For internal APIs, the use of neutral format tools is recommended, allowing for the generation of documentation, mock APIs, functional tests, and clients, ideally centralized in a shared repository. For data sources, rather than relying solely on introspection to generate container code from a schema, an alternative method is preferred: storing external data in a key/value structure (like a map or hash). This mandates a crucial second layer of validation—a simple table-driven suite—to ensure the key/value map holds the minimum required data in the correct format.

### Guiding Development: Prototypes and Tracers

To manage complex requirements and shifting scope, two specialized, complementary techniques are utilized: Tracers and Prototypes. Both address risk but serve fundamentally different structural purposes.

**The Tracer Bullet Approach (Structural Integrity):** 
The tracer code is used when the goal is to establish the entire structural skeleton of a complex system where the target or environment is vaguely defined or likely to shift. The tracer code is defined as lean, complete, and a permanent structural part of the final system. When demonstrating system interactions or establishing an architectural backbone, a developer might construct a tracer using a trivial implementation of a complex algorithm (e.g., a simple first-come, first-served container packing routine) coupled with a simple, functional user interface. This ensures that as new functionality is added, the overall framework remains intact, guaranteeing consistent system behavior and structural fidelity. 

**The Prototyping Approach (Exploratory Risk Analysis):** 

Prototypes, conversely, are intended for exploratory analysis and risk reduction at the lowest possible cost. They generate disposable code and are primarily used for intelligence gathering—the reconnaissance phase. Prototypes are designed to answer only a few specific questions, making them inherently faster and cheaper than building a full production application. They can be built using non-coding methods, such as a whiteboard drawing, a paint program mock-up, or Post-it notes for mapping dynamic workflows. In code, high-level scripting languages like Python or Ruby are often preferred for rapid development. The fundamental goal of a prototype must be **learning**, not the production of ready-to-ship code. Consequently, developers can strategically ignore details regarding correctness (using dummy data), completeness (functioning only in a limited sense, constrained to one preselected input/menu item), robustness (allowing for incomplete error checking, potentially causing crashes), and style (excessive documentation or comments are unnecessary).

**Distinction and Usage:**

Prototypes should be used for anything that carries risk—including architecture definition, unproven functionality, external data structures, performance bottlenecks, or UI design. In contrast, a tracer is superior when the development environment requires high fidelity to the structure; it provides a persistent, structurally sound skeleton that proves the entire system’s integration model works.

### Project Scheduling and Estimating Metrics

Accurate estimation—of time, storage, or performance—is a core professional skill that requires structured methodology. When beginning any estimate, the professional must first understand the specific context and whether high accuracy or a rough ballpark figure is required. The choice of unit is paramount; stating '130 working days' implies a far greater degree of accuracy than stating 'about six months.'

To maintain temporal consistency, strict guidelines dictate how time must be quoted: 1–15 days must be quoted in Days; 3–6 weeks must be quoted in Weeks; 8–20 weeks must be quoted in Months; and 20+ weeks must be quoted in Months. For instance, 125 working days (25 weeks) must be stated as 'about six months.'

All estimations are fundamentally built upon models of the problem. The initial phase involves consulting individuals who have successfully solved similar problems, adapting their experience even if an exact match is impossible. The process mandates creating a rudimentary, bare-bones mental model of the system (e.g., conceptualizing a server and traffic arrival for response time estimates). This modeling process is beneficial because it often uncovers underlying patterns. After creating the core model, the professional must decompose it into smaller components and identify the mathematical rules governing their interaction. For every component, necessary parameters must be identified and assigned a specific value, focusing heavily on parameters that are multiplicative or divisive, as these are generally more significant than additive ones. Since estimates often build upon each other, this reliance point is the largest potential source of error.
