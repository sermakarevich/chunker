# property-based-testing-and-design

**Parent:** [[content/L1/advanced-software-design-and-security-principles|advanced-software-design-and-security-principles]] — Software development requires adherence to foundational principles including DRY, orthogonality, and least privilege, demanding developers use interfaces to control global data and encrypt sensitive information like passwords and PII. Testing relies on Property-Based Testing (which identifies failing parameters used to create dedicated regression unit tests) and Design by Contract (DBC), alongside secure password policies per NIST guidelines (e.g., minimum 64 characters, accepting Unicode). Naming elements must clearly reflect intent, and system security requires rapid patching and using vetted, third-party cryptographic services.

When a property-based test fails, the developer should identify the specific parameters used by the test function and use those values to create a separate, regular unit test. This new unit test provides two benefits: first, it allows the developer to focus on the problem without the additional calls made by the property-based testing framework; and second, the unit test acts as a regression test. Because property-based tests generate random values, there is no guarantee that the test will use the exact failing values upon subsequent runs. Creating a dedicated unit test that explicitly forces the use of these failing values ensures that the specific bug will not slip through. Furthermore, property-based tests assist in design because they encourage the developer to think about the code in terms of invariants and contracts—that is, thinking about what must remain unchanged and what must always be true. This extra perspective has a strong effect on the code, helping to remove edge cases and highlighting functions that store data in an inconsistent state. Consequently, property-based testing is complementary to unit testing because they address different concerns, and utilizing both is recommended.
