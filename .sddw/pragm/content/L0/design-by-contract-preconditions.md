# design-by-contract-preconditions

**Parent:** [[content/L1/software-design-principles-and-process|software-design-principles-and-process]] — Software development requires adherence to principles like DRY, which mandates single, authoritative knowledge representations, and employing techniques such as PERT (optimistic, most likely, pessimistic times) and 'Eating the Elephant' (iterative thin slices) to manage uncertainty.

Regarding Design by Contract (DBC), the primary question is determining whether the caller or the routine being called is responsible for checking the precondition. When DBC is implemented as part of the language, the responsibility rests with neither party: the precondition is tested by the system behind the scenes after the calling code invokes the routine but before the routine's execution begins. Therefore, if explicit checking of parameters is required, the calling code must perform it, because the routine itself cannot see parameters that violate its precondition. For example, consider a program that reads a number from the console, calculates its square root by calling the `sqrt` function, and then prints the result. The `sqrt` function has a precondition that requires its argument must not be negative. If the user enters a negative number at the console, the calling code must ensure that the negative number is never passed to `sqrt`. The calling code has options for handling this failure: it could terminate, issue a warning and read a different number, or change the negative number to a positive number and append 'i' to the result returned by `sqrt`. Regardless of the chosen method, managing the input failure is the responsibility of the calling code, not `sqrt`'s. By formally expressing the domain of the square root function within the precondition of the `sqrt` routine, the system shifts the burden of correctness to the calling code, which is where the responsibility properly belongs. This allows the programmer to design the `sqrt` routine secure in the knowledge that its input will be within the required range.

Furthermore, DBC works well with the concept of crashing early. By using an `assert` or DBC mechanism to validate the preconditions, postconditions, and invariants, a programmer can crash early and report more accurate information about the problem. For example, if a method calculates square roots and requires a DBC precondition that restricts the domain to positive numbers, languages that support DBC will yield an informative error (such as `sqrt_arg_must_be_positive`) along with a stack trace when a negative parameter is passed. This behavior is superior to the alternative in other languages like Java, C, and C++, where passing a negative number to `sqrt` merely returns the special value NaN (Not a Number). If the developer attempts mathematical operations on NaN at a later point in the program, the results may be surprisingly difficult to diagnose. It is significantly easier for a developer to find and diagnose the problem by crashing early, specifically at the site of the problem.
