# orthogonality-principles

**Parent:** [[content/L1/orthogonality-design-principles-and-anti-patterns|orthogonality-design-principles-and-anti-patterns]] — The synthesis details the principles of orthogonality, outlining how component separation improves productivity and reduces risk, and discusses technical strategies for maintaining this separation in development and data handling.

To apply the principle of orthogonality to system design, developers should first understand that orthogonal systems are composed of a set of cooperating modules, where each module implements functionality independently of the others. This concept can be described using terms such as modular, component-based, or layered systems.

Layered approaches are particularly powerful for designing orthogonal systems because each layer uses only the abstractions provided by the layers below it. This design provides great flexibility in changing underlying implementations without affecting the code and reduces the risk of runaway dependencies between modules. A useful test for orthogonal design is asking: If one dramatically changes the requirements behind a particular function, how many modules are affected? In a truly orthogonal system, the expected answer should be 'one.' For example, modifying a button on a Graphical User Interface (GUI) panel should not necessitate a change in the database schema, and adding context-sensitive help should not change the billing subsystem.

When considering a complex system, such as one monitoring and controlling a heating plant, if the original requirement for a graphical user interface (GUI) is changed to include a mobile interface for engineers to monitor key values, an orthogonally designed system requires changing only the modules associated with the user interface; the underlying logic for controlling the plant remains unchanged. By carefully structuring the system, developers should be able to support both interfaces using the same underlying codebase.

Furthermore, developers must assess how decoupled their design is from real-world changes. Utilizing external identifiers that developers do not control—such as telephone numbers, postal codes, Social Security Numbers, government IDs, email addresses, and domains—introduces risks because these identifiers could change at any time. Developers must not rely on the properties of external things they cannot control.

**Toolkits and Libraries:** Developers must exercise caution when integrating third-party toolkits and libraries to preserve system orthogonality. When adopting any toolkit or library, developers should evaluate whether it forces changes into the code that should not be present. An object persistence scheme is considered orthogonal if it is transparent; if it requires creating or accessing objects in a special way, it is not. Isolating such details from the core code also makes it easier to change vendors in the future. For instance, the Enterprise Java Beans (EJB) system illustrates orthogonality: instead of application code having to delineate the start and end of each transaction, EJB expresses this information declaratively using annotations outside the methods that do the work. This allows the same application code to run in different EJB transaction environments without modification. The Decorator Pattern serves as another example of this style, demonstrating how to add functionality without changing existing structures, a technique applicable in nearly every programming language and requiring only programming discipline.
