# incremental-development-and-testing

**Parent:** [[content/L1/adaptive-software-development-principles|adaptive-software-development-principles]] — Effective software development demands an incremental, adaptive approach, integrating principles like DRY, orthogonality, and reversibility, alongside mandatory thorough testing via Design by Contract and property-based testing; furthermore, testing provides critical communication benefits, as evidenced by a developer who realized the primary value of testing stems from the act of considering its impact on the code structure.

To build software effectively, developers must adopt an incremental approach, continuously building small pieces of end-to-end functionality while learning about the problem and involving the customer at each stage to guide the process.

Although Test-Driven Development (TDD) is beneficial, developers must periodically step back to view the overall architecture. Focusing solely on TDD can lead to writing excessive code that does not advance the solution. For example, in 2006, Ron Jeffries, a leading figure in the agility movement, documented his test-driven coding of a Sudoko solver. Although Jeffries refined the board's representation and refactored the object model across several posts, he eventually abandoned the project, demonstrating how a developer can become distracted by minor details, even with the positive feedback of passing tests.

In contrast, Peter Norvig suggested an alternative strategy: instead of being driven by tests, the developer should begin by understanding how the problem is traditionally solved (e.g., using constraint propagation), then focus on refining the algorithm. This approach was demonstrated by Norvig addressing board representation in just a dozen lines of code, which directly flowed from his discussion of notation.

While tests help drive development, without a clear destination, the process risks becoming circular. Component-based development—the goal of having generic software components combined as easily as common integrated circuits (ICs)—is only reliable if developers ensure that the components are known to be dependable, requiring standardized voltages, interconnects, and timing. Therefore, developers must incorporate testability into the software from the outset, thoroughly testing every piece before integrating them.

Software unit testing is analogous to chip-level testing for hardware: it verifies a module's behavior in isolation. A unit test establishes an artificial environment, invokes routines within the module, and checks the results against known values or against previous runs (a process called regression testing).
