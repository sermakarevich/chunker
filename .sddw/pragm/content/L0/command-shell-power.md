# command-shell-power

**Parent:** [[content/L1/estimation-productivity-and-data-standards|estimation-productivity-and-data-standards]] — Developers must master estimation techniques like using PERT (optimistic, most likely, pessimistic time) and practicing iterative development (Eating the Elephant Technique) by focusing on refining schedules based on recorded experience. Technical proficiency requires using plain text for self-describing data (over binary formats), utilizing the command shell as the primary workspace (through aliases and functions), and always anticipating the need to operate beyond the limitations of IDEs or GUIs.

The command shell serves as the primary workspace for a programmer manipulating text files, functioning analogously to a workbench for a woodworker. From the shell prompt, a developer can invoke their full repertoire of tools, using pipes to combine them in ways that original developers never conceived. Using the shell, developers can launch applications, debuggers, browsers, editors, and utilities, and they can also search for files, query the status of the system, and filter output. Furthermore, developers can build complex macro commands for frequently performed activities by programming the shell.

For programmers who primarily use Graphical User Interface (GUI) environments and integrated development environments (IDEs), relying solely on clicking and pointing might seem sufficient. While GUI interfaces are wonderful and can be faster and more convenient for simple operations—such as moving files, reading/writing email, or building/deploying a project—relying exclusively on GUIs prevents the developer from accessing the full capabilities of the environment. Specifically, developers lose the ability to automate common tasks, utilize the full power of available tools, or combine those tools to create customized macro tools.

GUI environments are typically limited to the specific capabilities intended by their designers. If the developer needs to operate beyond the defined model, the developer often finds this difficult. Since Pragmatic Programmers engage in tasks that extend beyond just writing code, developing object models, writing documentation, or automating the build process, encountering a limitation is common. For example, if a developer needs to integrate a code preprocessor (such as one implementing design-by-contract or multi-processing pragmas) into an IDE, the developer cannot perform this unless the IDE designer explicitly provided the necessary hooks.

To boost productivity, developers should gain familiarity with the command shell. For instance, to create a list of all unique package names explicitly imported by Java code, a developer can execute the following sequence of commands, storing the result in a file named `list`: `grep '^import ' *.java | sed -e's/.*import *//' -e's/;.*$//' | sort -u >list`. Though this process might appear daunting to a novice, investing energy in becoming familiar with the shell quickly improves productivity. Developers should play around with the command shell to realize its utility.

**Customizing the Shell:**

Similar to a woodworker customizing a workspace, a developer should customize the shell, which often involves changing the configuration of the terminal program. Common customizations include:

*   **Setting color themes:** Developers may spend many hours selecting from available themes.
*   **Configuring a prompt:** The prompt, which signals that the shell is ready for commands, can be configured to display various information. Personal preferences often favor simple prompts that include a shortened current directory name, version control status, and the time.
*   **Creating aliases and shell functions:** Developers can simplify workflows by converting frequently used commands into simple aliases. For example, to create an alias for updating a Linux system (`sudo apt-get update` followed by `sudo apt-get upgrade`), a developer can use: `alias apt-up='sudo apt-get update && sudo apt-get upgrade'`. Similarly, to ensure that accidentally deleting files requires confirmation, a developer can set an alias: `alias rm ='rm -iv'`. 
*   **Command completion:** Most shells automatically complete command and file names when the user presses Tab. Developers can extend this functionality by configuring the shell to recognize the entered command and offer context-specific completions, some even customizing completions based on the current directory.

Because developers will spend significant time interacting with the shell, they should treat it like a customized home. The concept of the command shell supports the power of plain text, which is also a recommended practice. Developers should also challenge themselves by identifying manual tasks currently performed using a GUI and investigating whether an automated shell command can replace them. Furthermore, when moving to a new environment, developers should inquire about available shells and attempt to migrate their preferred shell to the new system. If the current shell cannot solve a encountered problem, developers should investigate alternative shells that might perform better.
