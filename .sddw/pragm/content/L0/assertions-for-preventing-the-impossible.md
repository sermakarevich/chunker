# assertions-for-preventing-the-impossible

**Parent:** [[content/L1/software-development-best-practices|software-development-best-practices]] — Developers must adopt principles like DRY, PERT, and the Tracer Bullet method, and use assertions to proactively detect impossible software outcomes, while managing limited resources through structured allocation/deallocation patterns.

The principle of Assertive Programming suggests that developers should not rely on self-deception, believing that certain impossible outcomes—such as a parameter being negative, a log failure, or a result being null—cannot occur. Instead, whenever a developer thinks, "of course that could never happen," the developer should incorporate code to explicitly check for that possibility using assertions. In many programming language implementations, a form of `assert` can perform this Boolean condition check. For example, if a parameter or a result should never be null, the developer can explicitly check for it using `assert (result != null)`. For a more detailed check in Java, the developer should include a descriptive string, such as `assert result != null && result.size() > 0 : "Empty result from XYZ";`. Assertions are also valuable for verifying the operation of an algorithm; for instance, after running a sorting algorithm named `my_sort` on a list of books, the developer can assert that `is_sorted?(books)` is true. However, developers must not use assertions as a substitute for real error handling. Assertions are designed to check for conditions that should never happen; using them for input validation, such as asserting that a user's input character (`ch`) must be either 'Y' or 'N', constitutes a poor practice. Furthermore, because most assert implementations terminate the process upon failure, if a developer needs to free resources, the developer must use a `catch` block to handle the assertion's exception or trap the exit. The developer must ensure that the code executed during the brief period of resource cleanup does not depend on the information that triggered the assertion failure.

Special caution is needed when assertions have side effects, meaning that evaluating the condition itself changes the state of the program. For example, asserting `iter.nextElement() != null` inside a `while` loop will have the side effect of moving the iterator past the element currently being fetched, causing the loop to process only half of the elements in the collection. Instead, the developer should retrieve the object first, and then assert the object's null status, like this: `Object obj = iter.nextElement(); assert(obj != null);`. Failing to address this side effect problem constitutes a kind of Heisenbug—a debugging technique that alters the system's behavior. 

Another common misunderstanding is the belief that assertions add overhead and are only necessary during debugging. Developers must understand that a program runs in a dangerous production world, where issues like rats chewing communications cables or log files filling storage partitions can occur. Therefore, the first line of defense for developers should be checking for any possible error, and the second line of defense involves using assertions to detect missed bugs. Turning off assertions when delivering a program to production, even if the program passed all tests, is equivalent to crossing a high wire without a net. If performance issues arise, developers should only disable the specific assertions that cause unacceptable overhead, leaving the rest in place.

Developers should use assertions in production, as shown by a former neighbor's startup company that made network devices. This company successfully achieved stability and fixed obscure, hard-to-reproduce bugs by leaving well-crafted assertions in their production releases. This level of feedback, provided to real users under actual conditions, allowed the developer team to fix critical issues, resulting in remarkably stable software and leading to the company's acquisition for hundreds of millions of dollars.
