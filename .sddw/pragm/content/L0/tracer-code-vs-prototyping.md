# tracer-code-vs-prototyping

**Parent:** [[content/L1/flexible-architecture-and-tracer-development|flexible-architecture-and-tracer-development]] — The comprehensive development strategy requires establishing flexible architecture through abstraction (hiding third-party APIs) and componentization (breaking code into parts), while adopting 'tracer development'—an iterative process where skeleton code is built end-to-end to provide immediate, visible feedback, contrasting with conventional 'dead reckoning' methods.

The tracer code approach addresses a different problem: understanding how the application operates as a whole. When a development team wishes to show users how interactions will work in practice and simultaneously provide developers with an architectural skeleton to build code upon, tracer code is useful. For example, the team might construct a tracer consisting of a trivial implementation of the container packing algorithm (such as first-come, first-served) and a simple but functional user interface. Once all the components within the application are integrated, the resulting framework can be shown to both users and developers. Over time, the team can add new functionality and complete stubbed routines, but the core framework remains intact, ensuring the system continues to behave as it did when the initial tracer code was completed. The distinction between tracer code and prototyping is important enough to warrant repetition: Prototyping generates disposable code, while tracer code is lean but complete, and it forms part of the final system's skeleton. Think of prototyping as the reconnaissance and intelligence gathering that happens before a single tracer bullet is fired.
