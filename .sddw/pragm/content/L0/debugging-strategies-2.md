# debugging-strategies-2

**Parent:** [[content/L1/software-development-principles-and-debugging-protocols|software-development-principles-and-debugging-protocols]] — Software development mandates using the DRY principle, abstracting third-party APIs, and employing iterative planning techniques like PERT and the 'Eating the Elephant Technique'; debugging requires reproducing the bug using single commands, employing advanced techniques like binary chopping (for stacks, datasets, and versions), and utilizing tracing statements or the Rubber Ducking method to understand pre-current state behavior.

The most effective way to begin fixing a bug is to ensure that the bug can be reproduced. If a developer cannot reproduce the bug, they will not know if the fix was successful. Furthermore, debugging is easiest when the bug can be reproduced using a single command, rather than requiring a long series of steps.

**Tip 31: Failing Test Before Fixing Code**
Sometimes, by forcing a developer to isolate the specific circumstances that display the bug, the developer gains insight into how to fix it, and the act of writing the test itself informs the solution. If a developer encounters a problem, they should use a debugger and a failing test to reliably trigger the problem. Before attempting to analyze the failure, the developer must first ensure they can see the incorrect value within the debugger, thereby avoiding the time waste of hours spent tracking down a bug only to discover that a particular run of the code worked correctly.

When looking into why a value is incorrect, the developer should examine the local stack environment by knowing how to move up and down the call stack. Keeping pen and paper nearby can help developers jot down notes, particularly when a clue is found and a developer must follow that clue, otherwise, the developer risks losing a significant amount of time while trying to trace back to the starting point.

If a developer encounters a stack trace that appears to scroll infinitely, a quicker method than examining every single stack frame is using a binary chop. A binary chop is an efficient technique used for debugging, similar to how a binary search finds a specific value in a sorted array. Instead of performing a linear search (which averages half the entries until a value is found or a value greater than the target is found), a developer uses a divide-and-conquer approach. For example, when facing a massive stack trace and trying to find which function corrupted the value in error, the developer chooses a stack frame near the middle. If the error is visible there, the problem lies in the frames before; otherwise, it is in the frames after. By repeating this procedure, even with 64 frames in the stack trace, the developer can determine the cause in at most six attempts.

Developers can also apply the binary chop technique when dealing with data that exhibits bugs. If a developer finds bugs that appear on certain datasets, the developer should split the dataset into two, running the application with each half. The developer should continue dividing the data until a minimum set of values is obtained that consistently exhibits the problem.

Similarly, if a team introduces a bug during a series of releases, the team can use the binary chop technique. The developer must create a test that causes the current release to fail, then select a release that is halfway between the current version and the last known working version. Running the test again and determining whether the test fails or passes helps the developer narrow the search scope. The author notes that having good version control is beneficial because many version control systems will automate this process, selecting the appropriate releases for the developer.

**Sensitivity to Input Values**
If a program works fine with all test data but crashes after a week in production when fed a specific dataset, the developer should not only look at the place the program crashes and work backward. It is often easier to begin with the data: the developer should obtain a copy of the problematic dataset and feed it through a locally running copy of the application, confirming that the crash still occurs. Then, the developer should binary chop the data until the input values that lead to the crash are exactly isolated.
