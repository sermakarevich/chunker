# security-best-practices-and-least-privilege

**Parent:** [[content/L1/advanced-software-design-and-security-principles|advanced-software-design-and-security-principles]] — Software development requires adherence to foundational principles including DRY, orthogonality, and least privilege, demanding developers use interfaces to control global data and encrypt sensitive information like passwords and PII. Testing relies on Property-Based Testing (which identifies failing parameters used to create dedicated regression unit tests) and Design by Contract (DBC), alongside secure password policies per NIST guidelines (e.g., minimum 64 characters, accepting Unicode). Naming elements must clearly reflect intent, and system security requires rapid patching and using vetted, third-party cryptographic services.

A core principle in software development is applying the principle of least privilege: developers should only use the minimum necessary permission level for the shortest amount of time. Developers should never automatically acquire the highest permission level, such as root or Administrator. If the high level of privilege is necessary, developers must take it, complete the minimum necessary work, and then relinquish the privilege quickly to reduce risk. This principle, articulated by Jerome Saltzer in 1974, guides system design by advising that every program and every privileged user of the system should operate using the least amount of privilege necessary to complete the job. For example, Unix-derived login programs initially execute with root privileges but promptly drop the high level of privilege to that of the authenticated user.

This concept extends beyond operating system privilege levels; applications should implement different, finely grained levels of access, allowing sensitive resources to be partitioned into categories, and granting individual users permissions for only certain categories. This technique minimizes the attack surface, restricting the scope of attack vectors both by time and by privilege level, ensuring that less is more.

Regarding initial settings, default settings for applications or users on a site should be set to the most secure possible values. While these secure defaults might not be the most convenient or user-friendly values, it is better to leave the trade-off decisions between security and convenience to individual users. For instance, the default for password entry should be to hide the password by replacing each character with an asterisk, which is a sensible default in public or crowded settings. However, some users might require seeing the password, perhaps for accessibility; in such cases, where there is little risk of onlookers, that remains a reasonable user choice.

Developers must encrypt sensitive data, including personally identifiable information, financial data, passwords, and other credentials, whether the data resides in a database or an external file. If this data is exposed, encryption provides an additional layer of safety.

When managing project resources, developers should store all necessary project components under version control, with one major exception: Never commit secrets, API keys, SSH keys, encryption passwords, or other credentials alongside the source code in version control. Instead, these keys and secrets must be managed separately, typically via config files or environment variables during the build and deployment process.

When considering password security, developers must understand that common security practices often conflict with common sense. Strict password policies, for example, can actually lower overall security. The National Institute of Standards and Technology (NIST) provides the following recommendations, and developers should follow them:
*   Do not restrict password length to less than 64 characters; NIST recommends 256 as a suitable maximum length.
*   Do not truncate a user's chosen password.
*   Accept all printable ASCII characters, space, and Unicode; developers should not restrict special characters like `[]();&%$#` or `/`, as these special characters are unlikely to compromise the system.
*   Do not provide password hints to unauthenticated users, nor should the system prompt for specific types of information (such as “what was the name of your first pet?”).
*   Do not disable the paste function in the browser or password managers. Such actions fail to increase security and instead motivate users to create simpler, shorter passwords that are much easier to compromise; both the US NIST and UK National Cyber Security Centre require verifiers to allow paste functionality for this reason.
*   Do not impose arbitrary composition rules, such as mandating a specific mix of upper and lower case, numerics, or special characters, or prohibiting repeating characters.
*   Do not arbitrarily require users to change their passwords after a set length of time, restricting such changes only when there is a valid reason, such as after a security breach.

Ultimately, developers should encourage long, random passwords with a high degree of entropy, as imposing artificial constraints limits entropy and encourages poor password habits, leaving user accounts vulnerable to takeover.

Concerning system maintenance, developers must apply security patches quickly. Waiting to update a system, even if it causes a temporary application breakage, is a poor choice because the system remains vulnerable to a known exploit. This principle applies to every network-connected device, including phones, cars, appliances, personal laptops, developer machines, build machines, production servers, and cloud images, because historical data shows that the largest data breaches were caused by systems that lagged behind on updates.

Finally, concerning cryptography, developers must remember that common sense often fails when dealing with cryptography. The most important rule is never to perform crypto work personally. Even simple elements like passwords can be managed improperly. When engaging with cryptography, developers must only rely on reliable resources: well-vetted, thoroughly examined, well-maintained, frequently updated, and preferably open-source libraries and frameworks. For authentication, the pragmatic approach is to use a third-party authentication provider, such as services offered by email, phone, or social media providers. These external providers spend their time maintaining their systems' security, making them experts in a field the local developer is not.
