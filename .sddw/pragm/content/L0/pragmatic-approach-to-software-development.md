# pragmatic-approach-to-software-development

**Parent:** [[content/L1/software-development-architecture-best-practices|software-development-architecture-best-practices]] — Successful software development requires an adaptive, context-specific approach guided by continuous feedback loops, utilizing principles like DRY, orthogonality, and least privilege to manage architectural complexity. Teams must facilitate communication instantly (using 'tracer bullets' to ensure all skills traverse the whole system end-to-end), automate all processes, and adhere to stringent security practices such as encrypting all sensitive data, utilizing third-party authentication, and never committing credentials alongside source code.

To successfully develop software, teams must reject adopting methodologies or processes simply because successful companies like Spotify, Netflix, Stripe, or GitLab use them. Instead, teams must adopt a context-specific approach by evaluating if a particular development method, framework, or testing technique is actually well-suited for the current job and team. The context should include market conditions, constraints, opportunities, expertise, organization size, management, user base, and requirements. Artifacts, superficial structures, policies, processes, and methods alone are insufficient for successful development. The core goal of software development is to deliver working software that provides users with new capability immediately, rather than weeks, months, or years from now. Improvement should focus on continuously shortening the delivery cycle: from years, to months, to weeks, to a single day, and eventually, on-demand delivery. When transitioning to continuous development, teams must first build a robust infrastructure, using the main trunk of the version control system and employing techniques like feature switches to selectively release test features. Initially, teams may start by combining Scrum for project management with technical practices from eXtreme Programming (XP), or, if the team is more disciplined and experienced, might consider Kanban and Lean techniques for both team processes and larger governance issues. Teams must investigate and try these approaches themselves, but care must be taken not to overinvest in any single methodology, as overreliance can create calcification, making it difficult to adapt quickly. The central principle is that no single plan works for all, and the best practice is to adapt the best elements from various methodologies. Developers should apply this pragmatic approach by piloting ideas with a small team or set of teams, keeping only the elements that function effectively and discarding the rest as overhead or waste.
