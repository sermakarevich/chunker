# testing-and-automation-principles

**Parent:** [[content/L1/software-development-best-practices-2|software-development-best-practices-2]] — Modern software development requires continuous, adaptive practices built on architectural rigor (DRY, orthogonality, abstraction) and process automation (version control, full automation). Testing must be comprehensive, utilizing unit tests for single modules, integration tests for subsystems, validation for user needs, and performance testing for scale. Developers must employ Test-Driven Development (TDD) and Property-Based Testing (PBT) to enhance code quality and resilience, while always adhering to strong ethical principles regarding user safety and system transparency.

To ensure code reliability and system integrity, developers must adopt practices that transcend merely finding bugs. The primary benefits of testing are derived when developers thoughtfully consider and write the tests, not merely when running them. This practice influences the code's initial design. For example, by imagining how a function to query a database for a list of people who watch over 10 videos a week should be tested, developers realize the need to pass the database instance into the function (instead of relying on a global database) and identify the necessity of passing the field name used for qualification as a parameter. A core benefit of writing tests is that it compels developers to view the code from the perspective of a client, rather than from the author's perspective. Furthermore, considering tests helps developers clarify vague understandings of requirements, forcing developers to think about boundary conditions and error conditions before writing any code. Structured Programming: Test-Driven Development (TDD) is a structured programming cycle that operationalizes these ideas. The basic TDD cycle involves five steps: 1) Deciding on a small piece of functionality to add; 2) Writing a test that will pass once the functionality is implemented; 3) Running all tests and confirming that the only failure is the one just written; 4) Writing the smallest amount of code needed to get the test to pass, followed by verifying that the tests run cleanly; and 5) Refactoring the code—improving either the test or the function—while ensuring the tests still pass. Developers must keep this cycle very short, ideally only taking minutes. While TDD is highly beneficial for beginners, developers must avoid becoming overly dependent on the practice. Over-reliance on TDD can lead to excessive time spent ensuring 100% test coverage or writing redundant tests, such as creating a failing test referencing a class name followed by an empty class definition, which results in a test that performs no function.

Property-Based Testing is a powerful technique where a computer generates test data to explore how the code handles unexpected states. Developers should use property-based testing techniques to generate test data according to the contractual agreements and invariants of the code under test. This technique allows developers to validate assumptions by defining properties that must hold true for all possible inputs, rather than writing specific examples. For instance, when testing a sorting function, a developer can establish properties that the sorted list must have the same number of elements as the original list (length is an invariant) and that no element in the result can be greater than the element that follows it. The process of property-based testing involves using a strategy to generate a wide array of random inputs, for example, running a test case one hundred times using a different random list each time, simulating the development of numerous individual tests. Consequently, property-based testing is complementary to unit testing because the two techniques address different concerns, and utilizing both is recommended.

Regarding bugs, developers must implement a principle of continuous improvement: if a bug slips through existing tests, developers must add a new test to trap that specific bug every single time—no matter how trivial the bug or how much the developer complains that the bug will never happen again. Automated tests must be modified to check for that particular bug perpetually. 

In modern development, relying on scripted, automatic procedures is essential. Whether using simple shell scripts with rsync and ssh, or comprehensive solutions such as Ansible, Puppet, Chef, or Salt, developers must not rely on manual intervention. For example, if developers are at a client site where all developers used the same IDE, the system administrator might provide instructions for installing add-on packages. If every developer's machine was loaded slightly differently, subtle differences in the application’s behavior could occur when different developers ran the same code, resulting in bugs appearing on one machine but not others. Therefore, developers must understand that manual procedures are not repeatable like computers. A shell script or program will execute the same instructions, in the same order, time after time. Since automated procedures are under version control, developers can examine changes to the build/release procedures over time. Furthermore, developers cannot build a project on an anonymous cloud server unless the build is fully automatic, nor can the development team deploy automatically if manual steps are involved. Introducing any manual steps, even temporarily, compromises the entire system foundation. 

By combining version control, rigorous testing, and full automation, a project establishes a firm foundation, allowing the development team to focus on delighting users. Developers must remember that the ultimate goal of the project is not merely delivering working software in a timely manner; users are not motivated by code, but rather by solving a core business problem within the context of their objectives and budget. Developers must therefore ask the critical question: “How will you know that we’ve all been successful a month (or a year, or whatever) after this project is done?” The answer will reveal the business value—whether it is customer retention, data quality, or cost savings—that truly matters, as the software is only a means to these ends. Throughout development, developers must proactively analyze the user requirements in light of these underlying business expectations and continuously think about how to deliver against them.
