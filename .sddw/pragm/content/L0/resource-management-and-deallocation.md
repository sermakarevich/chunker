# resource-management-and-deallocation

**Parent:** [[content/L1/software-development-best-practices|software-development-best-practices]] — Developers must adopt principles like DRY, PERT, and the Tracer Bullet method, and use assertions to proactively detect impossible software outcomes, while managing limited resources through structured allocation/deallocation patterns.

To manage resources in code—such as memory, transactions, threads, network connections, or files—developers must recognize that all these resources have limited availability. Beyond managing logging files or debug files, developers must also consider the resources they leave behind, such as data written to a database, and implement a mechanism to expire them. For any finite resource created, developers must consider how to maintain resource balance.

Regarding resource allocation, the basic pattern can be extended to routines that require multiple resources simultaneously. Developers should follow two rules: 1) Deallocate resources in the opposite order from which they were allocated. This prevents orphaned resources if one resource references another. 2) When allocating the same set of resources in different parts of the code, always allocate the resources in the same order to minimize the possibility of deadlock. (A deadlock occurs if Process A claims resource1 and waits for resource2, while Process B has claimed resource2 and waits for resource1, causing both processes to wait forever.)

This basic pattern—that the routine that allocates a resource must also be responsible for deallocating it—applies regardless of the resource type (transactions, network connections, memory, files, threads, windows). However, modern languages allow for developing this concept further.

In object-oriented languages, the concept of resource management mirrors an object's constructor (which allocates the resource) and its destructor (which removes the resource from the scope). Therefore, when developing in an object-oriented language, developers should encapsulate resources within classes. Every time a specific resource type is needed, the developer should instantiate an object of that class. When the object goes out of scope or is reclaimed by the garbage collector, the object's destructor should automatically deallocate the wrapped resource. This approach is particularly beneficial when working with languages where exceptions might interfere with resource deallocation.

When using languages that support exceptions, guaranteeing resource deallocation requires careful planning. Developers generally have two choices: 1) Utilize variable scope (such as stack variables in C++ or Rust). With usual scoping rules, a variable's memory is reclaimed when the variable exits scope (via return, block exit, or exception). Furthermore, developers can hook into the variable's destructor to clean up any external resources. For example, in Rust, the variable `accounts` will automatically close the associated file when it exits the scope block. 2) Utilize a `finally` clause within a `try...catch` block. A `finally` clause ensures that the specified code runs regardless of whether an exception was raised in the `try` block or caught in the `catch` block.

Developers must avoid the Exception Antipattern, which involves attempting to deallocate a resource in the `finally` block if the initial resource allocation fails and raises an exception. The correct pattern requires performing the allocation before the `begin` block and wrapping the processing logic in the `begin...finally` structure.

When the basic resource allocation pattern is insufficient, such as in programs using dynamic data structures, developers must establish a semantic invariant for memory allocation. Developers must explicitly decide which entity is responsible for data in an aggregate data structure. Developers have three main options for handling deallocation: 1) The top-level structure is responsible for freeing any contained substructures, which then recursively delete their contents. 2) The top-level structure is simply deallocated, allowing any unreferenced contained structures to become orphaned. 3) The top-level structure refuses to deallocate itself if it contains any substructures. The choice among these options depends on the specific circumstances of each data structure, and the developer must implement the chosen decision consistently. If implementing these options in a procedural language like C, which does not make data structures active, developers should write a separate module for each major structure to provide standard allocation and deallocation facilities. This module can also offer debugging features like serialization, deserialization, and traversal hooks.

Finally, developers should implement code that actively checks that resources are properly freed. For most applications, this involves producing wrappers for each resource type to track all allocations and deallocations. At critical points in the program logic, developers should use these wrappers to confirm that the resources are in the expected state. For a long-running request-servicing program, a good place to check resource usage is at the top of the main processing loop, right before waiting for the next request. A lower, but equally useful, approach involves investing in tools that check running programs for memory leaks.
