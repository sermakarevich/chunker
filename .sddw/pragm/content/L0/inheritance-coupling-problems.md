# inheritance-coupling-problems

**Parent:** [[content/L1/program-as-transformation-pipeline|program-as-transformation-pipeline]] — Programming is best viewed as a data transformation pipeline (like a Unix pipe: `find . -type f | xargs wc -l | sort -n | tail -5`), which minimizes coupling by passing data continuously through functional steps; to achieve polymorphism and structure robust code, developers should favor interfaces/protocols, delegation, and mixins/traits over inheritance to avoid architectural coupling.

When designing software, developers should be aware of the pitfalls of relying on inheritance, as this concept can lead to significant coupling. The discussion of inheritance originated with Simula 67 in 1969, initially serving as an elegant solution for queuing multiple types of events on a single list using prefix classes. For example, a `link` class, which acted as a prefix class, allowed both `car` and `bicycle` classes to be added to a list of waiting items at a traffic light. In this Simula approach, the mental model dictated that the instance data and implementation of the `link` class were prepended to the implementations of `car` and `bicycle`, making `link` function almost as a container. This structure provided polymorphism, as both `car` and `bicycle` implemented the `link` interface because they both contained the `link` code.

Over time, the Smalltalk school emerged. Alan Kay, one of the creators of Smalltalk, noted in a 2019 Quora answer that Smalltalk used inheritance for purely behavioral design. The Simula approach, which suggested that inheritance was a method for combining types, persisted in languages like C++ and Java, while the Smalltalk school, which treated inheritance as a dynamic organization of behaviors, was seen in languages like Ruby and JavaScript. Consequently, modern Object-Oriented (OO) developers utilize inheritance for one of two primary reasons: avoiding typing or expressing class relationships (e.g., declaring that a `Car` is a type of `Vehicle`).

However, both types of inheritance introduce substantial problems.

**Problems with Inheritance for Sharing Code:**
Inheritance introduces coupling, not only between the child class and the parent, but also between the parent’s parent, and so on. Furthermore, the code that uses the child class becomes coupled to all the ancestors. For example, in a given system, if the `Vehicle` class has methods like `stop` and `move_at(speed)`, and the `Car` class inherits from `Vehicle`, the top-level code calls `my_ride.move_at(30)`. If the developer in charge of `Vehicle` changes the API—for instance, changing `move_at` to `set_velocity` and changing the instance variable `@speed` to `@velocity`—this API change is expected to break clients of the `Vehicle` class. However, the top-level code assumes it is using a `Car`, not `Vehicle`, yet the implementation still breaks because the name of the instance variable is purely an internal implementation detail, resulting in significant coupling.

**Problems with Inheritance for Defining Types:**
Viewing inheritance as a means of defining new types, as depicted in class hierarchy diagrams, can lead to unnecessary complexity. Such diagrams, layer-upon-layer added to express even small differentiations between classes, can make the application more brittle, since changes can propagate across many layers. A worse problem arises with multiple inheritance. A `Car` might be a type of `Vehicle`, but it could also be a type of `Asset`, `InsuredItem`, or `LoanCollateral`. Modeling this correctly would necessitate multiple inheritance. Historically, C++ was criticized for multiple inheritance in the 1990s due to some questionable disambiguation semantics, leading many modern OO languages to omit it.
