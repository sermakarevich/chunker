# blackboard-system-architecture

**Parent:** [[content/L1/concurrency-architectural-patterns-and-coding-practices|concurrency-architectural-patterns-and-coding-practices]] — Software development requires continuous, non-mechanical critical decision-making, underpinned by architectural principles like DRY, orthogonality, and decoupling; for concurrency, the Actor Model (featuring independent state, mailboxes, and message-only communication) and Blackboard Systems (allowing asynchronous coordination by independent agents) are preferred alternatives to traditional shared resources, even if they increase system complexity.

Erlang, along with its successor Elixir, provides actor implementations for most programming languages, suggesting that developers consider using these tools for concurrent implementations. Regarding architectural design, the content discusses the concept of 'Blackboards,' which can be used by detectives (or independent agents/actors) to coordinate and solve large, complex problems, such as a murder investigation. The chief inspector initiates the process by posting a single question—for example, 'H. Dumpty (Male, Egg): Accident? Murder? Did Humpty really fall, or was he pushed?'—onto a large blackboard. As data accumulates, detectives (agents) contribute findings, witness statements, forensic evidence, or observations, which are then posted to the blackboard. This process continues across all shifts until the case is resolved. Key features of the blackboard approach include: (1) Detectives do not need knowledge of each other's existence; they simply watch the board for new information and add their own findings. (2) The agents may come from different disciplines, possessing varied educational levels and expertise, and do not necessarily work within the same precinct. (3) The agents share a common goal (solving the case), but this is the only connection they require. (4) The agents may arrive and depart during the process, and may work different shifts. (5) The blackboard imposes no restrictions on the types of content posted; acceptable submissions include pictures, sentences, or physical evidence. This exemplifies a form of laissez faire concurrency, where detectives are independent processes, agents, or actors. Some agents store facts on the blackboard, while others retrieve facts, potentially combining or processing them, and subsequently adding new information to the board, gradually guiding the group toward a conclusion.

Computer-based blackboard systems were initially designed for complex artificial intelligence applications, such as speech recognition and knowledge-based reasoning systems. One early example was David Gelernter’s Linda, which stored facts as typed tuples, allowing applications to write new tuples and query for existing tuples using pattern matching.

Later developments included distributed blackboard-like systems, such as JavaSpaces and T Spaces. These systems allow storing active Java objects—not merely data—on the blackboard. Users can retrieve objects by using partial matching of fields (via templates and wildcards) or by subtype. For example, searching a blackboard containing `Person` objects using an `Author` template with the last name 'Shakespeare' retrieves Bill Shakespeare (the author) but excludes Fred Shakespeare (the gardener).

In the context of large-scale applications, such as processing mortgage or loan applications, the blackboard approach offers an elegant solution to complex difficulties, including: 1) The odiously complex and constantly changing legal requirements from federal, state, and local governments; 2) The asynchronous and variable arrival order of data (e.g., credit checks may take time while name and address may be immediately available); 3) The potential for data gathering to occur by different people, distributed across different time zones, or automatically by other systems; 4) The fact that while certain data may be dependent on other data (e.g., title search requires proof of ownership or insurance); and 5) The realization that arriving new data may trigger new policies and raise new questions.

While a workflow system could attempt to handle every possible combination and circumstance, leading to complexity that must be reorganized as regulations change, a blackboard, when combined with a rules engine that encapsulates legal requirements, provides an elegant solution. Because the rules engine posts the output of any set of rules back to the blackboard, the blackboard approach ensures that the order of data arrival is irrelevant, and feedback loops are easily managed. 

Messaging systems, such as Kafka and NATS, can function similarly to blackboards and can also support the running of multiple actors. While utilizing the actor and/or blackboard and/or microservice approach removes a whole class of potential concurrency problems, this benefit carries a cost: the resulting architectures are harder to reason about because much of the action is indirect. Therefore, developers must maintain a central repository of message formats and/or APIs, preferably one that can generate code and documentation. Furthermore, good tooling is necessary to trace messages and facts as they progress through the system, such as propagating a unique trace ID to all involved actors so that developers can reconstruct events from log files. While these systems are more troublesome to deploy and manage due to more moving parts, this complexity can be mitigated by the fact that the system is more granular, allowing individual actors to be updated without requiring the replacement of the entire system.
