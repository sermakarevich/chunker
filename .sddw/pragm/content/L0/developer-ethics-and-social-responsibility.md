# developer-ethics-and-social-responsibility

**Parent:** [[content/L1/software-development-best-practices-2|software-development-best-practices-2]] — Modern software development requires continuous, adaptive practices built on architectural rigor (DRY, orthogonality, abstraction) and process automation (version control, full automation). Testing must be comprehensive, utilizing unit tests for single modules, integration tests for subsystems, validation for user needs, and performance testing for scale. Developers must employ Test-Driven Development (TDD) and Property-Based Testing (PBT) to enhance code quality and resilience, while always adhering to strong ethical principles regarding user safety and system transparency.

Software has evolved dramatically over the last two decades, transforming from a peripheral curiosity into a core necessity for modern businesses. This growth means that software has profoundly influenced global life, with programs now integrated everywhere—from Twitter assisting with planning revolutions and processors preventing vehicle skidding, to smartphones enabling users to no longer memorize minor daily details. Developers hold a significant degree of power through this capability, meaning they bear an extraordinary responsibility.

This responsibility is amplified when developing for embedded devices, which utilize an order of magnitude more computers than laptops, desktops, or data centers. These embedded computers frequently control life-critical systems, such as power plants, cars, or medical equipment. Poor design or implementation in even a simple central heating control system or home appliance can lead to death. Furthermore, even nonembedded systems can yield both positive and negative outcomes: social media may promote peaceful revolution or incite hate, and big data can simplify shopping while simultaneously eliminating perceived privacy. Banking systems, for example, make loan decisions that fundamentally alter people’s lives, and nearly any system possesses the potential to monitor its users.

The necessary vigilance required by this power means that every piece of delivered code must address two critical questions: First, have developers protected the user from harm? This involves making provisions, such as applying ongoing security patches to a baby monitor, ensuring manual control remains available if an automatic central heating thermostat fails, and limiting data collection to only what is necessary while encrypting all personal data. Developers must acknowledge that while no one is perfect, if a developer cannot truthfully list all possible negative consequences and prove that provisions were made to protect the users from those consequences, the developer bears some responsibility when negative outcomes occur.

Second, developers must consider the Golden Rule: would they be willing to be a user of the software? The developer must determine if they would be comfortable sharing their details or if they would be willing to be driven by an autonomous vehicle. If an individual is involved in a project that skirts ethical boundaries, that developer is responsible, regardless of the degree of separation or how well the developer rationalizes involvement. Developers should therefore consider these moral principles when designing software, actively building the future—for themselves and future descendants—to ensure it is a future everyone wants to inhabit. The process of creation and development is continuous, requiring developers to maintain awareness and have the courage to say 'no' to ideas that contradict this ethical ideal. The ultimate professional advice for developers is to focus on the act of building and to have fun.
