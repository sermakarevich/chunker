# plain-text-and-developer-tools

**Parent:** [[content/L1/estimation-productivity-and-data-standards|estimation-productivity-and-data-standards]] — Developers must master estimation techniques like using PERT (optimistic, most likely, pessimistic time) and practicing iterative development (Eating the Elephant Technique) by focusing on refining schedules based on recorded experience. Technical proficiency requires using plain text for self-describing data (over binary formats), utilizing the command shell as the primary workspace (through aliases and functions), and always anticipating the need to operate beyond the limitations of IDEs or GUIs.

Every developer, when starting a project, begins with a basic set of good-quality tools. For example, a woodworker might need rules, gauges, saws, planes, chisels, drills, braces, mallets, and clamps. These tools should be intentionally chosen, built to last, perform specific jobs with minimal overlap among tools, and feel comfortable in the developer's hands. Following the initial setup, the developer must commit to a process of continuous learning and adaptation. Each technical tool will have its own unique quirks and require special handling; the developer must sharpen each tool in a unique way, or hold each tool just so. Over time, each tool will wear according to its use until the grip fits the developer’s hand, and the cutting surface aligns perfectly with the angle at which the developer holds the tool. At this point, the tools become extensions of the developer's hands. Over time, the developer might add new tools, such as biscuit cutters, laser-guided miter saws, or dovetail jigs, which are all excellent pieces of technology. However, the developer can expect that the developer will feel most comfortable using one of those original tools. Tools amplify a developer's talent: the better the tools and the better the developer knows how to use them, the more productive the developer can be.

Developers should start with a basic set of generally applicable tools. As the developer gains experience and encounters special requirements, the developer should add to this basic set. Like the woodworker, developers must expect to regularly add to their toolbox. Developers must always be on the lookout for better ways of doing things. If the developer encounters a situation where the current tools cannot cut it, the developer should make a note to look for something different or more powerful that would help. The developer needs to let necessity drive the acquisitions. Many new programmers make the mistake of adopting a single power tool, such as a particular integrated development environment (IDE), and never leaving its cozy interface. The developer needs to understand that the developer needs to be comfortable beyond the limits imposed by an IDE. The only way for the developer to achieve this is to keep the basic tool set sharp and ready for use.

This section discusses investing in the developer's own basic toolbox. First, the discussion examines the raw materials, which are knowledge. Developers gather requirements as knowledge, and then the developer expresses that knowledge in designs, implementations, tests, and documents. The developer believes that the best format for persistently storing knowledge is plain text. With plain text, developers give themselves the ability to manipulate knowledge, both manually and programmatically, using virtually every tool at the developer's disposal.

The problem with most binary formats is that the context necessary to understand the data is separate from the data itself, which constitutes an artificial divorce of the data from its meaning. The data may be as meaningless as if it were encrypted without the application logic to parse it. With plain text, however, developers can achieve a self-describing data stream that is independent of the application that created it.

Plain text is defined as printable characters in a form that conveys information. It can be as simple as a shopping list (e.g., milk, lettuce, coffee) or as complex as the source of this book (which is itself written in plain text). However, the information part is important; plain text like `hlj;uijn bfjxrrctvh jkni'pio6p7gu;vh bjxrdi5rgvhj` or `Field19=467abe` is not useful because the reader does not know the significance of the numbers or characters. Developers prefer plain text to be understandable to humans.

Plain text does not imply unstructured text; structured formats such as HTML, JSON, and YAML are also plain text, as are the majority of fundamental protocols on the net, including HTTP and SMTP. These protocols maintain this plain-text format for several reasons: providing insurance against obsolescence, leveraging existing tools, and facilitating easier testing. Human-readable forms of data and self-describing data will outlive all other forms of data and the applications that created them. As long as the data survives, users will have a chance to use the data, potentially long after the original application that wrote it is defunct. Developers can parse such a file with only partial knowledge of its format; with most binary files, users must know all the details of the entire format to parse it successfully. For example, if a developer is given a data file from a legacy system and only needs to extract a list of clients’ Social Security numbers, recognizing the format of a Social Security number allows the developer to quickly write a small program to extract that data, even if the developer has no information on anything else in the file. If, however, the file were formatted using seemingly random characters (e.g., `AC27123456789B11P` or `XY43567890123QTYL`), the significance of the numbers would be much harder to recognize.

While the previous example used the field identifier `<FIELD10>`, using descriptive tags like `<SOCIAL-SECURITY-NO>123-45-6789</SOCIAL-SECURITY-NO>` makes the task trivial and ensures that the data will outlive any project that created it.

Plain text offers two additional advantages: it allows developers to leverage existing tools, as virtually every tool in the computing universe—from version control systems and editors to command-line tools—can operate on plain text. The Unix Philosophy enabled this by creating small, sharp tools, each designed to perform one thing well, using a common underlying format: the line-oriented, plain-text file. This standard is vital because it allows developers to keep all system administration databases (like users and passwords, or networking configuration) as plain-text files. When a system crashes, developers may only be in a minimal environment to restore the system. This situation emphasizes the simplicity of plain text. Furthermore, plain text is easier to search; for example, a quick `grep -r backup /etc` can tell developers which configuration file manages the system backups. If a large application deployment configuration file is in plain text, the developer can place it under a version control system, automatically keeping a history of all changes. File comparison tools such as `diff` and `fc` allow developers to see at a glance what changes have been made, while `sum` allows the developer to generate a checksum to monitor the file for accidental or malicious modification.

Using plain text to create synthetic data for system tests means that developers can simply add, update, or modify the test data without having to create special tools. Similarly, plain-text output from regression tests can be trivially analyzed with shell commands or a simple script. Ultimately, even in future blockchain-based intelligent agents negotiating data interchange, the ubiquitous text file will remain the standard, which is necessary for all parties to communicate.
