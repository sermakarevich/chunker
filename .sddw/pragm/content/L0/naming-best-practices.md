# naming-best-practices

**Parent:** [[content/L1/advanced-software-design-and-security-principles|advanced-software-design-and-security-principles]] — Software development requires adherence to foundational principles including DRY, orthogonality, and least privilege, demanding developers use interfaces to control global data and encrypt sensitive information like passwords and PII. Testing relies on Property-Based Testing (which identifies failing parameters used to create dedicated regression unit tests) and Design by Contract (DBC), alongside secure password policies per NIST guidelines (e.g., minimum 64 characters, accepting Unicode). Naming elements must clearly reflect intent, and system security requires rapid patching and using vetted, third-party cryptographic services.

In software development, developers must treat naming elements—including applications, subsystems, modules, functions, and variables—as critical because the names reveal significant information about the code's intent and underlying assumptions. Developers should adopt the practice of pausing to consider the 'motivation' or 'role' of every piece of code being created. Considering the role of a variable or function forces the developer to think about what is special about it, what specific task it performs, and what components it interacts with; this focused consideration often clarifies the code's logic and makes the code more sensible.

The science behind names suggests that the human brain processes and understands written words very quickly, giving names a high priority when making sense of information. For example, when authenticating users for a site selling jewelry made from old graphics cards, using the variable `user` is better than using a generic name like `container` or `buyer`, as `user` constantly reminds the developer of the person's role. Similarly, when defining a method that applies a discount, a specific name like `applyDiscount(Percentage discount)` is clearer than `deductPercent(double amount)`, as the name clearly communicates the intent, and using a custom `Percentage` type parameter (instead of a raw `double`) documents the expected value range (e.g., between 0 and 100 or 0.0 and 1.0).

When developing modules, developers should select descriptive names that reflect the functionality. For instance, instead of calling a function that calculates a number in the Fibonacci sequence merely `fib(n)`, calling it `Fib.of(n)` or `Fib.nth(n)` provides context, such as `Fib.of(0)` resulting in `0` and `Fib.nth(20)` resulting in `4181`.

While striving for clarity, developers must understand that coding culture dictates naming conventions. For example, some programming languages traditionally use single letters like `i`, `j`, or `k` as loop increment variables in C, and developers working in that environment should adhere to this norm, even if the convention is not followed in other languages. Furthermore, language communities vary in style, preferring either `camelCase` (with embedded capital letters) or `snake_case` (with embedded underscores) for separating words. Developers must honor the local coding culture and be aware that some languages allow a subset of Unicode characters in names.

Developers must prioritize consistency within a project's vocabulary; creating a project glossary listing terms with special team meaning helps the entire team use the jargon consistently. An effective strategy is to encourage pair programming, as this helps jargon spread organically. Because code changes, meaning may subtly shift, so developers must remain vigilant about updating names. When a name no longer expresses the intended meaning, it must be corrected immediately. If a developer encounters a misleading name, the developer must first fix the fundamental design flaw (an ETC violation) and then rename the offending component to make renaming easy and frequent.
