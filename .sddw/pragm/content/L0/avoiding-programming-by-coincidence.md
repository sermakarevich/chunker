# avoiding-programming-by-coincidence

**Parent:** [[content/L1/software-architecture-and-programming-practices|software-architecture-and-programming-practices]] — Robust software development requires structured adaptation using principles like DRY, orthogonality, and reversibility, while strictly adhering to decoupled APIs and using Interfaces/Protocols to manage global data. Developers must prevent 'programming by coincidence' by testing assumptions, planning the application, and actively listening to nonconscious 'lizard brain' signals when encountering resistance or complexity, which may require prototyping. In software development, it is critical for developers to avoid programming by coincidence, which is the act of relying on luck or accidental successes. This process, analogous to a soldier advancing through a minefield only to be hit by a mine despite initial probes revealing nothing, leads to false conclusions and disastrous outcomes. Instead, developers must practice programming deliberately. The danger of coincidence is evident when writing code that appears to function during limited testing but fails later because the developer did not understand the underlying mechanism of the code. 

One common pitfall is relying on 'accidents of implementation,' where the program works only because of undocumented error or boundary conditions. For instance, if a developer calls a routine with bad data and codes based on the routine's specific response, subsequent 'fixing' of that routine can cause the developer's code to break. Another example involves GUI rendering frameworks, where calling routines like `paint()`, `invalidate()`, `validate()`, `revalidate()`, and `repaint()` sequentially might appear to work, even though these routines were never designed to be called in that specific order. The developer should recognize that merely because a method works now does not guarantee its reliability, as the underlying boundary condition might be accidental, undocumented behavior could change with library updates, and additional calls only increase the risk of new bugs and slow down the code.

When writing code that other developers will call, developers should adhere to the basic principles of good modularization and ensure that small, well-documented interfaces are used. A well-specified contract, such as those detailed in Design by Contract, can help eliminate misunderstandings. Conversely, when calling routines, developers must rely only on documented behavior. If documentation is insufficient, the developer must explicitly document the assumed behavior. Furthermore, developers must be cautious of temporal issues, such as the ‘only off by one’ time zone discrepancy in a large project handling hardware data collection units. The developers on that project initially treated the inconsistency as a coincidence, resulting in the codebase accumulating untenable `+1` and `-1` statements, ultimately leading to the project being scrapped due to a fundamental flaw. 

Developers must also be wary of 'phantom patterns'—the human tendency to see causes and patterns even when the underlying occurrences are merely coincidental (e.g., assuming patterns in lottery numbers or stock trading). While this pattern recognition is natural, developers must treat system failures and intermittent errors (like a log file showing an intermittent error every 1,000 requests) with skepticism, recognizing that such errors may be complex race conditions or simple bugs. Similarly, testing that passes on one machine but fails on another may indicate environmental differences or simple coincidence.

To ensure robust code, developers should follow these guidelines:
1. **Prioritize the necessary:** Always be aware of the actions being taken. If a developer cannot explain the code in detail to a junior programmer, they may be relying on coincidences.
2. **Proceed from a plan:** Developers should always plan the application, whether using a written plan, a whiteboard, or a napkin, rather than proceeding without guidance.
3. **Rely on verifiable facts:** Developers must assume the worst if they cannot definitively determine that a dependency or assumption is reliable. The development team should document all assumptions, using Design by Contract to help clarify assumptions for both internal use and communication with other developers.
4. **Test the assumptions:** Developers must not only test the code but also the assumptions. Instead of guessing, developers should write an assertion to test the assumptions; if the assertion passes, the developer has improved the code documentation; if the assertion fails, the developer has identified a potential flaw.
5. **Address scope and history:** Developers must not let existing code dictate future code, understanding that all code can be replaced if it is no longer appropriate. Developers must plan for refactoring, which ensures that previous implementation choices do not inappropriately constrain future development.
