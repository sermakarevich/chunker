# big-o-notation-algorithmic-complexity

**Parent:** [[content/L1/software-architecture-and-programming-practices|software-architecture-and-programming-practices]] — Robust software development requires structured adaptation using principles like DRY, orthogonality, and reversibility, while strictly adhering to decoupled APIs and using Interfaces/Protocols to manage global data. Developers must prevent 'programming by coincidence' by testing assumptions, planning the application, and actively listening to nonconscious 'lizard brain' signals when encountering resistance or complexity, which may require prototyping. Algorithm performance must be analyzed using Big-O notation, which estimates resource consumption (time/memory) by identifying growth rates like $O(N^2)$ or $O(N)$ for predictive scaling.

Algorithmic analysis is crucial for understanding how an application's performance will scale with increased input size. Instead of focusing only on implementation, developers must estimate the resources that algorithms consume, including time, processor cycles, and memory. The technique used for this estimation is called Big-O notation, which is a mathematical method for dealing with approximations. When stating that a routine sorts records in $O(N^2)$ time, for example, the Big-O notation means that the worst-case time taken will vary as the square of the input size ($N$). The notation places an upper bound on the value of the resource being measured. For instance, if a function takes $O(N 	ext{ log } N)$ time, the upper bound of the time will not grow faster than $N$ multiplied by the logarithm of $N$. When analyzing algorithms, the convention is to remove all low-order terms and constant multiplying factors because the highest-order term will dominate the value as $N$ increases. This means that while one algorithm may be 1,000 times faster than another, the Big-O notation alone cannot convey actual numbers for time or memory; it only indicates how these values will change as the input changes.

Different complexities relate to different growth rates. For example, if a routine takes one second to process 100 records, an algorithm with $O(N^2)$ complexity will take about four times longer to process 200 records. If the code complexity is $O(N 	ext{ log } N)$, processing 1,000 records might take about ten seconds. By contrast, an algorithm with $O(N)$ complexity will show a linear increase to ten seconds when processing 1,000 records, while an $O(N^2)$ algorithm would take 100 seconds. Using an exponential algorithm, however, might result in a runtime that takes years to complete.

Figure 3 illustrates common Big-O notations and the corresponding running times: $O(1)$ (Constant time, such as accessing an element in an array or running simple statements); $O(	ext{log } N)$ (Logarithmic time, like binary search); $O(N)$ (Linear time, like sequential search); $O(N 	ext{ log } N)$ (Worse than linear, average runtime for algorithms like quicksort or heapsort); $O(N^2)$ (Square time, typical for selection and insertion sorts); $O(N^3)$ (Cubic time, often seen in matrix multiplication); and $O(2^N)$ (Exponential time, used for problems like the traveling salesman problem or set partitioning). The Big-O notation can represent any resource used by an algorithm, such as time or memory consumption.

Developers can use Big-O notation to predict how an algorithm will perform when scaling up. For example, a routine that takes one second to process 100 records and is $O(N^2)$ will take 100 seconds to process 1,000 records (a 10x increase in input yields a 100x increase in time). Conversely, an algorithm with $O(N)$ complexity will only increase the runtime ten-fold when increasing the input to 1,000 records.
