# property-based-testing-strategies

**Parent:** [[content/L1/adaptive-software-development-principles|adaptive-software-development-principles]] — Effective software development demands an incremental, adaptive approach, integrating principles like DRY, orthogonality, and reversibility, alongside mandatory thorough testing via Design by Contract and property-based testing; furthermore, testing provides critical communication benefits, as evidenced by a developer who realized the primary value of testing stems from the act of considering its impact on the code structure.

Software testing is a fundamental part of the programming process, not a function reserved for other departments or staff. Therefore, developers should plan on thoroughly testing all written software, as all software written will be tested, either by the development team or by eventual users. Performing preliminary consideration can significantly minimize maintenance costs and reduce help-desk calls.

Developers have three main choices regarding testing: Test First, Test During, or Test Never. Among these, Test First, which includes Test-Driven Design, is typically the most effective choice because it ensures that testing is integrated into the development process. When Test First is not convenient, Test During coding serves as a viable fallback, where the developer writes some code, refines it, writes tests for it, and then proceeds to the next section. The least effective choice is 'Test Later,' which is functionally equivalent to 'Test Never.'

For a culture of testing to succeed, all tests must pass consistently. Developers should treat test code with the same care applied to production code, ensuring that it remains decoupled, clean, and robust. Developers must avoid relying on unreliable things, such as the absolute position of widgets in a Graphical User Interface (GUI) system, exact timestamps in a server log, or the precise wording of error messages, because testing for these aspects will create fragile tests. Instead, developers should shift to having the computer, which does not share human preconceptions, automate the testing process.

In the context of Design by Contract (DBC), code modules operate with formal contracts, ensuring that the code performs only what it claims to do. DBC guarantees that when input is provided, the module will make certain guarantees about the outputs it produces. Additionally, code invariants describe properties that must remain true about a piece of state when that state passes through a function. By incorporating these invariants and contracts—which are collectively referred to as properties—developers can automate testing using property-based testing.

Property-based testing allows developers to validate assumptions by defining properties that must hold true for all possible inputs, rather than writing specific examples. For example, when testing a sorting function, a developer can establish the properties that the sorted list must have the same number of elements as the original list (length is an invariant) and that no element in the result can be greater than the element that follows it. Languages often have dedicated frameworks for this purpose. The process of property-based testing involves using a strategy to generate a wide array of random inputs, such as running a test case one hundred times with a different random list each time, simulating the development of numerous individual tests. For instance, when processing an order, a property test could assert that if items are taken from the warehouse, the total quantity removed plus the quantity currently in the warehouse must equal the original stock level. If this property fails, the automated test successfully locates a faulty assumption, such as realizing that the initial `in_stock` function only checked for at least one item being present, while the system should have ensured there were enough units to complete the order. This method is powerful because the automated test can fail due to a fault in a core assumption, even if the bug was not directly related to the feature being tested.

Property-based tests are powerful because the developer establishes rules for generating inputs and defines assertions for validating output, allowing the test suite to autonomously discover failures. This process can fail in various ways: the test may pass, an assertion may fail, or the code may fail entirely because it cannot handle the inputs provided by the test generator.
