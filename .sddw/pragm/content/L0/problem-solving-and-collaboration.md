# problem-solving-and-collaboration

**Parent:** [[content/L1/software-design-processes-and-architecture|software-design-processes-and-architecture]] — The context details resilient software development practices, emphasizing adherence to DRY, orthogonality, and reversibility, while detailing processes for problem-solving, such as techniques for identifying real constraints, taking breaks, and using 'Rubber Ducking.' It also extensively covers collaborative models like Pair and Mob Programming, noting that team structure is governed by Conway's Law, and advises on iterative design using a continuous feedback loop.

When facing an intractable problem, developers should first enumerate all possible avenues, explaining why a certain path cannot be taken, before dismissing any idea, regardless of how unusable or stupid it sounds. To solve a puzzle, individuals must recognize both the placed constraints and the available degrees of freedom, as the solution often resides within those freedoms. The critical task is not to think outside the box, but to identify the real constraints—to find the box itself.

When facing a complex problem, developers should categorize and prioritize the constraints. Following a method similar to how woodworkers begin a project by cutting the longest pieces first, developers should identify the most restrictive constraints first and then fit the remaining constraints within those limitations.

If a developer is working on a problem that feels unnecessarily difficult or 'impossible,' taking a break or switching to a different task is beneficial. While the conscious brain is aware of the problem, the amazing associative neural net lurking below consciousness can provide the answer when the developer deliberately distracts themselves. If distraction is not possible, the next best approach is to seek external explanation; asking a colleague questions such as: Why is the developer solving this problem? What benefit does solving the problem provide? Are the existing problems related to edge cases that can be eliminated? Is there a simpler, related problem that can be solved? This process is an example of 'Rubber Ducking' in practice.

Louis Pasteur observed that 'In the fields of observation, chance only favors the prepared mind,' emphasizing that for problem-solving 'eureka' moments, the nonconscious brain requires sufficient raw material from prior experiences. A practical way to feed the brain is to give it feedback on what works and what does not work during daily work, such as using an Engineering Daybook. Additionally, when designing code, developers must recognize the need to challenge preconceived notions and evaluate whether the apparent constraints are real, hard-and-fast rules. The core principle is to determine the real, limiting constraints of the problem.

Finally, when working with 'impossible' projects, the most successful approach is to work closely with users; the users should be considered part of the development team. This collaborative process embodies what might be called pair programming or mob programming: one person types the code while one or more other team members comment, ponder, and solve problems together. This collaborative method surpasses endless meetings, memos, and overly legalistic documentation, representing a powerful way of developing software.
