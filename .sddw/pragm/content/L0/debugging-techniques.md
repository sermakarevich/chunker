# debugging-techniques

**Parent:** [[content/L1/software-development-principles-and-debugging-protocols|software-development-principles-and-debugging-protocols]] — Software development mandates using the DRY principle, abstracting third-party APIs, and employing iterative planning techniques like PERT and the 'Eating the Elephant Technique'; debugging requires reproducing the bug using single commands, employing advanced techniques like binary chopping (for stacks, datasets, and versions), and utilizing tracing statements or the Rubber Ducking method to understand pre-current state behavior.

When debugging, developers must move beyond simply understanding the state of the program at the current moment. Debuggers can only show how the program arrived at the current state by displaying a stack trace. However, the stack trace typically fails to explain what the program was doing before reaching the current call chain, especially when dealing with event-based systems.

Tracing statements provide necessary diagnostic messages printed to the screen or a file (e.g., messages like “got here” or “value of x = 2”). Although tracing is a primitive technique compared to modern IDE-style debuggers, it remains highly effective for diagnosing several classes of errors that debuggers cannot detect. Tracing is particularly invaluable in systems where time is a critical factor, such as concurrent processes, real-time systems, and event-based applications. Developers can utilize tracing statements to drill down into the code by adding tracing statements as they descend the call tree. To maximize utility, developers should ensure that trace messages adhere to a regular, consistent format, which allows for automatic parsing. For instance, if a developer needs to track a resource leak, such as unbalanced file opens and closes, the developer could trace every open and every close to a log file. Processing this log file using text processing tools or shell commands makes it easy to identify the original opening that caused the leak.

Another useful technique for diagnosing problems is the Rubber Ducking method: the developer explains the problem to another person who watches the screen. The other person does not need to speak; merely observing the explanation, step by step, often causes the problem to become apparent. When explaining the problem to another person, the developer must explicitly state assumptions that the developer assumes when reviewing the code alone. Verbalizing these assumptions can yield new insight into the problem. If a person is not available, a substitute such as a rubber duck, teddy bear, potted plant, or similar inanimate object can be used.

When debugging, the code under review is often a mixture of application code written by the developer and other team members, third-party products (including databases, connectivity layers, web frameworks, specialized communications, or algorithms), and the platform environment (which includes the operating system, system libraries, and compilers). While it is technically possible that a bug resides in the OS, the compiler, or a third-party product, the developer should not begin by assuming this. It is far more likely that the bug resides within the application code currently under development. The general rule is that it is more productive to assume that the application code incorrectly calls into a library than to assume that the library itself is fundamentally broken. Even if the problem is actually with a third party, the developer must still eliminate their own application code before submitting a bug report. For example, on one project, a senior engineer initially believed the `select` system call was broken on a Unix system, spending weeks writing workarounds; however, the problem was discovered and corrected in minutes after the engineer was forced to read the `select` documentation. Developers should use the phrase “select is broken” as a gentle reminder whenever a developer starts blaming the system for a fault that is likely to be the developer’s own mistake.

**General Debugging Tips and Considerations:**

*   **Assumption Checking:** If a system stops working after a developer 'changed only one thing,' that single change was likely responsible, either directly or indirectly, regardless of how farfetched the connection seems. Sometimes, the causal change occurs outside the developer's control: new versions of the OS, compiler, database, or any other third-party software can corrupt previously correct code, leading to new bugs. Bugs for which a developer had a workaround may become fixed, thus breaking the workaround. Since APIs and functionalities change, the developer must always retest the system under these new conditions, and developers should monitor the schedule carefully when considering an upgrade. 
*   **The Element of Surprise:** When a developer is surprised by a bug, the developer must reevaluate every assumption the developer holds. For example, if the developer is working on a discount calculation algorithm that was previously believed to be 'bulletproof,' the developer must test all boundary conditions.
