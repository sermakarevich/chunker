# pragmatic-starter-kit

**Parent:** [[content/L1/software-development-best-practices-2|software-development-best-practices-2]] — Modern software development requires continuous, adaptive practices built on architectural rigor (DRY, orthogonality, abstraction) and process automation (version control, full automation). Testing must be comprehensive, utilizing unit tests for single modules, integration tests for subsystems, validation for user needs, and performance testing for scale. Developers must employ Test-Driven Development (TDD) and Property-Based Testing (PBT) to enhance code quality and resilience, while always adhering to strong ethical principles regarding user safety and system transparency.

The core of modern software development, much like the evolution from Model-T Ford cars to modern vehicles, demands processes that are automatic and repeatable. Manually following detailed procedures—whether for building, testing, deploying, or completing project paperwork—is insufficient because these processes must maintain consistency and repeatability. The 'Pragmatic Starter Kit' identifies three critical, interrelated topics that support every project regardless of methodology, language, or technology stack: Version Control, Regression Testing, and Full Automation. 

**Driving with Version Control:** All necessary project components must be stored under version control. Version control is crucial because it allows build machines and/or clusters to be created on demand as ephemeral spot instances in the cloud. Furthermore, deployment configuration must also be stored under version control, allowing the release to production process to be handled automatically. At the project level, version control drives both the build and release process: build, test, and deployment must be triggered by commits or pushes to the version control system and built in a container within the cloud. Releasing to staging or production should be specified using a tag in the version control system, making releases a low-ceremony, continuous delivery process not tied to a single build machine or developer's machine.

**Routinely and Continuously Testing:** Pragmatic Programmers are proactive in finding bugs, rather than just subconsciously knowing where the code will break. Testing should be treated like fishing with a net, using small, fine unit tests to find minor bugs and large, coarse integration tests to catch major defects. The goal is to write unit tests—in fact, a good project may have more test code than production code—because the time spent producing test code is cheaper in the long run and significantly increases the chance of producing a product with close to zero defects. Passing all tests provides a high degree of confidence that a piece of code is fully functional.

**The Scope of Testing:**
*   **Unit Testing:** A unit test involves writing code that exercises a single module. Unit testing is foundational; all modules must pass their individual unit tests before developers can proceed to integration.
*   **Integration Testing:** Integration testing verifies that major project subsystems work well together. If good contracts are established and components are well tested, integration issues can be easily detected; otherwise, integration is often the single largest source of bugs in a system. Integration testing is considered an extension of unit testing, focusing on how entire subsystems honor their contracts.
*   **Validation and Verification:** When an executable user interface or prototype is available, developers must validate whether the product meets user needs and functional requirements, asking if the solution is what the user actually needs, not just what the user stated they wanted. A bug-free system that answers the wrong question is not useful. Developers must be mindful of end-user access patterns and how these patterns differ from developer test data.
*   **Performance Testing:** Performance, or stress testing, should check if the software meets performance requirements under real-world conditions, specifically checking the expected number of users, connections, or transactions per second and assessing scalability.

**Testing the Tests:** Since developers cannot write perfect software, they also cannot write perfect test software. Developers must proactively test their test suites by deliberately introducing bugs into the source tree and verifying that the existing tests successfully report the failure. At a high level, developers can use tools like Netflix’s Chaos Monkey to disrupt services and test the application's resilience.

**Testing Thoroughly:** There is no definitive way to know if the codebase has been tested thoroughly enough, as it is impossible to test every possible state. While tools can provide code coverage analysis, developers should recognize that state coverage, which identifies all potential logical states the program may have, is significantly more important than simply executing every line of code. For example, a function accepting two integers (each 0-999) has 1,000,000 logical states, but simply running the three lines of code only proves that the line of code works, not that all logical states are covered.
