# technical-exercise-answers

**Parent:** [[content/L1/adaptive-software-design-principles|adaptive-software-design-principles]] — Modern system design requires adherence to the DRY principle and prioritizing orthogonality and reversibility, while mandating the use of structured patterns like the blackboard system or protocols/interfaces. Development process must adopt an abstract, user-story-based approach, utilizing methods like TDD and PBT for validation, and ensuring security compliance via least privilege and third-party authentication.

The following technical exercises provide solutions or conceptual answers to development problems. For instance, **Answer 1 (from exercise 1)** suggests that the class `Split2` is superior because the class is more orthogonal. This design concentrates on the specific task of splitting lines and ignores extraneous details, which allows the code to be easily developed and flexible, enabling `Split2` to successfully split lines originating from a file, from another routine, or directly passed via the environment.

**Answer 2 (from exercise 2)** examines coupling and orthogonality in different language paradigms. In Object-Oriented (OO) languages, features such as multiple inheritance, exceptions, operator overloading, and parent-method overriding (via subclassing) present opportunities to increase coupling in nonobvious ways. While general coupling between code and data is desirable (a concept called cohesion), poorly focused classes can still lead to problematic interfaces. Conversely, Functional languages encourage writing many small, decoupled functions that transform data, which is beneficial in theory and often in practice. However, if careful attention is not paid, a change to the data format required by one function can cause failures later in the data transformation stream. Languages with strong type systems can mitigate these data format coupling issues. 

**Answer 3 (from exercise 3)** suggests a low-tech method for brainstorming or planning: drawing cartoons (e.g., a car, a phone, and a house) with markers on a whiteboard, even if the art is simple stick-figure outlines. The team can use Post-it notes to describe the contents of target pages, refining the drawings and placements of the Post-it notes throughout the meeting process. 

**Answer 4 (from exercise 4)** details the creation of a parser table-driven language structure designed for extensibility. Each entry in the parser table must contain three pieces of information: the command letter (`cmd`), a flag indicating if an argument is required (`hasArg`), and the name of the routine to call (`func`). The core program reads a line from standard input (`stdin`), uses a function `findCommand` to look up the command, checks if an argument is needed and available (using `getArg`), and then calls the appropriate `cmd->func` handler. 

**Answer 5 (from exercise 5)** notes that the current problem has already been solved in a previous exercise, specifically by writing an interpreter for the external language, which includes the internal interpreter functionality, exemplified by the `doXxx` functions. 

**Answer 6 (from exercise 6)** provides two definitions for specifying time using Backus-Naur Form (BNF). The initial definition covers three structures: `time` (allowing `hour ampm`, `hour : minute ampm`, or `hour : minute`); `ampm` (defining `am` or `pm`); `hour` (allowing `digit` or `digit digit`); and `minute` (allowing `digit digit` or `digit`). A superior definition accounts for that hours must range from 00 to 23 and minutes must range from 00 to 59, defining more precise rules for the structure of `hour` and `minute` fields.
