# tracer-bullets-software-development

**Parent:** [[content/L1/flexible-architecture-and-tracer-development|flexible-architecture-and-tracer-development]] — The comprehensive development strategy requires establishing flexible architecture through abstraction (hiding third-party APIs) and componentization (breaking code into parts), while adopting 'tracer development'—an iterative process where skeleton code is built end-to-end to provide immediate, visible feedback, contrasting with conventional 'dead reckoning' methods.

Topic 12: Tracer Bullets

When developing software, developers often use the metaphor of hitting a target. Instead of literally firing a device at a shooting range, the concept illustrates the complexity of aiming for a goal in a shifting world. When aiming with certain devices, developers only receive one chance to aim and then must see whether the attempt hits the bullseye. However, using the concept of tracer bullets, which are loaded alongside regular ammunition, provides a better approach.

Tracer bullets ignite phosphorus when fired, leaving a visible pyrotechnic trail from the gun to the object that the bullet hits. If the tracer rounds are hitting the intended target, the regular bullets are also hitting it. Soldiers use these tracer rounds to refine aim, providing pragmatic, real-time feedback under actual conditions. This same principle applies to software development projects, especially when building something that has never existed before. Developers can use the term tracer bullet development to visually illustrate the need for immediate feedback under actual conditions, particularly when the goal is moving.

When building a complex system, developers may face many unknowns: the users may provide vague requirements because they have never seen a system like the one being built; developers may be using algorithms, techniques, languages, or libraries that are unfamiliar; and projects are long enough that the working environment will surely change before the project is complete. The classic, traditional response is to specify the system thoroughly by producing reams of paper detailing every requirement, thereby constraining the environment. This involves firing the project using dead reckoning—a single large calculation executed up front, followed by hoping the effort succeeds.

Pragmatic Programmers, however, prefer using the software equivalent of tracer bullets. Tracer bullets work because they operate in the same environment and under the same constraints as the real bullets, allowing the gunner to get immediate feedback and offering a relatively cheap solution. Therefore, to achieve this effect in code, developers must seek a method that quickly, visibly, and repeatedly moves the project from the requirement to some aspect of the final system. Developers should look for the important requirements—the ones that define the system—and focus on areas where the team has doubts or perceives the biggest risks. These identified areas should be the first areas that the team codes.

**Using Tracer Bullets to Find the Target**

Given the current complexity of projects, especially those with numerous external dependencies and tools, tracer bullets become even more important. The very first tracer bullet action is simply for the team to create the project, add a “hello world!” example, and ensure that the code compiles and runs. Subsequently, the team must identify areas of uncertainty within the overall application and add the skeleton code needed to make the system function.

The development process used in a complex client-server database marketing project—which required the ability to specify and execute temporal queries across various relational and specialized databases, using a client UI in random language A with libraries in a different language—demonstrated the value of tracer code. The initial build involved developing the front-end framework, the libraries for representing the queries, and the structure for converting a stored query into a database-specific query. By putting these parts together, the team only started by submitting a query that listed all the rows in a table, but this initial test proved that the UI could communicate with the libraries, the libraries could serialize and unserialize a query, and the server could generate SQL from the result. In the following months, the team gradually developed this basic structure by augmenting each component of the tracer code in parallel. For example, when the UI added a new query type, the library grew, and the SQL generation became more sophisticated.

Tracer code is not disposable; the team writes it for permanence. It includes all the error checking, structuring, documentation, and self-checking features found in any piece of production code, but the code itself is not fully functional. Once the team achieves an end-to-end connection among all components of the system, the team can assess how close the system is to the target and adjust the approach if necessary. Once the system is on target, adding full functionality becomes simple.

Tracer development supports the idea that a project is never finished, making it an incremental approach. This contrasts with the conventional, heavy engineering approach, where code is divided into modules, coded in isolation, combined into subassemblies, and only then presented to the user and tested as a complete application.

The tracer code approach offers several advantages:

*   **Users see something working early:** If the developers successfully communicate the progress, users understand they are viewing an immature system and will be excited by visible progress. Furthermore, users can contribute to the project as it progresses, increasing their buy-in. Users can also help the developers determine how close the project is to the target during each iteration.
*   **Developers build a structure to work in:** By embodying all the end-to-end interactions of the application in code, the team avoids having to create all the necessary logic
