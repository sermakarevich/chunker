# agile-feedback-and-design-principles

**Parent:** [[content/L1/software-design-processes-and-architecture|software-design-processes-and-architecture]] — The context details resilient software development practices, emphasizing adherence to DRY, orthogonality, and reversibility, while detailing processes for problem-solving, such as techniques for identifying real constraints, taking breaks, and using 'Rubber Ducking.' It also extensively covers collaborative models like Pair and Mob Programming, noting that team structure is governed by Conway's Law, and advises on iterative design using a continuous feedback loop.

The value of input data is not to dictate actions; instead, the input data guides the process of self-determination. These decisions are inherently contextual, depending on numerous factors, including who the developer is, the characteristics of the development team, the specific application, the tooling, the company, the customer, and the external environment. Since no fixed, static plan can survive this uncertainty, the focus must shift to managing uncertainty by gathering and acting on continuous feedback. Following this agile approach involves three recursive steps: first, determining the current state; second, making the smallest possible, meaningful step toward the desired state; and third, evaluating the outcome and correcting any introduced failures. These steps must be applied repeatedly at every level of development, even when making seemingly trivial decisions, as the continuous feedback loop can significantly improve the system's design. For example, a developer, initially naming a variable `user` when retrieving an account owner, might recognize that a more specific name like `owner` is appropriate. However, applying the feedback loop further suggests that the development team does not need the entire account owner object; instead, finding only the `email` address directly is the superior design, which successfully reduces coupling between the code and the account handling logic. The feedback loop also applies at the highest project level: successful work often occurs when the development team takes a single step in response to client requirements, only to realize that the intended development was unnecessary, or that the optimal solution does not require software at all. Furthermore, the team must apply the feedback loop to review its processes and operational effectiveness; a team that fails to continuously experiment with its process is not an agile team. This process of continuous improvement is what drives good design, because the measure of design is how easy the resulting system is to change. To make the feedback loop efficient, the mechanism for fixing what is broken (Step 3) must be as painless as possible. Therefore, practicing good design makes changes easy and adjustments possible at every level, which is the essence of agility.

Project-level success requires moving beyond individual coding philosophy and considering larger issues. Developing concurrent applications requires managing mutable external resources (such as databases, file systems, and service APIs), which introduces potential race conditions due to shared state. To solve this, developers must ensure resource access is atomic. While using semaphores or mutual exclusion works, a more robust solution is to centralize resource access by creating a method within the resource object itself (e.g., `get_pie_if_available()` within the `display_case` class). This method must use a `try...ensure` block to guarantee the semaphore is always released, even if an exception occurs during processing. Furthermore, when dealing with multiple necessary resources, such as needing both pie and ice cream for an apple pie à la mode, the developer must treat the combination as a single, new resource. The code handling the acquisition and release of all component resources should be moved into a dedicated module to manage this atomically. On the topic of team structure, a pragmatic team is defined as a small, stable entity, ideally containing fewer than 10-12 members.
