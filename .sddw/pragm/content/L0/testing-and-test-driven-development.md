# testing-and-test-driven-development

**Parent:** [[content/L1/software-architecture-scaling-design|software-architecture-scaling-design]] — Modern software development requires adherence to foundational principles like DRY and orthogonality, using techniques like Abstraction and Componentization to ensure reversibility and decoupling. Algorithmic performance is estimated using Big-O notation to predict scaling issues (e.g., $O(N^2)$ vs. $O(N 	ext{ log } N)$), and the development process relies on Test-Driven Development (TDD) and iterative Refactoring to maintain quality.

Testing is fundamentally more than just finding bugs; the major benefits of testing occur when developers think about and write the tests, not when developers run them. This practice influences the design of the code. For instance, when a developer writes a function to query a database to return a list of people who watch more than 10 videos a week on a site, initial consideration of tests guides the development process. By first imagining how the function would be tested, developers realize the need for passing the database instance into the function (rather than using a global database) and identify the need to pass the field name used for qualification as a parameter. This initial thinking about tests leads to two discoveries and changes the API of the method. A core benefit of testing is that it forces developers to view the code from the perspective of a client, not the author. Furthermore, considering tests helps developers clarify nebulous understandings of the requirements, forcing developers to think about boundary conditions and error conditions before writing code. Test-Driven Development (TDD) is a structured programming cycle that operationalizes this idea. The basic TDD cycle involves five steps: 1) Deciding on a small piece of functionality to add; 2) Writing a test that will pass once the functionality is implemented; 3) Running all tests and verifying that the only failure is the one just written; 4) Writing the smallest amount of code needed to get the test to pass, then verifying that the tests run cleanly; and 5) Refactoring the code—improving either the test or the function—while ensuring the tests still pass. This cycle should remain very short, ideally only taking minutes. While TDD is highly beneficial for beginners, developers must avoid becoming overly reliant on the practice. Over-reliance can manifest in spending excessive time ensuring 100% test coverage or writing many redundant tests, such as creating a failing test that simply references a class name (which fails) followed by an empty class definition (which passes), resulting in a test that does nothing. Designers must also avoid adopting extreme development methodologies: the top-down approach assumes the developer can express the entire requirement upfront, which is often not possible; conversely, the bottom-up approach assumes that building an initial layer of abstractions will eventually lead to a single top-level solution, but this fails because the developer does not know where the layers are heading. Instead, developers should aim to build end-to-end.
