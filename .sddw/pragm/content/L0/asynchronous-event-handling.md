# asynchronous-event-handling

**Parent:** [[content/L1/advanced-software-architecture-principles|advanced-software-architecture-principles]] — The context details multiple rigorous software development methodologies, including applying DRY, achieving reversibility by abstracting external services (like databases) to simply provide persistence, and managing global resources via controlled APIs. Project planning mandates using techniques like PERT (optimistic, most likely, pessimistic time estimates) and 'Eating the Elephant' (iterative slicing), while continuous guidance requires 'Tracer Bullet' methods and utilizing advanced event architectures like Pubsub, State Machines, and Stream processing (e.g., using `rxjs` for parallel stream processing).

To manage asynchronous events, developers have three primary architectural patterns: state machines, the Observer pattern, and Publish/Subscribe (Pubsub). 

**State Machines:**
Developers can implement a state machine by defining a transition table, where each entry is a two-element list containing the next state and the name of an action to take. For instance, a pattern shown in `event/strings_fsm.rb` might define transitions for a look-for-string mechanism: The `look_for_string` state uses the quote mark (`"`) to transition to the `:in_string` state, while the `:default` transition keeps the state at `look_for_string` and uses the `:ignore` action. In the `:in_string` state, the quote mark (`"`) transitions back to `look_for_string` with the `:finish_current_string` action, the backslash (`\`) transitions to the `:copy_next_char` state with the `:add_current_to_string` action, and the `:default` transition keeps the state at `:in_string` and uses the `:add_current_to_string` action. The system processes events (input characters) by looping through the events, triggering a transition using the defined table, and running the associated action name. The result of each transition is both a new state and the name of an action, allowing developers to select specific code to run before looping to the next event. While the initial example code is basic, developers could expand it using anonymous functions or function pointers for actions, or wrapping the state machine in a separate class with its own state.

State machines are valuable for complex workflows; for example, when signing up a user on an app, the system state can be kept in external storage and used to drive a state machine through numerous transitions such as entering details, validating email, or agreeing to specific legislative warnings.

**The Observer Pattern:**
This pattern establishes a source of events, called the observable, and maintains a list of interested clients, known as observers. An observer registers its interest with the observable, typically by passing a reference to a function to be called. When the event occurs, the observable iterates through its list of observers and calls the function passed to it, passing the event as a parameter. An example module demonstrates this in Ruby: the `Terminator` module stores callbacks in a `CALLBACKS` array and, when `Terminator.exit(exit_status)` is called, it iterates through all registered callbacks, calling each one and passing the `exit_status` (e.g., 99) as an argument. Since developers simply push a function reference onto a list and call those functions upon an event, this pattern is considered simple and often proves unnecessary to use a library.

However, the Observer pattern introduces coupling because every observer must explicitly register with the observable, and calling the callbacks synchronously within the observable can create performance bottlenecks.

**Publish/Subscribe (Pubsub):**
Pubsub generalizes the Observer pattern by solving the problems of coupling and performance. In the Pubsub model, the system consists of publishers and subscribers connected via channels. The implementation detail of these channels is hidden from the developer's code, regardless of whether the channels are implemented as a library, a process, or a distributed infrastructure. Each channel has a name. Subscribers register interest in one or more of these named channels, and publishers write events to them. Unlike the Observer pattern, the communication between the publisher and subscriber occurs outside the developer's code and can be potentially asynchronous.

Pubsub offers significant decoupling, making it easier for developers to add and replace code, potentially while the application is running, without altering existing code. This pattern is considered a great technology for decoupling the handling of asynchronous events. However, the Pubsub model is still essentially a message passing system, and its inherent difficulty in tracing the relationship between a publisher and its subscribers means that developers cannot easily see which subscribers are involved with a particular message.

**Reactive Programming, Streams, and Events:**
Reactive programming, which developers may encounter when using tools like spreadsheets, treats data changes as reactive; updating one cell formula referencing another cell causes the first to update. When dealing with events, streams provide a mechanism to treat events as a continuous collection of data. Streams are useful because developers can treat them like any other collection: manipulating, combining, filtering, and performing other data-related operations. Furthermore, streams can be asynchronous, allowing the developer's code to respond to events as they arrive.

The current de facto baseline for reactive event handling is defined on http://reactivex.io. For example, the `rxjs` library in JavaScript demonstrates stream manipulation by zipping two streams: a static list of five animal names and a dynamic `Observable.interval(500)` timer. The resulting `combined` stream only emits a value every 500ms, containing one item from the first input stream and one item from the second. When multiple streams are involved, developers can use operators like `mergeMap` to process separate streams in parallel. An example fetching user information from the public REST interface https://reqres.in/api/users/ demonstrates this, where fetching data for users with IDs 3, 2, and 1 resulted in three separate streams that were processed in parallel; the results showed the three requests being processed asynchronously, with the user ID 2 returning in 82ms, followed by the other two within 50ms and 51ms, respectively.
