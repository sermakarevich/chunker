# data-transformation-pipelines

**Parent:** [[content/L1/program-as-transformation-pipeline|program-as-transformation-pipeline]] — Programming is best viewed as a data transformation pipeline (like a Unix pipe: `find . -type f | xargs wc -l | sort -n | tail -5`), which minimizes coupling by passing data continuously through functional steps; to achieve polymorphism and structure robust code, developers should favor interfaces/protocols, delegation, and mixins/traits over inheritance to avoid architectural coupling.

Thinking of code as a series of transformations offers a highly liberating approach to programming. The process involves transforming data from an initial input into a final output, much like an industrial assembly line. For example, a sequence of transformations to find the five longest files in a directory tree using a Unix pipeline demonstrates this: `find . -type f` lists all files in or below the current directory (`.`) to standard output; `xargs wc -l` reads lines from standard input and passes them as arguments to the `wc -l` program, which counts the lines in each file argument and writes each result as "count filename" to standard output; `sort -n` sorts standard input assuming each line starts with a number (`-n`), writing the result to standard output; and `tail -5` reads standard input and writes only the last five lines to standard output. When analyzing the data flow, the original requirement of finding the "top 5 files in terms of lines" thus becomes a sequence of transformations: directory name $\to$ list of files $\to$ list with line numbers $\to$ sorted list $\to$ highest five + total $\to$ highest five. This transformation sequence is analogous to an industrial assembly line, where raw data enters one end and the finished information product exits the other.

When designing code, developers should adopt the view of programs as processes that transform inputs into outputs. This perspective allows developers to structure the code as a series of chained transformations. When implementing such transformations, developers can write the code as a series of assignments: `const content = File.read(file_name);` followed by `const lines = find_matching_lines(content, pattern)` and then `const result = truncate_lines(lines)`. 

While some languages, referred to here as Language X, may not support built-in pipelines, the concept of pipelines, which has been around for a long time, can be applied through function chaining. This process involves taking an initial input, passing its output to the next transformation, and so on. For instance, the program function `anagrams_in(word)` demonstrates chaining four transformations: `word` $\to$ `all_subsets_longer_than_three_characters()` $\to$ `as_unique_signatures()` $\to$ `find_in_dictionary()` $\to$ `group_by_length()`. The entire process is a simple chain of transformations required to meet the problem's requirement. This technique makes the code highly readable, as each step takes input from the previous transformation and passes its output to the next.

This transformational model provides architectural advantages over Object-Oriented (OO) programming. Instead of data residing in separate pools and changing states across multiple objects, the model treats data as a continuous flow, like a mighty river. This data flow becomes a peer to functionality, forming a pipeline: a sequence of code $\to$ data $\to$ code $\to$ data…. This approach significantly reduces coupling because a function can be reused anywhere its parameters match the output of some other function.

In scenarios involving error handling, the convention is to never pass raw values between transformations. Instead, developers must wrap the values in a data structure (or type) which also indicates if the contained value is valid. For example, in Haskell, this wrapper is called `Maybe`, and in F# and Scala it is called `Option`. Generally, there are two ways to write the code: developers can handle checking for errors inside the individual transformations or outside them. When using languages that support pattern matching in function calls, such as Elixir, the language can be structured to automatically handle errors, where an error at any point in the pipeline immediately becomes the final value of the pipeline. Otherwise, developers must use specialized helper functions (bind functions) to defer running subsequent pipeline functions until the preceding steps have successfully completed, ensuring that the entire pipeline accurately reflects the data flow and error handling.
