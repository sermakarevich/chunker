# least-privilege-security-practices

**Parent:** [[content/L1/software-development-architecture-best-practices|software-development-architecture-best-practices]] — Successful software development requires an adaptive, context-specific approach guided by continuous feedback loops, utilizing principles like DRY, orthogonality, and least privilege to manage architectural complexity. Teams must facilitate communication instantly (using 'tracer bullets' to ensure all skills traverse the whole system end-to-end), automate all processes, and adhere to stringent security practices such as encrypting all sensitive data, utilizing third-party authentication, and never committing credentials alongside source code.

The principle of least privilege dictates that developers must only use the minimum necessary permission level for the shortest amount of time. Developers must never automatically acquire the highest permission level, such as root or Administrator. If the highest level of privilege is necessary, developers must take it, complete the minimum necessary work, and then relinquish the privilege quickly to reduce risk. Jerome Saltzer formalized this principle in 1974, advising that every program and every privileged user of the system should operate using the minimum amount of privilege necessary to complete the job. This concept extends beyond operating system privilege levels; applications must implement different, finely grained levels of access, allowing sensitive resources to be partitioned into categories and granting individual users permissions for only certain categories. Implementing this technique minimizes the attack surface, thereby restricting the scope of attack vectors both by time and by privilege level, ensuring that less is more. Regarding initial settings, the default settings for applications or users on a site should always be set to the most secure possible values. While secure defaults might not be the most convenient or user-friendly options, leaving the trade-off decisions between security and convenience to individual users is acceptable. For example, the default for password entry should be to hide the password by replacing each character with an asterisk, which is a sensible default in public or crowded settings; however, if some users require seeing the password (for instance, due to accessibility), that remains a reasonable user choice in settings where there is little risk of onlookers. Developers must encrypt sensitive data, including personally identifiable information, financial data, passwords, and other credentials, regardless of whether the data resides in a database or an external file. If this data is exposed, encryption provides an additional layer of safety. When managing project resources, developers must store all necessary project components under version control, with one major exception: never committing secrets, API keys, SSH keys, encryption passwords, or other credentials alongside the source code in version control. Instead, developers must manage these keys and secrets separately, typically via config files or environment variables during the build and deployment process. When considering password security, developers must understand that common security practices often conflict with common sense. The National Institute of Standards and Technology (NIST) provides specific recommendations that developers must follow: (1) Do not restrict password length to less than 64 characters; NIST recommends 256 as a suitable maximum length. (2) Do not truncate a user's chosen password. (3) Accept all printable ASCII characters, space, and Unicode; developers should not restrict special characters like `[]();&%$#` or `/`, because these special characters are unlikely to compromise the system. (4) Do not provide password hints to unauthenticated users, nor should the system prompt for specific types of information (such as “what was the name of your first pet?”). (5) Do not disable the paste function in the browser or password managers, because such actions fail to increase security and instead motivate users to create simpler, shorter passwords that are much easier to compromise; both the US NIST and UK National Cyber Security Centre require verifiers to allow paste functionality for this reason. (6) Do not impose arbitrary composition rules, such as mandating a specific mix of upper and lower case, numerics, or special characters, or prohibiting repeating characters. (7) Do not arbitrarily require users to change their passwords after a set length of time, restricting such changes only when there is a valid reason, such as after a security breach. Ultimately, developers should encourage long, random passwords with a high degree of entropy, because imposing artificial constraints limits entropy and encourages poor password habits, leaving user accounts vulnerable to takeover. Concerning system maintenance, developers must apply security patches quickly. Waiting to update a system, even if it causes a temporary application breakage, is a poor choice because the system remains vulnerable to a known exploit. This principle applies to every network-connected device, including phones, cars, appliances, personal laptops, developer machines, build machines, production servers, and cloud images, because historical data shows that the largest data breaches were caused by systems that lagged behind on updates. Finally, when dealing with cryptography, developers must remember that common sense often fails when dealing with cryptography. The most important rule is never to perform crypto work personally. Even simple elements like passwords can be managed improperly. When engaging with cryptography, developers must only rely on reliable resources: well-vetted, thoroughly examined, well-maintained, frequently updated, and preferably open-source libraries and frameworks. For authentication, the pragmatic approach is to use a third-party authentication provider, such as services offered by email, phone, or social media providers. These external providers spend their time maintaining their systems' security, making them experts in a field the local developer is not.
