# managing-temporal-coupling

**Parent:** [[content/L1/software-architecture-decoupling-concurrency|software-architecture-decoupling-concurrency]] — Software architecture demands decoupling through protocols/interfaces, delegation, and mixins to avoid coupling and manage ephemeral state, while reliable development requires adhering to DRY, utilizing external configuration services for dynamic parameters, and managing complex workflows using concurrency analysis via activity diagrams and semaphores.

When considering software architecture, developers must recognize that time is a critical design element, not just the time scheduled for project completion. The temporal dimension includes two key aspects: concurrency (multiple things happening at the same time) and ordering (the relative sequence of events). Most programmers naturally approach design linearly, following a sequential pattern—for example, requiring Method A to always execute before Method B, or demanding that only one report runs at a time. This linear thinking, however, leads to temporal coupling, which limits flexibility and realism. For instance, requiring the user to wait for the screen to redraw before a button click is received is a temporal dependency.

To achieve greater flexibility and reliability, software must allow for concurrency and think about decoupling time or order dependencies. Doing this can improve workflow analysis, architecture, design, and deployment.

**Analyzing Workflow for Concurrency:**
Developers can model and analyze application workflows to determine what events can happen simultaneously and what events must happen in a strict order. Activity diagrams are a useful notation for this purpose. An activity diagram consists of rounded boxes representing actions. The arrow leaving an action leads to either another sequential action (which can start after the first action completes) or to a thick line called a synchronization bar. Once all actions leading into a synchronization bar are complete, the system can proceed along any exiting arrows. Any action that has no incoming arrows can be started at any time.

For instance, designing software for a robotic piña colada maker involves several steps: (1) Open blender, (2) Open piña colada mix, (3) Put mix in blender, (4) Measure 1/2 cup white rum, (5) Pour in rum, (6) Add 2 cups of ice, (7) Get glasses, and (8) Get pink umbrellas. If a bartender strictly followed these steps sequentially, the process would be inefficient. However, by analyzing the workflow using an activity diagram, developers realize that the top-level tasks (1, 2, 4, 10, and 11) can all happen concurrently at the start, and subsequently, tasks 3, 5, and 6 can happen in parallel.

**Concurrency in Documentation Generation:**
This principle also applies to internal processes, such as generating a book written in plain text. The finished text is fed through a pipeline of processors—some look for specific constructs (like bibliography citations, index entries, or special markup for tips), while others operate on the entire document. Since many processors require external information (reading or writing files, or piping through external programs), these relatively slow tasks provide an opportunity to exploit concurrency: each step in the pipeline can execute concurrently, reading from the previous step and writing to the next.

Additionally, some parts of the process, such as the conversion of mathematical formulae, are processor-intensive. Since every equation is independent of the others, the process can leverage parallelism by converting each formula in its own parallel process and collecting the results as they become available, significantly speeding up the build process on multicore machines.

**Identifying Opportunities for Concurrency and Parallelism:**
Activity diagrams identify potential areas for concurrency, but they do not determine if these parallel activities are worth exploiting. For example, in the piña colada maker scenario, the bartender would need five hands to run all potential initial tasks at once. The design process, therefore, involves finding activities that take time, but not critical processing time. Activities like querying a database, accessing an external service, or waiting for user input typically stall a program, and identifying these delays represents an opportunity to perform more productive work than the CPU equivalent of idling.

It is vital to distinguish between concurrency and parallelism: concurrency is a software mechanism, while parallelism is a hardware concern. If multiple processors are available, the optimal strategy is to split independent work into chunks, process each chunk in parallel, and then combine the results. A compiler for the Elixir language exemplifies this, where the compiler splits a project into modules and compiles each one in parallel; if one module depends on another, the compilation pauses until the dependency is compiled, leading to speedy execution that utilizes all available cores.

Finally, after identifying potential areas for concurrency and parallelism, the complex challenge is safely implementing these structures.
