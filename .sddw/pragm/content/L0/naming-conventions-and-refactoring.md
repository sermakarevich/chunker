# naming-conventions-and-refactoring

**Parent:** [[content/L1/adaptive-software-development-principles-2|adaptive-software-development-principles-2]] — Software development demands treating naming elements (functions, variables, etc.) with extreme precision, prioritizing specificity and congruence with user domain terminology; architecturally, systems must adhere to DRY, orthogonality, and reversibility, managing global data through controlled APIs and interfaces; regarding requirements, developers must adopt a role of discovery, viewing initial statements as invitations for exploration rather than final specifications, favoring abstract user stories and iterative feedback (prototypes) over detailed, monolithic documents.

When developers encounter a function or method with an overly generic name, developers should rename it to express all of the actions the function or method truly performs. Renaming the function or method to be more specific makes it an easier target for refactoring. In examples, developers should consider using more specific names, such as 'buyer' instead of the more traditional and generic 'user.' Teams should also ask whether the names used in the system are congruent with user terminology from the domain. If the names are not congruent with user terms, developers must understand why this causes a Stroop-effect style cognitive dissonance for the team. Furthermore, developers must analyze whether the names in the system are difficult to change, and then determine what actions can be taken to fix that particular broken window of naming inconsistency.
