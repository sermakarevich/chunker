# version-control-systems

**Parent:** [[content/L1/development-knowledge-management-techniques|development-knowledge-management-techniques]] — Effective software development requires developers to adopt DRY principles, leverage abstract architectural components, and prioritize continuous feedback loops (like the 'Tracer Bullet' approach). Crucial technical practices include mandating the use of plain text for persistence, achieving editor fluency through dedicated keyboard training, and utilizing advanced Version Control Systems (VCS) for collaboration and state recovery, while debugging requires rigorous, user-centric testing to find root causes, not just visible symptoms.

Version Control Systems (VCS) provide essential mechanisms for collaboration and maintaining the history of software development. Unlike single-user features like the 'undo' key, which recovers mistakes made over a short period, a VCS acts as a project-wide time machine, allowing users to recover code and documentation from previous versions, even those dating back weeks.

Understanding a VCS requires viewing it both as a historical repository of changes and as a central platform for team collaboration, deployment pipelines, and issue tracking.

**Warning against Poor Practices:**

1. **Shared Directories:** Sharing project source files across a network, whether internally or via cloud storage, is unviable. Teams using this method risk constantly corrupting each other’s work, losing changes, and breaking builds. Users must instead employ version control.
2. **Network/Cloud Repositories:** Storing the main repository on a network or cloud drive, while seeming like a compromise for accessibility and off-site backup, is even riskier. Because the VCS uses a set of interacting files and directories, if two instances simultaneously make changes, the overall state can become corrupted, potentially causing significant damage.

**Core Functions of a VCS:**

*   **Tracking Changes:** VCS systems meticulously keep track of every change made in source code and documentation. With proper configuration, users can always revert to previous versions of the software. Furthermore, a VCS can track detailed information, answering questions such as: Who made changes in this line of code? What is the difference between the current version and last week’s? How many lines of code were changed in this release? Which files are modified most often? This data is invaluable for bug-tracking, audits, performance analysis, and quality assessment.
*   **Releases:** A VCS identifies software releases, ensuring that users can always regenerate a specific release independently of any changes that occur later.
*   **Collaboration:** VCS systems permit two or more users to work concurrently on the same set of files, even making concurrent changes to the same file. The system then manages the merging of these changes when the files are submitted back to the repository. Despite initial apparent risk, this methodology proves effective on projects of all sizes.

**Best Practices and Advanced Features:**

1. **Universal Use:** Version control must be used for absolutely everything, even when working on a single-person, one-week 'throw-away' prototype. This includes not just source code, but also documentation, phone number lists, vendor memos, makefiles, build and release procedures, and simple shell scripts used for tidying up log files. Ideally, all day-to-day work is secured in a repository.
2. **Branching:** VCS systems offer a powerful feature: the ability to isolate development into specialized branches. Users can create a branch at any point in the project’s history, and all work done within that branch remains isolated from all other branches. Later, the user can merge the completed branch back into another target branch, adding all the contained changes. Since multiple people can work on a single branch, branches effectively function as little clone projects. The isolation benefit is clear: if one developer develops feature A in one branch, and a teammate works on feature B in a separate branch, the work does not interfere with each other.
3. **Project Hub Integration:** While VCS systems can be decentralized (with developers cooperating peer-to-peer), establishing a central repository is highly recommended. Using a central host allows developers to take advantage of integrations that streamline project flow. Key features to look for in a hosting provider include: good security and access control, an intuitive user interface (UI), the ability to perform operations via the command line (for automation), automated builds and tests, good support for branch merging (sometimes called pull requests), issue management (ideally integrated into commits and merges for metrics tracking), and good reporting (e.g., a Kanban board-like display for pending issues and tasks). Many teams configure their VCS so that a push to a specific branch automatically builds the system, runs the tests, and, if successful, deploys the new code into production. The ability to roll back is always guaranteed, even with complex pipelines.

**Challenges and Preparation:**

When implementing VCS, developers must proactively learn the necessary commands for recovery, rather than waiting for a disaster. Developers should also treat the entire laptop environment—including installed applications, system configuration, and user preferences (dotfiles)—as items that need to be captured in a VCS (hosted off the laptop) to facilitate disaster recovery. The best approach involves consciously exploring all features of the current VCS and hosting provider, such as feature branches, pull/merge requests, continuous integration (CI), build pipelines, and continuous deployment (CD). Furthermore, developers should use version control for non-project artifacts as well.
