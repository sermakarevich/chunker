# power-editing-editor-fluency

**Parent:** [[content/L1/development-knowledge-management-techniques|development-knowledge-management-techniques]] — Effective software development requires developers to adopt DRY principles, leverage abstract architectural components, and prioritize continuous feedback loops (like the 'Tracer Bullet' approach). Crucial technical practices include mandating the use of plain text for persistence, achieving editor fluency through dedicated keyboard training, and utilizing advanced Version Control Systems (VCS) for collaboration and state recovery, while debugging requires rigorous, user-centric testing to find root causes, not just visible symptoms.

Power Editing: Editors are an extension of the programmer’s hand, making the ability to manipulate text effortlessly crucial, as text constitutes the basic raw material for programming. Previously, the book recommended using a single editor for all purposes—including writing code, documentation, memos, and system administration—but the current advice suggests that developers are free to use as many editors as desired, provided the developer works toward fluency in each one. Achieving editor fluency significantly benefits development; specifically, making editing only 4% more efficient while editing for 20 hours a week can gain an additional week of work over the course of a year. More importantly, fluency eliminates the need to think about the mechanics of editing, significantly reducing the gap between a thought and its appearance in the editor buffer, which allows programming thoughts to flow more naturally.

Fluency involves several skills: when editing text, a fluent developer must be able to move and make selections by character, word, line, and paragraph. When editing code, a fluent developer must be able to move by various syntactic units (such as matching delimiters, functions, or modules), reindent code after changes, comment and uncomment blocks of code with a single command, and perform undo and redo changes. Furthermore, a fluent developer must be able to split the editor window into multiple panels, navigate to a particular line number, sort selected lines, search for both strings and regular expressions while repeating previous searches, temporarily create multiple cursors based on a selection or pattern match, edit the text at each cursor in parallel, and display compilation errors or run the current project’s tests.

To move toward fluency, the recommended approach is not mastering every command in a particular powerful editor, but rather learning the commands that make the developer’s life easier. The recipe involves a simple process: first, developers must observe themselves while editing. Every time a repetitive action is performed, the developer should adopt the habit of thinking, “There must be a better way,” and then finding an alternative method. Once a new, useful feature is discovered, the developer must solidify that feature into muscle memory through conscious repetition, ideally using it many times a day, which usually results in the feature being used without conscious thought after about a week.

When growing the editor skills, most powerful code editors are built around a basic core that can be augmented through extensions. If a developer encounters an apparent limitation in the editor, the developer should search for an extension that fills the need; often, another user has already developed a solution. Developers can take this further by digging into the editor’s extension language to automate repetitive tasks, which often only requires a line or two of code. In advanced cases, developers might write a full-blown extension and, if successful, publish it so that other people with similar needs can benefit.

Finally, developers should tackle specific technical challenges: first, disabling autorepeat and learning keyboard key sequences to move, select, and delete by characters, words, lines, and blocks, despite the difficulty. Second, developers should spend one week editing using only the keyboard, eliminating the use of a mouse or trackpad, forcing the learning of previously unused keyboard shortcuts. Third, developers should look for opportunities to integrate tools, such as previewing a final layout (a PDF file) directly alongside the original text within the editor buffer. If no suitable plugin or extension exists, the recommendation is for the developer to build the necessary feature themselves.
