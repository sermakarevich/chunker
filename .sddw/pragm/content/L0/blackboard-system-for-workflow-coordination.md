# blackboard-system-for-workflow-coordination

**Parent:** [[content/L1/adaptive-software-design-principles|adaptive-software-design-principles]] — Modern system design requires adherence to the DRY principle and prioritizing orthogonality and reversibility, while mandating the use of structured patterns like the blackboard system or protocols/interfaces. Development process must adopt an abstract, user-story-based approach, utilizing methods like TDD and PBT for validation, and ensuring security compliance via least privilege and third-party authentication.

When designing systems that involve processing multiple chunks of work, the choice of architectural pattern depends on whether the workflow requires feedback. If simple scheduling of a workload among parallel processes is sufficient, a shared work queue may be adequate. However, if the results of one processed chunk affect other chunks—for example, in machine vision applications or complex 3D image-warp transforms—a blackboard system is recommended. For group calendaring, a blackboard system is also suitable; one can post scheduled meetings and availability to the board because the system features autonomous entities, feedback from decisions is crucial, and participants may enter and exit the system. Developers should consider partitioning this blackboard system based on the search criteria: junior staff might only require information about the immediate office, human resources might only need data for English-speaking offices worldwide, and a CEO might require access to all data ('the whole enchilada'). Regarding data formats, the system can ignore formats or languages that are not understood. The system only needs to understand different formats for offices that hold meetings with each other, preventing the need to expose all participants to the full transitive closure of all possible data formats. This targeted understanding minimizes coupling where it is necessary, thereby preventing artificial constraints. A network monitoring tool can be designed similarly to a mortgage or loan application program. In this case, user-submitted trouble reports and automatically generated statistics both post data to the blackboard. A human or software agent can analyze the blackboard to diagnose network failures; for example, two errors on a line might only be cosmic rays, but 20,000 errors indicate a hardware problem. This model mirrors how detectives solve a murder mystery, allowing multiple entities to analyze the blackboard and contribute ideas toward solving network problems.
