# design-by-contract

**Parent:** [[content/L1/advanced-software-development-methodologies|advanced-software-development-methodologies]] — Professional development necessitates adopting principles like DRY, Design by Contract (DBC) which formalizes function expectations into Preconditions, Postconditions, and Class Invariants, and continuous process refinement using methods like PERT and the Tracer Bullet approach.

The principle of Design by Contract (DBC) is a concept that applies the formal structure of contracts—which define mutual rights, responsibilities, and repercussions for failure—to ensure the correctness of software modules. Initially developed by Bertrand Meyer for the Eiffel language, DBC is a technique focused on documenting and agreeing upon the expected behavior of software modules to guarantee program correctness. A correct program is defined as one that performs no more and no less than it claims to do. The heart of DBC involves documenting and verifying this claim. Every function or method in a software system operates with specific expectations regarding the state of the world before execution, and it must be able to make a statement about the state of the world upon completion. Meyer formalizes these expectations and claims into three categories:

1. **Preconditions:** These define what must be true for a routine to be successfully called, representing the routine's necessary requirements. The caller is responsible for ensuring that the preconditions are met, and a routine should never be called when its preconditions are violated.
2. **Postconditions:** These describe what the routine guarantees to achieve, representing the state of the world when the routine concludes. The existence of a postcondition guarantees that the routine will conclude; consequently, infinite loops are forbidden. 
3. **Class Invariants:** A class ensures that this specific condition remains true from the perspective of any calling code. Although a class's internal processing may temporarily violate the invariant, the invariant must be true by the time the routine exits and control returns to the caller. It is important to note that a class cannot provide unrestricted write-access to any data member involved in maintaining the invariant.

The contract between a routine and any potential caller can thus be summarized: if the caller meets all the routine’s preconditions, the routine guarantees that all postconditions and invariants will be true upon completion. If either the calling party or the routine fails to uphold the terms of the contract, a previously agreed-upon remedy is invoked, such as raising an exception or terminating the program. Failure to adhere to the contract is defined as a bug and should never happen; thus, preconditions should not be used for things like validating user input.

Modern languages, such as Clojure, provide strong support for these concepts, including pre- and post-conditions and comprehensive instrumentation via specs. For instance, a banking function designed to process a deposit using simple pre- and post-conditions might check if the deposited amount is greater than 0.00 and if the account is open; the function then guarantees that the returned value (the new transaction) can be found among the account's transactions. If the deposit amount is negative, the system will raise a runtime exception, indicating a failed assertion. Similarly, if the specified account is not open and valid, an exception will be raised. Other languages, like Elixir, use guard clauses to dispatch function calls based on available bodies; for example, if calling the `accept_deposit` function with an amount greater than 100,000 triggers additional steps, calling it with an amount less than or equal to zero results in a `FunctionClauseError`, indicating that no function clause matches the provided arguments.
