# event-streams-and-data-transformation

**Parent:** [[content/L1/advanced-software-architecture-principles|advanced-software-architecture-principles]] — The context details multiple rigorous software development methodologies, including applying DRY, achieving reversibility by abstracting external services (like databases) to simply provide persistence, and managing global resources via controlled APIs. Project planning mandates using techniques like PERT (optimistic, most likely, pessimistic time estimates) and 'Eating the Elephant' (iterative slicing), while continuous guidance requires 'Tracer Bullet' methods and utilizing advanced event architectures like Pubsub, State Machines, and Stream processing (e.g., using `rxjs` for parallel stream processing).

Event streams treat data changes as continuous collections of data, providing a mechanism to handle asynchronous events. This approach is valuable because it allows developers to manipulate, combine, filter, and perform other data-related operations on the event data, just as they would with any other standard collection. Furthermore, event streams can be asynchronous, enabling the developer's code to respond to events as they arrive. 

Regarding the management of event streams, the following points are relevant: 

*   **Event Streams and User IDs:** In previous examples, the list of user IDs in the observable users was static. However, the user ID list does not have to be static. Instead, the developer can generate an observable event containing the user ID every time a user's session is created and utilize this dynamic observable event stream instead of a static one. This development pattern is a powerful abstraction because developers no longer need to manage time as a separate concern. Event streams unify synchronous and asynchronous processing through a common, convenient Application Programming Interface (API). 
*   **Ubiquity of Events:** Events are encountered in many contexts. Some examples of obvious events include a button click or a timer expiring. Other examples include less obvious events, such as a user logging in or a line in a file matching a specific pattern. Regardless of the source, code that is structured around events is generally more responsive and better decoupled than code structured in a linear manner.

**Programmatic Focus: Data Transformation:**

Deming noted that if a person cannot describe what they are doing as a process, the person does not know what they are doing. All programs transform data by converting an input into an output. When designing, developers often focus primarily on classes, modules, data structures, algorithms, languages, and frameworks. This focus on code often misses the primary point: developers need to return to thinking of programs as processes that transform inputs into outputs. When adopting this perspective, many previous concerns evaporate. The resulting structure becomes clearer, error handling becomes more consistent, and coupling decreases significantly.

To illustrate this principle, one can use a series of transformations, such as finding the five longest files in a directory tree using a Unix pipeline: `find . -type f | xargs wc -l | sort -n | tail -5`. This sequence involves several distinct transformations: `find . -type f` lists all files in or below the current directory (`.`) to standard output; `xargs wc -l` reads lines from standard input and passes them as arguments to the `wc -l` program, which counts the lines in each file argument and writes each result as "count filename" to standard output; `sort -n` sorts standard input assuming each line starts with a number (`-n`), writing the result to standard output; and `tail -5` reads standard input and writes only the last five lines to standard output.

When analyzing the data flow, the original requirement of finding the "top 5 files in terms of lines" becomes a sequence of transformations: directory name $\to$ list of files $\to$ list with line numbers $\to$ sorted list $\to$ highest five + total $\to$ highest five. This process is analogous to an industrial assembly line, where raw data enters one end and the finished information product exits the other.

**Finding Transformations (Top-Down Approach):**

When analyzing a problem, the easiest way to identify necessary transformations is to begin by defining the required inputs and outputs. This establishes the overall program function. For example, to create a word game website that finds all words made from a given set of letters (e.g., "lvyin"), the input is the set of letters, and the output is a list of three-letter words, four-letter words, and so on (e.g., "ivy, lin, nil, yin"). The solution relies on a dictionary that groups words by a signature, which is chosen such that all words containing the same letters share the same signature. The simplest signature function is generating the sorted list of letters in the word. The anagram finder therefore decomposes into four transformations: 1) Initial input ("ylvin"); 2) All combinations of three or more letters; 3) Signatures of those combinations; and 4) The resulting list of dictionary words matching those signatures.

Further detail is provided by looking at the first step: taking a word and creating a list of all combinations of three or more letters. This step itself consists of multiple transformations: Initial input (e.g., "vinyl") $\to$ Convert to characters (v, i, n, y, l) $\to$ Get all subsets (e.g., $\text{[]}, [v], [i], \ldots, [v,i], [v,n], [v,y], \ldots, [v,i,n], [v,i,y], \ldots, [v,n,y,l], [i,n,y,l], [v,i,n,y,l]$) $\to$ Filter for subsets longer than three characters (e.g., $[v,i,n], [v,i,y], \ldots, [i,n,y,l], [v,i,n,y,l]$) $\to$ Convert back to strings (e.g., $\text{vin}, 	ext{viy}, \ldots, 	ext{inyl}, 	ext{vinyl}$). Many functional languages support the pipeline operator (`|>`), which automatically takes the value on its left and inserts it as the first parameter of the function on its right, conceptually representing data flowing from one transformation to the next.
