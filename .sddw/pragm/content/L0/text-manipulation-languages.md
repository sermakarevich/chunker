# text-manipulation-languages

**Parent:** [[content/L1/software-development-principles-and-debugging-protocols|software-development-principles-and-debugging-protocols]] — Software development mandates using the DRY principle, abstracting third-party APIs, and employing iterative planning techniques like PERT and the 'Eating the Elephant Technique'; debugging requires reproducing the bug using single commands, employing advanced techniques like binary chopping (for stacks, datasets, and versions), and utilizing tracing statements or the Rubber Ducking method to understand pre-current state behavior.

Pragmatic Programmers manipulate text using methods analogous to how woodworkers shape wood. While previous sections discussed specialized tools, such as shells, editors, and debuggers—tools similar to a woodworker's chisels, saws, and planes, each designed for one or two specific jobs—developers sometimes need a general-purpose text manipulation tool. These text manipulation languages are to programming what routers are to woodworking; they are noisy, messy, and can feel somewhat brute force. Making mistakes with these tools can ruin entire pieces, and some people believe they have no place in the professional toolbox. However, when used correctly, text manipulation languages, like routers, can be incredibly powerful and versatile, allowing developers to quickly trim, make joints, and carve. Although these tools take time to master, they offer surprising finesse and subtlety when used properly.

Fortunately, several excellent text manipulation languages exist. Unix developers (and macOS users) often utilize the power of their command shells, enhanced with tools such as `awk` and `sed`. Programmers who prefer a more structured toolset may prefer languages such as Python or Ruby. These languages are crucial enabling technologies because they allow developers to rapidly construct utilities and prototype ideas—jobs that might otherwise take five to ten times longer using conventional programming languages. The factor of multiplying time is particularly important for experimentation; spending 30 minutes trying out a creative idea is much better than spending five hours, and spending a day automating important components is acceptable, while spending a week may not be. For instance, in the book *The Practice of Programming* (KP99), Kernighan and Pike built the same program using five different languages, noting that the Perl version was the shortest, requiring only 17 lines compared with 150 lines in C. Using Perl, a programmer can manipulate text, interact with programs, communicate over networks, drive web pages, perform arbitrary precision arithmetic, and write programs with unusual stylistic elements like Snoopy swearing.

To demonstrate the wide-ranging applicability of text manipulation languages, the Pragmatic Bookshelf itself is largely built around text manipulation. For instance, the build system uses Ruby and Rake tasks for authors, editors, layout people, and support staff to coordinate the building of PDFs and ebooks. When including and highlighting code, the developers used the DRY principle (referring to Topic 9, DRY—The Evils of Duplication) to avoid manually copying and pasting code from tested programs, which would have guaranteed forgetting to update an example if the corresponding program changed. Instead, the developers employed Ruby to run a simple script that extracts a named segment of a source file, performs syntax highlighting, and converts the result into the typesetting language used for the book. Similarly, a simple script handles the website update process by extracting the table of contents and uploading it to the book's web page, and another script extracts sections for use as samples. The inclusion of equations is handled by a Python script that converts LaTeX math markup into formatted text, and the process for index generation involves a Ruby script that collates and formats entries marked up directly in the text itself. Therefore, in a highly practical sense, the Pragmatic Bookshelf relies on text manipulation. Following the advice to keep all content in plain text means that using these languages to manipulate that text provides numerous benefits. The chapter suggests three practical exercises: 1. Writing a script to convert a directory of `.yaml` configuration files into corresponding `.json` files, such that `database.yaml` becomes `database.json` with valid JSON content. 2. Writing a script that scans all source files for `camelCase` names, following a required change from `camelCase` to `snake_case`. 3. Expanding on the second exercise by adding the ability to automatically change those variable names in one or more files, ensuring a backup of the originals is kept in case of failure.
