# software-development-architecture-process-principles

**Parent:** [[content/L3/architecture-process-knowledge-management-systematics|architecture-process-knowledge-management-systematics]] — Software development mandates rigorous adherence to DRY, orthogonality, and reversibility in architecture, while managing uncertainty using PERT, the 'Eating the Elephant Technique,' and the 'Tracer Bullet' approach. Professionals must maintain mastery of the Command Shell and prioritize plain text for knowledge persistence, coupled with disciplined debugging techniques like binary chopping and structured use of engineering daybooks.

The professional practice of software development is governed by a set of adaptable design methodologies and structured knowledge management principles, designed to mitigate the inherent risks posed by the accelerating pace of technological change. These foundational principles govern both the internal structure of technical knowledge within a system and the systematic process by which complex projects are initiated, scheduled, and maintained. 

### Foundational Architectural Principles and Knowledge Management

At the core of reliable software development is the principle of **DRY (Don't Repeat Yourself)**. This cornerstone concept mandates that every single piece of knowledge within a system must possess one, single, unambiguous, and authoritative representation. Violating DRY—by expressing the same knowledge in two or more places—introduces severe system instability, as the knowledge requires constant, routine maintenance due to varied external factors. These factors include client meeting adjustments, regulatory updates, or algorithm failures. This mandate extends beyond mere code duplication, applying to all representations of knowledge throughout the entire system, including specifications, processes, and data representations.

In the domain of data structures, DRY adherence is critical. If a `Line` class initially includes redundant fields such as `Point start`, `Point end`, and a separate `double length`, and `length` is inherently derived from the other two points, the correct practice is to eliminate the redundant `length` field entirely. Instead, the `length` must be calculated via a derived method: `double length() { return start.distanceTo(end); }`. Furthermore, even when developers optimize for performance using caching, the impact of violating DRY must be strictly localized, and all external interactions must be managed exclusively via accessor functions (e.g., `setStart`, `getStart`) to simplify future modifications and uphold Meyer’s Uniform Access principle. The concept of **orthogonality** is equally paramount, requiring the system to be composed of cooperating modules where each module functions independently of the others, thereby significantly localizing changes and reducing overall system risk and increasing productivity.

For managing interactions with external systems, centralized knowledge management is vital. For **Across Internal APIs**, developers must centralize these APIs in a single repository and utilize tools that specify the API in a neutral format. This centralization is essential because it facilitates the generation of crucial artifacts, including documentation, mock APIs, and functional tests across multiple programming languages. For **Across External APIs**, formal documentation standards such as OpenAPI should be leveraged; if these established standards are absent, the most responsible action for programmers is to proactively create and publish them. When integrating **Data Sources**, developers should maximize the use of introspection capabilities to minimize duplication. This is achieved by generating necessary containers directly from the underlying schema, rather than writing manual, redundant storage code. An **Alternative Data Structure Strategy** might involve adopting key/value data structures (like a map or dictionary) instead of fixed structures (like structs or classes), but this strategy requires a secondary, crucial defense mechanism: implementing a simple table-driven validation suite to confirm that all required data is present and correctly formatted. To maximize flexibility, developers must also adopt a mindset of **reversibility**, which involves treating critical architectural decisions—such as committing to a specific third-party vendor’s database or adopting a particular architectural pattern—as temporary drafts rather than definitive, permanent commitments.

To facilitate this architectural flexibility, two strategies are paramount: **Abstraction**, which requires proactively hiding all interactions with third-party APIs behind dedicated abstraction layers, and **Componentization**, which involves systematically breaking down the code into discrete, manageable components, irrespective of the system's eventual deployment scale.

### Formalizing Correctness: Design by Contract (DBC) and Defect Management

Complementing architectural principles is the formal process of ensuring module correctness through **Design by Contract (DBC)**. Originating from Bertrand Meyer and initially designed for the Eiffel language, DBC is a technique that applies the structure of formal contracts—which define mutual rights, responsibilities, and repercussions for failure—to ensure that a software module performs no more and no less than it explicitly claims to do. The contract formalized by DBC is systematically broken down into three distinct, verifiable categories:

1. **Preconditions:** These define what *must* be true for a routine to be called successfully, serving as the routine's necessary input requirements. The calling party bears the responsibility of ensuring these preconditions are met, and a routine must never be invoked when its preconditions are violated. Violation by either party is classified as a bug.
2. **Postconditions:** These describe what the routine *guarantees* to achieve, representing the expected state of the world immediately upon the routine's conclusion. The existence of a postcondition guarantees that the routine will indeed conclude; consequently, infinite loops are strictly forbidden.
3. **Class Invariants:** This mandates that a specific condition must remain true from the perspective of any calling code. While a class’s internal processing may temporarily breach this invariant, it must be true by the time the routine exits and control returns to the caller. Importantly, a class cannot provide unrestricted write-access to any data member involved in maintaining the invariant.

By adhering to these contracts, the routine guarantees that all postconditions and invariants hold true upon completion, provided the caller meets all preconditions. If failure occurs, a pre-agreed-upon remedy, such as raising an exception or terminating the program, is invoked. Modern languages provide strong support for these concepts; for instance, Clojure utilizes pre- and post-conditions and comprehensive instrumentation via specs. For detailed runtime checks, developers should adopt **Assertive Programming**, which dictates that rather than relying on self-deception regarding impossible outcomes (like a negative parameter or a null result), they must explicitly incorporate code to check for every possibility. For example, if a result should never be null, the developer must use `assert (result != null)`. A more comprehensive check in Java might use `assert result != null && result.size() > 0 :`.

Furthermore, recognizing that software defects are inherent is critical. Detectable failures include receiving a `nil` value, finding an empty list, or encountering a missing key in a hash. Other errors, such as network or filesystem failures leading to corrupted data, or complex logic errors that only manifest after millions of instructions, underscore the necessity that every case/switch statement must include a default clause.

### Project Planning, Uncertainty, and Iterative Cycles

Managing highly uncertain or shifting requirements demands specialized project scheduling techniques. Best practice involves moving away from the reliance on a single time estimate and instead utilizing a range of scenarios. The **Program Evaluation Review Technique (PERT)**, originally developed for the U.S. Navy's Polaris submarine project, is a structured methodology requiring the estimation of three specific time values for every single task: an optimistic time, a most likely time, and a pessimistic time. These tasks must be systematically organized into a dependency network to accurately determine the overall project likelihood and prevent the common error of merely padding a single number with uncertainty. 

Complementing this structured approach is the **'Eating the Elephant Technique,'** which acknowledges that true planning must be inherently iterative. This cyclical process requires repeatedly completing thin slices of functionality through a mandated four-step cycle: 1) Analyze requirements; 2) Analyze risks (with an explicit mandate to prioritize the riskiest items earlier); 3) Design, implement, and integrate the functionality; and 4) Validate the functionality with the users. Critically, after each cycle, the team must refine the initial guess regarding the number of required iterations, thereby increasing confidence and improving the schedule estimate with every piece of completed functionality.

When the ultimate system goal remains highly nebulous, the **'Tracer Bullet'** approach is utilized. Instead of attempting 'dead reckoning' by specifying the system exhaustively upfront, this method demands continuous, real-time feedback. It requires identifying and focusing all development effort on the most critical requirements—those possessing the highest risks or deepest doubts—in order to move the project rapidly from abstract conceptual requirements toward a tangible, working aspect of the final system. It is crucial to distinguish tracer code from disposable prototypes. While prototyping exists purely to generate lessons learned and measure risk exposure (analogous to building a model out of balsa wood), tracer code is written with the explicit intention of permanence. When implemented correctly, the initial foundation—such as integrating a front-end framework, necessary libraries, and a server's capability to generate SQL—can be built incrementally, even if the overall functionality later undergoes substantial expansion.

Finally, promoting **Early User Visibility and Buy-in** is a vital part of process discipline. By consistently communicating progress, users are encouraged to view the system as immature—a state that typically generates excitement—and can actively contribute to its development, which greatly increases their sense of ownership. The core framework must be meticulously kept intact, regardless of the complexity of the functionality layered on top, distinguishing the disciplined effort from a simple prototype.

### Knowledge Capture and Tooling

Beyond architecture and process, successful professional development requires sophisticated knowledge capture and tooling. Developers must commit to continuous learning, striving for mastery of fundamental core tools, even when working within powerful Integrated Development Environments (IDEs). The **Command Shell** is posited as the primary, superior workspace, surpassing exclusive reliance on GUIs and IDEs because the latter are inherently restricted to their defined capabilities. Proficiency in the shell allows developers to combine tools using pipes in complex ways and automate routine tasks, dramatically improving productivity. Core practices include customizing the shell with aliases (e.g., `alias apt-up='sudo apt-get update && sudo apt-get upgrade'`) and configuring functional prompts.

Regarding data format, the most critical choice is the use of **plain text**. Plain text is strongly preferred over binary formats because of its superior portability and resilience. Its main advantage lies in its ability to be self-describing and independent of the application that originally created it, thereby preventing an 'artificial divorce' between the data and its context. Although structured formats like JSON, HTML, and YAML are utilized, they are plain text, providing 'insurance against obsolescence.' This standardization enables developers to manipulate knowledge manually and programmatically using virtually every available utility, pioneering the Unix Philosophy through small, sharp, line-oriented, plain-text tools, which allows file comparison utilities like `diff` to instantly reveal changes. On a personal level, authors recommend utilizing **engineering daybooks** (or journals) as a technique for professional knowledge capture. This physical, paper-based method, initially observed when a colleague worked with engineers at a small computer manufacturer, serves as a centralized repository for recordings, including accomplished work, new knowledge acquired, sketches of ideas, and readings from meters. This daybook method provides the benefit of **Reliability over Memory** compared to digital files or wikis, due to the unique cognitive process associated with the act of writing.
