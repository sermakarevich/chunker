# advanced-system-architecture-and-process

**Parent:** [[content/L3/holistic-software-development-philosophy|holistic-software-development-philosophy]] — The comprehensive approach to modern software development requires adhering to architectural principles like DRY, orthogonality, and decoupling using Interfaces/Protocols. Processes must employ Test-Driven Development (TDD), Property-Based Testing (PBT), and Design by Contract (DBC) to ensure robustness, while algorithmic complexity is managed using Big-O notation. Security requires the principle of least privilege, encryption of PII, and utilizing third-party providers for authentication, and tackling complex issues requires externalizing thought through techniques like Pair/Mob Programming or the rubber ducking method.

The professional practice of modern software development demands a highly adaptive and resilient approach, transforming mere coding into a continuous cycle of structured adaptation and rigorous critical thinking across architectural design, process flow, and knowledge management. This deep requirement starts with establishing foundational principles of code structure and data management, aiming to mitigate risks associated with technology's accelerating pace of change. A core principle is **DRY (Don't Repeat Yourself)**, which mandates that every piece of knowledge within a system must possess a single, unambiguous, and authoritative representation. Violating DRY poses significant instability risks because knowledge is susceptible to constant maintenance across multiple specifications, processes, and programs due to external pressures, such as client meetings, regulatory shifts, or algorithmic failures. For example, if a `Line` class redundantly defines fields like `Point start`, `Point end`, and `double length`, the redundancy of `length` must be eliminated. Instead, the length must be derived and calculated dynamically through a method, such as `double length() { return start.distanceTo(end); }`. This principle extends into data handling: when dealing with **Data Sources**, developers are advised to utilize introspection capabilities to minimize duplication by generating necessary containers directly from the existing schema, rather than relying on manual storage code. When employing an **Alternative Data Structure Strategy**, developers might opt for key/value data structures (like a map or dictionary) over fixed structures (structs or classes). Critically, this advanced method mandates a secondary defense: the implementation of a simple table-driven validation suite to guarantee that all required data is present and correctly formatted. Furthermore, to maintain data integrity and strict adherence to best practices, like Meyer’s Uniform Access principle, the impact of temporary optimizations, such as caching, must be strictly localized, and all external interactions must always be managed through dedicated accessor functions (e.g., `setStart`, `getStart`).

Building upon this foundation, system architecture must prioritize **orthogonality**, ensuring that the system is composed of cooperating modules where each component functions independently of the others. This dramatically increases productivity by localizing changes and substantially reducing systemic risk. Developers must cultivate a mindset of **reversibility**, treating critical decisions—such as committing to a specific third-party vendor’s database or adopting a specific architectural pattern—as temporary drafts rather than permanent, irreversible commitments. To maximize flexibility and maintain adaptability, three structured techniques guide the developer: 1) **Abstraction:** proactively hiding all interactions with external third-party APIs behind dedicated abstraction layers. 2) **Componentization:** systematically breaking down the code into discrete, manageable components, irrespective of the deployment scale. 3) **Guidance:** ensuring the code remains perpetually adaptable to change, adhering to the principle that there are no final decisions in development. 

This adaptive methodology necessitates adherence to **Decoupling**, which means maintaining separation between distinct concepts through sound design principles. Coupling is highly problematic because it links together things that must inevitably change in parallel, making maintenance exceptionally difficult. Developers must treat mutable external resources—including databases, file systems, and service APIs—as constituting global data. Since any change in the implementation of global data can couple the code across the entire system, any resource deemed important enough to be global must be enclosed within an API that the developers fully control. Key techniques to achieve this separation include **Interfaces and Protocols** (or traits), which specify abstract sets of required behaviors. For example, a `Drivable` protocol might mandate `double getSpeed()` and `void stop()`, while a `Locatable` protocol might require consistency across multiple components. This abstract approach allows different components—such as a `Car` and a `Phone`—to be treated generically and managed together in a list of items adhering to the same protocol.

When managing concurrent operations in a shared resource environment, developers must employ advanced architectural patterns. While many languages provide library support for mutual exclusion using constructs like mutexes, monitors, or semaphores, some, such as Rust, enforce concurrency support at the language level by enforcing data ownership, ensuring only a single variable or parameter can reference mutable data at any given moment. For mutable state shared across processes, the **Actor Model** provides a powerful alternative. An actor is defined as an independent virtual processor that maintains its own local, private state and processes messages from its mailbox sequentially. Upon receiving a message, the actor processes it fully, and it can only process one message at a time. While processing, the actor has the capability to create other actors, send messages to known actors, and establish a new state. Processors are a more generalized virtual processor, often handled by the operating system, but here, the specialized process type is constrained by convention to behave exactly like an actor.

Core Principles of Actors include three foundational rules: 1) **Distributed Authority:** No single entity controls the entire system; no single point schedules the sequence of events or orchestrates the transfer of information from raw data to the final output. 2) **Encapsulation of State:** All system state must reside either within incoming messages or within the local, private state of each individual actor. Messages can only be examined by the recipient, and the local state remains inaccessible to external entities. 3) **Unidirectional Messaging:** All message transfers are strictly unidirectional; actors cannot reply directly. If a response is needed, the sending actor must embed its own mailbox address within the message itself. Actors execute concurrently, asynchronously, and crucially, they share no state, regardless of whether physical processors are available or the runtime system handles context switching.

An excellent illustration is the Diner Scenario, involving the `customer`, `waiter`, and `pie case` actors. The external system initiates the process by sending a 'hungry for pie' message to the `customer`. The customer responds by dispatching an 'order' message to the waiter. The waiter then instructs the `pie case` to perform a 'get slice' operation. If successful, the pie case sends a 'put on table' message to the customer and a 'add to order' message to the waiter. If the pie case detects zero slices (checking `state.slices.length` is zero), it sends an 'error' message back to the waiter, who subsequently sends a 'no pie left' message to the customer, completing the negative flow. The implementation utilizes a wrapper structure where keys map to incoming message types and values hold the corresponding function.

Alternative advanced coordination mechanisms include **Blackboard Systems**. These systems allow multiple independent agents (or actors) to solve problems without needing direct knowledge of each other. Agents simply monitor a central 'blackboard' for new information and contribute their own findings. Key features include: (1) Agents do not need mutual knowledge; they only monitor and post. (2) Agents can come from different disciplines and do not need to belong to the same organization. (3) They share only a single, common goal (e.g., solving the case). (4) Agents may arrive and depart, and may work varying shifts. (5) The blackboard imposes no limitations on content types (pictures, full sentences, evidence).

Historically, computer-based blackboard systems began with AI applications like David Gelernter’s Linda, which stored facts as typed tuples. Distributed versions like JavaSpaces and T Spaces allowed storing active Java objects, retrievable by using partial field matching (via templates and wildcards) or by subtype. In large-scale applications, such as processing loan applications, the blackboard approach elegantly solves multiple complex difficulties: 1) the inherently complex and constantly changing legal requirements from federal, state, and local governments; 2) the asynchronous and variable arrival order of data (e.g., credit checks vs. immediate name/address); 3) potential data gathering by distributed people or automated systems; 4) data dependency (e.g., title search requires proof of ownership); and 5) the realization that new arriving data may trigger new policies and raise new questions. Combining a blackboard with a rules engine that posts its output back ensures the order of data arrival is irrelevant, and feedback loops are easily managed.

Messaging systems like Kafka and NATS function similarly to blackboards and can support multiple actors. While using the actor, blackboard, and/or microservice approach eliminates a class of potential concurrency problems, it introduces the cost of architectures being significantly harder to reason about because the action is indirect. Therefore, developers must proactively maintain a central repository of message formats and/or APIs, ideally one capable of generating code and comprehensive documentation. Furthermore, good tooling is necessary to trace messages and facts as they move through the system, such as propagating a unique trace ID to all involved actors to enable the reconstruction of events from log files. This complexity, though high, is mitigated by the system's increased granularity, allowing individual components to be updated without replacing the entire system.

Beyond architecture, continuous process refinement requires continuous vigilance. Developers must never engage in **programming by coincidence**, which relies on luck rather than demonstrable understanding. This failure mode is particularly dangerous when code passes limited testing but fails later due to overlooked underlying mechanisms. Developers must be wary of relying on 'accidents of implementation,' such as a routine designed for a bad input data that happens to work, or sequentially calling GUI rendering routines (like `paint()`, `invalidate()`, `validate()`, `revalidate()`, and `repaint()`) that merely appear to work now. Even temporal issues, like the 'only off by one' time zone discrepancy encountered in projects handling hardware data collection units, can accumulate into fundamental flaws.

To counteract coincidence, developers must practice deliberate programming. This involves adhering to principles like **Design by Contract** to eliminate misunderstandings, especially when creating a well-specified contract for other developers to call. When writing code for external use, developers must ensure good modularization and utilize small, well-documented interfaces. Conversely, when calling routines, developers must rely only on documented behavior. If documentation is insufficient, the developer must explicitly document the assumed behavior. Furthermore, developers must guard against 'phantom patterns'—the human tendency to see causes and patterns in mere coincidence (like lottery numbers). Testing that passes on one machine but fails on another may simply indicate environmental differences or coincidence.

For robust, reliable code, several guidelines must be followed: 1) **Prioritize the necessary:** The code must be explainable in detail to a junior programmer. 2) **Proceed from a plan:** The application must always be planned, whether using a written plan, a whiteboard, or a napkin. 3) **Rely on verifiable facts:** Developers must assume the worst if they cannot definitively determine reliability. All assumptions must be documented, using Design by Contract. 4) **Test the assumptions:** Developers must test not only the code but the assumptions; if the assertion passes, documentation improves; if it fails, a flaw is identified. 5) **Address scope and history:** Developers must not let existing code constrain future development, planning for refactoring, as all code can be replaced if inappropriate.

Furthermore, developers must manage internal nonconscious signals, drawing from concepts of instinct. The gut feeling or 'lizard brain' response is critical, signaling a deeper issue (e.g., nervousness, queasiness) when faced with difficulty (e.g., the blank page or a struggling code module). To actively listen to this inner signal, the developer should first stop coding and allow time and space for organization—such as taking a walk, having lunch, chatting, or sleeping on the issue. If these methods fail, the issue must be externalized: creating doodles about the code, explaining it to a coworker (especially a non-programmer), or talking to a rubber duck. If persistence is required, the final resort is prototyping: identifying a manageable aspect to explore (e.g., testing a new framework's data binding) or examining an algorithm on specific edge cases. If working on existing code, the developer should stash it and prototype something similar, beginning the process by writing actionable tests.

Finally, when assessing program performance, **Algorithmic analysis** uses Big-O notation, a mathematical method that establishes an upper bound on resource consumption (time, processor cycles, memory). This notation is used to predict scaling by focusing on the highest-order term. Key complexities include: $O(1)$ (Constant time, e.g., array access); $O(	ext{log } N)$ (Logarithmic time, e.g., binary search); $O(N)$ (Linear time, e.g., sequential search); $O(N 	ext{ log } N)$ (Average quicksort/heapsort); $O(N^2)$ (Square time, e.g., selection/insertion sorts); $O(N^3)$ (Cubic time, e.g., matrix multiplication); and $O(2^N)$ (Exponential time, e.g., traveling salesman problem). For example, an $O(N^2)$ routine processing 100 records for one second will take 100 seconds for 1,000 records, demonstrating the critical need for algorithmic consideration to prevent years-long runtimes from exponential complexity.
