# system-architecture-and-process-rigor

**Parent:** [[content/L3/holistic-software-development-philosophy|holistic-software-development-philosophy]] — The comprehensive approach to modern software development requires adhering to architectural principles like DRY, orthogonality, and decoupling using Interfaces/Protocols. Processes must employ Test-Driven Development (TDD), Property-Based Testing (PBT), and Design by Contract (DBC) to ensure robustness, while algorithmic complexity is managed using Big-O notation. Security requires the principle of least privilege, encryption of PII, and utilizing third-party providers for authentication, and tackling complex issues requires externalizing thought through techniques like Pair/Mob Programming or the rubber ducking method.

The professional practice of modern software development demands a holistic, adaptive, and deeply thoughtful approach that extends far beyond mere coding syntax. This systematic approach requires meticulous attention to architectural principles, robust knowledge management, process flow, and continuous critical thinking. The goal is to transform the development cycle into a continuous process of structured adaptation and rigorous critical thinking, fundamentally mitigating the risks associated with technology’s rapidly accelerating pace of change. Developers must adopt a philosophy of structured adaptation, recognizing that few things in complex systems are permanent or fully defined from the outset.

### System Architecture, Design, and Maintainability

At the architectural level, the system must adhere to several foundational principles. The core concept of **DRY (Don't Repeat Yourself)** dictates that every piece of knowledge within a system must maintain a single, unambiguous, and authoritative representation. Violations of DRY are dangerous because knowledge is inherently unstable, demanding constant maintenance across multiple specifications, processes, and programs when external changes occur—such as client meetings, regulatory shifts, or algorithm failures. For instance, if a `Line` class redundantly defines fields like `Point start`, `Point end`, and `double length`, the redundancy of `length` must be eliminated. Instead, the length must be derived and calculated using a method, such as `double length() { return start.distanceTo(end); }`. This principle applies broadly to **Data Sources**; developers should leverage introspection capabilities to minimize duplication by generating necessary containers directly from the schema, rather than writing manual storage code. Furthermore, when considering an **Alternative Data Structure Strategy**, such as employing key/value data structures (like a map or dictionary) instead of fixed structures (structs or classes), this method critically requires a secondary defense: a simple table-driven validation suite to ensure all required data is present and correctly formatted. To maintain data integrity and align with best practices like Meyer’s Uniform Access principle, temporary optimizations, such as caching, must be strictly localized, and all external interactions must always be managed through accessor functions (e.g., `setStart`, `getStart`).

To maximize flexibility and productivity, the architecture must prioritize **orthogonality**, meaning that the system is composed of cooperating modules where each function operates independently of the others. This localizes changes and significantly reduces systemic risk. Developers must cultivate a mindset of **reversibility**, viewing critical architectural decisions—such as adopting a specific architectural pattern or committing to a third-party vendor’s database—as temporary drafts rather than permanent commitments. To maximize flexibility, three structured techniques are highly recommended: 1) **Abstraction:** proactively hiding all interactions with third-party APIs behind dedicated abstraction layers; 2) **Componentization:** systematically breaking down code into discrete, manageable components, irrespective of the deployment scale; and 3) **Guidance:** ensuring the code remains always adaptable to future change, recognizing that no decisions are final.

This systemic necessity mandates strict adherence to **Decoupling**, which emphasizes keeping separate concepts separate through sound design principles, since coupling links together things that must change in parallel, thereby complicating maintenance. Developers must treat mutable external resources—including databases, file systems, and service APIs—as constituting global data. Recognizing that any change in the global data’s implementation can couple the code across the entire system, any resource deemed important enough to be global must be enveloped in an API that the developers fully control. Techniques to achieve this include **Interfaces and Protocols** (or traits), which specify abstract sets of required behaviors. For example, a `Drivable` interface might mandate `double getSpeed()` and `void stop()`, without providing implementation code. This allows components, such as a `Car` or a `Phone`, to be managed generically; if both implement a `Locatable` protocol, they can be managed together in a unified list of `Locatable` items.

### Code Validation, Testing, and Process Flow

Testing is not merely about finding bugs; its major benefits arise when developers actively think about and write the tests, a process that profoundly influences the design of the code. Developers must plan to thoroughly test all written software, as all code written will ultimately be tested, whether by the development team or the eventual users. Methodologically, developers must treat test code with the same diligence applied to production code, ensuring it remains decoupled, clean, and robust. Developers should avoid creating fragile tests by relying on unreliable aspects, such as the absolute position of widgets in a Graphical User Interface (GUI) system, exact timestamps in a server log, or the precise wording of error messages. Instead, the process must shift to having the computer automate testing, removing human preconceptions.

A powerful and structured approach is **Test-Driven Development (TDD)**. The basic TDD cycle involves five steps: 1) Deciding on a small piece of functionality; 2) Writing a test that is designed to fail before implementation; 3) Running all tests and verifying that the only failure is the one just written; 4) Writing the smallest amount of code necessary to make the test pass, followed by verifying clean test execution; and 5) Refactoring the code—improving the test or the function—while ensuring the tests remain passing. This cycle should be kept very short, ideally taking only minutes. Developers should only initiate refactoring when specific conditions arise: discovering a violation of the DRY principle, identifying an area that could be made more orthogonal, recognizing that initial requirements have drifted, or when real-world use reveals features that are more or less important than originally assumed. To guarantee external behavior hasn’t changed, developers must rely on good, automated unit testing that validates the code’s behavior.

Complementing TDD, **Property-Based Testing (PBT)** validates assumptions by defining abstract properties that must hold true for all possible inputs, rather than writing specific test examples. For instance, when testing a sorting function, a developer might assert two properties: the resulting list must maintain the same number of elements as the original list (an invariant), and no element in the result can be greater than the element immediately following it. PBT generates a wide array of random inputs—for example, running a test case one hundred times with a different random list each time—to simulate numerous individual cases. If a test fails, the resulting failure can pinpoint a fault in a core assumption. Developers must remember that while the fastest theoretical algorithm may not be the best solution for small input sets (e.g., insertion sort vs. quicksort), the utility of PBT is that it forces the developer to think in terms of invariants and contracts—what must remain unchanged and what must always be true.

This brings focus to **Design by Contract (DBC)**, where code modules operate using formal contracts, ensuring the code performs only what it explicitly claims to do. DBC guarantees that provided the input meets the required conditions, the module will make defined guarantees about its outputs. Furthermore, **code invariants** describe properties that must consistently remain true about a piece of state as that state passes through a function. Property-based testing is powerful because the developer defines the rules for generating inputs and the assertions for validating output, allowing the test suite to autonomously discover failures. Developers must also be wary of 'programming by coincidence,' the dangerous practice of relying on luck or accidental successes, such as building code based on an undocumented error response, or assuming functionality based on successful testing on one machine but failure on another.

### Algorithmic Complexity and Scalability Analysis

Understanding code performance requires **Algorithmic analysis**. Developers must estimate the resources an application consumes—including time, processor cycles, and memory—using **Big-O notation**. Big-O is a mathematical method for approximations that establishes an upper bound on resource consumption. It is conventional to remove all low-order terms and constant multiplying factors, as the highest-order term dominates as $N$ increases. While Big-O cannot convey actual numbers for time or memory, it indicates *how* these values change as the input changes. Key complexities include: $O(1)$ (Constant time, e.g., array access); $O(	ext{log } N)$ (Logarithmic time, e.g., binary search); $O(N)$ (Linear time, e.g., sequential search); $O(N 	ext{ log } N)$ (Average runtime for quicksort or heapsort); $O(N^2)$ (Square time, typical for selection/insertion sorts); $O(N^3)$ (Cubic time, often seen in matrix multiplication); and $O(2^N)$ (Exponential time, seen in problems like the traveling salesman problem).

Developers can use this notation to predict scaling. For example, if an $O(N^2)$ routine takes one second to process 100 records, it will take 100 seconds to process 1,000 records (a 10x input increase yields a 100x time increase), whereas an $O(N)$ algorithm would only increase runtime ten-fold. Predicting performance requires understanding that simple loops from 1 to $N$ result in $O(N)$ runtime, while nesting one loop inside another results in $O(N^2)$.

### Requirements Gathering and Semantic Precision

The developer's role requires adopting an inquisitive, adaptive stance in understanding needs. Developers must understand that requirements rarely appear on the surface; they are often buried beneath assumptions. Simply being told what to do is insufficient; the most valuable attribute is helping people understand what they actually want. When clients present a need, it should be treated not as an absolute requirement, but as an *invitation to explore*. The process must be one of continuous discovery, where the developer questions the initial statement of need by looking for potential loopholes and edge cases. For example, if a publisher mandates free shipping on orders of $50 or more, the developer must ask whether the $50 total includes tax, if the limit applies only to paper books or if it can include ebooks, and how frequently the limit might change. This is critical for achieving semantic precision.

Naming conventions are critical because the names must accurately reveal the code’s intent and underlying assumptions. The developer should pause to consider the 'motivation' or 'role' of every piece of code. Using a specific name like `applyDiscount(Percentage discount)` is superior to `deductPercent(double amount)` because it clearly communicates the intent, and using a custom `Percentage` type parameter documents the expected value range. Developers must prioritize consistency within a project's vocabulary, using a formal project glossary. If a developer encounters a misleading name, the foundational flaw (an ETC violation) must be fixed first, and *then* the component should be renamed.

### Security and Privilege Management

A core principle guiding robust system design is the **principle of least privilege**: developers must only use the minimum necessary permission level for the shortest amount of time. This principle suggests that developers should never automatically acquire the highest level of privilege (like root or Administrator) and must quickly relinquish it after completing minimum necessary work. This concept extends beyond operating system levels; applications should implement different, finely grained levels of access, allowing sensitive resources to be partitioned. Security practices require setting secure defaults for applications or users, as leaving the trade-off decision to the individual is preferable. Developers must encrypt sensitive data—including personally identifiable information, financial data, passwords, and other credentials—regardless of whether it resides in a database or an external file. 

Regarding passwords, the National Institute of Standards and Technology (NIST) provides guidelines: 1) Do not restrict password length to less than 64 characters (recommending 256 as a maximum); 2) Do not truncate a user's chosen password; 3) Accept all printable ASCII characters, space, and Unicode; 4) Do not provide password hints to unauthenticated users; 5) Do not disable the paste function; 6) Do not impose arbitrary composition rules or prohibit repeating characters; and 7) Do not arbitrarily require password changes after a set length of time, restricting such changes only when a valid reason, such as a security breach, exists. Developers should encourage long, random passwords with a high degree of entropy.

From a maintenance perspective, developers must apply security patches quickly. Waiting to update a system, even if it causes temporary application breakage, is dangerous because the system remains vulnerable to a known exploit. Concerning cryptography, the paramount rule is never to perform crypto work personally; developers must rely on reliable third-party, well-vetted, and preferably open-source libraries and frameworks. For authentication, using a third-party provider (like email or social media) is preferred because these external services specialize in system security.

### Internal Reflection and Process Improvement

Developers must also manage internal nonconscious signals, drawing inspiration from the concept that instincts signal a deeper issue, such as resistance when facing a blank page or difficulty when coding. When the code acts as a signal, suggesting the current situation is harder than it should be, this may point to a flawed structure, incorrect problem-solving, or the developer creating excessive bugs. To actively listen to this internal signal, the developer must first stop coding and allow time and space for the brain to organize itself—e.g., taking a walk, having lunch, or sleeping on the issue. If these methods fail, the issue should be externalized by creating doodles about the code, explaining the problem to a coworker (especially one who is not a programmer), or using the 'rubber duck' technique. The final resort is prototyping, where an aspect of the project is explored, such as testing a new framework's data binding capability or examining an algorithm on specific edge cases, ideally starting by writing actionable tests.
