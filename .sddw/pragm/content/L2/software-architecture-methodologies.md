# software-architecture-methodologies

**Parent:** [[content/L3/pragmatic-programmer-knowledge-portfolio-management|pragmatic-programmer-knowledge-portfolio-management]] — Professional development is viewed as a holistic, self-driven endeavor requiring a 'Pragmatic Programmer' mindset and adopting 'Kaizen' (continuous improvement). Technical mastery demands rigorous knowledge portfolio management, including diversifying skills, committing to learning one new language annually, and expanding reading scope to include nontechnical books. Architecturally, adherence to DRY, high orthogonality, and using the Command Shell (with plain text) are mandated, while project uncertainty should be managed using PERT, 'Eating the Elephant,' and the 'Tracer Bullet' method.

The professional practice of software development mandates adopting highly adaptable design methodologies and structured knowledge management to mitigate the inherent risks associated with technology’s accelerating pace of change. These principles govern everything from how technical knowledge is structured within a system (Don't Repeat Yourself - DRY) to how complex projects are scheduled, initiated, and maintained through iteration and continuous feedback.

### Architectural and Design Principles

Fundamental to reliable development is the adherence to **DRY (Don't Repeat Yourself)**, a cornerstone concept mandating that every single piece of knowledge within a system must possess a single, unambiguous, and authoritative representation. Violating DRY—by expressing the same knowledge in two or more places—causes severe issues, as knowledge is inherently unstable and requires constant maintenance due to client meeting adjustments, regulatory updates, or algorithm failures. This maintenance is a constant, routine part of the development process, requiring developers to locate and adjust encapsulated knowledge across specifications, processes, and programs. The principle extends beyond mere code to all representations of knowledge within the entire system. 

In data structures, DRY adherence is critical. For example, if a `Line` class initially uses redundant fields (e.g., defining `Point start`, `Point end`, and `double length`), the `length` is inherently derived from the other points. The correct practice is to eliminate the redundant field and calculate it using a method: `double length() { return start.distanceTo(end); }`. When developers violate DRY for performance optimization, such as caching, the impact must be strictly localized, and external interactions must be managed via accessor functions (e.g., `setStart`, `getStart`) to simplify future additions and align with Meyer’s Uniform Access principle. 

When dealing with interfaces to the external world, DRY violations are common. For **Across Internal APIs**, developers should centralize APIs in a single repository and use tools that specify the API in a neutral format, allowing generation of documentation, mock APIs, and functional tests across multiple languages. For **Across External APIs**, formal documentation standards like OpenAPI should be utilized; if these are lacking, the most responsible action is for programmers to create and publish them. When dealing with **Data Sources**, developers should leverage introspection capabilities to minimize duplication by generating necessary containers directly from the schema, rather than writing manual storage code. An **Alternative Data Structure Strategy** involves using key/value data structures (like a map or dictionary) instead of fixed structures (structs or classes), but this requires a second layer of defense: a simple table-driven validation suite to ensure required data is present in the correct format. 

The necessity of highly orthogonal system structures is also paramount. Orthogonality means the system is composed of cooperating modules where each module functions independently of the others, which greatly increases productivity by localizing changes and reduces risk. 

Architecturally, developers must adopt a mindset of **reversibility**, treating critical decisions (like committing to a specific third-party vendor’s database) as temporary drafts rather than permanent commitments. To achieve maximum flexibility, they must employ **Abstraction**, proactively hiding all interactions with third-party APIs behind dedicated abstraction layers, and **Componentization**, systematically breaking down code into discrete, manageable components, regardless of deployment scale.

### Project Development and Uncertainty Management

Handling uncertainty requires structured approaches to project scheduling. Instead of relying on a single estimate, best practice dictates using a range of scenarios. The **Program Evaluation Review Technique (PERT)**, initially used for the U.S. Navy's Polaris submarine project, is a methodology that requires estimating three time values for every task: an optimistic time, a most likely time, and a pessimistic time. These tasks must be organized into a dependency network to determine the overall project likelihood and mitigate the common error of padding a single number with uncertainty. Complementing this is the **'Eating the Elephant Technique,'** which acknowledges that true planning must be done iteratively. The process involves repeatedly completing thin slices of functionality through a cycle: 1) Analyze requirements; 2) Analyze risks (and prioritize riskiest items earlier); 3) Design, implement, and integrate the functionality; and 4) Validate the functionality with the users. After each cycle, the team refines the initial guess regarding the number of required iterations, increasing confidence and improving the schedule estimate with every bite taken.

### Guiding Principles and Development Processes

To achieve reliable outcomes when the goal is shifting, the **'Tracer Bullet'** approach is used. Rather than attempting 'dead reckoning' by specifying the system exhaustively upfront, the tracer bullet method demands continuous, real-time feedback. This requires identifying and focusing effort on the most critical requirements—those with the highest risks or deepest doubts—to move the project quickly from abstract requirements toward a tangible, working aspect of the final system. 

Tracer code is fundamentally different from disposable prototypes. While prototyping (like building a model out of balsa wood) generates code purely for lessons learned and risk exposure, tracer code is written intending for permanence. It must include rigorous error checking, structured documentation, and self-checking features. When implemented correctly, the initial foundation—such as integrating a front-end framework, necessary libraries, and a server's ability to generate SQL—can be built piece by piece, even if the functionality later expands (e.g., a UI team adding support for a new query type increases the complexity of the associated library).

Furthermore, the process encourages **Early User Visibility and Buy-in**. By communicating progress, users view the system as immature (which typically generates excitement) and can actively contribute to its development, increasing their sense of ownership. The core framework remains intact even as complex functionality is added, distinguishing it from a simple prototype.

### Knowledge Persistence and Tooling Mastery

The development toolkit must be expertly managed. Developers must commit to continuous learning and adaptation, maintaining mastery of fundamental core tools, even if they are constrained by powerful tools like an Integrated Development Environment (IDE). The **Command Shell** serves as the primary, superior workspace, superior to relying exclusively on GUIs and IDEs, because GUIs are inherently limited to their defined capabilities. Proficiency in the shell allows developers to combine tools using pipes in complex ways and automate common tasks, improving productivity dramatically. Customizing the shell with aliases (e.g., `alias apt-up='sudo apt-get update && sudo apt-get upgrade'`) and functional prompts is a core practice.

When managing knowledge, the most critical choice is using **plain text**. Plain text is preferred over binary formats because it offers superior portability and resilience. Its key advantage is its ability to be self-describing and independent of the application that created it, avoiding an 'artificial divorce' between data and context. While structured formats like JSON, HTML, and YAML are used, they remain plain text, providing 'insurance against obsolescence.' Plain text allows developers to manipulate knowledge manually and programmatically using virtually every available utility, which pioneered the Unix Philosophy by creating small, sharp, line-oriented, plain-text tools. This standardization is essential for tasks like running `grep -r backup /etc` and allows file comparison tools like `diff` to instantly reveal changes.

Finally, the professional value of a programmer is tied to their ability to adapt and manage a comprehensive **knowledge portfolio**. This requires continuous investment, committing to learning at least one new programming language annually to broaden conceptual thinking, and dedicating time to reading a technical book each month, eventually expanding to nontechnical books to consider the human element. Developing a reliable understanding requires applying rigorous critical analysis to the diverse stream of technical articles consumed.
