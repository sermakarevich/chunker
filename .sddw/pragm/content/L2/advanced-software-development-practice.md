# advanced-software-development-practice

**Parent:** [[content/L3/architecture-process-knowledge-management-systematics|architecture-process-knowledge-management-systematics]] — Software development mandates rigorous adherence to DRY, orthogonality, and reversibility in architecture, while managing uncertainty using PERT, the 'Eating the Elephant Technique,' and the 'Tracer Bullet' approach. Professionals must maintain mastery of the Command Shell and prioritize plain text for knowledge persistence, coupled with disciplined debugging techniques like binary chopping and structured use of engineering daybooks.

The professional practice of software development necessitates a holistic and rigorous approach to managing knowledge, mitigating inherent risks, and adapting to the accelerating pace of technological change. This discipline encompasses sophisticated architectural principles, formalized project management under uncertainty, detailed debugging protocols, and the mastery of underlying tooling and knowledge persistence.

**Architectural Integrity and Knowledge Management**

The cornerstone of reliable development is strict adherence to **DRY (Don't Repeat Yourself)**, mandating that every piece of knowledge within a system must possess a single, unambiguous, and authoritative representation. Violating DRY is critically problematic because knowledge is inherently unstable, requiring continuous, routine maintenance to accommodate adjustments stemming from diverse sources such as client meeting changes, regulatory updates, or algorithm failures. This maintenance burden forces developers to locate and adjust encapsulated knowledge across specifications, processes, and program code. This principle must extend beyond merely duplicating code.

In data structure design, DRY adherence requires eliminating redundant fields. For instance, if a `Line` class initially defines `Point start`, `Point end`, and a separate `double length`, the `length` field is redundant, as it can be derived mathematically from the other two points. The correct, principle-guided practice is to remove the redundant field and implement a calculating method: `double length() { return start.distanceTo(end); }`. Even when developers violate DRY for perceived performance gains, such as through caching, the resulting impact must be strictly localized. External interactions must adhere to accessor functions (e.g., `setStart`, `getStart`) to simplify future expansions and align with Meyer’s Uniform Access principle. 

When interfacing with external systems, centralized knowledge control is mandatory. For **Across Internal APIs**, developers must centralize these APIs within a single repository and utilize tools that specify the API in a neutral format. This standardization is vital because it enables the automatic generation of documentation, mock APIs, and functional tests across multiple programming languages. For **Across External APIs**, employing formal documentation standards like OpenAPI is non-negotiable; if such standards are absent, the most responsible action for professional programmers is to proactively create and publish them. In handling **Data Sources**, developers should minimize duplication by leveraging introspection capabilities to generate necessary data containers directly from the schema, rather than writing manual storage code. An **Alternative Data Structure Strategy** involves using key/value data structures (like a map or dictionary) instead of fixed structures (structs or classes), but this strategic choice requires a crucial secondary defense: implementing a simple table-driven validation suite to guarantee that all required data is present and formatted correctly.

The system architecture must also possess high **orthogonality**. Orthogonality ensures that the system is composed of cooperating modules where each module functions independently of the others. This structural quality significantly enhances productivity by localizing changes and mitigating overall systemic risk. Furthermore, developers must cultivate a mindset of **reversibility**, treating all critical development decisions—such as committing to a specific third-party vendor’s database—as temporary drafts rather than definitive, permanent commitments. To maximize flexibility, the strategy mandates **Abstraction**, proactively hiding all interactions with third-party APIs behind dedicated abstraction layers, complemented by **Componentization**, which involves systematically breaking down the code into discrete, manageable components, regardless of the project's scale.

**Project Management Under Uncertainty and Iteration**

Handling project uncertainty demands abandoning single-estimate forecasting in favor of structured, range-based approaches. The **Program Evaluation Review Technique (PERT)**, originally developed for the U.S. Navy's Polaris submarine project, is a structured methodology requiring the estimation of three distinct time values for every task: an optimistic time, a most likely time, and a pessimistic time. These tasks must be systematically organized into a dependency network to accurately determine the overall project likelihood, thus avoiding the significant error of arbitrarily padding a single number with uncertainty. Complementing this structured planning is the **'Eating the Elephant Technique,'** which models planning as a series of iterative cycles. This iterative process involves repeatedly completing thin slices of functionality through four defined steps: 1) Analyze requirements; 2) Analyze risks (with mandatory priority given to the riskiest items first); 3) Design, implement, and integrate the functionality; and 4) Validate the functionality with the end users. With every completed cycle, the team refines its initial guess regarding the total number of required iterations, consequently increasing confidence and refining the schedule estimate iteratively.

When the ultimate requirements or goals are constantly shifting, the **'Tracer Bullet'** approach is utilized. Instead of engaging in exhaustive upfront specification, often referred to as 'dead reckoning,' the tracer bullet method demands continuous, real-time feedback. This requires identifying and concentrating development effort on the most critical requirements—those with the deepest doubts or highest risks—to rapidly transition the project from abstract, theoretical requirements toward a tangible, working aspect of the final system. **Tracer code** is conceptually distinct from a disposable prototype (such as a balsa wood model). While prototyping aims solely for lessons learned and risk exposure, tracer code is written with the explicit intention of permanence. It must incorporate rigorous error checking, structured documentation, and self-checking features. For example, the foundational elements, like integrating a front-end framework or enabling a server to generate SQL, can be built incrementally, even if the system's functionality later expands (e.g., if a UI team adds support for a new query type, the associated library complexity increases). 

This process emphasizes **Early User Visibility and Buy-in**. By consistently communicating progress, users are empowered to view the system as immature—a state that typically generates excitement—and become active contributors, thereby increasing their sense of ownership. This ensures the core framework remains robust even as complex functionality is overlaid, distinguishing it from a temporary prototype. 

**Advanced Debugging and System Verification**

Debugging must be treated as general problem-solving, recognizing that defects, or 'bugs,' are inherent due to misunderstood requirements or coding errors. Developers must adopt a mindful state before beginning, eliminating defensive mechanisms and avoiding the danger of myopia by stepping back to analyze potential root causes behind the observed symptoms. The focus must always be on fixing the root problem, not assigning blame. The reliability of bug reports is not guaranteed; if they originate from a third party, it may be necessary to observe the user in action to acquire sufficient detail. One documented instance showed that an application crashed only when a tester painted a stroke from the upper right corner to the lower left corner, illustrating that artificial tests (like a single bottom-to-top brush stroke) fail to simulate realistic end-user usage patterns. Therefore, systematic testing of both boundary conditions and real-world scenarios is critical. 

To systematically locate errors, developers must ensure the bug can be **reproduced**, ideally using a single command. The principle of **Failing Test Before Fixing Code** dictates that the act of writing the test itself must inform the solution. When examining complex failure scenarios, specialized techniques are crucial: 1) To navigate the call stack, developers must know how to move up and down the stack, and keeping physical paper nearby aids in tracking clues. For an infinitely scrolling stack trace, the **binary chop** technique is employed, allowing the problem to be determined in at most six attempts by selecting a stack frame near the middle. 2) When diagnosing data-related bugs, the dataset should be split repeatedly in half until a minimum set of values consistently exhibits the problem. 3) When a system fails in production with a specific problematic dataset, the developer must obtain a copy of that data and binary chop the data itself until the input values leading to the crash are isolated. 

In addition to stack tracing, developers must use **Tracing statements** (e.g., “got here” or “value of x = 2”)—diagnostic messages printed to a screen or file—to track program state, which is especially important for concurrent, real-time, or event-based applications. These trace messages should follow a consistent format for automatic parsing. The **Rubber Ducking method**, which involves explaining the problem step-by-step to a person or object, is highly effective as it forces the developer to explicitly state all assumptions. While external elements like the OS or compiler might be suspected, the professional rule of thumb is that it is most productive to assume the bug resides within the application code itself. 

**Pragmatic Tools, Persistence, and Learning**

Mastery of the development toolkit requires committing to continuous learning and adapting, even when using powerful Integrated Development Environments (IDEs). The **Command Shell** is recognized as the primary, superior workspace, surpassing exclusive reliance on GUIs and IDEs, because shell scripting allows developers to combine diverse tools using pipes in complex ways and automate common tasks. Core best practices involve customizing the shell with aliases (e.g., `alias apt-up='sudo apt-get update && sudo apt-get upgrade'`) and establishing functional prompts.

Regarding knowledge persistence, **plain text** remains the paramount choice over all binary formats. Plain text offers superior portability and resilience because it is self-describing and independent of the application that originally created it, thus preventing an 'artificial divorce' between the data and its context. Though structured formats like JSON, HTML, and YAML are utilized, their plain text nature provides 'insurance against obsolescence.' This standardization allows developers to manipulate knowledge both manually and programmatically using virtually every available utility, reviving the Unix Philosophy. This is demonstrated by utilities like `grep -r backup /etc` and comparison tools like `diff` which instantly reveal changes.

Beyond technical mastery, professional value is tied to maintaining a comprehensive **knowledge portfolio**. This requires dedication to continuous intellectual pursuit: committing to learning at least one new programming language every year to broaden conceptual thinking, and dedicating time to reading a technical book every month, with eventual expansion to non-technical literature to consider the human element. Furthermore, for professional knowledge capture, authors recommend utilizing physical **engineering daybooks** as a centralized repository for notes, sketches, and acquired knowledge, as the physical act of writing offers unique cognitive benefits over typing.

**Collaborative Development and Version Control (VCS)**
VCS systems are indispensable not just as a historical repository, but as the central platform for team collaboration, deployment pipelines, and issue tracking. Developers must treat the VCS as the sole mechanism for sharing work.

**Critical Warnings:** Sharing project source files via network or cloud storage is unviable due to the high risk of corruption, lost changes, and broken builds. Storing the main repository on a network or cloud drive is even riskier, as simultaneous changes from different instances can corrupt the complex, interacting state managed by the VCS.

**Core Functions and Best Practices:** VCS systems meticulously record every change in source code and documentation, allowing users to revert to any previous version. They provide granular data essential for audits, such as determining: Who modified this line? What is the difference between the current version and last week's? How many lines changed in this release? And which files are modified most often? This data is crucial for bug-tracking and performance analysis. VCS inherently identifies software releases, guaranteeing that a specific version can always be regenerated independently of later changes. It enables multiple users to work concurrently on the same files, with the system robustly managing the merging of these changes. 

Crucially, VCS must be used for **absolutely all work**—even single-person, one-week 'throw-away' prototypes. This includes not only source code but also documentation, vendor memos, makefiles, build/release procedures, and simple shell scripts. The advanced feature of **Branching** allows development to be isolated; users can create a branch at any point, ensuring that all work remains separated from other branches, functioning effectively as small, dedicated clone projects. 

For advanced readiness, while decentralized VCS is possible, establishing a central repository is highly recommended. A central host provides features like good security, an intuitive UI, command-line access (essential for automation), automated builds and tests, and strong support for branch merging (e.g., pull requests). Many advanced teams configure their VCS such that a simple push to a specific branch automatically triggers a build, runs tests, and if successful, deploys the new code, thereby guaranteeing the ability to roll back. Developers must also proactively treat the entire laptop environment—including system configuration and user preferences (dotfiles)—as items needing capture in a VCS for comprehensive disaster recovery.
