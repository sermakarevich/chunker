# concurrency-architectural-patterns-and-coding-practices

**Parent:** [[content/L2/advanced-system-architecture-and-process|advanced-system-architecture-and-process]] — Modern software development requires adhering to DRY, orthogonality, and decoupling using protocols (like Drivable), while managing concurrency via the Actor Model (which uses unidirectional messaging and private state) or Blackboard Systems. Developers must avoid 'programming by coincidence' by diligently testing underlying assumptions, and must analyze performance using Big-O notation to predict resource scaling.

The modern software development life cycle demands an intensely adaptive and resilient approach, moving beyond mere coding to encompass continuous critical thinking across architectural design, process flow, and knowledge management. This required rigor starts with foundational principles of code structure and data management, establishing a baseline of disciplined technical architecture. A core principle is **DRY (Don't Repeat Yourself)**, which mandates that every piece of knowledge within a system must possess a single, unambiguous, and authoritative representation. Violations of DRY pose significant instability risks because knowledge is susceptible to constant maintenance across multiple specifications, processes, and programs due to external pressures—be it client meetings, regulatory shifts, or algorithmic failures. For example, if a `Line` class redundantly defines fields such as `Point start`, `Point end`, and `double length`, the redundancy of `length` must be eliminated. Instead, the length must be derived and calculated dynamically through a method, such as `double length() { return start.distanceTo(end); }`. This principle extends deeply into data handling. When dealing with **Data Sources**, developers are advised to utilize introspection capabilities to minimize duplication by generating necessary containers directly from the existing schema, rather than relying on manual storage code. Architecturally, implementing an **Alternative Data Structure Strategy** might involve key/value data structures (like a map or dictionary) instead of fixed structures (structs or classes). However, this advanced method critically mandates a secondary defense: the implementation of a simple table-driven validation suite to guarantee that all required data is present and correctly formatted. To maintain data integrity and strict adherence to best practices like Meyer’s Uniform Access principle, the impact of temporary optimizations, such as caching, must be strictly localized, and all external interactions must always be managed through dedicated accessor functions (e.g., `setStart`, `getStart`).

Building upon this, system architecture must prioritize **orthogonality**, ensuring the system is composed of cooperating modules where each component functions independently of the others. This dramatically increases productivity by localizing changes and substantially reducing systemic risk. Developers must cultivate a mindset of **reversibility**, treating critical decisions—such as committing to a specific third-party vendor's database or adopting a specific architectural pattern—as temporary drafts rather than permanent, irreversible commitments. To maximize flexibility and maintain adaptability, developers are guided by three structured techniques: 1) **Abstraction:** proactively hiding all interactions with external third-party APIs behind dedicated abstraction layers. 2) **Componentization:** systematically breaking down the code into discrete, manageable components, irrespective of the deployment scale. 3) **Guidance:** ensuring the code remains perpetually adaptable to change, adhering to the principle that there are no final decisions in development.

This adaptive methodology necessitates adhering to **Decoupling**, which means maintaining separation between distinct concepts through sound design principles. Coupling is problematic because it links together things that must inevitably change in parallel, thereby making maintenance exceptionally difficult. Developers must treat mutable external resources—including databases, file systems, and service APIs—as constituting global data. Recognizing that any change in the implementation of global data can couple the code across the entire system, any resource deemed important enough to be global must be enclosed within an API that the developers fully control. Key techniques to achieve this separation include **Interfaces and Protocols** (or traits), which specify abstract sets of required behaviors. For example, a `Drivable` protocol might mandate `double getSpeed()` and `void stop()`, while a `Locatable` protocol might require consistency across multiple components. This approach allows different components—such as a `Car` and a `Phone`—to be treated generically and managed together in a list of items adhering to the same protocol.

Given the complexities of managing concurrency in a shared resource environment, developers must adopt alternative architectural patterns. While many programming languages provide library support for mutual exclusion using constructs like mutexes, monitors, or semaphores, some, such as Rust, enforce concurrency support at the language level. Rust achieves this by enforcing data ownership, ensuring that only a single variable or parameter can reference a specific piece of mutable data at any given moment. Furthermore, while functional languages simplify concurrency by making data immutable, they encounter challenges when required to interact with the real, mutable world. Therefore, the **Actor Model** provides a recommended and powerful alternative for managing concurrent operations. 

**Actors and Processes:**
An actor is defined as an independent virtual processor that maintains its own local, private state. Every actor includes a mailbox: when a message arrives in the mailbox, and the actor is otherwise idle, it becomes active, processes the message, and continues processing subsequent messages until the mailbox is empty. While an actor processes a message, it has the capability to create other actors, send messages to known actors, and establish a new state for its subsequent message processing. A process is a more generalized virtual processor, usually implemented by the operating system for general concurrency. The specialized type of process referenced here is one constrained by convention to behave exactly like an actor.

**Core Principles of Actors:**
Actors are designed around three foundational rules: 1) **Distributed Authority:** No single entity controls the entire system; nothing schedules the sequence of events or orchestrates the transfer of information from raw data to the final output. 2) **Encapsulation of State:** All system state must reside either within incoming messages or within the local, private state of each individual actor. Messages can only be examined by the recipient who reads them, and the local state remains inaccessible to any external entity. 3) **Unidirectional Messaging:** All message transfers are strictly unidirectional; actors cannot reply directly. If a response is required, the sending actor must embed its own mailbox address within the message itself. Crucially, an actor processes each incoming message entirely to completion, and it can process only one message at a time.

As a consequence, actors execute concurrently, asynchronously, and critically, they share no state. Whether sufficient physical processors are available (allowing an actor to run on each) or if only a single processor is available (requiring the runtime system to handle context switching), the underlying code executed within the actors remains fundamentally identical. 

**Modeling Complex Systems with Actors (The Diner Scenario Example):**
To demonstrate this architecture, a system modeling a diner is constructed using three distinct actors: the `customer`, the `waiter`, and the `pie case`. The message flow dictates the interaction: The external system sends a 'hungry for pie' message to the customer. The customer responds by dispatching an 'order' message to the waiter, requesting pie. Subsequently, the waiter instructs the pie case to perform a 'get slice' operation for the customer. If the pie case confirms availability, it sends a slice to the customer and notifies the waiter to add the slice to the bill. Conversely, if the pie case finds no slices, it sends an 'error' message back to the waiter, and the waiter then apologizes to the customer. The implementation uses the Nact library in JavaScript and employs a wrapper structure where keys map to incoming message types and values hold the corresponding function to be executed.

*   **Customer Actor:** Expects three message types: 'hungry for pie' (from the external context), 'put on table' (from the pie case), and 'no pie left' (from the waiter). Upon receiving 'hungry for pie', the customer dispatches an 'order' message to the waiter. 
*   **Waiter Actor:** Processes 'order' messages (checking if the request is for pie). If so, the waiter dispatches a 'get slice' message to the pie case, passing explicit references to both the customer and the waiter. It also handles 'add to order' and 'error' messages. The 'error' message handler, specifically, dispatches a 'no pie left' message back to the customer, which triggers the waiter's apology routine. 
*   **Pie Case Actor:** Maintains an internal state represented by an array of all available pie slices. Upon receiving a 'get slice' message from the waiter, it checks if `state.slices.length` is zero. If slices exist, it uses `state.slices.shift()` to retrieve a slice, dispatches 'put on table' to the customer, dispatches 'add to order' to the waiter, and returns the updated state containing one less slice. If the array length is zero, the pie case dispatches an 'error' message to the waiter, returning the unmodified state.

This complex flow is exemplified by manually starting the three actors and passing them initial states (e.g., the pie case starts with `[]`).

Alternative advanced coordination mechanisms include **Blackboard Systems**. These systems, exemplified by the detective process of solving a complex crime, allow multiple independent agents (or actors) to coordinate and solve problems without needing direct knowledge of each other’s existence. A chief inspector posts an initial query onto a central 'blackboard.' As data accumulates, various agents—who may possess varied educational levels and expertise and come from different disciplines—contribute findings, witness statements, or forensic evidence to the board. The process continues until resolution. Key architectural features include: (1) Agents do not need to know each other's existence; they simply monitor the board for new information and post their own findings. (2) Agents can come from different disciplines and do not necessarily work within the same organizational precinct. (3) They share a single, common goal (solving the case), which is their only mandatory connection. (4) Agents may arrive and depart during the process, and they may work different shifts. (5) The blackboard imposes no limitations on the types of content posted, accepting submissions such as pictures, full sentences, or physical evidence.

Historically, computer-based blackboard systems began with AI applications like speech recognition. An early example is David Gelernter’s Linda, which stored facts as typed tuples, allowing applications to write new tuples and query for existing ones using pattern matching. Later, distributed versions like JavaSpaces and T Spaces allowed storing active Java objects on the blackboard. Users could retrieve objects by using partial matching of fields (via templates and wildcards) or by subtype—for example, querying a blackboard of `Person` objects using an `Author` template with the last name 'Shakespeare' could retrieve Bill Shakespeare but exclude Fred Shakespeare.

In large-scale applications, such as processing mortgage or loan applications, the blackboard approach offers an elegant solution to multiple complex difficulties: 1) The inherently complex and constantly changing legal requirements from federal, state, and local governments; 2) The asynchronous and variable arrival order of data (e.g., credit checks taking time while name and address are immediate); 3) The potential for data gathering by different people distributed across different time zones, or automated by other systems; 4) The fact that while some data may depend on other data (e.g., title search requires proof of ownership or insurance); and 5) The realization that new arriving data may trigger new policies and raise new questions. While a traditional workflow system could attempt to handle every possible combination, leading to reorganization complexity as regulations change, combining a blackboard with a rules engine that encapsulates legal requirements provides an elegant solution. Because the rules engine posts its output of any set of rules back to the blackboard, this approach ensures that the order of data arrival is irrelevant, and feedback loops are easily managed.

Messaging systems, such as Kafka and NATS, function similarly to blackboards and can also support the running of multiple actors. While using the actor, blackboard, and/or microservice approach eliminates a class of potential concurrency problems, it introduces a different cost: the resulting architectures are significantly harder to reason about because much of the action is indirect. Therefore, developers must proactively maintain a central repository of message formats and/or APIs, preferably one capable of generating code and comprehensive documentation. Furthermore, good tooling is necessary to trace messages and facts as they move through the system, such as propagating a unique trace ID to all involved actors to enable the reconstruction of events from log files.

While these systems are more challenging to deploy and manage due to their increased number of moving parts, this complexity is mitigated by the fact that the system is more granular, allowing individual actors or components to be updated without necessitating the replacement of the entire system.

Beyond architecture, the continuous process of writing code requires constant vigilance. Conventional wisdom incorrectly assumes that once a project reaches the coding phase, the work becomes purely mechanical—a mere transcription of the design into executable statements. This limited view is arguably the single biggest reason for software project failures, often resulting in systems that are ugly, inefficient, poorly structured, unmaintainable, or incorrect. Coding is inherently non-mechanical; it requires continuous decision-making, careful thought, and judgment to ensure the program remains accurate, productive, and functional over time. Developers who do not actively think about their code are programming by coincidence, meaning the code might function but for no specific, demonstrable reason.

The scope of involvement must be comprehensive. Regarding performance, while most code executes quickly, developers must recognize algorithms with the potential to slow down even the fastest processors. Developers must think critically about all code, including their own, and constantly identify opportunities for improvement in their programs and designs (refactoring). Moreover, testing is not solely a process of finding bugs; it provides crucial feedback on the code's design, API structure, and coupling. Therefore, the major benefits of testing accrue when the developer actively thinks about and writes the tests, rather than merely running them. Developers must also acknowledge their own inherent biases when executing tests (property-based testing). Critically, the resulting code must be highly readable and easily reason about because the real world contains 'bad actors' attempting to breach the system and cause harm. Finally, the process of naming things is challenging, as the names chosen often define the reality of the created system. Throughout the entire coding process, developers must maintain constant awareness of any potential semantic drift, much like a good, safe driver who constantly reviews the situation and proactively positions themselves to handle unexpected events, preventing a disaster even when the task is routine.
