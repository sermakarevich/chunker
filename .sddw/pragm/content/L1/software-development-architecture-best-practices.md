# software-development-architecture-best-practices

**Parent:** [[content/L2/adaptive-software-development-principles-3|adaptive-software-development-principles-3]] — Modern software development requires an adaptive, resilient process where architectural rigor (DRY, Orthogonality, Decoupling) and iterative feedback loops drive continuous improvement. Teams must prioritize cross-functional collaboration (e.g., Pair/Mob Programming) and adhere to strict operational practices, including automating deployment, encrypting all sensitive data, and applying the principle of least privilege to manage security and access.

The professional practice of modern software development demands a comprehensive, adaptive, and resilient approach that treats the development process itself as a continuous cycle of structured adaptation and rigorous critical thinking. Success hinges not on adhering to isolated methodologies, but on adapting a context-specific approach that evaluates if any given process, framework, or technique is genuinely suited for the current job and team. While adopting the practices of successful companies (like Spotify, Netflix, Stripe, or GitLab) can be inspiring, teams must resist simply implementing them because of hype or external pressure. The core goal must always be the rapid delivery of working software, continuously shortening the delivery cycle—from years to months, to weeks, and ideally, to on-demand capability.

### Agile Practices, Feedback, and Adaptability

The fundamental guiding philosophy is the management of uncertainty through continuous feedback. Decisions must be inherently contextual, depending on numerous factors, including the developer, the specific development team, the target application, the available tooling, the company's structure, the customer, and the external environment. Since no fixed, static plan can withstand this uncertainty, the focus must shift to gathering and acting on continuous feedback. This involves three recursive steps: first, determining the current state; second, making the smallest possible, meaningful step toward the desired state; and third, evaluating the outcome and correcting any introduced failures. This feedback loop must be applied repeatedly at every level, even when decisions seem minor—for instance, realizing that finding only the `email` address directly is superior to passing the entire account owner object, successfully reducing coupling between the code and account handling logic. The feedback loop is also critical at the highest project level, sometimes leading to the realization that the intended development was unnecessary or that the optimal solution requires no software at all. Furthermore, continuous improvement must apply to the team's operational process; a team that neglects to experiment with its own process is not truly agile. Good design, therefore, is measured by how easily the resulting system can be changed, making the feedback mechanism (Step 3) as painless as possible.

In addition to process adaptation, project-level success requires tackling large architectural issues. Developing concurrent applications necessitates managing mutable external resources (like databases, file systems, and service APIs), which inherently introduces potential race conditions due to shared state. To ensure resource access is atomic, developers should centralize the access mechanism by creating a method within the resource object itself (e.g., `get_pie_if_available()` within the `display_case` class). This method must utilize a `try...ensure` block to guarantee the semaphore is released even if an exception occurs. When a complex process requires multiple necessary resources (e.g., pie and ice cream for apple pie à la mode), the combination must be treated as a single, new, atomic resource, and its acquisition/release code moved into a dedicated module. All such resources important enough to be global must be wrapped in an API that the developers control, achieving robust **Decoupling**.

### Foundational Architectural Principles and Data Integrity

The technical architecture must build upon several foundational principles. **DRY (Don't Repeat Yourself)** is paramount: every piece of knowledge in a system must have a single, unambiguous, and authoritative representation. Violations of DRY are problematic because knowledge is unstable, requiring continuous maintenance across specifications, processes, and programs due to external changes (like regulatory shifts or algorithm failures). For example, if a `Line` class redundantly defines fields like `Point start`, `Point end`, and `double length`, the length redundancy must be eliminated; instead, it must be calculated using a method, such as `double length() { return start.distanceTo(end); }`. The principle extends to **Data Sources**, where developers should leverage introspection capabilities to minimize duplication by generating necessary containers directly from the schema, rather than writing manual storage code. When using an **Alternative Data Structure Strategy**, such as key/value structures (map or dictionary) instead of fixed structures (structs or classes), a secondary defense is critically mandated: a simple table-driven validation suite to guarantee all required data is present and correctly formatted. Data integrity and alignment with best practices like Meyer’s Uniform Access principle require that temporary optimizations (like caching) be strictly localized, and external interactions must always be managed through accessor functions (e.g., `setStart`, `getStart`).

Furthermore, the architecture must prioritize **orthogonality**, meaning the system consists of cooperating modules that function independently of one another, thereby localizing changes and reducing systemic risk. Developers should cultivate a mindset of **reversibility**, treating major decisions (such as committing to a specific third-party database or adopting a pattern) as temporary drafts, not permanent commitments. To maximize flexibility, developers are guided by three structured techniques: 1) **Abstraction:** proactively hiding all interactions with third-party APIs behind dedicated abstraction layers. 2) **Componentization:** systematically breaking down code into discrete, manageable components, irrespective of deployment scale. 3) **Guidance:** ensuring the code remains adaptable to change, remembering that there are no final decisions.

To achieve sound **Decoupling**, developers should use **Interfaces and Protocols** (or traits) to specify abstract sets of required behaviors—for example, a `Drivable` protocol might require `double getSpeed()` and `void stop()`—without providing implementation code. This allows components (like a `Car` and a `Phone`) that implement a common protocol (e.g., `Locatable`) to be managed generically together in a list.

### Team Structure, Communication, and Process Flow

Successful software development demands a highly adaptive and cohesive team structure. The team itself must communicate clearly with the outside world. To project an image of competence, teams should develop a distinct identity—a memorable, off-the-wall project name—and use it consistently when communicating, creating a brand for the work. The team's presentation should be positive, with well-prepared meetings and documentation that is crisp, accurate, and consistent, avoiding the pitfalls of appearing sullen, reticent, or producing messy documents with inconsistent terminology.

Crucially, minimizing friction and duplication of effort requires instantaneous and frictionless communication. If physical co-location is possible, simple proximity suffices; for remote teams, reliable messaging apps or electronic means are mandatory. Waiting a week for a meeting to ask a simple question creates significant friction, making the continuous, frictionless flow of information—regarding progress, problems, insights, and learnings—essential for staying DRY.

Regarding project execution, a common misconception is that the activities of understanding requirements, designing architecture, coding (frontend/server), and testing can happen separately. This is incorrect; all activities represent different views of the same core problem. Artificially separating roles or creating specialized teams introduces unnecessary 'gates' and 'handoffs,' which are considered waste under Lean principles. Programmers who are too far removed from the actual users often lack the necessary context to make informed decisions. The recommended approach is the **'tracer bullets' method**: developing individual features, no matter how small, that traverse the entire system end-to-end. This requires the team to possess all necessary skills—including frontend, UI/UX, server, DBA, and QA—and to be accustomed to collaborating. Using this method allows the team to implement small functionality quickly and receive immediate feedback on both the quality of team communication and the delivery process.

To manage resources and skill growth, teams must dedicate scheduled time to review operational effectiveness and failures, making changes accordingly. While personal learning is beneficial, spreading skills across the entire team is more effective, requiring planned knowledge sharing through informal brown-bag lunches or formal training sessions. Furthermore, developers should reject the temptation to add excessive features or complexities to the product; the structure should give the team just enough support to deliver the project's value.

Finally, all aspects of development and production deployment must be automated. Automation should cover tasks like code formatting, running automated tests during continuous builds, and reliably and repeatedly deploying the system. The team must incorporate the skills necessary to build and deploy these automation tools.

### Security, Privilege, and Credentials Management

Security must be built into the architecture at all levels. The **principle of least privilege** dictates that developers must only use the minimum necessary permission level for the shortest duration. Developers must never automatically acquire the highest privilege level (such as root or Administrator). If the highest level is necessary, it must be used, the minimum required work completed, and the privilege immediately relinquished to reduce risk. This principle extends beyond OS levels; applications must implement finely grained levels of access, partitioning sensitive resources and granting users permissions only for specific categories. The default settings for both applications and users must always be set to the most secure possible values, even if they are not the most user-friendly (e.g., default password entry should hide characters with asterisks). All sensitive data, including PII, financial data, passwords, and credentials, must be encrypted, whether it resides in a database or an external file.

When managing project resources, all necessary components must be stored under version control, with one critical exception: **never commit secrets, API keys, SSH keys, encryption passwords, or other credentials** alongside the source code. These must instead be managed separately, typically via config files or environment variables during build and deployment. When setting password security, common practices must be balanced with security science. NIST provides specific guidelines: (1) Do not restrict password length to less than 64 characters; 256 is recommended as a suitable maximum length. (2) Do not truncate a user's chosen password. (3) Accept all printable ASCII characters, space, and Unicode; special characters like `[]();&%$#` or `/` should not be restricted. (4) Do not provide password hints to unauthenticated users or prompt for specific personal information (like a pet’s name). (5) The paste function must not be disabled in browsers or password managers, as this merely encourages the creation of shorter, easier-to-compromise passwords (both US NIST and UK National Cyber Security Centre require this). (6) Do not impose arbitrary composition rules (e.g., requiring a specific mix of upper/lower/numeric characters). (7) Do not arbitrarily mandate password changes after a set time, restricting such changes only when a valid security breach reason exists. Developers should encourage long, random passwords with high entropy, as artificial constraints limit entropy. 

Regarding system maintenance, developers must apply security patches quickly. Waiting to update, even if causing temporary breakage, is poor practice because the system remains vulnerable to known exploits. This applies to every connected device (phones, cars, appliances, laptops, servers, cloud images) because historical data shows the largest data breaches come from outdated systems. Finally, when engaging with cryptography, developers must never perform the work personally; they must rely solely on reliable resources: well-vetted, thoroughly examined, well-maintained, frequently updated, and preferably open-source libraries. For authentication, the pragmatic approach is to use a third-party provider (via email, phone, or social media) because these external services are experts in system security and are actively maintaining them.
