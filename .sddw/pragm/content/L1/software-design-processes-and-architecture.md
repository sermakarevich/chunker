# software-design-processes-and-architecture

**Parent:** [[content/L2/adaptive-software-development-principles-3|adaptive-software-development-principles-3]] — Modern software development requires an adaptive, resilient process where architectural rigor (DRY, Orthogonality, Decoupling) and iterative feedback loops drive continuous improvement. Teams must prioritize cross-functional collaboration (e.g., Pair/Mob Programming) and adhere to strict operational practices, including automating deployment, encrypting all sensitive data, and applying the principle of least privilege to manage security and access.

The professional practice of modern software development demands a highly adaptive and resilient approach, necessitating meticulous attention to architectural principles, knowledge management, and process flow to mitigate the risks associated with technology’s accelerating pace of change. These practices transform the development process into a continuous cycle of structured adaptation and rigorous critical thinking. The core philosophical guidance is that there can never be a single, static plan, necessitating a constant focus on managing uncertainty through continuous feedback and adaptation. This 'Agile' mindset is defined as a style of practice, meaning a team responds to change and setbacks with agility, embodying values such as 'Individuals and interactions over processes and tools,' 'Working software over comprehensive documentation,' and 'Customer collaboration over contract negotiation.'

### Architectural and Data Integrity Principles

The technical architecture must adhere to several foundational principles. Chief among them is **DRY (Don't Repeat Yourself)**, which dictates that every single piece of knowledge within a system must maintain a single, unambiguous, and authoritative representation. Violations of DRY are problematic because knowledge is inherently unstable, requiring constant maintenance across specifications, processes, and programs due to external changes like client meetings, regulatory shifts, or algorithm failures. For example, if a `Line` class redundantly defines fields like `Point start`, `Point end`, and `double length`, the redundancy of `length` must be eliminated. Instead, the length must be derived and calculated using a method, such as `double length() { return start.distanceTo(end); }`. This principle extends beyond code, impacting data structures. When dealing with **Data Sources**, developers should leverage introspection capabilities to minimize duplication by generating necessary containers directly from the schema, rather than writing manual storage code. Furthermore, an **Alternative Data Structure Strategy** might involve using key/value data structures (like a map or dictionary) instead of fixed structures (structs or classes), but this method critically mandates a secondary defense: a simple table-driven validation suite to ensure that all required data is present in the correct format.

To maintain data integrity and align with best practices like Meyer’s Uniform Access principle, the impact of temporary optimizations, such as caching, must be strictly localized, and external interactions must always be managed through accessor functions (e.g., `setStart`, `getStart`).

In general, the system architecture must prioritize **orthogonality**, meaning the system is composed of cooperating modules where each functions independently of the others. This greatly increases productivity by localizing changes and reducing systemic risk. Architecturally, developers must cultivate a mindset of **reversibility**, viewing critical decisions—such as committing to a specific third-party vendor’s database or adopting a specific architectural pattern—as temporary drafts rather than permanent commitments. To maximize flexibility, three structured techniques are guided: 1) **Abstraction:** proactively hiding all interactions with third-party APIs behind dedicated abstraction layers. 2) **Componentization:** systematically breaking down code into discrete, manageable components, regardless of deployment scale. 3) **Guidance:** ensuring the code remains always adaptable to change, remembering that there are no final decisions. This necessitates adhering to **Decoupling**, which emphasizes keeping separate concepts separate through sound design principles. Coupling links together things that must change in parallel, making maintenance difficult. Developers must treat mutable external resources, such as databases, file systems, and service APIs, as constituting global data. Recognizing that any change in the global data’s implementation can couple the code across the system, any resource important enough to be global must be wrapped in an API that the developers control. Techniques to achieve this include **Interfaces and Protocols** (or traits), which specify abstract sets of required behaviors—for example, `Drivable` might require `double getSpeed()` and `void stop()`—without providing implementation code. This allows components to be treated generically, such that if both a `Car` and a `Phone` implement a `Locatable` protocol, they can be managed together in a list of `Locatable` items. When implementing functional contracts, developers should favor the use of these abstract definitions.

Moreover, developing concurrent applications requires managing mutable external resources, which introduces potential race conditions due to shared state. To solve this, developers must ensure resource access is atomic. While using semaphores or mutual exclusion works, a more robust solution is to centralize resource access by creating a method within the resource object itself (e.g., `get_pie_if_available()` within the `display_case` class). This method must use a `try...ensure` block to guarantee the semaphore is always released, even if an exception occurs during processing. Furthermore, when dealing with multiple necessary resources, such as needing both pie and ice cream for an apple pie à la mode, the developer must treat the combination as a single, new resource. The code handling the acquisition and release of all component resources should be moved into a dedicated module to manage this atomically.

### Process, Constraints, and Problem Solving

When approaching any complex or intractable problem, developers must adopt a systematic mindset. Firstly, it is crucial to enumerate *all* possible avenues, explaining why certain paths cannot be taken, before dismissing any idea, regardless of how unusable or impractical it might sound. To solve a puzzle, individuals must recognize both the placed constraints and the available degrees of freedom; the solution often resides within those freedoms, rather than requiring one to think outside the box. A highly effective method for tackling complex problems is to identify the most restrictive constraints first and fit the remaining constraints within those limitations, similar to how woodworkers begin a project by cutting the longest pieces.

If a problem feels unnecessarily difficult or 'impossible,' abandoning the direct approach is recommended. Taking a break or switching to a different task allows the associative neural net lurking below consciousness to provide the answer when the developer deliberately distracts themselves. If distraction is not feasible, the next best approach is to seek external explanation through techniques like 'Rubber Ducking,' which involves asking detailed questions: Why is the developer solving this problem? What benefit does solving the problem provide? Are the existing problems related to edge cases that can be eliminated? Is there a simpler, related problem that can be solved?

Louis Pasteur observed that 'In the fields of observation, chance only favors the prepared mind,' emphasizing that for problem-solving 'eureka' moments, the nonconscious brain requires sufficient raw material from prior experiences. Practical ways to feed the brain include giving feedback on what works and what does not work during daily work, such as using an Engineering Daybook. Developers must also actively challenge preconceived notions and evaluate whether the apparent constraints are real, hard-and-fast rules, aiming always to determine the real, limiting constraints of the problem.

Finally, successful projects, particularly those perceived as 'impossible,' require deep collaboration. The most successful approach is to work closely with users, considering them integral parts of the development team. This collaborative model exemplifies pair programming or mob programming. **Pair Programming**, an eXtreme Programming practice, involves two developers working together on a single problem: one operates the keyboard (the typist), while the second developer considers the problem. Duties can be switched. This process benefits from varied backgrounds and perspectives, allowing the developer considering the problem to focus on higher-level concerns while the typist handles low-level syntax details. The presence of an actively observing second developer helps prevent bad habits, such as naming variables generically (e.g., `foo`), by providing inherent peer pressure and increasing overall software quality. **Mob Programming** extends this by involving more than two developers working simultaneously on one problem, with typically only one typist. Mobs can easily include individuals not conventionally considered core developers, such as users, project sponsors, and testers. Collaborative practices are highly effective regardless of the working style—whether in the same office or remotely, alone, in pairs, or in mobs. When adopting these collaborative practices, the team must define who is included: Is participation limited to developers, or can extended members like users, testers, and sponsors be included? Starting recommendations include building the code, not criticizing the individual's ego; starting small, perhaps with a mob of only 4-5 people or beginning with short pair sessions; criticizing the code (e.g., “Let’s look at this block”) rather than the person; listening actively to and seeking to understand diverse viewpoints; and conducting frequent retrospectives to facilitate continuous improvement. The process of continuous improvement is crucial, as the measure of design is how easy the resulting system is to change. The team should initially research and understand the suggested rules and guidelines and is recommended to start by coding a simple exercise rather than jumping into a complex production project.

### Team Structure and Process Optimization

On a project-level view, the team's internal communication structures profoundly constrain the resulting system architecture, a principle formalized by **Conway's Law** (Melvin Conway, 1967). This law states that organizations designing systems are constrained to produce designs that are copies of the communication structures of the organization. Consequently, the social structures and communication pathways of the development team and the larger organization will be mirrored in the resulting application, website, or product. Historical studies show that teams with no communication pathways tend to create siloed, 'stove-pipe' systems. Conversely, teams split into two groups often result in clear client/server or frontend/backend architectural divisions. Furthermore, geographically distributed teams tend to produce more modular, distributed software. Crucially, development teams that include users in the process will produce software that clearly reflects that involvement, while teams that exclude users will reflect that exclusion. A pragmatic team is defined as a small, stable entity, ideally containing fewer than 10-12 members, and quality is fundamentally a team issue, meaning the team must take responsibility for the product's quality.

To make the continuous feedback loop efficient, the mechanism for fixing what is broken (Step 3 in the recursive cycle) must be as painless as possible. The process of continuous feedback—determining the current state, making the smallest possible, meaningful step toward the desired state, and evaluating the outcome and correcting any introduced failures—must be applied repeatedly at every level of development, even when making seemingly trivial decisions. This highly detailed approach ensures that good design makes changes easy and adjustments possible at every level, which is the essence of agility.
