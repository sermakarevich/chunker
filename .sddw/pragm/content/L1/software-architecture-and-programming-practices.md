# software-architecture-and-programming-practices

**Parent:** [[content/L2/advanced-system-architecture-and-process|advanced-system-architecture-and-process]] — Modern software development requires adhering to DRY, orthogonality, and decoupling using protocols (like Drivable), while managing concurrency via the Actor Model (which uses unidirectional messaging and private state) or Blackboard Systems. Developers must avoid 'programming by coincidence' by diligently testing underlying assumptions, and must analyze performance using Big-O notation to predict resource scaling.

The professional practice of modern software development demands a highly adaptive and resilient approach, necessitating meticulous attention to architectural principles, knowledge management, and process flow to mitigate the risks associated with technology’s accelerating pace of change. This process transforms development into a continuous cycle of structured adaptation and rigorous critical thinking, requiring developers to constantly review the situation, check for potential problems, and proactively position themselves to handle unexpected events, similar to a good driver reviewing traffic. The inherent complexity necessitates adhering to foundational architectural principles, chief among them being **DRY (Don't Repeat Yourself)**. This principle demands that every piece of knowledge within a system must maintain a single, unambiguous, and authoritative representation. Violating DRY is detrimental because knowledge is unstable, requiring constant, difficult maintenance across specifications, processes, and programs due to external changes, such as client meetings, regulatory shifts, or algorithm failures. For example, if a `Line` class redundantly defines fields like `Point start`, `Point end`, and `double length`, the redundancy of `length` must be eliminated. Instead, the length must be derived and calculated using a method, such as `double length() { return start.distanceTo(end); }`. This principle extends deeply into data handling. When managing **Data Sources**, developers should leverage introspection capabilities to minimize duplication by generating necessary containers directly from the schema, rather than writing manual storage code. An **Alternative Data Structure Strategy** might involve utilizing key/value data structures (like a map or dictionary) instead of fixed structures (structs or classes). However, this approach mandates a critical secondary defense: a simple table-driven validation suite to ensure that all required data is present in the correct format. To maintain data integrity and align with best practices like Meyer’s Uniform Access principle, the impact of temporary optimizations, such as caching, must be strictly localized, and external interactions must always be managed through accessor functions (e.g., `setStart`, `getStart`).

In terms of system design, architecture must prioritize **orthogonality**, meaning the system comprises cooperating modules where each functions independently of the others. This significantly boosts productivity by localizing changes and reducing systemic risk. Developers must cultivate a mindset of **reversibility**, viewing critical decisions—such as committing to a specific third-party vendor’s database or adopting a specific architectural pattern—as temporary drafts rather than permanent commitments. To maximize flexibility and manage dependencies, three structured techniques are essential: 1) **Abstraction:** proactively hiding all interactions with third-party APIs behind dedicated abstraction layers. 2) **Componentization:** systematically breaking down code into discrete, manageable components, regardless of deployment scale. 3) **Guidance:** ensuring the code remains always adaptable to change, recognizing that there are no final decisions.

This structural approach necessitates strict adherence to **Decoupling**, which mandates keeping separate concepts separate through sound design principles. Coupling is dangerous because it links together things that must change in parallel, making maintenance extremely difficult. Developers must treat mutable external resources—including databases, file systems, and service APIs—as constituting a global data set. Recognizing that any change in the global data’s implementation can couple the code across the entire system, any resource deemed important enough to be global must be rigorously wrapped in an API that the developers fully control. Techniques to achieve this include **Interfaces and Protocols** (or traits), which specify abstract sets of required behaviors. For instance, `Drivable` might mandate `double getSpeed()` and `void stop()`, without providing implementation code. This allows components to be managed generically; if both a `Car` and a `Phone` implement a `Locatable` protocol, they can be managed together in a unified list of `Locatable` items.

Beyond structural design, the developer’s mental state and approach to problem-solving are critical. Developers must never engage in **programming by coincidence**, which is the act of relying on luck or accidental successes. This dangerous practice is analogous to a soldier being hit by a mine despite initial probes revealing nothing, leading to false conclusions and disastrous outcomes. The danger is particularly evident when code appears to function during limited testing but fails later because the developer failed to understand the underlying mechanism. Pitfalls include relying on 'accidents of implementation,' where the program only works due to undocumented error or boundary conditions. A prime example is a routine that is called with bad data, and the code is built based on the routine's specific, undocumented response; subsequent ‘fixing’ of that routine can cause the developer's original code to break. Similarly, calling GUI rendering routines like `paint()`, `invalidate()`, `validate()`, `revalidate()`, and `repaint()` sequentially might *appear* to work, but the developer must recognize that merely working now does not guarantee reliability because the underlying boundary condition might be accidental or the behavior could change with library updates. Even temporal issues, like the ‘only off by one’ time zone discrepancy encountered in a project handling hardware data collection units, can accumulate into untenable errors, potentially causing the entire project to be scrapped due to a fundamental flaw.

To counteract coincidence, developers must practice deliberate programming, adhering to principles like **Design by Contract** to help eliminate misunderstandings, especially when creating a well-specified contract that other developers will call. When writing code that other developers will call, developers must ensure good modularization and utilize small, well-documented interfaces. Conversely, when calling routines, developers must rely only on documented behavior. If documentation is insufficient, the developer must explicitly document the assumed behavior. Furthermore, developers must be wary of 'phantom patterns'—the human tendency to see causes and patterns even when the underlying occurrences are merely coincidental (such as assuming patterns in lottery numbers or stock trading). Even system failures and intermittent errors, such as a log file showing an intermittent error every 1,000 requests, must be treated with skepticism, as they may be complex race conditions or simple bugs. Testing that passes on one machine but fails on another may simply indicate environmental differences or coincidence.

For robust, reliable code, several guidelines must be followed: 
1. **Prioritize the necessary:** The developer must always be able to explain the code in detail to a junior programmer. 
2. **Proceed from a plan:** The application must always be planned, whether using a written plan, a whiteboard, or a napkin. 
3. **Rely on verifiable facts:** Developers must assume the worst if they cannot definitively determine that a dependency or assumption is reliable. All assumptions must be documented, using Design by Contract to clarify assumptions for both internal use and communication with other developers.
4. **Test the assumptions:** Developers must not only test the code but also the assumptions. Instead of guessing, they should write an assertion to test the assumptions; if the assertion passes, the code documentation is improved; if it fails, a potential flaw has been identified.
5. **Address scope and history:** Developers must not let existing code constrain future development, understanding that all code can be replaced if it is no longer appropriate, and must plan for refactoring.

Simultaneously, the developer must manage internal nonconscious signals, drawing inspiration from Gavin de Becker’s work in *The Gift of Fear*. Our instincts, or our 'lizard brain' responses, are critical and are responses to patterns stored in the nonconscious brain, irrespective of whether the source is innate or learned through repetition. These instincts share the characteristic of having no words; they make a person feel, not think. When an instinct is triggered, the resulting feeling—such as nervousness, queasiness, or feeling overwhelmed—signals a deeper issue. Two common scenarios trigger this internal signal: facing the blank page (feeling resistance) or experiencing difficulty while coding. The resistance when starting a new module or project may be the accumulated experience trying to communicate the issue. Giving this nagging doubt time will likely make the concern clearer and more solvable. When coding difficulty increases, the code itself acts as a signal, suggesting that the current situation is harder than it should be. This may point to a flawed structure or design, solving the wrong problem, or the developer creating an excessive number of bugs. This self-sensing mechanism urges the developer to pay attention. To actively listen to the inner lizard brain, the developer must first stop coding and allow time and space for the brain to organize itself, perhaps by taking a walk, having lunch, chatting, or sleeping on the issue. If these methods fail, the developer should externalize the issue by creating doodles about the code, explaining the problem to a coworker (especially one who is not a programmer), or talking to a rubber duck. If the problem still persists, the final resort is prototyping: identifying an aspect of the project to explore, such as testing a new framework's data binding capability or examining an algorithm on specific edge cases. If working on existing resistant code, the developer should stash the existing code and prototype something similar instead. The prototyping process should begin by writing actionable tests.

Finally, understanding code performance requires **Algorithmic analysis**. Developers must estimate the resources an application consumes—including time, processor cycles, and memory—using Big-O notation. This is a mathematical method for dealing with approximations that establishes an upper bound on the value being measured. For example, when stating a routine sorts records in $O(N^2)$ time, it means the worst-case time will vary as the square of the input size ($N$). The convention is to remove all low-order terms and constant multiplying factors, as the highest-order term dominates the value as $N$ increases. While Big-O notation cannot convey actual numbers for time or memory (one algorithm might be 1,000 times faster than another), it indicates how these values change as the input changes.

Common complexities include: $O(1)$ (Constant time, e.g., accessing an element in an array); $O(	ext{log } N)$ (Logarithmic time, e.g., binary search); $O(N)$ (Linear time, e.g., sequential search); $O(N 	ext{ log } N)$ (Average runtime for quicksort or heapsort); $O(N^2)$ (Square time, typical for selection and insertion sorts); $O(N^3)$ (Cubic time, often seen in matrix multiplication); and $O(2^N)$ (Exponential time, used for problems like the traveling salesman problem).

Developers can use this notation to predict scaling. For instance, if an $O(N^2)$ routine takes one second to process 100 records, it will take 100 seconds to process 1,000 records (a 10x increase in input yields a 100x increase in time). Conversely, an $O(N)$ algorithm will only increase the runtime ten-fold when the input rises to 1,000 records. Exponential time complexity might result in runtimes taking years to complete.
