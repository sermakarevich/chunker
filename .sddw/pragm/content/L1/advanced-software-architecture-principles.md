# advanced-software-architecture-principles

**Parent:** [[content/L2/software-development-architecture-principles|software-development-architecture-principles]] — Software development mandates adherence to DRY, reversibility, and decoupling principles, utilizing techniques like interfaces/protocols, componentization, and abstraction to manage complexity, while the data flow must be treated as a continuous, trans-functional pipeline that requires structured error handling and rigorous external API documentation (e.g., OpenAPI).

The practice of professional software development mandates adopting highly adaptable design methodologies, structured knowledge management, and continuous adaptation to mitigate the risks inherent in technology’s accelerating pace of change. These principles guide how technical knowledge is structured, how complex projects are managed, and how developers interact with various data and computing resources. At the core of good design are several foundational principles, chief among them being **DRY (Don't Repeat Yourself)**, which stipulates that every single piece of knowledge within a system must possess a single, unambiguous, and authoritative representation. Violating DRY is problematic because knowledge is inherently unstable, requiring constant maintenance across specifications, processes, and programs due to adjustments in client meetings, regulatory changes, or algorithm failures. This principle extends beyond just code; for instance, if a `Line` class redundantly defines fields like `Point start`, `Point end`, and `double length`, the redundancy of `length` must be eliminated. Instead, the length must be derived and calculated using a method, such as `double length() { return start.distanceTo(end); }`. When developers violate DRY for performance optimization, such as caching, the impact must be strictly localized, and external interactions must be managed through accessor functions (e.g., `setStart`, `getStart`) to maintain alignment with Meyer’s Uniform Access principle. To prevent duplication when interfacing with the external world, **Across Internal APIs**, developers should centralize APIs in a single repository and use tools that specify the API in a neutral format, which facilitates the generation of documentation, mock APIs, and functional tests across multiple languages. For **Across External APIs**, formal documentation standards like OpenAPI must be used; if these are missing, the most responsible action is for programmers to create and publish them. Furthermore, when dealing with **Data Sources**, developers should leverage introspection capabilities to minimize duplication by generating necessary containers directly from the schema, rather than writing manual storage code. An **Alternative Data Structure Strategy** may involve using key/value data structures (like a map or dictionary) instead of fixed structures (structs or classes), but this method mandates a critical secondary defense: a simple table-driven validation suite to ensure that all required data is present in the correct format. The necessity of highly orthogonal system structures is also paramount, meaning the system is composed of cooperating modules where each functions independently of the others, greatly increasing productivity by localizing changes and reducing risk. Architecturally, developers must cultivate a mindset of **reversibility**, viewing critical decisions—such as committing to a specific third-party vendor’s database or a specific architectural pattern—as temporary drafts rather than permanent commitments. The danger of making such decisions is that the change might become extremely expensive or impossible to undo. To maximize flexibility, developers must follow three structured techniques: 1) **Abstraction:** proactively hiding all interactions with third-party APIs behind dedicated abstraction layers. 2) **Componentization:** systematically breaking down code into discrete, manageable components, regardless of deployment scale. 3) **Guidance:** ensuring the code is always adaptable to change, remembering that there are no final decisions. This requires adhering to the principle of **Decoupling**, which emphasizes keeping separate concepts separate through sound design principles, as coupling links together things that must change in parallel, making maintenance difficult. Developers must also treat mutable external resources, such as databases, file systems, and service APIs, as constituting global data, recognizing that any change in the global data’s implementation can couple the code across the system. Therefore, any resource important enough to be global must be wrapped in an API that the developers control. In general, developers must avoid relying on self-deception regarding impossible outcomes and must instead explicitly check for every possible outcome. When adapting a browser-based application to a mobile app, the change should ideally only require stripping out HTML rendering and replacing it with an API, minimizing the impact on the server side. Developers must always abstract the idea of a database to simply providing persistence as a service, which offers the flexibility to switch vendors or technologies. To achieve maximum flexibility, adopting an architectural approach that views persistence only as a service is necessary. This parallels the architectural decision of making a specific database's concept abstractly available. Another foundational technical practice is the superior utilization of the **Command Shell** as the primary workspace. Developers must commit to mastering fundamental core tools, making the shell superior to relying exclusively on GUIs and IDEs because GUIs are inherently limited to their defined capabilities. Proficiency in the shell allows developers to combine tools using pipes in complex ways and automate common tasks, significantly improving productivity. Complementing this, the use of **plain text** is the most critical choice for knowledge persistence due to its superior portability and resilience. Plain text is preferred over binary formats because it is self-describing and independent of the application that created it, thus avoiding an 'artificial divorce' between data and context. While structured formats like JSON, HTML, and YAML are utilized, they remain plain text, offering 'insurance against obsolescence.' This standardization is vital for tasks like running `grep -r backup /etc` and allows file comparison tools like `diff` to instantly reveal changes. Ultimately, the professional value of a programmer is linked to their ability to manage a comprehensive **knowledge portfolio**, requiring continuous investment, such as committing to learning at least one new programming language annually to broaden conceptual thinking, and dedicating time to reading a technical book each month, eventually expanding to nontechnical books to consider the human element. Developing a reliable understanding requires applying rigorous critical analysis to the diverse stream of technical articles consumed. 

### Project Planning and Uncertainty Management

Handling uncertainty requires structured project scheduling. Instead of relying on a single estimate, best practice dictates using a range of scenarios. The **Program Evaluation Review Technique (PERT)**, initially utilized for the U.S. Navy's Polaris submarine project, is a methodology requiring the estimation of three time values for every task: an optimistic time, a most likely time, and a pessimistic time. These tasks must be organized into a dependency network to determine the overall project likelihood and mitigate the common error of padding a single number with uncertainty. Complementing PERT is the **'Eating the Elephant Technique,'** which acknowledges that true planning must be done iteratively. This process involves repeatedly completing thin slices of functionality through a cycle: 1) Analyze requirements; 2) Analyze risks (and prioritize the riskiest items earlier); 3) Design, implement, and integrate the functionality; and 4) Validate the functionality with the users. After each cycle, the team refines the initial guess regarding the number of required iterations, increasing confidence and improving the schedule estimate with every bite taken. 

When the project goal is shifting, the **'Tracer Bullet'** approach is deployed. Rather than attempting 'dead reckoning' by specifying the system exhaustively upfront, this method demands continuous, real-time feedback. This involves identifying and focusing effort on the most critical requirements—those with the highest risks or deepest doubts—to rapidly move the project from abstract requirements toward a tangible, working aspect of the final system. Tracer code differs fundamentally from disposable prototypes; while prototyping generates code purely for lessons learned and risk exposure (like building a model out of balsa wood), tracer code is written with an intention of permanence. It must include rigorous error checking, structured documentation, and self-checking features. When correctly implemented, the initial foundation, such as integrating a front-end framework, necessary libraries, and a server's ability to generate SQL, can be built piece by piece, even if the functionality later expands. Furthermore, the process strongly promotes **Early User Visibility and Buy-in**, where users, by viewing the system as immature (which typically generates excitement), actively contribute to its development, thereby increasing their sense of ownership.
