# adaptive-software-development-principles

**Parent:** [[content/L2/system-architecture-and-process-rigor|system-architecture-and-process-rigor]] — Modern software development requires adherence to architectural principles like DRY, orthogonality, and decoupling, which are enforced by mechanisms such as Interfaces and Protocols, and advanced techniques like Property-Based Testing (PBT) and Design by Contract (DBC). The process must prioritize iterative learning (treating requirements as 'invitations to explore'), algorithmic efficiency analysis using Big-O notation (e.g., distinguishing $O(N)$ from $O(N^2)$), and rigorous security practices (e.g., Principle of Least Privilege and adhering to NIST password guidelines).

The modern professional practice of software development mandates a holistic, adaptive, and resilient approach, demanding meticulous attention to not only technical architecture and code structure but also knowledge management and robust process flows. The goal is to transform the development cycle into a continuous process of structured adaptation and rigorous critical thinking to mitigate risks posed by technology's accelerating pace of change. This comprehensive development process requires adopting an incremental approach, where developers build small pieces of end-to-end functionality while continually learning about the problem space and engaging the customer at every stage to guide the process. While adopting methodologies like Test-Driven Development (TDD) is beneficial, focusing exclusively on it risks producing excessive code that does not advance the core solution. For instance, Ron Jeffries documented his test-driven coding of a Sudoku solver in 2006; although he refined the board’s representation and refactored the object model across multiple posts, he eventually abandoned the project, illustrating how a developer can become distracted by minor details, even when encountering the positive feedback of passing tests. A more effective strategy, suggested by Peter Norvig, involves beginning by understanding how the problem is traditionally solved (e.g., using constraint propagation), then dedicating focus to refining the algorithm, a technique demonstrated when Norvig addressed board representation in just a dozen lines of code, stemming directly from his initial discussion of notation. Therefore, while tests drive development, the process must not become circular without a clear destination.

Architecturally, the system must adhere to foundational principles like **DRY (Don't Repeat Yourself)**, which requires that every piece of knowledge maintains a single, unambiguous, and authoritative representation. Violations of DRY create instability, necessitating constant maintenance across specifications, processes, and programs due to external shifts such as regulatory changes, client meetings, or algorithm failures. For example, if a `Line` class redundantly defines `Point start`, `Point end`, and `double length`, the redundancy of `length` must be eliminated; instead, it must be derived and calculated using a method like `double length() { return start.distanceTo(end); }`. This principle extends beyond basic code structure. When managing **Data Sources**, developers should use introspection capabilities to minimize duplication by generating necessary containers directly from the schema, rather than writing manual storage code. Furthermore, when considering an **Alternative Data Structure Strategy**, such as employing key/value data structures (like a map or dictionary) instead of fixed structures (structs or classes), this method requires a secondary defense: a simple table-driven validation suite to ensure all required data is present and correctly formatted. Adhering to best practices like Meyer’s Uniform Access principle mandates that the impact of temporary optimizations, such as caching, must be strictly localized, and external interactions must always be mediated through accessor functions (e.g., `setStart`, `getStart`).

To maximize system flexibility and productivity, the architecture must prioritize **orthogonality**, meaning that the system is composed of cooperating modules where each module functions independently of the others, thereby localizing changes and significantly reducing systemic risk. Developers must cultivate a mindset of **reversibility**, treating critical decisions—such as adopting a specific architectural pattern or committing to a third-party vendor’s database—as temporary drafts rather than permanent commitments. This flexibility is maximized through three structured techniques: 1) **Abstraction:** proactively hiding all interactions with third-party APIs behind dedicated abstraction layers; 2) **Componentization:** systematically breaking down the code into discrete, manageable components, irrespective of the deployment scale; and 3) **Guidance:** ensuring the code remains adaptable to change, maintaining the awareness that no decisions are final.

This systemic necessity mandates strict adherence to **Decoupling**, which is the practice of keeping separate concepts separate through sound design. Coupling occurs when components link together things that must change in parallel, making maintenance overly difficult. Developers must treat mutable external resources—including databases, file systems, and service APIs—as constituting global data. Recognizing that any change in the global data’s implementation can couple the code across the entire system, any resource deemed important enough to be global must be enveloped in an API that the developers fully control. Techniques used to achieve this include **Interfaces and Protocols** (or traits), which specify abstract sets of required behaviors. For example, a `Drivable` interface might mandate `double getSpeed()` and `void stop()`, without providing implementation code. This allows components to be treated generically: if both a `Car` and a `Phone` implement a `Locatable` protocol, they can be managed together in a list of `Locatable` items.

Testing, as a core element of the programming process, is not a function reserved for separate departments; developers must plan for thorough testing of all written software, as all code written will ultimately be tested, either by the development team or the end users. Developers are given three main choices for testing: Test First, Test During, or Test Never. 'Test Later' is the least effective choice, as it is functionally equivalent to 'Test Never.' **Test First**, which encompasses Test-Driven Design (TDD), is typically the most effective because it ensures that testing is integral to the development cycle. If TDD is not immediately convenient, 'Test During' serves as a viable fallback, where the developer writes some code, refines it, writes tests for it, and then proceeds to the next section. 

Methodologically, developers must treat test code with the same diligence applied to production code, ensuring it remains decoupled, clean, and robust. To avoid creating fragile tests, developers must avoid relying on unreliable aspects, such as the absolute position of widgets in a Graphical User Interface (GUI), exact timestamps in a server log, or the precise wording of error messages. Instead, they should transition to having the computer automate the testing process, removing human preconceptions. The concept of **Design by Contract (DBC)** guides modules to operate using formal contracts, ensuring the code only performs what it explicitly claims to do. DBC guarantees that provided the input meets the required conditions, the module will make defined guarantees about its outputs. Furthermore, **code invariants** describe properties that must consistently remain true about a piece of state as that state passes through a function. By implementing these invariants and contracts—collectively called properties—developers can leverage **property-based testing** (PBT). PBT allows developers to validate core assumptions by defining properties that must hold true for all possible inputs, rather than merely writing specific examples. For instance, when testing a sorting function, a developer might assert two properties: the resulting list must have the same number of elements as the original list (maintaining length as an invariant), and no element in the result can be greater than the element immediately following it. PBT involves a strategy to generate a wide array of random inputs—for example, running a test case one hundred times with a different random list each time—simulating the testing of numerous individual cases. This method is highly powerful because it can fail due to a fault in a core assumption, even if the bug was not directly related to the feature currently being tested. For example, when processing an order, a property test could assert that if items are taken from the warehouse, the total quantity removed plus the quantity currently in the warehouse must equal the original stock level. If this property fails, the automated test locates a fault in a core assumption, such as realizing the initial `in_stock` function only checked for at least one item being present, when the system should have ensured sufficient units existed to complete the order. Property-based tests are powerful because the developer sets the rules for generating inputs and defines the assertions for validating output, allowing the test suite to autonomously discover failures. The test process can fail in three ways: the test passes, an assertion fails, or the code fails entirely because it cannot handle the inputs provided by the test generator. 

In addition to structural integrity, effective software development requires continuous critical reflection on the practice of testing itself. An experienced developer, who had spent 45 years coding and 30 years writing automated tests, once paused testing to observe its impact. They found the impact surprisingly small, leading them to conclude that the majority of the benefit derived from testing comes not from the act of executing tests, but from the critical act of considering the tests and their impact on the code itself. While the developer noted that the code remained testable even without formal tests, they also acknowledged that ignoring testing overlooks the fact that tests serve as a crucial method of communication with other developers, especially when code is shared or relies on peculiar external dependencies. This insight suggests that although developers may achieve fluency to perform critical thinking without writing tests after decades, they should still formally write them to maintain robust communication and technical rigor.
