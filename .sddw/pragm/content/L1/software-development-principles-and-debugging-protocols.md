# software-development-principles-and-debugging-protocols

**Parent:** [[content/L2/advanced-software-development-practice|advanced-software-development-practice]] — Professional development requires strict adherence to DRY principles and high orthogonality, utilizing methods like PERT and the 'Tracer Bullet' for planning, and mastering specialized techniques like binary chopping and 'Rubber Ducking' for debugging. Knowledge management demands prioritizing plain text persistence, using the Command Shell as the primary workspace, and employing VCS universally for version control and recovery.

The professional practice of software development requires a highly adaptable and structured methodology for managing inherent risks, accelerating change, and mitigating defects. This comprehensive approach covers architectural design principles, project management under uncertainty, debugging methodologies, and core knowledge persistence practices.

### Architectural Rigor and Knowledge Management

At the foundation of reliable development is the principle of **DRY (Don't Repeat Yourself)**, which mandates that every piece of knowledge within a system must have a single, unambiguous, and authoritative representation. Violating DRY is severely problematic because knowledge is inherently unstable and requires constant maintenance due to factors like client meeting adjustments, regulatory updates, or algorithm failures. This maintenance burden necessitates developers locating and adjusting encapsulated knowledge across specifications, processes, and programs. The principle extends beyond mere code: in data structures, if a `Line` class redundantly defines `Point start`, `Point end`, and a `double length`, the correct practice is to eliminate the redundant `length` field and calculate it using a method (e.g., `double length() { return start.distanceTo(end); }`). Even when optimization like caching leads to DRY violations, the impact must be strictly localized, and external interactions must be managed via accessor functions (e.g., `setStart`, `getStart`) to simplify future additions and align with Meyer’s Uniform Access principle. When managing interfaces, **Across Internal APIs**, developers must centralize APIs in a single repository and utilize tools that specify the API in a neutral format, facilitating the generation of documentation, mock APIs, and functional tests across multiple languages. For **Across External APIs**, leveraging formal documentation standards like OpenAPI is critical; if these standards are absent, the most responsible action is for programmers to create and publish them. For **Data Sources**, developers should minimize duplication by leveraging introspection capabilities to generate necessary containers directly from the schema, rather than writing manual storage code. Furthermore, an **Alternative Data Structure Strategy** suggests using key/value data structures (such as a map or dictionary) instead of fixed structures (structs or classes), but this requires a second defense layer: a simple table-driven validation suite to ensure required data is present and correctly formatted.

The system structure must also achieve high **orthogonality**, meaning modules cooperate but function independently of each other, thereby greatly increasing productivity by localizing changes and reducing risk. Architecturally, developers must adopt a mindset of **reversibility**, treating critical decisions (like committing to a specific third-party vendor’s database) as temporary drafts, and must employ **Abstraction** to proactively hide all interactions with third-party APIs behind dedicated abstraction layers. This is complemented by **Componentization**, systematically breaking down code into discrete, manageable components, regardless of the eventual deployment scale.

In terms of knowledge persistence, **plain text** remains the superior choice over binary formats due to its unmatched portability and resilience. Plain text is self-describing and independent of the application that created it, avoiding an 'artificial divorce' between data and context. Although structured formats like JSON, HTML, and YAML are plain text, they offer 'insurance against obsolescence' and allow developers to manipulate knowledge manually and programmatically using nearly every available utility, following the Unix Philosophy. This standardization is vital for tasks like running `grep -r backup /etc` and enabling file comparison tools like `diff` to instantly reveal changes.

### Project Planning and Iterative Development

Managing project uncertainty requires structured, iterative approaches. Best practice dictates using a range of scenarios rather than a single estimate. The **Program Evaluation Review Technique (PERT)**, originally applied to the U.S. Navy's Polaris submarine project, is a methodology that demands estimating three time values for every task: an optimistic time, a most likely time, and a pessimistic time. These tasks must be mapped into a dependency network to determine the overall project likelihood and prevent the common error of padding a single number with uncertainty. Complementing this is the **'Eating the Elephant Technique,'** which models planning as an iterative cycle: 1) Analyze requirements; 2) Analyze risks (prioritizing the riskiest items early); 3) Design, implement, and integrate functionality; and 4) Validate the functionality with the users. Following each cycle, the team refines the initial guess of required iterations, increasing confidence with every measurable step.

When the goal is shifting, the **'Tracer Bullet'** approach is employed. Rather than attempting 'dead reckoning' by exhaustive upfront specification, this method requires continuous, real-time feedback. Effort must focus on the most critical requirements—those with the highest risks or deepest doubts—to quickly move the project from abstract concepts toward a tangible, working aspect. **Tracer code** must be written with an intention of permanence, containing rigorous error checking, structured documentation, and self-checking features, fundamentally differentiating it from disposable prototypes. For instance, the initial foundation, such as integrating a front-end framework or a server’s ability to generate SQL, can be built piece by piece, even if the functionality later expands, such as a UI team adding support for a new query type. This process also emphasizes **Early User Visibility and Buy-in**, wherein users view the system as immature (generating excitement) and actively contribute, increasing their ownership, while the core framework remains intact despite adding complex functionality.

### Advanced Debugging and Systematic Troubleshooting

Debugging requires adopting a mindful state, which involves turning off defensive mechanisms, tuning out project pressures, and achieving mental comfort. The first rule of debugging is to avoid panic; developers must resist the urge to fix only visible symptoms (myopia) and instead step back to determine the root cause, recognizing that the actual fault may be several steps removed. Before starting, developers should work on code that built cleanly, free of compiler warnings, as this is more productive than chasing theoretical problems. Furthermore, if bug reports originate from a third party, developers may need to observe the user who reported the bug in action to gather sufficient detail. For example, when testers found a large graphics application crashed only when painting a stroke using a specific brush, gathering all parties in the room and having the tester select the brush tool and paint a stroke from the upper right corner to the lower left corner replicated the crash. This mandates two critical behaviors: first, interviewing the user to acquire detailed data; and second, realizing that artificial tests (like a programmer's single brush stroke from bottom to top) are insufficient, requiring systematic, brutal testing of both boundary conditions and realistic end-user usage patterns.

To systematically find errors, developers must ensure the bug can be **reproduced**, ideally using a single command. If the bug cannot be reproduced, the fix cannot be verified. A powerful technique is the **Failing Test Before Fixing Code**: by forcing the isolation of the specific circumstances that display the bug, the act of writing the test itself can inform the solution. When examining incorrect values, developers should examine the local stack environment by knowing how to move up and down the call stack, and keeping pen and paper nearby helps prevent losing track of clues. If faced with an infinitely scrolling stack trace, a much quicker method than linear examination is the **binary chop** technique. By choosing a stack frame near the middle and checking for the error, the problem can be determined in at most six attempts, even with 64 frames. This technique applies equally to datasets: if bugs appear on certain data, the dataset should be split in half repeatedly until a minimum set of values consistently exhibits the problem. Similarly, when a bug is introduced during releases, a test should be created for the current release to fail, and then a version halfway between the current version and the last known working version should be selected to narrow the search scope. Furthermore, if a program crashes in production with a specific dataset, the developer must obtain a copy of that problematic dataset and binary chop the data itself until the input values leading to the crash are isolated.

When understanding program failure, developers must move beyond the current state, as stack traces typically fail to explain what the program was doing before reaching the current call chain, especially in event-based systems. **Tracing statements** provide necessary diagnostic messages (e.g., “got here” or “value of x = 2”) printed to a screen or file, remaining highly effective for diagnosing errors in concurrent, real-time, or event-based applications. Trace messages should adhere to a consistent, regular format to allow for automatic parsing, such as tracking every open and every close to identify a resource leak like unbalanced file operations. For diagnostics, the **Rubber Ducking method**—explaining the problem step by step to another person (or a rubber duck/inanimate object)—is highly effective, forcing the developer to explicitly state all assumptions. While bugs *might* reside in the OS, compiler, or third-party product, the general rule is that it is more productive to assume the bug resides within the application code, particularly assuming the application code incorrectly calls into a library rather than assuming the library is fundamentally broken. The developer must always eliminate their own application code before submitting a bug report, even if they initially suspect the system. Critically, if a system stops working after a developer 'changed only one thing,' that single change was likely responsible, directly or indirectly. Developers must also retest the system when external elements like new OS, compiler, or database versions change, recognizing that bugs for which a developer had a workaround may become fixed, thus breaking the workaround.

### Pragmatic Tools and Professional Growth

Developers are encouraged to master the **Command Shell** as the primary, superior workspace, surpassing reliance solely on GUIs and IDEs, because shell scripting allows developers to combine tools using pipes in complex ways and automate common tasks. Customizing the shell with aliases (e.g., `alias apt-up='sudo apt-get update && sudo apt-get upgrade'`) and functional prompts is a core best practice. For general text manipulation, developers should leverage tools like `awk` and `sed` (for Unix/macOS) or languages such as Python or Ruby, which are crucial enabling technologies. These languages allow rapid construction of utilities and prototyping ideas—jobs that might otherwise take five to ten times longer using conventional languages. For example, Kernighan and Pike built the same program using five languages, noting that the Perl version was the shortest, requiring only 17 lines versus 150 lines in C. The Pragmatic Bookshelf itself exemplifies the power of text manipulation, using Ruby for its build system (Rake tasks) to coordinate activities like including and highlighting code, ensuring the DRY principle is maintained by extracting named segments of source files instead of manual copy-pasting. Similarly, Python scripts handle LaTeX math markup conversion, and Ruby scripts manage index generation, all confirming that keeping content in plain text allows these languages to provide numerous functional benefits.

Finally, professional value is tied to the ability to manage a comprehensive **knowledge portfolio**. This requires continuous learning: committing to learning at least one new programming language annually to broaden conceptual thinking, and dedicating time to reading a technical book each month, eventually expanding to non-technical books to consider the human element. Developing reliability requires applying rigorous critical analysis to the diverse stream of consumed technical articles.
