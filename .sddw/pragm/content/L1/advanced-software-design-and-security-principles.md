# advanced-software-design-and-security-principles

**Parent:** [[content/L2/system-architecture-and-process-rigor|system-architecture-and-process-rigor]] — Modern software development requires adherence to architectural principles like DRY, orthogonality, and decoupling, which are enforced by mechanisms such as Interfaces and Protocols, and advanced techniques like Property-Based Testing (PBT) and Design by Contract (DBC). The process must prioritize iterative learning (treating requirements as 'invitations to explore'), algorithmic efficiency analysis using Big-O notation (e.g., distinguishing $O(N)$ from $O(N^2)$), and rigorous security practices (e.g., Principle of Least Privilege and adhering to NIST password guidelines).

The professional practice of modern software development demands an inherently adaptive and resilient approach, requiring meticulous attention to architectural principles, knowledge management, and process flow. This systemic approach transforms the development process into a continuous cycle of structured adaptation and rigorous critical thinking, demanding developers not only to write functional code but also to build systems that are secure, maintainable, and clear in their intent. 

### System Architecture, Design, and Maintainability

At the architectural level, developers must adhere to several foundational principles. The core concept of **DRY (Don't Repeat Yourself)** dictates that every single piece of knowledge within a system must maintain a single, unambiguous, and authoritative representation. Violations of DRY are dangerous because knowledge is inherently unstable, demanding constant maintenance across multiple specifications, processes, and programs when external changes occur—such as regulatory shifts, client meetings, or algorithm failures. For instance, if a `Line` class redundantly defines fields like `Point start`, `Point end`, and `double length`, the redundancy of `length` must be eliminated. Instead, the length must be derived and calculated using a method, such as `double length() { return start.distanceTo(end); }`. The principle applies broadly to **Data Sources**; developers should leverage introspection capabilities to minimize duplication by generating necessary containers directly from the schema, rather than writing manual storage code. While an **Alternative Data Structure Strategy** might propose using key/value data structures (like a map or dictionary) instead of fixed structures (structs or classes), this method critically requires a secondary defense: a simple table-driven validation suite to guarantee that all necessary data is present in the correct format. To maintain data integrity and align with best practices like Meyer’s Uniform Access principle, temporary optimizations, such as caching, must be strictly localized, and all external interactions must always be managed through accessor functions (e.g., `setStart`, `getStart`).

In terms of overall system design, the architecture must prioritize **orthogonality**, ensuring the system is composed of cooperating modules where each functions independently of the others. This localizes changes and significantly reduces systemic risk, thereby boosting productivity. Furthermore, developers must cultivate a mindset of **reversibility**, viewing critical architectural decisions—such as adopting a specific architectural pattern or committing to a third-party vendor’s database—as temporary drafts rather than permanent commitments. To maximize flexibility, three structured techniques are highly recommended: 1) **Abstraction:** proactively hiding all interactions with third-party APIs behind dedicated abstraction layers. 2) **Componentization:** systematically breaking down code into discrete, manageable components, irrespective of the deployment scale. 3) **Guidance:** ensuring the code remains always adaptable to future change, remembering that no decision is final.

**Decoupling** is central to this methodology. It emphasizes keeping separate concepts separate through sound design, since coupling links together things that must change in parallel, thereby complicating maintenance. Developers must treat mutable external resources—including databases, file systems, and service APIs—as constituting global data. Because any change in the global data’s implementation can couple the code across the system, any resource deemed important enough to be global must be wrapped in an API that the developers fully control. This control is often achieved using **Interfaces and Protocols** (or traits), which define abstract sets of required behaviors—for example, requiring `double getSpeed()` and `void stop()` for a `Drivable` type—without providing the underlying implementation code. This allows components, such as a `Car` or a `Phone`, to be treated generically and managed together in a list of items implementing that protocol.

### Code Validation, Testing, and Correctness

Testing methodologies are crucial for achieving a culture of reliability. Software testing is a fundamental part of the programming process, meaning developers must plan to thoroughly test all written software, regardless of whether the testing is done by the development team or by the eventual users. Treating test code with the same care applied to production code is mandatory, ensuring it remains decoupled, clean, and robust. Developers should avoid creating fragile tests by relying on unreliable aspects, such as the absolute position of widgets in a Graphical User Interface (GUI) system, exact timestamps in a server log, or the precise wording of error messages. Instead, the process should shift to having the computer automate testing, free from human preconceptions. 

A powerful technique is **Property-Based Testing**, which validates assumptions by defining properties that must hold true for all possible inputs, rather than writing specific test examples. When testing a sorting function, for example, a developer can establish properties such as the sorted list having the same number of elements as the original list (length is an invariant) and no element in the result being greater than the element that follows it. The process involves using a strategy to generate a wide array of random inputs—potentially running a test case one hundred times with a different random list each time—simulating the development of numerous individual tests. This method allows the automated test to fail due to a fault in a core assumption, even if the bug was not directly related to the feature being tested. Property-based tests fail in various ways: the test may pass, an assertion may fail, or the code may fail entirely because it cannot handle the inputs provided by the test generator. 

Because property-based tests generate random values, if a test fails, the developer should identify the specific parameters used by the test function and use those values to create a separate, regular unit test. This new unit test provides two vital benefits: first, it allows the developer to focus on the problem without the additional calls made by the property-based testing framework; and second, the unit test serves as a critical regression test, guaranteeing that the specific bug will not slip through, since the property-based test may not use the exact failing values on subsequent runs. Utilizing both property-based and unit testing is recommended because they address different concerns. Furthermore, the practice of property-based testing is beneficial for design because it forces the developer to think in terms of invariants and contracts—that is, thinking about what must remain unchanged and what must always be true—helping to remove edge cases and highlighting functions that store data in an inconsistent state.

In the context of **Design by Contract (DBC)**, code modules operate with formal contracts, ensuring that the code performs only what it claims to do. DBC guarantees that when input is provided, the module will make certain guarantees about the outputs it produces. Additionally, code invariants describe properties that must remain true about a piece of state when that state passes through a function. By incorporating these invariants and contracts—collectively referred to as properties—developers can automate testing using property-based testing.

### Security and Privilege Management

A core principle guiding robust system design is the **principle of least privilege**: developers must only use the minimum necessary permission level for the shortest amount of time. Developers should never automatically acquire the highest level of privilege, such as root or Administrator. If a high level of privilege is necessary, developers must take it, complete the minimum necessary work, and then quickly relinquish the privilege to reduce risk. This principle was articulated by Jerome Saltzer in 1974, guiding system design by advising that every program and every privileged user of the system should operate using the least amount of privilege necessary to complete the job. This concept extends beyond operating system privilege levels; applications should implement different, finely grained levels of access, allowing sensitive resources to be partitioned and granting individual users permissions for only certain categories, minimizing the attack surface. 

Security practices must begin with initial settings; default settings for applications or users on a site should be set to the most secure possible values. While secure defaults might not be the most convenient, it is preferable to leave the trade-off decisions between security and convenience to individual users. For instance, the default for password entry should be hiding the password by replacing each character with an asterisk, which is a sensible default in public settings. Developers must also encrypt sensitive data—including personally identifiable information, financial data, passwords, and other credentials—regardless of whether the data resides in a database or an external file. 

When managing project resources, developers must store all necessary project components under version control, with one major, critical exception: Never commit secrets, API keys, SSH keys, encryption passwords, or other credentials alongside the source code. Instead, these items must be managed separately, typically via config files or environment variables during the build and deployment process. 

Regarding password security, common security practices sometimes conflict with common sense. The National Institute of Standards and Technology (NIST) offers guidelines that developers should follow: 1) Do not restrict password length to less than 64 characters; NIST recommends 256 as a suitable maximum length. 2) Do not truncate a user's chosen password. 3) Accept all printable ASCII characters, space, and Unicode; developers should not restrict special characters like `[]();&%$#` or `/`. 4) Do not provide password hints to unauthenticated users, nor should the system prompt for specific types of information (e.g., “what was the name of your first pet?”). 5) Do not disable the paste function in the browser or password managers, because such actions fail to increase security and instead motivate users to create simpler, shorter passwords. 6) Do not impose arbitrary composition rules (e.g., mandating a specific mix of upper/lower case, numerics, or special characters) or prohibiting repeating characters. 7) Do not arbitrarily require users to change their passwords after a set length of time, restricting such changes only when there is a valid reason, such as after a security breach. Ultimately, developers should encourage long, random passwords with a high degree of entropy, as artificial constraints limit entropy and encourage poor habits. 

From a systems maintenance perspective, developers must apply security patches quickly. Waiting to update a system, even if it causes temporary application breakage, is a poor choice because the system remains vulnerable to a known exploit. This principle applies universally to every network-connected device, including phones, cars, appliances, personal laptops, developer machines, build machines, production servers, and cloud images, because historical data demonstrates that the largest data breaches were caused by systems that lagged behind on updates. 

Concerning cryptography, the most important rule is never to perform crypto work personally. Developers must only rely on reliable resources: well-vetted, thoroughly examined, well-maintained, frequently updated, and preferably open-source libraries and frameworks. For authentication, the pragmatic approach is to use a third-party authentication provider, such as services offered by email, phone, or social media providers, as these external providers specialize in system security, making them experts in a field the local developer is not.

### Naming Conventions and Code Clarity

In software development, developers must treat naming elements—including applications, subsystems, modules, functions, and variables—as critical because the names reveal significant information about the code's intent and underlying assumptions. Developers should adopt the practice of pausing to consider the 'motivation' or 'role' of every piece of code being created. Considering the role of a variable or function forces the developer to think about what is special about it, what specific task it performs, and what components it interacts with, which often clarifies the code's logic and makes the code more sensible. 

Naming improves clarity for both humans and machines. The human brain processes and understands written words very quickly, giving names a high priority when making sense of information. For example, when authenticating users for a site selling jewelry made from old graphics cards, using the variable `user` is clearer than using a generic name like `container` or `buyer`. Similarly, when defining a method that applies a discount, a specific name like `applyDiscount(Percentage discount)` is superior to `deductPercent(double amount)`, as the former clearly communicates the intent, and using a custom `Percentage` type parameter (instead of a raw `double`) documents the expected value range (e.g., between 0 and 100 or 0.0 and 1.0). Furthermore, for function naming, calling a function that calculates a number in the Fibonacci sequence as `Fib.of(n)` or `Fib.nth(n)` provides valuable context. 

While striving for clarity, developers must adhere to local coding culture. For instance, some programming languages traditionally use single letters like `i`, `j`, or `k` as loop increment variables in C, and adherence to this norm is expected even if the convention is not followed in other languages. Similarly, developers must honor local style preferences, whether preferring `camelCase` (with embedded capital letters) or `snake_case` (with embedded underscores). Developers must also be aware that some languages permit a subset of Unicode characters in names. 

Crucially, developers must prioritize consistency within a project's vocabulary; creating a project glossary listing terms with special team meaning helps the entire team use the jargon consistently. An effective strategy to encourage this is pair programming, as it helps jargon spread organically. Because code semantics can shift, developers must remain vigilant about updating names. When a name no longer expresses the intended meaning, it must be corrected immediately. If a developer encounters a misleading name, the foundational flaw—an ETC violation—must be fixed first, and *then* the offending component should be renamed to make the process easy and frequent.
