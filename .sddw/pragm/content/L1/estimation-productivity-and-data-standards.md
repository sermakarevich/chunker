# estimation-productivity-and-data-standards

**Parent:** [[content/L2/software-architecture-methodologies|software-architecture-methodologies]] — Software development requires adhering to principles like DRY (Don't Repeat Yourself) and orthogonality, ensuring knowledge is singularly represented and modules function independently; complex projects must use range-based scheduling techniques (PERT) and iterative 'Eating the Elephant' approaches, while development should utilize continuous 'Tracer Code' for visible, measurable progress rather than disposable prototypes.

The process of high-level software development requires developers to manage inherent uncertainties, maximize productivity, and structure their knowledge in portable, machine-readable formats. Effective practice necessitates adopting a deep understanding of estimation techniques, the utility of plain text, and mastery of the command shell environment.

**Handling Uncertainty and Project Scheduling:**

Developers are frequently confronted with the challenge of estimating project timelines, which can be difficult due to inherent uncertainties. Rather than providing a single number, best practice dictates using a range of scenarios, as demonstrated by the 'Painting the Missile Technique.' Historically, when the U.S. Navy planned the Polaris submarine project, the team utilized the Program Evaluation Review Technique (PERT), a methodology that employs three estimates for every task: an optimistic time, a most likely time, and a pessimistic time. Tasks must be organized into a dependency network, and simple statistics are then applied to determine the overall project likelihood. Using this range-based approach helps mitigate the common estimation error of padding a single number with uncertainty, allowing the statistics to distribute the overall project uncertainty and yield better estimates for the whole project. It is crucial to note, however, that despite PERT's quantitative nature, people often over-rely on the perceived mathematical authority of the technique without having prior experience. Furthermore, if an estimate proves inaccurate, the primary action must be to determine the root cause of the discrepancy, which may stem from selecting parameters that do not reflect reality or relying on a flawed underlying model.

Complementing this structured approach is the 'Eating the Elephant Technique,' which acknowledges that true project planning often requires practical experience gained *during* the project itself. This is achieved by adopting incremental development, repeatedly completing thin slices of functionality through a cycle of four steps: 1) Analyze requirements; 2) Analyze risks (and prioritize the riskiest items earlier); 3) Design, implement, and integrate the functionality; and 4) Validate the functionality with the users. Initially, a developer might only have a vague idea of the number of iterations required, and this initial guess is not suitable for all projects except the most trivial ones. The process mandates that the team completes the coding and testing of the initial functionality and marks this as the end of the first iteration. Based on this recorded experience, the team refines the initial guess regarding the number of required iterations and the scope of work included in each. This refinement increases the team's confidence and improves the schedule estimate with every cycle. The process of iterative estimation is analogous to the idiom of 'eating an elephant,' taking the endeavor one bite at a time. When questioned about an estimate, the best professional practice is to delay the response by stating, 'I will get back to you,' as going through the structured steps generally yields superior results compared to providing a quick, definitive estimate.

**Tooling, Data Format, and Knowledge Persistence:**

Developers require a basic, high-quality set of 'tools' to begin any project, similar to a woodworker who needs rules, gauges, saws, planes, chisels, drills, braces, mallets, and clamps. These tools must be intentionally selected, designed to perform specific jobs with minimal overlap, and should feel comfortable to use. While the initial setup is vital, the developer must commit to continuous learning and adaptation. Tools eventually become extensions of the developer's hands, improving productivity—the better the tools and the better the developer knows how to use them, the more productive the developer can be. When new requirements necessitate different capabilities, the developer must not be deterred by the limitations of a single, powerful tool, such as an Integrated Development Environment (IDE). The developer must maintain a mastery of the fundamental, core tool set, as the ability to operate beyond the constraints imposed by an IDE is paramount. The process of acquiring new tools must be driven by necessity; if the current tools cannot achieve a specific outcome, the developer must proactively seek a different or more powerful alternative.

Central to the developer's toolkit is the management of knowledge. Developers gather requirements as knowledge and then express this knowledge through designs, implementations, tests, and documents. The most crucial choice in data format is using plain text, which is defined as printable characters conveying information. Plain text is the preferred format because it offers superior portability and resilience compared to binary formats. The key advantage of plain text is its ability to be self-describing and independent of the application that created it. In contrast, binary formats often create an 'artificial divorce' where the necessary context to understand the data is separated from the data itself. Using plain text allows developers to give themselves the ability to manipulate knowledge manually and programmatically using virtually every available tool.

Plain text does not imply unstructured data; structured formats like JSON, HTML, and YAML are also plain text. These formats are vital because they provide 'insurance against obsolescence,' leveraging existing tools and facilitating easier testing. For instance, using descriptive tags like `<SOCIAL-SECURITY-NO>123-45-6789</SOCIAL-SECURITY-NO>` makes data extraction trivial and ensures the data's longevity, even long after the original application is defunct. Plain text offers two major advantages: first, it enables developers to leverage existing tools, as nearly every computing utility—from version control systems and editors to command-line tools—can operate on it. The Unix Philosophy pioneered this standard by creating small, sharp tools designed to perform one thing well, using the common underlying format of the line-oriented, plain-text file. This standardization is critical for maintaining system administration databases (like users and passwords) and allows for easy retrieval using commands like `grep -r backup /etc`. Secondly, plain text makes data easy to search and compare; file comparison tools such as `diff` and `fc` instantly reveal changes, while `sum` generates a checksum, monitoring for accidental or malicious modifications. The ubiquitous text file will remain the standard for all parties, even in future blockchain-based intelligent agents negotiating data interchange.

**The Command Shell as the Primary Workspace:**

The command shell serves as the developer's primary workspace, analogous to a woodworker's workbench. From the shell prompt, a developer can invoke a vast repertoire of tools, combining them using pipes in complex ways that original developers might not have conceived. Using the shell, developers can not only launch applications, debuggers, browsers, and editors, but they can also search for files, query system status, and filter output. Furthermore, the shell allows for the building of complex macro commands for frequently repeated activities. While Graphical User Interface (GUI) environments and Integrated Development Environments (IDEs) are excellent for simple operations (like moving files or building a project), relying exclusively on them prevents the developer from accessing the full capabilities of the environment. GUI environments are inherently limited to the specific capabilities defined by their designers; if a developer needs to operate beyond that defined model, they often face significant difficulty. For example, if an IDE needs to integrate a code preprocessor implementing design-by-contract, the IDE designer must explicitly provide the necessary hooks.

To maximize productivity, developers must gain familiarity with the command shell. This proficiency allows the developer to automate common tasks and utilize the full power of available tools. For instance, listing all unique package names imported by Java code can be achieved with a sequence of commands: `grep '^import ' *.java | sed -e's/.*import *//' -e's/;.*$//' | sort -u >list`. Investing energy in the shell quickly improves productivity. Developers should treat the shell like a customized home, a practice that encourages challenging manual tasks currently done via GUI and investigating their automation replacement. Customizing the shell is a core practice, involving configuring the terminal program with personalized elements such as color themes, and setting up functional prompts (e.g., including a shortened current directory name, version control status, and the time). Developers can greatly simplify workflows by creating aliases and shell functions. An example of an alias is creating `alias apt-up='sudo apt-get update && sudo apt-get upgrade'` for updating a Linux system. Similarly, setting `alias rm ='rm -iv'` ensures that accidentally deleting files requires confirmation. Most shells offer automatic command and file name completion via Tab, and this functionality can be extended to recognize entered commands and offer context-specific completions, including customizing completions based on the current directory. When moving to a new environment, developers should inquire about available shells and attempt to migrate their preferred environment, investigating alternatives if the current shell cannot solve a problem.
