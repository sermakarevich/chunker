# software-development-best-practices

The professional practice of software development requires adopting highly adaptable design methodologies and structured knowledge management to mitigate the inherent risks associated with technology’s accelerating pace of change. These principles govern everything from how technical knowledge is structured within a system (Don't Repeat Yourself - DRY) to how complex projects are scheduled, initiated, and maintained through iteration and continuous feedback.

### Foundational Architectural Principles and Knowledge Management

At the core of reliable development is adherence to **DRY (Don't Repeat Yourself)**, which mandates that every single piece of knowledge within a system must possess a single, unambiguous, and authoritative representation. Violating DRY—by expressing the same knowledge in two or more places—creates severe maintenance issues because knowledge is inherently unstable and requires constant adjustment due to factors like client meeting changes, regulatory updates, or algorithmic failures. This maintenance process requires developers to locate and adjust encapsulated knowledge across specifications, processes, and programs.

The principle of DRY extends beyond mere code. In data structures, for example, if a `Line` class initially defines redundant fields (such as `Point start`, `Point end`, and a `double length`), the `length` is inherently derived from the start and end points. The correct adherence is to eliminate the redundant `length` field and calculate it using a method, such as `double length() { return start.distanceTo(end); }`. Even when developers use caching for perceived performance optimization, the impact must be strictly localized, and external interactions must be managed via accessor functions (e.g., `setStart`, `getStart`) to simplify future additions and align with Meyer’s Uniform Access principle. The concept of orthogonality is equally paramount; a system should consist of cooperating modules, each functioning independently of the others. This greatly increases productivity by localizing changes and reducing risk.

Regarding interfaces to the external world, DRY violations are common. For **Across Internal APIs**, developers must centralize APIs in a single repository and utilize tools that specify the API in a neutral format, enabling the generation of documentation, mock APIs, and functional tests across multiple languages. For **Across External APIs**, formal documentation standards like OpenAPI should be leveraged; if these standards are absent, the most responsible action for programmers is to create and publish them. When dealing with **Data Sources**, minimizing duplication requires leveraging introspection capabilities to generate necessary containers directly from the schema, rather than writing manual storage code. An **Alternative Data Structure Strategy** suggests using key/value data structures (like a map or dictionary) instead of fixed structures (structs or classes), but this choice necessitates a second layer of defense: a simple table-driven validation suite to ensure all required data is present and correctly formatted. To achieve maximum system flexibility, developers must adopt a mindset of **reversibility**, treating critical architectural decisions (such as committing to a specific third-party vendor’s database) as temporary drafts rather than permanent commitments. This requires proactive use of **Abstraction**, hiding all interactions with third-party APIs behind dedicated abstraction layers, and **Componentization**, systematically breaking down code into discrete, manageable components, regardless of deployment scale.

### Project Lifecycle, Uncertainty, and Process Discipline

Managing uncertainty requires structured project scheduling. Instead of relying on a single estimate, best practice mandates using a range of scenarios. The **Program Evaluation Review Technique (PERT)**, originally utilized for the U.S. Navy's Polaris submarine project, is a methodology that requires estimating three time values for every task: an optimistic time, a most likely time, and a pessimistic time. These tasks must be organized into a dependency network to determine the overall project likelihood and prevent the common error of padding a single number with undue uncertainty. Complementing this is the **'Eating the Elephant Technique,'** which acknowledges that true planning must be done iteratively. This process involves repeatedly completing thin slices of functionality through a cycle: 1) Analyze requirements; 2) Analyze risks (and prioritize the riskiest items earlier); 3) Design, implement, and integrate the functionality; and 4) Validate the functionality with the users. After every cycle, the team refines the initial guess regarding the number of required iterations, thereby increasing confidence and improving the schedule estimate incrementally.

To achieve reliable outcomes when goals are shifting, the **'Tracer Bullet'** approach is employed. Rather than attempting 'dead reckoning' by specifying the system exhaustively upfront, this method demands continuous, real-time feedback. It requires identifying and focusing effort on the most critical requirements—those with the highest risks or deepest doubts—to move the project quickly from abstract requirements toward a tangible, working aspect of the final system.

Tracer code is fundamentally different from disposable prototypes. While prototyping (e.g., building a model out of balsa wood) generates code purely for lessons learned and risk exposure, tracer code is written with the intention of permanence. It must incorporate rigorous error checking, structured documentation, and self-checking features. When implemented properly, the initial foundation—such as integrating a front-end framework, necessary libraries, and a server's ability to generate SQL—can be built piece by piece, even if the functionality later expands (for instance, a UI team adding support for a new query type increases the associated library complexity). Furthermore, the process encourages **Early User Visibility and Buy-in**. By communicating progress, users view the system as immature (which often generates excitement) and actively participate in its development, increasing their sense of ownership. The core framework remains intact even as complex functionality is added, differentiating it from a simple prototype.

### Defect Detection and Assertive Programming

The necessity of anticipating failure is paramount. When program failures occur, recognizing that software defects (or bugs) are inherent because no one writes perfectly functioning software is critical. Other entities—people or systems—can detect issues before the developer is aware of the problem. Examples of detectable failures include receiving a `nil` value, finding an empty list, encountering a missing key in a hash, or determining that a value expected to be a hash is actually a list. Other detectable failures include network or filesystem errors leading to empty or corrupted data, or a logic error that, occurring millions of instructions after the root cause, changes the expected value of a case statement selector, causing the program to unexpectedly hit a default case. This necessitates that every case/switch statement include a default clause. Developers should adopt **Assertive Programming**, which suggests not relying on self-deception regarding impossible outcomes (like a negative parameter, a log failure, or a null result). Instead, whenever a developer thinks, 'of course that could never happen,' they must explicitly incorporate code to check for that possibility using assertions. For instance, if a parameter or result should never be null, the developer should explicitly assert for it using `assert (result != null)`.
