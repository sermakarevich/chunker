**Parent:** [[content/L2/system-architecture-and-process-rigor|system-architecture-and-process-rigor]] — Modern software development requires adherence to architectural principles like DRY, orthogonality, and decoupling, which are enforced by mechanisms such as Interfaces and Protocols, and advanced techniques like Property-Based Testing (PBT) and Design by Contract (DBC). The process must prioritize iterative learning (treating requirements as 'invitations to explore'), algorithmic efficiency analysis using Big-O notation (e.g., distinguishing $O(N)$ from $O(N^2)$), and rigorous security practices (e.g., Principle of Least Privilege and adhering to NIST password guidelines).

The practice of modern software development demands a highly adaptive, resilient, and deeply thoughtful approach that extends far beyond mere coding syntax. This requires meticulous attention to architectural principles, robust knowledge management, process flow, and continuous critical thinking to mitigate the risks inherent in technology’s rapidly accelerating pace of change. Developers must adopt a philosophy of structured adaptation, recognizing that few things in complex systems are permanent or fully defined from the outset.

### Semantic Precision and Naming Conventions

The foundation of clear code starts with naming elements. Developers must treat all naming conventions—including applications, subsystems, modules, functions, and variables—as critically important because the names must accurately reveal the code’s intent and underlying assumptions. The primary guidance is to adopt the practice of pausing to consider the 'motivation' or 'role' of every piece of code being created. Considering the role of a variable or function forces the developer to think about what is special about it, what specific task it performs, and what components it interacts with; this focused consideration significantly clarifies the code's logic and enhances its overall sensibility. This principle is amplified by the fact that the human brain processes written words very quickly, giving names a high processing priority when making sense of information.

In practice, this means selecting names that are highly specific. For instance, when authenticating users for a site selling jewelry made from old graphics cards, using the variable `user` is superior to using a generic name like `container` or `buyer`, as `user` consistently reinforces the person's role. Similarly, when defining a method that applies a discount, a specific name like `applyDiscount(Percentage discount)` is clearer than `deductPercent(double amount)`, because the name communicates the precise intent, and using a custom `Percentage` type parameter (instead of a raw `double`) documents the expected value range, such as between 0 and 100 or 0.0 and 1.0.

Beyond function names, developers should also analyze if the system's names are congruent with the actual terminology used by the people in the domain. If they are not, developers must recognize that this dissonance creates a Stroop-effect style cognitive dissonance for the entire team. When encountering a function or method with an overly generic name, the fix is to rename it explicitly to express all of the actions it truly performs, which ultimately makes it an easier target for subsequent refactoring. Developers should also consider using the more specific name, such as 'buyer', rather than the traditional and generic 'user'.

While developers must honor the local coding culture and established language conventions—adhering, for example, to the use of single letters like `i`, `j`, or `k` for loop increment variables in C—they must also maintain consistency. An effective strategy for ensuring project vocabulary consistency is to create a formal project glossary listing terms with special team meaning. Because code changes and meaning can subtly shift, developers must remain vigilant about updating names; if a name no longer expresses the intended meaning, it must be corrected immediately. If a developer encounters a misleading name, the fundamental design flaw (an ETC violation) must be fixed first, and *then* the offending component should be renamed to facilitate easy and frequent renaming.

### Architectural Resilience and Data Integrity

The technical architecture must adhere to several foundational principles, chief among them being **DRY (Don't Repeat Yourself)**. This principle mandates that every piece of knowledge within a system must maintain a single, unambiguous, and authoritative representation. Violations of DRY are problematic because knowledge is inherently unstable, requiring constant maintenance across specifications, processes, and programs due to external changes, such as client meetings, regulatory shifts, or algorithm failures. For example, if a `Line` class redundantly defines fields like `Point start`, `Point end`, and `double length`, the redundancy of `length` must be eliminated. Instead, the length should be derived and calculated using a method, such as `double length() { return start.distanceTo(end); }`. The principle extends to how data structures are managed; developers should leverage introspection capabilities when dealing with **Data Sources** to minimize duplication by generating necessary containers directly from the schema, rather than writing manual storage code.

Furthermore, when implementing modules, developers should prioritize **orthogonality**, ensuring the system is composed of cooperating modules where each functions independently of the others. This localized change significantly reduces systemic risk and increases productivity. Architects must cultivate a mindset of **reversibility**, viewing critical decisions—such as committing to a specific third-party vendor’s database or adopting a specific architectural pattern—as temporary drafts rather than permanent commitments. To maximize flexibility, three structured techniques are guided: 1) **Abstraction:** proactively hiding all interactions with third-party APIs behind dedicated abstraction layers. 2) **Componentization:** systematically breaking down code into discrete, manageable components, irrespective of the deployment scale. 3) **Guidance:** ensuring the code remains always adaptable to change, recognizing that there are no final decisions.

This resilience requires adherence to **Decoupling**, which emphasizes keeping separate concepts separate through sound design principles. Coupling links together things that must change in parallel, making maintenance difficult. Developers must treat mutable external resources—such as databases, file systems, and service APIs—as constituting global data. Since any change in the global data’s implementation can couple the code across the entire system, therefore, any resource important enough to be global must be wrapped in an API that the developers control. Techniques to achieve this include **Interfaces and Protocols** (or traits), which specify abstract sets of required behaviors—for example, `Drivable` might require `double getSpeed()` and `void stop()`—without providing implementation code. This allows components to be managed generically, such as listing both a `Car` and a `Phone` if they both implement a `Locatable` protocol. Finally, maintaining data integrity and alignment with best practices like Meyer’s Uniform Access principle mandates that the impact of temporary optimizations, such as caching, must be strictly localized, and external interactions must always be managed through accessor functions (e.g., `setStart`, `getStart`).

### Requirements Discovery and Management

Modern software development requires developers to adopt a highly inquisitive, adaptive role in understanding needs, moving away from the outdated process of relying on rigid, exhaustive documentation. The reality is that requirements rarely appear on the surface; they are often deeply buried beneath layers of assumptions, misconceptions, politics, or sometimes they may not truly exist at all.

Developers must understand that simply being told what to do, or merely listening to users, is insufficient. The most valuable attribute of a developer is helping people understand what they actually want. When clients present a need—whether strategic or tactical—it should be treated not as an absolute requirement, but as an *invitation to explore*. The process must be one of continuous discovery, where the developer questions the initial statement of need by looking for potential loopholes and edge cases. For instance, if a publisher mandates free shipping on orders of $50 or more, the developer must ask whether the $50 total includes tax or current shipping charges, if the limit applies only to paper books or if it can include ebooks, what kind of shipping is offered (e.g., Priority or Ground), and how frequently the $50 limit might change. By interpreting the client's responses and providing continuous feedback, the developer guides the client to refine their own thinking. This exploration phase is critical, as ignoring potential loopholes can compromise the client's original intent. 

When facing seemingly impossible or complex projects, developers must leverage collective problem-solving, which means working together to solve problems while coding. The Agile Manifesto champions 'Individuals and interactions over processes and tools,' emphasizing that no specific process or tool can replace critical thinking, enabling the team to avoid 'analysis paralysis.'

Historically, the process involved obtaining a requirements specification, developing a design document, creating flowcharts and pseudo code, and writing code—a model that was costly and constrained by limited early machine capabilities. This process, which required achieving exact specifications, is generally flawed for the modern, messy, conflicted, and unknown world. Consequently, the developer must recognize that exact specifications are rare, if not impossible.

Regarding documentation, developers must treat working code as the best—perhaps the only—form of requirements documentation. However, structured documents should not be considered final deliverables for client sign-off. Detailed, monolithic requirements documents (like those expanding a two-minute explanation into inch-thick masterpieces) are dangerous. They are flawed because the client does not truly know what they want initially, and even if signed off, the client rarely reads the detailed technical document because they are motivated by solving a high-level problem, while programmers are interested in all the nuances. Instead, the preferred form for managing requirements is a short description, often called a **user story**, which fits on a virtual index card. These stories describe what a small portion of the application should do from the perspective of a user, and placing them on a board allows developers to track both status and priority.

While developers must document their understanding of client needs, these documents should function merely as mileposts to guide the implementation process and should not be the core source of truth. Crucially, requirements must be **abstract**; the simplest statement that accurately reflects the business need is best. While developers must capture underlying semantic invariants as requirements, current work practices or policies should be documented separately. Stating a policy, such as 'Only authorized users may access an employee record,' naturally leads to designing and implementing an access control system, and in this case, only the policy metadata needs updating, allowing the system to be well factored to support metadata.

Finally, successful requirements gathering must account for the human element, remembering that successful tools must adapt to the hands that use them. Early feedback, using prototypes or tracer bullets, allows clients to respond with a more realistic assessment, such as, “yes, it does what I want, but not how I want.”
