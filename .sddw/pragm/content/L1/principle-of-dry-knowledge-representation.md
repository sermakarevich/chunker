# principle-of-dry-knowledge-representation

**Parent:** [[content/L2/software-architecture-methodologies|software-architecture-methodologies]] — Software development requires adhering to principles like DRY (Don't Repeat Yourself) and orthogonality, ensuring knowledge is singularly represented and modules function independently; complex projects must use range-based scheduling techniques (PERT) and iterative 'Eating the Elephant' approaches, while development should utilize continuous 'Tracer Code' for visible, measurable progress rather than disposable prototypes.

The principle of Don’t Repeat Yourself (DRY) is a cornerstone concept, viewed as one of the most critical tools in a Pragmatic Programmer’s toolbox. At its core, DRY mandates that every single piece of knowledge within a system must possess a single, unambiguous, and authoritative representation. Violating this principle—having the same knowledge expressed in two or more places—is analogous to giving a computer two contradictory pieces of information, which can cause the entire program to fail. Since knowledge is inherently unstable and changes rapidly (due to client meeting adjustments, regulatory updates from governments, or algorithm failures revealed by testing), programmers spend a considerable amount of time in a state of maintenance. This maintenance is not solely confined to bug fixing or adding features after release; rather, it is a constant, routine part of the entire development process, driven by daily changes in understanding, new requirements, and the continuous evolution of existing ones due to changes in the operating environment. When performing maintenance, developers must locate and adjust the encapsulated pieces of knowledge embedded throughout the application, which is complicated because duplicating knowledge across specifications, processes, and programs is easy and leads to a 'maintenance nightmare' long before deployment.

To develop software reliably and ensure maintainability, developers must strictly adhere to the DRY principle, ensuring that any knowledge element is represented only once within the system's authoritative source. The concept of DRY extends far beyond merely writing code; it pertains to all representations of knowledge within the entire system.

In the context of data structures, adherence to DRY is crucial. For example, if a `Line` class is initially defined with `Point start`, `Point end`, and `double length`, this structure violates DRY because the `length` is inherently derived from the start and end points. Changing either point necessitates a corresponding recalculation and change to the length. A better practice, therefore, is to eliminate the redundant field and instead calculate the length using a method: `double length() { return start.distanceTo(end); }`. However, developers sometimes violate DRY for performance optimization, such as when caching data to avoid repeating expensive operations. In such scenarios, the key is to strictly localize the impact of the violation, ensuring that the improper state management is not exposed to the outside world. The implementation provided for a class with localized violation demonstrates this: the internal state requires private fields (`private double length`, `private Point start`, `private Point end`), and all external interactions must be managed via accessor functions (`setStart`, `setEnd`, `getStart`, `getEnd`, `getLength`). This practice is vital because when a module exposes a data structure, it couples all the code that uses that structure to the module's internal implementation. The use of accessor functions is essential because it simplifies future functionality additions and aligns with Meyer’s Uniform Access principle, which states that all services offered by a module must be accessible through a uniform notation, regardless of whether those services are implemented via storage or computation.

DRY violations also appear in how code interfaces with the external world. When writing code, the code interacts with external entities—be it through APIs of other libraries, remote service calls, or external data sources. Each of these interfaces risks introducing a DRY violation because the programmer’s code must possess knowledge that is also present in the external entity. Specifically, the code must independently maintain knowledge of the external representation, such as the API structure, the data schema, or the meaning of error codes. This kind of duplication is unavoidable, but several strategies exist to mitigate the risk. When dealing with **Across Internal APIs**, programmers should use tools that allow specifying the API in a neutral format, which can then generate documentation, mock APIs, functional tests, and API clients available in multiple languages. Ideally, all APIs should be centralized in a single repository for shared usage across teams. For **Across External APIs**, the formal documentation standards like OpenAPI are increasingly used, allowing developers to import the API specification into local tools for more reliable integration. If formal specifications are lacking, the most responsible action is for programmers to create and publish one, which assists both internal developers and facilitates external assistance in maintenance. 

When dealing with **Data Sources**, many sources provide the capability for introspection on their data schema. This capability minimizes the duplication between the data source and the application code. Instead of writing manual code to store data, developers can generate necessary containers directly from the schema, a process often handled automatically by modern persistence frameworks. An **Alternative Data Structure Strategy** involves avoiding fixed structures (like a struct or class) for external data, opting instead to load the data into a key/value data structure (such as a map, hash, or dictionary). While this approach carries the inherent risk of reducing the security of knowing the exact data structure, the developer must implement a second layer of defense: a simple table-driven validation suite to verify that the created map contains at least the required data in the required format. 

Perhaps the most challenging type of duplication, **Inter-Developer Duplication**, occurs between different developers on a project. Entire sets of functionality may be inadvertently replicated across the codebase, a duplication that could remain undetected for years, resulting in severe maintenance issues. For example, an audit of U.S. governmental computer systems for Y2K compliance uncovered over 10,000 programs, each containing a unique version of Social Security Number validation code. Addressing this duplication requires effort on two levels: The team must cultivate strong, tightly knit units with excellent communication practices at a high level. At the module level, the problem is more subtle, involving commonly needed functionality or data that doesn't fit into an obvious area of responsibility but is implemented multiple times. The most effective method for addressing inter-developer duplication is promoting active and frequent communication among all developers. Practical steps include establishing daily scrum standup meetings, setting up forums (like Slack channels) for discussing common problems (providing a non-intrusive, permanent record), appointing a team member to act as a project librarian to facilitate knowledge exchange, establishing a central source tree location for utility routines and scripts, and maintaining the habit of consistently reviewing other people's source code and documentation, either informally or during formal code reviews. Crucially, the developer must remember that this access and review process must be reciprocal, meaning the developer should also allow other team members to examine their own code.
