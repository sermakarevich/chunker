# development-knowledge-management-techniques

**Parent:** [[content/L2/advanced-software-development-practice|advanced-software-development-practice]] — Professional development requires strict adherence to DRY principles and high orthogonality, utilizing methods like PERT and the 'Tracer Bullet' for planning, and mastering specialized techniques like binary chopping and 'Rubber Ducking' for debugging. Knowledge management demands prioritizing plain text persistence, using the Command Shell as the primary workspace, and employing VCS universally for version control and recovery.

The professional practice of software development demands an adaptive and rigorous approach to managing knowledge, mitigating risks inherent in the technology's accelerating pace of change. These requirements span architectural design, project scheduling, tooling mastery, and the continuous management of human knowledge.

### Architectural and Knowledge Management Principles

Central to reliable development is the strict adherence to **DRY (Don't Repeat Yourself)**, which mandates that every single piece of knowledge within a system must possess a single, unambiguous, and authoritative representation. Violating DRY—by expressing the same knowledge in multiple locations—is highly problematic because knowledge is unstable and requires constant, routine maintenance to accommodate adjustments from client meetings, regulatory changes, or algorithm failures. This maintenance effort compels developers to locate and adjust encapsulated knowledge across specifications, processes, and programs. This principle extends beyond mere code representation to all forms of knowledge within the entire system.

In data structures, DRY adherence means eliminating redundant fields. For instance, if a `Line` class initially defined `Point start`, `Point end`, and `double length`, the `length` field is redundant, as it is mathematically derived from the other points. The correct practice is to remove the redundant field and calculate it using a method: `double length() { return start.distanceTo(end); }`. Even when developers violate DRY for apparent performance gains, such as through caching, the resulting impact must be strictly localized. External interactions must be managed via accessor functions (e.g., `setStart`, `getStart`) to simplify future expansions and align with Meyer’s Uniform Access principle. 

When interfacing with external systems, DRY violations are also common. For **Across Internal APIs**, developers must centralize these APIs in a single repository and utilize tools that specify the API in a neutral format. This standardization enables the generation of documentation, mock APIs, and functional tests across multiple programming languages. For **Across External APIs**, formal documentation standards, such as OpenAPI, must be used; if such standards are absent, the most responsible course of action for programmers is to proactively create and publish them. When dealing with **Data Sources**, developers should leverage introspection capabilities to minimize duplication by generating necessary data containers directly from the schema, rather than writing manual storage code. An **Alternative Data Structure Strategy** involves employing key/value data structures (like a map or dictionary) instead of fixed structures (structs or classes), but this requires a crucial secondary defense: a simple table-driven validation suite to guarantee that all required data is present and formatted correctly.

The system must also possess high orthogonality. Orthogonality dictates that the system is composed of cooperating modules where each module functions independently of the others. This architectural quality significantly enhances productivity by localizing changes and mitigating overall risk. Furthermore, developers must adopt a mindset of **reversibility**, treating critical development decisions (such as committing to a specific third-party vendor’s database) as temporary drafts rather than permanent commitments. To ensure maximum flexibility, the strategy involves **Abstraction**, proactively hiding all interactions with third-party APIs behind dedicated abstraction layers, and **Componentization**, systematically breaking down code into discrete, manageable components, regardless of the project’s scale.

### Project Development, Uncertainty, and Iteration

Handling project uncertainty requires abandoning single-estimate forecasting in favor of structured, range-based approaches. The **Program Evaluation Review Technique (PERT)**, originally developed for the U.S. Navy's Polaris submarine project, is a methodology demanding the estimation of three time values for every task: an optimistic time, a most likely time, and a pessimistic time. These tasks must be organized into a dependency network to accurately determine the overall project likelihood, thereby avoiding the error of arbitrarily padding a single number with uncertainty.

Complementing PERT is the **'Eating the Elephant Technique,'** which acknowledges that planning must proceed through iterative cycles. This process involves repeatedly completing thin slices of functionality through a structured cycle: 1) Analyze requirements; 2) Analyze risks (with priority given to the riskiest items first); 3) Design, implement, and integrate the functionality; and 4) Validate the functionality with the end users. With every completed cycle, the team refines its initial guess regarding the total number of required iterations, which increases confidence and refines the schedule estimate iteratively.

When the ultimate goal or requirements are constantly shifting, the **'Tracer Bullet'** approach is utilized. Instead of attempting exhaustive upfront specification ('dead reckoning'), the tracer bullet method requires continuous, real-time feedback. This involves identifying and concentrating development effort on the most critical requirements—those with the deepest doubts or highest risks—to rapidly transition the project from abstract theoretical requirements toward a tangible, working aspect of the final system.

Tracer code is distinct from disposable prototypes. While prototyping (such as building a model out of balsa wood) aims solely for lessons learned and risk exposure, tracer code is written with the explicit intention of permanence. It must incorporate rigorous error checking, structured documentation, and self-checking features. Properly implemented, the initial foundational elements—including integrating a front-end framework, necessary libraries, or a server's ability to generate SQL—can be built incrementally, even if the system's functionality expands later (for example, a UI team adding support for a new query type necessitates increased complexity in the associated library).

Furthermore, the process emphasizes **Early User Visibility and Buy-in**. By regularly communicating progress, users view the system as immature (a state that typically generates excitement) and become active contributors, thereby increasing their sense of ownership. This ensures the core framework remains robust even as complex functionality is overlaid, distinguishing it from a temporary prototype.

### Tooling, Text, and Persistence Mastery

The professional developer must master their development toolkit, committing to continuous learning and adaptation, even when constrained by powerful tools like an Integrated Development Environment (IDE). The **Command Shell** is recognized as the primary and superior workspace compared to relying exclusively on GUIs and IDEs. This superiority stems from the shell’s ability to combine diverse tools using pipes in complex ways and automate common tasks, dramatically improving productivity. Core practices include customizing the shell with aliases (e.g., `alias apt-up='sudo apt-get update && sudo apt-get upgrade'`) and setting up functional prompts.

When managing knowledge persistence, the paramount choice is the use of **plain text**. Plain text offers superior portability and resilience compared to binary formats. Its critical advantage is its ability to be self-describing and independent of the application that initially created it, thus preventing an 'artificial divorce' between data and its context. Although structured formats like JSON, HTML, and YAML are utilized, they remain plain text, providing 'insurance against obsolescence.' Plain text enables developers to manipulate knowledge both manually and programmatically using virtually every available utility, reviving the Unix Philosophy of small, sharp, line-oriented, plain-text tools, which allows comparison tools like `diff` to instantly reveal changes, and utilities like `grep -r backup /etc` to function effectively.

Finally, the developer’s professional value is intrinsically linked to the ability to manage a comprehensive **knowledge portfolio**. This necessitates continuous learning, specifically committing to learning at least one new programming language every year to broaden conceptual thinking, and dedicating time to reading a technical book every month. This intellectual pursuit eventually expands to non-technical literature, encouraging consideration of the human element. Developing a reliable understanding requires applying rigorous critical analysis to the diverse stream of technical articles consumed.

### Collaborative Development and Version Control

Version Control Systems (VCS) are indispensable mechanisms for both collaboration and maintaining the complete history of software development. A VCS functions not only as a historical repository of changes, far surpassing the single-user 'undo' key, but also as a central platform for team collaboration, deployment pipelines, and issue tracking.

**Critical Warnings against Poor VCS Practices:**
1. **Shared Directories:** Sharing project source files across a network (whether internal or via cloud storage) is unviable, as teams risk constantly corrupting work, losing changes, and breaking builds. Users must rely exclusively on version control.
2. **Network/Cloud Repositories:** Storing the main repository on a network or cloud drive is even riskier. Since the VCS operates using a complex set of interacting files and directories, simultaneous changes from two different instances can corrupt the overall state, potentially causing significant damage.

**Core VCS Functions and Best Practices:**
*   **Tracking Changes:** VCS systems meticulously record every change in source code and documentation, allowing users to revert to any previous version. They provide granular data essential for audits and quality assessments, answering questions like: Who modified this line? What is the difference between the current version and last week's? How many lines changed in this release? And which files are modified most often? This data is invaluable for bug-tracking, performance analysis, and quality assessment. 
*   **Releases:** VCS systems inherently identify software releases, guaranteeing that a specific release can always be regenerated independently of any later changes.
*   **Collaboration:** VCS enables multiple users to work concurrently on the same files, even making changes to the same file simultaneously. The system robustly manages the merging of these changes when the files are submitted back to the repository, a methodology proven effective for projects of all sizes.
*   **Universal Scope:** Version control must be used for absolutely all work—even single-person, one-week 'throw-away' prototypes. This includes not only source code but also documentation, phone number lists, vendor memos, makefiles, build and release procedures, and simple shell scripts. Ideally, all day-to-day work must be secured in a repository.
*   **Branching:** A powerful feature is the ability to isolate development into specialized branches. Users can create a branch at any point in the project’s history, and all work remains isolated from all other branches. The completed work can then be merged back into the target branch. Since multiple people can work on a single branch, branches function effectively as small, dedicated clone projects. This isolation is crucial: if one developer develops feature A in one branch and a teammate develops feature B in a separate branch, the work cannot interfere with each other.

**Advanced Integration and Readiness:**
While VCS can be decentralized (peer-to-peer cooperation), establishing a central repository is highly recommended. A central host provides access to integrations that streamline project flow. Key features of a hosting provider include: good security and access control; an intuitive user interface (UI); the ability to perform operations via the command line (essential for automation); automated builds and tests; strong support for branch merging (often called pull requests); integrated issue management (ideally linking issues to commits and merges for metrics tracking); and robust reporting (e.g., a Kanban board display for pending tasks). Many advanced teams configure their VCS such that a simple push to a specific branch automatically triggers a build, runs tests, and, if successful, deploys the new code into production, guaranteeing the ability to roll back.

Developers must proactively learn necessary commands for recovery and treat the entire laptop environment—including installed applications, system configuration, and user preferences (dotfiles)—as items needing capture in a VCS (hosted off the laptop) for disaster recovery.

### Debugging and Editor Fluency

Debugging requires recognizing that software defects, or 'bugs,' are inherent because no one writes perfectly functioning software, stemming from misunderstood requirements or coding errors. Since modern computer systems can only execute what they are told, developers must anticipate that debugging will consume a major portion of the workday. Debugging must be treated as general problem-solving, not a puzzle. Upon encountering a problem, the focus must be on fixing the root problem, rather than assigning blame. Before starting, developers must maintain a mindful state, eliminating defensive mechanisms, tuning out project pressures, and becoming mentally comfortable. The first rule is to avoid panic; instead, one must step back to analyze potential root causes behind the observed symptoms, resisting the danger of myopia.

Developers must gather all relevant data, as bug reporting is not an exact science. If reports originate from a third party, it may be necessary to observe the user who reported the bug in action to acquire sufficient detail. For instance, one case demonstrated that while a programmer claimed an application worked with a specific brush, a live demonstration by the tester painting a stroke from the upper right corner to the lower left corner caused the application to crash. This highlights two crucial points: the need to interview the user who reported the bug for detailed data, and the failure of artificial tests (like a single bottom-to-top brush stroke) to adequately simulate realistic end-user usage patterns, requiring systematic testing of both boundary conditions and real-world scenarios. When debugging, the developer must focus on understanding what the program *thinks* is occurring, rather than merely observing the visible output.

In parallel, developing **editor fluency** is essential because text is the basic raw material for programming. Achieving fluency significantly benefits development; even a 4% efficiency gain over 20 hours a week can yield an additional week of work per year. More critically, fluency minimizes the gap between a thought and its representation in the editor buffer, allowing programming thoughts to flow naturally. Fluency involves moving and making selections by character, word, line, and paragraph; for code, it means moving by syntactic units (e.g., matching delimiters, functions, or modules), reindenting, commenting/uncommenting blocks instantly, and performing undo/redo. 

Achieving fluency is less about mastering every command in one editor and more about learning commands that simplify life. The process involves observing repetitive actions and asking, “There must be a better way.” Once a useful feature is found, it must be solidified into muscle memory through conscious, frequent repetition. Developers should tackle specific challenges: first, disabling autorepeat and learning keyboard key sequences to manipulate text by blocks; second, spending one week editing using only the keyboard, banning the mouse, to force the learning of shortcuts; and third, seeking to integrate tools, such as previewing a PDF layout directly within the editor buffer, building the feature if a plugin does not exist.
