# program-as-transformation-pipeline

**Parent:** [[content/L2/software-development-architecture-principles|software-development-architecture-principles]] — Software development mandates adherence to DRY, reversibility, and decoupling principles, utilizing techniques like interfaces/protocols, componentization, and abstraction to manage complexity, while the data flow must be treated as a continuous, trans-functional pipeline that requires structured error handling and rigorous external API documentation (e.g., OpenAPI).

The modern practice of designing robust, adaptable software necessitates adopting a core perspective that views programs fundamentally as processes that transform data from an initial input into a final output. This transformative model provides significant architectural advantages, allowing developers to structure code as a chain of operations, similar to an industrial assembly line. This approach is strongly reinforced by the Unix philosophy, which champions the use of small, sharp, line-oriented, plain-text tools, enabling the chaining of multiple distinct transformations. A classic illustration is finding the five longest files in a directory tree, which involves a precise sequence of transformations: directory name $	o$ list of files $	o$ list with line numbers $	o$ sorted list $	o$ highest five + total $	o$ highest five. The general mechanism involves piping standard output from one tool to the standard input of the next, as demonstrated by the command pipeline: `find . -type f | xargs wc -l | sort -n | tail -5`. The process of data transformation is central; when analyzing any data flow, defining the required inputs and outputs is the most effective starting point. For example, designing an anagram finder from the letters 'lvyin' has an initial input (e.g., 'ylvin'), and the output is a list of possible words (e.g., 'ivy, lin, nil, yin'). The solution decomposes into required transformations: 1) Initial input ('ylvin'); 2) All combinations of three or more letters; 3) Signatures of those combinations; and 4) The resulting list of dictionary words matching those signatures. Even the initial step of generating combinations involves a pipeline of transformations: Initial input (e.g., 'vinyl') $	o$ Convert to characters (v, i, n, y, l) $	o$ Get all subsets (e.g., $	ext{[]}, [v], [i], 	ext{etc.} 	o [v,i,y], 	ext{etc.} 	o [v,n,y,l], [i,n,y,l], 	ext{etc., and finally } [v,i,n,y,l]$) $	o$ Filter for subsets longer than three characters (e.g., $[v,i,n], [v,i,y], 	ext{etc., and } [v,i,n,y,l]$) $	o$ Convert back to strings (e.g., $	ext{vin}, 	ext{viy}, 	ext{etc., and } 	ext{vinyl}$). Many functional languages support the pipeline operator (`|>`), which conceptually represents data flowing from one transformation to the next, enabling highly readable code through function chaining. When implementing such chains, developers can sequentially write code by generating the output of one function and passing it as the input to the next, such as `const content = File.read(file_name);` followed by `const lines = find_matching_lines(content, pattern)` and then `const result = truncate_lines(lines)`. This method treats data as a continuous flow, akin to a river, forming a pipeline where data becomes a peer to functionality: a sequence of code $	o$ data $	o$ code $	o$ data... This approach enhances reusability, as a function can be used anywhere its parameters match the output of some preceding function, thus significantly reducing coupling. Such a flow is particularly beneficial when addressing concerns like finding the top five longest files. The transformed data flow sequence maps the requirement to a series of steps: directory name $	o$ list of files $	o$ list with line numbers $	o$ sorted list $	o$ highest five + total $	o$ highest five. When considering the implementation of transformations, especially regarding data flow and error handling, conventions must be strictly followed: raw values should never pass directly between transformations. Instead, the values must be wrapped in a dedicated data structure or type, which explicitly indicates whether the contained value is valid. For instance, languages like Haskell use `Maybe`, while F# and Scala utilize `Option` for this purpose. Developers have two primary strategies for error management within a pipeline: they can incorporate error checking inside each individual transformation function, or they can rely on the language structure. In languages that support pattern matching in function calls, such as Elixir, the language can be designed to automatically manage errors, causing an error at any point in the pipeline to immediately become the final value. If pattern matching is not supported, specialized helper functions (bind functions) are required to defer the execution of subsequent pipeline functions until the preceding steps have successfully completed, ensuring the accuracy of the entire data flow and error handling process. The transformational model also has profound implications for object-oriented design patterns, offering alternatives to traditional inheritance that mitigate coupling risks. Inheritance, though it originated with concepts like Simula 67 (using prefix classes to allow both `car` and `bicycle` to be added to a list of waiting items), is problematic. While Smalltalk focused on using inheritance for purely behavioral design, modern OO languages often use it either to avoid typing or express class relationships (e.g., declaring `Car` is a type of `Vehicle`). The core issues with inheritance are twofold. First, for sharing code, it introduces substantial coupling not only between the child and parent but also across multiple ancestors. If a base class (`Vehicle`) changes its API—for example, renaming `move_at(speed)` to `set_velocity` and changing `@speed` to `@velocity`—it breaks the top-level code that assumes it is working with a `Car`, even though the name of the instance variable is merely an internal detail. Second, viewing inheritance solely as a mechanism for defining types (as seen in layer-upon-layer class hierarchy diagrams) increases brittleness. A worse issue arises with multiple inheritance; while a `Car` might be a type of `Vehicle`, it could equally be a type of `Asset` or `LoanCollateral`, which would necessitate multiple inheritance. To avoid these pitfalls, developers should favor the use of interfaces and protocols. Interfaces (in Java) or protocols/traits (in other languages) specify abstract sets of required behaviors—for example, `Drivable` might require `double getSpeed()` and `void stop()`—without providing implementation code. This establishes a contract: any class that implements the interface must include all specified methods. This structure allows any class implementing the interface (like both `Car` and `Phone` implementing `Locatable`) to be safely stored together in a list of that abstract type, enabling safe processing. Interfaces enable polymorphism without the constraints of inheritance. Instead of forcing a subclass to carry all the unused methods of a large parent class (e.g., an `Account` class inheriting from a massive `PersistenceBaseClass`), developers can use **Delegation**. In delegation, the `Account` class focuses solely on its core business logic, and a separate `AccountRecord` class handles persistence, calling the methods via delegation (`@repo.save()`). This completely isolates the client from the framework's internal API. To mitigate the boilerplate code that arises with delegation (such as writing a `find` method for every record class), developers should adopt **Mixins** and **Traits**. Mixins allow extending a class or object with new functionality without using inheritance. They create a set of functions and then append that set to an existing class, such as defining `class AccountRecord extends BasicRecord with CommonFinders` or `class OrderRecord extends BasicRecord with CommonFinders`, where `CommonFinders` supplies common methods like `find(id)` and `findAll()`. This principle of using interfaces/protocols, delegation, or mixins/traits is a language-agnostic capability and the best choice is always the technique that most accurately expresses the developer's architectural intent.
